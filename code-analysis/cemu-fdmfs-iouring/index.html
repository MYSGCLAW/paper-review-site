FDMFS + io_uring COPY_FILE_RANGE 오프로드 경로 분석

한 줄 요약

FDMFS는 CEMU CSD의 온보드 메모리(FDM, Functional Data Memory)를 얇은 first-fit 할당자 기반 VFS 파일시스템으로 노출하고, copy_file_range를 통해 호스트 파일과 이 디바이스 메모리 사이를 zero-copy로 오가며, IORING_OP_COPY_FILE_RANGE와 IOCB_HIPRI/iopoll 경로를 이용해 이 복사를 io_uring 비동기·busy-poll 방식으로 완료 통지한다.

분석 범위

job.json이 지정한 12개 경로(fs/fdmfs/*, io_uring/copy_file_range.{c,h}, io_uring/opdef.c, io_uring/Makefile, include/uapi/linux/io_uring.h, include/uapi/linux/magic.h, include/trace/events/fdmfs.h, fs/read_write.c, include/linux/fs.h, block/bio.c, block/blk-core.c)를 모두 읽었다. block/bio.c·block/blk-core.c는 checkout 범위에서는 CEMU 고유 로직이 없고, FDMFS/io_uring이 재사용하는 표준 커널 iopoll 인프라(iocb_bio_iopoll, bio_poll)만 담고 있음을 확인했다 — 이 부분은 "확인함" 항목이다.

이 세션에는 Bash 도구가 제공되지 않아 git log로 실제 커밋 메시지(fix fdmfs performance bug on io_uring read)의 diff를 직접 확인하지 못했다. repo/.git는 존재하지만 reflog에는 clone 기록 한 줄뿐이고 pack 객체를 디코딩할 수단이 없다. 해당 커밋 관련 서술은 현재 코드에서 관찰되는 정황 증거에 기반한 추정이며, 본문에 명확히 표시했다.

참고문서(references/text/cemu__csd-README.txt)는 FDMFS를 libfuse 기반 사용자공간 파일시스템(hw/femu/csd/fuse)으로 설명하는데, 실제 분석 대상 코드는 fs/fdmfs/*.c로 구현된 정식 커널 VFS 모듈이다. 이는 문서와 코드의 버전/구현 방식 불일치이며 별도 항목(아래 "문서-코드 불일치")으로 남긴다.

먼저 볼 코드 리딩 순서

  1. fs/fdmfs/fdmfs.h — fdmfs_sb_info, fdmfs_inode, fdm_region 구조체를 먼저 파악한다. FDMFS 전체가 "free-list 기반 메모리 영역 할당자 + 얇은 VFS 껍데기"라는 것을 이 헤더 하나로 알 수 있다.
  2. fs/fdmfs/super.c의 fdmfs_fill_super — 마운트 시 cemu_bdev/fdm_addr/fdm_size를 얻고 free region 리스트를 초기화하는 지점. FDMFS가 실제 디바이스 메모리를 어떻게 "발견"하는지 여기서 시작한다.
  3. fs/fdmfs/file.c의 fdmfs_fallocate — 파일에 실제 저장 공간(디바이스 메모리 영역)이 배정되는 유일한 지점. 여기를 봐야 "쓰기 전에 반드시 fallocate가 필요하다"는 설계를 이해할 수 있다.
  4. fs/fdmfs/inode.c의 fdmfs_iomap_begin — iomap->bdev/iomap->addr를 어떻게 채우는지 확인하면 일반 read/write가 실제로 어디로 가는지(디바이스 bio) 알 수 있다.
  5. fs/fdmfs/file.c의 fdmfs_copy_file_range / fdmfs_copy_file_range_kiocb — 이 저장소의 핵심 기능. 두 함수의 구조가 거의 동일하므로 동기 버전을 먼저 읽고 io_uring용 버전과 비교하면 빠르다.
  6. fs/read_write.c의 generic_copy_file_checks/vfs_copy_file_range — FDMFS가 어떻게 EXDEV 제약을 우회하며 일반 파일시스템과의 크로스 복사를 허용받는지 확인한다.
  7. io_uring/copy_file_range.c의 io_copy_file_range — SQE에서 커널 kiocb로 변환되는 지점이자, IOPOLL 여부에 따라 완료 콜백이 갈라지는 지점.
  8. io_uring/opdef.c의 IORING_OP_COPY_FILE_RANGE 항목 — iopoll/iopoll_queue/hash_reg_file 플래그의 의미를 확인하며 io_uring 코어가 이 오퍼레이션을 어떻게 다루는지 마무리로 확인한다.
  9. block/blk-core.c의 iocb_bio_iopoll — FDMFS의 .iopoll 콜백이 실제로 무엇을 하는지(디바이스 bio를 재조회해 poll)를 확인해 poll 체인의 마지막 고리를 완성한다.

로직 다이어그램

io_uring을 통한 비동기 COPY_FILE_RANGE 오프로드(예: ext4 파일 → FDMFS 파일)의 실행 흐름이다.

대응 관계:

  • App → IOU: io_uring_enter 시스템콜, SQE 필드는 io_copy_file_range_prep(io_uring/copy_file_range.c:36)이 off_in/off_out/len/copy_fd_in/flags로 파싱.
  • CFR → FDM: io_copy_file_range(io_uring/copy_file_range.c:83)가 kiocb.ki_flags |= IOCB_NOWAIT를 강제하고, IOPOLL 모드면 IOCB_HIPRI + copy_complete_iopoll을, 아니면 copy_complete를 등록한다.
  • FDM → Ext4: fdmfs_copy_file_range_kiocb(fs/fdmfs/file.c:44)는 FDM 쪽 주소를 bvec_set_virt(fdmfs_region_addr(inode) + fdm_off, size)로 구성하고, 상대 파일이 FDMFS가 아니면 call_read_iter/call_write_iter로 상대 파일의 f_op를 그대로 호출한다 — 즉 실제 I/O는 FDMFS가 아니라 상대 파일시스템의 read/write 경로에서 수행된다.
  • Ext4 → Poll: 상대 파일이 O_DIRECT + NOWAIT를 지원하면 -EIOCBQUEUED를 반환하고 나중에 kiocb->ki_complete를 호출한다.
  • Poll → IOU: 일반 모드는 copy_complete(io_uring/copy_file_range.c:67)가 io_task_work를 예약해 copy_req_complete → io_req_task_complete로 CQE를 채운다. IOPOLL 모드는 copy_complete_iopoll(io_uring/copy_file_range.c:49)이 req->iopoll_completed를 smp_store_release로만 세팅하고, io_uring의 IOPOLL 루프가 등록된 파일의 ->iopoll(FDMFS라면 fdmfs_fops.iopoll = iocb_bio_iopoll)을 반복 호출해 bio_poll로 완료를 감지한다.

핵심 흐름

입력 → 검증 (write/read)

  • fdmfs_rw_iter(fs/fdmfs/file.c:24)는 iocb->ki_pos % 512로 512바이트(=FDMFS_BLKSIZE) 정렬을 강제한다. 이는 뒤 단계에서 실제 블록 디바이스 bio를 만들기 때문에 필요한 제약이다.
  • 파일에 데이터를 쓰려면 그 전에 반드시 fdmfs_fallocate가 호출되어 있어야 한다. fallocate는 offset != 0, 4096 정렬 길이만 허용하는 매우 제한된 구현으로, 파일당 단일 연속 영역만 가질 수 있다(스파스 파일/부분 재할당 불가).

상태 변경 (실제 저장 위치)

  • fdmfs_fallocate는 sbi->fdm_free 리스트에서 요청 크기에 맞는 영역을 떼어 sbi->fdm_used에 옮기고 FDMFS_I(inode)->region에 연결한다. 이 순간부터 파일의 데이터는 디바이스(FDM) 주소 공간의 [region->off, region->off+region->size) 구간에 대응한다.
  • 일반 write()는 iomap_dio_rw(iocb, iter, &fdmfs_iomap_ops, ...)를 거치며, fdmfs_iomap_begin(fs/fdmfs/inode.c:214)이 iomap->bdev = sbi->cemu_bdev, iomap->addr = region->off + offset을 채운다. 즉 커널 iomap direct-IO 계층이 이 정보로 CEMU NVMe 블록 디바이스(/dev/cemuX)를 향한 bio를 만들어 제출한다 — 데이터는 호스트 페이지 캐시가 아니라 CSD 온보드 메모리(FDM) 에 직접 쓰인다.
  • copy_file_range는 이 경로를 우회한다: fdmfs_region_addr()(fs/fdmfs/fdmfs.h:61)로 sbi->fdm_addr + region->off라는 CPU에서 직접 역참조 가능한 P2P 매핑 포인터를 얻어 bvec_set_virt로 iov_iter를 만들고, 상대 파일의 read_iter/write_iter에 그대로 넘긴다.

외부 호출/결과

  • 동기 syscall 경로: fs/read_write.c의 vfs_copy_file_range(1490행)는 file_in->f_op->copy_file_range == fdmfs_copy_file_range인 경우를 최우선으로 분기시켜 fdmfs_copy_file_range(동기 버전, kiocb를 스택에 init_sync_kiocb)를 호출한다.
  • 비동기 io_uring 경로: io_copy_file_range가 fdmfs_copy_file_range_kiocb(io_uring 전용 kiocb 버전)를 호출하고, -EIOCBQUEUED면 IOU_ISSUE_SKIP_COMPLETE로 반환해 나중에 ki_complete 콜백으로 CQE를 채운다.

중요한 설계 판단

  1. FDMFS는 "파일시스템"이라기보다 디바이스 메모리 할당자 + VFS 어댑터다. journaling, 블록 비트맵, 디스크 inode 테이블이 전혀 없다. 슈퍼블록에 저장되는 정보는 매직넘버 4바이트뿐이고(mkfs.c), 디렉터리 트리·free-region 리스트는 전부 kzalloc된 커널 메모리에만 존재한다. super.c 주석 "FDMFS is a temporary file system, the data is not persistent"가 이를 명시한다. 이는 tmpfs/ext4-DAX와 근본적으로 목적이 다르다: tmpfs는 호스트 DRAM 페이지캐시에 데이터를 두고, ext4-DAX는 영속적 PMEM을 블록/저널 시맨틱으로 관리하지만, 둘 다 "이 파일이 CSD 하드웨어에서 어떤 (nsid, offset, len)에 대응하는가"를 표현할 방법이 없다. FDMFS는 fdmfs_get_memory_range()(EXPORT_SYMBOL_GPL, fs/fdmfs/file.c:235)로 이 정보를 CEMU NVMe 드라이버(drivers/nvme/host/cemu.h, 이 checkout 범위 밖)에 넘겨 CSF(계산 스토리지 함수) 실행에 쓸 수 있게 하는 것이 핵심 존재 이유로 보인다(추정 — cemu.c 실물은 미확인).
  1. copy_file_range가 오프로드 인터페이스로 선택된 이유는 코드에서 명확히 드러난다: copy_file_range(2)는 원래도 "두 파일 사이의 서버측/디바이스측 복사"를 표현하는 표준 POSIX 인터페이스이며, vfs_copy_file_range가 clone/copy를 파일시스템에 위임하는 훅을 이미 가지고 있다. FDMFS는 이 훅 하나만 구현하면 되고, fs/read_write.c:1442의 특별 분기(file_in->f_op->copy_file_range == fdmfs_copy_file_range)로 서로 다른 슈퍼블록(ext4 ↔ FDMFS) 간 복사에서 발생하는 EXDEV 거부를 우회시켜 준다. 새 syscall이나 ioctl을 만들 필요 없이 기존 애플리케이션 도구(cp --reflink류, sendfile 대체 등)와 자연스럽게 통합된다.
  1. copy_file_range의 실제 구현은 "복사"가 아니라 "한쪽은 포인터, 한쪽은 진짜 read/write"다. fdmfs_copy_file_range[_kiocb]는 FDM 영역을 bvec_set_virt로 감싼 iov_iter를 만들어 상대측 파일의 read_iter/write_iter를 그대로 호출한다. 즉 FDMFS 쪽에는 별도의 I/O가 발생하지 않고(이미 P2P 매핑된 메모리이므로), 상대 파일시스템 쪽에서만 실제 I/O가 발생하는 zero-copy 반쪽짜리 memcpy 구조다. fs/fdmfs/inode.c의 iomap 경로(일반 read/write)와는 별개의 데이터 경로임에 유의.
  1. IOPOLL 지원이 세 계층(FDMFS/블록/io_uring)에 걸쳐 있는 이유는 완료 통지 방식이 계층마다 다르기 때문이다. io_uring이 IOCB_HIPRI를 세팅하면(io_uring/copy_file_range.c:105) 하위 bio가 REQ_POLLED로 제출되고, 완료는 인터럽트가 아니라 애플리케이션 스레드가 직접 bio_poll()(block/bio.c)/blk_mq_poll()을 호출해 감지해야 한다. FDMFS는 fdmfs_fops.iopoll = iocb_bio_iopoll(block/blk-core.c:939, fs/fdmfs/file.c:257)로 자신이 발급한 bio를 poll 가능하게 노출하고, io_uring 코어는 opdef에 .iopoll = 1, .iopoll_queue = 1(io_uring/opdef.c:483-484)을 걸어 이 op을 poll 루프 대상으로 등록한다. 세 곳 중 하나라도 빠지면 poll 체인이 끊긴다.

읽으면서 주의할 점

  • fdmfs_fallocate는 파일당 단일 연속 영역만 허용하고(offset != 0 거부, truncate 미지원), FALLOC_FL_KEEP_SIZE가 아니면 즉시 i_size를 length로 세팅한다. 스파스 파일, 재할당, shrink가 전혀 불가능한 매우 제한된 시맨틱이다.
  • fdmfs_icreate에서 inode->i_blkbits = 1로 설정되는데(fs/fdmfs/inode.c:57), 실제 블록 크기는 FDMFS_BLKSIZE = 512(2^9)이다. i_blkbits=1은 블록 크기 2바이트를 의미해 fallocate의 i_blocks 계산((length+(1<<blkbits)-1) >> blkbits)이 비정상적으로 큰 값을 낼 수 있다 — 실사용에 영향이 있는지는 확인이 필요함(i_blocks가 통계용으로만 쓰인다면 무해할 수 있음).
  • fdmfs_deallocate의 free-region 병합 로직(fs/fdmfs/file.c:132-144)에서 두 번째 분기 조건 region->off + region->size < region->off는 오버플로가 없는 한 항상 거짓이 되는 것으로 보이며, "새 region이 다음 free region 앞쪽에 붙는" 케이스를 병합하지 못하고 있는 것으로 보인다 — 버그로 추정되나 확인이 필요함. 반복적인 fallocate/unlink 사용 시 프리 리스트 파편화를 유발할 수 있다.
  • fdmfs_open이 filp->f_mode |= FMODE_NOWAIT를 강제로 세팅(fs/fdmfs/file.c:15)한다. 이 플래그가 없으면 io_uring은 이 파일에 대한 모든 요청을 blocking 가능하다고 간주해 io-wq 워커 스레드로 우회시키는데, 이는 잘 알려진 io_uring 성능 저하 패턴(인라인 완료 대신 스레드 스케줄링 오버헤드 추가)이다. 언급된 "fix fdmfs performance bug on io_uring read" 커밋의 실체는 이 FMODE_NOWAIT 설정 또는 fdmfs_rw_iter의 inode_lock_shared 도입일 가능성이 높다고 추정하지만, git 로그에 접근하지 못해 확정할 수 없다.
  • fdmfs_rw_iter가 read/write 모두 inode_lock_shared(공유 락)를 쓴다 — 일반적인 VFS 관례(쓰기는 보통 exclusive lock)와 다르다. FDMFS는 write 시점에 메타데이터(할당)를 바꾸지 않으므로(할당은 별도 fallocate 단계) 안전하다는 설계로 보이나, 동시 write/write 경쟁 시 CSD 하드웨어 레벨에서의 순서 보장이 없다는 점은 애플리케이션이 책임져야 한다.
  • fdmfs_kill_sb가 빈 함수다 — 언마운트 시 fdm_free/fdm_used 리스트와 inode 트리를 해제하지 않는 것으로 보인다(리소스 누수 가능성, 확인 필요).

문서-코드 불일치

references/text/cemu__csd-README.txt 103~185행은 FDMFS를 hw/femu/csd/fuse의 libfuse 기반 사용자공간 구현으로 설명하고, 별도 수정 커널(fuse_modify.patch)과 libfuse_modify.patch를 요구한다. 그러나 실제 분석 대상은 커널 내장 VFS 모듈(fs/fdmfs/*.c, Kconfig에 CONFIG_FDMFS_FS)이며 FUSE와 무관하다. 두 구현은 이름(FDMFS)과 목적(디바이스 메모리를 파일로 노출)만 같고 구조가 다르다 — CEMU 프로젝트가 초기 FUSE 프로토타입에서 커널 네이티브 파일시스템으로 재작성했을 가능성이 높다(추정). 이 분석 문서는 코드(커널 모듈) 기준으로 작성되었으며, README의 FUSE 관련 빌드 절차는 이 코드베이스에는 적용되지 않는다.
