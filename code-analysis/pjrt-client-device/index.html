PjRtClient / PjRtDevice / PjRtMemorySpace 계층 분석

한 줄 요약

`pjrt_client.h`는 PJRT의 순수 인터페이스(계약)만 정의하고, 실제로 반복되는 버퍼 생성·실행 파이프라인 로직은 `common_pjrt_client.h/.cc`의 `CommonPjRtClient` / `CommonPjRtLoadedExecutable` / `CommonPjRtBufferImpl`이 담당하는 2계층 구조이며, `PjRtDevice`는 데이터 버퍼를 직접 알지 못하고 항상 `PjRtMemorySpace`를 경유해서만 파악한다.

분석 범위

계획 단계에서 분리된 다음 8개 파일만 깊게 분석했다. 상위 PJRT 전체(컴파일러, C API 바인딩, 구체 백엔드인 CPU/GPU/TPU 클라이언트)는 이 문서의 범위 밖이며, 필요한 경우만 짧게 언급한다.

  • `repo/xla/pjrt/pjrt_client.h` — `PjRtMemorySpace`, `PjRtDevice`, `PjRtClient`, `PjRtBuffer`, `PjRtLoadedExecutable` 등 핵심 추상 인터페이스 (1757줄)
  • `repo/xla/pjrt/pjrt_client.cc` — 위 인터페이스의 최소 공통 구현 (254줄, 매우 얇음)
  • `repo/xla/pjrt/pjrt_device_description.h` — `PjRtDeviceDescription`, `PjRtMemorySpaceDescription` (정적 메타데이터 계약)
  • `repo/xla/pjrt/common_pjrt_client.h` / `.cc` — `CommonPjRtClient`, `CommonPjRtLoadedExecutable`, `CommonPjRtBufferImpl` 등 raw-buffer 기반 공통 구현 계층 (헤더 875줄, cc 3000줄 이상)
  • `repo/xla/pjrt/local_device_state.h` — **실제로는 `#include "xla/pjrt/se/local_device_state.h"` 한 줄뿐인 forwarding 헤더**. 대상 파일이 이번 체크아웃에 없어 클래스 본체는 확인 불가(확인 필요).
  • `repo/xla/pjrt/pjrt_client_utils.h` / `.cc` — `ShapeSpec` → `Shape` 변환 유틸리티 함수 하나

전체 구조

상위 PJRT 관점에서 이 8개 파일은 "C++ 계약(pjrt_client.h) + 공통 구현(common_pjrt_client.h)" 레이어에 해당하고, 그 위/아래에 `pjrt_compiler.h`(컴파일러), `pjrt_executable.h`(executable 메타데이터), `c/pjrt_c_api.h`(C ABI), 그리고 실제 하드웨어별 서브클래스(StreamExecutor 기반 CPU/GPU 클라이언트 등, 이번 분석 범위 밖)가 붙는다. `pjrt_client.h`가 정의하는 `PjRtClient`/`PjRtDevice`/`PjRtMemorySpace`/`PjRtBuffer`/`PjRtLoadedExecutable`은 플랫폼 무관 순수 가상 인터페이스이고, `common_pjrt_client.h`는 이 인터페이스를 상속해 여러 백엔드(주로 raw-buffer 모델을 쓰는 백엔드)가 공유하는 실행/전송 파이프라인을 구현한다.

공통 구현 레이어

공통 구현이 제공하는 기능 목록

`CommonPjRtClient`(`common_pjrt_client.h:62`)는 `PjRtClient`를 상속하며, 여러 raw-buffer 기반 백엔드가 반복 구현하던 다음 파이프라인들을 한 번만 구현해 제공한다.

  • **호스트→디바이스 버퍼 생성**: `BufferFromHostBuffer`(`:260-271`), `BufferFromHostLiteral`(`:273-275`, 구현은 `common_pjrt_client.cc:276-304`) — shape 정규화 → 크기 계산 → 원시 버퍼 할당 → 선형화 전송 → `DefineBuffer`로 `PjRtBuffer` 래핑까지의 전 과정을 조립(orchestrate)
  • **버퍼 생성 변형**: `CreateUninitializedBuffer`(`:294-295`), `CreateAliasBuffer`(`:277-279`), `CreateViewOfDeviceBuffer`(`:297-300`)
  • **실행 파이프라인**: `CommonPjRtLoadedExecutable`(`:473`)이 `Execute`/`ExecuteSharded`/`ExecutePortable`을 `Prepare`(인자 검증+출력 버퍼 할당) / `Launch`(실제 디스패치) 두 단계로 분리 구현하고, OOM 발생 시 `ExecutePrepareWithOomRetries`(`common_pjrt_client.cc:1737-1762`)로 재시도
  • **메모리 공간 간 복사**: `CopyToMemorySpace`의 Direct/FallbackThroughLiteral/SyncThroughLiteral 변형, `CopyToCpuMemorySpace`/`CopyFromCpuToMemorySpace`
  • **크로스호스트 전송**: `MakeCrossHostReceiveBuffers`, `CrossHostSendBuffers`, `CrossHostReceiveBuffers`(`:335-364`)
  • **Future/Promise 계측 배선**: `TrackFuture`, `CreateProfiledFuture`, `CreateLinkedUserPromise`(`:205-221`) — TraceMe 프로파일링 자동 삽입
  • **버퍼 구현체**: `CommonPjRtBufferImpl`(`:774`)이 `PjRtBuffer`의 device()/client() 역유도, `Bitcast`, `ToLiteral`/`LazyToLiteral`, `Delete`, `AcquireExternalReference` 등을 공통 구현

백엔드별로 오버라이드해야 하는 지점

`CommonPjRtClient`의 다음 멤버는 기본 구현이 `absl::UnimplementedError`를 반환하는 "채워야 하는" 순수/가상 함수다.

  • `AllocateRawBuffer` / `AllocateRawBufferForExecute`(`:141-156`) — 실제 디바이스 메모리 할당
  • `LinearizeInto` / `LinearizeHostBufferInto`(`:168-172`, `:316-323`) — 호스트 데이터를 디바이스 레이아웃으로 변환·전송
  • `GetOnDeviceBytesCount`, `GetMemorySpaceKindForShape`, `GetDeviceAddressAlignment`(`:122-136`, `:435-438`) — 크기/정렬 계산
  • `async_work_runner()`(`:68`) — 기본값이 `nullptr`이라 백엔드가 채우지 않으면 `LaunchOnDevice`(`:109-112`)가 널 포인터 역참조로 죽는다
  • `CommonPjRtLoadedExecutable::LoadRawExecutable`(`:684-687`, 순수 가상) — 실제 실행 가능한 `PjRtRawLoadedExecutable` 로드
  • `CommonPjRtLoadedExecutable::input_output_alias_config()`(`:693-694`, 순수 가상) — 별칭 설정 제공
  • `CommonPjRtClient::client()`(하위 `CommonPjRtLoadedExecutable::client()`도 순수 가상 `:539`)

문서에 언급되지 않은 이유(추정)

`common_pjrt_client.h`/`.cc`, `pjrt_client_utils.h`/`.cc` 모두 파일 헤더에 `Copyright 2025`가 찍혀 있는 반면, 참고 문서가 인용하는 `pjrt_client.h` 커밋은 그보다 앞선 것으로 추정되는 해시다. 즉 이 공통 구현 계층은 문서 작성 시점 이후에 도입된 리팩터링으로 추정된다 — 원래 개별 백엔드(StreamExecutor 기반 CPU/GPU 등)에 중복 구현되어 있던 `BufferFromHost*`/`Execute` 파이프라인을 `CommonPjRtClient`/`CommonPjRtLoadedExecutable`/`CommonPjRtBufferImpl`로 추출한 것으로 보인다. 문서는 여전히 `PjRtClient` 순수 인터페이스 레벨 설명에 머물러 있어 이 중간 계층을 다루지 않는 것으로 보인다(정확한 도입 시점은 git log 확인이 필요함).

먼저 볼 코드 리딩 순서

1. **`pjrt_device_description.h` 전체** — `PjRtDeviceDescription`/`PjRtMemorySpaceDescription`은 정적 메타데이터만 담는 가장 작은 계약이라, "기기 = ID + kind + attributes"라는 최소 감각을 먼저 잡기 좋다. 2. **`pjrt_client.h`의 `PjRtMemorySpace`(L89) → `PjRtDevice`(L148) → `PjRtClient`(L546) 순서** — 문서가 설명하는 3대 컴포넌트의 실제 가상 인터페이스와 역포인터 멤버(`client()`, `devices()`, `memory_spaces()`)를 한 번에 확인할 수 있다. 3. **`pjrt_client.h`의 `PjRtBuffer`(L1122)** — Client/Device/MemorySpace가 감싸는 실제 데이터 단위이며, `memory_space()`/`device()`/`client()` 역포인터 체인이 여기서 완성된다. 4. **`pjrt_client.h`의 `PjRtLoadedExecutable`(L1404), 특히 `Execute`/`ExecuteSharded`/`ExecutePortable`** — `Compile`/`CompileAndLoad`가 반환하는 실행 가능 객체의 최소 인터페이스를 확인한다. 5. **`pjrt_client.cc` 전체(254줄, 짧음)** — 헤더의 순수 가상 인터페이스에 실제 구현이 얼마나 적은지 체감한다. `pjrt_client.h/.cc` 자체는 "계약"일 뿐이라는 사실을 확인하는 단계. 6. **`common_pjrt_client.h`의 `CommonPjRtClient` 선언부(L62-449)** — `PjRtClient`를 상속하며 무엇을 override했고, 어떤 새 순수 가상(`AllocateRawBuffer`, `LinearizeInto`, `LoadRawExecutable` 등)을 추가했는지가 "백엔드가 채워야 할 지점" 지도가 된다. 7. **`common_pjrt_client.cc`의 `BufferFromHostLiteral`(L276-304)** — 호스트→디바이스 버퍼 생성이 실제로 어떤 순서로 동작하는지 확인하는 가장 쉬운 진입점. 8. **`common_pjrt_client.cc`의 `ExecutePrepare`/`ExecuteLaunch`/`ExecuteHelperOnSingleDevice`/`ExecuteSharded`(L1555-1876)** — `Compile` 이후 `Execute`가 실제로 Prepare/Launch 두 단계로 어떻게 쪼개지고 OOM 재시도가 어떻게 삽입되는지 확인하는 핵심 실행 경로. 9. **`local_device_state.h` + `pjrt_client_utils.h`/`.cc`(마지막, 가장 짧음)** — 이 두 파일이 사실상 분석 범위의 "주변부"임을 마지막에 확인한다. `local_device_state.h`는 실제 정의가 없는 forwarding 헤더이고, `pjrt_client_utils.h`/`.cc`는 `ShapeSpec→Shape` 변환용 작은 유틸일 뿐이다.

핵심 흐름

**입력 → 변환/검증 → 상태 변경 → 외부 호출/결과** 두 가지 대표 경로:

1. **버퍼 생성**: 프레임워크가 `BufferFromHostLiteral(literal, memory_space)`을 호출 → `MakeDefaultShapeForMemorySpace`로 shape/레이아웃을 해당 메모리 공간 기준으로 정규화(검증) → `GetOnDeviceBytesCount`로 필요 바이트 계산 → `AllocateRawBuffer`(상태 변경: 디바이스 메모리 실제 할당, 백엔드 구현) → `LinearizeInto`(외부 호출: 호스트→디바이스 DMA/커널) → `DefineBuffer`로 `CommonPjRtBufferImpl` 생성해 결과 반환. 2. **실행**: 프레임워크가 `Execute(argument_handles, options)` 호출 → `ExecutePrepareWithOomRetries`가 `ExecutePrepare` 반복(검증: `PrepareArguments`가 donation 규칙과 인자 개수를 확인) → 상태 변경: `AllocateOutputBuffersWithInputReuse`로 출력 버퍼 확보, `LoadRawExecutable`로 실행 가능한 객체 로드 → `ExecuteLaunch`가 `PjRtRawLoadedExecutable::Execute`를 호출(외부 호출: 실제 디바이스 커맨드 발행) → `CreateOutputs`로 결과 `PjRtBuffer` 벡터 반환.

두 흐름 모두 "Prepare(검증/할당, 실패 시 재시도 가능) → Launch/전송(성공 보장)" 패턴을 공유하는 것이 특징이다.

주요 파일과 역할

| 파일 | 역할 | |---|---| | `pjrt_client.h` | 플랫폼 무관 순수 인터페이스: `PjRtMemorySpace`, `PjRtDevice`, `PjRtClient`, `PjRtBuffer`, `PjRtLoadedExecutable`, `PjRtExecutableForwarder` | | `pjrt_client.cc` | 위 인터페이스의 극히 얇은 기본 구현(C API 델리게이터, `UnsafeBufferPointer`, `HostShape`/`ToLiteral` 헬퍼) | | `pjrt_device_description.h` | `PjRtDevice`/`PjRtMemorySpace`가 위임하는 정적 메타데이터 계약 | | `common_pjrt_client.h` | raw-buffer 기반 공통 구현의 선언: `CommonPjRtClient`, `CommonPjRtLoadedExecutable`, `CommonPjRtBufferImpl`, `CommonPjRtRawBufferImpl` | | `common_pjrt_client.cc` | 버퍼 전송·복사·크로스호스트 전송·2단계 Execute 파이프라인의 실제 구현체 | | `local_device_state.h` | `xla/pjrt/se/local_device_state.h`로의 forwarding만 존재. 실제 `LocalDeviceState`는 이번 분석 범위 밖 | | `pjrt_client_utils.h`/`.cc` | `ConvertShapeSpecsToShapes` 단일 유틸 함수 |

중요한 설계 판단

  • **계약과 구현의 분리**: `pjrt_client.h`는 어떤 상태도 강제하지 않는 순수 인터페이스로 유지하고, 반복되는 로직은 `CommonPjRtClient`로 추출했다. 이는 새 백엔드가 처음부터 모든 걸 구현하지 않고 `CommonPjRtClient`만 상속해도 되도록 하는 템플릿 메서드 패턴이다.
  • **Prepare/Launch 분리와 OOM 재시도**: `ExecutePrepare`가 실패(특히 `ResourceExhausted`)할 수 있는 부분을 `ExecuteLaunch`(성공이 보장되는 실제 디스패치)와 분리해, `ExecutePrepareWithOomRetries`에서 안전하게 재시도할 수 있게 했다(`common_pjrt_client.cc:1737-1762`).
  • **소유권의 단방향성**: 역포인터(`client()`, `devices()`)는 원시 포인터일 뿐이며, 실제 수명 관리 책임은 `PjRtClient` 쪽에 있다는 것이 `pjrt_client.h:494-495`의 클래스 주석에 명시되어 있다("It is the responsibility of the client of this API to keep the PjRtClient alive as long as any of the other runtime objects are alive").
  • **Device는 MemorySpace를 경유**: `PjRtDevice`가 버퍼를 직접 참조하지 않는 설계는 한 디바이스가 여러 메모리 종류(HBM, pinned host memory 등)를 가질 수 있는 구조를 자연스럽게 지원한다.
  • **`PjRtExecutableForwarder` 어댑터**: AoT 컴파일을 지원하지 않는 플러그인이 `PjRtLoadedExecutable`만 갖고 있어도 `PjRtExecutable` 인터페이스를 만족시킬 수 있도록 하는 어댑터 패턴(`pjrt_client.h:1674-1752`).

읽으면서 주의할 점

  • 참고 문서의 GitHub 라인 번호(`#L486`, `#L102`, `#L72`)는 현재 코드와 맞지 않는다 — 함수/클래스 이름으로 검색해서 대응시켜야 한다.
  • `local_device_state.h`는 8개 대상 파일 중 유일하게 나머지 7개 파일과 직접적인 연결이 없다(grep 결과 `LocalDeviceState`/`local_device_state`가 나머지 파일 어디에도 등장하지 않음). 이 파일만으로는 `PjRtDevice`와의 관계를 코드로 증명할 수 없다.
  • `ABSL_DEPRECATED`가 붙은 `PjRtDevice::id()`, `PjRtBuffer::device()`, `PjRtClient::GetHostAllocator()` 등은 레거시 API이므로 새 코드를 읽을 때 우선순위를 낮춰야 한다.
  • `CommonPjRtClient`의 많은 가상 함수가 기본값으로 `absl::UnimplementedError`를 반환하는 "옵트인" 패턴이라, 컴파일은 통과하지만 런타임에 실패할 수 있다. 어떤 구체 백엔드가 무엇을 실제로 구현했는지는 이 파일들만으로는 알 수 없다(확인 필요).

더 파볼 질문

  • 실제 구체 백엔드(예: StreamExecutor 기반 CPU/GPU 클라이언트)가 `CommonPjRtClient`를 상속하는지, 상속한다면 `AllocateRawBuffer`/`LinearizeInto`/`LoadRawExecutable`을 어떻게 구현하는지?
  • `xla/pjrt/se/local_device_state.h`의 실제 `LocalDeviceState` 클래스는 어떤 `PjRtDevice` 서브클래스가 소유하며, 스트림/이벤트 풀을 어떻게 다루는지?
  • `common_pjrt_client.h/.cc`가 실제로 도입된 커밋 시점과, 이 리팩터링 이전/이후 상태인 백엔드가 각각 무엇인지(git log 확인 필요)?
  • 이번 분석 범위 밖인 `PjRtRawBuffer`(`raw_buffer.h`)와 `AbstractTrackedDeviceBuffer`(`abstract_tracked_device_buffer.h`)가 실제 디바이스 메모리 참조 카운팅/donation을 어떻게 처리하는지?
