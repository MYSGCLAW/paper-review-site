CSF 실행 엔진 — 프로그램 수명주기와 실행 모델 (hw/femu/csd/compute.c)

한 줄 요약

compute.c는 NVMe Computational Storage 커맨드(Load Program / Program Activation / Program Execute)를 받아 Program 구조체를 INVALID → LOADING → LOADED → ACTIVATED로 전이시키고, eBPF(uBPF VM)와 shared library(dlopen) 두 백엔드 중 하나로 ComputeJob을 만들어 direct/indirect 두 실행 모델에 따라 스케줄러(sched.h)와 compute thread에 위임하는 CSD 에뮬레이션의 핵심 디스패처다.

분석 범위

  • repo/hw/femu/csd/compute.c (전체, 1338줄) — 프로그램 로드/활성화/실행/언로드 핸들러, 실행 스레드, indirect 파이프라인
  • repo/hw/femu/csd/compute.h — Program, ComputeNamespace, MemoryRangeSet 등 상태를 소유하는 타입
  • repo/hw/femu/csd/sched.h — ComputeJob, ComputeEngine, CsfScheduler 인터페이스 선언 (구현체 sched.c는 분석 범위 밖)
  • repo/hw/femu/csd/nvme_csd.h — NVMe CS 커맨드용 타입 별칭 (nvme_pind_t 등, 구조체 정의는 없고 전방선언만 존재)
  • repo/hw/femu/csd/memory.h — FDM(Functional Data Memory) 로컬 메모리 I/O 커맨드 정의
  • repo/hw/femu/csd/README.md — 실행 방법, CSF 런타임 시뮬레이션 방식 설명

참고자료(references/)의 code-structure.md, application-development.md와 대조하며 분석했다. sched.c, memory.c, fuse/(FDMFS) 구현은 이번 분석 대상에 포함되지 않으므로, 그 내부 동작은 헤더에 선언된 인터페이스 수준에서만 다룬다.

전체 구조

CEMU는 FEMU(QEMU 기반 SSD 에뮬레이터)에 femu_mode=4로 CSD 확장을 얹은 것이다(README.md). NVMe 네임스페이스 하나가 ComputeNamespace(compute.h:95)를 ns->private로 소유하며, 이 구조체가 프로그램 슬랩(Slab programs), 메모리 레인지 셋 슬랩(Slab mrs), 스케줄러(ComputeEngine *ce), 그리고 nr_thread개의 compute_thread를 묶는다. compute_ns_ops(compute.c:1325)가 FEMU의 커맨드셋 등록 지점이며 nvme_register(NVME_CSI_COMPUTE, &compute_ns_ops)(compute.c:1337)로 CS 커맨드 셋이 연결된다.

먼저 볼 코드 리딩 순서

  1. compute.h의 Program/ComputeNamespace/enum ProgramState/enum ProgramType — 이 파일이 다루는 모든 상태가 이 두 구조체에 모여 있으므로, 필드 이름을 먼저 눈에 익혀야 이후 함수를 읽을 때 헤맨다.
  2. compute.c:57 compute_init() — 네임스페이스 초기화 시 프로그램 슬랩과 nr_thread개의 to_csd 링, compute thread, sched_init()이 어떻게 생성되는지 보여주는 진입점이다.
  3. compute.c:338 load_program() — 프로그램 상태 전이(INVALID→LOADING→LOADED)의 유일한 진입점이므로 수명주기 이해의 핵심이다.
  4. compute.c:224 load_shared_lib() / compute.c:257 load_ubpf() — 두 실행 백엔드의 로딩 경로 차이(dlopen/dlsym vs uBPF VM + JIT)가 명확히 갈리는 지점이다.
  5. compute.c:468 program_activation() — LOADED→ACTIVATED 전이와, 실행 중 잡을 고려하지 않는(TODO로 남은) deactivate 로직을 확인한다.
  6. compute.c:513 program_execute() — is_indirect 분기가 시작되는 지점이며, ComputeJob이 생성되어 sched_alloc_job/sched_enqueue_job으로 넘어가는 실제 코드다. 이 함수가 가장 길고 복잡하므로 direct 경로(else 블록, compute.c:730)부터 먼저 읽고 indirect 블록(compute.c:593)을 나중에 읽는 것을 권한다.
  7. compute.c:856 run_functional_modeling() — eBPF/shared lib 실행 시점의 실제 분기와 runtime 측정 로직이 있는 함수다.
  8. compute.c:1218 compute_thread() — direct 잡은 동기 실행, indirect 잡은 별도 스레드(indirect_main)로 위임하는 디스패치 지점이다.
  9. compute.c:1005 indirect_main() — indirect 모델의 청크 단위 I/O-compute-I/O 상태 머신. 가장 복잡하므로 마지막에 읽는다.

핵심 흐름

1. 로드 (입력 → 상태 변경)

load_program()(compute.c:338)은 sel==0(load)/그 외(unload)로 나뉜다. load 경로는 loff(load offset)로 청크 전송을 구분한다.

  • loff==0(최초 청크): program->type/size/state=LOADING/runtime/runtime_scale/is_indirect를 커맨드 필드에서 채운다(compute.c:381-398). 주목할 점: program->code에 DMA로 쓰여지는 것은 실제 바이너리가 아니라, so_name/ebpf_path와 program_name을 널문자로 구분한 경로 문자열이다(주석 compute.c:220-223 "user data format for shared library: | so_name | csf_name |"). load_ubpf()(compute.c:257)도 이 경로 문자열을 parse_program()으로 파싱한 뒤 open()/read()로 호스트 파일시스템에서 직접 eBPF ELF를 읽는다(compute.c:267-280). 즉 실제 CSF 바이너리 전송은 에뮬레이트하지 않고, "호스트가 이미 접근 가능한 파일 경로"를 device가 열어보는 방식으로 단순화되어 있다 — 이는 원문 코드에서 직접 확인한 사실이며, 실제 NVMe 장치라면 있을 수 없는 지름길이다.
  • 전송이 끝나면(program->load_size == program->size) program->type에 따라 load_shared_lib()(compute.c:224) 또는 load_ubpf()(compute.c:257)를 호출하고 성공 시 state = PROGRAM_STATE_LOADED(compute.c:447)로 전이한다.
  • 재로드는 jobs_running != 0이면 NVME_PROGRAM_IN_USE로 거부된다(compute.c:370-374).

2. eBPF vs Shared Library 로딩 차이

Shared Library (PROGRAM_TYPE_SHARED_LIB)eBPF (PROGRAM_TYPE_EBPF)
로더load_shared_lib() (compute.c:224)load_ubpf() (compute.c:257)
메커니즘dlopen(so_name, RTLD_NOW) + dlsym(handle, program_name)ubpf_create() + ubpf_load_elf()
JIT없음. dlsym으로 얻은 함수 포인터를 shared_lib.jit_fn에 그대로 저장(이름은 jit_fn이지만 실제로는 네이티브 함수)load->jit 플래그가 참이면 ubpf_compile() 호출, 실패 시 NVME_INVALID_PROGRAM_DATA (compute.c:296-302)
언로드dlclose() (compute.c:308)ubpf_destroy() (compute.c:325)

JIT 판단 지점은 호스트가 Load Program 커맨드에 실어 보내는 load->jit 정수 필드이며(compute.c:345, 434), 전송이 완료되어 load_ubpf()가 호출되는 시점에 1회 결정된다. shared library는 애초에 네이티브 코드이므로 JIT 여부 개념 자체가 없다.

실행 시점 차이는 run_functional_modeling()(compute.c:856)의 switch (program->type)에서 갈린다(compute.c:879-895).

  • PROGRAM_TYPE_SHARED_LIB: program->shared_lib.jit_fn(&job->args) 직접 호출.
  • PROGRAM_TYPE_EBPF: program->ebpf.jit_fn == NULL이면 ubpf_exec(vm, args, &res)(인터프리터), 아니면 program->ebpf.jit_fn(&job->args)(JIT 네이티브 함수) 직접 호출.

3. 활성화와 실행 (검증 → ComputeJob 생성 → 스케줄러/스레드 위임)

program_execute()(compute.c:513)는 먼저 program->state == PROGRAM_STATE_ACTIVATED를 검증하고(compute.c:544), is_indirect 여부에 따라 필수 파라미터(numr, rsid, dlen)를 서로 다르게 검증한다(compute.c:548-567).

메모리 레인지 확보 후, sched_alloc_job(compute_ns(req->ns), req)(compute.c:755)로 req->job(ComputeJob)을 얻고 job->program, job->args.{numr,mr_addr,mr_len,cparam1,cparam2,data_buffer,buffer_len}, job->user_runtime을 채운다(compute.c:756-765).

여기서 direct/indirect 분기가 명확히 갈린다(compute.c:766-776):

if (!program->is_indirect) {
    sched_enqueue_job(compute_ns(req->ns), req);   // CU 스케줄러에 즉시 등록
}
femu_ring_enqueue(...to_csd[thread]..., &req, 1);   // 두 경우 모두 실행 스레드로 전달
if (!program->is_indirect) {
    qatomic_inc(&program->jobs_running);            // direct만 여기서 refcount 증가
}

indirect job은 이 시점에는 스케줄러에 등록되지 않고, 청크별 하위 job이 indirect_main() 내부에서 req_dup()(compute.c:969, 내부에서 sched_alloc_job+sched_enqueue_job 호출)을 통해 개별적으로 스케줄러에 등록된다.

compute_thread()(compute.c:1218)는 to_csd 링에서 request를 꺼내 job_is_indirect(job)(sched.h:309)로 분기한다: direct는 run_functional_modeling()을 스레드 안에서 동기 실행하고, indirect는 indirect_thread_pool(최대 MAX_INDIRECT_JOBS=16, compute.h:10)에서 슬롯을 할당받아 indirect_main을 별도 조인 가능 스레드로 기동한다(compute.c:1256-1263).

4. Indirect 모델: LBA → 장치 메모리 매핑과 chunk 처리

indirect 모델은 항상 rsid(MemoryRangeSet)를 요구하며(compute.c:552-555), LBA(sre: slba/nlb 리스트, NvmeCopyFormat)와 실제 FDM/NVM 백엔드 주소 사이의 매핑은 두 단계로 이뤄진다.

  1. 주소 매핑: memory_range_set_management()(compute.c:92)가 MRS 생성 시점에 각 MemoryRange의 addr를 backend_addr(mns->backend, sb)(compute.c:141)로 미리 계산해 mrs->mr_addr[i]에 저장한다. 이건 compute.c가 직접 수행하며 sched.c에는 위임하지 않는다.
  2. 청크 단위 I/O 생성: indirect_main()(compute.c:1005)이 task->iter_in[i]/iter_out[i](struct sre_iter)를 순회하며 next_sres()(compute.c:942)로 한 번에 처리할 NvmeCopyFormat(slba/nlb) 목록을 뽑아 n->to_ftl[1]에 실제 I/O 요청을 넣는다(compute.c:1068, 1156, 1176). 이 I/O는 FTL(FEMU의 기존 flash 시뮬레이션 계층, 분석 범위 밖)이 처리하고, 완료되면 indirect_task.ring을 통해 indirect_main으로 통지된다.

chunk size(chunk_nlb)는 program_execute()가 처리하는 NvmeProgramExecuteCmd.chunk_nlb(+1, compute.c:525)로, 호스트 애플리케이션이 매 실행 커맨드마다 지정한다(READM/application-development.md의 prep_nvme_uring_program_execute(... chunk_nlb ...) 참고). 다만 무한정 크게 잡을 수는 없고, chunk_nlb * 512 > mr_len[1]이면 거부된다(compute.c:723-726) — 즉 청크 크기는 MRS 생성 시 준비해 둔 입력 버퍼(mr[1]) 용량 내로 제한된다. nr_concurrent_chunks(동시 처리 스트림 수)는 data_buffer의 raw 정수로 전달되며(compute.c:603), mrs->numr이 이 값으로 나누어떨어져야 한다(compute.c:631-634) — 각 스트림이 동일한 개수의 메모리 레인지를 나눠 갖는 구조다.

각 청크는 stage 0(input I/O) → stage 1(compute, run_functional_modeling() 재사용) → stage 2(output I/O, destination==1일 때만) 상태를 순환하며, 우선순위 큐(pqueue, expire_time 기준)로 I/O·compute 완료 시각을 시뮬레이션한다(compute.c:1073-1186).

5. Runtime 측정과 용도

run_functional_modeling()(compute.c:856)은 두 개의 타이머를 쓴다.

  • enter_compute_section() 진입 대기 시간(time_cost)을 실제 호출 전에 빼서 runtime(사용자가 지정했거나 program->runtime으로 물려받은 기대 실행시간)을 보정한다(compute.c:869-877).
  • 실제 CSF 함수 호출(shared_lib.jit_fn/ubpf_exec/ebpf.jit_fn) 전후로 realtime(호스트 wall-clock 실측)을 측정한다(compute.c:878-897).
  • 만약 보정된 runtime이 0(=호스트/프로그램이 명시적 runtime을 지정하지 않음)이면, runtime = realtime * program->runtime_scale로 실측치를 스케일링해 시뮬레이션 값을 만든다(compute.c:898-902). program->runtime_scale은 로드 시 runtime_scale/10.0으로 설정되며, README의 csf_runtime_scale(기본 3) 파라미터와 연결된다.
  • 최종 runtime은 set_sched_runtime(job, runtime)(sched.h:300)으로 job->stat.exec_time에 기록되고, job_has_finished()(sched.h:257, job->runtime >= job_exec_time(job))를 통해 스케줄러가 "이 job이 CU를 얼마나 점유하는지" 판단하는 데 쓰인다. 즉 실제 호스트 실행은 항상 실측 wall-clock 만큼 빠르게 끝나지만, 스케줄러 관점의 "논리적 점유 시간"은 이 계산된 runtime으로 재구성되어 에뮬레이션 정확도를 만든다.
  • enter_compute_section/leave_compute_section(compute.c:804-832)은 첫 진입 스레드가 pause_all_vcpus()+cpu_disable_ticks()로 게스트 VM 클럭을 멈추고, 마지막으로 빠져나가는 스레드가 재개한다 — nr_thread개의 compute thread가 동시에 실행돼도 게스트 타이머가 실제 wall-clock 실행 시간만큼 앞서가지 않도록 하는 QEMU 타이밍 보정 장치다(코드 자체의 주석과 구조로부터 확인; 정확한 설계 의도는 "추정").

6. 여러 CSD 구분: pid / nsid / pind

  • pind(program index, compute.h:59): 네임스페이스 내부 프로그램 슬랩 슬롯 번호(1..MAX_PIND=1024). program_get(ns, pind) = slab_at(program_slab(ns), pind-1)(compute.h:116-119)로 조회하며, 호스트가 Load 커맨드에서 직접 지정한다(동적 할당 아님).
  • nsid(compute.h:60): init_program_slab()(compute.c:37)에서 네임스페이스 초기화 시 ns->id로 고정 설정된다. 프로그램 슬랩 자체가 ComputeNamespace 소유이므로 pind 공간은 네임스페이스별로 독립이다 — 서로 다른 네임스페이스(=서로 다른 "CSD" 개념)의 동일 pind 번호는 완전히 다른 슬롯을 가리킨다.
  • pid(program identifier, compute.h:59, uint64_t): load->pit(program identifier type) 플래그가 참일 때만 의미를 가진다. 최초 청크(loff==0)에서 pit==1이면 program->pid = pid를 기록하고(compute.c:383-385), 후속 청크(loff!=0)에서 pit==1 && pid != program->pid면 mismatch 에러를 낸다(compute.c:406-410). 즉 pid는 주소 지정용이 아니라 여러 DMA 청크로 나뉜 로드 요청이 같은 논리적 프로그램에 속하는지 검증하는 식별 태그다.

정리하면 nsid가 CSD(네임스페이스) 경계, pind가 그 안의 슬롯 핸들, pid가 슬롯에 적재된 프로그램의 논리적 신원 확인용이라는 3단 구조다.

중요한 설계 판단

  • CSF 바이너리 전송을 경로 문자열 전달로 대체. parse_program()(compute.c:185)이 파싱하는 것은 실제 코드가 아니라 파일 경로/심볼 이름 문자열이며, load_ubpf()는 호스트 파일시스템을 open()으로 직접 읽는다(compute.c:267). 실제 NVMe 장치라면 불가능한 방식으로, 에뮬레이션 편의를 위한 명시적 단순화로 보인다(원문 코드에서 확인, 설계 의도는 추정).
  • direct/indirect가 스케줄러 등록 타이밍부터 다르다. direct는 program_execute() 시점에 한 번에 sched_enqueue_job + jobs_running++이 이뤄지지만, indirect는 최초 job이 스케줄러를 거치지 않고 indirect_main이 청크마다 req_dup()으로 새 job을 만들어 개별적으로 등록한다(compute.c:766-776, 969-978). 이는 indirect 모델이 "하나의 execute 커맨드 = 여러 개의 실제 I/O+compute 사이클"이라는 근본적으로 다른 실행 단위를 가진다는 것을 반영한다.
  • runtime 측정을 실측+스케일링 하이브리드로 설계. 호스트가 명시적 runtime을 주면 그대로(약간의 락 대기 보정만) 쓰고, 안 주면 실제 wall-clock 실행시간에 runtime_scale을 곱한다(compute.c:856-910). 이는 정확한 CSF 성능 프로파일이 없어도 합리적인 시뮬레이션 타이밍을 만들기 위한 절충으로 보인다(추정).
  • compute thread 진입 시 vCPU를 일시정지. enter_compute_section/leave_compute_section(compute.c:804-832)이 nr_thread개의 동시 실행 스레드를 refcount로 관리하며 게스트 타이머를 멈춘다 — CSF 실제 실행이 게스트 관점에서 "순간적"으로 보이도록 만드는 타이밍 정합성 장치로 추정된다.

읽으면서 주의할 점

  • 언로드 시 실행 중 job 보호가 불완전하다. load_program()의 unload 경로(compute.c:449-463)는 program->state == PROGRAM_STATE_ACTIVATED면 femu_err()로 로그만 남기고 그대로 진행해 program->code를 free()하고 unload_ubpf()/unload_shared_lib()로 VM/so 핸들을 파괴한다. jobs_running 체크가 아예 없으며 주석으로 // TODO add jobs_running check(compute.c:451)가 명시돼 있다 — 실행 중인 compute_thread나 indirect_main이 동시에 program->ebpf.vm/program->shared_lib.jit_fn을 참조하면 use-after-free 가능성이 있다. 재로드(sel==0)는 jobs_running != 0을 검사해 거부하는 것과 대비된다(compute.c:370-374) — 로드/언로드 간 보호 수준이 비대칭이다.
  • 비활성화(deactivate)도 실행 중 job을 막지 않는다. program_activation()의 sel==0 분기는 jobs_running 체크 코드가 주석처리돼 있다(compute.c:486-491, // TODO fix jobs_running when cu>1). 즉 활성 job이 있어도 무조건 state = PROGRAM_STATE_LOADED로 내려간다.
  • jobs_running 감소 지점이 이 파일에 없다. qatomic_inc(&program->jobs_running)은 확인되지만(compute.c:775, 1070), 감소(decrement)는 sched.c의 job 완료 처리(주석 compute.c:572 "free in sched.c:job_finish()"가 힌트)에서 이뤄지는 것으로 보이며 이번 분석 범위 밖이라 직접 확인하지 못했다 — 확인 필요.
  • indirect job의 jobs_running 증가 시점이 청크마다 분산돼 있다. compute.c:1070에서 청크별로 증가하므로, program_execute() 리턴 시점에는 아직 jobs_running이 0일 수 있다. load_program()의 재로드 가드(jobs_running != 0 체크)가 indirect job 생성 초기 타이밍에 이 값을 정확히 반영하는지는 레이스 조건 관점에서 재검토가 필요해 보인다(추정, 확인 필요).
  • nvme_csd.h에는 실제 커맨드 구조체 정의가 없다. NvmeLoadProgramCmd, NvmeProgramExecuteCmd, NvmeMemoryRangeSetManageCmd 등은 전방선언만 있고 실 정의는 다른 파일(예: nvme-def.h)에 있을 것으로 추정되며, 이번 분석 경로에는 포함되지 않아 필드 의미 일부(cdft, pit 등 세부 인코딩)는 헤더만으로는 완전히 검증하지 못했다.

더 파볼 질문

  • sched.c의 sched_alloc_job/sched_enqueue_job/sched_job_finish_indirect 구현을 봐야 jobs_running 감소 시점과 job_slab 회수 로직, 그리고 ComputeJob->stat이 스케줄러 알고리즘(fifo/rr/grouped)에서 실제로 어떻게 소비되는지 확인할 수 있다.
  • memory.c가 memory.h의 NVME_CMD_MEMORY_{READ,WRITE,COPY,FILL}을 어떻게 처리하며, 이것이 compute.c의 MRS/FDM 접근과 어떤 관계인지(같은 backend_addr 경로를 공유하는지) 확인이 필요하다.
  • 언로드/비활성화 시 jobs_running/state 경쟁 조건이 실제로 트리거 가능한지(예: program_execute()가 state==ACTIVATED 검증을 통과한 직후, 다른 스레드가 unload를 수행하는 TOCTOU) 실제 재현 테스트로 검증할 가치가 있다.
  • README의 "Run CSF on ARM 서버" 옵션(arm_server_url)이 run_functional_modeling()의 실측 경로와 어떻게 스위칭되는지는 compute.c에서 직접 보이지 않는다 — 별도 조건부 컴파일 코드(--enable-femu-arm-server)를 확인해야 한다.
