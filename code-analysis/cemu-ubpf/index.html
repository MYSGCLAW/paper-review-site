uBPF 실행 환경과 CEMU의 in_mem/out_mem 확장

한 줄 요약

ubpf-cemu는 upstream iovisor/ubpf(인터프리터+JIT 겸용 유저스페이스 eBPF VM)에 단 4개 커밋·약 76줄만 얹어, "단일 mem/mem_len 버퍼" 모델을 "번호가 매겨진 다중 메모리 영역 배열(numr/mr_addr[]/mr_len[]) + 스칼라 파라미터(cparam1/2) + 보조 버퍼(data_buffer)"로 일반화한 것이며, 이 구조체(struct ubpf_jit_args)가 CEMU 쪽 struct cemu_args와 1:1 대응하는 ABI 계약 역할을 한다.

분석 범위

  • 코드: repo/vm(인터프리터·JIT·공개 헤더), repo/ubpf_plugin(conformance 테스트 러너), repo/bpf(upstream 샘플 .bpf.c), repo/README.md
  • 참고자료: references/text/cemu__repo-map.txt(3-저장소 교차 지도, ubpf-cemu 변경분 요약), references/text/cemu__application-development.txt(CSF 개발 가이드, struct cemu_args 정의), references/text/cemu__code-structure.txt(ComputeJob/Program 정의)
  • 제약: 이 체크아웃은 .git/info/sparse-checkout으로 vm, ubpf_plugin, bpf, README.md만 받아왔고, git 로그/diff를 조회할 수 있는 셸 도구가 제공되지 않아 커밋 단위 diff는 직접 확인하지 못했다. tests/cemu/kernel/(vadd, lz4, knn, grep, sql 예제와 cemu_def.h)은 이 저장소 스코프 밖이며 실제로 검색해도 존재하지 않는다 — 아래 해당 부분은 참고 문서에만 근거한 것으로 표시한다.

ubpf_vm.c가 실행 엔진의 중심이고, JIT는 ubpf_compile()을 통해 네이티브 함수 포인터(ubpf_jit_fn)를 만들어내는 별도 경로다. 두 경로 모두 동일한 struct ubpf_jit_args*를 입력으로 받지만, 안전성 검사 여부가 다르다(뒤에서 상술).

먼저 볼 코드 리딩 순서

  1. repo/README.md — 이 프로젝트가 upstream iovisor/ubpf임을 확인하고, "인터프리터 + x86-64/Arm64 JIT"라는 기본 구조를 배경지식으로 잡는다.
  2. repo/vm/inc/ubpf.h (특히 L61~75, L216~243) — struct ubpf_jit_args, ubpf_jit_fn, 그리고 ubpf_exec/ubpf_compile 시그니처. CEMU가 upstream과 갈라지는 지점이 전부 이 파일에 응집돼 있다.
  3. repo/vm/ubpf_int.h — 공개 헤더 뒤에 숨겨진 struct ubpf_vm(옵션 opaque 타입)의 실제 필드를 확인한다.
  4. repo/vm/ubpf_vm.c의 ubpf_exec() (L289~343) — args를 reg[1]에 실어 eBPF 레지스터 r1으로 전달하는 지점과, numr/mr_addr/mr_len/data_buffer/buffer_len을 지역 변수로 풀어내는 부분.
  5. 같은 파일의 BOUNDS_CHECK_LOAD/BOUNDS_CHECK_STORE 매크로(L549~613)와 bounds_check() 함수(L1119~1176) — 실제 메모리 안전성 검사 로직.
  6. repo/vm/ubpf_jit.c의 ubpf_compile()(L53~104) — JIT가 언제 호출되고 vm->translate(아키텍처별 함수 포인터)로 어떻게 위임되는지.
  7. repo/vm/ubpf_jit_x86_64.c의 map_register()(L100~103)와 EBPF_OP_LDX*/STX* 케이스(L589~625) — 인터프리터와 달리 bounds_check 호출이 전혀 없다는 것을 코드에서 직접 확인한다(grep으로도 매치 없음).
  8. repo/ubpf_plugin/ubpf_plugin.cc(L150~181)와 repo/vm/test.c(L325~347) — 실제로 ubpf_jit_args를 채우고 ubpf_exec/jit_fn을 호출하는 참조 구현.
  9. references/text/cemu__application-development.txt의 struct cemu_args 정의 — vm/inc/ubpf.h의 struct ubpf_jit_args와 필드를 1:1 대조.

핵심 흐름

입력 → 검증 → 상태 변경 → 외부 호출/결과 관점:

  • 인터프리터: ubpf_exec()가 args 포인터를 reg[1](=eBPF r1, 함수 첫 인자)에 넣고 명령어를 하나씩 디스패치한다. LDX*/STX*/ST* 계열 명령마다 bounds_check()를 호출해 주소가 (a) args 구조체 자신, (b) mr_addr/mr_len 배열 메타데이터, (c) mr_addr[i]..+mr_len[i] 중 하나, (d) data_buffer..+buffer_len, (e) 스택, (f) 등록된 vm->bounds_check_function 중 하나에 속하는지 확인한다(ubpf_vm.c:1119-1176). 어느 것에도 속하지 않으면 -1을 반환하며 즉시 중단한다.
  • JIT: ubpf_compile()이 vm->translate(아키텍처별 코드젠)를 호출해 네이티브 함수를 만든다. 이 네이티브 함수는 System V/AArch64 호출 규약상 첫 번째 인자 레지스터(x86-64는 RDI)에 그대로 eBPF r1을 매핑하므로(ubpf_jit_x86_64.c:82-103, 168), long long (*ubpf_jit_fn)(struct ubpf_jit_args*) 시그니처가 자연스럽게 성립한다. 그러나 LDX*/STX* 처리부(ubpf_jit_x86_64.c:589-625)에는 bounds_check 호출이 전혀 없다 — JIT 경로는 순수한 네이티브 load/store다.
  1. 입력: 호출자(테스트 하네스 또는 추정상 CEMU compute.c)가 eBPF 바이트코드를 ubpf_load()(ubpf_vm.c:154)로 적재한다. 이때 validate()(ubpf_vm.c:906)가 opcode 화이트리스트, 레지스터 번호(src<=10, dst<=9), 점프 목적지, CALL 대상 존재 여부만 검사한다 — 이것은 커널 eBPF의 verifier가 하는 "모든 경로에서의 메모리 접근 안전성 증명"과는 다르다. 코드 주석 자체가 이를 명시한다: /* HACK runtime bounds check … Needed since we don't have a verifier yet. */(L544-548).
  2. 변환/검증: 실행 직전 호출자는 struct ubpf_jit_args를 채운다(numr, mr_addr[], mr_len[], 선택적으로 cparam1/2, data_buffer/buffer_len). 이 구조체 자체가 사실상 "이 실행에서 접근 가능한 메모리 화이트리스트"다.
  3. 상태 변경(실행):
  4. 외부 호출/결과: EXIT에서 r0을 bpf_return_value(인터프리터) 혹은 함수 반환값(JIT, long long)으로 넘긴다. CEMU 문서 기준으로 이 반환값은 "처리한 바이트/블록 수"라는 CSF 관례를 따른다(참고자료).

중요한 설계 판단

  • 단일 버퍼 → 다중 영역 배열로 일반화한 이유(질문 1): upstream uBPF는 커널 소켓필터 계보를 이어받아 "패킷 하나 = 연속된 버퍼 하나"(mem/mem_len) 모델이면 충분했다. 반면 CEMU의 CSF는 스토리지 커맨드를 처리하며 입력과 출력이 서로 다른 파일(FDMFS)/NVMe LBA에서 매핑된, 논리적으로 분리된 장치 메모리 영역이다. 이를 하나의 버퍼로 합치려면 입력 뒤에 출력을 이어붙이는 임시방편 오프셋 규약이 필요해 zero-copy 접근을 해친다. 그래서 numr+mr_addr[]+mr_len[]으로 N개의 독립된 영역을 인덱스로 구분하고, 스칼라 값은 cparam1/2로, 커맨드에 실린 소규모 부가 데이터(예: 간접 모델의 LBA 리스트, 참고자료)는 별도 data_buffer/buffer_len으로 분리했다. references/text/cemu__repo-map.txt에 따르면 실제 커밋 히스토리는 in_mem/out_mem이라는 이름의 2-필드 버전(커밋 "add in_mem and out_mem for ubpf programs")에서 시작해, 이후 커밋("Change ubpf signature")에서 지금 보이는 배열 기반 numr/mr_addr/mr_len 구조로 다시 일반화됐다. 즉 job 설명의 "in_mem/out_mem"은 현재 코드 상태가 아니라 그 이전 단계의 명칭이며, 최종 코드는 더 일반화된 형태다 — 문서/이름과 현재 코드 간 버전 차이로 별도 기록해둔다.
  • 호환성(질문 5): 깨졌다. ubpf_exec(const struct ubpf_vm*, void *mem, size_t mem_len, uint64_t*)라는 upstream 시그니처가 ubpf_exec(const struct ubpf_vm*, struct ubpf_jit_args*, uint64_t*)(ubpf.h:229)로 바뀌었고, ubpf_jit_fn도 인자 없는/포인터+길이 시그니처에서 struct ubpf_jit_args* 단일 인자로 바뀌었다. 저장소 자체의 conformance 하네스(ubpf_plugin.cc, test.c)도 이 변경에 맞춰 패치됐다(repo-map 기준 각각 +13/+12줄). upstream 코드를 그대로 링크하면 컴파일이 깨진다.
  • 인터프리터 vs JIT(질문 3): ubpf_compile()은 호출자가 명시적으로 요청할 때만 실행되며(ubpf_plugin.cc의 --jit 플래그, 애플리케이션 가이드의 use_jit = true), 실패 시(ubpf_translate_null, 지원되지 않는 아키텍처) NULL을 반환해 폴백 없이 그대로 실패한다 — 자동 폴백은 코드 상 존재하지 않는다. 성능 차이는 코드에서 직접 벤치마크하진 않았지만 구조적으로 명확하다: 인터프리터는 명령어당 switch 디스패치 + bounds_check 함수 호출(레지스터 4개 스필 + 루프)을 매번 수행하는 반면, JIT는 네이티브 코드로 직접 로드/스토어한다.

읽으면서 주의할 점

  • JIT 경로에는 메모리 안전성 검사가 없다. ubpf_jit_x86_64.c의 LDX*/STX* 처리부는 bounds_check를 호출하지 않는다(grep으로 확인, 매치 0건). 이는 upstream ubpf 자체의 알려진 한계(JIT는 신뢰된 프로그램을 가정)이지만, CEMU 맥락에서는 "호스트가 다운로드한 CSF가 FDM 범위를 벗어나 임의의 프로세스 메모리를 읽고 쓸 수 있는가"라는 질문으로 직결된다. 이 저장소만으로는 CEMU가 JIT용 CSF를 로드/활성화하는 단계에서 별도 오프라인 검증을 하는지 확인할 수 없다 — 확인 필요(관련 코드는 hw/femu/csd/compute.c, 스코프 밖).
  • __attribute__((packed)) 구조체 계약이 두 저장소에 나눠 정의돼 있다. vm/inc/ubpf.h의 struct ubpf_jit_args와 tests/cemu/kernel/cemu_def.h(참고자료 기준, 이 저장소엔 없음)의 struct cemu_args는 이름이 다른 동일 레이아웃이다. 둘 중 하나만 바뀌면 컴파일러가 감지하지 못하는 오프셋 불일치가 생긴다.
  • bpf/ 디렉터리는 CEMU의 CSF 예제가 아니다. map.bpf.c/rel_64_32.bpf.c는 upstream ubpf 자체의 맵·릴로케이션 conformance 테스트용 샘플이다. vadd/lz4/knn/grep/sql 같은 CEMU CSF 예제는 tests/cemu/kernel/에 있다고 문서에 명시돼 있으나, 이 저장소 어디에도 실제 존재하지 않는다(전체 검색 결과 0건) — 질문 6에 대한 답은 전적으로 참고 문서(application-development.md)에 근거한 것이며 코드로 직접 검증하지 못했다.
  • CEMU eBPF는 커널 eBPF의 verifier·맵·helper 생태계를 갖고 있지 않다(질문 4): helper는 ubpf_register()로 등록된 최대 64개 함수 인덱스뿐이고(MAX_EXT_FUNCS, ubpf_vm.c:34), 맵은 test.c/bpf.h에 있는 BPF_MAP_TYPE_ARRAY 전용 토이 구현뿐이다(이것도 테스트 하네스 전용으로 보이며, 실제 CEMU 프로덕션 경로에서 맵을 지원하는지는 스코프 밖이라 확인 필요). 검증기가 없다는 것은 소스 주석에 명시된 사실이다.
