NVMe CSD 커맨드 경로 — CEMU가 FEMU에 붙는 지점

한 줄 요약

CEMU는 FEMU의 기존 커맨드 디스패치 골격(nvme-admin.c/nvme-io.c)을 전혀 건드리지 않고, NvmeNamespace.csi(Command Set Identifier)와 CommandSetOps 함수 포인터 테이블이라는 FEMU의 기존 확장 지점에 NVME_CSI_COMPUTE/NVME_CSI_MEMORY라는 새 네임스페이스 타입을 "끼워 넣는" 방식으로 Computational Storage를 구현한다. 다만 hw/femu/csd/compute.c, sched.c, memory.c — 실제 opcode 스위치문과 ComputeJob 생성 로직이 들어있는 파일들 — 은 이번 분석 대상 경로에 포함되어 있지 않아, 이 문서의 상당 부분은 "CEMU가 FEMU에 붙는 인터페이스"까지만 코드로 직접 확인했고 그 안쪽은 참고 문서와 구조적 추론으로 보완했다.

분석 범위

job.json이 지정한 11개 파일만 대상으로 한다: nvme-admin.c, nvme-io.c, nvme.h, nvme-def.h, nvme-util.c, femu.c, param.c/param.h, csd/nvme_csd.h, csd/compute.h, intr.c.

중요한 제약: references/text/cemu__repo-map.txt와 cemu__code-structure.txt에 따르면 CSD의 실질적 admin/io 핸들러(program_load, program_execute 등)와 스케줄러(CsfScheduler, ComputeUnit)는 hw/femu/csd/compute.c·sched.c·memory.c에 있는데, 이 파일들은 이번 job의 paths에 없다. 실제로 저장소 전체를 grep해도 program_load, program_execute, ComputeJob 등의 함수 정의는 이번 범위의 파일에는 존재하지 않고 nvme.h의 타입 선언에만 등장한다. 따라서 "Execute Program이 어떻게 실행되는가"의 마지막 구간(실제 스케줄링·CSF 실행)은 코드로 확인하지 못했고, 구조체 설계(compute.h)와 참고 문서로부터의 추론임을 본문에서 명확히 구분한다.

호스트 측 정의(linux-cemu의 include/linux/nvme.h, drivers/nvme/host/cemu.c)도 이 저장소에는 없으므로, 호스트 관련 서술은 전부 참고 문서(cemu__repo-map.txt, cemu__application-development.txt) 기반이다.

전체 구조

CEMU가 FEMU 위에 얹는 구조는 두 개의 독립적인 "모드 축"이 공존하는 형태다.

  • 레거시 축 — femu_mode (nvme.h:1384-1392): FEMU_OCSSD_MODE(0)/BBSSD(1)/NOSSD(2)/ZNSSD(3)/FEMU_CSD_MODE(4). 기존 FEMU가 "이 드라이브 전체가 어떤 SSD 종류인가"를 나타내던 전역 스위치다. 이번 분석 범위 안에서는 nvme-util.c:134(OCSSD(n) 체크 하나)와 femu.c:112(ZNS 컨트롤러 시작), init_nvm_backend(백엔드에 태그로 전달)에서만 참조된다. CSD 전용 분기(if (CSD(n)))는 이번 범위 파일 어디에도 없다 — FEMU_CSD_MODE는 정의만 되어 있고 opcode 라우팅에는 관여하지 않는다.
  • 신규 축 — NvmeNamespace.csi (nvme.h:125-132): NVM(0)/ZONED(2)/MEMORY(3)/COMPUTE(4). 실제 커맨드 디스패치는 전부 이 축으로 이루어진다. nvme-admin.c의 nvme_admin_cmd와 nvme-io.c의 nvme_io_cmd는 모두 마지막에 ns->ops->admin_cmd/ns->ops->io_cmd로 위임하고, ns->ops는 command_set_ops[csi](전역 배열, nvme.h:1046)에서 온다.
  • 각 CSI 구현체는 nvme_register(csi, ops) 매크로(nvme.h:1048-1051, __attribute__((constructor)))로 자기 CSI 슬롯을 스스로 등록한다. bbssd/zns/ocssd와 compute/memory는 이 지점에서 완전히 대칭이며 서로의 존재를 모른다 — 이것이 "CEMU가 기존 SSD 모드와 공존하는 방식"의 핵심이다.
  • 네임스페이스가 실제로 어떤 CSI인지는 커맨드라인이 아니라 JSON 설정 파일에서 온다. femu_props(femu.c:701-704)에는 config_file 문자열 property 하나만 있고, param.c의 parse_config()가 이 JSON을 읽어 ns->csi = nvme_csi_from_str(...), ns->ops = command_set_ops[ns->csi]를 설정한다. fdm_size, nr_cu, nr_thread, csf_runtime_scale, time_slice 같은 CSD 파라미터도 ns->ops->decode_params(param, ns) 콜백(compute.c/memory.c 안, 범위 밖)으로 네임스페이스별 JSON parameters 객체에서 파싱된다.

먼저 볼 코드 리딩 순서

  1. hw/femu/nvme.h의 NvmeNamespace, CommandSetOps, command_set_ops[], nvme_register 매크로 (1033~1103행) — CSD가 FEMU에 꽂히는 유일한 인터페이스이므로 가장 먼저 봐야 전체 그림이 잡힌다.
  2. hw/femu/param.c의 parse_config() (313~393행) — 네임스페이스가 어떻게 csi/ops를 부여받는지, CSD 파라미터가 JSON에서 어떻게 흘러들어오는지 여기서 확정된다. README의 -device femu,femu_mode=4,... CLI 예시와 실제 코드가 다르다는 것도 여기서 확인된다.
  3. hw/femu/nvme-def.h의 opcode enum들 (7~57행, 376~474행) — 상단의 NvmeIoCommands(레거시 0xa1~0xf8 CSD 오퍼레이션)와 하단의 NvmeComputeAdminCommands/NvmeComputeIoCommands(CSI별 opcode, 0x01/0x21~0x23)가 서로 다른 세대의 설계임을 비교해서 봐야 한다.
  4. hw/femu/csd/compute.h — Program, ComputeJob, MemoryRangeSet, ComputeNamespace 구조체. 실제 구현(compute.c)은 없지만 이 헤더만으로 CSF 수명주기와 잡 모델을 파악할 수 있다.
  5. hw/femu/nvme-io.c의 nvme_process_sq_io → nvme_io_cmd (44~113행, 534~588행) — SQE를 꺼내서 ns->ops->io_cmd로 넘기고, 반환값(finished)에 따라 즉시 완료할지 미룰지 갈리는 지점. 비동기 완료의 시작점이다.
  6. hw/femu/nvme-io.c의 nvme_process_cq_cpl (158~275행) — expire_time 기반 우선순위 큐(pqueue)로 완료를 지연시키는 메커니즘. CSF 실행 시간이 긴 경우의 완료/인터럽트 타이밍이 여기서 결정된다.
  7. hw/femu/intr.c의 nvme_isr_notify_io/nvme_isr_notify_admin (183~200행) — 완료 큐 갱신 후 실제로 게스트에 인터럽트를 쏘는 마지막 단계.
  8. hw/femu/femu.c의 nvme_start_ctrl 안 csd_memory_region 등록 로직 (69~82행) — multi-CSD/P2P의 실체.

FemuCtrl은 여러 NvmeNamespace를 소유하고(nvme.h:1283 namespaces 배열), 각 네임스페이스는 csi에 따라 command_set_ops[]에서 채워진 CommandSetOps를 가리킨다(param.c:358). NvmeNamespace.private(nvme.h:1075)가 compute.h의 ComputeNamespace*로 캐스팅되어 사용된다는 것은 compute.h:106-109의 인라인 헬퍼(program_slab, memory_range_set_get 등이 ns->private를 ComputeNamespace*로 캐스팅)로 확인된다. NvmeRequest.job(nvme.h:937)이 비동기 실행 중인 ComputeJob과 원래의 NVMe 요청을 잇는 유일한 끈이다. csd_memory_region[64](nvme.h:1069-1070 extern 선언, femu.c:16-19 정의)는 프로세스 내 여러 FemuCtrl 인스턴스(=여러 CSD)가 서로를 찾기 위한 전역 레지스트리다.

핵심 흐름

1) Execute Program이 ComputeJob이 되기까지 — 코드로 확인된 부분과 추정의 경계

확인됨: 게스트가 opcode NVME_CMD_COMPUTE_EXEC (0x01)(nvme-def.h:377, NvmeComputeAdminCommands가 아니라 NvmeComputeIoCommands이므로 I/O SQ로 제출)를 담은 NvmeProgramExecuteCmd(nvme-def.h:390-408, pind/rsid/numr/cparam1,2/group/chunk_nlb/runtime 필드)를 IO 서브미션 큐에 넣으면, nvme_process_sq_io(nvme-io.c:44)가 이를 꺼내 req->ns를 nsid로 확정하고 nvme_io_cmd(n, &cmd, req)(nvme-io.c:534)를 호출한다. nvme_io_cmd는 ns->csi == NVME_CSI_COMPUTE이면 다른 어떤 스위치도 거치지 않고 곧바로 ns->ops->io_cmd(ns, cmd, req)로 위임한다(nvme-io.c:538-539). 이 반환값이 함수 시그니처 주석(nvme.h:1040 // return 1 when the command is finished after function call)대로 완료 여부를 결정한다.

추정(compute.c 미확인): 그 이후 compute.h에 정의된 program_get(ns, pind)(compute.h:116-119, slab_at로 인덱싱)로 Program*을 찾고, 상태가 PROGRAM_STATE_ACTIVATED인지 확인한 뒤 ComputeJob(compute.h:106-115 근방에 정의는 없지만 code-structure.md에 구조가 나와 있음: jit_fn, in/out, req 백포인터, program, ce)을 만들어 스케줄러(sched.c, 범위 밖)로 넘길 것으로 보인다. 이 과정은 code-structure.md 참고 문서와 compute.h의 타입 정의만으로 추론한 것이며, 실제 스위치문·program_execute() 함수 본문은 이번 분석에서 읽지 못했다.

2) opcode 정의가 장치/호스트 사이에서 어떻게 "합의"되는가, 그리고 어긋나면?

nvme_csd.h는 실제 opcode 값을 전혀 담고 있지 않다 — 순수한 전방 선언(struct NvmeCmdLoadProgram; 등)과 nvme_pit_t/nvme_pid_t 같은 typedef만 있다(csd/nvme_csd.h:1-49). 실제 opcode·구조체는 nvme-def.h에 있고, 이것이 참고 문서(repo-map.txt)가 말하는 호스트 쪽 linux-cemu/include/linux/nvme.h(88 LOC diff)와 "손으로" 대응되어야 하는 대상이다. 이 저장소에는 호스트 헤더가 없어 필드 단위 diff는 확인 불가능하다. 구조적으로 확인된 사실은:

  • 두 저장소 사이에 공유 IDL이나 코드 생성기가 없다 — 완전히 독립된 두 C 헤더를 사람이 동기화한다.
  • 장치 쪽 방어는 static_assert(sizeof(...) == 64, ...)(예: nvme-def.h:194,209,255 등) 뿐이며, 이는 "총 크기가 64바이트인가"만 검증할 뿐 필드 순서·오프셋 불일치는 잡지 못한다.
  • 어긋났을 때의 실제 실패 모드는 두 갈래로 보인다. (a) opcode 값 자체가 어긋나면 ns->ops->io_cmd/admin_cmd 내부의 opcode 스위치(compute.c, 범위 밖)가 default 분기로 빠져 에러 상태 코드를 반환할 것으로 추정된다 — nvme-admin.c:1118-1123의 전역 admin 디스패치가 보여주는 것과 같은 패턴(ns->ops->admin_cmd가 없거나 실패하면 NVME_INVALID_OPCODE | NVME_DNR). (b) opcode는 맞는데 이후 필드(pind, rsid 등)가 어긋나면 더 위험하다 — compute.h:116-124의 program_get/memory_range_set_get은 slab_at(slab, pind - 1) 형태로 범위 체크 없이 인덱싱하는 것으로 보인다(이 헬퍼 함수들 안에는 bound check 코드가 없다). 실제 bound check가 compute.c에 별도로 있는지는 확인하지 못했으므로 "확인 필요"로 남긴다.

3) CSF 실행이 오래 걸릴 때 인터럽트/완료는 언제 발생하는가

nvme-io.c의 완료 경로는 CSD 전용 로직이 아니라 FEMU가 이미 갖고 있던 지연 시뮬레이션 공용 경로를 그대로 재사용한다.

  1. nvme_process_sq_io(44행)는 finished가 1일 때만 femu_ring_enqueue(n->to_poller[req->poller], ...)로 완료 큐 후보에 올린다(103-105행). CSD Execute Program처럼 오래 걸리는 커맨드는 io_cmd가 0을 반환해 이 시점에는 큐에 올라가지 않을 것으로 추정된다(compute.c 미확인).
  2. nvme_process_cq_cpl(158행)은 to_poller 링에서 요청을 꺼내 pqueue(우선순위 큐, 우선순위 = req->stat.expire_time)에 넣고, now >= req->stat.expire_time인 것만 순서대로 꺼내 nvme_post_cqe로 CQE를 기록한다(216-238행). 220-229행의 주석 처리된 코드가 NVME_CMD_CSD_EXEC opcode에 대해 req->stat.exec.runtime/queueing_time/nr_context_switch 같은 필드를 로깅하려 했던 흔적을 남기고 있다 — 이는 CSD 요청도 이 동일한 expire_time 지연 모델을 통과하도록 설계되었음을 강하게 시사한다.
  3. 큐가 빈 시점까지 처리한 뒤(259-274행) nvme_isr_notify_io(n->cq[index_poller])(intr.c:188)를 호출해 실제 인터럽트를 쏜다. intr.c:188-200을 보면 cq->irq_enabled && cq->virq이면 irqfd(event_notifier_set)로 KVM에 직접 알리고, 아니면 MSI-X/MSI/legacy PCI 인터럽트로 폴백한다.

즉 인터럽트는 SQE가 디큐된 시점이 아니라, req->stat.expire_time(장치가 시뮬레이션한 CSF 완료 시각)에 도달한 뒤 폴러 스레드가 다음 nvme_process_cq_cpl 반복에서 그 요청을 발견했을 때 발생한다. 이 expire_time을 CSF의 실제/시뮬레이션 런타임으로 설정하는 코드(csf_runtime_scale 적용 지점)는 compute.c/sched.c에 있을 것으로 추정되며 이번 범위에서 직접 확인하지 못했다. csd-README.txt가 설명하는 "기능 시뮬레이션과 성능(타이밍) 시뮬레이션의 분리"가 바로 이 expire_time 필드를 통해 기존 I/O 지연 모델에 접목되는 구조로 보인다.

4) 여러 CSD(multi-CSD)는 어디서 처리되는가

femu.c:69-82의 nvme_start_ctrl 안에서, 네임스페이스 수가 3개(NVM+Memory+Compute, 즉 하나의 완전한 CSD 구성)일 때 전역 배열 csd_memory_region[64](femu.c:19, nvme.h:1061-1070에 타입/extern 선언)에 자신의 BAR2(slm_mem) DMA 주소, 백엔드 메모리 주소·크기, FemuCtrl* 자기 자신을 스핀락(csd_memory_region_lock)으로 보호하며 등록한다. FemuCtrl.drive_id는 nvme_init_ctrl_str(femu.c:491-500)에서 전역 원자 카운터로 부여되어 여러 CSD 인스턴스를 구분한다. 그리고 nvme-io.c:202-213의 nvme_process_cq_cpl 안에서 req->p2p_req가 참이면 "memory copy p2p request, the request is finished in another drive"라는 주석과 함께 이 폴러는 PCIe 지연만 더하고 실제 CQE 게시는 다른 드라이브에서 일어난다고 명시하고 있다. 즉 multi-CSD 간 P2P 메모리 복사는 이 전역 csd_memory_region 테이블을 조회해 목적지 드라이브의 백엔드에 직접 접근하고, 완료 처리는 그 목적지 드라이브 자신의 폴러 스레드가 담당하는 구조로 보인다(실제 조회/디스패치 코드는 memory.c/backend 쪽, 범위 밖).

중요한 설계 판단

  • CSD를 "opcode 스위치문 추가"가 아니라 "새 CSI 네임스페이스 타입 추가"로 구현했다. 기존 FEMU 코드(nvme-admin.c/nvme-io.c)를 한 줄도 침습하지 않고 ns->csi/CommandSetOps 확장점만 사용한 것은, ocssd/bbssd/zns와 정확히 동일한 패턴을 재사용한 것이며 코드 충돌 위험이 구조적으로 낮다.
  • 완료 지연(latency) 모델을 새로 만들지 않고 기존 expire_time/pqueue 메커니즘에 얹었다. CSF 실행이 아무리 길어도 인터럽트 처리 경로 자체는 일반 read/write와 동일한 코드를 탄다 — 이는 "기능 시뮬레이션과 성능 시뮬레이션의 분리"(repo-map.txt)라는 CEMU의 핵심 철학이 코드 레벨에서 어떻게 구현되는지 보여주는 지점이다.
  • 런타임 설정이 CLI property가 아니라 JSON 파일 기반이다. femu_props에 config_file 하나만 등록돼 있다는 사실은, README에 나온 -device femu,fdm_size=...,nr_cu=... 같은 커맨드라인 예시가 실제로는 동작하지 않거나(최소한 이 코드 버전 기준으로는) 문서가 오래되었음을 시사한다. 아래 "읽으면서 주의할 점"에서 별도로 다룬다.
  • femu_mode와 csi라는 두 개의 독립된 축을 유지한 것은 하위호환을 위한 최소 침습 설계로 보인다 — 기존 NVM 네임스페이스의 백엔드 선택은 여전히 femu_mode가 담당하고, CSD의 커맨드 라우팅은 전적으로 csi가 담당한다. 다만 이 때문에 FEMU_CSD_MODE 값 자체는 이번 범위 코드에서 거의 아무 역할도 하지 않는 "의미상 마커"에 가깝다.

읽으면서 주의할 점

  • 문서-코드 불일치 (1): CLI 파라미터 vs JSON 설정. csd-README.txt는 -device femu,devsz_mb=...,femu_mode=4,fdm_size=... 형태의 QEMU 커맨드라인 예시를 보여주지만, femu.c:701-704의 femu_props에는 config_file 문자열 property 하나만 정의돼 있다. fdm_size, nr_cu, nr_thread, csf_runtime_scale, time_slice는 실제로는 param.c의 parse_config()가 읽는 JSON 파일의 parameters/네임스페이스별 parameters 객체에서 온다. README가 오래된 CLI 인터페이스를 설명하고 있거나, -device 프로퍼티 파싱이 QEMU 표준 property 시스템이 아닌 다른 경로(확인 못함)로 이루어지는 것일 수 있다 — 실행 방법을 그대로 따라 하기 전에 실제 run-csd.sh와 JSON 설정 파일을 함께 확인해야 한다.
  • 문서-코드 불일치 (2): 레거시 flat opcode의 생존. nvme-def.h 상단의 NVME_CMD_CSD_DOWNLOAD(0xa1)~NVME_CMD_CSD_DELETE_GROUP(0xf8)는 CSI 도입 이전의 초기 설계로 보이며, nvme-io.c:220의 if (req->cmd.opcode == NVME_CMD_CSD_EXEC) 분기 내부가 통째로 주석 처리돼 있어 사실상 죽은 코드다. 신규 CSI-scoped opcode(NvmeComputeAdminCommands/NvmeComputeIoCommands, 0x01/0x21~0x23)와 혼동하지 말 것 — 새 커맨드를 추가할 때는 후자를 기준으로 삼아야 한다.
  • compute.h의 슬랩 인덱싱 헬퍼에 범위 체크가 안 보인다. program_get/memory_range_set_get(116-124행)은 호스트가 보낸 pind/rsid를 그대로 slab_at(slab, id - 1)에 넘긴다. compute.c에서 별도로 검증하는지 확인하지 못했으므로, 이 헬퍼만 보고 "안전하다"고 단정하면 안 된다.
  • 핵심 구현 파일이 이번 분석 범위 밖이다. compute.c/sched.c/memory.c를 읽지 않고서는 opcode 스위치문, ComputeJob 생성/스케줄링, MRS 관리의 실제 동작을 확정할 수 없다. 이 문서의 3장·4장 상당 부분은 compute.h 타입 정의 + code-structure.md/repo-map.txt 참고 문서로 재구성한 것이다.
