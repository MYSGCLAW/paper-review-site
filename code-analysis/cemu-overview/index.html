CEMU 전체 구조와 CSD 실행 경로 분석

한 줄 요약

CEMU는 QEMU/FEMU 위에 hw/femu/csd/(Program·ComputeJob·CsfScheduler·MemoryRangeSet)라는 얇은 CSD 확장 레이어를 얹어 "실제 계산은 host CPU에서 진짜로 수행하고, 실행 시간만 시뮬레이션 배율로 재해석"하는 방식으로 Computational Storage Drive를 에뮬레이션하며, 이 장치 로직·호스트 드라이버 스택(linux-cemu)·eBPF 실행기(ubpf-cemu)가 세 개의 독립 저장소로 나뉘어 있다.

분석 범위

  • 분석 대상: repo/README.md, repo/README-FEMU.md, repo/docs/cemu/{code-structure,application-development}.md, repo/hw/femu/csd/{README.md,compute.h,sched.h,memory.h,nvme_csd.h}, repo/hw/femu/{nvme.h,param.h}, repo/tests/cemu/**
  • 중요한 제약: 이 체크아웃에는 hw/femu/csd/*.c(compute.c, sched.c, memory.c 등)와 hw/femu/nvme-admin.c / nvme-io.c가 포함되어 있지 않다(Glob으로 확인, 헤더만 존재). 즉 이 문서는 헤더에 선언된 자료구조·함수 시그니처와 tests/cemu/의 실제 동작 코드를 근거로 하며, .c 구현부의 세부 로직(락 처리, 실제 스케줄링 알고리즘 본문 등)은 코드로 직접 확인하지 못했다 — 이 부분은 "확인이 필요함"으로 표시한다.
  • 참고 문서(references/text/)는 코드-구조 문서(code-structure.md), 애플리케이션 개발 가이드(application-development.md), 그리고 job 준비를 위해 미리 작성된 사전 조사 문서 repo-map.md를 포함한다. repo-map.md는 저자 문서가 아니라 분석자용 사전 조사이므로, 본문에서는 저자 문서(code-structure.md/application-development.md)와 별도로 취급한다.

README.md의 Quick Start(69~99행)는 마운트 후 노출되는 세 NVMe 네임스페이스를 보여준다: /dev/nvme0n1(NVM, 일반 SSD와 동일), /dev/nvme0m2(Memory, FDM을 파일시스템으로 노출), /dev/nvme0c3(Compute, CSF 다운로드/실행). 이 3분할은 hw/femu/nvme.h의 enum NvmeCsi { NVME_CSI_NVM, NVME_CSI_ZONED, NVME_CSI_MEMORY, NVME_CSI_COMPUTE }(125~132행)와 enum NvmeNsImpl(1023~1031행)에 그대로 매핑된다. 즉 CSD = 일반 SSD 네임스페이스 + 메모리 네임스페이스 + 컴퓨트 네임스페이스이며, 이는 femu_mode가 FEMU_CSD_MODE = 4(nvme.h:1389)일 때 활성화된다(static inline bool CSD(FemuCtrl *n), nvme.h:1424).

먼저 볼 코드 리딩 순서

  1. docs/cemu/code-structure.md — 전체 아키텍처 개관 문서. 이후 어떤 파일에서 무엇을 찾아야 하는지 지도 역할을 하므로 가장 먼저 읽는다.
  2. hw/femu/nvme.h의 enum NvmeCsi / enum NvmeNsImpl / FEMU_CSD_MODE(125행, 1023행, 1389행) — CSD가 FEMU의 기존 모드·네임스페이스 체계에 어떻게 끼워드는지 확인한다. 여기를 먼저 봐야 "CSD 전용 코드"와 "FEMU 공통 코드"의 경계가 보인다.
  3. hw/femu/csd/compute.h — Program, ComputeNamespace, MemoryRangeSet 구조체. "무엇이 실행되는가"를 정의하는 핵심 자료구조다.
  4. hw/femu/csd/sched.h — ComputeJob, ComputeUnit, CsfScheduler, JobGroup. "언제, 어느 코어에서, 어떤 순서로 실행되는가"를 정의한다.
  5. hw/femu/csd/memory.h + nvme_csd.h — FDM 메모리 I/O 커맨드와, CSD 관련 타입의 전방선언만 모아둔 헤더. NVMe CSD 프로토콜의 데이터 형태를 파악한다.
  6. tests/cemu/cemu_ioctl.h + tests/cemu/kernel/cemu_def.h — 유저스페이스 ioctl ABI(struct ioctl_download/ioctl_execute/ioctl_create_mrs)와 CSF 함수 시그니처(struct cemu_args)를 확인한다. 이 두 헤더가 "호스트 앱이 CSD와 대화하는 언어"다.
  7. tests/cemu/util.c + vadd_example.cpp — 실제 호스트 앱이 DOWNLOAD → ACTIVATE → CREATE_MRS → EXECUTE를 어떤 디바이스 노드·어떤 시스템콜로 호출하는지 실행 가능한 코드로 확인한다. 문서와 코드가 갈라지는 지점이 바로 여기서 드러난다.
  8. tests/cemu/kernel/vadd.c — vadd()(direct)와 vadd_indirect()(indirect) 두 함수를 나란히 두고 있어 두 실행 모델의 차이를 코드 수준에서 대조할 수 있다.
  9. tests/cemu/test_indirect.cpp — LBA 매핑(get_file_mappings, FIEMAP 기반)과 data_buffer에 실린 NvmeCopyFormat2 배열, chunk_nlb/user_runtime 파라미터까지 indirect 모델의 실제 데이터 흐름을 끝까지 추적할 수 있다.

각 화살표의 코드 대응:

  • IOCTL_CEMU_DOWNLOAD / ACTIVATE / CREATE_MRS는 tests/cemu/cemu_ioctl.h에 정의된 CEMU 전용 ioctl로, linux-cemu의 UAPI 헤더와 짝을 이룬다(cemu_ioctl.h 1~5행 주석에 원본 경로 명시).
  • Program 상태 전이는 hw/femu/csd/compute.h의 enum ProgramState(29~34행)와 Program 구조체(58~86행)에 대응하며, Program.type이 PROGRAM_TYPE_EBPF/PROGRAM_TYPE_SHARED_LIB(36~42행)에 따라 union { ebpf; shared_lib; }(75~85행) 중 하나가 채워진다.
  • ComputeJob 생성과 큐잉은 hw/femu/csd/sched.h의 ComputeJob(35~61행), sched_enqueue_job()/sched_alloc_job()(318~321행 함수 선언)에 대응한다.
  • 스케줄러 결정은 CsfScheduler 인터페이스(178~235행)의 pick_next_job/enqueue_job 콜백, 그리고 ComputeUnit(113~124행)의 time_slice/context_switch_time에 대응한다.
  • 가장 중요한 실측 지점: program_execute 시점에 실제로 dlopen된 함수 포인터나 uBPF JIT 함수(ubpf_jit_fn jit_fn)가 호출되어 실제 계산이 수행된다(기능 시뮬레이션). 이후 그 실행 시간이 Program.runtime_scale / csf_runtime_scale(파라미터, hw/femu/csd/README.md 41행)로 스케일링되어 CSD가 "이 정도 시간이 걸렸다"고 보고하는 값(성능 시뮬레이션)으로 바뀐다.
  • IOCTL_CEMU_EXECUTE는 cemu_ioctl.h에 정의되어 있지만, 실제 vadd_example.cpp/test_indirect.cpp는 이를 쓰지 않고 표준 NVMe passthru(opcode 0x01, nsid 3)를 /dev/ngXn3로 직접 보낸다 — 이 차이는 아래 "저자 문서와 코드가 다른 지점"에서 별도로 다룬다.

핵심 흐름

입력 → 변환/검증 → 상태 변경 → 외부 호출/결과 관점에서 정리하면:

  1. 입력: 호스트 앱이 .so 또는 .bpf.o 파일 경로와 커널 이름 문자열을 ioctl_download 구조체(cemu_ioctl.h 15~25행)에 담아 /dev/nvme0cN에 IOCTL_CEMU_DOWNLOAD ioctl로 전달한다(tests/cemu/util.c의 prep_shared_library/prep_ebpf, 281~300행).
  2. 변환/검증: 드라이버(linux-cemu/drivers/nvme/host/cemu.c, 이 저장소 범위 밖)가 NVMe Admin "Load Program" 커맨드로 변환해 QEMU 경계를 넘기고, hw/femu/csd/compute.h의 Program이 PROGRAM_STATE_LOADING으로 생성된다. Program.type에 따라 dlopen+dlsym(공유 라이브러리) 또는 uBPF 로더(+선택적 JIT)로 분기한다 — 이 분기 로직 자체는 compute.c(범위 밖)에 있고, 헤더는 그 결과를 담을 union 슬롯만 정의한다.
  3. 상태 변경: IOCTL_CEMU_ACTIVATE로 PROGRAM_STATE_ACTIVATED가 되고, IOCTL_CEMU_CREATE_MRS(ioctl_create_mrs, cemu_ioctl.h 41~47행)로 FDMFS 파일 디스크립터 배열이 MemoryRangeSet(compute.h 19~27행)으로 등록되어 rsid를 받는다.
  4. 외부 호출/결과: 실행 시점에는 표준 NVMe passthru 커맨드(struct nvme_program_execute_cmd, tests/cemu/util.h 76~93행, opcode 0x01/nsid 3)를 /dev/ngXn3에 NVME_IOCTL_IO_CMD ioctl 또는 io_uring IORING_OP_URING_CMD로 보낸다. QEMU 안에서 nvme-io.c(범위 밖)가 ComputeJob을 만들고 sched.c(범위 밖)의 CsfScheduler가 ComputeUnit에 배치하며, 실제 함수가 host CPU에서 실행되어 MemoryRangeSet의 mr_addr[]를 직접 읽고 쓴 뒤, 반환값(direct 모델: 처리 바이트 수, indirect 모델: write-back 블록 수)이 NvmeCqe 완료로 앱에 전달된다.

중요한 설계 판단

  1. 기능 시뮬레이션과 성능 시뮬레이션의 분리. Program은 runtime(측정/설정된 실행시간)과 runtime_scale을 별도로 갖고(compute.h 71~73행), ComputeJob도 runtime과 user_runtime(사용자가 미리 알려준 추정치)을 분리해 갖는다(sched.h 47~48행). 실제 계산은 host CPU가 dlopen된 함수나 uBPF(JIT 가능)로 진짜로 수행하고, 그 결과 나온 실행시간을 csf_runtime_scale로 스케일링해 "CSD라면 이만큼 걸렸을 것"이라는 시뮬레이션 시간으로 재해석한다(hw/femu/csd/README.md "CSF Runtime Simulation" 절, 68~89행). 이 분리 덕분에 CEMU는 실제 연산 결과의 정확성(기능 정합성)과 SSD-class 프로세서의 성능 특성(느린 코어 모사)을 동시에 얻는다 — ARM 서버에 원격 실행시켜 실측치를 받아오는 arm_server 옵션도 같은 축의 확장이다.
  2. 인덱스 핸들 + Slab 패턴. pind/rsid는 포인터가 아니라 1-base 정수 인덱스이며 program_get()/memory_range_set_get()(compute.h 116~124행)이 이를 Slab 내부 객체로 변환한다. ioctl ABI를 통해 포인터 대신 정수를 유저스페이스로 노출할 수 있어 커널/게스트/호스트 경계를 넘나들기 안전하다.
  3. CsfScheduler 플러그인 인터페이스로 스케줄링 정책을 교체 가능하게 설계했다(sched.h 178~235행: init/fini/pick_next_job/enqueue_job/dequeue_job/switched_from/pick_thread). code-structure.md가 언급하는 FIFO/RR/Grouped-FIFO/Grouped-RR 4종 구현이 이 인터페이스의 구체화다. 새 스케줄링 알고리즘을 추가하려면 이 구조체를 채워 extern CsfScheduler xxx_scheduler로 노출하면 된다 — 문서에 없는 확장 지점 활용법.
  4. nvme_register(csi, ops) 생성자 매크로(nvme.h 1048~1051행)로 각 네임스페이스 타입(NVM/ZONED/MEMORY/COMPUTE)의 admin/io 핸들러를 프로그램 시작 시점에 정적 등록한다. 새 CSI를 추가하는 확장 지점이며, command_set_ops[NVME_CSI_MAX] 테이블이 디스패치 테이블 역할을 한다.
  5. Direct/Indirect 두 실행 모델을 같은 struct cemu_args로 통일하되 numr(메모리 범위 개수)로 구분한다. vadd.c를 보면 direct는 numr==2(output/input), indirect는 numr==3(output/input/global_mem 고정 슬롯)로 강제해 CSF 작성자가 자신의 함수가 어느 모델인지 인자 개수로 알 수 있게 했다.

읽으면서 주의할 점

  • 이 체크아웃은 .c 구현부가 빠진 헤더/문서/테스트 스냅샷이다. program_load, program_execute, sched_enqueue_job의 실제 로직(락 순서, 에러 처리, 재시도)은 코드로 확인하지 못했다 — 실제 수정 작업을 하려면 compute.c/sched.c/memory.c/nvme-admin.c/nvme-io.c를 별도로 확보해야 한다.
  • application-development.md의 "Using ioctl (Synchronous)" 예제와 실제 코드가 다르다. 문서(295~247행)는 ioctl(ctl_fd, IOCTL_CEMU_EXECUTE, &exec)를 실행 경로로 제시하지만, vadd_example.cpp와 test_indirect.cpp는 이 ioctl을 전혀 쓰지 않고 표준 NVMe passthru(/dev/ngXn3 + NVME_IOCTL_IO_CMD 또는 io_uring IORING_OP_URING_CMD, opcode 0x01/nsid 3)로 실행 커맨드를 보낸다. IOCTL_CEMU_EXECUTE가 정의만 되고 실제로 쓰이지 않는 레거시/대안 경로인지, 아니면 문서가 아직 반영 안 된 구버전 설명인지는 확인이 필요함 — linux-cemu 드라이버 소스가 있어야 두 경로가 결국 같은 admin/io 핸들러로 합류하는지 판단 가능하다.
  • P2P(cross-drive) 기능이 저자 공식 문서에 전혀 없다. test_p2p.cpp, nvme.h의 csd_memory_region[64]/get_mem_ctrl(), NvmeRequest.p2p_req/mem_ctrl 필드는 모두 코드에 존재하지만 code-structure.md/application-development.md 어디에도 설명이 없다. 이 기능을 건드릴 때는 문서를 신뢰하지 말고 코드와 repo-map.md(사전조사 문서)가 언급한 linux-cemu 커밋 로그(add cemu multi-drive p2p support)를 함께 참고해야 한다.
  • hw/femu/csd/README.md와 docs/cemu/*.md는 서로 다른 세대의 문서다. 전자는 FEMU 베이스 CSD 지원(파라미터, ARM 원격 서버, FDMFS 빌드 절차) 문서이고, 후자가 CEMU 고유 아키텍처 문서다. 파라미터 이름(fdm_size, nr_cu, csf_runtime_scale 등)은 전자에만 나열되어 있으므로 실행 파라미터를 찾을 때는 전자를 봐야 한다.
  • indirect 모델은 MAX_INDIRECT_JOBS = 16(compute.h 10행)이라는 하드코딩된 상한이 있다. 청크를 병렬 처리할 때 이 값을 넘기면 어떻게 동작하는지는 .c 구현부 확인이 필요하다.
  • nvme_csd.h는 순수 전방선언 헤더이므로 이것만 봐서는 어떤 함수가 이 타입들을 실제로 사용하는지 알 수 없다 — 반드시 compute.h/sched.h와 함께 읽어야 의미가 생긴다.

더 파볼 질문

  • hw/femu/csd/compute.c의 program_execute()가 direct/indirect 분기를 어디서 하고, indirect 모델에서 MAX_INDIRECT_JOBS를 초과하는 청크를 어떻게 큐잉하는가?
