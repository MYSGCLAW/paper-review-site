장치 메모리(FDM)와 Memory Range Set — hw/femu/csd/memory.c 분석

한 줄 요약

FDM은 QEMU 프로세스 힙에 aligned_alloc으로 잡은 평범한 DRAM 버퍼(SsdBackend.logical_space)이고, NVMe 네임스페이스 모델 위에 NVME_CSI_MEMORY라는 별도 커맨드셋으로 얹혀 있으며, MemoryRangeSet은 이 DRAM 위의 여러 구간을 하나의 rsid로 묶어 CSF 실행 시마다 주소를 재전달하지 않도록 하는 "이름 붙은 스캐터-개더 목록"이다.

분석 범위

이번 체크아웃(repo/)은 git sparse-checkout으로 다음 6개 경로만 포함한다.

  • hw/femu/csd/memory.c, memory.h, nvme_csd.h, compute.h
  • hw/femu/backend/(backend.c, backend.h, dram.c)
  • hw/femu/dma.c

references/raw/cemu/repo-map.md에 따르면 MRS 생성/삭제 admin 커맨드, fdm_size 파라미터 파싱, CSF 실행 엔진은 각각 hw/femu/csd/compute.c, hw/femu/param.c, hw/femu/nvme-admin.c에 있는데 이번 체크아웃에는 포함되어 있지 않다. 이 파일들에 대한 설명은 코드 확인이 아니라 참고 문서(code-structure.md, repo-map.md, application-development.md) 기반 추정이며, 본문에서 "문서 근거"로 명시한다.

전체 구조

CEMU는 하나의 NVMe 컨트롤러(FemuCtrl) 아래 세 개의 네임스페이스를 노출한다(CEMU-README.md 108-98행 예시: /dev/nvme0n1=NVM, /dev/nvme0m2=Memory, /dev/nvme0c3=Compute). memory.c는 이 중 Memory 네임스페이스(nsid=2, CSI=NVME_CSI_MEMORY)의 I/O 커맨드 핸들러 전체이고, 실제 데이터는 hw/femu/backend/의 SsdBackend가 들고 있다. dma.c는 이 백엔드와 호스트(게스트) 메모리 사이의 PRP 기반 DMA를 담당한다.

먼저 볼 코드 리딩 순서

  1. hw/femu/csd/nvme_csd.h — MemoryRange, MemoryRangeSet, Program 등 CSD 전역에서 쓰는 타입/전방 선언 목록. 전체 그림의 색인 역할이라 먼저 봐야 한다.
  2. hw/femu/csd/memory.h — NvmeMemoryReadCmd/WriteCmd/FillCmd/CopyCmd와 opcode enum. Memory 네임스페이스가 받는 실제 온-와이어 커맨드 포맷을 먼저 알아야 memory.c가 읽힌다.
  3. hw/femu/backend/backend.h + dram.c — FDM이 결국 SsdBackend.logical_space(DRAM aligned_alloc)라는 것을 가장 먼저 확인하면 이후 backend_addr() 호출들이 전부 "그냥 포인터 산술"임을 알 수 있다.
  4. memory.c의 memory_init → check_param → memory_read/write/fill — 표준 단일 구간 읽기/쓰기 경로. 가장 단순한 흐름이라 여기서 프로그래밍 모델을 잡는다.
  5. hw/femu/dma.c의 nvme_map_prp, dma_read_prp, dma_write_prp — PRP1/PRP2가 어떻게 QEMUSGList로 바뀌고 실제 게스트 물리 메모리 접근(dma_buf_read/write, pci_dma_sglist_init)으로 이어지는지.
  6. memory.c의 memory_region_backend_addr + memory_copy — cdft 2/3/4 분기, FTL 스레드로 넘기는 비동기 경로, 컨트롤러 간(P2P 유사) 주소 변환.
  7. hw/femu/csd/compute.h의 MemoryRangeSet/MemoryRange/ComputeNamespace — MRS의 실제 필드와 slab_at(..., rsid-1) 인덱싱 규칙. 단, 생성/삭제 로직(compute.c)은 이번 범위 밖이므로 구조체 근거로만 추론한다.
  8. references/raw/cemu/code-structure.md, repo-map.md, application-development.md — 체크아웃에 없는 compute.c/nvme-admin.c/param.c가 무엇을 하는지 문서로 보완하고, 위에서 읽은 실제 코드와 대조.

로직 다이어그램

1) 표준 Memory Read/Write 명령 흐름 (코드로 직접 확인됨)

sequenceDiagram participant Host as "호스트 / io_uring" participant Core as "FEMU NVMe 큐 (범위 밖)" participant Mem as "memory.c memory_io_cmd" participant Dma as "dma.c dma_read_prp / dma_write_prp" participant Be as "backend/dram.c logical_space" Host->>Core: "NVMe I/O 커맨드, nsid=2, opcode=MEMORY_READ 등" Core->>Mem: "memory_io_cmd(ns, cmd, req)" Mem->>Mem: "check_param(sb, len) 정렬/경계 검사" Mem->>Dma: "dma_read_prp(n, backend_addr(ns->backend, sb), rl, prp1, prp2)" Dma->>Dma: "nvme_map_prp: prp1/prp2 를 QEMUSGList 로 변환" Dma->>Be: "dma_buf_read/write 로 게스트 물리 메모리와 backend 포인터 교환" Mem-->>Core: "req->cqe.n.result = 처리 길이"
  • Host→Core→Mem: 표준 NVMe 제출 경로 자체는 이번 범위 밖(FEMU core), memory_io_cmd(memory.c:213)가 CSI 디스패치의 실제 진입점.
  • Mem→Mem: check_param(memory.c:7)이 4바이트 정렬과 sb+len <= ns->backend->size 두 가지만 검사한다.
  • Mem→Dma: memory_read(memory.c:20)/memory_write(memory.c:50)가 각각 dma_read_prp/dma_write_prp(dma.c:137, dma.c:112)를 호출.
  • Dma→Be: backend_addr()(backend.h:34)는 logical_space + offset일 뿐이고, 실제 게스트-호스트 데이터 이동은 nvme_map_prp(dma.c:23)가 만든 QEMUSGList를 통해 QEMU의 dma_buf_read/write가 수행한다.

2) memory_copy의 세 가지 경로 (fdm→fdm은 동기, nvm↔fdm은 FTL로 위임)

flowchart TD A["memory_copy 진입, cdft 필드 확인"] --> B{"cdft 값?"} B -->|"2: nvm to fdm"| C["req->sres = malloc, FTL 스레드 ring에 enqueue"] B -->|"3: fdm to nvm"| C B -->|"4: fdm to fdm"| D["cf[128] 스택 버퍼 사용, 동기 처리"] D --> E["memory_region_backend_addr 로 sdaddr를 backend 포인터로 변환"] E --> F["backend_rw_internal 로 다른 네임스페이스 backend와 직접 memmove"] C -.-> G["ftl.c 스레드(범위 밖)가 실제 NAND<->FDM 전송 완료"]
  • memory_copy(memory.c:127)는 copy->cdft로 세 포맷을 구분한다. cdft==2/3은 req->sdaddr에 backend 포인터를 저장해두고 femu_ring_enqueue(n->to_ftl[...], ...)로 FTL 스레드에 넘기고 즉시 리턴(비동기), cdft==4는 이 함수 안에서 backend_rw_internal(backend.c:81)로 동기 완료한다.
  • memory_region_backend_addr(memory.c:104)는 전역 배열 csd_memory_region[](정의부는 이번 체크아웃에 없음, 확인 필요)을 순회해 sdaddr가 어느 컨트롤러(FemuCtrl *n)의 어느 backend 오프셋인지 찾아준다 — 컨트롤러 간(P2P) 주소 변환 지점.

모듈/객체 관계도

flowchart LR NS["NvmeNamespace (FEMU 코어, 범위 밖) / nsid, csi, backend, private"] NS -->|"csi = NVME_CSI_MEMORY"| MNS["MemoryNamespace / memory.h"] NS -.->|"csi = compute csi (문서 근거)"| CNS["ComputeNamespace / compute.h"] NS -->|"ns->backend"| BE["SsdBackend / backend.h / logical_space, size"] BE -->|"FEMU_DRAM_BACKEND"| DRAM["dram.c / aligned_alloc(4096, size)"] CNS -->|"Slab mrs"| MRS["MemoryRangeSet / compute.h / rsid, numr, lock"] MRS -->|"mr 배열"| MR["MemoryRange / compute.h / nsid, sb, len, addr"] MR -.->|"nsid로 참조"| NS MNS -.->|"memory_copy cdft=4"| REG["csd_memory_region[] 전역 레지스트리 / 정의 위치 확인 필요"]
  • NS→MNS, NS→CNS: 하나의 NvmeNamespace가 CSI 값에 따라 ns->private에 서로 다른 타입(MemoryNamespace 또는 ComputeNamespace)을 꽂는 구조. memory_init(memory.c:240)이 MemoryNamespace를 할당하는 실제 코드이고, Compute 쪽 초기화 함수는 이번 체크아웃에 없어 점선(문서 근거)으로 표시했다.
  • CNS→MRS: ComputeNamespace.mrs(compute.h:97)는 Slab 컨테이너이고 memory_range_set_get()(compute.h:121)가 slab_at(&cns->mrs, rsid-1)로 인덱싱한다. 즉 MRS 저장소는 Memory 네임스페이스가 아니라 Compute 네임스페이스 소유다 — application-development.md에서 IOCTL_CEMU_CREATE_MRS를 Compute 디바이스(/dev/nvme0c3) fd에 보내는 것과 일치.
  • MR-.->NS: MemoryRange.nsid(compute.h:13) 필드 덕분에 하나의 MRS 안 개별 구간이 서로 다른 네임스페이스(FDM=2, 심지어 NVM=1)를 가리킬 수 있다.

핵심 흐름

입력 → 검증 → 상태 변경 → 결과 (단일 read/write/fill)

  1. 입력: 호스트가 NVMe I/O 큐에 NvmeMemoryReadCmd/WriteCmd/FillCmd(memory.h:14-69, 모두 64바이트 고정 크기, static_assert로 강제)를 nsid=2로 제출.
  2. 변환/검증: check_param()(memory.c:7)이 4바이트 정렬 여부와 sb+len <= ns->backend->size 두 조건만 확인. 오버플로 방지용 별도 검사는 없음(아래 "위험한 edge case" 참고).
  3. 상태 변경: backend_addr(ns->backend, sb)로 얻은 포인터에 대해 dma_read_prp/dma_write_prp(read/write) 또는 memset(fill, backend_fill in backend.h:38)이 직접 DRAM을 수정.
  4. 외부 호출/결과: DMA 함수 내부에서 nvme_map_prp가 PRP를 QEMUSGList로 바꾸고, QEMU의 dma_buf_read/write(또는 CMB 경로의 qemu_iovec_to/from_buf)가 실제 게스트 물리 메모리를 건드린다. 완료 후 req->cqe.n.result에 처리 바이트 수를 채우고 NVME_SUCCESS 반환.

memory_copy (다중 구간 복사)

  1. 입력: NvmeMemoryCopyCmd(memory.h:14)에 sdaddr(목적지), prp1/prp2(소스 range entry 테이블 위치), nr(구간 수, 8비트), cdft(포맷)가 실린다.
  2. 변환/검증: memory_region_backend_addr가 sdaddr를 실제 backend 포인터로 바꾸고, 이 주소가 속한 컨트롤러(req->mem_ctrl)를 알아낸다.
  3. 상태 변경: cdft==2/3(nvm↔fdm)은 range entry들을 담을 버퍼를 malloc하고 FTL 스레드 링에 요청을 넘겨 비동기 처리(실제 NAND I/O는 이번 범위 밖). cdft==4(fdm→fdm)는 각 range entry에 대해 nvme_find_namespace로 소스 네임스페이스를 찾고 backend_rw_internal로 동기 memmove.
  4. 외부 호출/결과: cdft==4는 이 함수 안에서 req->status/req->cqe.n.result를 직접 채우고 반환값 1로 즉시 완료를 알린다. cdft==2/3은 0을 반환해 완료를 뒤로 미룬다 — memory_io_cmd의 다른 모든 브랜치가 동기(주석 // TODO: io command should be async)인 것과 대비되는 유일한 비동기 분기.

중요한 설계 판단

  • FDM = 그냥 DRAM, NAND 시뮬레이션 없음: dram.c의 init_dram_backend는 aligned_alloc(4096, b->size) 한 줄이 전부다. bbssd/ftl.c(범위 밖)가 지연/마모 등 NAND 타이밍을 시뮬레이션하는 것과 달리, Memory 네임스페이스는 그런 시뮬레이션 계층이 없는 "순수 스크래치패드"로 설계되어 있다 — CSF가 빠르게 접근해야 하는 작업 메모리라는 성격과 일치한다.
  • CSI 기반 커맨드셋 등록: nvme_register(NVME_CSI_MEMORY, &memory_command_set_ops)(memory.c:258)로 Memory 네임스페이스는 표준 NVM 네임스페이스와 동일한 큐/구조를 쓰되, 커맨드 디스패치 테이블만 다르게 꽂는다. 이는 기존 FEMU의 단일 네임스페이스 가정을 깨지 않고 CSD 확장을 얹은 방식으로 보인다(추정 — FEMU core의 CSI 디스패치 코드는 범위 밖).
  • MRS는 Compute 네임스페이스가 소유: MRS 배열이 데이터를 담는 Memory 네임스페이스가 아니라 Compute 네임스페이스의 Slab mrs에 있다는 점은, MRS가 "메모리 자체"가 아니라 "CSF 실행 시 넘길 인자 목록의 사전 등록"이라는 성격을 보여준다. MemoryRange.nsid 필드 덕분에 한 MRS 안에서 FDM과 NVM 구간을 섞어 참조할 수 있다.
  • PRP DMA 함수 네이밍이 방향 직관과 반대: memory_read는 dma_read_prp, memory_write는 dma_write_prp를 쓰는데, 이는 "prp가 가리키는 호스트 메모리 기준"이 아니라 "ptr(로컬 backend 버퍼) 기준" 명명이다(dma_write_prp=호스트→ptr에 "써넣는다", dma_read_prp=ptr을 "읽어서" 호스트로 보낸다). memory_copy 안 주석 // dma read sre가 실제로는 dma_write_prp 호출인 것으로 이 규칙이 내부적으로 일관됨을 재확인할 수 있다.

읽으면서 주의할 점

  • memory.h가 선언한 get_mem_ctrl()과 memory.c가 정의한 memory_region_backend_addr()는 이름이 다르다. 헤더의 void *get_mem_ctrl(uint64_t sdaddr, uint64_t len, FemuCtrl **n, uint16_t *ret); 선언에 대응하는 정의가 memory.c에 보이지 않는다(정의된 것은 동일 시그니처의 static 함수 memory_region_backend_addr). 실제 get_mem_ctrl 심볼이 다른 파일(범위 밖, 예: 삭제되었거나 다른 이름으로 정리된 레거시 선언일 가능성)에 있는지 확인이 필요함.
  • memory_copy의 cf[128] 스택 버퍼 크기 vs copy->nr(8비트 필드, 최대 255) 불일치. cdft==4(fdm→fdm) 분기에서는 sres가 여전히 스택 배열 cf[128]을 가리키는 상태로 dma_write_prp(n, (void*)sres, sizeof(NvmeCopyFormat) * copy->nr, ...)가 호출된다(memory.c:135-136, 164). copy->nr이 128을 넘는 값(최대 255까지 가능)으로 오면 스택 버퍼 오버플로/이후 루프에서의 OOB 읽기가 발생할 수 있는 것으로 보인다. cdft==2/3만 req->sres를 malloc으로 재할당하므로 이 위험은 cdft==4 경로에만 해당한다. 이는 코드를 직접 읽고 확인한 사실이며, 실제 익스플로잇 가능성(호스트가 임의로 nr>128을 넣을 수 있는지, 상위 admin/queue 계층에서 별도 제한이 있는지)은 nvme-io.c(범위 밖)까지 봐야 확정할 수 있어 확인이 필요함으로 표시한다.
  • check_param은 정수 오버플로를 명시적으로 막지 않는다. sb % 4 || len % 4와 sb + len > ns->backend->size만 검사하므로, 이론상 sb(uint64)가 매우 큰 값이고 len(uint32)과 더했을 때 wraparound가 나면 검사를 우회할 여지가 있는지는 필드 폭(uint64 + uint32)상 실질적으로 도달 불가능해 보이지만(추정), 방어적 코드로서 명시적 오버플로 체크는 없다.
  • MRS 생성/삭제와 pthread_spinlock_t lock의 실제 사용처(락 획득/해제 지점)는 이번 체크아웃에 없다. compute.h가 보여주는 것은 필드 존재(in_use, lock)와 슬랩 인덱싱 규칙(slab_at(slab, rsid-1))뿐이다. 동시성 제어의 구체적 스코프(생성 중 조회 차단인지, 삭제 중 실행 차단인지)는 추정이며 hw/femu/csd/compute.c를 읽어야 확정된다.
  • fdm_size → ns->size 연결 경로는 이번 파일들에서 직접 추적되지 않는다. csd-README.md가 fdm_size를 QEMU 디바이스 프로퍼티(MB 단위)로 문서화하고, repo-map.md는 이 파싱이 hw/femu/param.c에 있다고 밝히지만, 그 값이 어떻게 Memory 네임스페이스의 ns->size로 전달되어 memory_init의 init_backend(&ns->backend, FEMU_DRAM_BACKEND, NULL, ns->size) 호출까지 이어지는지는 이번 체크아웃 범위 밖이라 확인이 필요함.

더 파볼 질문

  • FDM은 SSD의 NAND와 어떻게 다르며 왜 별도로 존재하는가? 코드로 확인된 사실: FDM(Memory 네임스페이스)과 NVM 네임스페이스 둘 다 SsdBackend/FEMU_DRAM_BACKEND를 쓰지만(memory_init), NVM 쪽은 bbssd/ftl.c(범위 밖)의 NAND 타이밍 시뮬레이션과 FTL 매핑을 거치는 반면 Memory 네임스페이스는 memory.c에서 그런 계층 없이 backend 포인터에 직접 접근한다. 즉 물리적으로는 둘 다 DRAM이지만, NAND처럼 느리고 페이지/블록 단위인 것처럼 시뮬레이션되는 저장소(NVM)와, 빠르고 바이트 단위로 접근 가능한 스크래치패드(FDM)라는 시뮬레이션 계층의 차이로 구분된다(코드+문서 종합 추정).
  • Memory Range Set은 어떤 문제를 풀기 위한 추상화인가? application-development.md의 "Create MRS once and reuse them for multiple executions to reduce overhead" 문구와 compute.h의 mr_addr/mr_len이 cemu_args.mr_addr/mr_len과 필드명까지 그대로 대응하는 것으로 보아, 매 실행(execute)마다 입출력 구간 목록을 통째로 재전달하는 대신 한 번 등록해 rsid 하나로 재사용하는 것이 목적이다(문서 근거, compute.c 미확인).
  • CSF 실행 시 입력 데이터는 어떤 경로로 FDM에 들어오는가? memory_write/memory_fill로 직접 채우거나, memory_copy(cdft=2, nvm→fdm)로 NVM 네임스페이스에서 복사해 들어온다 — 코드로 확인됨. 이후 CSF에 실제로 어떻게 매핑되는지(compute.c의 ComputeJob 생성 부분)는 범위 밖.
  • 호스트가 장치 메모리를 직접 읽고 쓰는 경로는 무엇인가? (1) 표준 NVME_CMD_MEMORY_READ/WRITE + PRP DMA(dma.c) — 코드로 확인됨. (2) csd_memory_region[]/memory_region_backend_addr 기반 컨트롤러 간 주소 변환 — 이는 test_p2p 예제(문서 언급)와 연결되는 P2P 성격의 경로로 보이나, 이 레지스트리의 정의·등록 코드가 범위 밖이라 P2P DMA(BAR 노출 등)의 정확한 메커니즘은 확인이 필요함.
  • 메모리 네임스페이스는 여러 CSD 사이에서 어떻게 구분되는가? memory_copy의 req->mem_ctrl != n 검사와 memory_region_backend_addr의 FemuCtrl **n 출력 파라미터는 컨트롤러(즉 CSD 인스턴스, FemuCtrl) 단위로 backend가 구분됨을 보여준다. repo-map.md의 "add cemu multi-drive p2p support" 커밋 언급과 함께, 여러 CSD 간 구분이 이 메커니즘 위에서 이뤄지는 것으로 보이나 세부는 nvme.c/드라이버 쪽(범위 밖) 확인 필요.
  • MRS의 수명은 누가 관리하며 누수 가능성은 없는가? Slab mrs가 고정 용량 슬랩이고 in_use 플래그가 있는 것으로 보아 삭제 시 in_use=0으로 되돌리는 재사용 모델로 추정되지만, 삭제 admin 커맨드 핸들러(compute.c)가 이번 범위에 없어 실제 해제/재사용/누수 여부는 확인 불가 — 다음 분석 대상으로 hw/femu/csd/compute.c의 memory_range_set_management 핸들러를 명시적으로 추천한다.
