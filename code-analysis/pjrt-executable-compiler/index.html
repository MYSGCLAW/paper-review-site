PjRtExecutable / PjRtCompiler / Phase Compile 모듈 분석

한 줄 요약

PjRtCompiler는 XlaComputation 또는 MLIR 모듈을 받아 아직 로드되지 않은 PjRtExecutable을 만들어내는 컴파일 계약을 정의하고, PjRtExecutable은 그 산출물의 메타데이터 조회·레이아웃 조회·직렬화 책임만 가지는 반면, 실제 실행 가능한 PjRtLoadedExecutable과 Execute/ExecuteSharded/ExecutePortable은 이번 분석 대상(repo/xla/pjrt/pjrt_client.h가 스냅샷에 존재하지 않음) 밖의 상위 계층에 있다.

분석 범위

job.json이 지정한 12개 파일만 깊게 분석했다.

xla/pjrt/pjrt_executable.h / .cc
xla/pjrt/pjrt_compiler.h / .cc
xla/pjrt/pjrt_layout.h
xla/pjrt/layout_mode.h / .cc
xla/pjrt/partial_program_utils.h / .cc
xla/pjrt/pjrt_phase_compile_sample_plugin.h / .cc
xla/pjrt/stream_executor_executable.h

이 저장소 스냅샷(repo/xla/pjrt/)에는 위 12개 파일만 존재하고 pjrt_client.h, maybe_owning_mlir_module.h, xla/pjrt/se/stream_executor_executable.h, pjrt_c_api_phase_compile_*.h 등은 포함되어 있지 않다. PjRtLoadedExecutable, Execute/ExecuteSharded/ExecutePortable의 정확한 현재 시그니처는 이 저장소에서 직접 확인할 수 없으며, 아래 내용은 참고 문서(references/text/pjrt__PJRT C++ Device API 개요 ... .txt)의 인용에 근거한 것임을 명시한다(확인이 필요함).

전체 구조

상위 PJRT 구조에서 이 모듈이 차지하는 위치는 짧게만 언급한다. PJRT는 프레임워크(JAX 등) ↔ 디바이스 플러그인 사이의 C API이며, C++ 계층(pjrt_client.h, pjrt_executable.h, pjrt_compiler.h)이 이를 감싼다. 이번 분석 범위는 그중 "컴파일 산출물"(PjRtExecutable)과 "컴파일러"(PjRtCompiler/PjRtPhaseCompiler) 두 축이며, PjRtClient/PjRtLoadedExecutable/PjRtBuffer 등 실행 계층은 forward declaration(class PjRtClient;, pjrt_compiler.h:214)으로만 등장한다.

먼저 볼 코드 리딩 순서

  1. pjrt_compiler.h:447-479의 PjRtCompiler::Compile 두 오버로드 — 이 모듈 전체의 진입 계약(입력/출력 타입)을 먼저 확정하기 위함.
  2. pjrt_executable.h:356-449의 PjRtExecutable — Compile이 실제로 반환하는 타입이 어떤 책임을 지는지 확인하기 위함.
  3. pjrt_executable.h:86-152의 CompileOptions와 :157-164의 LoadOptions — 컴파일에 무엇을 넘기고, 이후 별도 Load 단계에 무엇이 넘어가는지(범위 밖) 구분하기 위함.
  4. layout_mode.h/layout_mode.cc의 LayoutMode — 컴파일 "전"에 레이아웃이 어떻게 요청되는지(mhlo.layout_mode 속성) 이해하기 위함.
  5. pjrt_executable.cc:408-456의 GetParameterLayouts/GetOutputLayouts와 pjrt_layout.h의 PjRtLayout — 컴파일 "후" 레이아웃이 어떻게 조회·직렬화되는지 4단계와 짝지어 보기 위함.
  6. pjrt_compiler.h:444-596의 PjRtPhaseCompiler/CompilationPhaseFunctions — 일반 Compile과 별개인 단계별 컴파일 확장점의 골격을 파악하기 위함.
  7. pjrt_phase_compile_sample_plugin.h/.cc — 6번의 골격을 실제 플러그인이 어떻게 구현하는지 구체 예시로 확인하기 위함.
  8. partial_program_utils.h/.cc — 단계별 컴파일 산출물이 C API 경계를 넘을 때 쓰는 직렬화 유틸을 확인하기 위함.
  9. pjrt_compiler.cc의 PjRtCompilerRegistry/PjRtCompile 자유 함수 — 플랫폼 문자열로부터 실제 PjRtCompiler 인스턴스를 찾아 1번의 Compile을 호출하는 오케스트레이션을 마지막에 확인하기 위함.

2) Phase Compile 흐름 (PjRtPhaseCompiler::RunPhases)

flowchart TD P1["RunPhases options input_programs topology phases_to_run"] --> P2["phases_to_run 목록을 순서대로 순회"] P2 --> P3["phase_map_ 에서 name 으로 CompilationPhaseFunctions 조회"] P3 --> P4["validator options programs 호출"] P4 --> P5{"검증 통과"} P5 -- 실패 --> P6["오류 상태 반환"] P5 -- 성공 --> P7["compiler options programs topology 호출"] P7 --> P8["출력 PjRtPartialProgramProto 를 다음 phase 입력으로 대입"] P8 --> P2
  • 각 phase의 validator/compiler는 CompilationPhaseFunctions(pjrt_compiler.h:511-532)에 담긴 std::function이며, pjrt_phase_compile_sample_plugin.cc의 PhaseValidator/PhaseCompiler가 그 구체 예시다.
  • RunPhases 구현은 pjrt_compiler.cc:252-283. 한 phase의 출력(programs)이 다음 phase의 입력으로 그대로 대입되는 파이프라인 구조다.

두 오버로드는 입력 표현(XlaComputation 대 MLIR 모듈)만 다르고 반환 타입은 동일하다. client는 nullptr일 수 있어(pjrt_compiler.h:499,504), 실행 가능한 클라이언트 없이도 "다른 토폴로지를 대상으로 한" AOT 컴파일이 가능하도록 설계되어 있다(추정: 크로스 컴파일/캐시 워밍 용도).

레이아웃 결정 시점 — 두 단계로 나뉜다.

  • 부가 근거: IsEarlyExitCompilation(pjrt_executable.cc:159-169)은 env_option_overrides의 xla_early_exit_with_layouts 또는 debug_options().xla_early_exit_with_layouts()를 검사한다. 이는 kAuto 모드에서 "컴파일러가 레이아웃만 결정하고 조기 종료"하는 별도 컴파일 모드가 존재함을 시사한다(추정: 프론트엔드가 auto 레이아웃 결과만 얻어 최종 프로그램을 다시 빌드하는 용도).
  1. 컴파일 이전(입력 측): LayoutMode(layout_mode.h:34-63)가 mhlo.layout_mode MLIR 속성의 C++ 표현이다. Mode::kDefault/kUserSpecified/kAuto 세 값을 가지며, kUserSpecified일 때만 user_layout이 채워진다(layout_mode.cc:31-40의 CHECK로 강제). 참고 문서의 로그 예시(... 개요.txt:342)에서 mhlo.layout_mode = "default"가 실제 MLIR 함수 인자 속성으로 나타나는 것을 확인할 수 있다. CompileOptions::argument_layouts(pjrt_executable.h:88)는 XlaComputation 경로에서 프로그램 전체 인자 레이아웃을 미리 지정하는 대안 경로다.
  2. 컴파일 이후(출력 측): PjRtExecutable::GetParameterLayouts/GetOutputLayouts(pjrt_executable.cc:408-456)가 GetHloModules()로 얻은 HloModule::entry_computation_layout()을 FlattenedParameterLayouts/FlattenedResultLayouts로 펼친 뒤 각 Layout을 PjRtLayout(pjrt_layout.h:36-79)으로 감싸 반환한다. 즉 "실제로 확정된" 레이아웃은 컴파일 이후에만 알 수 있으며, kAuto였던 값도 이 시점엔 구체적인 Layout으로 고정되어 있다. PjRtLayout::Serialize()/Deserialize()는 Layout::ToString()/ParseLayout()을 그대로 감싼 것으로, C API 경계를 넘어 문자열로 레이아웃을 주고받기 위한 것으로 보인다(추정).

결과 타입 — Compile은 absl::StatusOr<std::unique_ptr<PjRtExecutable>>을 반환한다. PjRtExecutable(pjrt_executable.h:356)은 순수 가상 함수가 num_replicas/num_partitions/SizeOfGeneratedCodeInBytes/name/GetHloModules/GetParameterMemoryKinds/GetOutputMemoryKinds 뿐이고(:360-407), GetOutputShapes/GetOutputElementTypes/GetOutputDimensions/GetParameterLayouts/GetOutputLayouts/GetParameterShardings/GetOutputShardings는 GetHloModules() 위에 쌓은 기본 구현(pjrt_executable.cc)이다 — 템플릿 메서드 패턴으로, 백엔드는 소수의 순수 가상 함수만 구현하면 나머지 조회 기능을 공짜로 얻는다.

SerializeExecutable/GetAbiVersion/FingerprintExecutable/GetCompileOptions/GetCompiledMemoryStats/GetCostAnalysis는 전부 기본값이 absl::UnimplementedError인 가상 함수다(pjrt_executable.h:417-448). 즉 직렬화·핑거프린트는 선택적 기능으로 설계되어 있다. 문서(... 개요.txt:234)는 "컴파일된 아티팩트와 실행 옵션을 가져와 실행 파일을 저장하고 필요에 따라 로드할 수 있도록 직렬화/역직렬화하는 방법을 알고 있습니다"라고 명시한다. 이를 종합하면 직렬화의 목적은 (a) 컴파일된 실행 파일을 디스크나 캐시에 저장했다가 재실행 시 재컴파일 없이 로드하는 컴파일 캐싱, (b) 컴파일 환경과 실행 환경이 분리된 배포/AOT 워크플로(client == nullptr로 컴파일 가능한 것과 부합) 두 목적으로 보인다 — 이는 문서 문구와 시그니처 설계에서 도출한 추정이며, 실제 캐시 구현(예: JAX persistent compilation cache)은 분석 범위 밖이라 확인이 필요하다.

실행 메서드 3종 비교

Execute/ExecuteSharded/ExecutePortable은 이번 분석 대상 파일에 정의되어 있지 않다(PjRtLoadedExecutable은 pjrt_client.h에 있으며 이 저장소 스냅샷에는 해당 파일이 없다). 아래 표는 참고 문서(... 개요.txt:237-247)에 인용된 시그니처를 그대로 옮긴 것이며, 실제 헤더와 대조 확인이 필요하다.

ExecuteSharded와 ExecutePortable은 시그니처가 거의 동일해 보이지만 전제 조건이 다르다: 전자는 컴파일 시점에 확정된 디바이스 배치를 그대로 재생하는 것이고, 후자는 애초에 특정 디바이스에 묶이지 않은(compile_portable_executable) 실행 파일을 원하는 디바이스로 "이식"해서 돌리는 것이다. 이 구분은 pjrt_executable.h:97-101의 주석("Beware: on GPUs, sometimes an executable compiled for one device doesn't run on another")과도 부합한다.

핵심 흐름 (Phase Compile 확장 패턴)

입력 → 검증 → 변환 → 결과

pjrt_phase_compile_sample_plugin.cc가 보여주는 패턴을 코드가 실제로 움직이는 순서로 재구성하면:

  1. 등록: SamplePhaseCompiler::RegisterAllPhases()(:170-175)가 단 하나의 phase "stablehlo_to_optimized_stablehlo"(kPhaseName, :66)를 PhaseCompiler/PhaseValidator 함수 쌍과 함께 RegisterPhase(PjRtPhaseCompiler 기반 클래스, pjrt_compiler.cc:228-246)로 등록한다.
  2. 검증: RunPhases가 phase를 실행하기 전 PhaseValidator(:98-112)를 호출해 입력 PjRtPartialProgramProto의 program_format()이 "bytecode"인지 확인한다. 아니면 즉시 InvalidArgumentError.
  3. 변환: PhaseCompiler(:114-166)가 각 입력 프로그램을 StablehloTypeSerialization::Deserialize로 MLIR ModuleOp로 되돌린 뒤, mlir::PassManager + createStablehloAggressiveSimplificationPass로 최적화하고, 다시 Serialize로 bytecode 문자열로 만든다.
  4. 결과 태깅: 새 PjRtPartialProgramProto에 producer_phase = kPhaseName, consumer_phases = ["some_next_phase"], version = "1.0"을 채워 다음 phase가 이 출력을 소비할 수 있음을 명시한다(:155-162).
  5. 일반 Compile은 의도적으로 미구현: SamplePhaseCompiler::Compile(XlaComputation...)과 Compile(MaybeOwningMlirModule...)(:177-195) 둘 다 UnimplementedError만 반환한다. 즉 이 컴파일러 구현체는 오직 RunPhases 경로로만 동작하도록 설계되어 있다 — PjRtCompiler의 일반 Compile 계약과 PjRtPhaseCompiler의 단계별 계약이 서로 배타적으로 쓰일 수 있음을 보여주는 예시다.
  6. C API 노출: PJRT_PhaseCompile_Get_Compiler(:197-211)가 SamplePhaseCompiler를 생성·등록하고 PJRT_PhaseCompiler 구조체로 감싼다. CreateSamplePhaseCompileExtension(:218-222)이 이를 PJRT_PhaseCompile_Extension으로 묶고, GetSamplePhaseCompilePjrtApi(:247-257)가 이 확장을 플러그인의 PJRT_Api에 붙인다. 즉 C++ PjRtPhaseCompiler는 플러그인 내부 구현이고, 프레임워크 쪽은 PJRT C API 확장(pjrt_c_api_phase_compile_extension.h, 분석 범위 밖)을 통해서만 접근한다.
  7. C ABI 경계 직렬화: partial_program_utils.cc의 ConvertCharBuffersToPjRtPartialProgramProtos/ConvertPjRtPartialProgramProtosToCharBuffers(:33-91)가 const char** + size_t* ↔ vector<PjRtPartialProgramProto> 변환을 담당한다. PjRtPartialProgramProto는 C++ 객체라 C ABI를 직접 넘길 수 없으므로, phase 사이/plugin-host 경계마다 이 직렬화를 거치는 것으로 보인다(추정 — 실제 RunPhases용 C API 함수 구현은 pjrt_c_api_phase_compile_internal.h에 있으며 분석 범위 밖).

중요한 설계 판단

  • Compile은 "로드되지 않은" 산출물만 만든다. PjRtCompiler::Compile의 반환 타입은 PjRtExecutable이고, 주석이 명시적으로 "must be loaded by a compatible client before execution"라고 못박는다(pjrt_compiler.h:451-452). LoadOptions(pjrt_executable.h:157-164)의 존재도 별도 Load 단계가 있음을 뒷받침한다. 이는 컴파일과 로드/실행을 분리해, client == nullptr인 상태로도(pjrt_compiler.h:499,504) 다른 토폴로지를 겨냥한 AOT 컴파일이 가능하게 하려는 설계로 보인다.
  • 직렬화 관련 기능은 전부 옵트인. SerializeExecutable, GetAbiVersion, FingerprintExecutable, GetCompileOptions, GetCompiledMemoryStats, GetCostAnalysis가 순수 가상이 아니라 기본값이 UnimplementedError인 일반 가상 함수다. 최소 기능만 구현한 백엔드도 PjRtExecutable을 구현할 수 있게 하되, 캐싱/배포를 지원하려는 백엔드는 이를 오버라이드하도록 유도한다.
  • Phase compile은 일반 Compile과 배타적으로 쓰일 수 있다. SamplePhaseCompiler는 RegisterAllPhases만 구현하고 두 Compile 오버로드는 의도적으로 UnimplementedError를 반환한다. PjRtCompiler::AsPhaseCompiler()가 기본값 nullptr이고 PjRtPhaseCompiler에서만 this를 반환하도록(pjrt_compiler.h:477-478, 578-579) fallible downcast 패턴을 쓴 것도, "이 컴파일러가 단계별 컴파일을 지원하는지"를 호출부가 안전하게 판별하게 하려는 설계다.
  • phase 실행 순서는 등록 순서가 아니라 호출자가 지정한다. RunPhases(..., phases_to_run)은 phase_names_(등록 순서)가 아니라 인자로 받은 phases_to_run 리스트 순서대로 순회한다(pjrt_compiler.cc:252-283). GetPhaseNames()는 단지 등록된 phase 목록을 조회하는 용도일 뿐, 실행 순서를 강제하지 않는다.

읽으면서 주의할 점

  • PjRtLoadedExecutable이 PjRtExecutable의 서브클래스라는 근거가 이번 분석 파일 안에는 없다. 참고 문서(... 개요.txt:235)는 "PjRtLoadedExecutable는 ... PjRtExecutable 의 서브클래스입니다"라고 명시하지만, 이 저장소 스냅샷에는 pjrt_client.h가 없어 실제 class PjRtLoadedExecutable : public PjRtExecutable 선언을 확인할 수 없다. 분석 대상 12개 파일 중 PjRtLoadedExecutable을 언급하는 파일은 하나도 없다. 즉 이 상속 관계는 문서에서만 확인했고 코드로는 검증하지 못했다(확인 필요).
  • 문서와 코드의 표현 차이: 참고 문서 231행은 "PjRtCompiler 또는 PjRtClient::Compile 메서드의 책임은 ... PjRtLoadedExecutable을 반환하는 것"이라고 뭉뚱그려 설명하지만, 실제 PjRtCompiler::Compile(pjrt_compiler.h:453-460)은 PjRtExecutable을 반환한다. PjRtLoadedExecutable을 직접 반환하는 것은 (문서에 같이 언급된) PjRtClient::Compile이라는 다른 메서드로 추정되며, 이는 분석 범위 밖이다. 이 문서 문장을 PjRtCompiler::Compile에 그대로 적용하면 오해할 수 있다.
  • Execute/ExecuteSharded/ExecutePortable 표는 참고 문서 인용이지 실제 헤더 대조 결과가 아니다. 버전에 따라 인자 순서나 ExecuteOptions 위치 등이 바뀌었을 수 있으니 실제 사용 전에는 pjrt_client.h로 직접 확인이 필요하다.
  • stream_executor_executable.h는 단순 forwarding 헤더라 이 파일 자체에는 로직이 없다. 실질적 StreamExecutorExecutable(PjRtExecutable의 또 다른 구현체로 추정)은 xla/pjrt/se/stream_executor_executable.h에 있으며 이번 스냅샷에 포함되어 있지 않다.
  • pjrt_phase_compile_sample_plugin.cc가 사용하는 PjRtPartialProgramProto의 program/program_format/producer_phase/consumer_phases/version 필드는 proto 정의 파일을 직접 읽지 않고 사용 코드에서 역추적한 것이다(추정).

더 파볼 질문

  • PjRtClient::Compile(문서에 언급, 코드 미확인)이 내부적으로 PjRtCompiler::Compile + Load를 합성하는 wrapper인지, 아니면 완전히 별개의 구현 경로인지 — pjrt_client.h를 확보해 확인 필요.
  • RunPhases를 C API 경계 너머에서 실제로 호출하는 pjrt_c_api_phase_compile_internal.h/.cc의 구현이 partial_program_utils.cc의 변환 함수를 정확히 어느 지점에서 호출하는지.
  • IsEarlyExitCompilation(pjrt_executable.cc:159-169)이 트리거하는 "레이아웃만 계산하고 조기 종료"하는 실제 컴파일 경로가 어느 서비스 코드(HLO 컴파일 파이프라인)에 있는지.
  • CompileOptions::compiler_variant와 PjRtCompilerRegistry의 variant_name이 실제로 어떤 시나리오(예: 원격 컴파일 서비스 vs 인프로세스)에 쓰이는지 구체 사용처.
