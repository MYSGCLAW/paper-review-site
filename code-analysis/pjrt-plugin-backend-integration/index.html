PJRT Plugin/Backend 통합 메커니즘 분석 — example_plugin, xla_cpu, xla_gpu, xla_tpu

한 줄 요약

`xla/pjrt/plugin/example_plugin`은 "PJRT 플러그인 통합" 문서의 1~3단계(C API 구현 → `GetPjRtApi` 구현 → 테스트)를 가장 얇은 형태로 구현한 참조 코드이며, 실제 프로덕션 백엔드(`xla_cpu`, `xla_gpu`, `xla_tpu`)는 `cpu/`·`gpu/`에 있는 무거운 레거시 `PjRtClient` 구현체를 `GetXlaPjrtCpuClient`/`GetXlaPjrtGpuClient` 같은 얇은 어댑터로 감싸고, `static_registration.cc`/`dynamic_registration.cc`가 그 결과물(`PJRT_Api*`)을 전역 레지스트리에 등록하는 3층 구조로 되어 있다.

분석 범위

계획서 지시에 따라 `xla/pjrt/plugin/` 아래 4개 플러그인 디렉터리(example_plugin, xla_cpu, xla_gpu, xla_tpu)와 그 하위 실제 구현이 위치한 `xla/pjrt/cpu/`, `xla/pjrt/gpu/`의 클라이언트/컴파일러 헤더·cc 파일만 깊게 읽었다. `pjrt_client.h`, `common_pjrt_client.h`, `pjrt_c_api_wrapper_impl.h`, `static_registration.h`, `dynamic_registration.h`, `plugin_names.h`, `pjrt_c_api_cpu_internal.h`, `pjrt_c_api_gpu_internal.h` 등은 이번 저장소 스냅샷(`repo/`)에 실제로 포함되어 있지 않았다(각각 Glob으로 확인, 매칭 0건) — **이는 이번 분석용으로 잘라낸 부분 체크아웃이기 때문으로 추정**되며, 해당 파일들의 동작은 참고 문서와 이를 호출하는 `.cc` 파일들의 사용 패턴으로부터 추론했다. 이 추론 부분은 본문에 "확인 필요"로 표시했다.

전체 구조

상위 PJRT는 프레임워크(JAX 등)가 오직 `PJRT_Api*`(C ABI, `pjrt_c_api.h`)만 보고 백엔드와 통신하도록 설계되어 있다. 이번 분석 대상인 `plugin/` 디렉터리는 그 `PJRT_Api*`를 "누가, 어떻게 만들어서, 언제 프로세스에 노출하는가"를 담당하는 계층이며, 실제 텐서 연산·컴파일·디바이스 관리 로직(`PjRtClient`의 구체 서브클래스)은 여기 포함되지 않고 `xla/pjrt/cpu/`, `xla/pjrt/gpu/` 같은 별도 디렉터리에 있다. 즉 `plugin/` 레이어는 "등록/노출" 관심사만 가지며, "연산 구현" 관심사와 명확히 분리되어 있다.

먼저 볼 코드 리딩 순서

1. **`example_plugin/myplugin_cpp_pjrt.h`, `.cc`** — 가장 얇은 `xla::PjRtClient` 구현체(`MypluginPjrtClient`)를 먼저 봐야 PJRT의 최소 필수 가상 함수 집합(`platform_name`, `device_count`, `devices` 등)을 감을 잡을 수 있다. 2. **`example_plugin/myplugin_c_pjrt_internal.cc`** — C++ 클라이언트를 C API(`PJRT_Client_Create`)로 감싸는 지점이자 `GetMyPluginPjrtApi()`(사실상 문서의 `GetPjRtApi`)가 정의된 핵심 파일이므로 두 번째로 본다. 3. **`example_plugin/myplugin_c_pjrt.h`, `.cc`** — 위 내부 구현을 외부에 노출하는 얇은 공개 진입점(`GetPjrtApi`)이라 구조를 완결짓는 데 짧게 확인한다. 4. **`xla_cpu/xla_cpu_pjrt_client.h`, `.cc`** — example_plugin과 대비되는 "실전" 어댑터 패턴(`GetXlaPjrtCpuClient` → 레거시 `GetPjRtCpuClient` 위임)을 확인해 등록 레이어와 구현 레이어의 경계를 이해한다. 5. **`cpu/cpu_client.h`의 `PjRtCpuClient` 클래스 선언부** — `CommonPjRtClient`를 상속하는 실제 무거운 구현체 형태를 보고, example_plugin의 `MypluginPjrtClient`(직접 `PjRtClient` 상속)와 상속 계층이 다르다는 점을 대조한다. 6. **`xla_cpu/cpu_static_registration.cc`, `xla_gpu/gpu_static_registration.cc`** — `REGISTER_PJRT_PLUGIN` 매크로 한 줄로 정적 등록이 끝나는 패턴을 본 뒤, 7. **`xla_tpu/tpu_dynamic_registration.cc`** — `REGISTER_DYNAMIC_PJRT_PLUGIN` 매크로와 대조하여 static/dynamic 등록의 코드량·의미 차이를 확인한다. 8. **`gpu/se_gpu_pjrt_client.h`의 `StreamExecutorGpuClient` 클래스 선언부** — CPU와 다른 상속 계층(`PjRtStreamExecutorClient`)을 확인해 CPU/GPU 레거시 구현이 서로 다른 중간 베이스 클래스를 쓴다는 사실을 파악한다.

핵심 흐름

1) example_plugin 워크스루

**1단계 대응 코드 — PJRT C API 인터페이스 구현** 문서의 "옵션 A: PJRT C API 직접 구현" vs "옵션 B: C++ API 구현 + C→C++ 래퍼 사용"이 그대로 예제에 나타난다.

  • 옵션 B에 해당하는 코드가 `myplugin_cpp_pjrt.h`/`.cc`이다. `MypluginPjrtClient`는 `xla::PjRtClient`를 직접 상속하고, `platform_name`/`process_index`/`platform_id`/`device_count`/`devices`/`memory_spaces`/`platform_version` 등 최소 가상 함수만 오버라이드한다(`myplugin_cpp_pjrt.cc:30-81`). 디바이스가 0개(`device_count() { return 0; }`)인 뼈대뿐인 구현이라는 점에서 이 파일이 "보일러플레이트 최소 집합"을 보여주기 위한 것임을 알 수 있다.
  • 옵션 A에 해당하는, "C++ PJRT 클라이언트의 일부가 아닌 C API 메서드"들이 `myplugin_c_pjrt_internal.cc`에 직접 구현되어 있다: `PJRT_MypluginExecuteContext_Create`, `PJRT_MypluginDeviceTopology_Create`(둘 다 `UnimplementedError` 반환), 그리고 확장 포인트인 `PJRT_MyPlugin_CreateExampleExtensionCpp`/`PJRT_MyPlugin_DestroyExampleExtensionCpp`(라인 48-76). 즉 example_plugin은 옵션 A와 옵션 B를 **동시에** 보여준다 — 클라이언트 생성 자체는 C++ 래퍼(옵션 B)로 하고, C++ `PjRtClient` 인터페이스에 없는 나머지 C API 콜백만 옵션 A 방식으로 직접 채운다. 이는 실제로 대부분의 XLA 백엔드가 취하는 표준 패턴이다.

**2단계 대응 코드 — GetPjRtApi 구현** 문서 예시 의사코드(`GetPjrtApi()` → `static const PJRT_Api pjrt_api = pjrt::CreatePjrtApi(...)`)와 `myplugin_c_pjrt_internal.cc:78-97`의 `GetMyPluginPjrtApi()`가 거의 1:1로 대응한다. `pjrt::CreatePjrtApi`에 `PJRT_MypluginClient_Create`(1단계에서 만든 클라이언트 팩토리), `PJRT_MypluginExecuteContext_Create`, `PJRT_MypluginDeviceTopology_Create`, `PJRT_Plugin_Initialize_NoOp`, 그리고 example extension을 함수 포인터로 넘겨 `static const PJRT_Api`를 한 번만 만들고 캐시된 포인터를 반환한다. 이 `GetMyPluginPjrtApi()`를 다시 감싸는 공개 진입점이 `myplugin_c_pjrt.h`가 선언하고 `myplugin_c_pjrt.cc:21`이 구현하는 `extern "C" const PJRT_Api* GetPjrtApi()`다 — 프레임워크가 `dlsym`으로 찾는 심볼은 바로 이 이름이다(문서 3단계 "JAX에서 플러그인 사용" 절의 `GetPjrtApi` 규약과 일치).

**3단계 대응 코드 — C API 구현 테스트** 문서는 `RegisterPjRtCApiTestFactory`(`pjrt_c_api_test.h`)를 통해 기본 PJRT C API 동작 테스트를 실행하라고 안내한다. 이 헤더/테스트 등록 파일은 이번 저장소 스냅샷에 포함되어 있지 않아 example_plugin 디렉터리 안에서 실제 테스트 파일을 확인하지 못했다 — **확인 필요**. `README.md`는 "PJRT 플러그인을 만드는 데 필요한 모든 보일러플레이트 코드의 기초"라는 한 줄 설명만 담고 있어 문서 3단계에 대응하는 코드가 별도 BUILD 타겟(`myplugin_c_api_test` 류)으로 존재할 것으로 추정되나, 이번 체크아웃 범위 밖이다.

**C API 직접 구현 vs C++ 래퍼 경로 비교**

2) CPU/GPU 실제 구현과 등록 레이어 관계

**cpu 레이어 관계도** `cpu_client.cc`의 `GetPjRtCpuClient(CpuClientOptions options)`(`cpu_client.cc:219`)가 실제 디바이스 개수 계산(`CpuDeviceCount()`), 스레드풀 크기 결정, `CpuTopologyDescription`/`PjRtCpuDevice` 생성까지 모두 담당하는 무거운 팩토리다. 반환 타입 `PjRtClient*`는 실제로 `PjRtCpuClient`(`cpu_client.h:86`, `CommonPjRtClient` 상속)이며, 이 클래스는 `Compile`, `CompileAndLoad`, `CreateRawBufferChannel` 등 PJRT C++ API 전체 표면을 구현한다. `plugin/xla_cpu/xla_cpu_pjrt_client.cc:27-31`의 `GetXlaPjrtCpuClient(CpuClientOptions options)`는 이 `GetPjRtCpuClient`를 그대로 호출해 반환할 뿐이다(`return xla::GetPjRtCpuClient(options);`, TODO 주석으로 "PJRT Sandwich로 감쌀 예정"이라 적혀 있어 향후 변경 가능성을 코드 자체가 명시). 즉 **래핑 관계**이며, `cpu_client.h:561`의 `GetPjRtCpuClient` 선언에는 `ABSL_DEPRECATED("Use public XLA:CPU GetXlaPjRtCpuClient instead")`가 붙어 있다 — 레거시 함수는 아직 살아있지만 신규 코드는 `plugin/xla_cpu`의 진입점을 쓰도록 유도하는 이행기 상태임을 코드가 스스로 보여준다.

**gpu 레이어 관계도** `se_gpu_pjrt_client.cc`의 `GetStreamExecutorGpuClient(const GpuClientOptions& options)`(헤더 선언 `se_gpu_pjrt_client.h:260`)가 실제 `StreamExecutorGpuClient`(`se_gpu_pjrt_client.h:112`, `PjRtStreamExecutorClient` 상속)를 만드는 무거운 팩토리다. `plugin/xla_gpu/xla_gpu_pjrt_client.cc:27-33`의 `GetXlaPjrtGpuClient(GpuClientOptions options)`는 `use_tfrt_gpu_client`일 때 `use_async_dispatch`를 강제로 켜는 정책 한 줄을 추가한 뒤 `GetStreamExecutorGpuClient(options)`를 그대로 호출한다. CPU와 마찬가지로 **래핑 관계**이지만, CPU 쪽과 달리 GPU 쪽 어댑터에는 옵션 보정 로직이 한 줄 들어 있다는 점이 차이다. 아키텍처적으로 주목할 점: CPU 실구현(`PjRtCpuClient`)은 `CommonPjRtClient`를, GPU 실구현(`StreamExecutorGpuClient`)은 `PjRtStreamExecutorClient`를 상속한다 — 서로 다른 중간 베이스 클래스를 쓰는 이질적 구조다. `CommonPjRtClient`/`PjRtStreamExecutorClient`의 실제 정의는 이번 체크아웃에 없어(둘 다 Glob 매칭 0건) 상세 비교는 못했지만, **CPU가 상대적으로 더 최근에 리팩터링된 공통 베이스로 옮겨갔고 GPU는 아직 과거 StreamExecutor 기반 클래스를 쓰는 과도기로 추정**된다.

**registration 흐름** `cpu_static_registration.cc`와 `gpu_static_registration.cc`는 각각 3줄짜리 파일로, `#include "xla/pjrt/c/pjrt_c_api_cpu_internal.h"` / `pjrt_c_api_gpu_internal.h`를 통해 (이번 스냅샷에는 없는) `pjrt::cpu_plugin::GetCpuPjrtApi()` / `pjrt::gpu_plugin::GetGpuPjrtApi()`를 얻고, `REGISTER_PJRT_PLUGIN(kCpuPjrtName, ...)` / `REGISTER_PJRT_PLUGIN(kGpuPjrtName, ...)` 매크로 한 줄로 등록을 끝낸다. 이 `GetCpuPjrtApi`/`GetGpuPjrtApi`는 example_plugin의 `GetMyPluginPjrtApi`와 동일한 역할(2단계, `pjrt::CreatePjrtApi(...)` 래핑)을 수행할 것으로 강하게 추정되지만, 정의 파일이 스냅샷 밖이라 **확인 필요**로 남긴다. `REGISTER_PJRT_PLUGIN` 매크로 자체(`static_registration.h`)도 이번 스냅샷에 없어 정확한 구현은 확인하지 못했으나, 이름과 사용 맥락(전역 `static` 객체의 생성자에서 이름→`PJRT_Api*` 맵에 등록하는 C++ static-initialization 패턴)으로 미루어 **프로세스가 이 `.cc` 파일이 포함된 바이너리를 로드하는 시점(정적 링크·동적 링크 모두 가능, 단 실행 파일에 심볼이 정적으로 포함됨)에 자동 등록된다**고 추정한다.

주요 파일과 역할

| 파일 | 역할 | |---|---| | `example_plugin/myplugin_cpp_pjrt.h/.cc` | 최소 `xla::PjRtClient` 구현체(`MypluginPjrtClient`), 문서 1단계 옵션 B | | `example_plugin/myplugin_c_pjrt_internal.cc` | C API 콜백 구현 + `GetMyPluginPjrtApi`, 문서 1단계 옵션 A + 2단계 | | `example_plugin/myplugin_c_pjrt.h/.cc` | 외부에 노출되는 `GetPjrtApi` 심볼 | | `xla_cpu/xla_cpu_pjrt_client.h/.cc` | CPU 레거시 클라이언트를 감싸는 얇은 진입점 `GetXlaPjrtCpuClient` | | `xla_cpu/cpu_static_registration.cc` | CPU `PJRT_Api`를 정적으로 레지스트리에 등록 | | `xla_gpu/xla_gpu_pjrt_client.h/.cc` | GPU 레거시 클라이언트를 감싸는 얇은 진입점 `GetXlaPjrtGpuClient` (옵션 보정 로직 포함) | | `xla_gpu/gpu_static_registration.cc` | GPU `PJRT_Api`를 정적으로 레지스트리에 등록 | | `xla_tpu/tpu_dynamic_registration.cc` | 런타임 라이브러리 경로 환경변수(`TPU_LIBRARY_PATH`)로 TPU 플러그인을 동적 등록 | | `cpu/cpu_client.h/.cc` | 실제 CPU 백엔드 구현: `PjRtCpuClient`, `PjRtCpuExecutable`, `PjRtCpuLoadedExecutable` | | `cpu/cpu_pjrt_compiler.h` | CPU 전용 `PjRtCompiler` 구현(`CpuPjRtCompiler`) | | `gpu/se_gpu_pjrt_client.h/.cc` | 실제 GPU 백엔드 구현: `StreamExecutorGpuClient`, `StreamExecutorGpuDevice` | | `gpu/se_gpu_pjrt_compiler.h` | GPU 전용 `PjRtCompiler` 구현(`StreamExecutorGpuCompiler`) |

중요한 설계 판단

  • **등록 레이어와 구현 레이어의 명확한 분리**: `plugin/xla_cpu`, `plugin/xla_gpu`는 옵션·정책 보정과 `PJRT_Api*` 노출만 담당하고, 실제 디바이스/버퍼/컴파일 로직은 전혀 갖지 않는다. 이는 `cpu_client.cc`/`se_gpu_pjrt_client.cc`가 core XLA와 강하게 결합된 레거시 코드인 반면, `plugin/` 레이어는 신규 등록 인프라(정적/동적 레지스트리, `plugin_names.h`)와 결합되도록 설계를 나눈 것으로 보인다.
  • **CPU에는 `ABSL_DEPRECATED`가 붙은 레거시 `GetPjRtCpuClient`가 남아있고 GPU에는 붙어있지 않다** — CPU 쪽이 마이그레이션이 더 진행된 상태이고 GPU는 아직 과도기임을 시사한다(추정).
  • **example_plugin은 옵션 A/B를 혼합**해서 "필수는 C++ 클라이언트, 나머지 잡다한 콜백은 C API 직접 구현"이라는 실전 패턴을 정확히 보여준다 — 순수 옵션 A만으로 example을 만들지 않은 것은 문서가 권장하는 실제 워크플로우를 반영한 판단으로 보인다.
  • **`REGISTER_PJRT_PLUGIN` vs `REGISTER_DYNAMIC_PJRT_PLUGIN`**은 이름으로 미루어 "빌드 시점에 심볼을 링크할지" vs "런타임에 공유 라이브러리를 경로로 찾아 로드할지"의 차이로 설계된 것으로 보이며, TPU처럼 별도 배포/버전 관리가 필요한 백엔드에 동적 등록이 적합하다는 판단이 반영된 것으로 추정된다.

읽으면서 주의할 점

  • 이번 체크아웃에는 `static_registration.h`, `dynamic_registration.h`, `plugin_names.h`, `pjrt_c_api_wrapper_impl.h`, `pjrt_c_api_cpu_internal.h`, `pjrt_c_api_gpu_internal.h`, `common_pjrt_client.h`, `myplugin_c_pjrt_internal.h`(구현부만 있고 헤더가 없음) 등 매크로/베이스 클래스 정의 파일이 실제로 존재하지 않는다. 이 문서에서 이들에 대해 서술한 내용은 전부 호출부 코드와 명명 규칙, 참고 문서로부터의 **추론**이며 단정이 아니다.
  • `example_plugin`의 `MypluginPjrtClient`는 디바이스가 0개인 뼈대이므로, 실제 새 백엔드를 만들 때는 `devices_`/`memory_spaces_`를 채우는 로직과 `Compile`/`Execute` 오버라이드가 추가로 필요하다(이 예제만으로는 실행 가능한 백엔드가 아님).
  • CPU와 GPU가 서로 다른 중간 베이스 클래스(`CommonPjRtClient` vs `PjRtStreamExecutorClient`)를 쓴다는 점 때문에, 두 구현체의 오버라이드 목록을 1:1로 비교하며 "표준 인터페이스"를 유추하면 오해가 생길 수 있다.

더 파볼 질문

  • `PJRT 예` 문서가 언급한 CPU/GPU 플러그인은 `plugin/xla_cpu`, `plugin/xla_gpu`와 **래핑 관계**다: `cpu_client.cc`/`se_gpu_pjrt_client.cc`가 실구현체를 만들고, `plugin/xla_cpu`/`plugin/xla_gpu`는 그 팩토리를 호출만 하는 얇은 어댑터다. 대체 관계가 아니다.
  • C API 직접 구현(옵션 A)은 "`xla::PjRtClient`가 표현할 수 없는 콜백"(예: `PJRT_Plugin_Initialize`, extension 관련 훅)에만 국한해서 쓰고, 클라이언트 본체는 옵션 B(C++ + 래퍼)로 만드는 것이 example_plugin이 보여주는 실전 선택 기준이다. 순수 옵션 A는 XLA C++ 코드를 전혀 링크할 수 없는 완전 독립 플러그인에서만 필요할 것으로 추정된다.
  • `REGISTER_PJRT_PLUGIN`(static)은 바이너리 빌드 시점에 `PJRT_Api*`를 심볼로 포함시켜 프로세스 시작과 동시에 사용 가능하게 하고, `REGISTER_DYNAMIC_PJRT_PLUGIN`(dynamic, TPU 사례)은 `TPU_LIBRARY_PATH` 같은 환경변수 문자열을 등록해 두었다가 실제 사용 시점에 `dlopen`류로 로드하는 지연 로딩 방식으로 추정된다(매크로 정의 파일 부재로 확인 필요).
  • 새 백엔드 추가 시 최소로 구현해야 하는 것은 (1) `xla::PjRtClient` 서브클래스에서 최소 가상 함수 집합(`platform_name`, `process_index`, `platform_id`, `device_count`, `devices`, `addressable_devices`, `memory_spaces`, `platform_version`) — `MypluginPjrtClient`가 정확히 이 최소 집합을 보여줌, (2) 클라이언트를 생성하는 팩토리 함수, (3) 이를 감싸는 `Get*PjrtApi()`(`pjrt::CreatePjrtApi` 호출), (4) 공개 `extern "C" const PJRT_Api* GetPjrtApi()` 심볼. 문서 4단계(JAX 통합: `jax_plugins` 네임스페이스/entry_point, `initialize()`)는 Python 쪽 절차라 이번 C/C++ 분석 범위 밖이지만, 그 전제 조건이 바로 위 4가지임을 코드가 뒷받침한다.
