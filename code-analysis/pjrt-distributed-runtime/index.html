XLA PJRT distributed/ 모듈 분석 — 멀티 호스트 조정 런타임

한 줄 요약

xla/pjrt/distributed/는 PJRT 자체의 컴퓨팅 로직과는 독립된 랑데부/바리어/KV 저장소 기반 클러스터 부트스트랩 레이어로, 여러 프로세스가 자신의 로컬 디바이스 토폴로지를 교환해 하나의 GlobalTopologyProto를 만들고, 이를 GPU 백엔드가 PjRtClient 생성 시점에 사용할 수 있도록 준비하는 역할을 한다. (이 checked-out 스코프에는 그 GPU 백엔드 코드 자체는 포함되어 있지 않아, 연결부는 코드 근거 + 추정으로 설명한다.)

분석 범위

  • distributed.h/.cc — 서비스/클라이언트 생성 진입점(팩토리 함수)
  • service.h/.cc — DistributedRuntimeService, CoordinationServiceImpl (코디네이터 프로세스 측)
  • client.h/.cc — DistributedRuntimeClient, DistributedRuntimeCoordinationServiceClient (워커 프로세스 측)
  • key_value_store_interface.h, in_memory_key_value_store.h/.cc — KV 저장소 추상화와 단일 프로세스용 구현
  • topology_util.h/.cc — 로컬/글로벌 토폴로지 교환 및 GPU 토폴로지 구성
  • protocol.proto — 토폴로지 페이로드 메시지 정의

이 저장소 스니펫은 repo/.git/info/sparse-checkout에 명시된 대로 위 12개 파일만 체크아웃되어 있다(확인 완료). 따라서 xla/pjrt/gpu/se_gpu_pjrt_client.cc 등 상위 PJRT GPU 클라이언트 코드는 로컬에 존재하지 않으며, 이 문서의 "PjRtClient와의 연결점" 섹션은 이 디렉터리 내부의 타입/주석 근거 + 명명 관례를 통한 추정으로 작성했다.

전체 구조

이 모듈은 크게 3계층으로 나뉜다.

  1. RPC/코디네이션 계층 — service.*, client.*가 xla/pjrt/distributed/coordination/(스코프 밖) 의 CoordinationService/CoordinationServiceAgent를 감싸는 얇은 어댑터다. 실제 barrier·heartbeat·KV RPC 로직은 이 스코프에 없다.
  2. KV 추상화 계층 — key_value_store_interface.h가 정의하는 KeyValueStoreInterface는 코디네이션 서비스에 종속되지 않은 범용 Get/Set 인터페이스이며, InMemoryKeyValueStore(단일 프로세스)와 DistributedKeyValueStore(client.cc 내부, gRPC 기반)가 이를 구현한다.
  3. 토폴로지 계층 — topology_util.*가 KV 저장소를 이용해 각 노드의 LocalTopologyProto를 모아 GlobalTopologyProto/GpuTopologyProto로 합성한다. protocol.proto는 이 계층의 페이로드 메시지만 정의한다.

먼저 볼 코드 리딩 순서

  1. key_value_store_interface.h의 KeyValueStoreInterface, MultiProcessKeyValueStore — 이 모듈 전체가 결국 "Get/Set 가능한 KV 저장소 하나"를 중심으로 돌아간다는 것을 먼저 이해해야 나머지 코드가 쉬워진다.
  2. in_memory_key_value_store.cc — absl::Mutex::AwaitWithTimeout으로 blocking get을 구현한 가장 단순한 구현체라, KV 저장소의 동작 계약(존재하지 않으면 timeout 후 NotFound)을 코드 20줄 이내로 파악할 수 있다.
  3. protocol.proto — DeviceProto/LocalTopologyProto/GlobalTopologyProto의 필드를 먼저 보면 이후 topology_util.cc의 로직이 "무엇을 조립하는지" 바로 이해된다.
  4. topology_util.cc의 ExchangeTopologies — 이 모듈의 핵심 비즈니스 로직이며, KV 저장소를 이용한 리더 기반 랑데부 패턴을 직접 볼 수 있다.
  5. topology_util.cc의 BuildGlobalTopology, BuildGpuTopology — partition_index/global_device_id 할당 규칙과 GPU 대칭성 검사를 이해하면 왜 이 정보가 PjRtClient 쪽에 필요한지 짐작할 수 있다.
  6. service.h/.cc — DistributedRuntimeService::Get → CoordinationServiceImpl 생성 → BuildAndStart → StartRpcThread 순서로 코디네이터 프로세스의 초기화 흐름을 본다.
  7. client.cc의 DistributedRuntimeCoordinationServiceClient — Connect()/Shutdown()과 KV API들이 전부 coord_agent_(스코프 밖 CoordinationServiceAgent)로 위임되는 얇은 어댑터임을 확인한다.
  8. client.cc 하단의 DistributedKeyValueStore, GetDistributedKeyValueStore — client가 어떻게 KeyValueStoreInterface로 재포장되어 topology_util.cc 같은 소비자에게 넘어가는지 연결고리를 확인한다.
  9. distributed.h/.cc — 위 모든 조각을 gRPC 채널/자격증명과 함께 묶는 최상위 팩토리 함수(GetDistributedRuntimeService, GetDistributedRuntimeClient)를 마지막에 본다.

서비스/클라이언트 역할 분리

service 초기화 순서 (service.h/.cc)

DistributedRuntimeService::Get(address, credentials, options) (service.cc:84)가 유일한 생성 경로다.

  • EnableCoordinationService(options)가 xla::CoordinationService(스코프 밖 타입)를 num_tasks=num_nodes, cluster_register_with_barrier=true, heartbeat_timeout, shutdown_barrier_timeout 등으로 구성.
  • 4-스레드 ThreadPool(coord_compute_pool_) 생성.
  • GrpcCoordinationServiceImpl(스코프 밖)을 builder에 등록하고 방금 만든 coord_service_ 인스턴스를 주입.
  1. grpc::ServerBuilder를 만들고 AddListeningPort(address, credentials)로 리스닝 주소를 등록 (메시지 크기 제한 해제: SetMax{Receive,Send}MessageSize(-1)).
  2. DistributedRuntimeService 생성자(service.cc:102)가 CoordinationServiceImpl(options, &builder)를 생성한다. 이 안에서:
  3. builder->BuildAndStart()로 실제 gRPC 서버가 리스닝을 시작한다.
  4. coord_impl_->StartRpcThread()가 별도 스레드에서 HandleRPCsLoop()을 돌려 RPC를 처리한다.

즉 "gRPC 서버 빌드 → 코디네이션 서비스 등록 → 서버 시작 → RPC 처리 스레드 시작" 순서가 고정되어 있고, 서버가 완전히 뜬 뒤에야 RPC 루프가 붙는다. ~CoordinationServiceImpl()(service.cc:69)은 coord_service_를 먼저 파괴해 진행 중인 RPC를 정리한 뒤 coord_rpc_service_->Shutdown()을 부르는 순서를 명시적으로 지킨다(주석: "Service object must be destroyed to clear all pending RPCs before shutting down the RPC service").

client 연결 흐름 (client.h/.cc)

DistributedRuntimeCoordinationServiceClient 생성자(client.cc:98)는:

  1. Options(node_id, heartbeat_timeout, shutdown_timeout, init_timeout 등)를 CoordinationServiceAgent::Config로 변환.
  2. NewGrpcCoordinationClient(channel)로 리더(서비스)에 대한 gRPC 클라이언트 생성.
  3. CoordinationServiceAgent::Create(env, node_id, config, leader_client, missed_heartbeat_callback)로 실제 에이전트(coord_agent_)를 만든다. 실패 시 생성자 자체는 죽지 않고 LOG(ERROR)만 남긴 채 coord_agent_가 null로 남는 점이 눈에 띈다(아래 "주의할 점" 참고).

이후 Connect()(client.cc:129)는 coord_agent_->Connect() 한 줄로 위임되며, 이 호출이 "모든 노드가 cluster register barrier에 도달할 때까지" 블록한다(cluster_register_with_barrier=true가 service 쪽에 설정돼 있으므로). 타임아웃 시 DeadlineExceeded를 잡아 "일부 태스크가 크래시/미스케줄됐을 수 있다"는 힌트 로그를 남긴다(client.cc:134-143).

missed_heartbeat_callback의 기본값(client.h:78-88)은 LOG(FATAL)로 프로세스를 통째로 죽이는 것이며, 테스트에서만 오버라이드 가능하다는 점이 이 모듈의 "fail-fast" 철학을 보여준다.

KV store 사용 지점

  • DistributedRuntimeClient의 KV류 가상 함수(BlockingKeyValueGet, KeyValueTryGet, KeyValueIncrement, KeyValueDirGet, KeyValueSet, KeyValueDelete, AsyncKeyValueGet)는 전부 coord_agent_의 동명 메서드로 그대로 전달된다(client.cc:157-211). 즉 KV 저장소의 실제 저장 위치는 코디네이션 서비스 프로세스(코디네이터) 안이며, client는 RPC 클라이언트일 뿐이다.
  • client.cc:283의 DistributedKeyValueStore(내부 클래스)가 DistributedRuntimeClient를 KeyValueStoreInterface로 재포장하며, 모든 키에 prefix_를 붙인다. GetDistributedKeyValueStore(client, prefix)(client.cc:316)로 노출된다 — 이것이 topology_util.cc 같은 소비자가 실제로 받는 타입이다.
  • WaitAtBarrier(barrier_id, timeout, nodes)(client.h:161)는 이름이 명시하듯 바리어 전용 API이며, 코드상 topology_util.cc는 이걸 직접 쓰지 않고 대신 KV Set/Get을 리더-팔로워 패턴으로 직접 조합해 랑데부를 구현한다(아래 "핵심 흐름" 참고) — barrier API와 topology-exchange 로직이 서로 다른 동기화 메커니즘을 쓴다는 점은 설계상 주목할 부분.

PjRtClient와의 연결점

연결 지점 코드 근거

이 스코프 안에서 직접 확인 가능한 근거는 다음과 같다(모두 xla/pjrt/gpu/... 등 상위 파일을 열지 않고도 이 디렉터리 내부 문서/타입에서 확인됨):

  • key_value_store_interface.h:31-37의 클래스 주석: "In the multi-node case, the caller of PjRtClient can provide a key-value store accessible across nodes... The caller is responsible for avoiding key collisions between different users of key-value store." — KeyValueStoreInterface가 PjRtClient 생성자에게 넘겨지는 파라미터로 설계되었음을 텍스트로 명시.
  • key_value_store_interface.h:66-70의 MultiProcessKeyValueStore{key_value_store, process_index, process_count} — KV 저장소 하나에 프로세스 랭크/총수를 묶은 구조체로, PjRtClient 팩토리 함수의 파라미터 형태와 정확히 일치하는 모양이다(추정: 실제 시그니처는 스코프 밖).
  • topology_util.h:70-75의 BuildGpuTopology(GlobalTopologyProto, gpu::GpuTargetConfig, cpu::TargetMachineOptions) -> GpuTopologyProto에서 반환 타입 GpuTopologyProto는 xla/service/gpu_topology.pb.h에 정의되어 있고(include만 확인, 정의 파일은 스코프 밖), 이름 규칙상 GPU PjRtClient/PjRtTopologyDescription 쪽에서 소비될 가능성이 높다.
  • protocol.proto의 DeviceProto에 fabric_uuid("NVLink"/"SM100 Blackwell" 명시), compute_capability, shared_memory_per_block_optin(CUDA 드라이버 enum 이름을 그대로 딴 필드) 등 CUDA/NVIDIA 특화 필드가 다수 존재 — 이 프로토콜이 범용이 아니라 사실상 GPU 백엔드 전용으로 설계되었음을 강하게 시사한다.

확인이 필요함: 실제로 ExchangeTopologies/BuildGpuTopology를 호출하는 코드(예상 위치: xla/pjrt/gpu/se_gpu_pjrt_client.cc 또는 gpu_topology.cc)는 이번 sparse-checkout에 포함되어 있지 않아 grep으로 직접 검증하지 못했다. 위 결론은 코드 내부 주석과 타입 설계, 명명 관례에 근거한 추정이다.

이 모듈이 없을 때(단일 호스트) vs 있을 때(멀티 호스트)의 차이

ExchangeTopologies(topology_util.cc:284) 안에 이 분기가 명시적으로 코드화되어 있다:

  • 단일 호스트 (num_nodes == 1): KV 저장소를 전혀 쓰지 않는다. global_topology에 자신의 local_topology를 그대로 복사하고, global_device_id를 local_device_ordinal로 바로 지정한다(topology_util.cc:294-300). kv_store가 nullptr이어도 동작한다.
  • 멀티 호스트 (num_nodes > 1): CHECK(kv_store != nullptr)(topology_util.cc:302)로 KV 저장소가 필수임을 강제한다. 이후 리더(node_id==0)가 전 노드의 로컬 토폴로지를 KV에서 모아 BuildGlobalTopology로 합성하고 다시 KV에 발행하며, 나머지 노드는 그 결과를 KV에서 blocking get 한다. partition_index/global_device_id도 network proximity, fabric_uuid, boot_id 등 여러 호스트에 걸친 정보를 종합해 결정된다(단일 호스트에서는 이런 계산이 전혀 필요 없음).

즉 이 디렉터리는 "멀티 호스트일 때만 활성화되는 조정 계층"이며, 단일 호스트 실행 경로는 이 모듈을 사실상 우회한다 — num_nodes==1 분기가 그 근거다.

핵심 흐름

입력 → 각 프로세스가 node_id, num_nodes, 서비스 address, 자신의 LocalTopologyProto(로컬 디바이스 목록)를 갖고 시작.

변환/검증 →

  • client.cc의 DistributedRuntimeCoordinationServiceClient::Connect()가 클러스터 등록 바리어를 통과시켜 "모든 노드가 살아있음"을 검증.
  • topology_util.cc:GetAllLocalTopologies가 KV에서 모은 각 노드의 직렬화된 LocalTopologyProto를 파싱, 실패 시 최대 10개까지 에러 메시지를 모아 InternalError로 반환.
  • SameLocalTopology(topology_util.cc:71)로 재시작 시 기존 값과 새 값이 다르면 명시적으로 InternalError를 내는 idempotency 체크.

상태 변경 →

  • KV 저장소(InMemoryKeyValueStore 또는 DistributedKeyValueStore)에 local_topology/{platform}/{node_id}, global_topology/{platform} 키가 채워짐.
  • BuildGlobalTopology가 입력 local_topologies의 DeviceProto를 제자리에서 변형(partition_index, global_device_id 필드를 설정)한다는 점이 주석("Steals the contents")에 명시되어 있다 — 호출자는 이후 원본 local_topology를 재사용하면 안 된다.

외부 호출/결과 →

  • 최종적으로 GlobalTopologyProto(및 GPU라면 GpuTopologyProto)가 만들어지고, 이 값이 (스코프 밖) PjRtClient 생성 경로로 전달된다고 추정된다.
  • DistributedRuntimeClient::Shutdown()은 coord_agent_->Shutdown()을 호출해 종료 바리어(shutdown_timeout)를 통과시키며, 이 타임아웃이 만료되면 에러 상태를 반환하되 프로세스를 강제 종료하지는 않는다(연결 실패 시 missed_heartbeat_callback이 LOG(FATAL)인 것과 대비됨).

주요 파일과 역할

파일역할
distributed.h/.ccinsecure credentials 기반 기본 service/client 팩토리. 보안 자격증명이 필요하면 사용자가 직접 DistributedRuntimeService::Get/GetDistributedRuntimeClient(channel, options)를 호출하도록 분리돼 있음(주석 명시).
service.h/.cc코디네이터 프로세스용 gRPC 서버 래퍼. CoordinationServiceImpl이 실제 CoordinationService(스코프 밖) 인스턴스를 소유.
client.h/.cc워커 프로세스용 클라이언트 인터페이스와 CoordinationServiceAgent 어댑터 구현. KV store를 KeyValueStoreInterface로 재노출하는 GetDistributedKeyValueStore도 여기 위치.
key_value_store_interface.hKV 저장소 추상 인터페이스 + MultiProcessKeyValueStore 파라미터 구조체.
in_memory_key_value_store.h/.cc단일 프로세스/테스트용 KV 구현. AsyncGet은 미구현(UnimplementedError).
topology_util.h/.cc로컬 토폴로지 수집→글로벌 토폴로지 합성→GPU 토폴로지 산출 로직. 이 모듈의 실질적 "비즈니스 로직".
protocol.protoDeviceProto/LocalTopologyProto/GlobalTopologyProto 등 토폴로지 페이로드 메시지 정의. RPC(service/rpc)는 정의하지 않음.

중요한 설계 판단

  • KV 저장소를 barrier/RPC와 분리된 범용 인터페이스로 추상화: KeyValueStoreInterface는 코디네이션 서비스뿐 아니라 InMemoryKeyValueStore 같은 완전히 다른 구현으로도 교체 가능하게 설계됐다. 덕분에 topology_util.cc는 자신이 gRPC 위에서 도는지 단일 프로세스 테스트인지 몰라도 된다.
  • 토폴로지 교환에 전용 barrier API 대신 KV Get/Set 리더-팔로워 패턴 사용: WaitAtBarrier가 이미 존재함에도 ExchangeTopologies는 이를 쓰지 않고 node_id==0을 리더로 삼아 직접 Get-all/Build/Set 패턴을 구현했다. 동기화와 데이터 교환(글로벌 토폴로지 자체를 전달)을 한 번에 처리해야 했기 때문으로 보인다(추정).
  • num_nodes==1 최적화 분기: KV 저장소 의존성을 단일 호스트 경로에서 완전히 제거해, 로컬 단일 프로세스 실행 시 이 모듈의 RPC/동기화 오버헤드가 전혀 들지 않도록 했다.
  • fail-fast 기본 정책: missed_heartbeat_callback 기본값이 LOG(FATAL)(client.h:78-88)이라, 별도 설정 없이는 한 노드의 장애가 전체 프로세스 그룹을 종료시킨다. Options.recoverable이 이를 완화할 수 있는 플래그로 존재하지만 기본값은 false.
  • protocol.proto가 RPC를 정의하지 않는 설계: barrier/heartbeat/KV RPC는 범용 CoordinationService(TSL 계층, 스코프 밖)에 위임하고, 이 proto는 그 KV 저장소에 실릴 "페이로드 스키마"만 담당한다. 즉 이 디렉터리는 PJRT 전용 새 RPC 프로토콜을 만든 게 아니라 기존 TensorFlow/TSL 코디네이션 서비스를 재사용하는 얇은 어댑터에 가깝다.

읽으면서 주의할 점

  • protocol.proto 파일 상단 주석과 실제 메시지 정의가 어긋난다: 주석(protocol.proto:18-22)은 "barriers... health checking... GPU topology 공유"를 이 프로토콜의 목적으로 설명하지만, 실제로 정의된 메시지는 DeviceProto/LocalTopologyProto/GlobalTopologyProto뿐이며 barrier/heartbeat용 RPC나 메시지는 이 파일에 없다(다른 proto인 TSL coordination_service.proto가 담당, 스코프 밖). 문서를 읽고 이 파일에서 barrier RPC 정의를 찾으면 안 된다.
  • BuildGlobalTopology는 입력을 파괴적으로 변형한다: absl::Span<LocalTopologyProto> local_topologies를 매개변수로 받아 Swap으로 내용을 훔쳐간다(주석 "Steals the contents"). 호출 후 원본 벡터를 재사용하면 안 된다.
  • 클라이언트 생성자가 에이전트 생성 실패를 조용히 삼킬 수 있다: client.cc:117에서 CoordinationServiceAgent::Create가 실패하면 LOG(ERROR)만 남기고 coord_agent_는 null로 남는다. 이후 Connect() 등에서 coord_agent_->Connect()를 바로 호출하므로, 이 경로는 null 역참조 위험이 있다(스코프 안에서 명시적 null 체크는 보이지 않음 — "확인이 필요함").
  • InMemoryKeyValueStore::AsyncGet은 항상 실패한다: absl::UnimplementedError를 즉시 done()으로 전달한다. topology_util.cc는 AsyncGet을 쓰지 않아 문제없지만, 이 KV 구현을 다른 곳에서 async 경로로 재사용하면 조용히 깨질 수 있다.
  • ExchangeTopologies의 KV 재시도(AlreadyExists) 처리는 "같은 노드의 재시작"만 가정한다: 다른 노드가 우연히 같은 키를 다시 쓰는 경쟁 상황은 고려되어 있지 않다(키가 platform+node_id로 이미 유일하므로 설계상 문제는 없지만, 재시작 판별 로직(SameLocalTopology)이 실패하면 전체 교환이 InternalError로 죽는다는 점은 운영상 주의할 부분).

더 파볼 질문

  • distributed::Service와 distributed::Client는 각각 어느 프로세스에서 실행되는가? 이 스코프의 코드 자체는 프로세스 배치를 강제하지 않는다 — GetDistributedRuntimeService/GetDistributedRuntimeClient(distributed.h)는 완전히 독립된 팩토리 함수다. 다만 API 설계(address+node_id)로 미루어 보면 관례적으로 한 프로세스(보통 rank 0)가 서비스+클라이언트를 동시에 띄우고, 나머지 프로세스는 클라이언트만 띄워 그 주소로 접속하는 구조로 쓰인다(JAX의 jax.distributed.initialize 관례에 대한 일반 지식 기반 추정 — 그 상위 런처 코드는 스코프 밖).
  • protocol.proto가 정의하는 RPC는 무엇을 조율하는가? 엄밀히는 "정의하는 RPC가 없다." 이 파일은 메시지 스키마만 정의하며, 실제 barrier/heartbeat/KV RPC는 TSL CoordinationService(스코프 밖)가 담당한다. protocol.proto가 실질적으로 조율하는 것은 KV 저장소에 실리는 GPU 디바이스 토폴로지 페이로드의 직렬화 포맷뿐이다.
  • topology_util.cc가 만든 토폴로지 정보는 PjRtClient/PjRtDevice의 어느 코드 경로로 전달되는가? key_value_store_interface.h의 클래스 주석("caller of PjRtClient can provide a key-value store")과 BuildGpuTopology의 반환 타입(GpuTopologyProto, xla/service/gpu_topology.pb.h)으로 미루어 GPU PjRtClient 생성 함수의 파라미터로 흘러들어간다고 추정되나, 실제 호출부(xla/pjrt/gpu/se_gpu_pjrt_client.cc 등)가 이 sparse-checkout에 없어 grep으로 확정할 수 없다. 다음 분석 스코프에 xla/pjrt/gpu/se_gpu_pjrt_client.cc와 xla/pjrt/pjrt_client.h를 포함시키면 이 연결을 코드로 직접 검증할 수 있다.
  • 이 디렉터리가 4개 참고 문서 어디에도 언급되지 않은 이유: 참고 문서 4편은 모두 pjrt_c_api.h 기반 플러그인 C ABI(PJRT_Client_Create, PJRT_TopologyDescription_Create 등)를 다루는 플러그인 개발자용 가이드다(문서 내 유일한 관련 언급은 "PJRT 플러그인 통합" 문서의 PJRT_TopologyDescription_Create뿐이며, 이는 "[선택사항]"으로 표시되어 있다). 반면 xla/pjrt/distributed/는 C++ 전용 내부 구현이고, protocol.proto의 fabric_uuid/compute_capability/CUDA 드라이버 enum 이름을 그대로 딴 필드들이 보여주듯 사실상 GPU(NVIDIA) 백엔드 전용 구현체에 가깝다. PJRT의 C ABI 계약은 토폴로지 교환 메커니즘을 규정하지 않으므로(오직 opaque PJRT_TopologyDescription만 요구), 플러그인 통합 가이드가 이 특정 구현을 언급할 필요가 없었을 것으로 추정된다.
