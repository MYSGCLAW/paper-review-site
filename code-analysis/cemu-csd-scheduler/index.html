CEMU CSD 컴퓨트 유닛 스케줄러와 QoS 분석

한 줄 요약

hw/femu/csd/sched.c의 단일 sched_thread가 우선순위 큐(pqueue)로 ComputeUnit(가상 스케줄링 슬롯)을 순회하며 CsfScheduler 플러그인의 7개 콜백을 호출해 "언제 어떤 잡을 얼마 동안 실행 중인 것으로 칠지"를 결정하는 가상시간 기반 성능 시뮬레이터이며, 실제 CSF 코드 실행(기능 시뮬레이션)은 별도의 nr_thread개 스레드에서 비동기로 이루어지고 job->stat.exec_time을 통해서만 두 흐름이 연결된다.

분석 범위

  • hw/femu/csd/sched.c / sched.h: 스케줄러 코어 로직, ComputeEngine/ComputeUnit/ComputeJob/JobGroup 정의, CsfScheduler 인터페이스
  • hw/femu/csd/sched_fifo.c, sched_rr.c, sched_grouped_fifo.c, sched_grouped_rr.c: 4종 스케줄러 구현체
  • hw/femu/csd/compute.h: ComputeNamespace/ComputeParams/Program 정의 (nr_cu, nr_thread, csf_runtime_scale 필드)
  • hw/femu/timing-model/timing.c, timing.h: NAND 채널/칩 레벨 타이밍 모델 (CSD 잡 스케줄링과는 별개 계층)
  • hw/femu/statistics/statistics.c, statistics.h, femu_stat.py, analysis.py, plot_*.py: 공유메모리 기반 통계 export

범위 제약 (확인 필요): 이 체크아웃에는 hw/femu/csd/compute.c가 포함되어 있지 않다. 따라서 csf_runtime_scale이 실제로 어디서 곱해지는지, 기능 시뮬레이션 스레드(compute_threads)가 job->stat.exec_time을 어떻게 세팅하는지, CsfSchedulerOption 구조체의 필드(algo, time_slice, context_switch_time, take_care_of_group*)가 어디서 파싱되는지는 헤더의 선언과 references/text/cemu__csd-README.txt의 설명으로만 추정했다. 코드로 직접 확인한 부분과 README/구조 문서 기반 추정을 구분해 표기했다.

전체 구조

CSD 스케줄러는 두 개의 독립된 실행 축을 가진다.

  1. 성능 시뮬레이션 축 (sched.c의 sched_thread): ComputeEngine.cu[nr_cu]라는 가상 코어 배열을 우선순위 큐로 관리하며, 각 CU가 "다음에 리스케줄될 시각"(next_sched_time)이 되면 CsfScheduler 콜백을 호출해 다음 잡을 고르고 time_slice/context_switch_time만큼 시간을 전진시킨다. 잡의 실제 완료 여부는 job_exec_time()(=job->stat.exec_time)과 비교해서만 판단한다.
  2. 기능 시뮬레이션 축 (코드 범위 밖, compute.c 추정): sched_enqueue_job()이 scheduler->pick_thread()(또는 기본 라운드로빈 ce->next_thread)로 고른 OS 스레드에서 실제 eBPF/공유라이브러리 코드를 실행하고, 그 결과 실행시간을 set_sched_runtime()을 통해 job->stat.exec_time에 기록한다(추정).

이 두 축은 ComputeJob.stat(스핀락 보호)만을 공유 상태로 연결되어 있어, "가상 스케줄링 결정"과 "실제 실행 시간 측정"이 명확히 분리된다.

먼저 볼 코드 리딩 순서

  1. sched.h의 CsfScheduler 구조체 (L178~235) — 스케줄러 플러그인이 구현해야 할 7개 콜백의 계약을 먼저 이해해야 나머지 코드가 읽힌다.
  2. sched.h의 ComputeJob/ComputeUnit/ComputeEngine/JobGroup (L21~165) — 스케줄링 대상(잡), 스케줄링 단위(CU), 소유자(엔진), QoS 묶음(그룹)의 관계를 파악한다.
  3. sched.c의 sched_init() (L179~286) — csf_sched_option.algo/take_care_of_group으로 4종 스케줄러 중 하나를 고르는 선택 로직과 그룹 0/1/2 사전 할당을 확인한다.
  4. sched.c의 sched_thread() (L534~634) — 전체 스케줄러의 심장부. pqueue로 CU를 고르고 콜백을 호출하는 메인 루프.
  5. sched_fifo.c 전체 — 가장 단순한 구현체로, pick_next_job/enqueue_job이 어떻게 최소 구현되는지 기준점을 잡는다.
  6. sched_rr.c — FIFO와 비교하며 time_slice/원형 리스트로 라운드로빈이 어떻게 구현되는지 본다.
  7. sched_grouped_fifo.c, sched_grouped_rr.c — 그룹별 큐 분리와 group->prio 사용 방식의 차이(중요한 설계 판단 항목 참고)를 비교한다.
  8. sched.c의 job_switched_out/job_switched_in/job_finish (L288~531) — 잡 상태 전이와 통계(RequestStat) 기록 지점을 확인한다.
  9. statistics.h/statistics.c + femu_stat.py/analysis.py — C 쪽에서 채운 통계가 공유메모리를 거쳐 파이썬으로 어떻게 소비되는지 확인한다.

  • IO->>Stage: sched_alloc_job()/sched_enqueue_job()(sched.c:377,390)이 ComputeJob을 할당하고 stage_job()(sched.c:361)으로 링버퍼에 적재한다. 그룹 배정은 job_group_add()(sched.c:80)에서 발생한다.
  • IO->>Func: sched_enqueue_job() 내부에서 scheduler->pick_thread가 있으면 호출, 없으면 ce->next_thread 라운드로빈으로 실행 스레드를 고른다(sched.c:414~421). 실제 실행/exec_time 세팅은 범위 밖 코드(추정).
  • Sched->>Stage: enqueue_staged_job()(sched.c:439)이 새 잡을 모든 허용 CU(permited_cu_mask)의 enqueue_job 콜백에 등록하고, 반환값이 1(선점 필요)이면 해당 CU를 cu_mask에 표시한다.
  • Sched->>Plugin: pick_next_job: sched_thread()의 안쪽 while(1) 루프(sched.c:585~631)에서 매 CU마다 호출되며, time_slice/context_switch_time은 ComputeUnit의 기본값에서 시작해 스케줄러가 in-out 파라미터로 덮어쓸 수 있다.
  • Sched->>Sched: cu.next_sched_time: 이 값이 바로 "가상 시간"이 전진하는 지점이며, pqueue_change_priority()로 CU 우선순위 큐를 재정렬한다(sched.c:628~629).
  • Sched->>Plugin: dequeue_job: job_finish()(sched.c:288) 안에서 잡이 허용된 모든 CU에 대해 호출된다(sched.c:314~322).

핵심 흐름

입력: 호스트가 NVMe Program Execute 커맨드를 보내면 sched_alloc_job()이 ComputeJob을 슬랩 할당(ce->job_slab, sched.c:55~66)하고, sched_enqueue_job()이 exec->group(커맨드에 담긴 그룹 id)으로 JobGroup을 찾아 job_group_add()로 그룹의 jobs 큐에 연결한다. 기본 그룹은 sched_init()에서 id 0, 1, 2로 미리 3개가 할당되며 모두 DEFAULT_JOB_GROUP_PRIO(5)로 초기화된다(sched.c:239~253).

변환/검증: stage_job()이 링버퍼가 가득 찼는지 검사하고, 가득 찼으면 즉시 job_finish()로 실패 처리한다(sched.c:361~375). sched_thread의 enqueue_staged_job()은 job->permited_cu_mask를 확인해 해당 잡이 허용된 모든 CU의 enqueue_job 콜백을 호출하고, 모든 CU가 거부(pushed == 0)하면 NVME_DNR 상태로 잡을 종료시킨다(sched.c:476~482).

상태 변경: sched_thread 메인 루프는 pqueue에서 가장 이른 next_sched_time을 가진 CU를 꺼내 job_switched_out()(이전 잡 러닝타임 정산·그룹 active_runtime 갱신·완료시 job_finish 호출)과 pick_next_job()(다음 잡 선정)을 순서대로 호출한다. 잡이 처음 스케줄되면 그룹의 jobs_queueing을 감소, jobs_running을 증가시키고(sched.c:608~616), job_switched_in()이 last_sched_time을 now + context_switch_time으로 설정한다 — 이 값이 이후 job_switched_out에서 실제 러닝타임(runtime += now - last_sched_time) 계산의 기준점이 된다.

외부 호출/결과: 잡이 job_has_finished()(즉 job->runtime >= job_exec_time(job))를 만족하면 job_finish()가 NvmeRequest 통계를 채우고(req->stat.reqlat, exec.queueing_time, exec.runtime 등, sched.c:298~304) femu_ring_enqueue()로 to_poller 링에 완료 요청을 넣어 NVMe 컴플리션 처리 스레드로 넘긴다(sched.c:326~336). 이 시점에 job_group_remove()와 job_free()로 자원을 회수한다.

중요한 설계 판단

1. 새 스케줄링 알고리즘을 추가하려면 무엇을 구현해야 하는가? CsfScheduler 구조체(sched.h:178)를 채운 전역 인스턴스를 만들고 sched.h:241~244의 extern 목록과 sched.c의 sched_init() 선택 분기(sched.c:194~216, 현재는 algo 문자열 "fifo"/"rr" + take_care_of_group 불리언 조합만 인식)에 등록해야 한다. 필수 콜백은 pick_next_job, enqueue_job(sched.c:221~222에서 assert로 강제)뿐이고 나머지 5개(init/fini/dequeue_job/switched_from/pick_thread)는 옵션이다. enqueue_job은 잡이 허용된 CU마다 별도로 호출되므로(멀티 CU 허용 시) 스케줄러 내부에서 잡의 중복 등록을 스스로 관리해야 한다(rr/grouped_rr는 job->sched_info로 중복 등록을 assert로 방지).

2. 4종 스케줄러는 어떤 상황에서 유리한가?

스케줄러선점/시간분할그룹 인지특징
fifo없음 (잡이 끝날 때까지 점유)없음단순 처리량 위주, 짧은 잡이 긴 잡 뒤에서 굶주릴 수 있음

grouped_rrtime_slice 단위 순환, 그룹별 동적 계산그룹 단위 라운드로빈, group->prio는 QoS API가 아니라 요청의 chunk_nlb(전송 크기)로 매 도착마다 재계산(sched_grouped_rr.c:154~168)도착 간격/크기 기반의 적응형 시간분할이 필요할 때, 단 MAX_GROUPS=8 제한 있음

3. "가상 시간"은 어떻게 진행되며 벽시계 시간과 어떻게 분리되는가? sched_thread는 매 반복마다 clock_ns()(정의부는 이 체크아웃 범위 밖, 확인 필요)를 호출해 now를 얻고, cu->next_sched_time = now + context_switch_time + time_slice로 각 CU의 다음 스케줄 시각을 미리 계산해 우선순위 큐에 넣는다(sched.c:628~629). 즉 스케줄링 결정 자체는 실제 벽시계 시각을 폴링해서 이루어지므로 완전한 이산 이벤트 시뮬레이션이 아니라 벽시계 기반 근사(busy-loop polling) 방식이다(코드로 확인). 반면 잡의 "실제로 얼마나 오래 걸리는가"(job_exec_time)는 기능 시뮬레이션 스레드가 실행한 결과에 csf_runtime_scale(compute.h:89)을 곱해 결정되는 것으로 README(references/text/cemu__csd-README.txt "CSF Runtime Simulation" 절)에 기술되어 있으나, 그 곱셈이 일어나는 코드(compute.c 추정)는 이번 분석 범위에 없어 확인이 필요하다. 스케줄링(언제/얼마나 점유했다고 칠지)과 실행시간 산정(얼마나 걸릴지)이 이렇게 분리되어 있는 것이 "성능 시뮬레이션 vs 기능 시뮬레이션" 분리의 핵심이다.

4. deadline/bandwidth QoS는 보장인가 best-effort인가? 저장소 내 검색 결과 JobGroup.bandwidth와 JobGroup.deadline(sched.h:94~95)은 sched_job_group_set_qos()(sched.c:156)에서 설정될 뿐, 4개 스케줄러 어디에서도 읽히거나 시행되지 않는다(코드로 확인, compute.c가 범위 밖이라 그쪽에서 참조할 가능성은 완전히 배제할 수 없음 — 확인 필요). 실제로 스케줄링 결정에 반영되는 QoS 신호는 오직 prio뿐이며, 그마저도 grouped_fifo는 사용자가 설정한 prio(1~9)를 그대로 쓰지만 grouped_rr는 chunk_nlb로 매번 덮어써 사용자가 설정한 값이 실질적으로 무시된다(sched_grouped_rr.c:155). 위반 시 어떤 조치(드롭, 재시도, 에러 반환)도 코드에 없으므로 현재 구현상 deadline/bandwidth는 미시행(unenforced) 필드로 판단된다.

5. 경쟁 조건 방지 스케줄링 결정 자체는 sched_thread 단일 스레드가 CU 큐/그룹 큐를 순차적으로만 조작하므로 CU 간 경쟁은 구조적으로 없다. 실제 락이 필요한 지점은 스레드 경계를 넘는 공유 상태뿐이다: job->lock(스핀락, job->stat.exec_time을 기능 시뮬레이션 스레드가 쓰고 스케줄 스레드가 읽음, sched.h:249~255), group->lock(jobs_running/jobs_queueing/active_runtime 카운터, IO 스레드·스케줄 스레드 동시 접근), ce->staged_jobs.lock(IO 스레드가 push, 스케줄 스레드가 drain, sched.c:361~374, sched.c:445~489), ce->job_group_lock/슬랩 락(그룹 할당/해제).

6. 통계 수집과 export ComputeJobStat(시작/실행/종료시각, 컨텍스트 스위치 수)과 ComputeUnitStat(CU별 idle/exec 시간)이 statistics.h에 정의되고, job_finish()에서 RequestStat.exec(ExecStat: runtime, queueing_time, csf_id, thread, jit 여부 등)로 집계된다. femu_stat_init()이 shm_open+mmap으로 /dev/shm/FEMU-STAT-SHM-<drive_id>를 여는데, 실제 shm 생성은 analysis.py의 open_shm()이 먼저 수행한다(주석 "shm is created in analysis.py", statistics.c:21). femu_stat_add_req()가 링버퍼(FemuStat.req_stat[FEMU_STAT_CAPACITY])에 매 요청을 덮어쓰기 방식으로 기록하고(statistics.c:63~69), analysis.py가 stat.tail을 폴링하며 process()로 opcode별 분기 처리해 IOPS/지연분포/late-io/P2P 바이트 등을 에폭 단위로 집계한다. femu_stat.py의 open_shm()이 /* START */~/* END */ 마커 사이의 C 선언을 파싱해 cffi cdef를 자동 생성하는 점이 특징이다(코드-헤더 동기화를 매크로 파싱으로 강제).

읽으면서 주의할 점

  • sched.h의 CsfScheduleInfo, WokerThreadPool, CSE_CORE_COUNT, CSD_THREAD_COUNT(L16~26, L69~77)는 저장소 전체에서 다른 곳에 전혀 참조되지 않는다(grep 확인) — 죽은/레거시 타입으로 보이며 실제 흐름과 무관하다. 혼동하지 말 것.
  • sched_grouped_rr.c의 struct job_queue는 groups[MAX_GROUPS](8개 고정 배열)를 job->group->id로 직접 인덱싱한다(sched_grouped_rr.c:32~36, 139~140). 그런데 MAX_JOB_GROUPS는 1024이고 슬랩으로 그룹 id가 순차 할당된다(sched.c:236). 사용자가 8개 이상의 그룹을 만들고 grouped_rr를 쓰면 배열 범위를 벗어난 쓰기가 발생할 수 있다 — 실사용 전 반드시 확인/수정이 필요한 잠재적 버그로 보인다.
  • grouped_rr의 group->prio는 QoS API(sched_job_group_set_qos)가 아니라 매 잡 도착시 chunk_nlb * 10으로 덮어써진다(그룹 id 0 제외). sched_job_group_set_qos()로 설정한 prio는 grouped_rr에서는 사실상 무의미해진다는 점을 놓치기 쉽다.
  • job_group_free()(sched.c:117)는 아직 실행/대기 중인 잡이 있는 그룹을 삭제하면 해당 잡들을 그룹 0으로 강제 이관한다(sched.c:123~145) — 삭제 즉시 실패하지 않고 조용히(에러 로그만 남기고) 재배치되는 동작이므로, 그룹 삭제 후 QoS가 갑자기 바뀔 수 있음에 유의.
  • pick_thread는 4개 스케줄러 모두 구현하지 않으므로, 현재 제공된 스케줄러들은 항상 기본 라운드로빈(ce->next_thread)으로 기능 시뮬레이션 스레드를 배정한다(sched.c:414~421). CU(가상 코어, nr_cu)와 기능 시뮬레이션 스레드(nr_thread)는 서로 다른 카운트이며 기본 배정 로직도 서로 독립적이라는 점을 혼동하지 말 것.
  • compute.c가 범위 밖이므로 csf_runtime_scale의 실제 적용 지점, CsfSchedulerOption 파싱 위치, job->stat.exec_time 최초 설정 지점은 이 문서에서 코드로 검증하지 못했다.
