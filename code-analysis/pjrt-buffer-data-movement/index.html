PjRtBuffer 데이터 이동과 비동기 이벤트 분석

한 줄 요약

`PjRtBuffer`는 실제 데이터 소유·전송·동기화 책임을 `AbstractTrackedDeviceBuffer`(정의/사용 이벤트 추적) → `PjRtRawBuffer`(원시 디바이스 메모리 + 전송 프리미티브) → `PjRtDeviceEvent`(C-API 호환 비동기 값) 3단으로 위임하는 구조이며, `GetReadyFuture()`는 이 이벤트 체인을 `device_event_utils.h`의 `RunWhenReady`(참조 카운트 기반 join)로 모아 `Future<>`로 변환한다.

분석 범위

  • 대상: `xla/pjrt/pjrt_client.h`, `raw_buffer.{h,cc}`, `staging_buffer.{h,cc}`, `host_to_device_transfer_manager.{h,cc}`, `device_event.{h,cc}`, `device_event_utils.{h,cc}`, `abstract_tracked_device_buffer.{h,cc}`, `host_memory_allocator.h`, `host_memory_spaces.h`, `semaphore.{h,cc}`, `pjrt_future.h` (17개 파일, `repo/xla/pjrt/` 하위)
  • **주의**: 이 체크아웃은 위 파일들만 포함된 부분 스냅샷이다. `common_pjrt_client.h`, `xla/future.h`, CPU/GPU 백엔드 구현(`pjrt_stream_executor_client.*` 등)은 포함되어 있지 않아 직접 인용할 수 없고, 호출 관계로만 추론했다. 이 부분은 "확인 필요"로 표기한다.
  • 참고 문서: OpenXLA "PJRT C++ Device API 개요" 한국어 페이지(`references/text/...cpp_api_overview...`)를 코드와 대조했다.

전체 구조: 상위 PJRT와의 연결 (간략)

참고 문서는 `PjRtClient` → `PjRtDevice`/`PjRtMemorySpace` → `PjRtBuffer` 순의 소유 관계와, 버퍼가 "메모리 공간은 알지만 기기를 반드시 알 필요는 없다"는 점을 명시한다. 이번 분석 대상 모듈들은 이 계층 중 **`PjRtBuffer`의 구현 세부사항**(데이터가 실제로 어디에 있고 언제 준비되는지)에 해당하며, `PjRtClient`/`PjRtDevice`/`PjRtExecutable` 자체의 구현(컴파일, 실행 스케줄링)은 범위 밖이다. `CommonPjRtBuffer`(`abstract_tracked_device_buffer.h`)가 `PjRtBuffer`의 공용 구현 기반이며, 이를 구체화하는 `CommonPjRtClient`는 이번 체크아웃에 포함되지 않은 상위 오케스트레이터로만 참조된다 (`memory_space->client()`를 `absl::down_cast<CommonPjRtClient*>`하는 코드가 `abstract_tracked_device_buffer.cc:54,128`, `host_to_device_transfer_manager.cc:76` 등 여러 곳에서 등장).

1. 버퍼 생명주기

1.1 생성 경로

`pjrt_client.h`의 `PjRtClient`는 세 가지 생성 경로를 제공한다.

  • **동기 생성**: `BufferFromHostBuffer`(`pjrt_client.h:980`), `BufferFromHostLiteral`(`:1001`) — 즉시 `PjRtBuffer`를 반환하되, 내부 전송은 비동기일 수 있다. `HostBufferSemantics` enum(`:933-969`)이 호스트 데이터의 공유/복사/변경 가능 여부를 규정한다(아래 1.2 참고).
  • **비동기(지연) 생성**: `CreateBuffersForAsyncHostToDevice`(`:902`, `:915`)가 `AsyncHostToDeviceTransferManager`(`:827` 추상 클래스)를 반환한다. 이 매니저의 구체 구현이 `host_to_device_transfer_manager.cc`의 `CommonAsyncHostToDeviceTransferManager::Create`(`:64`)이다.
  • **초기화 없는/에러/별칭 생성**: `CreateUninitializedBuffer`, `CreateAliasBuffer`, `CreateErrorBuffer`(`:736-755`) — 아직 데이터가 없거나 에러 상태로 시작하는 버퍼.

`CommonAsyncHostToDeviceTransferManager::Create`의 핵심 단계(`host_to_device_transfer_manager.cc:64-174`): 1. 각 `ShapeSpec`마다 `client->CreateAllocationEventForTransfers(...)`로 **지연 할당 이벤트**를 만든다. 주석(`:87-91`)에 따르면 이는 원격 서버에서 데이터가 올 때까지 빈 HBM을 붙잡지 않기 위한 최적화다. 2. `client->AllocateRawBuffer(...)`로 `PjRtRawBufferRef`(원시 디바이스 메모리)를 확보한다. 3. `client->CreateLinkedEventPromise(...)`로 `definition_event_promise`/`definition_event` 쌍을 만든다. 4. `client->DefineBuffer(shared_device_shape, memory_space, raw_buffer, {definition_event})`로 실제 `PjRtBuffer`를 만들고, 이를 `RetrieveBuffer(int)`(`:213`)를 통해 즉시 호출자에게 반환할 수 있다. **이 시점에 정의 이벤트는 아직 미완료 상태이므로 버퍼는 "존재하지만 준비되지 않은" 상태다.**

1.2 전송 경로 (BufferFromHostBuffer / ToLiteral 왕복)

**Host → Device (쓰기, H2D):**

  • 일괄(one-shot) 버퍼는 `BufferFromHostBuffer` 경로를 통하며, `HostBufferSemantics`(`pjrt_client.h:933`)로 소유권 규칙을 정한다:
  • `kImmutableOnlyDuringCall`: 런타임은 호출이 끝나기 전에 데이터를 복사해야 하며, `on_done_with_host_buffer`가 반환 전에 호출됨.
  • `kImmutableUntilTransferCompletes`: 전송 완료(정의 이벤트 준비)까지 호스트 데이터를 참조할 수 있음.
  • `kImmutableZeroCopy` / `kMutableZeroCopy`: 버퍼가 호스트 메모리를 직접 별칭(alias)하며, 버퍼가 살아있는 동안 호출자가 데이터를 보존(및 mutable 버전은 런타임이 이를 변형 가능).
  • 참고 문서(`227`행)의 "버퍼 시맨틱: 공유/복사/변경 가능 여부"가 정확히 이 enum에 대응한다.
  • 분할 전송(streaming) 경로는 `AsyncHostToDeviceTransferManager::TransferLiteralToBuffer`/`TransferRawDataToSubBuffer`(`pjrt_client.h:851,878`)이며, 구현은 `host_to_device_transfer_manager.cc:218,323`이다. `TransferLiteralToBuffer`는 `client_->LinearizeInto(literal, device_shape, kImmutableUntilTransferCompletes, raw_buffer)`(`:263-268`)로 리터럴을 디바이스 레이아웃으로 변환하며 raw buffer에 기록하고, 완료 시 `definition_event.Set(transfer_event)`(`:306`)로 정의 이벤트를 채워 다운스트림 소비자를 unblock한다.

**Device → Host (읽기, D2H, `ToLiteral`):**

  • `PjRtBuffer::ToLiteral(MutableLiteralBase*)`(`pjrt_client.h:1224`)와 지연 할당 버전 `LazyToLiteral`(`:1230`)이 순수 가상 함수로 선언되어 있으며, 실제 구현은 이번 체크아웃 범위 밖(백엔드별 `CommonPjRtBuffer` 파생 클래스)이다.
  • 다만 D2H 전송의 저수준 빌딩 블록은 `raw_buffer.cc`의 `ToStagingBuffer()`(`:137-196`)에서 확인 가능하다. 이 함수는 두 갈래로 갈라진다:
  • `raw_buffer->GetHostPointer()`가 널이 아니면(호스트에서 직접 보이는 메모리, 예: CPU) — 복사 없이 `HostRawBufferPjRtStagingBuffer`로 그 포인터를 그대로 감싼다(제로카피).
  • 널이면 — 호출자가 제공한 `allocate_staging_buffer(size, memory_space)`로 별도 스테이징 메모리를 할당하고, `raw_buffer->CopyRawDeviceToHostAndReturnEvent(dst, 0, size)`로 실제 DMA 복사를 발행한 뒤, 그 이벤트가 완료되면 `WrappedPjRtStagingBuffer`를 채운다.
  • 즉 `ToLiteral`의 실질적 디바이스-메모리 판독은 `PjRtRawBuffer::CopyRawDeviceToHost[AndReturnEvent]`(`raw_buffer.h:150,176`)가 담당하고, `staging_buffer.h`는 그 결과를 담는 "임시 바이트 배열" 인터페이스(`data()`/`const_data()`만 존재, `staging_buffer.h:32-34`)일 뿐이다.

1.3 삭제 및 외부 참조 처리

  • `PjRtBuffer::Delete()`(`pjrt_client.h:1287`)는 순수 가상. 구현은 `CommonPjRtBuffer`가 내부적으로 `AbstractTrackedDeviceBuffer::Delete(memory_space)`(`abstract_tracked_device_buffer.cc:143-159`)를 사용하는 패턴을 따른다고 추정된다(직접 오버라이드는 이 파일셋에 없음 — **확인 필요**).
  • `AbstractTrackedDeviceBuffer::Delete`는 정의 이벤트 + 사용 이벤트를 모두 모은 뒤 `LockUsageEvents()`로 더 이상 사용 이벤트 추가를 막고, `raw_buffer.release()->DecrefAfter(std::move(events))`(`:157`)를 호출한다. `DecrefAfter`(`raw_buffer.cc:112-114`)는 `xla::RunWhenReady(events, [this](){ DropRef(); })`이므로 **메모리 참조 해제(refcount 감소)는 진행 중인 모든 연산이 끝난 뒤 지연 실행**된다 — 문서의 "Delete는 비동기 연산 완료 후 메모리를 지연 해제한다"는 설명과 일치.
  • **외부 참조**: `PjRtBuffer::AcquireExternalReference()`(`pjrt_client.h:1216`, `ExternalReference` 내부 클래스 `:1200`)와 `ReleaseDeviceMemoryOwnership`(`:1305`)이 소유권을 프레임워크(NumPy/DLPack 등)로 이전하는 API다. `CommonPjRtBuffer::ScopedHold`(`abstract_tracked_device_buffer.h:202`)가 이를 뒷받침하는 락 메커니즘:
  • Hold 종류는 `kUsage`, `kExternalReference`, `kDonation` 세 가지(`:204`).
  • **불변식**: 도네이션 홀드는 "쓰기 잠금"처럼 동작하여, 외부 참조가 살아있으면 도네이션을 거부하고(`abstract_tracked_device_buffer.cc:215-218`), 도네이션 홀드가 진행 중이면 다른 어떤 홀드도 블록된다(`WaitForOutstandingDonationHold`, `:189-194`).
  • `CommonPjRtBuffer::~CommonPjRtBuffer()`(`:176-180`)는 소멸 시 모든 홀드 카운트가 0이어야 함을 `CHECK_EQ`로 강제한다 — 즉 외부 참조가 살아있는 한 버퍼 객체 자체의 파괴도 막힌다는 설계.

2. 비동기 동기화 메커니즘

2.1 Future 개념과 GetReadyFuture 대응 코드

1. `tsl::MakePromise<void>()`로 `[definition_promise, definition_future]`를 만든다. 2. `raw_buffer()->GetRawBufferAsyncValue()`(할당 이벤트, `client->include_raw_buffer_in_ready_event()`가 true일 때만) + `definition_events()` 각각을 `dependencies`(`PjRtDeviceEventRefVector`)에 모은다. 이미 concrete인 이벤트는 제외한다(`:66,73`의 `IsConcrete()` 체크). 3. `xla::RunWhenReady(deps_span, ...)`(device_event_utils.h)로 이 모든 이벤트가 준비될 때 콜백을 실행하도록 등록한다. 콜백은 각 이벤트의 에러 여부를 취합해 `definition_promise.Set(status)`로 future를 완료시킨다.

  • `pjrt_future.h`는 `xla/future.h`로의 단순 재전달(`#include "xla/future.h"`, 실제 정의는 이번 체크아웃 범위 밖 — **확인 필요**)이며, `Future<T>`는 `tsl::MakePromise<T>()`로 생성되는 promise/future 쌍으로 코드 전반에서 관찰된다(예: `raw_buffer.cc:82`, `abstract_tracked_device_buffer.cc:56`).
  • `PjRtBuffer::GetReadyFuture()`(`pjrt_client.h:1394`)의 실제 구현 예시는 `AbstractTrackedDeviceBuffer::GetReadyFuture(PjRtMemorySpace*)`(`abstract_tracked_device_buffer.cc:52-106`)다:
  • **문서 대응**: 참고 문서(`226`행)의 `xla::Future<> ToLiteral(...)` 시그니처, `258`행의 "x 값이 필요하면 GetReadyFuture에서 반환된 future를 통해 준비되었다고 선언할 때까지 실행이 차단된다"는 서술이 이 구현과 정확히 일치한다.

2.2 device_event의 역할

`device_event.h`는 PJRT C API(`PJRT_DeviceEvent`)와 C++ 네이티브 `tsl::AsyncValue`를 모두 감쌀 수 있는 **경량 vtable 기반 래퍼**를 제공한다.

  • `PjRtDeviceEventPtr`(`:103`) — non-owning 핸들. `AndThen`, `GetErrorIfPresent`, `state()`, `down_cast<T>()`를 제공하며, 내부적으로 `IsCompatibleWithLocalAsyncValue()`(`device_event.cc:154`)로 "이 이벤트가 로컬 `tsl::AsyncValue`인가, 아니면 C API 너머의 외부 플러그인 이벤트인가"를 판별해 빠른 경로(직접 `AsyncValue::AndThen`)와 느린 경로(vtable 호출)를 나눈다.
  • `PjRtDeviceEventRef`(`:217`) — 소유(owning) 참조, RAII로 ref count 관리.
  • `PjRtDeviceEventPromise`/`PjRtDeviceEventPromiseRef`(`:298,320`) — 값을 나중에 채우는 프로미스(예: `TransferLiteralToBuffer`의 `definition_event_promise`).
  • `PjRtDeviceEventRefVector`/`PjRtDeviceEventSpan`(`:374,420`) — C API 호환 벡터/스팬(내부적으로 `PJRT_DeviceEventVector` C 구조체를 직접 감쌈).

**"실행 큐잉"과의 연결**은 `device_event_utils.h`의 `ScopedLauncher`(`:28`)가 담당한다:

  • `AddDependency(event)`가 호출될 때마다, 이벤트가 아직 미완료면 `state_`(참조 카운트된 `Callee` 객체)를 지연 생성하고 그 이벤트의 `AndThen`에 `state_`를 캡처한 람다를 건다(`device_event_utils.cc:64-77`).
  • `Callee`의 소멸자(`:39-45`)가 실제 task를 실행한다(옵션으로 `tsl::Executor`에 위임, 없으면 즉시 호출).
  • **설계 포인트**: 명시적 카운터 대신 **참조 카운팅 자체를 join/countdown-latch로 재사용**한다 — 각 의존성 이벤트가 완료되면 그 이벤트가 쥐고 있던 `state_` 참조 하나가 사라지고, 모든 의존성 + `ScopedLauncher` 자신이 소멸해야 `Callee` refcount가 0이 되어 task가 실행된다. `RunWhenReady`/`ExecuteWhenReady`(`:55-77`)가 이를 감싼 편의 함수다.
  • `AbstractTrackedDeviceBuffer::Delete`의 `DecrefAfter`, `GetReadyFuture`의 `RunWhenReady` 호출이 모두 이 메커니즘을 사용하므로, **device_event는 "이 데이터가 언제 준비/해제 가능한가"를 나타내는 토큰이고, ScopedLauncher/RunWhenReady가 그 토큰들을 모아 후속 작업을 큐잉하는 실행 계층**이다.

2.3 semaphore가 사용되는 지점

  • `Semaphore`(`semaphore.h:27`)는 표준적인 카운팅 세마포어로, `Acquire`/`TryAcquire`/`Release`와 RAII `ScopedReservation`(`:50`)을 제공한다. 구현(`semaphore.cc`)은 `absl::Mutex::LockWhen`으로 블로킹 획득을 구현하는 평범한 코드다.
  • **확인 사실**: 이번에 제공된 17개 파일 및 `repo/xla/pjrt` 전체를 grep한 결과, `semaphore.h`를 include하거나 `Semaphore`를 사용하는 곳은 `semaphore.cc/h` 자기 자신뿐이었다. `raw_buffer`, `abstract_tracked_device_buffer`, `host_to_device_transfer_manager`, `device_event*` 어디에서도 참조하지 않는다.
  • **해석(추정)**: 이 리포지토리 스냅샷이 `xla/pjrt/` 전체가 아니라 job.json에 명시된 파일만 포함한 부분 체크아웃이므로, `Semaphore`는 아마 CPU/GPU 등 백엔드별 클라이언트 구현(예: 과거 `pjrt_stream_executor_client.cc`류에서 동시 H2D 전송 개수를 제한하던 "transfer semaphore" 패턴)에서 소비될 가능성이 높다. 하지만 이 파일셋 안에서는 **직접적인 연결 고리를 확인할 수 없다** — 질문 4("device_event/semaphore가 실행 큐잉과 어떻게 연결되는가")에 대해, 실행 큐잉의 실질적 구현은 `device_event`+`ScopedLauncher`(2.2절)이며, `semaphore.h`는 이 스냅샷 기준으로는 동시성 처리량 제한용 범용 프리미티브로 존재하되 현재 분석 대상 모듈들과는 배선되어 있지 않다고 판단하는 것이 정확하다. **단정하지 않음 — 확인 필요.**

3. 계층 구조: raw_buffer vs abstract_tracked_device_buffer (질문 2 답변)

세 계층이 뚜렷하게 구분된다.

| 계층 | 파일 | 책임 | |---|---|---| | **원시 메모리/전송** | `raw_buffer.h/.cc` | `PjRtRawBuffer`(C++ 구현체) / `PjRtRawBufferInterface`(C API-호환 핸들, vtable 디스패치). "이 바이트가 어디 있고 어떻게 복사·슬라이스하는가"만 안다. `Slice`, `CopyRawHostToDevice[AndReturnEvent]`, `CopyTo`/`ScheduleCopyTo`, `MakeAllocationReadyEvent`. 논리적 shape, dtype, 홀드 개념이 전혀 없다. | | **정의/사용 이벤트 추적** | `abstract_tracked_device_buffer.h/.cc` — `AbstractTrackedDeviceBuffer` | `PjRtRawBufferRef` 하나를 감싸며 `definition_events_`(언제 데이터가 준비되는가)와 `usage_events_`(언제 안전하게 mutate/free 가능한가)를 추적. `GetReadyFuture`, `Delete`, `BlockForOperationsToComplete`를 제공. | | **PjRtBuffer 공개 API + 동시성 상태기계** | `abstract_tracked_device_buffer.h` — `CommonPjRtBuffer : public PjRtBuffer` | `AbstractTrackedDeviceBuffer`를 `device_buffer_`로 소유하고, `ScopedHold`(Usage/External/Donation)로 도네이션·삭제·외부공유 사이의 상호배제를 구현. `on_device_shape()`, `memory_space()`, `device()` 등 나머지 `PjRtBuffer` 순수 가상 함수는 **더 하위의 백엔드별 파생 클래스**(이 체크아웃에 없음)가 채운다. |

`PjRtRawBuffer`는 재사용 가능한 "메모리 소유권+전송" 단위이고(예: `CreateRawAliasOfBuffer`로 기존 `PjRtBuffer`에서 원시 뷰를 뽑아낼 수 있음, `raw_buffer.cc:116`), `AbstractTrackedDeviceBuffer`/`CommonPjRtBuffer`는 그 위에서 "언제 쓸 수 있는가", "누가 지금 쓰고 있는가", "donation 중인가"를 조율하는 동시성 레이어다.

질문 답변 요약

1. **"PjRtBuffer는 MemorySpace만 알고 Device는 모른다"는 문서 설명**: `pjrt_client.h:1188`에서 `memory_space()`는 순수 가상으로 필수 구현이지만, `device()`(`:1190`)는 `ABSL_DEPRECATED`는 아니지만 주석 `// TODO(b/277820585): remove device() after the migration is done.`이 붙어 있어 **레거시 하위호환용으로 남아있을 뿐 설계상 우선순위가 아님**이 코드에 그대로 드러난다. `AbstractTrackedDeviceBuffer`/`CommonPjRtBuffer` 어디에도 `device()`의 기본 구현이 없고, 오직 `memory_space_`(`abstract_tracked_device_buffer.h:379`)만 멤버로 보유한다 — memory_space를 통해서만 디바이스를 추이적으로 알 수 있다는 문서 설명과 일치. 2. **raw_buffer vs abstract_tracked_device_buffer 계층 차이**: 위 3절 참고. 3. **staging_buffer vs host_to_device_transfer_manager**: `PjRtStagingBuffer`(`staging_buffer.h`)는 **단일 linearize/delinearize 복사 1회**에서 쓰이는 임시 바이트 뷰 인터페이스(zero-copy 또는 raw buffer로부터의 D2H 복사 결과)일 뿐이며, `raw_buffer.cc:ToStagingBuffer()`가 이를 만든다. 반면 `CommonAsyncHostToDeviceTransferManager`(`host_to_device_transfer_manager.cc`)는 **버퍼 객체 자체의 탄생부터 여러 청크에 걸친 전송, 정의 이벤트 완료까지 전체 세션**을 관장하는 상위 오케스트레이터다 — "언제 어떤 바이트가 어디로 가는가"(staging_buffer)와 "전체 비동기 전송 세션의 상태 기계"(transfer manager)로 나뉜다. 4. **`x = foo(1)` 시나리오의 device_event/semaphore 구현**: 2.2절 `ScopedLauncher`/`RunWhenReady`가 실질적 구현. `x`가 반환된 시점엔 `definition_events_`가 아직 미완료 상태(`GetReadyFuture`가 pending future 반환)이며, 이후 `print(x+1)`처럼 실제 값이 필요해지는 지점에서 `.Await()`(future.h, 범위 밖)를 호출하면 그때 블록된다. `semaphore.h`는 이 흐름에 현재 직접 배선되어 있지 않음(2.3절 "확인 필요" 참고).

주요 파일과 역할

| 파일 | 핵심 타입/함수 | 역할 | |---|---|---| | `pjrt_client.h` | `PjRtClient`, `PjRtBuffer`, `AsyncHostToDeviceTransferManager`, `HostBufferSemantics` | 공개 API 표면 정의(순수 가상 다수) | | `raw_buffer.{h,cc}` | `PjRtRawBuffer`, `PjRtRawBufferInterface`, `ToStagingBuffer` | 원시 디바이스 메모리 + 전송 프리미티브 | | `staging_buffer.{h,cc}` | `PjRtStagingBuffer` | linearize/delinearize용 임시 바이트 뷰 | | `host_to_device_transfer_manager.{h,cc}` | `CommonAsyncHostToDeviceTransferManager` | 지연 H2D 버퍼 생성·청크 전송 오케스트레이션 | | `abstract_tracked_device_buffer.{h,cc}` | `AbstractTrackedDeviceBuffer`, `CommonPjRtBuffer`, `ScopedHold` | 정의/사용 이벤트 추적 + 홀드 기반 동시성 제어 | | `device_event.{h,cc}` | `PjRtDeviceEventPtr/Ref/Promise` | C API 호환 비동기 이벤트 래퍼 | | `device_event_utils.{h,cc}` | `ScopedLauncher`, `RunWhenReady`, `BlockUntilReady` | 이벤트 join 후 후속 작업 큐잉/블로킹 | | `semaphore.{h,cc}` | `Semaphore`, `ScopedReservation` | 범용 카운팅 세마포어(이 파일셋 내 미사용) | | `pjrt_future.h` | (재전달) | `xla/future.h`로의 alias, 정의는 범위 밖 | | `host_memory_allocator.h` | `HostMemoryAllocator`, `BasicHostMemoryAllocator` | 호스트측 정렬 메모리 할당(DMA 친화적) | | `host_memory_spaces.h` | `UnpinnedHostMemorySpace`, `PinnedHostMemorySpace`, `CpuDeviceMemorySpace` | 호스트 인접 메모리 종류별 `PjRtMemorySpace` 구현 |

중요한 설계 판단

  • **지연 할당(lazy allocation)**: `CommonAsyncHostToDeviceTransferManager`는 실제 HBM 할당을 첫 데이터 청크가 도착할 때까지 미룬다(`allocation_events_[i].reset()`, `host_to_device_transfer_manager.cc:233,354`). 원격 소스에서 오는 데이터를 기다리는 동안 빈 메모리를 점유하지 않기 위한 명시적 트레이드오프(주석 `:87-91`).
  • **refcount를 join 프리미티브로 재사용**: `ScopedLauncher`가 명시적 카운터 대신 `tsl::RCReference<Callee>`의 소멸을 트리거로 쓰는 것은 락-프리에 가까운 경량 설계지만, 디버깅 시 "왜 이 콜백이 실행되지 않았는가"를 추적하려면 refcount 그래프를 이해해야 하는 비용이 있다.
  • **C-API 이중화(vtable) 패턴 반복**: `PjRtMemorySpace`/`PjRtMemorySpaceCApiDelegator`, `PjRtRawBuffer`/`PjRtRawBufferInterface`, `PjRtDeviceEvent(Ptr/Ref)`/`PJRT_DeviceEvent` 세 곳 모두 동일한 "C++ 네이티브 구현과 C API 플러그인 구현을 같은 인터페이스로 다루기" 패턴을 쓴다. 즉 이 모듈들은 순수 C++ 편의 계층이면서 동시에 PJRT C API ABI 경계이기도 하다.
  • **Donation과 다른 홀드 간 상호배제를 카운터+Mutex.Await로 구현**: 락-프리 자료구조 대신 `absl::Mutex::Await(Condition)`를 사용해 단순성을 택했다(`WaitForOutstandingUsageHolds`/`WaitForOutstandingDonationHold`).

읽으면서 주의할 점

  • 이 체크아웃은 **`xla/pjrt/` 전체가 아니라 job.json에 명시된 17개 파일만 포함**한다. `common_pjrt_client.h`(`CommonPjRtClient::TrackFuture`, `CreateLinkedEventPromise`, `AllocateRawBuffer`, `DefineBuffer`, `LinearizeInto`, `AppendDescriptionToEvent` 등 핵심 오케스트레이션 함수가 전부 여기 정의됨)와 `xla/future.h`(`Future<T>`의 실제 정의)는 포함되지 않아, 이 문서의 여러 설명은 **호출 시그니처로부터의 추론**이다.
  • `semaphore.h`는 코드베이스 안에서 고아(orphan) 상태로 보이며, "실행 큐잉과 semaphore의 연결"을 코드에서 직접 증명할 수 없다. CPU/GPU 백엔드 구현을 추가로 확인해야 한다.
  • `PjRtBuffer::device()`는 존재하지만 deprecated 예정이므로, 새 코드를 작성/리뷰할 때 `memory_space()`를 우선해야 한다.
  • `ScopedHold`는 이동만 가능하고 복사 불가(`ScopedHold(const ScopedHold&) = delete`), 소멸자가 자동으로 hold를 drop하므로 스코프 관리 실수 시 데드락(도네이션 홀드가 영원히 안 풀림) 위험이 있다.

더 파볼 질문

  • `CommonPjRtClient::TrackFuture`, `CreateLinkedEventPromise`, `LinearizeInto`, `AllocateRawBuffer`의 실제 구현(`common_pjrt_client.h/.cc`, 이번 범위 밖)을 봐야 device_event가 실제 백엔드(CPU/GPU) 실행 스트림과 어떻게 매핑되는지 확정할 수 있다.
  • `Semaphore`가 실제로 어느 백엔드에서 사용되는지(예: 동시 H2D 전송 개수 제한) 전체 리포지토리에서 확인이 필요하다.
  • `use_stream_based_compaction_`(`abstract_tracked_device_buffer.h:150`, `.cc:491-527`)가 켜졌을 때의 usage event 압축 로직이 GPU 스트림 기반 백엔드에서 어떻게 쓰이는지는 `GetDefinitionStream()`을 실제로 채우는 구현체를 봐야 한다.
