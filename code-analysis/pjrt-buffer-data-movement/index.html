PjRtBuffer 데이터 이동과 비동기 이벤트 메커니즘

한 줄 요약

`PjRtBuffer`의 실제 구현은 `PjRtRawBuffer`(물리 메모리 + C API 브리지) → `AbstractTrackedDeviceBuffer`(정의/사용 이벤트 추적) → `CommonPjRtBuffer`(hold 기반 생명주기 관리)의 3단 계층으로 이루어지며, 이 위에서 `PjRtDeviceEventRef`/`PjRtDeviceEventPromiseRef`와 `xla::Future<>`가 결합해 "즉시 반환하지만 필요한 시점에만 블록하는" 비동기 실행 모델을 구현한다.

분석 범위

`job.json`에 지정된 18개 파일만 체크아웃된 저장소로, `xla/pjrt/` 아래 다른 파일(`common_pjrt_client.*`, `pjrt_client.cc`, `xla/future.h`, `cpu/`·`gpu/` 백엔드 등)은 이 스냅숏에 존재하지 않는다. 즉 이번 분석은 **PJRT 전체가 아니라, 최근 리팩터링으로 분리된 "저수준 데이터 이동/이벤트 레이어"만을 대상**으로 한다. 이 레이어를 실제로 사용하는 `CommonPjRtClient`(할당·직렬화·플랫폼별 로직)는 소스가 없어 시그니처와 호출 패턴으로만 역할을 추정했으며, 이런 부분은 본문에 "추정" 또는 "확인 필요"로 표시했다.

전체 구조 (상위 PJRT와의 연결, 짧게)

상위 PJRT 계층에서 사용자는 `PjRtClient`(`pjrt_client.h`)를 통해 `PjRtBuffer`를 만들고 `PjRtLoadedExecutable::Execute`에 넘긴다. `PjRtDevice`/`PjRtMemorySpace`/`PjRtClient`는 이번 분석 범위에서는 인터페이스 선언만 존재하고 구체 구현(`StreamExecutor` 기반 클라이언트, TPU/GPU 클라이언트 등)은 범위 밖이다. 이번 분석은 그 구현체들이 공통으로 딛고 서는 "buffer backing store" 계층 — `raw_buffer.h/.cc`, `abstract_tracked_device_buffer.h/.cc`, `device_event.h/.cc`, `device_event_utils.h/.cc`, `staging_buffer.h/.cc`, `host_to_device_transfer_manager.h/.cc`, `semaphore.h/.cc`, `host_memory_allocator.h`, `host_memory_spaces.h` — 를 집중적으로 다룬다.

먼저 볼 코드 리딩 순서

1. **`pjrt_client.h`의 `PjRtMemorySpace`/`PjRtDevice`/`PjRtBuffer` 선언(L89-303, L1122-1398)** — 전체 계층의 공개 계약과 `HostBufferSemantics`(L933-969), `AsyncHostToDeviceTransferManager`(L827-893)를 먼저 봐야 이후 구현 코드가 어떤 인터페이스를 채우는지 이해된다. 2. **`device_event.h`의 `PjRtDeviceEventRef`/`PjRtDeviceEventPromiseRef`** — 이 저장소 전체에서 가장 자주 등장하는 타입으로, C API(`PJRT_DeviceEvent`)와 `tsl::AsyncValue`를 감싼 참조 카운트 핸들이다. 이걸 모르면 다른 `.cc` 코드가 읽히지 않는다. 3. **`device_event_utils.h`의 `ScopedLauncher`/`RunWhenReady`/`BlockUntilReady`** — "N개 이벤트가 준비되면 콜백 실행"이라는 fan-in 조합자와, 실제로 스레드를 재우는 유일한 지점(`BlockUntilReady`)을 구분해서 봐야 "지연된 블로킹" 모델이 이해된다. 4. **`raw_buffer.h`의 `PjRtRawBufferInterface`/`PjRtRawBuffer`** — 물리 메모리 한 덩어리를 나타내는 가장 저수준 타입. `CopyRawHostToDeviceAndReturnEvent`, `Slice`, `CopyTo`/`ScheduleCopyTo` 시그니처를 눈에 익혀 둔다. 5. **`abstract_tracked_device_buffer.h`의 `AbstractTrackedDeviceBuffer`와 `CommonPjRtBuffer::ScopedHold`** — raw_buffer 위에 정의/사용 이벤트와 usage/external/donation hold 규칙을 얹는 곳. `ScopedHold`의 3가지 타입 설명 주석(L156-201)을 먼저 읽는다. 6. **`abstract_tracked_device_buffer.cc`의 `GetReadyFuture`와 `Delete`** — 실제 Future 생성·이벤트 fan-in·지연 해제 로직이 담긴 핵심 함수. 이번 분석에서 가장 중요한 두 함수다. 7. **`host_to_device_transfer_manager.cc`의 `CommonAsyncHostToDeviceTransferManager`** — 위의 모든 타입(raw_buffer, device_event, definition_event)이 실제로 조립되어 쓰이는 유일한 "완결된 사용 예"이므로, 여기서 개념들이 어떻게 맞물리는지 확인한다. 8. **`raw_buffer.cc`의 `ToStagingBuffer`** — `staging_buffer.h`의 추상 인터페이스가 실제로 어떻게 채워지는지 보여주는 유일한 코드. host-visible 여부에 따라 분기하는 로직을 본다. 9. **`semaphore.h/.cc`** — 마지막에 가볍게. 이 스코프 안에서는 호출부가 없는 독립 유틸리티이므로, 개념(카운팅 세마포어 + RAII 예약)만 확인하고 넘어가면 된다.

버퍼 생명주기

생성 경로

`pjrt_client.h`에는 최소 5개의 버퍼 생성 진입점이 있다: `BufferFromHostBuffer`/`BufferFromHostLiteral`(동기, L980-1014), `CreateUninitializedBuffer`(L736-740), `CreateAliasBuffer`(L744-748), `CreateBuffersForAsyncHostToDevice`(비동기, L902-929), `CreateViewOfDeviceBuffer`(외부 소유 메모리 뷰, L1028-1034), `MakeCrossHostReceiveBuffers`(L1059-1065). 이 경로들의 실제 구현부는 스코프 밖이지만, `host_to_device_transfer_manager.cc`에서 유일하게 전체 흐름을 확인할 수 있다:

1. `CommonAsyncHostToDeviceTransferManager::Create`(L64-174)가 각 `ShapeSpec`에 대해 `client->AllocateRawBuffer(...)`로 `PjRtRawBufferRef`를 얻고, `client->CreateLinkedEventPromise(...)`로 `definition_event`(promise+ref 쌍)를 만든 뒤, `client->DefineBuffer(shared_device_shape, memory_space, raw_buffer, {definition_event})`(L157-160)로 실제 `PjRtBuffer`를 만든다. `DefineBuffer` 자체는 스코프 밖이지만 시그니처로 보아 **`AbstractTrackedDeviceBuffer`를 만들어 `CommonPjRtBuffer`(=`PjRtBuffer`)로 감싸는 함수로 추정**된다. 2. 이때 실제 device 메모리는 아직 확보되지 않을 수 있다. `allocation_event`(`CreateAllocationEventForTransfers`가 반환)가 `ScopedEvent`로 보류되어 있다가, 첫 데이터가 도착할 때(`TransferLiteralToBuffer`/`TransferRawDataToSubBuffer`에서 `allocation_events_[buffer_index].reset()`) 비로소 소멸자를 통해 `SetStateConcrete()`가 호출되어 할당이 트리거된다(L232-234, L353-354, 주석: "lazy allocation design avoids holding on to empty, unusable HBM"). 3. 생성된 `PjRtBuffer`는 `RetrieveBuffer(buffer_index)`(L213-216)로 즉시 호출자에게 반환되며, 아직 `definition_event`가 미완료 상태이므로 "아직 준비 안 됨" 상태의 유효한 핸들이다.

전송 경로

  • **동기(`BufferFromHostBuffer`/`BufferFromHostLiteral`)**: `HostBufferSemantics` enum(4종, L933-969)이 호스트 메모리의 공유/복사/변경 가능성 계약을 규정한다 — `kImmutableOnlyDuringCall`(호출 중에만 유효, 복사 강제), `kImmutableUntilTransferCompletes`(전송 완료까지 유효), `kImmutableZeroCopy`(버퍼 생존 동안 유효, 런타임은 변경 안 함), `kMutableZeroCopy`(런타임도 변경 가능, 예: 출력 버퍼로 aliasing). 이 값에 따라 구현이 zero-copy 뷰를 만들지, 즉시 복사할지 결정한다(구체 구현은 스코프 밖).
  • **비동기(`AsyncHostToDeviceTransferManager`)**: `TransferLiteralToBuffer`는 `client_->LinearizeInto(literal, device_shape, ..., raw_buffer)`(L263-268, host layout→device layout 변환 + 전송, 구현은 스코프 밖이지만 개념적으로 `staging_buffer`를 사용할 것으로 추정)를, `TransferRawDataToSubBuffer`는 `raw_buffer->CopyRawHostToDeviceAndReturnEvent(...)`(L376-378, 이미 device layout으로 정렬된 원시 바이트)를 각각 호출한다. 두 경우 모두 반환된 `h2d_transfer_event`에 `AndThen`을 걸어, 전송이 끝나면 `definition_event.Set(transfer_event)`(L306, L439)로 버퍼를 ready 상태로 전이시킨다.
  • **역방향(`ToLiteral`, device→host)**: `PjRtBuffer::ToLiteral`/`LazyToLiteral`(`pjrt_client.h:1224-1231`)은 순수 가상 함수로 구현은 스코프 밖이지만, `raw_buffer.cc`의 `ToStagingBuffer` 헬퍼(L137-196)가 그 하부 메커니즘을 보여준다: raw buffer가 host-visible 포인터를 가지면(`GetHostPointer() != nullptr`, 예: CPU/pinned host) **zero-copy로 바로 그 메모리를 감싸는** `HostRawBufferPjRtStagingBuffer`를 반환하고, 그렇지 않으면 `allocate_staging_buffer` 콜백으로 새 스테이징 버퍼를 만들고 `raw_buffer->CopyRawDeviceToHostAndReturnEvent(...)`로 device→host 복사를 건 뒤 완료 시 스테이징 버퍼를 concrete로 만든다.

삭제/외부 참조 처리

  • `IsDeleted()`는 `CommonPjRtBuffer::device_buffer_ == nullptr` 여부다(`abstract_tracked_device_buffer.cc:409-412`).
  • 실제 `Delete()`는 스코프 밖이지만, 최종적으로 도달하는 `AbstractTrackedDeviceBuffer::Delete(memory_space)`(L143-159)가 핵심이다: `definition_events_`와 `usage_events_`를 모두 모은 뒤 `LockUsageEvents()`로 더 이상 새 usage event를 못 붙이게 막고, `raw_buffer`를 꺼내 `this`(unique_ptr 소유)를 파괴한 다음, **`raw_buffer.release()->DecrefAfter(std::move(events))`** 를 호출한다. `DecrefAfter`(`raw_buffer.cc:112-114`)는 `xla::RunWhenReady(events, [this]{ DropRef(); })`로 구현되어 있어 — 즉 실제 메모리 반납(`DropRef`)은 **모든 정의/사용 이벤트가 끝난 뒤로 지연**된다. 이는 `pjrt_client.h`의 `Delete()` 주석("메모리는 지연 해제되며, Delete는 잠깐 블록할 수는 있지만 스트림 연산 완료까지 블록하지는 않는다")과 정확히 대응한다.
  • **외부 참조**는 `CommonPjRtBuffer::ScopedHold`의 `kExternalReference` 타입(`abstract_tracked_device_buffer.h:204`)으로 구현된다. 홀드 규칙(L156-201 주석 + `GetBufferForDonationHoldLocked` L208-229)에 따르면: 도네이션 홀드는 미해제 외부 참조가 있으면 에러를 반환하고, 외부 참조가 있는 동안은 `Delete`/도네이션이 완전히 끝나지 않는다 — `AcquireExternalReference()`(`pjrt_client.h:1216-1217`, 순수 가상)가 이 `kExternalReference` 홀드로 구현될 것으로 추정된다(구체 코드는 스코프 밖).

Future 개념과 GetReadyFuture 대응 코드

`pjrt_future.h`는 실질적으로 `#include "xla/future.h"` 한 줄뿐인 **포워딩 헤더**다. 즉 현재 이 저장소에서 `PjRtFuture<T>`라는 이름은 사라지고 범용 `xla::Future<T>`(정의는 `xla/future.h`에 있으나 이번 체크아웃에는 포함되지 않음 — 확인 필요)로 통합되었다. `PjRtBuffer::GetReadyFuture()`(`pjrt_client.h:1394`)는 순수 가상이며, 실제 구현은 `AbstractTrackedDeviceBuffer::GetReadyFuture(PjRtMemorySpace*)`(`abstract_tracked_device_buffer.cc:52-106`)다. 여기서 `tsl::MakePromise<void>()`로 **호출할 때마다 새 Future/Promise 쌍**을 만들고, `raw_buffer`의 할당 이벤트 + 아직 concrete가 아닌 `definition_events_`를 의존성으로 모아 `RunWhenReady`에 넘긴다. 이는 `pjrt_client.h`의 주석("버퍼가 삭제/도네이션돼도 이미 반환된 Future는 유효하게 남는다")을 그대로 구현한 것 — 반환되는 Future는 내부 이벤트 객체와 독립적인 새 객체이기 때문이다.

device_event의 역할

`device_event.h`의 `PjRtDeviceEventRef`(소유)/`PjRtDeviceEventPtr`(비소유)/`PjRtDeviceEventPromiseRef`(쓰기 측)는 `PJRT_DeviceEvent` C 구조체를 감싼 참조 카운트 핸들이다. 로컬(in-process) 이벤트는 `tsl::AsyncValue`에 바로 매핑되고(`internal::GetBuiltinAsyncValueCApiFunctionTable`, `device_event.h:31-96`), 외부 C API 플러그인 이벤트는 vtable 디스패치로 동일한 인터페이스를 제공한다. `device_event_utils.h`의 `ScopedLauncher`/`RunWhenReady`/`ExecuteWhenReady`는 "여러 이벤트가 모두 준비되면 작업 실행"이라는 **논블로킹 fan-in 조합자**이며, `ScopedLauncher::AddDependency`가 각 의존성에 참조 카운트된 `Callee`를 매달아(`DeleteWhenReady`/`AndThen`) 마지막 참조가 사라질 때(`~Callee`)만 태스크를 실행한다(`device_event_utils.cc:34-105`). 반면 `BlockUntilReady`(`device_event_utils.cc:132-144`)만이 `absl::Notification`으로 **실제로 스레드를 재우는** 유일한 지점이다 — `AbstractTrackedDeviceBuffer::BlockForOperationsToComplete`(L108-124)가 이를 사용한다.

**문서 시나리오("x=foo(1) 실행 후 값이 필요할 때까지 블로킹을 미룬다")와의 대응**: `x` 생성 시점에는 `definition_events_`에 아직 완료되지 않은 `PjRtDeviceEventRef`(플러그인이 나중에 `Set`할 promise)만 들어있다. `CommonPjRtBuffer::GetBufferWithHold(kUsage)`/`AcquireScopedRawBuffer`(`abstract_tracked_device_buffer.cc:414-444`)는 이 미완료 이벤트를 **블록 없이** 그대로 다음 연산의 의존성으로 넘길 수 있으므로, `x`를 쓰지 않는 다른 연산은 계속 큐잉될 수 있다. `x`의 값이 실제로 필요해지는 지점(예: `ToLiteralSync`가 내부적으로 호출하는 `Future::Await`, 또는 `BlockForOperationsToComplete`)에 이르러서야 `BlockUntilReady`가 호출되어 스레드가 멈춘다.

semaphore가 사용되는 지점

`semaphore.h`/`semaphore.cc`는 `absl::Mutex::LockWhen` 기반의 표준 카운팅 세마포어(`Acquire`/`TryAcquire`/`Release`)와 RAII `ScopedReservation`/`ScopedAcquire`를 제공하는 **독립적인 유틸리티**다. 그러나 이번 체크아웃 범위(`xla/pjrt/` 지정 파일들과 저장소 전체를 재검색) 안에서 `Semaphore`를 실제로 사용하는 코드는 **한 곳도 발견되지 않았다** — `raw_buffer`, `abstract_tracked_device_buffer`, `host_to_device_transfer_manager`, `device_event*` 어디에도 참조가 없다. 이 계획 단계 스냅숏 안에서 semaphore는 device_event/실행 큐잉과 아직 배선되지 않은 leaf 유틸리티로 보이며, 일반적인 PJRT 패턴상 CPU/GPU 등 플랫폼별 클라이언트가 동시 전송/실행 개수를 제한하는 데 쓸 것으로 **추정**되지만, 해당 소비 코드(플랫폼별 `*_client.cc`)는 이번 분석 범위 밖이라 **확인이 필요함**.

핵심 흐름

  • **입력**: `BufferFromHostBuffer`류 API 호출 또는 `CreateBuffersForAsyncHostToDevice` + `TransferLiteralToBuffer/TransferRawDataToSubBuffer` 조합.
  • **변환/검증**: `HostBufferSemantics`로 소유권 계약 확인 → 필요 시 `LinearizeInto`(host layout→device layout, 내부적으로 스테이징 버퍼 경유 추정) → `PjRtRawBuffer`에 실제 바이트 기록(`CopyRawHostToDeviceAndReturnEvent`).
  • **상태 변경**: 전송 완료 이벤트를 `PjRtDeviceEventPromiseRef.Set(...)`으로 `definition_events_`에 반영 → `AbstractTrackedDeviceBuffer`가 이를 근거로 `GetReadyFuture()` 호출 시마다 새 Future를 발급.
  • **외부 호출/결과**: 소비자는 `ToLiteral`(내부적으로 `ToStagingBuffer` 경유해 device→host 복사) 또는 `Execute`의 인자로 버퍼를 그대로 사용하며, 값이 실제로 필요한 지점(`Await`/`BlockUntilReady`)에서만 동기 대기가 발생한다. `Delete`는 이벤트 완료 이후로 미뤄진 지연 해제(`DecrefAfter`)로 마무리된다.

중요한 설계 판단

  • **버퍼는 MemorySpace만 소유하고 Device는 간접 참조한다.** `CommonPjRtBuffer`(`abstract_tracked_device_buffer.h:379`)의 유일한 위치 관련 필드는 `PjRtMemorySpace* const memory_space_`이며 `PjRtDevice*` 필드가 없다. `PjRtBuffer::device()`는 여전히 순수 가상으로 남아있지만 코드 주석에 `// TODO(b/277820585): remove device() after the migration is done.`(`pjrt_client.h:1189-1190`)이 붙어 있어, **이 저장소는 "device() 제거"라는 마이그레이션이 진행 중인 과도기 상태**임을 보여준다. 반대로 `host_memory_spaces.h`의 세 `PjRtMemorySpace` 구현은 모두 `PjRtDevice* device_`를 실제로 들고 있어, "메모리 공간이 디바이스를 알고, 버퍼는 메모리 공간을 거쳐 디바이스에 도달한다"는 문서의 역포인터 구조를 그대로 구현한다.
  • **raw_buffer.h는 "메모리"를, abstract_tracked_device_buffer.h는 "생명주기/동기화"를 담당하도록 명확히 분리되어 있다.** `PjRtRawBuffer`는 C ABI(`PJRT_RawBuffer`)를 직접 상속해 플러그인 경계를 넘나들 수 있는 반면, `AbstractTrackedDeviceBuffer`/`CommonPjRtBuffer`는 순수 C++이며 C API 미러가 없다. 이 분리 덕분에 raw buffer 하나를 여러 논리 버퍼가 공유(슬라이싱, 도네이션, staging)하면서도 이벤트/hold 규칙은 상위 계층에서만 관리된다.
  • **staging_buffer와 host_to_device_transfer_manager는 서로 다른 시점을 책임진다.** `staging_buffer`는 **단일 복사 연산** 단위에서 "host-visible 포인터가 있으면 zero-copy, 없으면 스테이징 후 복사"라는 저수준 linearize/delinearize 원시 연산(`raw_buffer.cc: ToStagingBuffer`)을 담당하는 반면, `host_to_device_transfer_manager`는 **버퍼 생명주기/멀티청크 스트리밍** 단위에서 "데이터가 없는 상태로 버퍼 핸들을 즉시 내주고, 이후 여러 번에 걸쳐 청크를 흘려 넣는" 상위 정책(지연 할당, `TransferRawDataToSubBuffer`의 `is_last_transfer` 분기)을 담당한다. 후자는 개념적으로 전자를 호출해 실제 바이트 변환을 수행할 것으로 추정된다.
  • **메모리 해제는 항상 이벤트 완료 뒤로 지연된다.** `AbstractTrackedDeviceBuffer::Delete` → `PjRtRawBuffer::DecrefAfter` → `RunWhenReady(events, DropRef)`로 이어지는 체인은, "Delete는 즉시 반환하지만 실제 free는 진행 중인 async 작업이 끝난 뒤"라는 설계를 관통한다. 이는 GPU/TPU에서 스트림 상의 아직 실행 중인 커널이 참조하는 메모리를 조기 해제하지 않기 위한 전형적 패턴이다.

읽으면서 주의할 점

  • 이 체크아웃에는 `common_pjrt_client.h/.cc`, `pjrt_client.cc`, `xla/future.h`가 **없다**. `AllocateRawBuffer`, `DefineBuffer`, `LinearizeInto`, `CreateLinkedEventPromise`, `TrackFuture`, `WaitOnStream`, `Future<>`/`tsl::MakePromise` 등은 모두 이 파일들에서 온 것이며, 이번 문서에서 그 동작은 호출 패턴으로 **역추정**한 것이지 소스로 확인한 것이 아니다.
  • `Semaphore`는 문서 요구사항("device_event/semaphore가 실행 큐잉과 어떻게 연결되는지")과 달리 이 스코프 안에서는 device_event와 **전혀 연결되어 있지 않다.** 실제 배선을 확인하려면 `xla/pjrt/cpu/` 또는 `xla/pjrt/gpu/` 등 플랫폼별 클라이언트 소스가 필요하다.
  • 참고 문서(OpenXLA 공식 문서)는 `xla::Future<> ToLiteral(...)` 같은 최신 시그니처를 이미 반영하고 있어 코드와 일치하지만, `device_event.h`/`raw_buffer.h`/`staging_buffer.h` 같은 **내부 구현 세부사항은 공식 문서 어디에도 설명되어 있지 않다** — 참고자료 4개 문서 전체를 검색해도 "definition event", "raw buffer", "staging"에 대응하는 설명은 없었다. 즉 이 레이어는 공개 API 문서보다 한 단계 더 내부적인, 최근 도입된 리팩터링 계층으로 보인다(추정).
  • `AbstractTrackedDeviceBuffer::Delete`는 `std::unique_ptr<AbstractTrackedDeviceBuffer> device_buffer(this)`로 **자기 자신을 unique_ptr로 감싸 소멸시키는** 다소 특이한 패턴(L144)을 쓴다 — `this` 포인터 라이프타임에 유의해서 읽어야 한다.

더 파볼 질문

  • `CommonPjRtClient::DefineBuffer`/`AllocateRawBuffer`/`LinearizeInto`/`CreateLinkedEventPromise`의 실제 구현(`common_pjrt_client.cc`, 이번 스코프 밖)을 봐야 host→device 변환 로직과 raw buffer 할당 정책(재시도, OOM 처리)이 완전히 드러난다.
  • `Semaphore`의 실제 소비처(`xla/pjrt/cpu/cpu_client.cc` 또는 `xla/pjrt/gpu/*` 추정)를 찾아, 문서가 요구한 "semaphore와 실행 큐잉의 연결"을 실제로 검증할 필요가 있다.
  • `xla/future.h`의 `Future<T>`/`tsl::MakePromise<T>` 구현을 읽어, `PjRtFuture`에서 `xla::Future`로의 통합이 어떤 API 변화(콜백 체이닝, `Await()` 시맨틱 등)를 수반했는지 확인할 가치가 있다.
  • `use_stream_based_compaction_` 플래그(`abstract_tracked_device_buffer.h:150`)가 `AddUsageEvent`에서 스트림/시퀀스 기반으로 usage event를 압축하는 로직(`abstract_tracked_device_buffer.cc:491-509`)은 GPU 스트림 최적화로 보이는데, 이 최적화가 켜지는 조건(어느 백엔드가 `true`로 생성하는지)은 확인이 필요하다.
