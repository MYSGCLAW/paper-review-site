CEMU NVMe CSD 커널 드라이버 분석

한 줄 요약

cemu.c는 표준 NVMe PCI 드라이버(pci.c, core.c) 위에 "컴퓨트 네임스페이스(c3)"와 "메모리 네임스페이스(m2)"를 위한 별도의 블록/문자 디바이스 쌍을 직접 만들어 붙이고, pci.c의 nvme_prep_rq()와 core.c의 nvme_setup_cmd()에 조건부 분기(#ifdef CONFIG_NVME_CEMU)를 심어 일반 I/O bio를 CSD 전용 NVMe 커맨드(Load Program / Program Execute / Memory Read·Write·Copy)로 둔갑시키는 방식으로 동작한다.

분석 범위

job.json이 지정한 linux-cemu 저장소의 NVMe 호스트 드라이버 경로(drivers/nvme/host/{cemu.c,cemu.h,pci.c,core.c,ioctl.c,nvme.h,trace.h,Kconfig,Makefile}, include/uapi/linux/cemu_ioctl.h, include/linux/{cemu_ioctl.h,nvme.h}, include/linux/blk_types.h)만 다룬다. references/raw/cemu/repo-map.md에 따르면 이 저장소의 CEMU 고유 코드는 전체의 극히 일부(약 2,600 LOC)이므로, 위 파일들에 cemu/CEMU/CONFIG_NVME_CEMU 마커가 붙은 부분만 실제 확장 지점이고 나머지는 Linux 6.8 upstream 그대로다. 디바이스(에뮬레이터) 측 hw/femu/csd/nvme_csd.h는 이 job의 repo/ 트리에 포함돼 있지 않아 직접 열어 대조하지 못했다(아래 "대조" 항목에서 별도 표시).

전체 구조

세 개의 독립된 저장소(CEMU/linux-cemu/ubpf-cemu) 중 이 문서는 linux-cemu 호스트 드라이버만 다룬다. references/raw/cemu/repo-map.md가 정리한 전체 데이터 흐름 골격을 인용하면:

호스트 애플리케이션 (ioctl / io_uring cmd, cemu_ioctl.h)
  ▼
drivers/nvme/host/cemu.c  — /dev/nvmeXm2(블록), /dev/nvmeXc3(문자)
  ▼
drivers/nvme/host/core.c  — nvme_setup_cmd() 가 REQ_OP → nvme_command 변환
  ▼
drivers/nvme/host/pci.c   — nvme_prep_rq() 가 p2pdma 페이지/bio 플래그를 감지해
                              Memory Read/Write/Copy 커맨드로 재작성, PRP/SGL 매핑
  ▼
QEMU/FEMU 경계 (에뮬레이션된 CSD 컨트롤러, 이 job 범위 밖)

핵심은 cemu.c가 NVMe 표준 네임스페이스 스캔 경로(core.c의 nvme_scan_ns_list/nvme_alloc_ns)를 거치지 않고, PCI probe 시점에 컨트롤러 단위 quirk(NVME_QUIRK_CEMU)를 보고 cemu_dev_add()를 한 번 호출해 m2/c3 노드를 수동으로 만든다는 점이다. 이는 코드를 읽어 직접 확인한 사실이며, 뒤에서 근거를 제시한다.

먼저 볼 코드 리딩 순서

  1. include/uapi/linux/cemu_ioctl.h — 사용자 공간과의 유일한 공식 계약(ioctl 번호, struct ioctl_download/execute/create_mrs)을 먼저 외워두면 이후 커널 코드의 필드 이름이 바로 이해된다.
  2. drivers/nvme/host/cemu.h — struct cemu_dev, struct cemu_bio와 전역 배열 cemu_bdev[]/cemu_p2p_start[]/cemu_p2p_end[]. 이 배열들이 뒤에 나오는 멀티-드라이브 P2P의 핵심 자료구조이므로 구조부터 파악한다.
  3. cemu.c: cemu_dev_add() — 디바이스가 "어떻게 발견되고 등록되는지"의 실체. nvme_find_get_ns(ctrl, 1)로 실제 NVM 네임스페이스를 얻고, BAR2를 p2pdma 리소스로 등록하고, m2/c3 노드를 만드는 전체 순서를 여기서 본다.
  4. cemu.c: cemu_cdev_ioctl()과 하위 핸들러 (cemu_load_program, cemu_unload_program, cemu_program_activation, cemu_program_execute, cemu_create_mrs) — UAPI 구조체가 실제로 어떻게 소비되는지, 그리고 각 ioctl이 bio 경로를 타는지 admin passthrough를 타는지 구분해서 본다.
  5. core.c: nvme_setup_cmd() 및 인접 nvme_setup_load()/nvme_setup_execute() — cemu.c가 만든 bio(REQ_OP_LOAD_PROGRAM, REQ_OP_PROGRAM_EXECUTE)가 실제 struct nvme_command으로 변환되는 지점. 여기가 core.c의 CEMU 훅 전체다.
  6. pci.c: nvme_prep_rq()의 #ifdef CONFIG_NVME_CEMU 블록 — 일반 Read/Write bio가 p2pdma 페이지(is_pci_p2pdma_page)나 BIO_NVME_MEMORY_CMD 플래그를 갖고 있으면 Memory Read/Write/Copy 커맨드로 재작성되는 로직. P2P DMA 활용의 실제 코드다.
  7. pci.c: nvme_pci_probe()의 quirk 분기와 pci_ids[]의 0x8888 항목 — 일반 NVMe SSD와 CSD를 구분하는 유일한 기준(PCI Device ID + NVME_QUIRK_CEMU)을 확인한다.
  8. include/linux/nvme.h의 nvme_load_program_cmd/nvme_program_activation_cmd/nvme_program_execute_cmd/nvme_memory_range_set_manage_cmd/nvme_memory_range — cemu.c/core.c가 채우는 필드의 원형. 디바이스 측 nvme_csd.h와의 대조 기준점이 된다(단, 이 job에서는 후자를 직접 열람하지 못함).
  9. drivers/nvme/host/nvme.h의 NVME_QUIRK_CEMU, struct nvme_ctrl의 cemu_dev/cemu_p2p_start/end 와 include/linux/blk_types.h의 REQ_OP_LOAD_PROGRAM/REQ_OP_PROGRAM_EXECUTE/BIO_NVME_MEMORY_CMD — 마지막으로, 위 모든 훅이 어떤 전역 열거형 확장에 의존하는지 확인한다. 이 확장이 왜 upstream에 부담스러운지 판단하는 근거가 된다.

  • App→Cdev 화살표들은 실제로 include/uapi/linux/cemu_ioctl.h에 정의된 IOCTL_CEMU_* 매크로와 struct ioctl_download/ioctl_execute/ioctl_create_mrs에 대응한다.
  • Cdev→Bio→Core→Dev는 DOWNLOAD만 bio 경로(cemu_load_program, nvme_setup_load)를 타고, ACTIVATE/DEACTIVATE/UNLOAD/MRS는 nvme_submit_sync_cmd/__nvme_submit_sync_cmd로 admin queue에 직접 동기 제출된다는 차이를 나타낸다.
  • EXECUTE 화살표는 의도적으로 "io_uring 경로만 실제 커맨드 전송"이라고 표기했다. cemu_cdev_ioctl()의 IOCTL_CEMU_EXECUTE 분기(cemu_program_execute())는 프로그램 존재/활성 여부만 확인하고 아무 NVMe 커맨드도 보내지 않은 채 0을 반환한다(cemu.c:416-435). 실제로 REQ_OP_PROGRAM_EXECUTE bio를 만들어 제출하는 코드는 io_uring uring_cmd용 program_execute()(cemu.c:581-615)뿐이며, 이마저 execute.pind를 읽지 않고 pind=1을 하드코딩한다.

  • cemu_bdev_submit_bio(cemu.c:100-120)가 m2 블록 디바이스를 "가짜 디스크"로 만들어, 실제 큐 처리는 원본 NVM 네임스페이스(dev->nvme_bdev)에 위임하면서 BIO_NVME_MEMORY_CMD 플래그(blk_types.h:349)만 얹는다는 점이 핵심이다.
  • is_memory_copy_cmnd/is_memory_rw_cmnd/is_cemu_p2p_addr/cemu_addr은 모두 pci.c:896-936에 정의돼 있다.
  • is_cemu_p2p_addr()는 cemu_bdev_minor 범위 전체(즉 모든 등록된 CSD)를 순회하며 물리주소가 속한 디바이스를 찾는다 — 이것이 멀티-드라이브 P2P의 실체다.

  • struct cemu_dev(cemu.h:38)는 컨트롤러 1개당 1개 생성되며, prog_list로 여러 sysfs_program(프로그램 메타데이터, 로드된 CSF마다 1개)을 소유한다.
  • struct cemu_bio(cemu.h:14)는 커널 내부 bio 완료 동기화용 임시 객체로, bio->bi_private를 통해 cemu_dev나 카운터에 연결된다.
  • cemu_bdev[]/cemu_p2p_start[]/cemu_p2p_end[](cemu.h:70-72, 전역 정의는 cemu.c:33-35)는 디바이스 소유가 아닌 모듈 전역 배열로, pci.c가 어떤 CSD든 상관없이 참조할 수 있게 해준다 — 이것이 "multi-CSD/multi-drive P2P가 자료구조에 미친 영향"의 직접적 증거다: 원래 있었다면 cemu_dev 안에 갇혀 있을 정보를 전역으로 뽑아내 인덱스(dev->idx)로 관리하게 바뀐 흔적.
  • struct nvme_ctrl(nvme.h, drivers/nvme/host/nvme.h:401-404)에 추가된 cemu_dev/cemu_p2p_start/cemu_p2p_end 필드가 core/pci와 cemu.c를 잇는 유일한 공식 연결 고리다.

핵심 흐름

입력 → 변환/검증 → 상태 변경 → 외부 호출

  1. 입력: 사용자 공간이 /dev/nvmeXc3를 열고 include/uapi/linux/cemu_ioctl.h의 IOCTL_CEMU_* 중 하나를 호출한다. 모든 인자는 __u64로 인코딩된 사용자 포인터를 담고 있어(copy_from_user 필요), 커널은 항상 (void __user *)(uintptr_t) 캐스팅 후 접근한다(cemu.c 전반).
  2. 변환/검증: cemu_cdev_ioctl()이 커맨드 번호로 분기하고, 각 핸들러가 dev->lock 뮤텍스 하에서 dev->prog_list를 선형 탐색(__find_program_by_id/__find_program_by_name)해 대상 프로그램의 존재/상태를 검증한다.
  3. 상태 변경: LOAD는 dev->cur_pind를 증가시키며 새 sysfs_program을 만들고 sysfs 노드(/sys/fs/nvmeXm2/<name>)를 추가한다. ACTIVATE/DEACTIVATE는 prog->is_active 플래그만 갱신한다. UNLOAD는 sysfs_program을 리스트에서 제거하고 kfree한다.
  4. 외부 호출: 최종적으로 nvme_submit_sync_cmd()/__nvme_submit_sync_cmd()(admin queue, ACTIVATE/DEACTIVATE/MRS)나 submit_bio()(io queue, LOAD/EXECUTE-uring)를 통해 실제 NVMe 커맨드가 컨트롤러로 나간다. 이 지점에서 core.c: nvme_setup_cmd()가 block-layer REQ_OP를 struct nvme_command로 최종 직렬화한다.

질문에 대한 답

  • CSD를 쓰기 위한 정확한 ioctl 순서: open("/dev/nvmeXc3") → IOCTL_CEMU_DOWNLOAD(pind 획득) → IOCTL_CEMU_ACTIVATE → (필요 시) IOCTL_CEMU_CREATE_MRS → IOCTL_CEMU_EXECUTE(ioctl 또는 io_uring IORING_OP_URING_CMD) → 종료 후 IOCTL_CEMU_DEACTIVATE → IOCTL_CEMU_UNLOAD. IOCTL_CEMU_DELETE_MRS는 MRS를 만들었다면 정리 단계에서 호출한다. application-development.md에 나온 사용 순서와 코드 구현이 정확히 일치한다.
  • P2P DMA가 없애는 데이터 이동: cemu_p2pmem_setup()(cemu.c:645)이 CSD의 BAR2(SLM, CEMU_SLM_BAR=2)를 pci_p2pdma_add_resource()로 등록해 그 메모리를 일반 페이지처럼 취급되게 만든다. 그 결과 nvme_prep_rq()가 그 페이지를 가리키는 bio를 만나면 Memory Read/Write/Copy 커맨드로 변환해(pci.c:838-1024) 호스트 DRAM을 거치지 않고 CSD 내부(NAND ↔ 디바이스 메모리) 또는 서로 다른 두 CSD의 BAR 사이에서 직접 데이터가 이동하도록 한다. 즉, "디바이스→호스트 RAM→디바이스"의 왕복 복사(bounce buffer)와 그에 따른 호스트 CPU/메모리 대역폭 소모를 제거하는 것이 목적이다.
  • CSD를 일반 NVMe SSD와 구분하는 방법: 네임스페이스 Identify가 아니라 PCI Device ID로 구분한다. pci.c 최하단 pci_ids[] 테이블의 PCI_VDEVICE(INTEL, 0x8888) 항목(#ifdef CONFIG_NVME_CEMU, pci.c:3564-3570)에만 NVME_QUIRK_CEMU 비트가 설정되고, nvme_pci_probe()가 dev->ctrl.quirks & NVME_QUIRK_CEMU일 때만(pci.c:3265) cemu_dev_add()를 호출한다. 컴퓨트/메모리 네임스페이스 자체는 NVMe 표준 Identify Namespace 스캔(core.c: nvme_scan_ns/nvme_alloc_ns) 경로를 전혀 타지 않는다 — 코드에서 확인한 사실이며, cemu_dev_add()는 nvme_find_get_ns(ctrl, 1)로 nsid=1(NVM)의 기존 블록 디바이스만 얻어 재사용하고, m2/c3 노드는 직접 blk_alloc_disk()/cdev_add()로 새로 만든다. FIXME: nsid라는 주석이 이 부분이 하드코딩된 임시 처리임을 코드 스스로 인정하고 있다.
  • 실행 중인 프로그램을 unload하면: cemu_unload_program()(cemu.c:359)은 prog->is_active 상태를 확인하지 않고 곧바로 Load 커맨드(sel=1)를 admin queue에 동기 전송한다. 디바이스가 성공을 반환하면 sysfs_program을 리스트에서 제거하고 kfree한다. 활성 상태 프로그램을 먼저 deactivate하도록 강제하는 로직이 커널 쪽에는 없다 — 안전장치가 사실상 디바이스 펌웨어에 위임돼 있다(추정: 디바이스가 자체적으로 거부/직렬화할 가능성이 있으나 이 저장소만으로는 확인 불가).
  • 여러 CSD가 붙었을 때 프로그램/메모리 격리: 프로그램은 cemu_dev 단위(컨트롤러 1개당 1개)로 완전히 분리된 prog_list+mutex+sysfs kobject를 가져 강하게 격리된다. 반면 디바이스 메모리 주소 공간은 전역 배열 cemu_bdev[]/cemu_p2p_start[]/cemu_p2p_end[](CEMU_MAX_MINOR=64 슬롯)로 관리되어, pci.c: is_cemu_p2p_addr()가 모든 등록 CSD를 순회해 물리주소가 속한 디바이스를 찾는다 — 이는 PCI BAR 물리주소 공간이 시스템 전역에서 자연히 겹치지 않는다는 전제에 의존한 "격리 없는 공유 조회 테이블" 방식이다.
  • upstream 채택 가능성: 아래 "중요한 설계 판단"과 "읽으면서 주의할 점" 참고 — 결론적으로 연구용 프로토타입에 가깝다.

중요한 설계 판단

  • 네임스페이스 인식을 NVMe 표준 스캔이 아니라 컨트롤러 quirk로 대체: m2/c3는 Identify Namespace를 거치지 않고 probe 시점에 통째로 생성된다. 구현은 단순하지만, 실제 NVMe CS(Computational Storage) 스펙이 정의하는 "네임스페이스별 Command Set" 모델과는 괴리가 있다(추정: 진짜 스펙 준수 구현이라면 NVME_CSI_* 값과 nvme_update_ns_info()의 CSI 분기를 통해 인식되어야 하지만, 이 코드는 그 경로를 쓰지 않는다).
  • 블록 계층의 전역 enum req_op를 확장: REQ_OP_LOAD_PROGRAM/REQ_OP_PROGRAM_EXECUTE를 NVMe 드라이버 전용이 아니라 include/linux/blk_types.h의 공유 열거형에 추가했다. 이는 모든 블록 드라이버가 공유하는 네임스페이스를 침범하는 것이라, 다른 서브시스템의 향후 opcode 확장과 충돌 위험이 있고 upstream 관례상 받아들여지기 어렵다.
  • m2 블록 디바이스를 "리다이렉터"로 설계: cemu_bdev_submit_bio()가 자체 request_queue 처리 로직 없이 bio를 원본 NVM 네임스페이스로 재위임하는 방식은, FDMFS/O_DIRECT/mmap 같은 표준 파일-블록 스택을 그대로 재사용할 수 있게 해주는 영리한 설계다. 다만 디스크 두 개(m2, 원본 n1)가 큐를 공유하면서 논리 블록 크기를 1바이트로 강제(disk->queue->limits.logical_block_size = 1)하는 부분은 매우 특수한 해킹에 가깝다.
  • 전역 배열 기반 P2P 주소 조회: 멀티-드라이브 P2P를 위해 per-device 캡슐화를 깨고 전역 배열(cemu_bdev[] 등)로 뽑아낸 것은 기능적으로는 합리적이지만, 잠금 규율이 일관적이지 않다(아래 참고).

읽으면서 주의할 점

  • cemu_program_execute()(ioctl 경로)는 사실상 스텁이다. 프로그램 존재/활성 여부만 확인하고 실제 NVMe 커맨드를 전혀 보내지 않은 채 0을 반환한다(cemu.c:416-435). 동기 ioctl로 실행을 시도하는 사용자 코드는 조용히 아무 일도 일어나지 않는 것처럼 보일 수 있다.
  • program_execute()(io_uring 경로)는 pind=1을 하드코딩하고 execute.pind/rsid/cparam1,2 등 SQE 페이로드를 사용하지 않는다(ioctl_execute를 가리키는 코드가 주석 처리돼 있음, cemu.c:583). 여러 프로그램을 로드한 뒤 특정 프로그램을 지정해 실행하는 시나리오는 현재 코드로는 동작하지 않는다(추정: 개발 중인 상태로 보임).
  • 활성 상태 검사와 실제 I/O 제출 사이 락이 풀린다. cemu_program_execute()/program_execute() 모두 dev->lock을 짧게 잡아 존재/활성 여부만 확인한 뒤 즉시 unlock하고, 그 이후에 실제 bio 제출이 일어난다. 그 사이에 다른 스레드가 IOCTL_CEMU_DEACTIVATE나 IOCTL_CEMU_UNLOAD를 실행하면 TOCTOU 경쟁이 발생할 수 있다.
  • is_cemu_p2p_addr()(pci.c)가 g_lock 없이 전역 배열을 순회하는 반면, 배열에 대한 쓰기(cemu_dev_add/cemu_dev_remove)는 g_lock으로 보호된다. NULL 체크로 크래시는 피하지만 핫 리무브 타이밍에 따라 최신성 보장이 없는 lock-free 읽기다.
  • cemu_dev_mmap()은 __maybe_unused로 표시돼 있고 cdev_fops/cemu_bdev_ops 어디에도 연결돼 있지 않다. 죽은 코드로 보이며, 이전 설계(직접 mmap)에서 현재 설계(FDMFS + bio 리다이렉트)로 바뀐 흔적으로 추정된다.
  • REQ_OP_LOAD_PROGRAM bio 처리 중 prog_name[len] = download.runtime_scale;(cemu.c:231)처럼, 프로그램 이름 뒤에 runtime_scale 값을 문자로 덮어써 사실상의 "네임스페이스 접미사"를 만드는 코드가 있다. 의도적인 트릭으로 보이나 매우 읽기 어렵고, runtime_scale이 int인데 문자로 캐스팅되므로 값 범위에 따라 예기치 않은 바이트가 파일명에 들어갈 수 있다.
  • 디바이스 측 nvme_csd.h와의 커맨드 정의 대조는 이 job 범위에서 완전히 확인하지 못했다. include/linux/nvme.h의 opcode(0x21=MRS manage, 0x22=Load/Unload, 0x23=Activation, Execute는 0x01+nsid=3, Memory Copy는 0x01+nsid=2)는 CEMU 참고 문서(code-structure.md)가 설명하는 admin/IO 커맨드 목록과 이름 수준에서는 일치하지만, 필드 바이트 오프셋까지 일치하는지는 디바이스 헤더 원문 확인이 필요하다.

더 파볼 질문

  • 디바이스 측 hw/femu/csd/nvme_csd.h(CEMU 저장소, 이 job 범위 밖)를 직접 열어 nvme_load_program_cmd 등 5개 구조체의 바이트 오프셋이 호스트 측과 정확히 일치하는지 바이너리 레이아웃까지 대조가 필요하다.
  • cemu_program_execute()가 스텁으로 남아있는 이유가 설계 미완성인지, 아니면 io_uring 경로만 지원하기로 의도적으로 결정한 것인지 커밋 로그나 논문 본문에서 확인할 가치가 있다.
  • 프로그램 unload 시 활성 상태 검사를 생략한 것이 디바이스 펌웨어의 방어 로직을 신뢰한 설계인지, 아니면 단순 누락인지 CEMU 에뮬레이터의 Unload 처리 코드(hw/femu/nvme-admin.c, 이 job 범위 밖)를 봐야 판단할 수 있다.
