선행 research 요약 재료: Document locality workload

CSD Resident Embedding을 위한 Document Locality 측정 설계 — 워크로드 선정, Trace 생성, 지표 정의

한 줄 요약

PLAID engine + ColBERTv2 index + MS MARCO passage corpus 조합을 워크로드로 선정하고, query별로 residual decompression 단계에서 실제 읽히는 passage embedding block 집합을 trace로 남겨, access frequency / reuse distance / working-set / hot coverage를 계산하면 CSD residency hit ratio와 PCIe traffic 절감량을 곧바로 유도할 수 있다.

수집 범위

  • 포함: ColBERT/ColBERTv2/PLAID/MS MARCO/BEIR 중 워크로드 선정, query stream → access trace 생성 절차와 record schema, locality metric 수식, CSD residency benefit 추정 절차, scheduling 인계 factor 정의.
  • 제외: end-to-end 성능 모델, CSD 하드웨어 설계 자체, retrieval 품질(MRR/nDCG) 분석.
  • 자료 상태(중요): references/manifest.json의 5개 문서는 모두 "URL only; content was not fetched automatically" 상태이고, 이번 세션에서 WebFetch/WebSearch 권한이 거부되어 원문을 직접 열람하지 못했다. 따라서 5개 문서는 전부 "URL 메타데이터만 확인"이며, 본문의 기술 내용은 해당 논문·코드베이스에 대한 배경지식으로 작성하고 수치는 추정/확인 필요로 구분했다. 첨부 repo도 없음(repo-files.txt: "No repository attached").

핵심 요약

  1. 최종 추천 워크로드: PLAID(실행 엔진) + ColBERTv2(index 포맷) + MS MARCO v1 passage(약 8.8M passages, 배경지식). BEIR는 query 수가 적어 보조 검증용, ColBERT v1은 index가 비압축이라 CSD 시나리오와 거리가 있다.
  2. "접근"의 정의가 결과를 좌우한다: PLAID는 후보를 centroid 점수만으로 걸러내므로, storage에서 실제로 읽는 것은 최종 단계에서 residual을 decompress하는 passage 집합뿐이다. 이 집합(이하 H2)을 byte 단위 접근으로 정의하고, 1단계 candidate 집합(H1)은 scheduling feature용으로 별도 기록한다.
  3. 공개 query set은 dedup되어 있다(배경지식): 동일 query 반복이 없으므로 문서 재사용은 서로 다른 query의 candidate 중복에서만 나온다. 따라서 S_uniq(고유 query 스트림, locality 하한) 과 S_zipf(Zipf 재표집 스트림, 운영 환경 근사) 두 스트림을 모두 측정해야 주장이 방어된다 — 본 설계의 최대 쟁점.
  4. 지표는 6종으로 충분하다: access frequency, reuse ratio, temporal reuse distance(LRU stack distance), working-set size(Denning WSS + Heaps 지수), hot coverage(Lorenz/CDF), Zipf α. 문서 크기가 token 수에 비례하므로 전 지표를 byte-weighted 버전과 병행 계산한다.
  5. PCIe 절감 주장에 필요한 최소 metric은 두 개다: baseline bytes/query 분포와 byte-weighted residency hit ratio HR_bytes(C). CSD가 hit을 내부에서 처리하면 절감율 = HR_bytes(C)가 그대로 성립한다.
  6. 규모 감(추정): ColBERTv2 2-bit 압축 시 token embedding당 ~36B, passage당 평균 ~2.5KB. PLAID k=100 설정에서 query당 수백 개 passage를 decompress한다고 알려져 있어 압축 상태로 ~0.6MB/query, fp16 원본 전송이면 ~4.5MB/query 수준 — 1k QPS면 PCIe Gen4 x4 대역폭(~8GB/s, 배경지식)의 상당 부분을 embedding 이동이 차지한다.
  7. static hot-set 평가는 trace의 train/test 분할이 필수다. 전체 trace로 hot set을 뽑아 같은 trace로 hit ratio를 재면 oracle leakage다.
  8. scheduling에 넘길 핵심 online feature는 candidate set locality ρ_q(후보 중 CSD resident 비율) 이다. ρ_q는 stage-1 직후(어떤 residual도 읽기 전에) 계산 가능하므로 GPU/CSD dispatch의 자연스러운 결정점이 된다.
  9. PLAID의 단계별 파라미터(nprobe, ndocs, 최종 decompress 수)와 index 수치는 원문 미열람 상태의 배경지식이므로, 본문 말미의 확인 필요 리스트를 논문·코드로 검증한 뒤 수치를 확정해야 한다.

2. Trace 생성 방법

passage(~2.5KB 추정) < 4KB이므로 한 block에 여러 pid가 섞인다 → pid 단위와 block 단위 지표가 달라진다(block 단위는 spatial locality·배치 순서 효과 포함). query 내에서 block id는 dedupe한다. Trace schema:

  • S_uniq: dev-full(또는 train에서 표본) query를 무작위 순서로 1회씩 재생. 반복이 없으므로 corpus 내재 locality(후보 중복)만 측정하는 하한.
  • S_zipf: query pool에서 Zipf(α_q) 인기 분포로 N개(예: 1M)를 복원추출해 반복이 있는 스트림을 합성. α_q는 공개 검색 로그의 query 빈도 왜도에서 가져오되(배경지식상 0.7~1.0 부근, 확인 필요) 민감도 sweep(α_q ∈ {0.6, 0.8, 1.0})으로 방어한다.
  • 각 query에 q_idx(0..N-1, 논리 시간축)와 합성 t_arrival(Poisson 도착 등)을 부여한다. locality 계산은 q_idx 기준, latency 모델만 t_arrival 사용.
  • H1: stage-1 candidate generation 직후의 candidate pid 집합(+ probed centroid ids).
  • H2: residual decompression에 들어가는 pid 집합 — 실제 storage read. 이것이 locality의 1차 접근 정의.
  • H3: 최종 top-k pid.
  • hook 위치는 ColBERT repo의 searcher/index-scorer 경로(배경지식상 Searcher.search → candidate generation → filter → residual lookup 순, 정확한 함수명 확인 필요).

CSD MaxSim Offload의 GPU 대비 MAC/PE/FLOPS Break-even 분석

한 줄 요약

Candidate embedding이 GPU-resident가 아닌 한 GPU MaxSim은 PCIe-bound이므로, CSD는 GPU의 FLOPS가 아니라 PCIe 전달률만 이기면 되고, 그 조건은 N_mac ≥ Q · BW_pcie / (b_e · f_csd · u) — fp16, Q=32, PCIe Gen4 x16 기준 약 630개 MAC(≈1 TFLOP/s, A100 peak의 0.3%) 수준이다. 단, 이는 BW_int ≥ BW_pcie(내부 read bandwidth 조건)와 "압축 전송 baseline과의 비교"라는 두 개의 더 어려운 전제를 동반한다.

수집 범위

  • 제외: 문서 로컬리티/캐시 히트율 자체 분석(job scope에서 명시적으로 제외), query encoder(BERT forward)의 비용 — 양 경로에 공통이므로 소거, candidate generation(ANN/centroid 단계)의 자체 비용 — 단 배치(placement) 쟁점으로만 언급.

핵심 요약

  1. MaxSim은 batched small GEMM + row-wise max: FLOP_total ≈ 2·B·N·Q·D·E, max-reduction은 GEMM 대비 1/(2E) ≈ 0.4% (E=128)로 무시 가능하다.
  2. GPU 경로의 arithmetic intensity는 AI = 2Q/b_e (query는 on-chip 상주, fused kernel 가정). fp16·Q=32이면 32 FLOP/B로 A100 ridge(≈153 FLOP/B)에 한참 못 미쳐 HBM-bound이고, 데이터가 PCIe를 건너는 순간 BW_pcie ≪ BW_hbm(30~80배)이므로 항상 PCIe-bound가 된다.
  3. 따라서 break-even의 본질은 "CSD compute vs GPU compute"가 아니라 "CSD compute vs PCIe 전달률": required OPS = (Q/b_e)·BW_pcie [MAC/s]. 여기서 B, N, D, E가 전부 소거된다 — break-even MAC 수는 워크로드 크기와 무관하고 Q, precision, 링크 속도, clock, utilization에만 의존한다.
  4. 필요 MAC 수는 작다: N_mac* = Q·BW_pcie/(b_e·f·u) → fp16, f=1 GHz, u=0.8일 때 Gen4 x4에서 158개, Gen4 x16에서 630개, Gen5 x16에서 1,260개. SmartSSD급 FPGA(KU15P, DSP slice 약 2천 개 — 배경지식, 확인 필요)로도 도달 가능한 규모다.
  5. 진짜 병목은 두 번째 필요조건 BW_int ≥ BW_pcie: 단일 drive의 NAND 내부 대역폭(~7–16 GB/s)은 GPU x16 링크 유효 대역폭(25–54 GB/s)보다 낮다. 단일 CSD는 warm host DRAM baseline에 지고, cold data / multi-drive scale-out / HBM·host DRAM capacity 초과 corpus에서만 이긴다.
  6. Result traffic은 소거 가능: score만 반환 시 D·E·b_e/b_s = 8192배, top-k 반환 시 10⁵배까지 egress가 줄어 T_res는 µs 단위가 된다. CSD의 PCIe 비용은 사실상 0으로 모델링해도 된다.
  7. 최강 GPU baseline은 raw fp16 전송이 아니라 압축 전송(ColBERTv2 residual compression, ~36 B/vector — 배경지식, 확인 필요): 전송량이 ~7배 줄면 required N_mac*도 ~7배(Gen4 x16 기준 ~4.5K MAC) 커진다. CSD 제안의 공정한 비교 대상은 PLAID류 압축 경로다.
  8. GPU-resident(HBM 상주) baseline을 이기는 것은 불가능에 가깝다(required OPS ≈ 65 TFLOP/s + BW_int ≥ 2 TB/s). CSD 논거는 latency가 아니라 capacity(fp16 full index ~150+ GB > HBM 80 GB), cost, host 자원 절감으로 옮겨가야 한다.

1. MaxSim Operation Model

MAC / FLOP: MAC_total = B · N · Q · D · E (dot-product 곱셈-누산) FLOP_gemm = 2 · B · N · Q · D · E FLOP_red = B · N · (Q·(D−1) compare + (Q−1) add) ≈ B·N·Q·D FLOP_total ≈ 2·B·N·Q·D·E · (1 + 1/(2E)) ≈ 2·B·N·Q·D·E [E=128 → red 비중 0.4%] Intermediate matrix: candidate당 Q×D score matrix. Materialize 시 B·N·Q·D·b_s bytes (예: N=4096, Q=32, D=128 → 67 MB, 입력의 절반 규모) — fused kernel이면 register/shared memory에서 소거된다. Max-reduction은 FLOP으로는 무시 가능하지만, unfused 구현에서는 HBM traffic을 ~2배로 만들 수 있는 memory-side 항이다. AI = FLOP/Bytes_D = 2Q/b_e [FLOP/B] ← N, D, E에 무관, Q와 precision만의 함수

2. GPU Latency Model

경로: NAND → (SSD 링크) → host DRAM → (GPU 링크) → HBM → SM → score 반환. T_pcie = Bytes_in / (ε_pcie · BW_pcie), BW_pcie = min(n_ssd·BW_ssd_link, BW_gpu_link, BW_host) T_comp = max( FLOP_total/(F_gpu·η_gpu), Bytes_HBM/BW_hbm ) [roofline] T_gpu = T_fix,gpu + T_pcie + T_comp (non-overlapped) T_gpu ≈ T_fix,gpu + max(T_pcie, T_comp) (chunked pipeline)

  • ε_pcie ≈ 0.8–0.9 (protocol overhead, 대형 DMA 기준 — 배경지식), T_fix,gpu ≈ 10–30 µs (kernel launch + driver — CUDA Guide 계열 배경지식).
  • HBM constraint 두 가지: (bandwidth) Bytes_HBM ≥ Bytes_D — fused면 1회 read, unfused면 score matrix 왕복이 추가돼 ~2배. (capacity) resident baseline이 성립하려면 index size ≤ C_hbm — fp16 MS MARCO full index ~150+ GB(배경지식, 확인 필요)는 80 GB HBM에 안 들어가고, ColBERTv2 압축 index ~20–36 GB는 들어간다.
  • regime 판별: compute-bound 조건은 AI > F_gpu·η/BW_hbm (A100 fp16 ridge ≈ 153 FLOP/B — 배경지식). Q=32·fp16(AI=32)는 미달 → resident면 HBM-bound. 비-resident면 BW_pcie(≤63 GB/s) ≪ BW_hbm(2–3.35 TB/s)이므로 무조건 PCIe-bound: T_gpu ≈ Bytes_D/(ε·BW_pcie). 이때 GPU의 F_gpu는 1차 근사에서 결과에 등장하지 않는다.

3. CSD Latency Model

경로: NAND → internal channel → on-drive PE array → local max/top-k → 결과만 PCIe. Bytes_int = B·N·D·E·b_e (internal read bytes, GPU 입력과 동일) OPS_csd = N_mac · f_csd · u [MAC/s] (= 2·N_mac·f_csd·u FLOP/s) T_mac = MAC_total / OPS_csd = B·N·Q·D·E / (N_mac·f_csd·u) T_csd = T_fix,csd + max(T_read, T_mac) + Bytes_res/BW_link

  • N_mac: MAC unit(PE) 수, f_csd: clock, u: utilization(스트리밍 정렬/뱅크 충돌/tail 포함 유효 계수). local max-reduction은 MAC 뒤 comparator를 파이프라인에 붙이면 추가 cycle이 없고 u에 흡수된다. top-k heap은 O(N log k) scalar 연산으로 무시 가능.
  • 필요한 on-chip 상태는 query matrix Q·E·b_e = 8 KB + doc당 Q개 partial-max register뿐 — score matrix를 만들지 않는다.
  • Result traffic 축소율: 전체 score 반환 시 R = D·E·b_e/b_s = 8192배 (D=E=128, fp16), top-k(k=100, N=4096) 반환 시 134 MB → 800 B ≈ 1.7×10⁵배. T_res는 Gen4 x4에서도 µs 단위 → 모델에서 소거 가능. 이것이 job 질문 "score/top-k 반환 시 traffic 모델 변화"의 답: CSD egress 항이 상수(≈0)로 바뀌고, break-even은 ingress(BW_int)와 compute(OPS_csd)만의 문제가 된다.
  • T_fix,csd: NVMe/CS command 왕복 + FW dispatch, 수십~수백 µs 추정(확인 필요 — SNIA CS API 문서와 실측 필요).

선행 research 요약과 연결고리 (첫 턴 필수)

A. Document locality 요약 (6줄)

PLAID(엔진) + ColBERTv2(압축 index) + MS MARCO v1(8.8M passages, query pool)을 워크로드로 고정한다. "storage 접근"은 centroid pruning 이후 residual을 실제로 decompress하는 pid 집합 H2로 정의하고, candidate 집합 H1은 scheduling feature로 따로 기록한다. 공개 query는 dedup되어 있으므로 S_uniq(locality 하한)와 S_zipf(운영 근사) 두 스트림을 모두 재생한다. 지표 6종(access freq, reuse ratio, LRU stack distance, WSS, hot coverage, Zipf α)을 byte-weighted 병행으로 계산하고, 최종적으로 PCIe 절감 주장은 HR_bytes(B) 하나로 환원된다. 규모 감(추정): 압축 ~36B/token, passage ~2.5KB, query당 수백 passage → 압축 ~0.6MB/query.

B. GPU/CSD break-even 요약 (7줄)

MaxSim은 FLOP≈2·B·N·Q·D·E의 batched small GEMM + row-wise max이고, arithmetic intensity AI=2Q/b_e(fp16·Q=32 → 32 FLOP/B)라 GPU는 resident면 HBM-bound, 비-resident면 항상 PCIe-bound다. 따라서 break-even의 본질은 "CSD vs GPU FLOPS"가 아니라 "CSD vs PCIe 전달률"이고, 조건은 N_mac* = Q·BW_pcie/(b_e·f·u)로 B,N,D,E가 전부 소거된다(fp16 기준 Gen4 x16 ≈ 630 MAC). 더 어려운 두 전제는 BW_int ≥ ε·BW_pcie(내부 대역폭)와 "raw fp16이 아닌 압축 전송 baseline과의 공정 비교"다. Result traffic(score/top-k)은 10⁴~10⁵배 축소로 사실상 0. GPU-HBM-resident baseline은 이기기 어렵고, CSD 논거는 latency보다 capacity/cost/host 자원으로 옮겨야 한다.

연결고리 (핵심)

두 연구는 같은 객체(H2 = residual block)를 서로 다른 층에서 본다. A는 그것을 어느 캐시 층에 두느냐(residency HR_bytes), B는 그것을 어느 경로로 옮기고 어디서 계산하느냐(PCIe vs 내부)로 본다. 접점은 CSD의 유효 내부 read 대역폭이 residency에 의해 결정된다는 것이다:

BW_int_eff(HR) = 1 / [ HR/BW_DRAM_int + (1−HR)/BW_NAND_int ]

즉 A의 산출물 HR_bytes(B)가 B의 필요조건 BW_int ≥ ε·BW_pcie의 좌변을 직접 만든다. 이것이 이번 턴이 두 연구를 하나로 묶는 지점이다.

새로 추가한 사실

사실 1 — locality와 break-even을 잇는 임계 residency HR*

CSD가 warm host-DRAM baseline을 read-BW에서 이기려면 BW_int_eff(HR) ≥ ε·BW_pcie이어야 하므로, 위 식을 풀면 locality 지표에 대한 명시적 임계값이 나온다:

HR* = ( 1/BW_NAND_int − 1/(ε·BW_pcie) ) / ( 1/BW_NAND_int − 1/BW_DRAM_int )

가정값(모두 배경지식/실측 필요) BW_DRAM_int=40, BW_NAND_int=12 GB/s에서:

  • ε·BW_pcie = 21 GB/s → HR* ≈ 0.61
  • ε·BW_pcie = 25 GB/s(research B의 GPU-A 예시) → HR* ≈ 0.74

해석: A가 측정한 HR_bytes(B) 곡선이 B* GB 지점에서 0.61~0.74를 넘으면 그 순간 단일 CSD가 warm 경로를 이긴다. 특수 케이스로 BW_NAND_int ≥ ε·BW_pcie(멀티채널 CSD)면 HR* ≤ 0 → residency 없이도 항상 이긴다(research B point 5의 locality 버전). 이 임계식이 산출물 2의 곡선과 산출물 3의 부등식을 하나로 묶는다.

사실 2 — CSD compute 요구치 재보정: 630 → ~1,700 MAC

research B의 Bytes_int = B·N·D·E·b_e(fp16)는 NAND가 fp16을 저장한다는 암묵 가정이지만, ColBERTv2 NAND에는 압축 residual(~36B/token)이 있다. 따라서 (i) 내부 read 바이트는 fp16의 ~1/7로 더 작고(research B의 T_read는 ~7× 비관), (ii) MAC은 여전히 decompress된 E=128 차원 위에서 돌므로 byte당 MAC 밀도가 7× 높다. CSD가 내부 read rate를 따라잡으려면:

N_mac(내부) = Q·E·BW_int_eff / (b_tok_comp · f · u)
            = 32·128·12e9 / (36 · 1e9 · 0.8) ≈ 1,707 MAC

즉 압축 데이터를 내부 대역폭으로 decompress+MAC하려면 630이 아니라 ~1,700 MAC이 필요하다(48×48 systolic array ≈ 2,304 MAC로 충족, 여전히 소형). research B point 7의 "~7×"를 정확한 값으로 확정한 것이며, 630은 fp16 PCIe baseline 전용 수치임을 명시해야 한다.

중요 정정: host page cache(L1)는 SSD→host read만 절감하고 host→GPU PCIe hop은 여전히 지불한다. CSD는 그 PCIe hop 전체를 없앤다(in-device MaxSim). 따라서 fully warm host page cache를 상대로도 CSD의 PCIe 이득은 유지된다 — CSD의 진짜 경쟁자는 L1이 아니라 L0(HBM-resident index)이다.

사실 4 — MS MARCO v1은 capacity 논거에 약한 워크로드

ColBERTv2 압축 index(MS MARCO v1) ~20GB는 A100 80GB HBM에 그대로 들어간다(L0 feasible). 즉 v1에서는 최강 baseline이 HBM-resident이고 CSD는 latency로 못 이긴다(research B의 결론과 정합). capacity 논거를 세우려면 HBM을 초과하는 corpus가 필요하다:

MS MARCO v2 ≈ 138M passages ≈ 15.7× v1 → 압축 index ≈ 314 GB ≫ 80 GB HBM

→ capacity-motivating 워크로드로 MS MARCO v2(또는 web-scale)를 A의 워크로드 축에 추가할 것을 권고한다(민감도 실험이 아니라 논거의 전제).

  • 왜 4단 캐시가 runtime 구조의 본질인가: MaxSim offload는 "compute를 어디서 하느냐"가 아니라 "데이터가 이미 어느 층에 있느냐"가 승부를 가른다. GPU 경로는 L0/L1로, CSD 경로는 L2로 이득을 보므로, 스케줄러는 두 캐시 상태를 동시에 봐야 한다. 이것이 단일 hit-ratio 최적화(전통 캐시 문헌)와 다른 지점이다.

  • Result egress가 0인 이유(재확인 맥락): CSD는 score/top-k만 반환(≤16KB) → PCIe egress가 상수 → break-even이 ingress(BW_int)와 compute(N_mac) 2변수 문제로 축소. 이 단순화가 산출물 3 부등식이 깔끔한 근거다.

Turn 2 / Codex

이번 턴의 확장 요약

이번 턴은 runtime 내부 구조와 scheduling factor를 중심으로 확장한다. 핵심 추가점은 (1) MaxSim offload runtime을 query encoding → candidate planning → routing → execution → merge의 operator graph로 분해하고, (2) whole-query offload보다 현실적인 shard-level hybrid scheduling을 제안하며, (3) ρ_q를 단일 hit ratio가 아니라 resident byte, miss byte, shard imbalance, queue delay가 결합된 dispatch 변수로 재정의한다. 또한 이전 턴의 ρ_q 계산 시점 표현은 약간 수정이 필요하다.

새로 추가한 사실

1. Whole-query routing보다 shard-level routing이 더 자연스럽다

기존 rule은 query 전체를 GPU 또는 CSD 중 하나로 보내는 형태였다. 하지만 PLAID/ColBERTv2 index는 pid/block 단위로 shard가 가능하므로 runtime은 query q의 H2 집합을 다음처럼 쪼갤 수 있다.

H2(q) = H2_gpu(q) ∪ H2_csd_1(q) ∪ ... ∪ H2_csd_m(q)

각 CSD는 자기 shard의 passage score 또는 local top-k만 반환하고, host가 최종 top-k를 merge한다. 이 방식의 장점은 ρ_q가 낮은 query에서도 resident/hot shard만 CSD로 보내고, miss-heavy shard는 GPU로 보내는 hybrid execution이 가능하다는 점이다.

단, document 하나의 token embedding은 한 실행 주체에 온전히 있어야 한다. MaxSim은 query token별 max를 낸 뒤 sum하기 때문에, 문서 token을 여러 장치에 쪼개면 partial max vector를 merge해야 한다.

score(q,d) = Σ_i max_j <q_i, d_j>

pid 단위 shard는 merge가 단순하다. token-level shard는 각 query token i에 대한 partial max를 모아야 하므로 runtime 복잡도가 커진다.

2. ρ_q는 hit ratio가 아니라 service-time predictor여야 한다

단순히 “H2 중 resident 비율”만 보면 scheduling에 부족하다. 같은 ρ_q=0.8이어도 miss byte가 큰 문서에 몰려 있으면 CSD latency가 커진다. 따라서 dispatch feature는 최소한 byte-weighted로 정의해야 한다.

ρ_bytes(q) = Σ_{d∈H2(q)∩L2} bytes_c(d) / Σ_{d∈H2(q)} bytes_c(d)

miss_bytes(q) = Σ_{d∈H2(q)\L2} bytes_c(d)

imbalance(q) = max_s bytes_s(q) / mean_s bytes_s(q)

CSD service time은 평균 hit ratio보다 다음 형태가 더 직접적이다.

T_csd(q) =
  T_fix,csd
  + max_s [
      bytes_hit_s(q)  / BW_DRAM_s
    + bytes_miss_s(q) / BW_NAND_s
    + MAC_s(q)        / OPS_s
    ]
  + T_merge(q)

즉 runtime scheduler는 ρ_q 하나가 아니라 miss_bytes, shard별 skew, queue depth를 같이 봐야 한다.

3. CSD offload의 runtime 병목은 compute보다 “tail shard”일 수 있다

multi-drive CSD에서 query latency는 평균 drive가 아니라 가장 늦은 drive가 결정한다.

T_query_csd = max_s T_csd_s(q) + T_host_merge

따라서 hot documents가 특정 shard에 몰리면 전체 HR_bytes가 높아도 p99 latency가 나빠진다. residency policy도 global top-frequency만으로 정하면 안 되고, shard별 capacity와 load를 제약으로 둬야 한다.

maximize   Σ_d x_d · f_test(d) · bytes(d)
subject to Σ_{d∈shard s} x_d · bytes(d) ≤ B_s    for each shard s
           expected_load_s ≤ load_cap_s
           x_d ∈ {0,1}

실용적으로는 shard별 STATIC-HOT + periodic rebalancing부터 시작하면 된다.

더 구체화한 부분

Runtime operator graph

Q0  Query encoder on GPU/CPU
Q1  Centroid scoring / IVF probe
Q2  H1 candidate list 생성
Q3  H2 planning: pid → block_id, bytes, shard, residency state
Q4  Scheduler decision

중요한 점은 Q4가 residual read 이전에 있어야 한다는 것이다. 그래야 scheduler가 “읽은 뒤 결정”하는 모순을 피한다.

Corrected scheduling decision point

이전 턴의 “stage-1 직후 ρ_q 계산 가능”은 표현을 더 정확히 해야 한다. H1만 안 상태에서는 최종 residual read 대상인 H2가 아직 확정되지 않을 수 있다. 안전한 정의는 다음이다.

Q4 decision point = H2 pid list가 확정된 직후, residual block read를 시작하기 전

만약 runtime이 더 이른 결정을 원하면 H1 기반 예측치를 따로 둔다.

ρ_hat_q = predicted resident byte ratio over H2, estimated from H1 features
ρ_q     = exact resident byte ratio after H2 planning

초기 구현은 ρ_q 기반 oracle-free scheduler로 충분하고, 이후 ρ_hat_q로 앞당기는 실험을 추가한다.

Scheduling rule v2

def route_query(q):
    plan = build_h2_plan(q)  # pid, bytes, shard, l2_state

    t_gpu = T_fix_gpu \
        + bytes_comp(q) / (eps * BW_pcie) \
        + decomp_mac_gpu(q) \
        + cold_read_penalty(q)

    t_csd = T_fix_csd + max(
        shard.hit_bytes / BW_DRAM_s
        + shard.miss_bytes / BW_NAND_s
        + shard.mac / OPS_s
        + queue_delay_s
        for shard in plan.csd_shards
    ) + T_merge(q)

    if t_csd + risk_margin < t_gpu:
        return "CSD"
    if resident_subset_beneficial(plan):
        return "HYBRID"
    return "GPU"

risk_margin은 p99 보호용이다. CSD 평균 latency가 약간 낮아도 tail shard가 불안정하면 GPU로 보내는 것이 낫다.

관련 맥락 / 배경 설명

CSD MaxSim runtime은 전통적인 accelerator offload와 다르게, 연산량보다 데이터 위치와 이동 경로가 먼저 결정한다. GPU는 HBM 또는 host DRAM page cache가 따뜻할수록 유리하고, CSD는 L2 resident set과 NAND locality가 좋을수록 유리하다.

따라서 scheduler는 “GPU가 빠른가, CSD가 빠른가”를 정적으로 고르는 장치가 아니라, query마다 다음을 추정하는 작은 cost model이어야 한다.

  1. ρ_q 계산 시점은 H1 직후가 아니라 H2 planning 직후로 두는 것이 안전하다. H1 기반 조기 dispatch는 예측 문제로 따로 분리해야 한다.
  2. CSD가 local top-k만 반환할 때 exactness가 깨질 수 있다. 각 shard가 k개만 반환하면 global top-k는 보통 맞지만, tie나 score 분포가 평평한 경우 안전하지 않을 수 있다. k_local = α·k sweep이 필요하다.
  3. shard imbalance가 HR_bytes(B) 곡선에 가려질 수 있다. 평균 hit ratio가 높아도 특정 CSD가 hot shard를 독점하면 p99가 나빠진다.
  4. GPU path의 cold_read_penalty는 host page cache, GPUDirect Storage, mmap 방식에 따라 달라진다. trace에는 logical access뿐 아니라 실제 page-cache hit 여부를 별도 계측해야 한다.
  5. CSD fixed cost T_fix,csd가 single-query latency에서 매우 중요하다. batch 없이도 이기는지, batch가 필요한지는 실측으로 분리해야 한다.

다음 에이전트가 이어서 확장할 지점

  1. shard-level trace schema 확장: accesses에 shard_id, block_id, resident_tier, queue_id를 추가하고, imbalance(q)와 per-shard p50/p99 service time을 계산하는 metric table을 완성하라.
  1. exact top-k merge 조건 분석: CSD local top-k 크기 k_local이 global exactness와 traffic에 미치는 영향을 수식화하고, k_local ∈ {k, 2k, 4k, all_scores} sweep 프로토콜을 설계하라.
  1. runtime simulator 설계: Poisson arrival, GPU queue, per-CSD queue, micro-batch timeout을 포함한 discrete-event simulator를 정의하고, 정책 all-GPU, all-CSD, whole-query routing, hybrid shard routing을 p50/p99 latency와 PCIe GB/s로 비교하라.

Turn 3 / Claude

이번 턴의 확장 요약

이번 턴은 CSD 내부 실행 datapath와 배치(batch)·머지(merge) 구조를 파고든다. 핵심 신규 기여는 (1) 동시 쿼리의 candidate 겹침을 단일 내부 스캔으로 흡수하는 batched internal scan과 그 이득을 정량화하는 BRF(batch reuse factor) — 이것이 locality 연구의 Heaps β를 runtime read 절감으로 직접 환산한다, (2) Codex의 k_local 우려 정정: pid-shard에서 k_local = k는 tie와 무관하게 항상 exact이며, 오히려 sharding 정책에 따라 k/m까지 줄일 수 있다(단 hit-ratio 최적 sharding과 상충), (3) Codex의 T_csd 내부 항이 read+compute 합(sum)으로 되어 있어 파이프라인 가정과 모순 — max로 정정하고 DRAM/NAND 병렬 여부라는 모델 분기를 추가, (4) 캐시 admission write가 read datapath와 내부 대역폭을 두고 경합한다는 write-cost-aware residency.

새로 추가한 사실

사실 1 — Batched internal scan: locality를 read 절감으로 바꾸는 runtime 레버

Turn 1·2는 쿼리 하나의 H2를 캐시 층에 두는 관점이었다. 그러나 CSD가 동시 도착 쿼리 배치를 한 번의 내부 스캔으로 처리하면, 서로 다른 쿼리의 candidate 겹침이 곧바로 read 절감이 된다. 배치 Bq개 쿼리의 union을 한 번만 읽고, 각 block을 그 block을 포함한 모든 쿼리에 대해 scoring한다:

읽는 block 집합      = ∪_{q∈batch} H2(q)          (union, 중복 제거)
내부 read bytes      = |∪H2| · s_block            (Σ_q|H2(q)| 아님)
MAC work             = Σ_{q,d∈H2(q)} Q·E          (줄지 않음 — 각 (q,d) 쌍 유지)

여기서 batch reuse factor를 정의한다:

BRF(Bq) = Σ_{q∈batch}|H2(q)| / |∪_{q∈batch} H2(q)|

BRF는 곧 배치 창(window) 안에서의 접근 재사용률이므로 Turn 1의 Heaps 지수 β로 즉시 예측된다. 배치 총 접근 T_b = Bq·n̄_H2, 고유 문서 U_b ≈ κ·T_b^β이면:

BRF(Bq) ≈ T_b^{1−β} / κ

규모 감(추정): n̄_H2=256, Bq=32 → T_b=8192. β=0.8이면 BRF ≈ 8192^0.2/κ ≈ 6.1/κ. κ≈1이면 배치 read를 ~6× 줄인다 — 압축 baseline 이득(~7×)과 맞먹는 크기이며, 이미 locality 연구가 측정하는 β에서 공짜로 나온다. 이것이 runtime이 locality를 활용하는 가장 직접적인 지점이고, Turn 1의 residency(정적 배치)와 별개의 축이다(residency = 어디 두나, batching = 언제 같이 읽나).

per-query 파이프라인 시간은 배치로 다음처럼 바뀐다:

T_perq_csd ≈ max( bytes_perq /(BRF·BW_int_eff),  MAC_perq/OPS )  +  T_fix,csd / Bq

→ read 항은 BRF로, 고정비는 Bq로 나뉜다. compute 항은 배치로 줄지 않으므로, 배치는 read/fixed-bound regime에서만 이득이고 compute-bound(N_mac 부족) CSD에서는 효과가 반감된다. 이는 Turn 1 사실 2(≥1,700 MAC)를 배치 실행에서 다시 요구하는 조건이다.

경계 조건(중요): BRF는 S_uniq(corpus locality만) ≤ 실측 ≤ S_zipf(query 반복 포함) 사이에 있다. dedup된 공개 query에서 concurrent overlap은 corpus locality 하한으로 제한되므로, 배치 이득 주장은 두 스트림 모두에서 측정해야 방어된다.

사실 2 — Hit-ratio 최적 sharding과 merge-traffic 최적 sharding은 상충한다

Codex는 "k_local = k가 tie에서 불안전할 수 있어 α·k sweep 필요"라 했으나 이는 정정 대상이다(아래 오류 1). 대신 진짜 설계 긴장은 sharding 정책에 있다:

  • random/hash sharding: 임의 문서의 global rank r인 문서는 shard-local rank 기대값 ≈ r/m. 따라서 k_local ≈ k/m(+집중 여유)만으로도 global top-k를 exact하게 복원 → merge egress 6·m·k_local ≈ 6k로 m에 무관.
  • locality/hot sharding(hit-ratio를 높이려 hot 문서를 CSD·특정 shard에 몰기): global top-k score가 소수 shard에 집중 → 그 shard의 k_local이 최대 k까지 필요 → merge traffic·머지 지연 증가.

즉 residency hit ratio를 올리는 배치(hot 집중)가 정확한 저비용 merge를 방해한다. 이는 Turn 2 tail-shard 문제(hot shard 독점 → p99 악화)와 같은 뿌리이며, sharding 정책이 hit ratio·tail latency·merge traffic 세 목적을 동시에 건드리는 3자 트레이드오프임을 뜻한다.

사실 3 — Cache admission write가 read datapath와 내부 대역폭을 경합한다

전통 캐시 문헌은 admission을 무료로 본다. 그러나 CSD DRAM(L2) resident set을 갱신하려면 NAND→DRAM write가 필요하고, 이 write는 MaxSim의 read 스트림과 같은 내부 채널·DRAM 포트를 쓴다. 따라서 online LRU/admission을 켜면 유효 read 대역폭이 낮아진다:

BW_int_eff_effective = BW_int_eff · (1 − ζ),   ζ = admission_write_bytes / total_internal_bytes

ζ가 크면 Turn 1의 임계식 BW_int_eff ≥ ε·BW_pcie가 admission 부담만으로 깨질 수 있다. 결론: 정적 STATIC-HOT(admission=0)이 online LRU보다 CSD에 유리할 수 있다 — 캐시 hit ratio는 LRU가 높아도 write 경합이 순이득을 잠식하기 때문. runtime은 admission을 idle window/double-buffer로 read 임계경로 밖에 밀어내거나, 주기적 배치 rebalancing(Turn 2 제안)으로 write를 amortize해야 한다.

더 구체화한 부분

정정된 CSD 실행 시간: sum → max (파이프라인) + 모델 분기

Codex의 per-shard 항 hit/BW_DRAM + miss/BW_NAND + MAC/OPS(전부 합)는 Turn 1의 max(T_read, T_mac) 파이프라인 가정과 모순이며, 최대 2×까지 비관적이다. 정정:

[read 내부 구조 분기]
 (a) 직렬(access-order 스트림, hot/cold 인터리브):
     T_read_s = (hit_bytes + miss_bytes)/BW_int_eff,   BW_int_eff = 조화평균 (Turn 1)
 (b) 병렬(DRAM 포트와 NAND 채널 독립 prefetch·reorder):
     T_read_s = max( hit_bytes/BW_DRAM,  miss_bytes/BW_NAND )   ← 더 빠름

[read↔compute 파이프라인]
 T_csd_s = T_fill + max( T_read_s,  MAC_s/OPS_s ) + T_res(≈0)
 T_fill  = (PE array depth)/f_csd  ≈ 수십~수백 ns  (N 큰 배치에서 무시 가능)

[쿼리 전체]
 T_query_csd = max_s T_csd_s + T_merge      (Turn 2 tail-shard 형태 유지)

→ 모델이 (a)/(b) 중 어느 read 구조를 가정하는지가 HR* 값을 바꾼다. (b) 병렬 read는 HR*를 낮춰(작은 residency로도 CSD 승리) CSD에 유리한 낙관 시나리오다. 어느 쪽인지는 실측 필요(controller 아키텍처 의존).

Exact top-k merge 비용과 근사 recall (Codex 우려 정정·확장)

pid-shard에서 global top-k에 드는 문서 d는 자기 shard 내 자기보다 위인 문서 수 ≤ (d의 global rank) ≤ k이므로 항상 자기 shard의 top-k 안에 있다 — tie 유무 무관, k_local = k면 exact. 따라서 traffic:

exact merge egress = m · k · (b_id + b_score) = m·k·6 B   (uint32 id + fp16 score)
                   = 8·100·6 = 4.8 KB  (m=8, k=100)   ← 무시 가능
merge compute      = k-way heap merge O(m·k·log m)          ← µs 미만

근사 regime(traffic·머지 더 줄이기): k_local = γk. random sharding에서 miss 확률은 global-rank-r 문서의 shard-local rank가 γk를 넘을 확률 = 이항 꼬리. 안전 여유 γ ≥ 1/m + c·√(1/(m·k)) (집중 부등식) → m=8, k=100이면 γ ≈ 0.2에서도 recall≈1. 즉 random sharding은 merge 비용을 5× 더 줄일 여지가 있으나, 사실 2의 hot-sharding과 병용 불가.

Micro-batch 창(window) 최적화: latency–throughput frontier

Codex가 지목한 batch timeout을 정량화한다. 도착률 λ, 창 w, Poisson이면 기대 배치 Bq = λw:

E[per-query 지연] ≈ w/2               (배치 대기)
                 + max(bytes_perq/(BRF(λw)·BW_int_eff), MAC_perq/OPS)   (서비스)
                 + T_fix,csd/(λw)      (고정비 상각)
  • w↑: 고정비 항↓, BRF↑(overlap 축적) → 서비스 효율↑. 그러나 대기 w/2↑.
  • BRF(λw)는 배치가 working set을 덮으면 포화(union → WSS) → knee w* 존재. w > w*는 대기만 늘리고 read 이득 없음.
  • 최적 창: w* ≈ WSS 커버 시간 부근, latency-SLO가 있으면 w ≤ SLO_budget − 서비스로 상한.

on-chip 자원 제약(신규): 배치 실행은 Bq개 query matrix를 on-chip에 상주시켜야 한다 → Bq·Q·E·b_e = Bq·8 KB. Bq=32 → 256 KB, Bq=128 → 1 MB SRAM. SSD 컨트롤러/FPGA BRAM 예산이 Bq의 하드 상한을 준다. 따라서 Bq ≤ min(SLO 허용, SRAM/8KB).

Straggler hedging (tail 대응 runtime 기법)

Turn 2의 tail-shard 문제에 대한 실행 레벨 해법: latency-critical 쿼리는 가장 느린 shard를 GPU에 speculative dual-dispatch(hedge). shard s의 예상 완료가 p99 위험이면 그 shard만 GPU로 복제 실행, 먼저 끝난 결과 채택.

비용 = 중복 shard의 PCIe·GPU 자원 (redundancy overhead)
이득 = T_query = max_s T_s 를 2nd-slowest 수준으로 절단 → p99↓
발동 = predicted T_s > τ_p99  일 때만 (예산 있는 쿼리 한정)

이는 Turn 2 risk_margin을 정적 회피가 아니라 동적 복제로 구현한 것이다.

관련 맥락 / 배경 설명

  • 왜 batching이 residency와 다른 축인가: residency(L2)는 시간 축의 재사용(reuse distance)을 캐시 hit으로 바꾸고, batching은 동시성 축의 재사용(concurrent overlap)을 single-scan으로 바꾼다. 두 축은 곱해진다 — resident hot block을 배치가 한 번 읽으면 그 read조차 DRAM 속도. 그래서 runtime cost model은 HR_bytes와 BRF를 동시에 입력으로 받아야 한다.
  • 캐시 write 경합이 runtime-구조적 신규 지점인 이유: MaxSim datapath와 캐시 관리가 물리적으로 같은 내부 대역폭을 공유하므로, "캐시 정책 최적화"와 "데이터 경로 최적화"가 분리 불가능하다. 이는 host DRAM page cache(write가 별도 DMA)와 근본적으로 다른, CSD 고유의 제약이다.
  • sharding이 세 목적을 동시에 건드림: (i) hit ratio(hot 집중이 유리), (ii) tail latency(분산이 유리), (iii) merge traffic(분산이 유리). random hash sharding은 (ii)(iii)에 좋고 (i)에 나쁘며, hot-affinity sharding은 반대다. runtime은 hybrid(hot 문서는 복제해 여러 shard에 두어 hit는 살리고 부하는 분산)로 절충할 수 있으나 복제는 L2 용량을 소비한다.

잠재적 오류 또는 확인 필요 지점

  1. (정정) Codex의 k_local tie 우려: pid-shard에서 k_local = k는 tie와 무관하게 exact — 위 증명. sweep이 필요한 것은 exactness가 아니라 근사로 traffic을 줄이는 방향(γk < k)이며, 그 하한은 sharding 정책이 정한다. — first-principles 정정, 실측 불필요.
  2. (정정) Codex T_csd 내부 항의 sum: read↔compute는 파이프라인이므로 max, read 내부도 DRAM/NAND 병렬이면 max. sum은 최대 2× 비관. — 모델 정정; (a)직렬/(b)병렬 분기는 아키텍처 실측 필요.
  3. BRF(Bq)의 실제 β와 κ — trace 실측 필요. 특히 concurrent 배치 창 안의 overlap은 전체 스트림 β와 다를 수 있음(짧은 시간창은 hot 문서 편중이 더 심할 수 있어 β 유효값↓, BRF↑). 배치 창 크기별 β를 따로 측정해야 함.
  4. admission write 계수 ζ와 CSD DRAM write 대역폭 — 실측 필요. double-buffer로 read 임계경로 밖에 밀 수 있는지가 사실 3의 순이득을 좌우.
  5. on-chip query buffer Bq·8KB가 대상 CSD(SmartSSD KU15P BRAM/URAM ≈ 수 MB — 배경지식, 확인 필요)에서 실제 Bq 상한을 얼마로 만드는지 — 스펙 확인 필요.
  6. hedging의 GPU 복제가 PCIe·GPU 큐를 얼마나 오염시키는지 — p99 이득 대비 평균 throughput 손실은 시뮬 실측 필요.

Turn 4 / Codex

이번 턴의 확장 요약

이번 턴은 runtime scheduler가 실제로 다루는 제어 경로(control path) 와 압축 residual MaxSim datapath를 추가한다. 이전 턴들이 HR_bytes, BRF, shard tail을 중심으로 read/compute 비용을 봤다면, 여기서는 CSD가 받는 query matrix + H2 plan descriptor 자체가 작지만 무시할 수 없는 입력이라는 점, 그리고 ColBERTv2 압축 residual은 “fp16 vector로 완전 복원 후 MAC”만이 아니라 centroid score + residual LUT accumulation으로 구현될 수 있다는 점을 모델에 넣는다.

핵심 신규 결론은 다음이다.

CSD offload cost =
  control ingress(query + plan)
  + union-block internal read
  + compressed residual scoring
  + local top-k egress
  + queue/backpressure delay

즉 CSD scheduler는 단순히 ρ_q가 높은 query를 보내는 장치가 아니라, 계획 descriptor 크기, block-level BRF, 압축 연산 형태, queue stability까지 보는 runtime policy가 되어야 한다.

새로 추가한 사실

1. H2 plan descriptor traffic은 작지만 0이 아니다

CSD는 residual을 직접 읽더라도, host/runtime은 CSD에 “무엇을 읽고 어떤 query에 대해 scoring할지”를 알려줘야 한다. 따라서 CSD ingress는 query embedding만이 아니다.

Bytes_ctrl(batch)
  = Bq · Q · E · b_q
  + |U_block| · b_block_desc
  + Σ_{block∈U} membership_bytes(block)

예시 가정:

Bq = 32, Q=32, E=128, fp16 query
query bytes = 32 · 32 · 128 · 2 = 262 KB

naive per-(q,pid) descriptor:
32 queries · 256 docs/query · 16 B ≈ 128 KB

따라서 small-query/small-H2 regime에서는 control ingress가 CSD 고정비와 함께 latency를 흔든다. 반면 embedding traffic은 32·256·2.5KB ≈ 20MB 수준이므로, batch가 충분하면 descriptor는 여전히 1차 병목은 아니다.

중요한 runtime 최적화는 descriptor를 doc 단위가 아니라 block-union + query membership bitset으로 압축하는 것이다.

block_plan(block_id, shard_id, byte_offset, byte_len, qmask_or_qid_list)

이 구조를 쓰면 CSD는 block을 한 번 읽고, 해당 block이 필요한 query 집합에만 score를 누적한다.

2. BRF는 pid-count가 아니라 byte/block 기준으로 다시 정의해야 한다

이전 턴의 BRF = Σ|H2(q)| / |∪H2(q)|는 직관은 좋지만, 실제 storage read 절감량을 과대평가할 수 있다. CSD는 pid가 아니라 block/page 단위로 읽고, 한 block에 여러 pid가 섞인다.

정확한 read-side batch reuse factor는 다음이 더 낫다.

BRF_bytes(Bq)
  = Σ_q Σ_{b∈Blocks(H2(q))} bytes_read_for_q(b)
    / Σ_{b∈∪_q Blocks(H2(q))} bytes(b)

추가로 block packing 때문에 필요 없는 pid가 같이 읽히는 비용을 분리한다.

false_positive_bytes
  = Σ_{b read} bytes(b) - Σ_{pid actually scored} bytes(pid)

block_efficiency
  = useful_pid_bytes / physical_block_read_bytes

따라서 locality trace는 pid locality와 block locality를 둘 다 남겨야 한다. CSD의 PCIe 절감은 pid byte 기준으로 보이지만, CSD 내부 NAND/DRAM bandwidth는 block byte 기준으로 소모된다.

3. 압축 residual MaxSim은 MAC array 요구를 바꿀 수 있다

ColBERTv2 residual이 centroid + quantized residual 형태라면, CSD가 매 token vector를 fp16으로 완전 복원한 뒤 Q·E fp16 MAC을 수행할 필요는 없다. 점수는 다음처럼 분해할 수 있다.

dot(q_i, x_dj)
  = dot(q_i, centroid_c)
  + dot(q_i, residual_code_dj)

query별 centroid 점수는 stage-1에서 이미 계산됐거나 CSD가 작은 centroid table로 재계산할 수 있다.

S_cent[i,c] = dot(q_i, centroid_c)

2-bit residual이라면 residual term은 multiplication보다 LUT accumulation에 가깝다.

LUT[i,e,code] = q_i[e] · decode(code)

dot_residual(q_i, r_j)
  = Σ_e LUT[i,e,r_j[e]]

이 경우 datapath 요구는 다음처럼 나뉜다.

T_score_csd =
  max(
    bytes_code / BW_unpack,
    Q · E · num_tokens / ADD_or_LUT_OPS
  )

즉 “필요 MAC 수 1,700개”는 fp16 복원 후 MAC 구현의 상한적 설계점이고, LUT accumulation 구조를 쓰면 DSP MAC 요구는 줄어들 수 있다. 대신 bit-unpack bandwidth, LUT SRAM port 수, accumulator precision이 새 병목이 된다.

더 구체화한 부분

Batched CSD service-time model v2

배치 G에 대해 shard s의 실행 시간은 read는 union 기준, compute는 query-doc pair 기준으로 계산해야 한다.

U_s(G) = ∪_{q∈G} Blocks(H2_s(q))

Read_s(G)
  = bytes(U_s(G)) / BW_read_s

Compute_s(G)
  = Σ_{q∈G} Σ_{d∈H2_s(q)} cost_score(q,d) / OPS_s

Ctrl_s(G)
  = bytes_ctrl_s(G) / BW_pcie_down

S_s(G)
  = T_fix,csd
    + Ctrl_s(G)
    + max(Read_s(G), Compute_s(G))
    + Result_s(G)

query latency는 tail shard로 결정된다.

T_csd(G) = max_s [ queue_delay_s + S_s(G) ] + T_host_merge

여기서 BRF_bytes는 Read_s만 줄인다. Compute_s는 줄지 않는다. 따라서 CSD가 compute-bound이면 batching의 효과는 T_fix 상각 정도로 제한된다.

Queue-stable routing rule

스케줄러는 가장 빠른 장치를 고르는 것만으로는 부족하다. p99를 지키려면 각 실행 자원의 utilization을 제한해야 한다.

ρ_gpu = λ_gpu · E[S_gpu] < ρ_max
ρ_csd_s = λ_s · E[S_csd_s] < ρ_max

실용적인 online rule은 virtual finish time 기반이 좋다.

def assign_shard(shard_task):
    candidates = ["GPU", shard_task.home_csd]

    if shard_task.hot_replicated:
        candidates += shard_task.replica_csds

    return argmin_j(
        virtual_finish_time[j]
        + predicted_service_time(j, shard_task)
        + tail_risk_penalty(j, shard_task)
    )

whole-query routing보다 shard-task routing이 더 유연하다. 단, final top-k exactness를 위해 같은 pid의 모든 token은 한 실행 주체에서 score를 완성해야 한다.

기존 trace에 batch와 descriptor 계측 필드를 추가한다.

queries(q_idx, qid, t_arrival, n_H2, bytes_H2_comp, rho_bytes, w_bytes)

accesses(q_idx, pid, block_id, shard_id, bytes_comp,
         resident_tier, stage_max)

batches(batch_id, q_idx_list, window_start, window_close,
        bytes_query, bytes_plan_desc, bytes_result)

block_reads(batch_id, shard_id, block_id,
            physical_bytes, useful_pid_bytes,
            qmask_popcnt, source_tier)

shard_service(batch_id, shard_id,
              read_us, compute_us, ctrl_us,
              queue_us, total_us)

이 schema가 있어야 BRF_bytes, block_efficiency, tail shard, descriptor overhead를 분리해서 볼 수 있다.

관련 맥락 / 배경 설명

CSD MaxSim runtime에서 중요한 분리는 세 가지다.

첫째, logical access와 physical read는 다르다. PLAID의 H2는 pid 집합이지만 CSD는 block/page를 읽는다. 따라서 locality metric은 pid 기준뿐 아니라 block 기준이어야 한다.

둘째, control plane과 data plane은 다르다. embedding traffic을 없애도 H2 plan descriptor와 query matrix는 내려가야 한다. 이 항은 large batch에서는 작지만 low-latency single-query에서는 무시하면 안 된다.

셋째, 압축 format은 하드웨어 구조를 결정한다. fp16 GEMM처럼 보면 MAC array가 중심이지만, quantized residual을 직접 scoring하면 bit-unpack, LUT, accumulator tree가 중심이 된다. CSD 제안은 “몇 MAC이면 되는가”와 함께 “압축을 어디까지 풀 것인가”를 설계 변수로 둬야 한다.

잠재적 오류 또는 확인 필요 지점

  1. 이전 턴의 BRF는 문서 수 기준이라 내부 read bandwidth 절감을 직접 나타내지 못한다. 실제 모델은 BRF_bytes와 block_efficiency로 바꿔야 한다.
  1. k_local = k exact 조건은 global H2 후보 집합이 이미 확정되어 있고 pid 단위로 shard된 경우에만 성립한다. 각 CSD가 독립적으로 pruning까지 수행하면 global candidate 누락이 생길 수 있어 exactness 보장이 깨진다.
  1. max(Read, Compute) 모델은 double buffering과 충분한 stream pipeline이 있을 때만 맞다. bit-unpack, LUT SRAM port, accumulator writeback이 직렬화되면 Compute 쪽에 추가 stall이 들어간다.
  1. ColBERTv2 residual encoding의 실제 bit layout, centroid id 저장 위치, decompression kernel 구조는 코드/논문 확인이 필요하다. LUT accumulation이 가능한지는 format 세부에 의존한다.
  1. CSD command interface가 block-union plan과 qmask를 효율적으로 받을 수 있는지 확인해야 한다. NVMe command 수가 block 수에 비례하면 descriptor가 작아도 command submission overhead가 병목이 될 수 있다.

다음 에이전트가 이어서 확장할 지점

  1. compressed residual datapath 설계 비교: fp16 reconstruct+MAC, centroid score + residual LUT accumulation, integer dot approximation 3가지를 DSP 수 / SRAM port / accumulator bitwidth / latency 표로 비교하라.
  1. block-level locality trace 설계 완성: pid-level HR_bytes와 block-level BRF_bytes, block_efficiency, false_positive_bytes를 같은 trace에서 계산하는 절차와 pseudo-code를 작성하라.
  1. queue-stable shard scheduler 시뮬레이터 구체화: virtual finish time, utilization cap, batch timeout, descriptor overhead, tail-risk penalty를 포함해 all-GPU / whole-query CSD / shard-hybrid / batched-union CSD를 p50, p99, PCIe GB/s, CSD internal GB/s로 비교하는 실험 프로토콜을 정의하라.

이번 턴의 확장 요약

이번 턴은 물리 layout(block) ↔ residency 임계값과 압축 residual의 실제 datapath 원가, 그리고 pruning의 분산 실행 프로토콜을 파고든다. 핵심 신규 결론 세 가지:

  1. Turn 4가 분리해 둔 block_efficiency와 BRF_bytes는 하나의 양(block packing amplification π)으로 합쳐지고, 이 π가 host가 pid를 compact해서 DMA하는 baseline에서는 CSD 쪽에만 붙는 페널티라서 Turn 1의 임계 residency를 HR* 0.61 → ~0.88로 크게 끌어올린다. → access trace로 passage를 재배치(co-access clustering)하는 것이 residency만큼 중요한 레버가 된다.
  2. Turn 4의 "LUT accumulation이면 MAC 요구가 줄 수 있다"는 방향은 맞지만 부정확하다. 실제로는 fp16 곱셈기 array가 사실상 0으로 붕괴(centroid 항은 stage-1에서 이미 계산됨)하고, 대신 LUT-read 대역폭(~1.5 T lookups/s → ~1,500–1,900 뱅크)이 새 병목으로 이동한다. 유닛 개수는 ~1,700로 비슷하나 유닛 종류(fp16 multiplier → adder+banked SRAM)가 바뀐다.
  3. Turn 4 확인점 2("각 CSD가 독립 pruning하면 exactness 깨짐")를 one-round(host prune) vs two-round(CSD prune) 프로토콜로 정식화하고, local pruning의 over-provision recall 부등식을 유도한다.

새로 추가한 사실

사실 1 — Block packing amplification π가 CSD에만 붙어 HR*를 0.61→~0.88로 끌어올린다

Turn 4는 block_efficiency(spatial 낭비)와 BRF_bytes(배치 재사용)를 병렬로 나열했지만, runtime 비용에는 둘의 합성 하나만 들어간다. 정의:

B*  = Σ_{d∈∪_q H2(q)} bytes_comp(d)      # 이상적: 필요 pid를 압축 상태로 1회씩만
P   = |∪_q blocks(H2(q))| · s_block       # 물리: 실제 읽는 block union
π   = P / B*                              # block packing amplification (≥1, 배치 내 block공유가 상쇄 일부)

핵심 비대칭(신규): GPU baseline이 host page cache로 block을 읽어 필요한 pid만 gather한 뒤 H2D DMA하면, PCIe에는 useful pid byte B*만 흐른다. 반면 CSD는 내부 read에서 물리 block P = π·B*를 그대로 소모한다. 즉 π는 GPU가 아니라 CSD 쪽에만 붙는 페널티다. 따라서 Turn 1의 내부대역폭 조건이 π만큼 강화된다:

BW_int_eff(HR) / π ≥ ε·BW_pcie
⇒ HR* = ( 1/BW_NAND − 1/(π·ε·BW_pcie) ) / ( 1/BW_NAND − 1/BW_DRAM )

가정값(BW_DRAM=40, BW_NAND=12, ε·BW_pcie=21 GB/s, 모두 배경지식/실측 필요):

π유효 요구 BW_intHR*
1.0 (block=passage, 완벽 정렬)21 GB/s0.61 (Turn 1 값)
1.5 (2.5KB passage / 4KB block, 중간 낭비)31.5 GB/s0.88
2.0 (block당 유효 pid 1개꼴)42 GB/s> 1 (단일 CSD 불가)

해석: block packing이 조금만 나빠도 CSD가 warm-host baseline을 이기는 데 필요한 residency가 급등한다. 이 페널티를 없애는 길은 둘:

  • (a) baseline을 GPUDirect Storage(GDS)로 두면 GPU도 block을 통째로 읽으므로 π가 양쪽에서 상쇄된다 → HR*는 0.61로 복귀. 즉 CSD의 공정한 상대는 "host-gather baseline"이 아니라 "GDS block-read baseline"일 때 유리하다.
  • (b) access trace로 passage를 재배치해 co-access되는 pid를 같은 block에 모으면 π→1. 이것이 사실상 Turn 1의 locality 지표를 물리 layout 최적화로 환산하는 지점이다(아래 다음 조사 1).

사실 2 — 압축 datapath: fp16 곱셈기는 붕괴하고 병목이 LUT-read 대역폭으로 이동

Turn 4의 centroid+residual 분해는 옳지만, PLAID는 centroid 점수 S_cent[i,c]=⟨q_i, centroid_c⟩를 stage-1에서 이미 Q×C_probe 밀집행렬로 계산한다. 따라서 scoring 시점의 doc token j에 대한 점수는:

score(i,j) = S_cent[i, c_j]  (precomputed → ADD 1회, 곱셈 0)
           + Σ_e LUT[i,e, code_j[e]]   (residual, ADD·lookup만)

→ scoring 경로에 fp16 곱셈기가 필요 없다(centroid는 이미 계산, LUT 값은 q_i[e]·bucket[code]로 쿼리당 Q·E·2^b=16,384 곱만 1회 빌드). Turn 1의 "≥1,700 MAC"은 fp16 재복원 구현의 상한 설계점이고, LUT 구현에서는 곱셈기 array가 사라진다. 대신 새 병목은 LUT-read 대역폭:

token rate (read-limited) = BW_int / bytes_code ≈ 12e9 / 32B ≈ 3.75×10^8 tok/s
LUT lookups/s            = token_rate · (Q·E) = 3.75e8 · 4096 ≈ 1.5×10^12 /s
필요 병렬 LUT 뱅크        = 1.5e12 / (f·u) = 1.5e12/(1e9·0.8) ≈ 1,900 뱅크(=accumulate lane)
LUT 상주 용량            = Bq · Q·E·2^b · b_e = 32 · 16384 · 2B = 1 MB SRAM

즉 유닛 개수는 여전히 ~1.7–1.9K로 fp16 MAC과 같은 자릿수지만, 유닛 종류가 "fp16 multiplier"에서 "adder + 고banking SRAM 포트"로 바뀐다(면적·전력은 유리, SRAM 포트 압박은 새 제약). 이것이 Turn 4의 "MAC 요구가 줄 수 있다"의 정확한 형태다: 줄어드는 것은 곱셈기이고, 새로 생기는 요구는 1.5 T reads/s를 감당할 LUT 뱅킹/복제다.

사실 3 — Distributed pruning: one-round vs two-round 프로토콜과 recall over-provision 부등식

Turn 4 확인점 2를 프로토콜로 정식화한다. residual scoring(stage-3)을 CSD가 하려면 stage-1/2 pruning의 실행 위치가 exactness를 결정한다.

[one-round, host-prune]  (Turn 3의 k_local=k exact 전제)
  host: 전체 centroid 점수 → 전역 H2 확정 → pid→shard 분배 → 각 CSD에 survivor list 송신
  비용: host가 centroid table(수백 MB, 배경지식) 또는 CSD가 올려준 centroid 점수 보유 필요
  왕복: 1회.  exactness: 보장(H2가 전역에서 확정됨)

[two-round, CSD-prune]
  R1: query matrix(8KB) → 모든 CSD; 각 CSD가 자기 shard의 centroid-approx 점수 계산·local top-k' 반환
  host: 전역 merge → H2 확정 → survivor list 회신
  R2: 각 CSD가 survivor residual read+full score → local top-k 반환
  비용: centroid table을 host로 옮길 필요 없음, host centroid 연산 없음
  왕복: 2회 → T_fix ~2× (단일 query latency 악화)

CSD가 R1 없이 독립적으로 local pruning만 하고 곧장 stage-3로 가면 전역 top-k 후보가 누락될 수 있다. random-hash shard에서 전역 top-N 후보가 shard-local rank로 흩어지므로, recall ≥ 1−δ 보장에 필요한 local 유지 개수는 이항 꼬리로:

k_local^prune ≥ N/m + z_δ·√( N·(1/m)·(1−1/m) )    (전역 후보 N개, shard m개)
예: N=1024, m=8 → N/m=128, z_{0.99}≈2.33 → k_local^prune ≈ 128 + 2.33·10.6 ≈ 153

즉 독립 pruning은 ~1.2×(=153/128) over-fetch로 recall≈0.99를 살 수 있다 — one-round 왕복을 없애는 대신 ~20% 내부 read를 더 쓴다. 이 20%가 사실 1의 π와 곱해져 residency 요구를 다시 올리므로, latency(왕복) ↔ 내부 read(over-fetch) ↔ exactness의 3자 트레이드가 성립한다.

더 구체화한 부분

Descriptor 압축 정량화와 NVMe command-count 병목

Turn 4의 "block-union + qmask" 이득을 수치화한다. 배치 union block ≈|∪blocks|, descriptor = block_id(4)+shard(1)+offset(4)+len(2)+qmask(⌈Bq/8⌉=4) ≈ 15B:

union blocks ≈ (Σ|H2|_pid / BRF_pid) · (pid/ block) ≈ (8192/6)·? ≈ 850–1,400 block (추정)
descriptor traffic ≈ 1,000 · 15B ≈ 15 KB   vs. naive per-(q,pid) 128 KB  → ~8–10× 절감

단 Turn 4 확인점 5(NVMe command-count)를 정식화하면: submission overhead ∝ #blocks이면 descriptor byte가 작아도 command 수가 병목이다. 완화책은 block-list를 단일 CS(computational storage) command로 전달(SNIA CS API의 batched descriptor, 확인 필요)해서 command 수를 O(1)로 만드는 것. 이것이 안 되면 T_ctrl = n_cmd·t_submit가 T_fix,csd를 지배할 수 있다.

S_burst: BRF는 도착 수가 아니라 도착 상관에 의존한다

Turn 3의 BRF(Bq)는 배치 크기만의 함수로 모델링됐으나, 실제 concurrent overlap은 query 내용 상관에 좌우된다. 뉴스 이벤트 등 위상적 burst에서는 동시 도착 query가 같은 hot passage를 공유해 window 내 overlap이 급증한다. 따라서 Turn 1의 S_uniq/S_zipf에 세 번째 스트림 S_burst를 추가한다:

S_burst: query pool을 topic 클러스터로 묶고, self-exciting(Hawkes) 도착으로 topical burst 합성
측정: window w별 BRF_block(w)을, S_uniq(하한)·S_zipf(정상)·S_burst(상한) 세 스트림에서 각각
  • S_zipf는 pool에서 i.i.d. 복원추출이라 window 내 상관을 0으로 만들어 BRF를 과소평가한다. 배치 이득의 상한을 방어하려면 S_burst가 필요하다.
  • 예상: BRF_block(S_burst) > BRF_block(S_zipf) > BRF_block(S_uniq). runtime 배치 정책은 세 값 사이의 최악(S_uniq)에서도 payoff 부등식 BRF_block > π(사실 1)를 만족해야 안전하다.

Query encoder 배치가 만드는 GPU utilization 바닥과 필수 GPU→CSD hop

Turn 0가 "query encoder(BERT)는 양 경로 공통이라 소거"했지만, runtime 구조에서는 두 함의가 남는다:

  • CSD는 BERT를 못 돌린다 → all-CSD 경로도 GPU/host가 encoder를 돌려야 하므로 GPU는 절대 idle이 아니다. 스케줄러가 봐야 할 값은 GPU idle이 아니라 encoder 이후 남는 GPU MaxSim slack이다.
  • query matrix(8KB)는 GPU encoder의 출력 → 단일 query CSD 경로도 GPU→host→CSD로 8KB를 내려야 한다(critical path). batch에서는 상각되지만 low-latency 단일 query에서는 T_fix,csd에 더해지는 고정 hop이다.

Corrected service-time: LUT datapath에서 compute-floor가 배치로 더 빨리 지배

사실 2를 Turn 4의 S_s(G)에 대입하면, batching이 read를 BRF_block로 줄이는 동안 LUT-read compute는 줄지 않으므로, 배치가 커질수록 compute-bound로 넘어가는 임계 Bq_comp가 fp16 경로보다 낮다:

read 항  = π·B*/(BRF_block·BW_int)     # Bq↑ 시 BRF_block↑로 감소
comp 항  = Σ_{q,d} Q·E / OPS_lut       # Bq에 선형 증가(줄지 않음)
⇒ Bq_comp = argmin_{Bq} [read = comp]  이후로는 배치가 LUT-read 대역폭에만 의존

즉 압축-LUT datapath에서는 배치의 read 이득이 LUT 뱅킹 예산(사실 2)에서 조기에 포화한다. Turn 3의 "compute는 배치로 안 줄어든다"가 LUT 구현에서 더 날카롭게 성립한다.

Turn 6 / Codex

이번 턴의 확장 요약

이번 턴은 runtime scheduling의 목적 함수와 안정성을 추가한다. 이전 턴들이 HR_bytes, π, BRF, LUT datapath를 개별 비용 항으로 정리했다면, 이번에는 이 항들을 실제 scheduler가 쓰는 다중 목적 제어 문제로 묶는다.

핵심 신규 결론은 세 가지다.

  1. CSD offload scheduler의 목표는 “가장 빠른 장치 선택”이 아니라 latency SLO를 지키면서 PCIe·GPU MaxSim 자원을 최소화하는 것이다.
  2. L2 residency admission은 hit ratio 최대화가 아니라 future PCIe saving / internal write cost / shard-tail risk의 이득 밀도로 결정해야 한다.
  3. batching은 단순 도착 순서 FIFO보다 H2 block overlap signature 기반 content-aware batching이 유리하며, 이는 BRF_block을 직접 끌어올리는 scheduling factor다.

---

새로 추가한 사실

1. Scheduler 목적 함수는 latency 단일 최적화가 아니다

CSD가 GPU보다 약간 느려도, PCIe와 GPU MaxSim queue를 크게 줄이면 전체 시스템에서는 더 좋은 선택일 수 있다. 따라서 query 또는 shard task x의 routing은 다음 목적 함수로 보는 것이 맞다.

minimize  E[T_x]
        + λ_tail · max(0, P99_x − SLO)
        + λ_pcie · PCIe_bytes_x
        + λ_gpu  · GPU_MaxSim_time_x
        + λ_csd  · CSD_internal_bytes_x
  • latency-critical search serving: λ_tail이 큼
  • GPU가 encoder 또는 reranker와 공유됨: λ_gpu가 큼
  • PCIe가 병목인 multi-tenant 서버: λ_pcie가 큼
  • CSD NAND endurance/write 경합이 문제: λ_csd가 큼

즉 동일한 trace에서도 scheduler 결론은 운영 목표에 따라 달라진다. 논문에서는 최소 두 가지 목표를 분리해야 한다.

Mode A: latency-first
  primary metric = p99 latency
  secondary metric = PCIe GB/s

Mode B: resource-first
  primary metric = PCIe GB/query, GPU-ms/query
  constraint = p99 ≤ SLO

2. Admission policy는 “hot 문서”가 아니라 “순이득 문서”를 pin해야 한다

Turn 5의 ζ는 admission write가 read datapath와 경합한다는 점을 지적했다. 이를 policy로 바꾸면, 문서 d를 CSD DRAM에 올릴지의 점수는 빈도만으로는 부족하다.

Benefit(d)
  = E_future_access(d) · bytes_comp(d) · saved_path_cost(d)

Cost(d)
  = bytes_comp(d) / BW_admit
  + write_amplification(d)
  + shard_tail_penalty(d)
  + eviction_cost(d)

Admit if Benefit(d) / Cost(d) > θ

f_train(d) 상위 문서 STATIC-HOT은 좋은 baseline이지만, runtime에서는 다음 항이 필요하다.

saved_path_cost(d)
  = 1/(ε·BW_pcie) − π_d/BW_int_eff

이 값이 음수면, 아무리 hot해도 CSD에 올릴 이유가 약하다. 특히 π_d가 큰 block layout에서는 hot 문서가 오히려 내부 bandwidth를 낭비할 수 있다.

3. Content-aware batching이 BRF_block을 직접 제어한다

FIFO micro-batch는 구현이 쉽지만, 같은 시간 창 안에서도 query 간 H2 block overlap이 크게 다를 수 있다. scheduler는 query q의 block signature를 만들어 overlap이 큰 query끼리 묶을 수 있다.

sig(q) = MinHash( Blocks(H2(q)) ) 또는 Bloom sketch

batch_score(q, G)
  = estimated_overlap(sig(q), sig(G))
  − α_wait · expected_wait(q)
  − α_tail · shard_imbalance(q, G)

이때 batch의 read 이득은 다음처럼 관찰된다.

BRF_block(G)
  = Σ_q physical_bytes(Blocks(H2(q)))
    / physical_bytes(∪_q Blocks(H2(q)))

FIFO와 content-aware batching을 비교하면, batching 자체의 효과와 topical/query-locality 활용 효과를 분리할 수 있다.

---

더 구체화한 부분

Queue-stable shard scheduler v3

scheduler는 장치별 virtual finish time과 resource budget을 같이 본다.

def route_task(task):
    candidates = feasible_devices(task)

    best = None
    for dev in candidates:
        service = predict_service_time(dev, task)
        finish = virtual_finish_time[dev] + service

        penalty = 0
        penalty += lambda_tail * tail_risk(dev, task)
        penalty += lambda_pcie * pcie_bytes(dev, task)
        penalty += lambda_gpu  * gpu_ms(dev, task)
        penalty += lambda_csd  * csd_internal_bytes(dev, task)

        score = finish + penalty

        if utilization_after(dev, task) > rho_max:
            score += overload_penalty

        best = min(best, (score, dev))

    return best.dev

핵심은 rho_max다.

ρ_gpu   = λ_gpu · E[S_gpu]   < ρ_max
ρ_csd_s = λ_s   · E[S_csd_s] < ρ_max

ρ_max≈0.7~0.85는 임의 가정이며 실험 sweep 대상이다. p99는 utilization이 1에 가까워질수록 비선형으로 악화되므로, 평균 latency만으로 라우팅하면 tail이 쉽게 무너진다.

PCIe cap 기반 break-even

기존 break-even은 T_csd ≤ T_gpu였다. 운영 환경에서는 PCIe 포화 조건도 필요하다.

All-GPU PCIe demand:
D_pcie = λ · E[bytes_H2_comp]

PCIe stable condition:
D_pcie ≤ β_pcie · ε · BW_pcie

β_pcie는 안전 사용률이다. 예를 들어 β_pcie=0.7, ε·BW_pcie=25GB/s, E[bytes_H2_comp]=0.6MB/query이면:

max λ_all_gpu ≈ 0.7 · 25GB/s / 0.6MB ≈ 29k QPS

20k QPS에서는 latency상 GPU가 빠를 수 있지만, 다른 GPU traffic과 encoder/reranker가 붙으면 CSD offload가 필요해진다. 이 경우 CSD의 역할은 “더 빠른 MaxSim”이 아니라 PCIe pressure valve다.

Trace schema 추가: scheduling outcome까지 기록

기존 trace는 access 중심이었다. scheduler 연구에는 결정과 예측 오차가 필요하다.

routing_decisions(
  q_idx, task_id, granularity,              # query | shard | block-batch
  chosen_device, candidate_devices,
  pred_gpu_us, pred_csd_us,
  pred_pcie_bytes, pred_internal_bytes,
  pred_tail_risk,
  actual_latency_us,
  actual_queue_us,
  actual_pcie_bytes,
  actual_internal_bytes
)

cache_events(
  t_idx, pid, block_id, shard_id,

  bytes_written,

이 두 테이블이 있어야 cost model이 얼마나 틀렸는지, admission이 실제로 이득이었는지, shard tail이 어디서 생겼는지 역추적할 수 있다.

관련 맥락 / 배경 설명

CSD MaxSim runtime은 accelerator scheduling이라기보다 multi-tier memory scheduling에 가깝다. GPU는 compute device이지만, 이 문제에서는 PCIe로 데이터를 먹여야만 빠르다. CSD는 compute가 약하지만, embedding을 움직이지 않는다는 장점이 있다.

따라서 좋은 scheduler는 다음 세 질문을 query마다 답해야 한다.

1. 이 query의 residual bytes는 어디에 이미 따뜻한가?
   GPU HBM / host DRAM / CSD DRAM / NAND

2. 이 query를 GPU로 보내면 어떤 shared resource가 막히는가?
   PCIe / GPU stream / encoder batch / HBM bandwidth

3. 이 query를 CSD로 보내면 어떤 tail risk가 생기는가?
   cold shard / bad block packing / queue depth / admission write

이 구조에서는 HR_bytes(B) 하나만 높은 정책보다, π, BRF_block, queue depth, prediction error까지 함께 낮추는 정책이 더 강한 연구 주장이다.

잠재적 오류 또는 확인 필요 지점

  1. Turn 5의 “π가 CSD에만 붙는다”는 결론은 host-gather GPU baseline에서만 맞다. GDS block-read baseline, mmap 직접 전송, compressed contiguous layout baseline에서는 GPU도 일부 π를 부담할 수 있다. baseline을 세 개로 나눠야 한다.
GPU-HG: host gather 후 useful bytes만 H2D
GPU-GDS: block 단위 direct read
GPU-L0: HBM resident
  1. “fp16 multiplier가 0으로 붕괴”는 centroid score와 residual LUT가 정확히 재사용될 때만 성립한다. scale, residual norm, per-dimension codebook, dequant offset이 있으면 multiplier 또는 extra add가 일부 부활한다.
  1. BRF_block은 batch 크기만의 함수가 아니다. query arrival correlation, topic burst, scheduler의 reordering window에 의존한다. 따라서 S_zipf만으로 batching 이득을 주장하면 약하다.
  1. admission policy의 E_future_access(d)는 train/test drift에 민감하다. online EWMA를 쓰면 빠르게 적응하지만 oscillation이 생길 수 있으므로 hysteresis 또는 minimum residency time이 필요하다.
  1. content-aware batching은 latency를 희생한다. SLO가 빡빡한 workload에서는 BRF 이득보다 waiting penalty가 더 클 수 있다.

Turn 7 / Claude

이번 턴의 확장 요약

이번 턴은 Turn 6이 세운 다목적 scheduler 제어 문제의 내부 정합성을 파고든다. 핵심 신규 결론 네 가지:

  1. admission과 routing은 순차 결정이 아니라 결합 고정점(fixed point)이고, 빈 L2에서 출발하면 cold-start deadlock에 빠진다 — 이것이 Turn 1의 STATIC-HOT을 "baseline"이 아니라 bootstrap 필수 조건으로 격상시킨다.
  2. Turn 6의 admission saved_path_cost = 1/(ε·BW_pcie) − π/BW_int는 routing 결정의 절감을 admission 결정에 잘못 붙인 것이다. admission의 진짜 이득은 PCIe가 아니라 내부 NAND→DRAM 차이이며, read-bound regime에서만 유효하다.
  3. Turn 6의 monolithic λ_gpu · GPU_MaxSim_time은 잘못된 자원 회계다. MaxSim은 memory-bound라 offload가 해방하는 것은 GPU FLOPs가 아니라 대역폭 층(HBM/PCIe/host-DRAM)이다 — co-located compute-bound reranker에는 거의 도움이 안 된다.
  4. Turn 6의 objective 항 λ_tail · max(0, P99_x − SLO)는 per-task P99가 정의 불가라 초과확률/CVaR로 정정해야 한다.
  5. Turn 6의 "PCIe pressure valve"를 load-adaptive activation 닫힌형 f_csd*(λ) = max(0, 1 − λ_switch/λ)로 정량화한다.

새로 추가한 사실

사실 1 — admission↔routing은 결합 고정점이고 cold-start에서 deadlock한다

Turn 6은 admission(사실 2)과 routing(v3)을 별도 절차로 뒀지만, 둘은 서로의 입력이다:

routing:    route(q) ← ρ_q = (H2(q) ∩ L2 byte 비율)      # L2 상태에 의존
admission:  x_d ← Benefit(d) ∝ E_future_CSD_access(d)     # routing 결과에 의존

즉 L2 상태 → ρ_q → routing → CSD 접근 분포 → admission → L2 상태의 폐루프다. 이는 단순 최적화가 아니라 고정점 문제이고, 빈 L2에서 출발하면 다음 deadlock이 성립한다:

L2 = ∅  ⇒  ρ_q = 0  ∀q  ⇒  모든 query → GPU
         ⇒  어떤 pid도 CSD 내부에서 scoring되지 않음
         ⇒  E_future_CSD_access(d) = 0  ∀d
         ⇒  Benefit(d) = 0  ⇒  admission 영원히 미발동  ⇒  L2 = ∅  (닫힘)

함의(신규): Turn 1의 STATIC-HOT은 여러 admission 정책 중 하나의 baseline이 아니라, 고정점을 non-trivial 지점으로 부트스트랩하는 유일한 진입로다. offline f_train으로 L2를 미리 채워야 첫 query부터 ρ_q > 0이 되고 online admission이 신호를 받는다. 부트스트랩 없는 순수 online 정책(LRU/Benefit-Cost)은 탐색(exploration)을 강제 주입해야 한다:

route_explore(q) = CSD  with prob ε_explore   (residency 신호 생성용)
                 = route(q)  otherwise

ε_explore는 herding(모두 GPU로 쏠려 L2가 굶는 현상)을 깨는 최소 탐색률이다. 이는 Turn 6이 hand-wave한 "minimum residency time / hysteresis"의 근본 원인을 짚는다: 문제는 진동만이 아니라 부트스트랩 부재로 인한 흡수 상태(absorbing state)다.

사실 2 — admission 이득은 PCIe가 아니라 내부 NAND→DRAM 차이이고, read-bound에서만 유효 (Turn 6 정정)

Turn 6의 saved_path_cost(d) = 1/(ε·BW_pcie) − π_d/BW_int_eff는 "d를 GPU로 보내느냐 CSD로 보내느냐(routing)"의 per-byte 절감이지, "d를 L2에 올리느냐 NAND에 두느냐(admission)"의 절감이 아니다. 두 결정을 분리하면:

  • routing 결정은 이미 CSD 경로를 택하는 순간 PCIe를 없앤다 — d가 L2든 NAND든 무관하게 in-device scoring이면 PCIe hop은 0.
  • admission 결정은 이미 CSD로 라우팅된 d의 내부 read를 NAND에서 DRAM으로 옮길 뿐이다.

따라서 정정된 admission 이득은:

Benefit_admit(d) = E_future_CSD(d) · π_d · bytes_comp(d) · ( 1/BW_NAND − 1/BW_DRAM ) · 𝟙[read-bound]
Cost_admit(d)    = π_d · bytes_comp(d) / BW_admit  (fill write)
                 + eviction_cost(d) + ζ_contention(d)          # Turn 3·5의 write 경합
Admit if  Benefit_admit(d) / Cost_admit(d) > θ

핵심 gating(신규): 𝟙[read-bound]가 결정적이다. CSD가 compute-bound(T_mac ≥ T_read, Turn 5의 LUT datapath가 큰 배치에서 그러함)이면 T_csd = max(T_read, T_mac) = T_mac이므로 read를 DRAM으로 빨라지게 해도 임계경로가 안 줄어 admission 이득 = 0이다. 즉 Turn 6의 "hot 문서를 pin" 논리는 read-bound CSD에서만 성립하고, LUT-bound·MAC-bound CSD에서는 L2 예산을 write 경합만 늘리며 낭비한다. 이는 사실 3(offload가 해방하는 자원)과 정합한다: admission은 내부 read 대역폭이 binding일 때만 지렛대다.

사실 3 — offload가 해방하는 것은 GPU FLOPs가 아니라 대역폭 층이다 (Turn 6 λ_gpu 정정)

Turn 6의 objective는 λ_gpu · GPU_MaxSim_time_x를 단일 자원으로 뒀지만, MaxSim은 AI = 2Q/b_e = 32 FLOP/B < ridge 153이라 항상 memory-bound다. GPU가 MaxSim에 실제로 쓰는 자원을 층별로 분해하면(N=4096, research B sanity-check 수치 기준):

| 자원 | MaxSim 점유 | offload가 해방하는가 | |---|---|:--:| | GPU SM FLOPs | HBM-resident 시 ~20% (=32/153), non-resident 시 ~0% (SM은 PCIe 대기로 idle) | ✗ 거의 못 함 | | HBM BW | resident 시 T_comp≈67µs 동안 ~100% | ✅ (resident 시) | | PCIe BW | non-resident 시 전송 동안 ~100% | ✅✅ | | host DRAM BW | gather/copy read | ✅ |

→ 역설적 결론: CSD offload가 이기는 모든 regime(PCIe-bound)에서 GPU SM은 애초에 idle이라 해방되는 GPU 연산 능력은 무의미하고, GPU SM이 바쁜 유일한 regime(HBM-resident, ~20% FLOPs·100% HBM)에서는 CSD가 진다(Turn 1 산출물 4). 즉 CSD offload는 GPU 연산(FLOPs)을 유의미하게 비워주지 않는다 — 데이터 경로 대역폭(PCIe·host-DRAM·HBM)을 비워준다.

정정된 objective의 자원 항:

λ_gpu·GPU_MaxSim_time   →   λ_hbm·HBM_bytes_x + λ_pcie·PCIe_bytes_x + λ_hostdram·hostDRAM_bytes_x

운영 함의(신규): GPU가 compute-bound reranker(cross-encoder, ~수 ms/query)와 공유되면, MaxSim은 그 reranker의 FLOPs를 거의 안 뺏고 HBM BW만 조금 쓴다. 따라서 "GPU가 reranker와 공유되니 CSD로 offload해 GPU를 비운다"는 Turn 6의 λ_gpu 논거는 reranker가 HBM-bound일 때만 성립하고, 대개는 성립하지 않는다. CSD의 진짜 가치는 여전히 PCIe·host-DRAM 압력 완화(Mode B의 λ_pcie)에 있다.

더 구체화한 부분

Load-adaptive CSD activation: overflow 라우팅 닫힌형 (pressure valve 정량화)

Turn 6의 "pressure valve"를 QPS 함수로 확정한다. GPU-라우팅 query만 PCIe를 쓰고 CSD-라우팅은 ≈0이므로, PCIe 안정 조건 λ_gpu · E[bytes] ≤ β·ε·BW_pcie에서:

λ_switch      = β·ε·BW_pcie / E[bytes_H2_comp]        # PCIe-only 임계 QPS
f_csd*(λ)     = max(0, 1 − λ_switch/λ)                # 안정성만으로 CSD가 흡수해야 할 최소 비율

수치(β=0.7, ε·BW=25 GB/s, bytes=0.6 MB): λ_switch ≈ 29.2k QPS. → λ=50k에서 f_csd* ≈ 0.42(latency와 무관하게 42%는 CSD로 가야 PCIe가 안 터진다), λ=100k에서 f_csd* ≈ 0.71. 이 하한은 latency-first(Mode A)와 독립인 resource-first(Mode B) 바닥선이다.

λ ≈ λ_switch 근방에서 라우팅이 GPU↔CSD로 flapping하지 않도록 이력(hysteresis) 밴드:

λ ↑ λ_on  (= 1.05·λ_switch) 넘으면 CSD 활성화 시작
λ ↓ λ_off (= 0.90·λ_switch) 밑으로 떨어져야 비활성화
+ 최소 활성 dwell time T_dwell (사실 1의 흡수상태·L2 warm-up 보호)

per-task tail penalty 정정: P99_x → 초과확률/CVaR

Turn 6의 max(0, P99_x − SLO)는 P99가 스트림 전체의 분위수라 단일 task x에 정의되지 않는다. task의 자체 완료시간 분포 F_x(큐 대기 + block-packing 분산 + shard tail로 생기는)로 대체:

tail_penalty(x) = λ_tail · P[ C_x > SLO ]            # 초과확률, 또는
                = λ_tail · CVaR_α(C_x)               # 분포 꼬리 평균 (α=0.99)
C_x = queue_delay + T_fix + max_s( read_s, comp_s ) + T_merge   (분포로 취급)

이렇게 하면 tail_risk가 라우팅 argmin에 미분 가능한 스칼라로 들어가고, Turn 6의 tail_risk_penalty(dev, task)가 비로소 well-defined해진다.

content-aware batching: MinHash는 Jaccard지 containment가 아니다 (Turn 6 미세 정정)

Turn 6의 estimated_overlap = MinHash(Blocks(H2(q)), sig(G))는 Jaccard를 추정하지만, 배치에 q가 새로 추가하는 block = |Blocks(q) \ Blocks(G)| = |Blocks(q)|·(1 − containment(q→G))는 containment에 좌우된다. |Blocks(q)| ≪ |Blocks(G)|(작은 query가 큰 배치에 합류)일 때 Jaccard는 overlap을 과소평가한다. LSH-Ensemble류 containment 추정치를 써야 배치 이득이 정확히 예측된다:

marginal_read(q|G) ≈ |Blocks(q)| · (1 − containment_hat(q → G)) · s_block
batch_score(q,G)   = −marginal_read(q|G) − α_wait·wait(q) − α_tail·imbalance(q,G)

L2 cold-start warm-up 비용 (volatile DRAM의 숨은 항)

CSD DRAM(L2)은 휘발성이므로 power-cycle이나 resident-set 재계획(Turn 2의 periodic rebalancing) 때마다 NAND→DRAM 재충전이 필요하다:

T_warmup(B) = B / BW_NAND       # B=8GB, BW_NAND=12GB/s → ~0.67s

이 0.67초 동안 serving read BW가 잠식되어 p99 spike가 발생한다. 재계획 주기 T_epoch(예: 1h)로 상각하면 평균 overhead는 무시 가능(0.67s/3600s ≈ 0.02%)하나, 전이 구간 tail은 사실 1의 T_dwell·hysteresis가 보호해야 할 대상이다. 이는 host page cache(재충전이 OS·백그라운드 DMA)와 다른 CSD 고유 항이다.

관련 맥락 / 배경 설명

  • 왜 결합 고정점이 two-timescale 제어를 요구하는가: 사실 1의 폐루프는 cache-aware load balancing/affinity scheduling의 고전적 herding과 같은 구조다. 표준 안정화는 시간척도 분리다 — routing은 fast timescale(query별, ρ_q 순간값 반응), admission은 slow timescale(EWMA E_future, epoch 재계획). fast loop이 slow loop의 상태를 "거의 상수"로 보면 진동이 감쇠한다. Turn 6의 hysteresis/minimum-residency는 사실상 이 시간척도 분리의 임시 구현이며, 원리적으로는 admission을 stochastic approximation(Robbins-Monro, step size ↓)로 두면 수렴 보장을 논할 수 있다.
  • 자원 층 분해가 본질인 이유: 사실 3은 "GPU offload"라는 단어가 오도한다는 점을 드러낸다. MaxSim은 systolic FLOPs를 원하는 workload가 아니라 대역폭을 먹는 memory-streaming workload다. 그래서 이 연구의 정확한 프레이밍은 "GPU 연산을 CSD로 옮긴다"가 아니라 "embedding stream을 PCIe 밖(device 내부)에 가둔다"이다. λ 가중치도 연산이 아니라 대역폭 층별로 걸려야 한다.
  • admission gating과 datapath의 결합: 사실 2의 𝟙[read-bound]는 Turn 5의 datapath 선택(fp16-reconstruct vs LUT)과 직결된다. LUT datapath는 배치가 커지면 LUT-read-bound가 되어 admission 이득이 사라지고, fp16-reconstruct는 read-bound가 오래 유지되어 admission이 계속 유효하다. 즉 하드웨어 datapath 선택이 캐시 정책의 가치까지 결정한다 — Turn 5의 "format이 하드웨어를 정한다"의 상위 계층 버전.

  3. (정정) Turn 6 P99_x per-task: 분위수는 task별로 정의 불가 → 초과확률/CVaR. — 정의 정정, 실측 불필요.

  6. 사실 3의 "MaxSim SM FLOPs ~20%"는 A100 fp16 ridge 153·peak 312 TFLOP/s 배경지식 기반 — GPU 세대·tensor core 경로에 따라 다름, 확인 필요.
  7. L2 warm-up T_warmup가 serving과 실제로 얼마나 경합하는지(별도 채널로 background fill 가능한지)는 컨트롤러 아키텍처 실측 필요.

  1. two-timescale 제어의 안정성/수렴 증명 + 시뮬: routing(fast, ρ_q 반응)과 admission(slow, EWMA/stochastic-approx)을 시간척도 분리 제어로 정식화하고, 부트스트랩(STATIC-HOT) 유무·탐색률 ε_explore·step size에 대해 herding/deadlock 탈출과 진동 감쇠를 discrete-event 시뮬로 검증하라. 수렴 지표(L2 hit ratio의 정착 시간, 라우팅 flip률)를 정책별로 보고할 것.
  1. 자원-층 원장(ledger) 기반 Mode-B 재평가: query당 {SM-FLOPs, HBM-BW, PCIe-BW, host-DRAM-BW, CSD-internal-BW} 5층 소비를 정책(all-GPU / shard-hybrid / batched-CSD)별로 기록하는 원장을 만들고, GPU에 compute-bound cross-encoder reranker를 co-locate한 시나리오에서 CSD offload가 실제로 reranker throughput을 올리는지(즉 λ_gpu 가정이 성립하는지)를 측정하라. 사실 3의 "FLOPs는 안 비워진다"를 실험으로 확정/반증할 것.
  1. load-adaptive activation의 diurnal 시뮬: 하루 QPS 곡선(λ_switch를 오르내리는)에서 f_csd*(λ) = max(0, 1−λ_switch/λ) overflow 라우팅을 static split·all-GPU와 비교하고, λ_switch 교차점 p99 spike와 L2 cold-start warm-up 전이를 관측하라. flapping을 최소화하는 hysteresis 밴드 (λ_on, λ_off)와 T_dwell을 sweep해, baseline별(host-gather/GDS/L0) λ_switch 차이(확인점 4)가 스위칭 정책을 어떻게 바꾸는지 규명할 것.

Turn 8 / Codex

이번 턴의 확장 요약

이번 턴은 scheduler가 쓰는 예측값의 불확실성, CSD 내부 early-exit / cancellation, 그리고 PCIe·NVMe control-plane 병목을 추가한다. Turn 7은 admission-routing 고정점과 resource ledger를 잘 정리했지만, runtime scheduler가 실제로 실패하는 지점은 평균 cost model보다 prediction error와 deadline violation에 있다.

핵심 신규 결론은 다음이다.

CSD scheduler는
  predicted mean latency가 아니라
  deadline-aware upper-bound latency와 cancellation 가능한 작업 단위
를 기준으로 결정해야 한다.

또한 PCIe pressure valve 식은 byte bandwidth만 보면 부족하고, NVMe command rate / submission queue / completion queue / PCIe switch 공유까지 포함한 control-plane 안정 조건으로 확장해야 한다.

새로 추가한 사실

1. Scheduler는 point estimate가 아니라 uncertainty-aware decision을 해야 한다

기존 routing rule은 predicted_service_time 하나로 장치를 고른다. 하지만 CSD 경로는 다음 항들의 분산이 크다.

T_csd =
  T_queue
  + T_ctrl
  + max(T_read, T_score)
  + T_merge

특히 T_queue, T_ctrl, π, BRF_block, shard imbalance는 query마다 예측 오차가 크다. 따라서 latency-first 모드에서는 평균 예측값보다 upper confidence bound가 맞다.

UCB_csd(q) = μ_csd(q) + z_slo · σ_csd(q)
UCB_gpu(q) = μ_gpu(q) + z_slo · σ_gpu(q)

route to CSD only if:
  UCB_csd(q) + slack_guard < deadline(q)
  and UCB_csd(q) < UCB_gpu(q) + resource_credit(q)

여기서 resource_credit(q)는 PCIe 절감이나 host-DRAM 절감을 latency 손해로 환산한 값이다. 즉 resource-first 모드에서는 CSD가 약간 느려도 보낼 수 있지만, SLO slack 안에서만 허용한다.

추가 trace 필드:

prediction_log(
  q_idx, task_id, device,
  pred_mean_us, pred_std_us, pred_p95_us,
  actual_us,
  error_us,
  deadline_us,
  violated
)

이 필드가 있어야 scheduler가 단순히 “맞았는지”가 아니라 어느 항의 분산 때문에 p99가 깨졌는지를 학습할 수 있다.

2. CSD MaxSim은 query-level cancellation을 지원해야 tail을 줄일 수 있다

CSD 작업은 한번 시작하면 끝까지 도는 것으로 모델링됐지만, runtime 관점에서는 cancellation이 중요하다. GPU나 다른 CSD shard가 충분히 좋은 top-k threshold를 먼저 만들면, 느린 CSD shard의 남은 candidate 중 global top-k에 들어갈 가능성이 낮은 작업을 중단할 수 있다.

문서 또는 block 단위 upper bound를 둔다.

UB(q, d) = approximate_score(q, d) + residual_error_bound(d)

skip or cancel d if:
  UB(q, d) < current_global_topk_threshold

batch 내부에서는 block 단위로 더 자연스럽다.

UB(q, block) = max_{d in block} UB(q, d)

skip block for q if:
  UB(q, block) < τ_k(q)

이 방식은 두 가지 효과가 있다.

internal_read_saved  = cancelled_block_bytes
score_ops_saved      = cancelled_q_doc_pairs · Q · D · E
tail_reduction       = slow shard의 remaining work 절단

즉 CSD offload는 단순 streaming accelerator가 아니라, host가 threshold를 갱신하고 CSD가 남은 block을 취소할 수 있는 cooperative runtime이어야 한다.

3. λ_switch는 PCIe byte bandwidth뿐 아니라 command-plane cap도 포함해야 한다

Turn 7의 load-adaptive activation 식은 byte bandwidth 기준이었다.

λ_switch = β · ε · BW_pcie / E[bytes_H2]

하지만 CSD 경로에서는 embedding bytes는 줄어도 control command가 늘어난다. 특히 block-list descriptor를 여러 NVMe command로 쪼개면 command rate가 병목이 된다.

확장된 안정 조건:

λ_gpu · E[bytes_gpu]      ≤ β_bw  · BW_pcie
λ_csd · E[bytes_ctrl]     ≤ β_down · BW_pcie_down
λ_csd · E[n_cmd_csd]      ≤ β_cmd · IOPS_cmd
λ_csd · E[cq_entries]     ≤ β_cq  · CQ_rate

따라서 실제 switch point는 다음의 최소값이다.

λ_switch = min(
  β_bw  · BW_pcie / E[bytes_gpu],
  β_cmd · IOPS_cmd / E[n_cmd_gpu_or_csd],
  β_cq  · CQ_rate / E[cq_entries]
)

만약 CSD가 block 하나당 command 하나를 요구하면 n_cmd_csd ≈ |U_block|가 되어 byte 절감에도 불구하고 control-plane이 먼저 터진다. 그래서 Turn 5의 batched descriptor는 최적화가 아니라 CSD runtime의 필수 인터페이스 조건에 가깝다.

더 구체화한 부분

Deadline-aware batching

content-aware batching은 BRF_block을 올리지만 대기시간을 늘린다. 따라서 batch 구성은 FIFO나 overlap 최대화가 아니라 deadline 제약이 붙은 packing 문제다.

maximize   BRF_block(G) − α · imbalance(G)
subject to now + wait_q + service_p95(G) ≤ deadline_q   for all q in G
           SRAM_query(G) ≤ SRAM_budget
           n_cmd(G) ≤ cmd_budget

실용적 heuristic:

def admit_to_batch(q, open_batches):
    feasible = [
        G for G in open_batches
        if predicted_p95(G + q) <= min_deadline(G + q)
        and sram_bytes(G + q) <= SRAM_budget
        and command_count(G + q) <= CMD_budget
    ]

    if not feasible:
        return start_new_batch(q)

    return argmax_G(
        delta_brf_block(G, q)
        - alpha_wait * added_wait(G, q)
        - alpha_tail * added_imbalance(G, q)
    )

이렇게 해야 S_burst에서는 overlap 이득을 얻고, S_uniq처럼 overlap이 약한 경우에는 불필요한 batching으로 p99를 망치지 않는다.

Cancellation-aware service model

기존 모델:

S_s(G) = T_fix + T_ctrl + max(Read_s(G), Compute_s(G)) + Result_s(G)

확장 모델:

S_s(G)
  = T_fix
  + T_ctrl
  + max(
      Read_s(G)    · (1 − cancel_read_frac_s),
      Compute_s(G) · (1 − cancel_compute_frac_s)
    )
  + T_cancel_protocol
  + Result_s(G)

T_cancel_protocol은 host threshold update와 CSD stop signal 비용이다. cancellation이 이득이려면 다음이 필요하다.

saved_read_or_compute_time > T_cancel_protocol + pipeline_flush_cost

따라서 cancellation은 작은 query보다 large-H2, flat queue, slow-shard query에서 더 유리하다.

Score-margin scheduling feature

query별로 pruning 이후 score margin을 feature로 남긴다.

margin_k(q) = approx_score_k(q) − approx_score_{k+r}(q)
ub_gap(q)   = topk_threshold(q) − max_remaining_UB(q)

예상 효과:

large margin  → early exit 가능성 큼 → CSD 유리
small margin  → 거의 끝까지 scoring 필요 → GPU 또는 full CSD scan 필요

즉 ρ_bytes, π, BRF_block에 더해 score distribution shape이 scheduling factor가 된다.

지금까지의 논리는 주로 “어떤 bytes를 어디서 읽는가”였다. 하지만 production runtime에서는 “끝까지 읽어야 하는가”도 같은 수준으로 중요하다. PLAID 계열 검색은 이미 approximate score와 pruning threshold를 갖고 있으므로, CSD는 그 정보를 단순히 candidate list 생성에만 쓰지 말고 남은 work를 취소하는 signal로 써야 한다.

이 관점에서 CSD offload는 세 계층이 결합된 runtime이다.

data placement:     L2 residency, layout, replication
work formation:     H2 planning, batching, shard assignment
work termination:   threshold update, early exit, cancellation

앞의 두 계층만 있으면 평균 throughput 모델은 만들 수 있지만, p99를 설명하려면 세 번째 계층이 필요하다.

이번 턴의 확장 요약

이번 턴은 Turn 8의 세 축(uncertainty-aware routing, cancellation, control-plane cap)을 exactness-안전한 형태로 완성하고, 지금까지 8턴이 한 번도 다루지 않은 에너지/TCO break-even과 end-to-end Amdahl 상한을 새 축으로 넣는다.

핵심 신규 결론 다섯 가지:

  1. Turn 8의 cancellation은 "block을 읽은 뒤 취소"였지만, 진짜 레버는 residual을 읽기 전에 centroid-UB로 취소하는 것이다 — 이것이 binding resource(내부 read)를 직접 절감하고, PLAID의 stage-1/2 centroid pruning이 사실 residual-norm slack Q·δ_max를 뺀 lossy 버전의 안전 UB임을 드러낸다.
  2. Turn 8 확인점 5(margin calibration)를 calibrated recall margin ξ_δ로 완성해 "취소로 얻는 절감 ↔ recall 손실" 곡선을 수식화한다.
  3. 에너지 break-even은 latency break-even과 다른 부등식이고 CSD에 훨씬 관대하다 — Turn 7이 "단일 CSD는 warm host에 진다"고 한 read-bound regime에서도 CSD가 Joule/query로는 이긴다. 새 regime이 추가된다.
  4. Turn 8의 UCB = μ + z_slo·σ는 상수 z가 틀렸다. tail은 shard-max의 극단값이라 유효 계수가 √(2 ln m)로 커지고, 공유 BW_int가 공통모드 분산을 주입한다.
  5. Amdahl 상한: CSD가 MaxSim transfer를 0으로 만들어도 end-to-end speedup은 encoder+candgen 비중 때문에 1/(1−φ)(추정 ~2.8×)로 묶인다 — 단 이 상한은 Mode A(latency)에만 걸리고 Mode B(PCIe 압력)에는 안 걸린다.

새로 추가한 사실

사실 1 — Centroid-UB read-avoiding cancellation: residual을 읽기 전에 취소 (Turn 8 정정·완성)

Turn 8은 UB(q,block) < τ_k면 block을 skip한다고 했지만, UB를 block을 읽은 뒤의 실제 score로 잡았다(max_{d in block} approximate_score). 그러면 이미 read를 소모한 뒤이고, Turn 8 확인점 3("block에 high-score 하나만 있어도 통째로 읽어야")이 바로 그 한계다. 진짜 레버는 resident한 centroid 데이터만으로, residual read 이전에 상한을 계산하는 것이다.

MaxSim S(q,d) = Σ_i max_{j∈d} ⟨q_i, x_{dj}⟩에서 각 token을 x_{dj} = centroid_{c_j} + r_{dj}로 분해하면(Turn 4·5의 분해):

⟨q_i, x_{dj}⟩ = S_cent[i, c_j] + ⟨q_i, r_{dj}⟩
             ≤ S_cent[i, c_j] + ||q_i||·||r_{dj}||        (Cauchy-Schwarz)
             ≤ S_cent[i, c_j] + ||q_i||·δ_max             (δ_max = 양자화가 보장하는 residual norm 상한)

δ_max 항은 j에 무관한 상수이므로 문서 상한이 닫힌형으로 나온다:

UB_cent(q,d) = Σ_i max_{j∈d} S_cent[i, c_j]  +  δ_max · Σ_i ||q_i||
                └─ centroid id(=c_j)만 필요 ─┘     └─ 쿼리별 상수(정규화 시 = Q·δ_max) ─┘

핵심(신규): 첫 항은 d의 token들의 centroid 배정(IVF/doclens에 압축 저장, resident) 만으로 계산된다 → residual block을 한 byte도 읽지 않고 계산 가능. 따라서

if  UB_cent(q,d) < τ_k(q):   d의 residual block read를 아예 생략   → 내부 read 절감(binding resource)

이는 Turn 8의 compute-절감(scoring skip)보다 한 단계 강하다 — Turn 7 사실 2의 결론(binding resource는 내부 read)에 정확히 맞는 절감이다.

PLAID와의 관계(재프레이밍): PLAID의 stage-1/2 centroid pruning은 Σ_i max_j S_cent[i,c_j]만으로 후보를 거른다 — 즉 δ_max·Σ||q_i|| slack을 뺀 것이라 원리적으로 lossy(approximate)다. slack을 더하면 그 filter가 exactness-safe cancellation 상한으로 승격된다. 따라서 "얼마나 공격적으로 취소하느냐"는 곧 slack을 δ_max에서 얼마나 깎느냐의 knob이고, δ_max는 2-bit 양자화의 per-dimension 오차로 직접 결정된다(코드 확인 필요). 이는 Turn 5의 two-round pruning recall 문제도 통합한다: local pruning = local UB_cent 적용, recall 보장 = slack 크기의 함수(이항 over-provision과 별개 축).

사실 2 — 에너지/TCO break-even은 latency break-even과 다른 부등식이고 CSD에 관대하다

지금까지 8턴은 전부 latency·bandwidth였다. 그러나 데이터센터 CSD의 1차 동기는 흔히 pJ/byte(전력·TCO)다. GPU-HG 경로는 residual byte가 PCIe 1회 + host DRAM(write+read) + HBM(write+read) 를 건너지만, CSD 경로는 NAND read 1회 + on-chip SRAM/register에서 끝난다. per-byte movement 에너지:

E_move,gpu ≈ b · ( e_nand + e_pcie + 2·e_ddr + 2·e_hbm )
E_move,csd ≈ b · ( e_nand + e_sram_local )
CSD가 없애는 증분 = e_pcie + 2·e_ddr + 2·e_hbm  (per byte)

배경지식 상수(모두 확인 필요, 세대 의존): e_pcie ≈ 5 pJ/bit, e_ddr ≈ 10–20 pJ/bit, e_hbm ≈ 3–4 pJ/bit, e_sram ≈ 0.1 pJ/bit, fp16 MAC ≈ 1 pJ.

CSD가 없애는 증분 ≈ (5 + 2·15 + 2·4) pJ/bit ≈ 43 pJ/bit ≈ 344 pJ/byte
0.6 MB/query → ~0.2 mJ/query 절감  → 30k QPS면 movement 차이만 ~6 W
반면 CSD compute 추가비용 = MAC/byte · e_MAC = (Q/b_e)·1 pJ ≈ 16 pJ/byte  ≪ 344 pJ/byte

핵심(신규 regime): latency break-even은 BW_int ≥ ε·BW_pcie(Turn 4)라는 어려운 부등식을 요구하지만, 에너지 break-even은 e_csd_compute < e_pcie + DRAM 왕복이라는 거의 항상 참인 부등식이다(compute 16 pJ vs movement 344 pJ). 따라서 HBM-resident가 아닌 모든 regime에서 CSD가 에너지로는 이긴다 — Turn 7이 "단일 CSD는 warm host DRAM에 latency로 진다"고 한 read-bound regime조차 Joule/query·TCO로는 CSD 우세다. Turn 1의 regime 표(산출물 4)에 "latency loss / energy win" 칸을 추가해야 한다. 이는 CSD 논거를 "더 빠르다"에서 "같은 latency budget 안에서 pJ/query·서버당 W를 줄인다"로 넓히는, 지금까지 빠졌던 축이다.

사실 3 — Amdahl 상한: MaxSim transfer를 0으로 만들어도 end-to-end speedup은 1/(1−φ)

지금까지 모든 break-even은 MaxSim 경로 내부만 비교했다. 그러나 query 응답 전체는:

T_e2e = T_encode(BERT, GPU 전용, Turn 7 사실3) + T_candgen(stage1·2) + T_maxsim_path(stage3 data-move+score) + T_misc

CSD가 줄이는 것은 T_maxsim_path뿐이다. 절감 비중 φ = ΔT_maxsim / T_e2e이면 Mode A latency speedup ≤ 1/(1−φ).

research sanity-check 수치로 추정: T_maxsim_path(GPU-A) ≈ 5.4 ms, T_encode+candgen ≈ 3 ms(배경지식, 확인 필요) → φ ≈ 5.4/8.4 ≈ 0.64, 이상적 상한 ≈ 2.8×. CSD가 T_maxsim_path를 5.4→1.7 ms로 줄여도 실현 speedup = 8.4/(3+1.7) ≈ 1.79×. 즉 단일 query latency 이득은 애초에 <2×로 묶인다.

함의(신규):

  • Mode A(latency-first)에서 CSD의 가치는 Amdahl로 상한이 있으므로, encoder/candgen이 지배적인 워크로드(짧은 H2, 작은 corpus)에서는 CSD offload를 안 하는 게 맞다 — 이는 Turn 8의 UCB에 resource_credit만으로는 안 잡히는, query-구조적 gate다.
  • 반대로 Mode B(PCIe·에너지)에는 Amdahl이 안 걸린다: PCIe traffic 절감 = HR_bytes(Turn 1)이고 이는 encoder 시간과 무관하다. 따라서 "CSD는 latency device가 아니라 throughput/에너지 device"라는 Turn 7 프레이밍이 Amdahl로 다시 확증된다. routing objective에 φ_q(query별 MaxSim 비중)를 feature로 추가해야 Mode A에서 CSD 낭비를 막을 수 있다.

더 구체화한 부분

Cancellation-safe threshold: calibrated recall margin (Turn 8 확인점 5 완성)

Turn 8은 margin_k(q)가 calibration에 민감하다고만 지적했다. 안전 조건을 수식화한다. 취소에 쓰는 threshold는 k번째 true score의 하한(보수적) 이어야 false-negative가 안 난다. approximate score 오차 ε_d = S_approx(q,d) − S_exact(q,d)의 (1−δ) 분위수를 ξ_δ라 하면:

τ_k^safe(q) = τ_k^approx(q) − ξ_δ          # threshold를 ξ_δ만큼 낮춰 보수화
cancel d  ⟺  UB_cent(q,d) < τ_k^safe(q)
⇒ recall ≥ 1 − δ   (오차 꼬리를 ξ_δ가 덮으므로)

절감–recall 트레이드는 명시적이다:

saved_read_frac(δ) = Pr_d[ UB_cent(q,d) < τ_k^approx − ξ_δ ]      # δ↑(ξ_δ↓) → 더 많이 취소
recall(δ)          = 1 − δ

→ δ를 sweep해 "내부 read 절감 vs recall 손실" 파레토를 그린다. ξ_δ는 PLAID approx score와 full-residual MaxSim의 오차 분포 실측으로 calibrate(Turn 5 확인점 4·Turn 8 확인점 5의 통합). 2-bit 양자화 오차가 작으면 ξ_δ가 작아 취소가 공격적일 수 있다 — 이것이 사실 1의 δ_max slack과 같은 물리량의 두 얼굴이다(UB 쪽은 상한, threshold 쪽은 calibrated 하한).

Shard-max tail: UCB의 z는 상수가 아니라 √(2 ln m) (Turn 8 정정)

Turn 8의 UCB_csd = μ + z_slo·σ는 shard가 하나일 때만 맞다. query latency는 T_query = max_s T_s + T_merge(Turn 2·4)라 m개 shard의 극단값이다. iid 근사 T_s ~ N(μ_s, σ_s²)이면:

E[max_s T_s] ≈ μ_s + σ_s·√(2 ln m)
p99(max) 유효계수 z_eff(m) ≈ √(2 ln m) + z_0.99/√(2 ln m)
  m=8  → √(2 ln 8)  ≈ 2.04
  m=64 → √(2 ln 64) ≈ 2.88

→ shard를 8→64로 잘게 나누면 tail 계수가 ~2.0→2.9로 커진다. 즉 tight SLO에서는 fine sharding이 tail을 악화시키므로 latency-critical query는 touch하는 shard 수를 줄이는 쪽(whole-query 또는 hot-affinity 소수 shard) 이 유리하다 — Turn 4·5 sharding 3자 트레이드에 "tail-vs-#shard"라는 정량 축을 붙인다.

상관 보정(신규): shard들은 공유 BW_int와 admission-write 경합 ζ(Turn 3·5)로 양의 상관이다. 공통모드 성분 σ_common(전역 BW dip은 모든 shard를 같이 밀어냄)은 평균으로 상쇄되지 않는다:

z_eff·σ  →  √(2 ln m_eff)·σ_idio + z·σ_common,   m_eff ≈ 1 + (m−1)(1−ρ_corr)

→ Turn 8의 독립 UCB는 σ_common을 놓쳐 p99를 낙관적으로 과소평가한다. routing은 shard-idiosyncratic 분산과 공통모드 분산을 분리 추정해야 한다.

Long-running CS command의 head-of-line blocking (Turn 8 command-plane 확장)

Turn 8·Turn 5는 "block-list를 단일 CS command로 → n_cmd = O(1)"을 command-rate 병목의 해법으로 제시했다. 그러나 긴 CS command는 NVMe queue slot을 서비스 시간 S_cmd 동안 점유한다. queue depth QD이면 동시성 상한 = QD/S_cmd commands/s:

S_cmd(batch scan) ≈ 5 ms,  QD = 64  →  최대 12.8k long-cmd/s
이 값 < QPS 이면 SQ 포화 → 작은 명령(latency-critical query descriptor)이 큰 배치 scan 뒤에 HoL blocking

→ n_cmd = O(1)은 command rate에는 좋지만 HoL/priority에는 나쁘다. 올바른 설계는 (a) 배치당 O(1) command + (b) 다중 우선순위 NVMe queue(가중 라운드로빈 arbitration) + (c) 긴 scan을 preemptible/cancellable chunk로 분할(사실 1의 취소와 co-design)이다. 즉 command-plane 병목과 cancellation은 같이 설계해야 한다 — 긴 command를 쪼개면 취소 지점이 생기고, 취소를 지원하면 HoL도 완화된다.

관련 맥락 / 배경 설명

  • Late-interaction cancellation = block-max WAND의 MaxSim 판: 사실 1의 UB_cent로 하는 조기 종료는 IR의 threshold algorithm(Fagin TA/NRA)·block-max WAND(BMW)와 동형이다. BMW는 posting block의 max impact로 문서를 skip하고, 여기서는 centroid-max score가 그 "block-max impact" 역할을 한다. 차이는 aggregation이 Σ_i max_j(late interaction)라 term별 상한을 합쳐야 한다는 점 — 그래서 Σ_i max_{j∈d} S_cent[i,c_j] 형태가 자연스럽게 나온다. 이 연결은 "CSD가 무엇을 계산 안 해도 되는가"를 40년 된 top-k 문헌으로 정당화해준다.
  • 에너지가 진짜 데이터센터 driver인 이유: 사실 2는 이 연구의 서사를 바꾼다. HBM 80 GB에 안 들어가는 fp16 full index(~150 GB)를 CSD로 밀면 capacity(Turn 1 산출물4) 뿐 아니라 movement 에너지를 산다 — GPU 경로의 PCIe+2 DRAM 왕복이 byte당 compute의 20배 에너지이기 때문. latency가 지는 regime에서도 W/서버·pJ/query가 이기면 TCO 논거로 채택된다. 즉 CSD 제안의 방어선은 latency → capacity → 에너지/TCO 순으로 물러설수록 넓어진다.
  • Amdahl이 scheduling factor를 하나 더 만든다: 사실 3은 routing이 봐야 할 feature에 φ_q(query별 MaxSim 시간 비중)를 추가한다. ρ_q(residency)·π·BRF_block·margin_k에 더해, encoder가 지배적인 query는 CSD로 보내도 latency가 안 줄어드는 구조적 gate다. Mode A에서 이 gate가 없으면 CSD는 "이겨봐야 1.79×"인 query에 자원을 낭비한다.

Turn 10 / Codex

이번 턴의 확장 요약

이번 턴은 runtime 내부 구조를 실제 구현 단위로 쪼개고, scheduler를 고정 가중치 정책이 아니라 온라인 자원 가격(resource price)을 학습하는 제어기로 확장한다. 핵심은 CSD MaxSim offload가 단일 kernel offload가 아니라 다음 네 상태를 계속 조정하는 runtime이라는 점이다.

query arrival
→ candidate plan 생성
→ batch/block/shard task 형성
→ GPU 또는 CSD 실행
→ partial top-k merge / cancellation / feedback
→ routing·admission·batching policy 업데이트

새 결론은 세 가지다.

  1. CSD runtime에는 score ownership, version consistency, monotone threshold라는 정확성 invariant가 필요하다.
  2. scheduler의 λ_pcie, λ_tail, λ_csd는 고정 상수가 아니라, 관측 utilization으로 갱신되는 dual variable로 두는 편이 낫다.
  3. tail latency는 cancellation만으로는 부족하고, 일부 query에는 speculative GPU fallback / hedged execution이 필요하다.

---

새로 추가한 사실

1. Runtime 정확성 invariant가 없으면 scheduling이 top-k exactness를 깨뜨린다

지금까지는 latency와 bandwidth 중심이었지만, runtime 내부에서는 다음 invariant가 필요하다.

Invariant A: score ownership
  하나의 pid에 대한 full MaxSim score는 정확히 하나의 execution owner가 완성한다.
  owner ∈ {GPU, CSD_s}

Invariant B: version consistency
  query q가 보는 doc embedding, centroid id, residual block, tombstone bitmap은 같은 index_epoch에 속해야 한다.

Invariant C: monotone threshold
  τ_k(q)는 시간이 지날수록 증가만 해야 하며,
  cancellation은 항상 보수적 UB와 현재 τ_k 이하에서만 수행한다.

Invariant D: idempotent cancellation
  CSD task가 cancel된 뒤 늦게 completion을 보내도 host merge가 중복 반영하지 않아야 한다.

이 중 가장 위험한 것은 score ownership이다. shard-hybrid routing에서 같은 pid의 token 일부를 GPU가, 일부를 CSD가 scoring하면 max_j가 깨질 수 있다. 따라서 task granularity는 다음 중 하나로 고정해야 한다.

safe task unit = whole pid
              or contiguous doc-token span containing a complete pid
unsafe unit    = arbitrary residual block if block boundary cuts a pid

즉 block-level scheduling은 가능하지만, block이 pid를 쪼개는 layout이면 score ownership metadata가 추가로 필요하다.

---

2. 고정 λ scheduler보다 online dual-price scheduler가 더 자연스럽다

Turn 6의 목적 함수는 λ_tail, λ_pcie, λ_csd를 외부에서 정하는 형태였다. 하지만 실제 운영에서는 PCIe, CSD internal BW, command queue의 혼잡도가 시간에 따라 바뀐다. 따라서 λ를 고정하지 않고 resource price로 갱신한다.

자원 제약:

E[PCIe_bytes/s]      ≤ C_pcie
E[CSD_read_bytes/s]  ≤ C_int
E[NVMe_cmd/s]        ≤ C_cmd
P[latency > SLO]     ≤ δ_slo

라우팅 비용:

cost(dev, task)
  = predicted_latency(dev, task)
  + μ_pcie · bytes_pcie(dev, task)
  + μ_int  · bytes_internal(dev, task)
  + μ_cmd  · n_cmd(dev, task)
  + μ_slo  · violation_risk(dev, task)

dual update:

μ_r(t+1) = max(0, μ_r(t) + η · (usage_r(t) − target_r))

해석:

  • PCIe가 붐비면 μ_pcie↑ → CSD 선택이 늘어남
  • CSD NAND가 붐비면 μ_int↑ → GPU 또는 L2-hit query만 CSD로 감
  • command queue가 막히면 μ_cmd↑ → 작은 query의 CSD offload가 줄어듦
  • SLO violation이 늘면 μ_slo↑ → p95/p99 안전한 GPU fallback 증가

이 방식은 f_csd*(λ) 같은 닫힌형 threshold보다 일반적이다. 특히 byte bandwidth, command-plane, tail-risk가 서로 다른 시점에 병목이 될 때 자동으로 policy가 이동한다.

---

3. Hedged execution은 p99를 줄이지만 PCIe 절감률을 갉아먹는다

CSD tail이 큰 query는 처음부터 GPU와 CSD를 동시에 일부 실행하고, 먼저 충분한 top-k threshold를 만든 쪽을 기준으로 다른 쪽을 cancel할 수 있다.

hedge if:
  P[T_csd > deadline] · penalty_slo
  >
  extra_GPU_bytes / BW_pcie_price
  + wasted_CSD_work

구체적 정책:

normal:
  q → CSD only

hedged:
  q → CSD full task
  q → GPU fallback for high-risk shard subset or first M candidates
  first useful τ_k wins
  loser is cancelled

이 정책은 latency-first Mode A에서는 강하지만, resource-first Mode B에서는 조심해야 한다. hedging은 CSD의 핵심 장점인 PCIe traffic reduction을 일부 되돌리기 때문이다.

추가 trace 필드:

hedge_events(
  q_idx, task_id,
  primary_device, hedge_device,
  hedge_delay_us,
  cancelled_device,
  wasted_internal_bytes,
  wasted_pcie_bytes,
  latency_saved_us
)

Index update와 tombstone 비용

Retrieval index는 완전 static이 아닐 수 있다. 문서 삭제, passage 추가, embedding 재빌드가 있으면 CSD resident cache와 block layout이 깨진다.

최소 runtime 상태:

index_epoch
block_epoch(block_id)
resident_epoch(pid)
tombstone_bitmap(pid)
delta_index_location(pid)

query 실행 조건:

valid(q, pid) =
  resident_epoch(pid) == query_epoch
  and tombstone_bitmap(pid) == 0

업데이트가 많은 workload에서는 CSD의 장점이 줄어든다.

update_pressure =
  bytes_rewritten_per_sec / BW_admit
  + invalidated_resident_bytes / L2_capacity

따라서 CSD offload가 유리한 regime에 새 조건이 추가된다.

read_dominant workload:
  query_bytes_read_per_sec ≫ update_bytes_written_per_sec

update-heavy workload:
  admission churn + layout rewrite 때문에 CSD 이득 감소

Query tiling이 LUT datapath 병목을 바꾼다

Turn 5는 LUT bank 수가 Q·E에 비례한다고 봤다. 하지만 runtime은 query token을 tile로 나눠 SRAM port 요구를 줄일 수 있다.

Q = 32
Q_tile ∈ {4, 8, 16, 32}

한 token vector를 여러 query-tile에 대해 반복 처리하면:

LUT_ports_required ∝ Q_tile · E
passes_per_doc     = Q / Q_tile

두 구현 선택지가 생긴다.

Option A: full-Q streaming
  read residual once
  high LUT bank pressure
  low read amplification

Option B: Q-tiling
  fewer LUT banks
  either residual re-read or local token buffer 필요

Q-tiling break-even:

extra_read_time(Q_tile)
  < saved_area_or_frequency_gain(Q_tile)

즉 LUT datapath의 병목은 단순히 “1.9K bank 필요”가 아니라, SRAM bank 수와 residual 재사용 buffer 크기의 tradeoff로 다시 써야 한다.

---

관련 맥락 / 배경 설명

이 연구의 runtime은 GPU scheduler와 SSD FTL 사이 어딘가에 있다. GPU 쪽에서는 stream, kernel, HBM pressure를 보고, SSD 쪽에서는 block layout, command queue, internal read bandwidth를 본다. 기존 IR engine은 보통 이 두 계층을 분리하지만, CSD MaxSim에서는 분리하면 손해가 크다.

특히 Planner → Batcher → Router 순서가 고정이면 최적성이 떨어진다. routing을 먼저 하면 CSD batch overlap을 놓치고, batching을 먼저 하면 deadline이 빡빡한 query를 CSD에 묶어 p99를 망칠 수 있다. 따라서 실용 구현은 짧은 window 안에서 다음을 반복하는 형태가 된다.

candidate plan 생성
→ 가능한 batch 후보 몇 개 생성
→ 각 batch의 GPU/CSD cost 예측
→ dual-price로 선택
→ 실행 후 prediction error feedback

이 구조는 “정적 hot-set + threshold routing”보다 복잡하지만, 논문 기여로는 더 강하다. CSD offload의 성패가 평균 latency가 아니라 tail, command-plane, index consistency, admission churn에서 갈리기 때문이다.

잠재적 오류 또는 확인 필요 지점

  1. score ownership invariant는 실제 ColBERT/PLAID index block이 pid 경계를 보존하는지에 의존한다. block이 여러 pid를 포함하는 것은 문제 없지만, 하나의 pid가 여러 block에 걸치면 owner tracking이 필요하다.
  1. dual-price scheduler는 step size η에 민감하다. 너무 크면 GPU/CSD routing이 진동하고, 너무 작으면 diurnal load 변화에 늦게 반응한다.
  1. hedged execution은 p99를 낮출 수 있지만, PCIe와 GPU fallback bytes를 증가시킨다. Mode B에서는 손해일 수 있으므로 별도 ablation이 필요하다.
  1. index update 모델은 MS MARCO 같은 offline benchmark에는 거의 보이지 않는다. 운영형 corpus를 가정한 synthetic update stream이 필요하다.
  1. Q-tiling은 LUT bank 수를 줄이지만 residual read 재사용을 어렵게 만든다. CSD 내부 SRAM 용량과 DMA burst size를 모르면 최적 Q_tile을 확정할 수 없다.

---

다음 에이전트가 이어서 확장할 지점

  1. dual-price scheduler 시뮬레이터 설계: μ_pcie, μ_int, μ_cmd, μ_slo를 online으로 갱신하며 all-GPU / fixed-split / threshold-CSD / dual-price를 p50, p99, PCIe GB/s, command utilization, routing flip률로 비교하라.
  1. correctness-first runtime protocol: score ownership, index_epoch, tombstone, late completion dedupe, idempotent cancel을 포함한 host-CSD message schema와 state machine을 정의하고, cancellation과 hedging이 exact top-k를 깨지 않는 조건을 증명하라.
  1. Q-tiling LUT datapath sweep: Q_tile={4,8,16,32}에 대해 LUT bank 수, residual reread bytes, SRAM buffer 요구량, latency를 계산하고, BW_int=12GB/s와 FPGA SRAM 예산 안에서 read-bound와 LUT-bound 경계가 어디인지 도출하라.

Turn 11 / Claude

이번 턴의 확장 요약

이번 턴은 Turn 10이 세운 runtime 정확성 invariant / dual-price scheduler / hedged execution의 내부 정합성을 파고들어, 그중 두 개를 정정하고 세 개의 새 축(제어 평면 자체의 자원 예산, monotone-CRDT 병합, epoch RCU 용량 페널티)을 추가한다. 핵심 신규 결론 다섯 가지:

  1. Turn 10의 Invariant A(score ownership = whole pid)는 너무 강하다. MaxSim은 Σ_i max_j이고 max는 교환·결합적 monoid라, 한 pid의 token을 여러 owner로 쪼개도 partial-max 벡터를 병합하면 정확하다. 진짜 invariant는 "one owner per pid"가 아니라 "merger가 pid의 Q개 partial-max를 합산 확정하기 전엔 τ_k 비교에 넣지 않는다"이다 — 이는 finer load balancing을 허용한다.
  2. Turn 9의 calibration slack ξ_δ(recall)에 더해, 이종 하드웨어(GPU tf32 vs CSD fp16-LUT) 수치오차 η_num가 exactness에 추가 항으로 들어간다. τ_k는 ξ_δ + η_num만큼 낮춰야 한다.
  3. Turn 10의 hedge 비용 회계가 틀렸다. deferred/tied-request hedging을 쓰면 추가 PCIe = P[T_csd > t_hedge] · bytes_gpu로, t_hedge=p95면 avg ~30KB/query에 불과 — Mode B에서도 hedging은 싸다.
  4. Turn 10의 dual-price는 안정성 조건을 생략했다. η < 2/L(L = 수요-가격 탄력성)이고, 이 탄력성은 Turn 7의 λ_switch 근방에서 급증하므로 switch point에서 η를 줄여야 진동을 막는다.
  5. stale τ_k는 항상 exact-safe(monotone 성질)이며 효율만 저하시킨다 — cancellation broadcast 요구를 크게 완화한다.

새로 추가한 사실

사실 1 — Score ownership 정정: MaxSim은 max-monoid라 token 단위 분할이 exact다 (Turn 10 Invariant A 완화)

Turn 10은 "같은 pid의 token 일부를 GPU가, 일부를 CSD가 scoring하면 max_j가 깨진다"고 했으나, 틀렸다. MaxSim 정의:

S(q,d) = Σ_{i=1}^{Q} max_{j∈tokens(d)} sim(q_i, x_j)

token 집합을 tokens(d) = A ⊎ B로 분할하면 각 query token i에 대해

max_{j∈A⊎B} sim(q_i,x_j) = max( max_{j∈A} sim(q_i,x_j),  max_{j∈B} sim(q_i,x_j) )

즉 owner별로 길이-Q의 partial-max 벡터 p^A_i, p^B_i를 만든 뒤 merger가 p_i = max(p^A_i, p^B_i) (elementwise), S = Σ_i p_i로 합치면 정확히 동일하다. max는 교환·결합·멱등 monoid이므로 owner 수·순서·중복과 무관하다. 정정된 invariant:

Invariant A' (partial-max ownership):
  각 (pid, query-token i)에 대해 partial-max p_i는 여러 owner가 기여할 수 있다.
  merger는 pid의 Q개 partial-max가 "모든 owner로부터 도착 완료"되기 전에는
  그 pid의 S를 확정(τ_k 비교·top-k 삽입)하지 않는다.

함의(신규): block이 pid를 쪼개는 layout(Turn 5의 π>1 원인)도 owner-tracking 없이 안전하게 병렬 실행 가능하다. 대신 merger는 in-flight pid당 Q개 partial-max register + arrival counter를 유지해야 한다(pid당 Q·b_e + O(1) = ~64B, 수천 in-flight면 수백 KB — host DRAM에 무해). 이는 Turn 10이 "block이 pid를 쪼개면 metadata 추가 필요"라 한 것을 "metadata = per-pid partial-max accumulator 하나"로 정량화한다.

사실 2 — 이종 하드웨어 수치오차 η_num가 exactness 3번째 slack 항이다

partial-max를 GPU(tf32/fp16 tensor core)와 CSD(fp16-via-LUT 또는 int8 근사)가 나눠 계산하면, 같은 sim(q_i,x_j)도 rounding이 달라 near-tie에서 다른 j가 뽑히거나 합산 순서가 달라진다. 따라서 "GPU/CSD 혼합 owner의 exact top-k"는 엄밀히는 수치 tolerance까지만 exact다. Turn 9의 recall slack에 이 항을 더한다:

τ_k^safe(q) = τ_k^approx(q) − ξ_δ − η_num
  ξ_δ    : approx(centroid/quantized) vs full-residual 오차 분위수  (Turn 9)
  η_num  : owner 간 floating-point 재현오차 상한 (≈ Q · ε_mach · max|sim|)

수치 감(fp16 ε_mach≈2^-11, Q=32, |sim|≤1): η_num ≈ 32·5e-4 ≈ 0.016 — ξ_δ에 비하면 작지만, top-k 경계가 촘촘한 hard query에서는 무시하면 재현 불가능한 top-k가 나온다. 해결: (a) merger가 tie-break를 pid id로 결정론화, (b) partial-max·합산을 owner 무관 고정 precision(fp32 accumulate)으로 통일. 이것이 사실 1의 partial-max 병합이 요구하는 숨은 정밀도 계약이다.

사실 3 — 제어 평면 자체가 host CPU/DRAM 예산을 먹는다 (지금까지 8턴이 "scheduling 결정 = 무료"로 가정)

Turn 6~10은 routing/batching/merge를 latency에 넣었지만 그 결정을 계산하는 비용을 한 번도 회계하지 않았다. 고QPS에서 이 비용이 병목이 된다. 30k QPS·query당 |Blocks|≈1,000·MinHash 128 perm·shard m=64·top-k k=1,000 기준:

MinHash 서명(content-aware batching):  128 perm × 1,000 block = 1.28e5 hash/q
  → 30k QPS = 3.84e9 hash/s ≈ CPU 5–10코어 (hash ~2–3 ns)
전역 top-k merge:                       m·k = 6.4e4 비교/q
  → 30k QPS = 1.9e9 op/s ≈ CPU 1–2코어  (+ 임계경로 T_merge)
dual-price cost 평가:                    (m+1) 후보 × 소수 FLOP ≈ 2e6/s  → 무시 가능
containment/overlap 스코어링:            open_batch당 O(B) → 최악 O(B²)

→ 제어 평면만 ~7–14 host 코어 + host DRAM 대역폭을 요구한다. 이는 CSD가 "PCIe·host-DRAM을 비운다"는 Turn 7 서사와 부분 상충한다: embedding 이동은 사라지지만 scheduling metadata(서명·merge·telemetry)가 host DRAM 대역폭을 되먹는다. 정정된 원장에는 Turn 8~10의 5층(SM/HBM/PCIe/host-DRAM/CSD-int)에 더해 host-CPU-cycle·host-DRAM(control) 두 층을 추가해야 하고, 이 오버헤드가 f_csd*를 올릴수록(=CSD로 더 보낼수록) 커지는 역설이 있다(사실 1의 partial-max accumulator·서명 계산이 CSD 라우팅 query마다 발생).

사실 4 — Stale τ_k는 항상 exact-safe다: cancellation broadcast 요구 완화

Turn 8~10은 cancellation을 위해 τ_k를 in-flight owner에 계속 broadcast해야 한다고 암묵 가정했으나, monotone 성질이 이를 크게 완화한다. owner가 낡은 τ_k^old ≤ τ_k^cur를 써도:

{ d : UB(q,d) < τ_k^old }  ⊆  { d : UB(q,d) < τ_k^cur }

즉 취소하는 집합이 부분집합이라 절대 과-취소가 없다. true top-k 원소 d*는 UB(q,d*) ≥ S(q,d*) ≥ τ_k^cur ≥ τ_k^old이므로 τ_k^old 기준으로도 취소되지 않는다 → false-negative 불가. 따라서:

stale τ_k  ⇒  correctness 무손실,  효율(취소율)만 저하

설계 함의(신규): cancellation은 best-effort 비동기 broadcast로 충분하고 강한 일관성이 필요 없다. τ_k 갱신 왕복 지연 T_rtt와 shard scan 시간 T_scan의 비가 취소 세밀도를 정한다:

취소 checkpoint 수 ≈ T_scan / T_rtt
취소 효율 손실 ≈ E[τ_k가 stale인 구간에 취소 못 한 read bytes]

T_scan=5 ms, T_rtt=80 µs면 ~62 checkpoint로 촘촘하지만, T_scan=200 µs면 2–3회뿐 → Turn 9의 long CS command preemptible chunk 분할은 취소 세밀도를 위해서도 필요하다. 이는 Turn 10 Invariant D(idempotent cancel)와 결합해 일관성 프로토콜을 극도로 단순화한다(사실은 CRDT — 아래 참조).

더 구체화한 부분

Deferred(tied-request) hedging: PCIe 비용 정정과 수치

Turn 10의 hedge는 GPU 사본을 즉시 띄워 extra_GPU_bytes를 항상 지불하는 것처럼 모델링됐다. Dean & Barroso식 tied/deferred request로 바꾸면, primary(CSD)가 지연 t_hedge 안에 끝나면 GPU 사본을 아예 안 띄운다:

expected extra PCIe/query = P[T_csd > t_hedge] · bytes_gpu
t_hedge = p95(T_csd)  ⇒  P≈0.05,  bytes_gpu=0.6 MB  ⇒  ~30 KB/query 평균
총 CSD-경로 PCIe = 8 KB(query emb) + 30 KB(hedge 상각) = 38 KB  (all-GPU 0.6 MB의 ~1/16)

→ hedging은 Mode B에서도 PCIe 절감을 거의 되돌리지 않는다. Codex의 "hedging은 Mode B에서 손해" 결론은 immediate hedging에만 참이다. 최적 t_hedge는 tail penalty와 hedge PCIe의 균형:

t_hedge* = argmin_t [ λ_tail·P[T_csd > SLO | not hedged by t]  +  μ_pcie·P[T_csd > t]·bytes_gpu ]

t_hedge를 SLO slack과 μ_pcie(사실상 dual price)로 동적으로 정하면, PCIe가 붐빌 땐 t_hedge↑(hedge 자제), tail이 위험하면 t_hedge↓. Turn 10의 dual-price 프레임에 hedge가 자연스럽게 흡수된다.

Dual-price 안정성: η < 2/L과 switch point의 탄력성 급증 (Turn 10 정정)

Turn 10의 μ_r(t+1)=max(0, μ_r + η·(usage−target))는 step size 안정 조건이 없다. usage는 μ의 함수 usage_r(μ)이고, dual ascent 안정성은 gradient의 Lipschitz 상수 L = ‖∂usage/∂μ‖에 의존:

안정(비진동) 조건:  η < 2 / L,   L = 라우팅 수요-가격 탄력성

핵심(신규): L은 라우팅 무차별 경계(UCB_csd ≈ UCB_gpu) 근처 query 밀도에 비례한다. Turn 7의 λ ≈ λ_switch에서는 거의 모든 query가 무차별점 근처라 L이 급증 → 고정 η는 진동. 처방:

η_eff = η_0 / (1 + κ · elasticity_hat(t))     # 무차별점 밀도 추정으로 감쇠

이는 Turn 7의 hysteresis (λ_on, λ_off)와 같은 병을 다른 층에서 고치는 것 — hysteresis는 라우팅 출력에, adaptive η는 가격 갱신에 건다. 둘을 결합해야 fast(routing)·slow(price) 두 timescale 모두 안정.

병합기 = 유계 join-semilattice(CRDT) → Turn 10 open Q2(exactness 증명)에 대한 구성적 답

Turn 10은 "cancellation·hedging이 exact top-k를 안 깨는 조건을 증명하라"를 남겼다. 사실 1·4를 합치면 병합 상태가 CRDT임을 보여 증명이 닫힌다:

상태 = (Heap_k, τ_k),  Heap_k = 지금까지 확정된 top-k pid-score 집합
merge(H_a, H_b) = top_k( H_a ∪ H_b )          # 결합·교환·멱등 (집합 union + top-k)
τ_k = Heap_k의 k번째 값                          # merge에 대해 단조 증가
  • merge가 결합·교환·멱등 ⇒ 메시지 순서·중복·재전송 무관 수렴(Invariant D 자동 충족).
  • τ_k 단조 ⇒ 사실 4의 stale-safe cancellation.
  • partial-max(사실 1)도 per-(pid,i) max가 같은 성질 ⇒ 전체 병합이 두 층의 join-semilattice 합성.

따라서 "확정 전 모든 owner partial-max 도착"(Invariant A')만 지키면 cancellation·hedging·재전송이 있어도 결과가 sequential exact top-k와 동일함이 CRDT 수렴으로 증명된다. 강한 일관성·락 불필요.

Q-tiling 재분석: re-read가 아니라 SRAM 재분배 + 직렬 passes가 진짜 비용 (Turn 10 정정)

Turn 10은 Q-tiling(Option B)이 "residual 재-read 또는 token buffer"를 요구한다며 read 재사용을 걱정했다. 하지만 streaming은 doc 단위이므로 한 doc을 처리하는 동안 그 doc residual만 on-chip에 잡으면 재-read가 없다:

단일-doc buffer = doclen · bytes_code ≈ 80 tok · 32 B ≈ 2.5 KB  (사소)
→ Q/Q_tile passes를 이 buffer 위에서 반복  ⇒  NAND 재-read = 0

즉 binding resource(내부 read)는 Q-tiling과 무관하다 — Codex의 read-amplification 우려는 과장. 진짜 비용 두 가지:

(1) SRAM 재분배: LUT bank ∝ Q_tile·E·2^b·b_e  vs  token buffer ∝ doclen·bytes_code
    총 SRAM이 binding이면 Q-tiling은 LUT→buffer로 옮길 뿐 순이득이 없을 수 있다.
(2) 직렬 passes: passes_per_doc = Q/Q_tile 배 → per-doc compute latency 증가
    → T_score 상승. T_score > T_read이 되는 순간 compute-bound 전환(Turn 7 사실2).
Q-tiling break-even:  Q/Q_tile · t_pass(Q_tile) ≤ T_read_per_doc  (read-bound 유지 조건)

이는 Eyeriss dataflow taxonomy로 정확히 매핑된다: full-Q streaming = query-stationary(LUT에 query 상주, doc 1회 스트림), Q-tiling = doc-stationary(doc buffer 상주, query tile 스트림). 선택 기준은 "무엇이 재사용 가치가 큰가": Q개 query token은 모든 doc에 재사용(→query-stationary 유리), doc은 Q_tile 패스에만 재사용. 따라서 hot-set이 크면 query-stationary(full-Q), SRAM이 극히 부족한 소형 CSD면 doc-stationary(Q-tiling).

Epoch RCU: usable-L2 감소와 ADD가 강제하는 resharding (Turn 10 index_epoch 구체화)

Turn 10의 index_epoch/tombstone은 메커니즘이 없었다. 자연스러운 구현은 RCU/double-buffer:

update:  새 epoch block을 병렬 기록 → grace period(기존 query drain) → 구 block 회수
usable_L2 = C_L2 − ( Δepoch_hot_bytes + inflight_grace_bytes )
  • DELETE: tombstone 1 bit/pid로 저렴, layout 불변, π·shard map 유지.
  • ADD: delta index가 shard hash 분포를 바꿔 Turn 5 static sharding 가정을 깬다. 두 선택 — (a) overflow shard(신규 pid를 GPU-served로 두고 다음 rebuild까지 유예: CSD 이득 없지만 정확), (b) 즉시 resharding(admission churn·layout rewrite 폭증). update-heavy면 (a)가 지배해 CSD 유효 coverage가 시간에 따라 감쇠 → rebuild 주기 T_rebuild가 새 SLO 변수.
CSD 유효 coverage(t) = HR_bytes(B) · (1 − add_fraction·(t − t_rebuild)/T_rebuild)

→ Turn 10의 update_pressure에 "coverage 감쇠 → 주기적 rebuild 비용" 항을 추가. Turn 7의 warm-up T_warmup은 이 rebuild마다 재발생.

관련 맥락 / 배경 설명

  • 왜 CRDT 프레이밍이 본질인가: 사실 1·4를 CRDT(join-semilattice)로 묶으면, 이 runtime의 정확성이 분산 합의(Paxos/Raft)나 락이 아니라 monotone 수렴에 의존함이 드러난다. top-k는 유계 lattice, τ_k는 그 위의 단조 함수라, hedge/cancel/재전송은 전부 "더 많은 정보를 join"할 뿐이라 결과를 못 바꾼다. 이는 IR의 exhaustive vs early-termination이 같은 답을 주는 이유(WAND/BMW의 safe pruning)와 동형이며, Turn 9의 block-max WAND 연결의 정확성 측 짝이다.
  • 제어 평면 오버헤드가 서사를 조인다: 사실 3은 "embedding을 PCIe 밖에 가둔다"는 Turn 7 프레이밍의 한계다 — data plane을 비울수록 control plane(서명·merge·telemetry·partial-max accumulator)이 host를 먹는다. 진짜 승리는 "host를 완전히 비운다"가 아니라 "MB급 embedding 이동을 KB급 metadata 이동으로 교환한다"이고, 그 교환비(0.6 MB → 38 KB, 사실 3+hedge)가 CSD 논거의 정량 핵심이다.
  • Eyeriss dataflow가 CSD datapath 선택의 상위 언어다: Q-tiling 재분석은 Turn 5의 "format이 하드웨어를 정한다"를 dataflow 층으로 올린다 — query-stationary vs doc-stationary는 재사용 거리(reuse distance)를 실리콘에 새기는 결정이고, hot-set 크기(Turn 1 locality)와 SRAM 예산이 그 선택을 함께 정한다. locality 지표가 캐시 정책(Turn 7) → layout(Turn 5) → 이제 datapath dataflow까지 3층을 관통한다.

잠재적 오류 또는 확인 필요 지점

  1. (정정) Turn 10 Invariant A: max가 monoid라 pid는 token 단위로 분할·partial-max 병합 가능(사실 1). "one owner per pid"는 불필요; merger의 per-pid accumulator 비용만 실측하면 됨 — arrival-counter/accumulator 메모리 실측 필요.
  2. (정정) Turn 10 hedge 회계: immediate가 아니라 deferred(tied-request)면 PCIe 추가 = P[T>t_hedge]·bytes로 작다(더 구체화 1). Mode B "hedge 손해" 결론은 immediate에만 유효 — *T_csd 분포·t_hedge* 실측 필요*.
  3. (정정) Turn 10 dual-price: η < 2/L 안정 조건 누락, λ_switch 근방 탄력성 급증으로 진동 위험(더 구체화 2). — *elasticity_hat 추정기·κ sweep 시뮬 필요*.
  4. η_num의 실제 크기: 사실 2는 fp16 배경지식 추정. 실제 GPU(tensor core 경로) vs CSD LUT의 재현오차는 accumulate precision·tie-break 정책에 의존 — 혼합 owner 동일-query 재현성 실측 필요.
  5. 제어 평면 코어 수(사실 3): MinHash/merge 코어 추정은 perm 수·block 수·m·hash 구현에 민감. SIMD MinHash·GPU 서명 오프로드로 낮출 여지 — 실측 및 서명 오프로드 대안 필요.
  6. stale-τ_k 효율 손실(사실 4): correctness는 증명됐으나 취소율 저하량은 T_scan/T_rtt와 score 분포에 의존 — *T_rtt 실측 + 취소율 시뮬 필요*.
  7. RCU usable-L2 페널티: Δepoch_hot_bytes + inflight_grace_bytes 크기는 update rate·grace period 길이에 의존, overflow-shard vs resharding 임계 QPS 미확정 — update stream 합성 + rebuild 주기 sweep 필요.

Turn 12 / Codex

이번 턴의 확장 요약

이번 턴은 Turn 11의 partial-max 병합 가능성을 받아들이되, 그 결론이 runtime 설계에 주는 부작용을 더 구체화한다. 핵심은 다음이다.

  1. max monoid 덕분에 pid를 token 단위로 쪼개도 정확성은 유지할 수 있지만, partial-max 벡터를 host로 많이 반환하면 CSD의 PCIe 절감 논리가 무너진다.
  2. cancellation과 hedging이 exact top-k를 깨지 않으려면 단순 CRDT 병합만으로 부족하고, 각 pid가 EXACT 또는 PRUNED로 끝났음을 보이는 completion certificate가 필요하다.
  3. scheduler는 평균 latency와 dual price만 보면 부족하며, 실제 CSD에서는 NAND die/channel hotspot, GC stall, thermal throttling 때문에 locality placement 자체가 tail latency를 만든다.
  4. 따라서 다음 단계의 runtime 모델은 locality hit ratio가 아니라 hit ratio × topology balance × result-traffic bound × certificate cost를 함께 최적화해야 한다.

새로 추가한 사실

1. Partial-max token sharding은 정확하지만 PCIe result traffic을 폭증시킬 수 있다

Turn 11의 핵심 정정, 즉 MaxSim의 token 집합 분할은 수학적으로 맞다.

S(q,d) = Σ_i max_j sim(q_i, x_dj)

tokens(d) = A ∪ B
max_{j∈A∪B} sim(q_i,x_j)
  = max(
      max_{j∈A} sim(q_i,x_j),
      max_{j∈B} sim(q_i,x_j)
    )

따라서 owner별 partial-max vector를 만들고 elementwise max로 합치면 정확하다.

하지만 이 구조를 cross-device split으로 쓰면 새로운 병목이 생긴다. 각 owner가 host merger에 pid별 partial vector를 보내야 하기 때문이다.

Bytes_partial
  = N_split_docs · n_owner · Q · b_acc

예시:

N_split_docs = 4096
Q            = 32
b_acc        = 2B(fp16) 또는 4B(fp32)
n_owner      = 2

Bytes_partial(fp16) = 4096 · 2 · 32 · 2B ≈ 524 KB
Bytes_partial(fp32) = 4096 · 2 · 32 · 4B ≈ 1.0 MB

이는 Turn 1의 압축 H2 baseline ~0.6 MB/query와 같은 규모다. 즉 partial-max를 host로 많이 올리면 “CSD는 top-k만 반환해서 PCIe를 줄인다”는 핵심 장점이 사라진다.

새 조건:

cross-device token split is useful only if

Bytes_partial + Bytes_control + Bytes_topk
  << Bytes_H2_gpu_baseline

따라서 token sharding은 기본적으로 다음 범위에 제한해야 한다.

권장:
  같은 CSD 내부 PE 간 token split
  같은 GPU kernel 내부 warp/block 간 token split
  같은 drive 내부 shard 간 partial reduction

주의:
  GPU와 CSD가 같은 pid의 token을 나눠 scoring하고 host가 Q-vector를 병합

비권장:
  많은 pid에 대해 partial-max vector를 PCIe로 반환하는 cross-device split

결론적으로 Turn 11의 Invariant A'는 정확성 측면에서는 강력하지만, PCIe 절감 측면에서는 result-traffic bound가 추가 invariant로 필요하다.

Invariant E: bounded partial egress
  host로 반환되는 partial-max bytes는
  saved embedding bytes의 작은 상수 비율 이하여야 한다.

2. CRDT 병합만으로는 cancellation 정확성이 닫히지 않고 completion certificate가 필요하다

Turn 11의 CRDT 관점은 merge 순서·중복·late completion에는 충분하다. 하지만 cancellation은 “없는 정보”를 결과에서 제외하는 행위라서, 단순 join만으로는 정확성 증명이 끝나지 않는다.

각 pid는 반드시 둘 중 하나의 terminal state를 가져야 한다.

EXACT(pid):
  모든 token partition의 partial-max가 도착했고,
  score_exact = Σ_i max_owner p_owner[i] 로 확정됨

PRUNED(pid):
  아직 계산하지 않은 partition까지 포함한 upper bound가
  현재 safe threshold보다 낮다는 certificate가 있음

partial execution 중인 pid의 안전한 pruning 조건은 다음처럼 쓴다.

a_i(d) = processed partitions에서 얻은 현재 partial max
R(d)   = 아직 읽지 않은 token/block partition 집합

UB_partial(q,d)
  = Σ_i max(
      a_i(d),
      max_{j∈R(d)} S_cent[i, c_j] + residual_slack_j(q)
    )

PRUNED if:
  UB_partial(q,d) < τ_k^safe(q)

여기서:

τ_k^safe(q) = τ_k^approx(q) − ξ_δ − η_num
  • ξ_δ: approximate score calibration 오차
  • η_num: GPU/CSD 이종 수치오차
  • residual_slack_j(q): residual norm 또는 codebook 기반 upper-bound slack

이 구조가 있어야 late completion, cancellation, hedging이 섞여도 host merger가 pid를 안전하게 닫을 수 있다.

추가 trace schema:

doc_terminal_events(
  q_idx,
  pid,
  terminal_state,        # exact | pruned
  n_partitions_total,
  n_partitions_scored,
  ub_at_prune,
  tau_safe_at_prune,
  slack_model_id,
  epoch_id
)

이 테이블이 없으면 “취소로 몇 byte를 아꼈는가”는 볼 수 있어도, 그 취소가 어떤 bound에 의해 안전했는지를 재현할 수 없다.

3. Document locality는 NAND topology locality와 다르다

지금까지 locality는 주로 pid/block frequency였다.

HR_bytes(B)
BRF_block(G)
reuse distance
hot coverage

하지만 CSD 내부에서는 같은 hot set이라도 어느 NAND channel/die/plane에 놓였는가가 tail latency를 바꾼다. hot document가 한 channel에 몰리면 L2 hit ratio가 높아도 miss traffic의 p99가 나빠질 수 있다.

topology-aware load:

load_c
  = Σ_{d assigned to channel c}
      λ_d · π_d · bytes_comp(d) · miss_prob(d)

placement 목표:

minimize   max_c load_c
maximize   HR_bytes(B)
minimize   Σ_q shard_touch_count(q)

즉 admission은 단순히 Benefit/Cost가 아니라 channel balance 항을 가져야 한다.

Score_admit(d, c)
  = Benefit_admit(d) / Cost_admit(d)
    − α_hotspot · projected_load_c
    − α_tail    · shard_tail_risk(d, c)

새로운 scheduling factor:

topology_skew(q)
  = max_c bytes_q,c / mean_c bytes_q,c

topology_skew(q)가 큰 query는 CSD 평균 service time이 좋아 보여도 특정 die/channel tail에 걸릴 가능성이 높다. deadline routing에서는 rho_bytes(q)보다 이 값이 더 강한 gate가 될 수 있다.

4. Dual price는 soft control이고, burst에는 credit lease가 필요하다

Turn 10·11의 dual-price scheduler는 평균 usage를 target으로 밀어내는 데 좋다. 하지만 sudden burst에서는 가격이 오르기 전에 queue가 먼저 찬다.

따라서 실제 executor 앞단에는 hard credit이 필요하다.

credit_pcie(t)
credit_cmd(t)
credit_csd_read(t)
credit_host_cpu(t)

각 credit은 시간당 refill된다.

credit_r(t+Δ)
  = min(CreditCap_r, credit_r(t) + refill_r · Δ − reserved_r)

task admission 조건:

execute task only if:
  required_credit_r(task) ≤ available_credit_r
  for all constrained resource r

dual price는 “어느 경로가 더 비싼가”를 결정하고, credit lease는 “지금 당장 넣어도 큐가 터지지 않는가”를 보장한다.

route_score(dev, task)
  = UCB_latency(dev, task)
    + μ_pcie · bytes_pcie
    + μ_cmd  · n_cmd
    + μ_int  · bytes_internal
    + μ_cpu  · host_cycles
    + μ_slo  · violation_risk

feasible(dev, task)
  = all required credits are available

이렇게 나눠야 λ_switch 근방에서 가격 진동이 줄고, NVMe SQ/CQ 같은 hard limit을 넘지 않는다.

더 구체화한 부분

Certificate-aware runtime state machine

pid별 accumulator는 다음 상태를 가진다.

INIT
  → PARTIAL(pid, seen_mask, partial_max[Q])
  → EXACT(pid, score)
  → PRUNED(pid, ub, tau_safe, proof_id)

전이 규칙:

INIT → PARTIAL:
  first owner returns partial-max vector

PARTIAL → PARTIAL:
  merge by elementwise max
  seen_mask |= owner_partition_mask

PARTIAL → EXACT:
  seen_mask == all_partitions

INIT/PARTIAL → PRUNED:
  UB_partial(q,d) < τ_k^safe(q)

EXACT → TOPK_MERGE:
  score can enter global heap

PRUNED:
  never enters heap, but proof is logged

중요한 점은 PRUNED가 EXACT보다 약한 상태가 아니라, top-k 후보가 될 수 없다는 증명으로 닫힌 terminal state라는 것이다.

Cross-device split break-even

cross-device split이 tail을 줄이는 이득:

Saved_tail_time
  = T_slowest_unsplit − T_parallel_split

추가 비용:

Extra_cost
  = Bytes_partial / BW_pcie
    + T_merge_partial
    + host_cpu_partial
    + extra_control_bytes / BW_pcie

사용 조건:

Use cross-device split only if:

Saved_tail_time
  > Extra_cost + λ_pcie · Bytes_partial + λ_cpu · host_cpu_partial

대부분의 query에서는 pid 단위 또는 shard 단위 routing이 낫고, cross-device token split은 다음 경우에만 쓸 만하다.

1. 매우 긴 문서 또는 outlier passage
2. 특정 shard straggler가 p99를 지배하는 query
3. partial vector가 host가 아니라 같은 device 내부에서 reduce되는 경우
4. top-k 근처 소수 pid에만 split을 적용하는 경우

T_csd = T_ctrl + max(T_read, T_score) + T_result

확장:

관련 맥락 / 배경 설명

이번 확장의 핵심은 “정확성은 monoid로 풀리지만, 비용은 monoid로 사라지지 않는다”는 점이다. MaxSim의 max 구조 덕분에 token 분할, late completion, 중복 메시지는 수학적으로 다룰 수 있다. 그러나 partial result를 어디서 병합하느냐에 따라 PCIe traffic이 MB급으로 되돌아올 수 있다.

따라서 CSD runtime의 원칙은 다음처럼 정리된다.

1. 가능한 한 device 내부에서 partial-max를 reduce한다.
2. host에는 finalized top-k 또는 certificate만 올린다.
3. partial vector를 host로 올리는 경로는 tail hedge용 예외로 둔다.
4. 모든 pruning은 UB certificate로 닫는다.
5. locality placement는 frequency뿐 아니라 NAND topology balance를 본다.

이 관점에서 document locality 연구도 한 단계 바뀐다. 단순히 “어떤 문서가 자주 읽히는가”가 아니라, “자주 함께 읽히는 문서들이 같은 channel에 몰리는가”, “hot block이 GC/write 경합을 만드는가”, “resident set이 topology tail을 줄이는가”까지 trace에서 재구성해야 한다.

---

잠재적 오류 또는 확인 필요 지점

  1. Turn 11의 partial-max 병합은 정확하지만, host로 partial vector를 대량 반환할 경우 PCIe 절감이 사라질 수 있다. Bytes_partial을 반드시 원장에 넣어야 한다.
  1. CRDT 병합은 finalized score에는 충분하지만, pruned document에는 UB_partial < τ_k^safe certificate가 필요하다. negative information을 증명 없이 버리면 exactness 주장이 약해진다.
  1. residual_slack_j(q)를 δ_max · ||q_i||로 둘지, centroid별/codebook별 tighter bound로 둘지는 실측해야 한다. slack이 느슨하면 read-before-prune 이득이 거의 없다.
  1. topology-aware placement는 실제 CSD가 channel/die mapping을 얼마나 노출하는지에 의존한다. FTL이 물리 배치를 숨기면 host scheduler는 근사 telemetry만 쓸 수 있다.
  1. credit lease의 refill rate와 cap을 잘못 잡으면 dual-price보다 더 보수적인 병목이 된다. burst protection과 평균 throughput 사이 sweep이 필요하다.
  1. GC와 thermal throttling은 MS MARCO offline replay만으로는 보이지 않는다. 장시간 QPS trace와 background write/admission workload를 같이 넣어야 한다.

다음 에이전트가 이어서 확장할 지점

  1. Certificate-aware exact runtime simulator: pid accumulator state machine을 구현해 EXACT/PRUNED terminal event를 기록하고, centroid-UB slack·τ_k^safe·stale threshold 조건에서 recall, saved read bytes, certificate count, host merge cost를 함께 측정하라.
  1. Partial-result traffic bound 실험: pid 단위 routing, intra-CSD token split, cross-device token split, deferred hedge split을 비교해 Bytes_partial, PCIe GB/query, p99, host CPU cycles를 측정하라. 특히 Bytes_partial << Bytes_H2_gpu 조건이 깨지는 지점을 찾아라.
  1. Topology-aware locality placement: 기존 HR_bytes(B) trace에 channel/die mapping과 GC/thermal telemetry를 붙여 STATIC-HOT, balanced-hot, replicated-hot, dual-price+credit 정책을 비교하라. 결과는 hit ratio가 아니라 p99, topology_skew, internal GB/s, admission write interference로 정리하라.

Turn 13 / Claude

이번 턴의 확장 요약

이번 턴은 Turn 12의 네 축(Invariant E 부분전송, completion certificate, NAND topology, credit lease)을 받되, 그중 두 수치를 정정하고 지금까지 12턴이 한 번도 계량하지 않은 세 항을 추가한다. 핵심은 "CSD는 PCIe 밖에 embedding을 가둔다"는 서사가 다운링크(query 배포)와 제어 평면 에너지라는 두 곳에서 새는 것을 정량화하는 것이다.

핵심 신규 결론 다섯 가지:

  1. Turn 12의 Bytes_partial = N_split_docs·n_owner·Q·b_acc(512KB~1MB)는 후보 전체를 split한다는 strawman이다. centroid-UB 선행 pruning 후 cross-device split은 top-k 근처 straggler O(k′)개에만 걸어야 하므로 실효 egress는 1~2 order 작다(~10–40KB). Invariant E는 옳지만 임계값이 훨씬 관대하다.
  2. "CSD는 query 8KB만 내려보낸다"는 12턴 내내 반복된 가정이 sharding에서 깨진다: 후보가 여러 drive에 흩어지면 각 drive가 전체 query를 받아야 해 다운링크 = 8KB·n_drives. 이것이 Turn 7의 "scale-out으로 BW_int를 키운다"와 정면 충돌하며 유효 drive 수 상한(~71)을 만든다.
  3. 제어 평면 host-CPU 에너지가 Turn 9의 movement 에너지 절감을 압도한다: content-aware batching의 MinHash가 host에서 돌면 ~50–100W, movement 절감은 ~6W → 에너지-win 서사가 조건부로 무너진다.
  4. Turn 9/12의 τ_k^safe = τ_k^approx − ξ_δ는 hedge로 GPU-exact와 CSD-approx owner가 섞이면 느슨하다. per-doc slack의 k번째 order statistic이 더 타이트하면서 여전히 safe다.
  5. Turn 11 RCU의 epoch-snapshot은 residual뿐 아니라 centroid table에도 걸려야 certificate가 유효하다.

새로 추가한 사실

사실 1 — Query fan-out: "8KB만 내려간다"가 sharding에서 다운링크 병목이 된다

research seed부터 Turn 12까지 CSD 다운링크는 query embedding 8KB + control로 고정 가정됐다. 그러나 corpus를 n_drives개로 shard하면(Turn 7의 scale-out), 한 query의 후보 pid들은 여러 drive에 흩어지고, 각 drive는 자기 후보를 전체 Q개 query token에 대해 scoring해야 하므로 그 drive 전체가 8KB query를 받아야 한다.

query 하나의 후보 k(~1000)개가 n_drives에 균등 분산되면
  touched_drives ≈ min(n_drives, k) = n_drives     (n_drives ≤ k)
downstream_bytes/query = 8KB · n_drives  (+ block-list descriptor)

즉 다운링크는 상수 8KB가 아니라 drive 수에 선형이다. 수치(임의 가정 λ=30k QPS, β_down=0.7, BW_pcie_down=25 GB/s):

downstream 안정: λ · 8KB · n_drives ≤ β_down · BW_pcie_down
⇒ n_drives ≤ β_down·BW_pcie_down / (λ·8KB)
           = 0.7·25e9 / (30000·8192) ≈ 71 drives      # 단일 host root complex 기준

함의(신규): Turn 7은 "scale-out으로 aggregate BW_int를 키워 BW_int ≥ ε·BW_pcie를 만족"한다고 했지만, drive를 늘릴수록 다운링크 query fan-out이 커져 반대 방향으로 압력을 준다. 두 힘의 최적점이 존재한다:

upstream(read):   per-query T_read = bytes_H2 / (n_drives · BW_int_drive)   → n_drives↑ 유리
downstream(query): 8KB · n_drives                                            → n_drives↑ 불리 (β_down 소모)

결정적 성질: doc-side 전송을 CSD가 0으로 만들수록 query fan-out이 남은 다운링크의 지배항이 된다. 이것은 "마지막까지 최적화 안 한 항이 병목이 된다"의 전형이고, 12턴이 놓친 다운링크 예산 축이다. fp16 대신 int8 query 전송(4KB)·PCIe multicast(하드웨어 지원 시)·"query를 drive에 broadcast 후 내부 fan-out"으로 완화하지만, NVMe는 native multicast가 약해 8KB·n_drives가 실질 하한이다(실측 필요: multicast/peer-DMA 지원 여부).

부수 항(관련): 각 drive는 stage-1 centroid scoring q·centroids를 자기 shard 위에서 수행한다. centroid table이 drive마다 복제되면 Q·nprobe·E MAC이 n_drives배 중복된다(nprobe=8192 기준 ~33 MMAC/query·drive, MaxSim의 ~1.5%지만 복제 시 누적). centroid도 shard해 복제 낭비를 없애야 한다 — Turn 5 sharding 결정에 "centroid 복제 vs 분할" 축 추가.

사실 2 — 제어 평면 host-CPU 에너지가 movement 에너지 절감을 압도한다 (Turn 9 energy-win 조건화)

Turn 9는 CSD가 movement 에너지 ~344 pJ/byte를 절감해 ~6 W(30k QPS, 0.6MB/query)를 아낀다고 했고, Turn 11은 content-aware batching MinHash가 host ~7–14 코어를 먹는다고 했다. 이 둘을 같은 원장에 놓으면 부호가 뒤집힌다.

movement 절감 = 344 pJ/B · 0.6 MB/q · 30k QPS ≈ 0.206 mJ/q · 30k ≈ 6.2 W   (Turn 9)
MinHash 제어 = 128 perm · 1000 block = 1.28e5 hash/q · 30k = 3.84e9 hash/s
  @ ~2.5 ns/hash/core → ~9–10 코어,  @ 5–10 W/코어 → 48–96 W               (Turn 11 확장)

핵심(신규): content-aware batching을 host CPU에서 순진하게 돌리면 그 에너지(50–100W)가 CSD가 아끼는 movement 에너지(6W)를 8–16배 초과한다. 즉 Turn 9의 "energy-win regime"은 무조건 성립하는 게 아니라 제어 평면이 싸다는 조건부다. 세 가지 탈출로:

(a) content-aware batching 비활성화     → BRF_block 이득 포기(read-bound 손해), 에너지 win 회복
(b) SIMD MinHash                        → ~5–8× → ~1.5 코어(8–15W), 6W와 동급(순이득 미미)
(c) 저비용 overlap 추정(bottom-k sketch, LSH bucket, perm↓)  → 정확도↔에너지 sweep

에너지 회계의 비대칭성이 본질이다: content-aware batching이 아끼는 것은 내부 NAND read(양 경로 공통항)의 BRF배뿐이고, CSD가 이미 없앤 PCIe/DRAM/HBM 왕복은 건드리지 않는다. 따라서 이 최적화는 latency(read-bound) 이득은 크지만 에너지 관점에선 host-CPU 세금이다. Turn 11 사실 3의 "제어 평면이 host를 되먹는다"를 에너지 부호까지 확정한다: 원장에 host-CPU-energy 항이 없으면 energy-win regime 경계를 낙관적으로 과대평가한다.

사실 3 — Mixed exact/approx owner에서 per-doc slack이 더 타이트하고 여전히 safe (Turn 9/12 τ_k^safe 정정)

Turn 9/12는 pruning threshold를 τ_k^safe = τ_k^approx − ξ_δ − η_num로 전역 상수 ξ_δ를 뺐다. 그러나 hedge(Turn 10·11)로 GPU가 exact score, CSD가 centroid-approx score를 내는 doc이 heap에 섞이면, GPU-exact entry는 slack이 0인데도 전역 ξ_δ를 함께 깎여 threshold가 불필요하게 낮아진다(=under-pruning).

정정: doc별 하한 lb_d = score_d − slack_d를 만들고 그 집합의 k번째 order statistic을 쓴다.

slack_d = 0            (GPU-exact owner)
        = ξ_δ          (CSD centroid/quantized-approx owner)
        = η_num        (혼합 owner, 수치오차만)
τ_k^safe(q) = k-th largest of { score_d − slack_d : d ∈ Heap }

안전성 증명(신규, 1줄): true_d ≥ lb_d ∀d ⇒ order statistic의 elementwise 지배로 k-th largest(true) ≥ k-th largest(lb) ⇒ τ_k^safe는 참 k번째 score의 하한 ⇒ false-negative 불가. 동시에 전역 τ_k^approx − max_d slack_d보다 크거나 같아(exact entry가 slack을 안 깎으므로) 더 공격적으로 취소한다.

함의: hedge가 GPU-exact 결과를 heap에 넣을수록 τ_k^safe가 올라가 CSD 쪽 취소가 더 aggressive해진다 — hedge와 cancellation이 협력적임을 드러낸다(Turn 10은 둘을 경쟁 관계로만 봤다). 단, heap entry마다 slack_d(=owner/모델 id)를 저장해야 하므로 Turn 12 certificate schema에 owner_class, slack_model_id를 heap 항목 수준으로 내려야 한다.

더 구체화한 부분

Turn 12 Bytes_partial 정정: 후보 전체가 아니라 straggler O(k′)개

Turn 12는 N_split_docs = 4096(≈ 후보 전체)로 512KB~1MB egress를 계산해 Invariant E를 정당화했다. 그러나 cross-device token split은 모든 후보에 걸 이유가 없다 — Turn 12 자신의 조건 4("top-k 근처 소수 pid에만 split")와 모순이다. 올바른 순서:

1. centroid-UB(Turn 9)로 후보 대량 pruning     → 생존 doc ~ O(k) (k=1000)
2. 생존 doc 중 shard-straggler로 p99를 지배하는 소수만 cross-device split  → k′ ≪ k
Bytes_partial = k′ · n_owner · Q · b_acc
  예: k′=64, n_owner=2, Q=32, b_acc=2B → 8.2 KB   (Turn 12의 512KB 대비 ~64×↓)

추가 압축(신규): partial-max 벡터의 대부분 token은 한 owner가 지배해 상대 owner 값은 near-tie에서만 병합에 영향을 준다. owner B는 p^B_i > (τ 기여 하한)인 token만 sparse하게 보내면 되므로, 실효 전송은 Q가 아니라 contested token 수 q_tie ≪ Q에 비례한다. 따라서 Invariant E(bounded partial egress)는 정책적으로 자동 만족되며, 512KB는 "만약 후보 전체를 무차별 split하면"의 비현실적 상한이다.

Topology 시너지: L2 residency가 NAND channel load를 공짜로 평탄화한다 (Turn 12 사실 3 확장)

Turn 12는 topology balance를 admission의 별도 항 −α_hotspot·projected_load_c로 붙였다. 그러나 hot-set residency(Turn 1 STATIC-HOT)와 topology balance는 대체로 정렬된다:

load_c = Σ_{d∈channel c} λ_d · π_d · bytes(d) · miss_prob(d)
L2에 f(d) 상위 doc을 pin ⇒ 그 doc의 miss_prob ≈ 0 ⇒ load_c에서 제거
∴ 가장 hot한(=가장 큰 load 기여) doc이 먼저 NAND load에서 빠짐 → skew 자동 완화

함의(신규): residency가 latency·PCIe뿐 아니라 NAND channel skew까지 동시에 줄이는 3-in-1 지렛대다. 단 hotspot은 L2 cutoff 바로 아래 warm tier로 이동한다 — L2에 안 들어가지만 여전히 자주 읽히는 doc들이 새 channel 병목이 된다. 따라서 topology-aware placement는 전체가 아니라 warm tier(residency 실패한 상위 접근 doc)에만 걸면 된다.

역방향 결합(신규): admission fill write와 GC는 특정 channel에 몰린다(Turn 3/5 ζ). hot doc을 read-hot channel과 같은 channel에 write하면 read-write 간섭이 생긴다. 즉 placement는 read-load와 write-load(admission target)를 동시에 분산하는 이분(bipartite) 균형이다:

minimize max_c ( read_load_c + w · write_load_c )
  write_load_c = admission_fill_bytes_c/BW_admit + GC_bytes_c

read만 보는 Turn 12의 load_c는 write-hot channel에 read-hot을 얹는 최악 배치를 못 피한다.

RCU epoch-snapshot은 centroid table에도 걸린다 (Turn 11 사실 완성 + Turn 12 certificate)

Turn 11 RCU는 residual block double-buffer만 다뤘다. 그러나 사실 1에서 drive가 UB_cent를 계산하려면 centroid table·IVF·doclens도 읽는다. update가 centroid를 재학습/재배정하면(대규모 rebuild) centroid table도 epoch가 바뀐다. certificate 유효성:

valid_certificate(q, pid) requires  epoch(centroid_table) = epoch(residual) = epoch(doclens) = query_epoch

DELETE(tombstone)는 centroid 불변이라 안전하지만, centroid 재학습을 포함한 rebuild는 in-flight query의 centroid read snapshot까지 pin해야 한다. 이는 Turn 11의 usable_L2 페널티에 centroid table 이중화(수백 MB·배경지식)를 더한다. PRUNED certificate는 그래서 (pid, τ_safe, UB, slack_model_id, epoch_id) 5-tuple이어야 재현·감사 가능(Turn 12 schema에 slack_model_id·epoch_id 있음 — centroid_epoch를 명시 분리 권장).

관련 맥락 / 배경 설명

  • 다운링크가 서사를 조이는 이유: Turn 7~12는 CSD의 PCIe 이득을 "up-link egress(top-k만 올림)"로만 봤다. 사실 1은 down-link(host→device)를 처음 계량한다. PCIe는 full-duplex라 up/down이 별도 lane이지만, sharding이 넓어질수록 down-link query fan-out이 up-link 절감과 같은 규모가 된다(8KB·71 = 0.57MB ≈ baseline 0.6MB). 즉 "embedding을 KB metadata로 교환한다"(Turn 11)는 명제의 정확한 형태는 "up-link embedding을 없애되 down-link query를 drive 수만큼 복제한다"이고, 순이득은 bytes_H2 − 8KB·n_drives다. 이 부호가 scale-out 폭을 정하는 새 설계 변수다.
  • 에너지 방어선의 숨은 세금: Turn 9는 CSD 방어선을 latency→capacity→energy로 넓혔다. 사실 2는 그 가장 넓은 방어선(energy)조차 제어 평면 CPU 세금에 노출됨을 보인다. 정확한 명제는 "CSD가 movement 에너지를 아낀다"가 아니라 "CSD가 movement 에너지를 아끼되, 그 절감보다 제어 평면(특히 content-aware batching) 에너지가 크지 않은 한에서만 순이득"이다. 이는 이 연구의 원장이 반드시 {SM, HBM, PCIe, host-DRAM, CSD-int, host-CPU-cycle, host-CPU-energy} 7층이어야 하는 이유다(Turn 11 6층 + energy).
  • hedge와 cancellation의 협력(사실 3의 배경): GPU-exact hedge가 heap에 들어가면 τ_k^safe가 올라 CSD 취소가 빨라진다 — 즉 hedge의 부작용(PCIe 소모)이 cancellation의 read 절감으로 부분 상쇄된다. 이는 Turn 10의 "hedge vs 절감" 경쟁 프레임을 "hedge → 더 타이트한 threshold → 더 많은 CSD read 절감"의 협력 프레임으로 바꾼다. Fagin TA/NRA에서 정확한 후보 하나가 threshold를 올려 나머지를 대량 pruning하는 것과 동형이다.

잠재적 오류 또는 확인 필요 지점

  1. (정정) Turn 12 Bytes_partial: N_split_docs를 후보 전체(4096)로 잡은 것은 strawman. centroid-UB 선행 pruning + straggler-only split이면 실효 k′·n_owner·q_tie·b_acc ≈ 8–40KB. — k′(split 대상 straggler 수)·q_tie(contested token) 분포 실측 필요.
  2. (신규) query fan-out 8KB·n_drives: PCIe/NVMe가 multicast·peer-to-peer broadcast를 지원하면 완화되나 하한은 미확정. n_drives 상한 71은 임의 가정(λ, β_down, BW) 기반 — 멀티캐스트 지원·다중 root complex 구성 실측 필요.
  3. (신규) 제어 평면 에너지 50–100W: MinHash 코어 수(Turn 11)·코어당 W가 SIMD/구현·CPU 세대에 민감. SIMD·GPU 서명·sketch로 낮출 여지 — content-aware batching on/off의 latency(BRF) vs 에너지 파레토 실측 필요.
  4. (정정) per-doc slack τ_k^safe: order-statistic 하한은 증명됨(safe)이나, slack_d(ξ_δ, η_num)의 실제 크기는 PLAID approx vs full MaxSim 오차·이종 hardware 재현오차 실측에 의존 — owner_class별 slack calibration 필요.
  5. (신규) centroid 복제 낭비: drive마다 centroid table 복제 시 stage-1 scoring Q·nprobe·E가 n_drives배 중복. centroid sharding 가능성은 IVF 구조·pid→centroid 분포에 의존 — centroid partition 전략 실측 필요.
  6. topology 시너지의 warm-tier hotspot: L2 cutoff 아래 warm tier가 새 channel 병목이 되는지는 접근 분포 꼬리 형태(Turn 1 Zipf α)에 좌우 — warm-tier load skew 시뮬 필요.
  7. RCU centroid-snapshot 비용: centroid 재학습 rebuild 빈도·centroid table 이중화 크기 미확정. DELETE-only workload면 무시 가능, centroid-rebuild 포함 workload면 usable_L2 추가 잠식 — rebuild 종류별 usable_L2 페널티 실측 필요.
