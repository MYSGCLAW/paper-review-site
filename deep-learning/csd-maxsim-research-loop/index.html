선행 research 요약 재료: Document locality workload

CSD Resident Embedding을 위한 Document Locality 측정 설계 — 워크로드 선정, Trace 생성, 지표 정의

한 줄 요약

PLAID engine + ColBERTv2 index + MS MARCO passage corpus 조합을 워크로드로 선정하고, query별로 residual decompression 단계에서 실제 읽히는 passage embedding block 집합을 trace로 남겨, access frequency / reuse distance / working-set / hot coverage를 계산하면 CSD residency hit ratio와 PCIe traffic 절감량을 곧바로 유도할 수 있다.

수집 범위

  • 포함: ColBERT/ColBERTv2/PLAID/MS MARCO/BEIR 중 워크로드 선정, query stream → access trace 생성 절차와 record schema, locality metric 수식, CSD residency benefit 추정 절차, scheduling 인계 factor 정의.
  • 제외: end-to-end 성능 모델, CSD 하드웨어 설계 자체, retrieval 품질(MRR/nDCG) 분석.
  • 자료 상태(중요): references/manifest.json의 5개 문서는 모두 "URL only; content was not fetched automatically" 상태이고, 이번 세션에서 WebFetch/WebSearch 권한이 거부되어 원문을 직접 열람하지 못했다. 따라서 5개 문서는 전부 "URL 메타데이터만 확인"이며, 본문의 기술 내용은 해당 논문·코드베이스에 대한 배경지식으로 작성하고 수치는 추정/확인 필요로 구분했다. 첨부 repo도 없음(repo-files.txt: "No repository attached").

핵심 요약

  1. 최종 추천 워크로드: PLAID(실행 엔진) + ColBERTv2(index 포맷) + MS MARCO v1 passage(약 8.8M passages, 배경지식). BEIR는 query 수가 적어 보조 검증용, ColBERT v1은 index가 비압축이라 CSD 시나리오와 거리가 있다.
  2. "접근"의 정의가 결과를 좌우한다: PLAID는 후보를 centroid 점수만으로 걸러내므로, storage에서 실제로 읽는 것은 최종 단계에서 residual을 decompress하는 passage 집합뿐이다. 이 집합(이하 H2)을 byte 단위 접근으로 정의하고, 1단계 candidate 집합(H1)은 scheduling feature용으로 별도 기록한다.
  3. 공개 query set은 dedup되어 있다(배경지식): 동일 query 반복이 없으므로 문서 재사용은 서로 다른 query의 candidate 중복에서만 나온다. 따라서 S_uniq(고유 query 스트림, locality 하한) 과 S_zipf(Zipf 재표집 스트림, 운영 환경 근사) 두 스트림을 모두 측정해야 주장이 방어된다 — 본 설계의 최대 쟁점.
  4. 지표는 6종으로 충분하다: access frequency, reuse ratio, temporal reuse distance(LRU stack distance), working-set size(Denning WSS + Heaps 지수), hot coverage(Lorenz/CDF), Zipf α. 문서 크기가 token 수에 비례하므로 전 지표를 byte-weighted 버전과 병행 계산한다.
  5. PCIe 절감 주장에 필요한 최소 metric은 두 개다: baseline bytes/query 분포와 byte-weighted residency hit ratio HR_bytes(C). CSD가 hit을 내부에서 처리하면 절감율 = HR_bytes(C)가 그대로 성립한다.
  6. 규모 감(추정): ColBERTv2 2-bit 압축 시 token embedding당 ~36B, passage당 평균 ~2.5KB. PLAID k=100 설정에서 query당 수백 개 passage를 decompress한다고 알려져 있어 압축 상태로 ~0.6MB/query, fp16 원본 전송이면 ~4.5MB/query 수준 — 1k QPS면 PCIe Gen4 x4 대역폭(~8GB/s, 배경지식)의 상당 부분을 embedding 이동이 차지한다.
  7. static hot-set 평가는 trace의 train/test 분할이 필수다. 전체 trace로 hot set을 뽑아 같은 trace로 hit ratio를 재면 oracle leakage다.
  8. scheduling에 넘길 핵심 online feature는 candidate set locality ρ_q(후보 중 CSD resident 비율) 이다. ρ_q는 stage-1 직후(어떤 residual도 읽기 전에) 계산 가능하므로 GPU/CSD dispatch의 자연스러운 결정점이 된다.
  9. PLAID의 단계별 파라미터(nprobe, ndocs, 최종 decompress 수)와 index 수치는 원문 미열람 상태의 배경지식이므로, 본문 말미의 확인 필요 리스트를 논문·코드로 검증한 뒤 수치를 확정해야 한다.

프로토콜 / 동작 흐름

participant M as CSD DRAM (resident set) participant N as CSD NAND (residual blocks)

  • 상태/객체 순서: query handle(qid, q_idx) → centroid score 행렬(임시) → H1 candidate pid 리스트 → pruning 후 H2 pid 리스트 → residual block read 요청(pid→(chunk, offset, len) 변환은 doclens 누적합) → decompress된 embedding(임시) → top-k 결과로 완료.
  • CSD 경계: baseline(일반 SSD)에서는 H2 residual bytes가 PCIe를 건너 host/GPU로 간다. CSD 설계에서는 H2 읽기·decompress·MaxSim이 장치 내부에서 끝나고 PCIe에는 Q(내려감)와 top-k(올라옴)만 흐른다. 즉 H2 집합의 byte 합이 곧 절감 대상 traffic이다.
  • centroid table과 IVF/doclens는 크기가 작아(수십~수백 MB 추정) 무조건 resident로 두고, locality 측정 대상에서 제외한다(쟁점 2 참조).

2. Trace 생성 방법

passage(~2.5KB 추정) < 4KB이므로 한 block에 여러 pid가 섞인다 → pid 단위와 block 단위 지표가 달라진다(block 단위는 spatial locality·배치 순서 효과 포함). query 내에서 block id는 dedupe한다. Trace schema:

  • S_uniq: dev-full(또는 train에서 표본) query를 무작위 순서로 1회씩 재생. 반복이 없으므로 corpus 내재 locality(후보 중복)만 측정하는 하한.
  • S_zipf: query pool에서 Zipf(α_q) 인기 분포로 N개(예: 1M)를 복원추출해 반복이 있는 스트림을 합성. α_q는 공개 검색 로그의 query 빈도 왜도에서 가져오되(배경지식상 0.7~1.0 부근, 확인 필요) 민감도 sweep(α_q ∈ {0.6, 0.8, 1.0})으로 방어한다.
  • 각 query에 q_idx(0..N-1, 논리 시간축)와 합성 t_arrival(Poisson 도착 등)을 부여한다. locality 계산은 q_idx 기준, latency 모델만 t_arrival 사용.
  • H1: stage-1 candidate generation 직후의 candidate pid 집합(+ probed centroid ids).
  • H2: residual decompression에 들어가는 pid 집합 — 실제 storage read. 이것이 locality의 1차 접근 정의.
  • H3: 최종 top-k pid.
  • hook 위치는 ColBERT repo의 searcher/index-scorer 경로(배경지식상 Searcher.search → candidate generation → filter → residual lookup 순, 정확한 함수명 확인 필요).

queries 테이블 (query당 1행)

accesses 테이블 ((q_idx, pid)당 1행; H2 기준이 기본, stage_max로 H1도 표현)

docs 테이블 (정적, pid당 1행)

이 schema로 §3의 모든 지표가 trace 재스캔 없이 groupby/집계만으로 계산되고, §5의 scheduling feature도 같은 field에서 파생된다.

CSD MaxSim Offload의 GPU 대비 MAC/PE/FLOPS Break-even 분석

한 줄 요약

Candidate embedding이 GPU-resident가 아닌 한 GPU MaxSim은 PCIe-bound이므로, CSD는 GPU의 FLOPS가 아니라 PCIe 전달률만 이기면 되고, 그 조건은 N_mac ≥ Q · BW_pcie / (b_e · f_csd · u) — fp16, Q=32, PCIe Gen4 x16 기준 약 630개 MAC(≈1 TFLOP/s, A100 peak의 0.3%) 수준이다. 단, 이는 BW_int ≥ BW_pcie(내부 read bandwidth 조건)와 "압축 전송 baseline과의 비교"라는 두 개의 더 어려운 전제를 동반한다.

수집 범위

  • 포함: MaxSim의 MAC/FLOP 연산 모델, GPU 실행 경로(T_gpu)와 CSD 실행 경로(T_csd)의 latency equation, break-even inequality와 required MAC/PE/OPS 유도, sanity-check 수치 예시, sensitivity sweep 설계.
  • 제외: 문서 로컬리티/캐시 히트율 자체 분석(job scope에서 명시적으로 제외), query encoder(BERT forward)의 비용 — 양 경로에 공통이므로 소거, candidate generation(ANN/centroid 단계)의 자체 비용 — 단 배치(placement) 쟁점으로만 언급.

핵심 요약

  1. MaxSim은 batched small GEMM + row-wise max: FLOP_total ≈ 2·B·N·Q·D·E, max-reduction은 GEMM 대비 1/(2E) ≈ 0.4% (E=128)로 무시 가능하다.
  2. GPU 경로의 arithmetic intensity는 AI = 2Q/b_e (query는 on-chip 상주, fused kernel 가정). fp16·Q=32이면 32 FLOP/B로 A100 ridge(≈153 FLOP/B)에 한참 못 미쳐 HBM-bound이고, 데이터가 PCIe를 건너는 순간 BW_pcie ≪ BW_hbm(30~80배)이므로 항상 PCIe-bound가 된다.
  3. 따라서 break-even의 본질은 "CSD compute vs GPU compute"가 아니라 "CSD compute vs PCIe 전달률": required OPS = (Q/b_e)·BW_pcie [MAC/s]. 여기서 B, N, D, E가 전부 소거된다 — break-even MAC 수는 워크로드 크기와 무관하고 Q, precision, 링크 속도, clock, utilization에만 의존한다.
  4. 필요 MAC 수는 작다: N_mac* = Q·BW_pcie/(b_e·f·u) → fp16, f=1 GHz, u=0.8일 때 Gen4 x4에서 158개, Gen4 x16에서 630개, Gen5 x16에서 1,260개. SmartSSD급 FPGA(KU15P, DSP slice 약 2천 개 — 배경지식, 확인 필요)로도 도달 가능한 규모다.
  5. 진짜 병목은 두 번째 필요조건 BW_int ≥ BW_pcie: 단일 drive의 NAND 내부 대역폭(~7–16 GB/s)은 GPU x16 링크 유효 대역폭(25–54 GB/s)보다 낮다. 단일 CSD는 warm host DRAM baseline에 지고, cold data / multi-drive scale-out / HBM·host DRAM capacity 초과 corpus에서만 이긴다.
  6. Result traffic은 소거 가능: score만 반환 시 D·E·b_e/b_s = 8192배, top-k 반환 시 10⁵배까지 egress가 줄어 T_res는 µs 단위가 된다. CSD의 PCIe 비용은 사실상 0으로 모델링해도 된다.
  7. 최강 GPU baseline은 raw fp16 전송이 아니라 압축 전송(ColBERTv2 residual compression, ~36 B/vector — 배경지식, 확인 필요): 전송량이 ~7배 줄면 required N_mac*도 ~7배(Gen4 x16 기준 ~4.5K MAC) 커진다. CSD 제안의 공정한 비교 대상은 PLAID류 압축 경로다.
  8. GPU-resident(HBM 상주) baseline을 이기는 것은 불가능에 가깝다(required OPS ≈ 65 TFLOP/s + BW_int ≥ 2 TB/s). CSD 논거는 latency가 아니라 capacity(fp16 full index ~150+ GB > HBM 80 GB), cost, host 자원 절감으로 옮겨가야 한다.

세부 정리

1. MaxSim Operation Model

MAC / FLOP: MAC_total = B · N · Q · D · E (dot-product 곱셈-누산) FLOP_gemm = 2 · B · N · Q · D · E FLOP_red = B · N · (Q·(D−1) compare + (Q−1) add) ≈ B·N·Q·D FLOP_total ≈ 2·B·N·Q·D·E · (1 + 1/(2E)) ≈ 2·B·N·Q·D·E [E=128 → red 비중 0.4%] Intermediate matrix: candidate당 Q×D score matrix. Materialize 시 B·N·Q·D·b_s bytes (예: N=4096, Q=32, D=128 → 67 MB, 입력의 절반 규모) — fused kernel이면 register/shared memory에서 소거된다. Max-reduction은 FLOP으로는 무시 가능하지만, unfused 구현에서는 HBM traffic을 ~2배로 만들 수 있는 memory-side 항이다. AI = FLOP/Bytes_D = 2Q/b_e [FLOP/B] ← N, D, E에 무관, Q와 precision만의 함수

2. GPU Latency Model

경로: NAND → (SSD 링크) → host DRAM → (GPU 링크) → HBM → SM → score 반환. T_pcie = Bytes_in / (ε_pcie · BW_pcie), BW_pcie = min(n_ssd·BW_ssd_link, BW_gpu_link, BW_host) T_comp = max( FLOP_total/(F_gpu·η_gpu), Bytes_HBM/BW_hbm ) [roofline] T_gpu = T_fix,gpu + T_pcie + T_comp (non-overlapped) T_gpu ≈ T_fix,gpu + max(T_pcie, T_comp) (chunked pipeline)

  • ε_pcie ≈ 0.8–0.9 (protocol overhead, 대형 DMA 기준 — 배경지식), T_fix,gpu ≈ 10–30 µs (kernel launch + driver — CUDA Guide 계열 배경지식).
  • HBM constraint 두 가지: (bandwidth) Bytes_HBM ≥ Bytes_D — fused면 1회 read, unfused면 score matrix 왕복이 추가돼 ~2배. (capacity) resident baseline이 성립하려면 index size ≤ C_hbm — fp16 MS MARCO full index ~150+ GB(배경지식, 확인 필요)는 80 GB HBM에 안 들어가고, ColBERTv2 압축 index ~20–36 GB는 들어간다.
  • regime 판별: compute-bound 조건은 AI > F_gpu·η/BW_hbm (A100 fp16 ridge ≈ 153 FLOP/B — 배경지식). Q=32·fp16(AI=32)는 미달 → resident면 HBM-bound. 비-resident면 BW_pcie(≤63 GB/s) ≪ BW_hbm(2–3.35 TB/s)이므로 무조건 PCIe-bound: T_gpu ≈ Bytes_D/(ε·BW_pcie). 이때 GPU의 F_gpu는 1차 근사에서 결과에 등장하지 않는다.

3. CSD Latency Model

경로: NAND → internal channel → on-drive PE array → local max/top-k → 결과만 PCIe. Bytes_int = B·N·D·E·b_e (internal read bytes, GPU 입력과 동일) OPS_csd = N_mac · f_csd · u [MAC/s] (= 2·N_mac·f_csd·u FLOP/s) T_mac = MAC_total / OPS_csd = B·N·Q·D·E / (N_mac·f_csd·u) T_csd = T_fix,csd + max(T_read, T_mac) + Bytes_res/BW_link

  • N_mac: MAC unit(PE) 수, f_csd: clock, u: utilization(스트리밍 정렬/뱅크 충돌/tail 포함 유효 계수). local max-reduction은 MAC 뒤 comparator를 파이프라인에 붙이면 추가 cycle이 없고 u에 흡수된다. top-k heap은 O(N log k) scalar 연산으로 무시 가능.
  • 필요한 on-chip 상태는 query matrix Q·E·b_e = 8 KB + doc당 Q개 partial-max register뿐 — score matrix를 만들지 않는다.
  • Result traffic 축소율: 전체 score 반환 시 R = D·E·b_e/b_s = 8192배 (D=E=128, fp16), top-k(k=100, N=4096) 반환 시 134 MB → 800 B ≈ 1.7×10⁵배. T_res는 Gen4 x4에서도 µs 단위 → 모델에서 소거 가능. 이것이 job 질문 "score/top-k 반환 시 traffic 모델 변화"의 답: CSD egress 항이 상수(≈0)로 바뀌고, break-even은 ingress(BW_int)와 compute(OPS_csd)만의 문제가 된다.
  • T_fix,csd: NVMe/CS command 왕복 + FW dispatch, 수십~수백 µs 추정(확인 필요 — SNIA CS API 문서와 실측 필요).

4. Break-even MAC/PE 조건

첫 턴이므로 두 선행 research를 요약·연결한 뒤, 두 연구를 하나의 방정식으로 잇는 새 결과를 도출한다. 핵심 신규 기여는 (1) locality 지표 HR_bytes(B)와 break-even이 만나는 임계 residency HR* ≈ 0.61–0.74, (2) NAND가 압축 residual을 저장한다는 사실에서 나오는 CSD compute 요구치 재보정(630 → ~1,700 MAC), (3) HBM/host-DRAM/CSD-DRAM/NAND 4단 캐시 모델과 그 위의 dispatch rule이다. 산출물 1~5를 모두 채운다.

---

선행 research 요약과 연결고리 (첫 턴 필수)

A. Document locality 요약 (6줄)

PLAID(엔진) + ColBERTv2(압축 index) + MS MARCO v1(8.8M passages, query pool)을 워크로드로 고정한다. "storage 접근"은 centroid pruning 이후 residual을 실제로 decompress하는 pid 집합 H2로 정의하고, candidate 집합 H1은 scheduling feature로 따로 기록한다. 공개 query는 dedup되어 있으므로 S_uniq(locality 하한)와 S_zipf(운영 근사) 두 스트림을 모두 재생한다. 지표 6종(access freq, reuse ratio, LRU stack distance, WSS, hot coverage, Zipf α)을 byte-weighted 병행으로 계산하고, 최종적으로 PCIe 절감 주장은 HR_bytes(B) 하나로 환원된다. 규모 감(추정): 압축 ~36B/token, passage ~2.5KB, query당 수백 passage → 압축 ~0.6MB/query.

B. GPU/CSD break-even 요약 (7줄)

MaxSim은 FLOP≈2·B·N·Q·D·E의 batched small GEMM + row-wise max이고, arithmetic intensity AI=2Q/b_e(fp16·Q=32 → 32 FLOP/B)라 GPU는 resident면 HBM-bound, 비-resident면 항상 PCIe-bound다. 따라서 break-even의 본질은 "CSD vs GPU FLOPS"가 아니라 "CSD vs PCIe 전달률"이고, 조건은 N_mac* = Q·BW_pcie/(b_e·f·u)로 B,N,D,E가 전부 소거된다(fp16 기준 Gen4 x16 ≈ 630 MAC). 더 어려운 두 전제는 BW_int ≥ ε·BW_pcie(내부 대역폭)와 "raw fp16이 아닌 압축 전송 baseline과의 공정 비교"다. Result traffic(score/top-k)은 10⁴~10⁵배 축소로 사실상 0. GPU-HBM-resident baseline은 이기기 어렵고, CSD 논거는 latency보다 capacity/cost/host 자원으로 옮겨야 한다.

연결고리 (핵심)

두 연구는 같은 객체(H2 = residual block)를 서로 다른 층에서 본다. A는 그것을 어느 캐시 층에 두느냐(residency HR_bytes), B는 그것을 어느 경로로 옮기고 어디서 계산하느냐(PCIe vs 내부)로 본다. 접점은 CSD의 유효 내부 read 대역폭이 residency에 의해 결정된다는 것이다:

BW_int_eff(HR) = 1 / [ HR/BW_DRAM_int + (1−HR)/BW_NAND_int ]

즉 A의 산출물 HR_bytes(B)가 B의 필요조건 BW_int ≥ ε·BW_pcie의 좌변을 직접 만든다. 이것이 이번 턴이 두 연구를 하나로 묶는 지점이다.

새로 추가한 사실

사실 1 — locality와 break-even을 잇는 임계 residency HR*

CSD가 warm host-DRAM baseline을 read-BW에서 이기려면 BW_int_eff(HR) ≥ ε·BW_pcie이어야 하므로, 위 식을 풀면 locality 지표에 대한 명시적 임계값이 나온다:

HR* = ( 1/BW_NAND_int − 1/(ε·BW_pcie) ) / ( 1/BW_NAND_int − 1/BW_DRAM_int )

가정값(모두 배경지식/실측 필요) BW_DRAM_int=40, BW_NAND_int=12 GB/s에서:

  • ε·BW_pcie = 21 GB/s → HR* ≈ 0.61
  • ε·BW_pcie = 25 GB/s(research B의 GPU-A 예시) → HR* ≈ 0.74

해석: A가 측정한 HR_bytes(B) 곡선이 B* GB 지점에서 0.61~0.74를 넘으면 그 순간 단일 CSD가 warm 경로를 이긴다. 특수 케이스로 BW_NAND_int ≥ ε·BW_pcie(멀티채널 CSD)면 HR* ≤ 0 → residency 없이도 항상 이긴다(research B point 5의 locality 버전). 이 임계식이 산출물 2의 곡선과 산출물 3의 부등식을 하나로 묶는다.

사실 2 — CSD compute 요구치 재보정: 630 → ~1,700 MAC

research B의 Bytes_int = B·N·D·E·b_e(fp16)는 NAND가 fp16을 저장한다는 암묵 가정이지만, ColBERTv2 NAND에는 압축 residual(~36B/token)이 있다. 따라서 (i) 내부 read 바이트는 fp16의 ~1/7로 더 작고(research B의 T_read는 ~7× 비관), (ii) MAC은 여전히 decompress된 E=128 차원 위에서 돌므로 byte당 MAC 밀도가 7× 높다. CSD가 내부 read rate를 따라잡으려면:

N_mac(내부) = Q·E·BW_int_eff / (b_tok_comp · f · u)
            = 32·128·12e9 / (36 · 1e9 · 0.8) ≈ 1,707 MAC

즉 압축 데이터를 내부 대역폭으로 decompress+MAC하려면 630이 아니라 ~1,700 MAC이 필요하다(48×48 systolic array ≈ 2,304 MAC로 충족, 여전히 소형). research B point 7의 "~7×"를 정확한 값으로 확정한 것이며, 630은 fp16 PCIe baseline 전용 수치임을 명시해야 한다.

사실 3 — 4단 캐시 모델 (runtime 구조의 핵심)

두 경로는 서로 다른 캐시로 이득을 본다:

| 층 | 이름 | GPU 경로가 쓰는가 | CSD 경로가 쓰는가 | |----|------|:--:|:--:| | L0 | GPU HBM (index pin) | ✅ (fits면) | ❌ | | L1 | host DRAM page cache | ✅ (SSD read 절감) | ❌ | | L2 | CSD DRAM (resident set, ρ_q) | ❌ | ✅ | | L3 | NAND | fallback | fallback |

중요 정정: host page cache(L1)는 SSD→host read만 절감하고 host→GPU PCIe hop은 여전히 지불한다. CSD는 그 PCIe hop 전체를 없앤다(in-device MaxSim). 따라서 fully warm host page cache를 상대로도 CSD의 PCIe 이득은 유지된다 — CSD의 진짜 경쟁자는 L1이 아니라 L0(HBM-resident index)이다.

사실 4 — MS MARCO v1은 capacity 논거에 약한 워크로드

ColBERTv2 압축 index(MS MARCO v1) ~20GB는 A100 80GB HBM에 그대로 들어간다(L0 feasible). 즉 v1에서는 최강 baseline이 HBM-resident이고 CSD는 latency로 못 이긴다(research B의 결론과 정합). capacity 논거를 세우려면 HBM을 초과하는 corpus가 필요하다:

MS MARCO v2 ≈ 138M passages ≈ 15.7× v1 → 압축 index ≈ 314 GB ≫ 80 GB HBM

→ capacity-motivating 워크로드로 MS MARCO v2(또는 web-scale)를 A의 워크로드 축에 추가할 것을 권고한다(민감도 실험이 아니라 논거의 전제).

산출물 5 — scheduling factor & decision rule 초안

online feature (stage-1 직후, 어떤 residual도 읽기 전 계산 가능): n_H2, bytes_c(q), ρ_q(L2 hit 비율), w_q(L1 warm 비율), Q_gpu/Q_csd(큐 깊이).

def route(q):
    # 예상 서비스 시간
    t_gpu = T_fix_gpu + bytes_c(q)/(eps*BW_pcie) \
            + (0 if w_q(q)>θ_warm else bytes_c(q)/BW_ssd)      # cold면 SSD read 가산
    bw_eff = 1/(rho_q(q)/BW_DRAM_int + (1-rho_q(q))/BW_NAND_int)
    t_csd = T_fix_csd + bytes_c(q)/bw_eff
    # 큐 대기 포함 유효 완료시각
    if Q_csd*t_csd_avg + t_csd < Q_gpu*t_gpu_avg + t_gpu:
        return "CSD"
    return "GPU"

decision rule 요약: (a) w_q 높음(host warm) → GPU 유리, (b) ρ_q > HR* & cold → CSD 유리, (c) 큐 불균형 시 load-balancing 항이 지배. batch 정책: latency-critical 단일 query는 GPU(warm), throughput-oriented 스트림은 CSD로 batch 합류 → T_fix,csd 상각. 이 rule은 산출물 2 스키마의 ρ_q·w_q에서 바로 파생된다.

관련 맥락 / 배경 설명

  • 왜 4단 캐시가 runtime 구조의 본질인가: MaxSim offload는 "compute를 어디서 하느냐"가 아니라 "데이터가 이미 어느 층에 있느냐"가 승부를 가른다. GPU 경로는 L0/L1로, CSD 경로는 L2로 이득을 보므로, 스케줄러는 두 캐시 상태를 동시에 봐야 한다. 이것이 단일 hit-ratio 최적화(전통 캐시 문헌)와 다른 지점이다.
  • 큐잉 관점: t_arrival(Poisson)을 부여하면 GPU·CSD-farm은 서비스 시간이 ρ_q에 의존하는 이질적 2-server가 된다. state-dependent routing이라 M/G/1 닫힌형이 어렵고 trace 기반 시뮬이 현실적이다.
  • Result egress가 0인 이유(재확인 맥락): CSD는 score/top-k만 반환(≤16KB) → PCIe egress가 상수 → break-even이 ingress(BW_int)와 compute(N_mac) 2변수 문제로 축소. 이 단순화가 산출물 3 부등식이 깔끔한 근거다.

Turn 2 / Codex

이번 턴의 확장 요약

이번 턴은 runtime 내부 구조와 scheduling factor를 중심으로 확장한다. 핵심 추가점은 (1) MaxSim offload runtime을 query encoding → candidate planning → routing → execution → merge의 operator graph로 분해하고, (2) whole-query offload보다 현실적인 shard-level hybrid scheduling을 제안하며, (3) ρ_q를 단일 hit ratio가 아니라 resident byte, miss byte, shard imbalance, queue delay가 결합된 dispatch 변수로 재정의한다. 또한 이전 턴의 ρ_q 계산 시점 표현은 약간 수정이 필요하다.

새로 추가한 사실

1. Whole-query routing보다 shard-level routing이 더 자연스럽다

기존 rule은 query 전체를 GPU 또는 CSD 중 하나로 보내는 형태였다. 하지만 PLAID/ColBERTv2 index는 pid/block 단위로 shard가 가능하므로 runtime은 query q의 H2 집합을 다음처럼 쪼갤 수 있다.

H2(q) = H2_gpu(q) ∪ H2_csd_1(q) ∪ ... ∪ H2_csd_m(q)

각 CSD는 자기 shard의 passage score 또는 local top-k만 반환하고, host가 최종 top-k를 merge한다. 이 방식의 장점은 ρ_q가 낮은 query에서도 resident/hot shard만 CSD로 보내고, miss-heavy shard는 GPU로 보내는 hybrid execution이 가능하다는 점이다.

단, document 하나의 token embedding은 한 실행 주체에 온전히 있어야 한다. MaxSim은 query token별 max를 낸 뒤 sum하기 때문에, 문서 token을 여러 장치에 쪼개면 partial max vector를 merge해야 한다.

score(q,d) = Σ_i max_j <q_i, d_j>

pid 단위 shard는 merge가 단순하다. token-level shard는 각 query token i에 대한 partial max를 모아야 하므로 runtime 복잡도가 커진다.

2. ρ_q는 hit ratio가 아니라 service-time predictor여야 한다

단순히 “H2 중 resident 비율”만 보면 scheduling에 부족하다. 같은 ρ_q=0.8이어도 miss byte가 큰 문서에 몰려 있으면 CSD latency가 커진다. 따라서 dispatch feature는 최소한 byte-weighted로 정의해야 한다.

ρ_bytes(q) = Σ_{d∈H2(q)∩L2} bytes_c(d) / Σ_{d∈H2(q)} bytes_c(d)

miss_bytes(q) = Σ_{d∈H2(q)\L2} bytes_c(d)

imbalance(q) = max_s bytes_s(q) / mean_s bytes_s(q)

CSD service time은 평균 hit ratio보다 다음 형태가 더 직접적이다.

T_csd(q) =
  T_fix,csd
  + max_s [
      bytes_hit_s(q)  / BW_DRAM_s
    + bytes_miss_s(q) / BW_NAND_s
    + MAC_s(q)        / OPS_s
    ]
  + T_merge(q)

즉 runtime scheduler는 ρ_q 하나가 아니라 miss_bytes, shard별 skew, queue depth를 같이 봐야 한다.

3. CSD offload의 runtime 병목은 compute보다 “tail shard”일 수 있다

multi-drive CSD에서 query latency는 평균 drive가 아니라 가장 늦은 drive가 결정한다.

T_query_csd = max_s T_csd_s(q) + T_host_merge

따라서 hot documents가 특정 shard에 몰리면 전체 HR_bytes가 높아도 p99 latency가 나빠진다. residency policy도 global top-frequency만으로 정하면 안 되고, shard별 capacity와 load를 제약으로 둬야 한다.

maximize   Σ_d x_d · f_test(d) · bytes(d)
subject to Σ_{d∈shard s} x_d · bytes(d) ≤ B_s    for each shard s
           expected_load_s ≤ load_cap_s
           x_d ∈ {0,1}

실용적으로는 shard별 STATIC-HOT + periodic rebalancing부터 시작하면 된다.

Runtime operator graph

Q0  Query encoder on GPU/CPU
Q1  Centroid scoring / IVF probe
Q2  H1 candidate list 생성
Q3  H2 planning: pid → block_id, bytes, shard, residency state
Q4  Scheduler decision
Q5a GPU path: residual fetch → decompress → MaxSim

중요한 점은 Q4가 residual read 이전에 있어야 한다는 것이다. 그래야 scheduler가 “읽은 뒤 결정”하는 모순을 피한다.

Corrected scheduling decision point

이전 턴의 “stage-1 직후 ρ_q 계산 가능”은 표현을 더 정확히 해야 한다. H1만 안 상태에서는 최종 residual read 대상인 H2가 아직 확정되지 않을 수 있다. 안전한 정의는 다음이다.

Q4 decision point = H2 pid list가 확정된 직후, residual block read를 시작하기 전

만약 runtime이 더 이른 결정을 원하면 H1 기반 예측치를 따로 둔다.

ρ_hat_q = predicted resident byte ratio over H2, estimated from H1 features
ρ_q     = exact resident byte ratio after H2 planning

초기 구현은 ρ_q 기반 oracle-free scheduler로 충분하고, 이후 ρ_hat_q로 앞당기는 실험을 추가한다.

Scheduling rule v2

def route_query(q):
    plan = build_h2_plan(q)  # pid, bytes, shard, l2_state

    t_gpu = T_fix_gpu \
        + bytes_comp(q) / (eps * BW_pcie) \
        + decomp_mac_gpu(q) \
        + cold_read_penalty(q)

    t_csd = T_fix_csd + max(
        shard.hit_bytes / BW_DRAM_s
        + shard.miss_bytes / BW_NAND_s
        + shard.mac / OPS_s
        + queue_delay_s
        for shard in plan.csd_shards
    ) + T_merge(q)

    if t_csd + risk_margin < t_gpu:
        return "CSD"
    if resident_subset_beneficial(plan):
        return "HYBRID"
    return "GPU"

risk_margin은 p99 보호용이다. CSD 평균 latency가 약간 낮아도 tail shard가 불안정하면 GPU로 보내는 것이 낫다.

관련 맥락 / 배경 설명

CSD MaxSim runtime은 전통적인 accelerator offload와 다르게, 연산량보다 데이터 위치와 이동 경로가 먼저 결정한다. GPU는 HBM 또는 host DRAM page cache가 따뜻할수록 유리하고, CSD는 L2 resident set과 NAND locality가 좋을수록 유리하다.

따라서 scheduler는 “GPU가 빠른가, CSD가 빠른가”를 정적으로 고르는 장치가 아니라, query마다 다음을 추정하는 작은 cost model이어야 한다.

잠재적 오류 또는 확인 필요 지점

  1. ρ_q 계산 시점은 H1 직후가 아니라 H2 planning 직후로 두는 것이 안전하다. H1 기반 조기 dispatch는 예측 문제로 따로 분리해야 한다.
  2. CSD가 local top-k만 반환할 때 exactness가 깨질 수 있다. 각 shard가 k개만 반환하면 global top-k는 보통 맞지만, tie나 score 분포가 평평한 경우 안전하지 않을 수 있다. k_local = α·k sweep이 필요하다.
  3. shard imbalance가 HR_bytes(B) 곡선에 가려질 수 있다. 평균 hit ratio가 높아도 특정 CSD가 hot shard를 독점하면 p99가 나빠진다.
  4. GPU path의 cold_read_penalty는 host page cache, GPUDirect Storage, mmap 방식에 따라 달라진다. trace에는 logical access뿐 아니라 실제 page-cache hit 여부를 별도 계측해야 한다.
  5. CSD fixed cost T_fix,csd가 single-query latency에서 매우 중요하다. batch 없이도 이기는지, batch가 필요한지는 실측으로 분리해야 한다.

다음 에이전트가 이어서 확장할 지점

  1. shard-level trace schema 확장: accesses에 shard_id, block_id, resident_tier, queue_id를 추가하고, imbalance(q)와 per-shard p50/p99 service time을 계산하는 metric table을 완성하라.
  1. exact top-k merge 조건 분석: CSD local top-k 크기 k_local이 global exactness와 traffic에 미치는 영향을 수식화하고, k_local ∈ {k, 2k, 4k, all_scores} sweep 프로토콜을 설계하라.
  1. runtime simulator 설계: Poisson arrival, GPU queue, per-CSD queue, micro-batch timeout을 포함한 discrete-event simulator를 정의하고, 정책 all-GPU, all-CSD, whole-query routing, hybrid shard routing을 p50/p99 latency와 PCIe GB/s로 비교하라.

Turn 3 / Claude

이번 턴의 확장 요약

이번 턴은 CSD 내부 실행 datapath와 배치(batch)·머지(merge) 구조를 파고든다. 핵심 신규 기여는 (1) 동시 쿼리의 candidate 겹침을 단일 내부 스캔으로 흡수하는 batched internal scan과 그 이득을 정량화하는 BRF(batch reuse factor) — 이것이 locality 연구의 Heaps β를 runtime read 절감으로 직접 환산한다, (2) Codex의 k_local 우려 정정: pid-shard에서 k_local = k는 tie와 무관하게 항상 exact이며, 오히려 sharding 정책에 따라 k/m까지 줄일 수 있다(단 hit-ratio 최적 sharding과 상충), (3) Codex의 T_csd 내부 항이 read+compute 합(sum)으로 되어 있어 파이프라인 가정과 모순 — max로 정정하고 DRAM/NAND 병렬 여부라는 모델 분기를 추가, (4) 캐시 admission write가 read datapath와 내부 대역폭을 두고 경합한다는 write-cost-aware residency.

새로 추가한 사실

사실 1 — Batched internal scan: locality를 read 절감으로 바꾸는 runtime 레버

Turn 1·2는 쿼리 하나의 H2를 캐시 층에 두는 관점이었다. 그러나 CSD가 동시 도착 쿼리 배치를 한 번의 내부 스캔으로 처리하면, 서로 다른 쿼리의 candidate 겹침이 곧바로 read 절감이 된다. 배치 Bq개 쿼리의 union을 한 번만 읽고, 각 block을 그 block을 포함한 모든 쿼리에 대해 scoring한다:

읽는 block 집합      = ∪_{q∈batch} H2(q)          (union, 중복 제거)
내부 read bytes      = |∪H2| · s_block            (Σ_q|H2(q)| 아님)
MAC work             = Σ_{q,d∈H2(q)} Q·E          (줄지 않음 — 각 (q,d) 쌍 유지)

여기서 batch reuse factor를 정의한다:

BRF(Bq) = Σ_{q∈batch}|H2(q)| / |∪_{q∈batch} H2(q)|

BRF는 곧 배치 창(window) 안에서의 접근 재사용률이므로 Turn 1의 Heaps 지수 β로 즉시 예측된다. 배치 총 접근 T_b = Bq·n̄_H2, 고유 문서 U_b ≈ κ·T_b^β이면:

BRF(Bq) ≈ T_b^{1−β} / κ

규모 감(추정): n̄_H2=256, Bq=32 → T_b=8192. β=0.8이면 BRF ≈ 8192^0.2/κ ≈ 6.1/κ. κ≈1이면 배치 read를 ~6× 줄인다 — 압축 baseline 이득(~7×)과 맞먹는 크기이며, 이미 locality 연구가 측정하는 β에서 공짜로 나온다. 이것이 runtime이 locality를 활용하는 가장 직접적인 지점이고, Turn 1의 residency(정적 배치)와 별개의 축이다(residency = 어디 두나, batching = 언제 같이 읽나).

per-query 파이프라인 시간은 배치로 다음처럼 바뀐다:

T_perq_csd ≈ max( bytes_perq /(BRF·BW_int_eff),  MAC_perq/OPS )  +  T_fix,csd / Bq

→ read 항은 BRF로, 고정비는 Bq로 나뉜다. compute 항은 배치로 줄지 않으므로, 배치는 read/fixed-bound regime에서만 이득이고 compute-bound(N_mac 부족) CSD에서는 효과가 반감된다. 이는 Turn 1 사실 2(≥1,700 MAC)를 배치 실행에서 다시 요구하는 조건이다.

경계 조건(중요): BRF는 S_uniq(corpus locality만) ≤ 실측 ≤ S_zipf(query 반복 포함) 사이에 있다. dedup된 공개 query에서 concurrent overlap은 corpus locality 하한으로 제한되므로, 배치 이득 주장은 두 스트림 모두에서 측정해야 방어된다.

사실 2 — Hit-ratio 최적 sharding과 merge-traffic 최적 sharding은 상충한다

Codex는 "k_local = k가 tie에서 불안전할 수 있어 α·k sweep 필요"라 했으나 이는 정정 대상이다(아래 오류 1). 대신 진짜 설계 긴장은 sharding 정책에 있다:

  • random/hash sharding: 임의 문서의 global rank r인 문서는 shard-local rank 기대값 ≈ r/m. 따라서 k_local ≈ k/m(+집중 여유)만으로도 global top-k를 exact하게 복원 → merge egress 6·m·k_local ≈ 6k로 m에 무관.
  • locality/hot sharding(hit-ratio를 높이려 hot 문서를 CSD·특정 shard에 몰기): global top-k score가 소수 shard에 집중 → 그 shard의 k_local이 최대 k까지 필요 → merge traffic·머지 지연 증가.

즉 residency hit ratio를 올리는 배치(hot 집중)가 정확한 저비용 merge를 방해한다. 이는 Turn 2 tail-shard 문제(hot shard 독점 → p99 악화)와 같은 뿌리이며, sharding 정책이 hit ratio·tail latency·merge traffic 세 목적을 동시에 건드리는 3자 트레이드오프임을 뜻한다.

사실 3 — Cache admission write가 read datapath와 내부 대역폭을 경합한다

전통 캐시 문헌은 admission을 무료로 본다. 그러나 CSD DRAM(L2) resident set을 갱신하려면 NAND→DRAM write가 필요하고, 이 write는 MaxSim의 read 스트림과 같은 내부 채널·DRAM 포트를 쓴다. 따라서 online LRU/admission을 켜면 유효 read 대역폭이 낮아진다:

BW_int_eff_effective = BW_int_eff · (1 − ζ),   ζ = admission_write_bytes / total_internal_bytes

ζ가 크면 Turn 1의 임계식 BW_int_eff ≥ ε·BW_pcie가 admission 부담만으로 깨질 수 있다. 결론: 정적 STATIC-HOT(admission=0)이 online LRU보다 CSD에 유리할 수 있다 — 캐시 hit ratio는 LRU가 높아도 write 경합이 순이득을 잠식하기 때문. runtime은 admission을 idle window/double-buffer로 read 임계경로 밖에 밀어내거나, 주기적 배치 rebalancing(Turn 2 제안)으로 write를 amortize해야 한다.

더 구체화한 부분

정정된 CSD 실행 시간: sum → max (파이프라인) + 모델 분기

Codex의 per-shard 항 hit/BW_DRAM + miss/BW_NAND + MAC/OPS(전부 합)는 Turn 1의 max(T_read, T_mac) 파이프라인 가정과 모순이며, 최대 2×까지 비관적이다. 정정:

[read 내부 구조 분기]
 (a) 직렬(access-order 스트림, hot/cold 인터리브):
     T_read_s = (hit_bytes + miss_bytes)/BW_int_eff,   BW_int_eff = 조화평균 (Turn 1)
 (b) 병렬(DRAM 포트와 NAND 채널 독립 prefetch·reorder):
     T_read_s = max( hit_bytes/BW_DRAM,  miss_bytes/BW_NAND )   ← 더 빠름

[read↔compute 파이프라인]
 T_csd_s = T_fill + max( T_read_s,  MAC_s/OPS_s ) + T_res(≈0)
 T_fill  = (PE array depth)/f_csd  ≈ 수십~수백 ns  (N 큰 배치에서 무시 가능)

[쿼리 전체]
 T_query_csd = max_s T_csd_s + T_merge      (Turn 2 tail-shard 형태 유지)

→ 모델이 (a)/(b) 중 어느 read 구조를 가정하는지가 HR* 값을 바꾼다. (b) 병렬 read는 HR*를 낮춰(작은 residency로도 CSD 승리) CSD에 유리한 낙관 시나리오다. 어느 쪽인지는 실측 필요(controller 아키텍처 의존).

Exact top-k merge 비용과 근사 recall (Codex 우려 정정·확장)

pid-shard에서 global top-k에 드는 문서 d는 자기 shard 내 자기보다 위인 문서 수 ≤ (d의 global rank) ≤ k이므로 항상 자기 shard의 top-k 안에 있다 — tie 유무 무관, k_local = k면 exact. 따라서 traffic:

exact merge egress = m · k · (b_id + b_score) = m·k·6 B   (uint32 id + fp16 score)
                   = 8·100·6 = 4.8 KB  (m=8, k=100)   ← 무시 가능
merge compute      = k-way heap merge O(m·k·log m)          ← µs 미만

근사 regime(traffic·머지 더 줄이기): k_local = γk. random sharding에서 miss 확률은 global-rank-r 문서의 shard-local rank가 γk를 넘을 확률 = 이항 꼬리. 안전 여유 γ ≥ 1/m + c·√(1/(m·k)) (집중 부등식) → m=8, k=100이면 γ ≈ 0.2에서도 recall≈1. 즉 random sharding은 merge 비용을 5× 더 줄일 여지가 있으나, 사실 2의 hot-sharding과 병용 불가.

Micro-batch 창(window) 최적화: latency–throughput frontier

Codex가 지목한 batch timeout을 정량화한다. 도착률 λ, 창 w, Poisson이면 기대 배치 Bq = λw:

E[per-query 지연] ≈ w/2               (배치 대기)
                 + max(bytes_perq/(BRF(λw)·BW_int_eff), MAC_perq/OPS)   (서비스)
                 + T_fix,csd/(λw)      (고정비 상각)
  • w↑: 고정비 항↓, BRF↑(overlap 축적) → 서비스 효율↑. 그러나 대기 w/2↑.
  • BRF(λw)는 배치가 working set을 덮으면 포화(union → WSS) → knee w* 존재. w > w*는 대기만 늘리고 read 이득 없음.
  • 최적 창: w* ≈ WSS 커버 시간 부근, latency-SLO가 있으면 w ≤ SLO_budget − 서비스로 상한.

on-chip 자원 제약(신규): 배치 실행은 Bq개 query matrix를 on-chip에 상주시켜야 한다 → Bq·Q·E·b_e = Bq·8 KB. Bq=32 → 256 KB, Bq=128 → 1 MB SRAM. SSD 컨트롤러/FPGA BRAM 예산이 Bq의 하드 상한을 준다. 따라서 Bq ≤ min(SLO 허용, SRAM/8KB).

Straggler hedging (tail 대응 runtime 기법)

Turn 2의 tail-shard 문제에 대한 실행 레벨 해법: latency-critical 쿼리는 가장 느린 shard를 GPU에 speculative dual-dispatch(hedge). shard s의 예상 완료가 p99 위험이면 그 shard만 GPU로 복제 실행, 먼저 끝난 결과 채택.

비용 = 중복 shard의 PCIe·GPU 자원 (redundancy overhead)
이득 = T_query = max_s T_s 를 2nd-slowest 수준으로 절단 → p99↓
발동 = predicted T_s > τ_p99  일 때만 (예산 있는 쿼리 한정)

이는 Turn 2 risk_margin을 정적 회피가 아니라 동적 복제로 구현한 것이다.

관련 맥락 / 배경 설명

  • 왜 batching이 residency와 다른 축인가: residency(L2)는 시간 축의 재사용(reuse distance)을 캐시 hit으로 바꾸고, batching은 동시성 축의 재사용(concurrent overlap)을 single-scan으로 바꾼다. 두 축은 곱해진다 — resident hot block을 배치가 한 번 읽으면 그 read조차 DRAM 속도. 그래서 runtime cost model은 HR_bytes와 BRF를 동시에 입력으로 받아야 한다.
  • 캐시 write 경합이 runtime-구조적 신규 지점인 이유: MaxSim datapath와 캐시 관리가 물리적으로 같은 내부 대역폭을 공유하므로, "캐시 정책 최적화"와 "데이터 경로 최적화"가 분리 불가능하다. 이는 host DRAM page cache(write가 별도 DMA)와 근본적으로 다른, CSD 고유의 제약이다.
  • sharding이 세 목적을 동시에 건드림: (i) hit ratio(hot 집중이 유리), (ii) tail latency(분산이 유리), (iii) merge traffic(분산이 유리). random hash sharding은 (ii)(iii)에 좋고 (i)에 나쁘며, hot-affinity sharding은 반대다. runtime은 hybrid(hot 문서는 복제해 여러 shard에 두어 hit는 살리고 부하는 분산)로 절충할 수 있으나 복제는 L2 용량을 소비한다.

잠재적 오류 또는 확인 필요 지점

  1. (정정) Codex의 k_local tie 우려: pid-shard에서 k_local = k는 tie와 무관하게 exact — 위 증명. sweep이 필요한 것은 exactness가 아니라 근사로 traffic을 줄이는 방향(γk < k)이며, 그 하한은 sharding 정책이 정한다. — first-principles 정정, 실측 불필요.
  2. (정정) Codex T_csd 내부 항의 sum: read↔compute는 파이프라인이므로 max, read 내부도 DRAM/NAND 병렬이면 max. sum은 최대 2× 비관. — 모델 정정; (a)직렬/(b)병렬 분기는 아키텍처 실측 필요.
  3. BRF(Bq)의 실제 β와 κ — trace 실측 필요. 특히 concurrent 배치 창 안의 overlap은 전체 스트림 β와 다를 수 있음(짧은 시간창은 hot 문서 편중이 더 심할 수 있어 β 유효값↓, BRF↑). 배치 창 크기별 β를 따로 측정해야 함.
  4. admission write 계수 ζ와 CSD DRAM write 대역폭 — 실측 필요. double-buffer로 read 임계경로 밖에 밀 수 있는지가 사실 3의 순이득을 좌우.
  5. on-chip query buffer Bq·8KB가 대상 CSD(SmartSSD KU15P BRAM/URAM ≈ 수 MB — 배경지식, 확인 필요)에서 실제 Bq 상한을 얼마로 만드는지 — 스펙 확인 필요.
  6. hedging의 GPU 복제가 PCIe·GPU 큐를 얼마나 오염시키는지 — p99 이득 대비 평균 throughput 손실은 시뮬 실측 필요.
