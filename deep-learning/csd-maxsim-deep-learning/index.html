CSD Resident Embedding을 위한 Document Locality 측정 설계 — 워크로드 선정, Trace 생성, 지표 정의

한 줄 요약

PLAID engine + ColBERTv2 index + MS MARCO passage corpus 조합을 워크로드로 선정하고, query별로 residual decompression 단계에서 실제 읽히는 passage embedding block 집합을 trace로 남겨, access frequency / reuse distance / working-set / hot coverage를 계산하면 CSD residency hit ratio와 PCIe traffic 절감량을 곧바로 유도할 수 있다.

수집 범위

  • 포함: ColBERT/ColBERTv2/PLAID/MS MARCO/BEIR 중 워크로드 선정, query stream → access trace 생성 절차와 record schema, locality metric 수식, CSD residency benefit 추정 절차, scheduling 인계 factor 정의.
  • 제외: end-to-end 성능 모델, CSD 하드웨어 설계 자체, retrieval 품질(MRR/nDCG) 분석.
  • 자료 상태(중요): references/manifest.json의 5개 문서는 모두 "URL only; content was not fetched automatically" 상태이고, 이번 세션에서 WebFetch/WebSearch 권한이 거부되어 원문을 직접 열람하지 못했다. 따라서 5개 문서는 전부 "URL 메타데이터만 확인"이며, 본문의 기술 내용은 해당 논문·코드베이스에 대한 배경지식으로 작성하고 수치는 추정/확인 필요로 구분했다. 첨부 repo도 없음(repo-files.txt: "No repository attached").

핵심 요약

  1. 최종 추천 워크로드: PLAID(실행 엔진) + ColBERTv2(index 포맷) + MS MARCO v1 passage(약 8.8M passages, 배경지식). BEIR는 query 수가 적어 보조 검증용, ColBERT v1은 index가 비압축이라 CSD 시나리오와 거리가 있다.
  2. "접근"의 정의가 결과를 좌우한다: PLAID는 후보를 centroid 점수만으로 걸러내므로, storage에서 실제로 읽는 것은 최종 단계에서 residual을 decompress하는 passage 집합뿐이다. 이 집합(이하 H2)을 byte 단위 접근으로 정의하고, 1단계 candidate 집합(H1)은 scheduling feature용으로 별도 기록한다.
  3. 공개 query set은 dedup되어 있다(배경지식): 동일 query 반복이 없으므로 문서 재사용은 서로 다른 query의 candidate 중복에서만 나온다. 따라서 S_uniq(고유 query 스트림, locality 하한) 과 S_zipf(Zipf 재표집 스트림, 운영 환경 근사) 두 스트림을 모두 측정해야 주장이 방어된다 — 본 설계의 최대 쟁점.
  4. 지표는 6종으로 충분하다: access frequency, reuse ratio, temporal reuse distance(LRU stack distance), working-set size(Denning WSS + Heaps 지수), hot coverage(Lorenz/CDF), Zipf α. 문서 크기가 token 수에 비례하므로 전 지표를 byte-weighted 버전과 병행 계산한다.
  5. PCIe 절감 주장에 필요한 최소 metric은 두 개다: baseline bytes/query 분포와 byte-weighted residency hit ratio HR_bytes(C). CSD가 hit을 내부에서 처리하면 절감율 = HR_bytes(C)가 그대로 성립한다.
  6. 규모 감(추정): ColBERTv2 2-bit 압축 시 token embedding당 ~36B, passage당 평균 ~2.5KB. PLAID k=100 설정에서 query당 수백 개 passage를 decompress한다고 알려져 있어 압축 상태로 ~0.6MB/query, fp16 원본 전송이면 ~4.5MB/query 수준 — 1k QPS면 PCIe Gen4 x4 대역폭(~8GB/s, 배경지식)의 상당 부분을 embedding 이동이 차지한다.
  7. static hot-set 평가는 trace의 train/test 분할이 필수다. 전체 trace로 hot set을 뽑아 같은 trace로 hit ratio를 재면 oracle leakage다.
  8. scheduling에 넘길 핵심 online feature는 candidate set locality ρ_q(후보 중 CSD resident 비율) 이다. ρ_q는 stage-1 직후(어떤 residual도 읽기 전에) 계산 가능하므로 GPU/CSD dispatch의 자연스러운 결정점이 된다.
  9. PLAID의 단계별 파라미터(nprobe, ndocs, 최종 decompress 수)와 index 수치는 원문 미열람 상태의 배경지식이므로, 본문 말미의 확인 필요 리스트를 논문·코드로 검증한 뒤 수치를 확정해야 한다.

프로토콜 / 동작 흐름

participant M as CSD DRAM (resident set) participant N as CSD NAND (residual blocks)

  • 상태/객체 순서: query handle(qid, q_idx) → centroid score 행렬(임시) → H1 candidate pid 리스트 → pruning 후 H2 pid 리스트 → residual block read 요청(pid→(chunk, offset, len) 변환은 doclens 누적합) → decompress된 embedding(임시) → top-k 결과로 완료.

  • centroid table과 IVF/doclens는 크기가 작아(수십~수백 MB 추정) 무조건 resident로 두고, locality 측정 대상에서 제외한다(쟁점 2 참조).

2. Trace 생성 방법

passage(~2.5KB 추정) < 4KB이므로 한 block에 여러 pid가 섞인다 → pid 단위와 block 단위 지표가 달라진다(block 단위는 spatial locality·배치 순서 효과 포함). query 내에서 block id는 dedupe한다. Trace schema:

  • S_uniq: dev-full(또는 train에서 표본) query를 무작위 순서로 1회씩 재생. 반복이 없으므로 corpus 내재 locality(후보 중복)만 측정하는 하한.
  • S_zipf: query pool에서 Zipf(α_q) 인기 분포로 N개(예: 1M)를 복원추출해 반복이 있는 스트림을 합성. α_q는 공개 검색 로그의 query 빈도 왜도에서 가져오되(배경지식상 0.7~1.0 부근, 확인 필요) 민감도 sweep(α_q ∈ {0.6, 0.8, 1.0})으로 방어한다.
  • 각 query에 q_idx(0..N-1, 논리 시간축)와 합성 t_arrival(Poisson 도착 등)을 부여한다. locality 계산은 q_idx 기준, latency 모델만 t_arrival 사용.
  • H1: stage-1 candidate generation 직후의 candidate pid 집합(+ probed centroid ids).
  • H2: residual decompression에 들어가는 pid 집합 — 실제 storage read. 이것이 locality의 1차 접근 정의.
  • H3: 최종 top-k pid.
  • hook 위치는 ColBERT repo의 searcher/index-scorer 경로(배경지식상 Searcher.search → candidate generation → filter → residual lookup 순, 정확한 함수명 확인 필요).

queries 테이블 (query당 1행)

accesses 테이블 ((q_idx, pid)당 1행; H2 기준이 기본, stage_max로 H1도 표현)

docs 테이블 (정적, pid당 1행)

이 schema로 §3의 모든 지표가 trace 재스캔 없이 groupby/집계만으로 계산되고, §5의 scheduling feature도 같은 field에서 파생된다.

CSD MaxSim Offload의 GPU 대비 MAC/PE/FLOPS Break-even 분석

한 줄 요약

Candidate embedding이 GPU-resident가 아닌 한 GPU MaxSim은 PCIe-bound이므로, CSD는 GPU의 FLOPS가 아니라 PCIe 전달률만 이기면 되고, 그 조건은 N_mac ≥ Q · BW_pcie / (b_e · f_csd · u) — fp16, Q=32, PCIe Gen4 x16 기준 약 630개 MAC(≈1 TFLOP/s, A100 peak의 0.3%) 수준이다. 단, 이는 BW_int ≥ BW_pcie(내부 read bandwidth 조건)와 "압축 전송 baseline과의 비교"라는 두 개의 더 어려운 전제를 동반한다.

수집 범위

  • 포함: MaxSim의 MAC/FLOP 연산 모델, GPU 실행 경로(T_gpu)와 CSD 실행 경로(T_csd)의 latency equation, break-even inequality와 required MAC/PE/OPS 유도, sanity-check 수치 예시, sensitivity sweep 설계.
  • 제외: 문서 로컬리티/캐시 히트율 자체 분석(job scope에서 명시적으로 제외), query encoder(BERT forward)의 비용 — 양 경로에 공통이므로 소거, candidate generation(ANN/centroid 단계)의 자체 비용 — 단 배치(placement) 쟁점으로만 언급.

핵심 요약

  1. MaxSim은 batched small GEMM + row-wise max: FLOP_total ≈ 2·B·N·Q·D·E, max-reduction은 GEMM 대비 1/(2E) ≈ 0.4% (E=128)로 무시 가능하다.
  2. GPU 경로의 arithmetic intensity는 AI = 2Q/b_e (query는 on-chip 상주, fused kernel 가정). fp16·Q=32이면 32 FLOP/B로 A100 ridge(≈153 FLOP/B)에 한참 못 미쳐 HBM-bound이고, 데이터가 PCIe를 건너는 순간 BW_pcie ≪ BW_hbm(30~80배)이므로 항상 PCIe-bound가 된다.
  3. 따라서 break-even의 본질은 "CSD compute vs GPU compute"가 아니라 "CSD compute vs PCIe 전달률": required OPS = (Q/b_e)·BW_pcie [MAC/s]. 여기서 B, N, D, E가 전부 소거된다 — break-even MAC 수는 워크로드 크기와 무관하고 Q, precision, 링크 속도, clock, utilization에만 의존한다.
  4. 필요 MAC 수는 작다: N_mac* = Q·BW_pcie/(b_e·f·u) → fp16, f=1 GHz, u=0.8일 때 Gen4 x4에서 158개, Gen4 x16에서 630개, Gen5 x16에서 1,260개. SmartSSD급 FPGA(KU15P, DSP slice 약 2천 개 — 배경지식, 확인 필요)로도 도달 가능한 규모다.
  5. 진짜 병목은 두 번째 필요조건 BW_int ≥ BW_pcie: 단일 drive의 NAND 내부 대역폭(~7–16 GB/s)은 GPU x16 링크 유효 대역폭(25–54 GB/s)보다 낮다. 단일 CSD는 warm host DRAM baseline에 지고, cold data / multi-drive scale-out / HBM·host DRAM capacity 초과 corpus에서만 이긴다.
  6. Result traffic은 소거 가능: score만 반환 시 D·E·b_e/b_s = 8192배, top-k 반환 시 10⁵배까지 egress가 줄어 T_res는 µs 단위가 된다. CSD의 PCIe 비용은 사실상 0으로 모델링해도 된다.
  7. 최강 GPU baseline은 raw fp16 전송이 아니라 압축 전송(ColBERTv2 residual compression, ~36 B/vector — 배경지식, 확인 필요): 전송량이 ~7배 줄면 required N_mac*도 ~7배(Gen4 x16 기준 ~4.5K MAC) 커진다. CSD 제안의 공정한 비교 대상은 PLAID류 압축 경로다.
  8. GPU-resident(HBM 상주) baseline을 이기는 것은 불가능에 가깝다(required OPS ≈ 65 TFLOP/s + BW_int ≥ 2 TB/s). CSD 논거는 latency가 아니라 capacity(fp16 full index ~150+ GB > HBM 80 GB), cost, host 자원 절감으로 옮겨가야 한다.

세부 정리

1. MaxSim Operation Model

MAC / FLOP: MAC_total = B · N · Q · D · E (dot-product 곱셈-누산) FLOP_gemm = 2 · B · N · Q · D · E FLOP_red = B · N · (Q·(D−1) compare + (Q−1) add) ≈ B·N·Q·D FLOP_total ≈ 2·B·N·Q·D·E · (1 + 1/(2E)) ≈ 2·B·N·Q·D·E [E=128 → red 비중 0.4%] Intermediate matrix: candidate당 Q×D score matrix. Materialize 시 B·N·Q·D·b_s bytes (예: N=4096, Q=32, D=128 → 67 MB, 입력의 절반 규모) — fused kernel이면 register/shared memory에서 소거된다. Max-reduction은 FLOP으로는 무시 가능하지만, unfused 구현에서는 HBM traffic을 ~2배로 만들 수 있는 memory-side 항이다. AI = FLOP/Bytes_D = 2Q/b_e [FLOP/B] ← N, D, E에 무관, Q와 precision만의 함수

2. GPU Latency Model

경로: NAND → (SSD 링크) → host DRAM → (GPU 링크) → HBM → SM → score 반환. T_pcie = Bytes_in / (ε_pcie · BW_pcie), BW_pcie = min(n_ssd·BW_ssd_link, BW_gpu_link, BW_host) T_comp = max( FLOP_total/(F_gpu·η_gpu), Bytes_HBM/BW_hbm ) [roofline] T_gpu = T_fix,gpu + T_pcie + T_comp (non-overlapped) T_gpu ≈ T_fix,gpu + max(T_pcie, T_comp) (chunked pipeline)

  • ε_pcie ≈ 0.8–0.9 (protocol overhead, 대형 DMA 기준 — 배경지식), T_fix,gpu ≈ 10–30 µs (kernel launch + driver — CUDA Guide 계열 배경지식).
  • HBM constraint 두 가지: (bandwidth) Bytes_HBM ≥ Bytes_D — fused면 1회 read, unfused면 score matrix 왕복이 추가돼 ~2배. (capacity) resident baseline이 성립하려면 index size ≤ C_hbm — fp16 MS MARCO full index ~150+ GB(배경지식, 확인 필요)는 80 GB HBM에 안 들어가고, ColBERTv2 압축 index ~20–36 GB는 들어간다.
  • regime 판별: compute-bound 조건은 AI > F_gpu·η/BW_hbm (A100 fp16 ridge ≈ 153 FLOP/B — 배경지식). Q=32·fp16(AI=32)는 미달 → resident면 HBM-bound. 비-resident면 BW_pcie(≤63 GB/s) ≪ BW_hbm(2–3.35 TB/s)이므로 무조건 PCIe-bound: T_gpu ≈ Bytes_D/(ε·BW_pcie). 이때 GPU의 F_gpu는 1차 근사에서 결과에 등장하지 않는다.

3. CSD Latency Model

경로: NAND → internal channel → on-drive PE array → local max/top-k → 결과만 PCIe. Bytes_int = B·N·D·E·b_e (internal read bytes, GPU 입력과 동일) OPS_csd = N_mac · f_csd · u [MAC/s] (= 2·N_mac·f_csd·u FLOP/s) T_mac = MAC_total / OPS_csd = B·N·Q·D·E / (N_mac·f_csd·u) T_csd = T_fix,csd + max(T_read, T_mac) + Bytes_res/BW_link

  • N_mac: MAC unit(PE) 수, f_csd: clock, u: utilization(스트리밍 정렬/뱅크 충돌/tail 포함 유효 계수). local max-reduction은 MAC 뒤 comparator를 파이프라인에 붙이면 추가 cycle이 없고 u에 흡수된다. top-k heap은 O(N log k) scalar 연산으로 무시 가능.
  • 필요한 on-chip 상태는 query matrix Q·E·b_e = 8 KB + doc당 Q개 partial-max register뿐 — score matrix를 만들지 않는다.
  • Result traffic 축소율: 전체 score 반환 시 R = D·E·b_e/b_s = 8192배 (D=E=128, fp16), top-k(k=100, N=4096) 반환 시 134 MB → 800 B ≈ 1.7×10⁵배. T_res는 Gen4 x4에서도 µs 단위 → 모델에서 소거 가능. 이것이 job 질문 "score/top-k 반환 시 traffic 모델 변화"의 답: CSD egress 항이 상수(≈0)로 바뀌고, break-even은 ingress(BW_int)와 compute(OPS_csd)만의 문제가 된다.
  • T_fix,csd: NVMe/CS command 왕복 + FW dispatch, 수십~수백 µs 추정(확인 필요 — SNIA CS API 문서와 실측 필요).

4. Break-even MAC/PE 조건

Break-even inequality (양쪽 모두 pipelined, 비-resident GPU baseline): T_csd ≤ T_gpu ⇔ max( Bytes_D/BW_int , MAC_total/OPS_csd ) ≤ Bytes_D/(ε·BW_pcie) [+ 고정항 무시] ⇔ (i) BW_int ≥ ε·BW_pcie …… 대역폭 조건 (ii) OPS_csd ≥ MAC_total/Bytes_D · ε·BW_pcie = (Q/b_e)·ε·BW_pcie …… compute 조건 (ii)에서 MAC_total/Bytes_D = (B·N·Q·D·E)/(B·N·D·E·b_e) = Q/b_e [MAC/B] — B, N, D, E가 전부 소거된다. 전송되는 1 byte당 수행해야 할 MAC이 Q/b_e로 고정되기 때문이다(각 doc token vector가 Q개 dot product에 참여). 이번 턴에서는 이전의 ρ_q 기반 스케줄링을 닫힌형 decision rule로 더 구체화하고, locality trace와 break-even 모델을 연결하는 trace-driven simulator spec을 추가한다. 핵심 변화는 세 가지다.

  1. ρ_q는 문서 개수 비율이 아니라 byte-weighted, 가능하면 compute-weighted residency 비율이어야 한다.
  2. 압축 residual baseline에서는 break-even compute 강도가 Q/b_e가 아니라 Q·D / b_tok_res로 바뀌어 CSD 필요 MAC 수가 약 7배 증가한다.
  3. H2 passage 접근을 pid 단위로만 보면 안 되고, 4KB block read amplification과 hot-doc 배치 효과를 별도 계수 A_block으로 모델에 넣어야 한다.

새로 추가한 사실

1. ρ_q의 올바른 정의

기존 ρ_q = resident candidate 비율은 문서 개수 기준이면 부정확하다. doclen이 가변이므로 다음처럼 byte 기준으로 정의해야 한다.

X_q      = Σ_{d∈H2(q)} bytes(d)
R_q(B)   = Σ_{d∈H2(q), d resident under capacity B} bytes(d)
ρ_q(B)   = R_q(B) / X_q

MaxSim compute까지 반영하려면 token 기준과 동일해진다.

C_q      = Σ_{d∈H2(q)} L_d · Q · D
C_res_q  = Σ_{resident d∈H2(q)} L_d · Q · D
ρ_comp_q = C_res_q / C_q

bytes(d) ∝ L_d이면 ρ_q ≈ ρ_comp_q지만, 압축률이 문서마다 다르거나 block read amplification이 있으면 둘이 달라진다.

2. 압축 residual baseline의 compute intensity

raw fp16 기준:

bytes/token = D · b_e
MAC/token   = Q · D
MAC/byte    = Q / b_e

ColBERTv2 residual 압축 기준:

bytes/token = b_tok_res       # 예: ~36B/token, 확인 필요
MAC/token   = Q · D
MAC/byte    = Q · D / b_tok_res

예시:

Q=32, D=128, b_e=2B
raw fp16:     32 / 2 = 16 MAC/B
residual 36B: 32·128 / 36 = 113.8 MAC/B

따라서 공정한 GPU baseline이 “압축 residual을 GPU로 전송 후 decompress/scoring”이면 CSD compute 조건은 약 113.8 / 16 = 7.1× 강해진다.

더 구체화한 부분

1. ρ* 문턱의 1차 유도

query q의 H2 byte를 X_q, resident 비율을 ρ_q, effective GPU PCIe BW를 BW_g = ε·BW_pcie, CSD 내부 BW를 BW_i, CSD MAC throughput을 F_c = N_mac·f·u라고 둔다.

Hybrid split 경로:

T_gpu_miss(q) = (1−ρ_q) · X_q / BW_g
T_csd_hit(q)  = max( ρ_q · X_q / BW_i,
                     ρ_q · X_q · I_mac / F_c )
T_hybrid(q)   = T_fix + max(T_gpu_miss, T_csd_hit) + queue

여기서 I_mac은 전송 byte당 MAC이다.

raw fp16:      I_mac = Q / b_e
compressed:    I_mac = Q·D / b_tok_res

CSD hit path가 read-bound이면:

T_csd_hit = ρ_q · X_q / BW_i

GPU miss와 균형점은:

(1−ρ_q) · X_q / BW_g = ρ_q · X_q / BW_i

ρ*_bw = BW_i / (BW_i + BW_g)

예시:

BW_i = 12 GB/s, BW_g = 25 GB/s
ρ*_bw = 12 / 37 = 0.324

즉 resident byte가 약 32% 이상이면 hit 처리와 miss 처리 시간이 균형을 이룬다. 단, 이것은 hybrid split의 균형점이지, pure GPU 대비 항상 이긴다는 뜻은 아니다.

CSD compute-bound이면:

T_csd_hit = ρ_q · X_q · I_mac / F_c
ρ*_comp = F_c / (F_c + BW_g · I_mac)

최종 문턱:

ρ*_q ≈ max(ρ*_bw, ρ*_comp, ρ*_queue, ρ*_sla)

관련 맥락 / 배경 설명

Block 단위 locality와 read amplification

passage residual은 평균 수 KB 이하일 수 있지만 NAND/SSD 내부 I/O는 보통 page/block 단위로 움직인다. 따라서 pid 단위 byte 절감만 보면 과대평가될 수 있다.

logical_bytes_q = Σ bytes(pid)
physical_bytes_q = Σ unique_block_bytes(block_id)
A_block(q) = physical_bytes_q / logical_bytes_q

CSD 내부 read 모델은 다음처럼 바뀐다.

T_read = A_block(q) · bytes_H2_residual(q) / BW_int

hot pid를 block-local하게 재배치하면 A_block이 내려간다. 따라서 연구 질문은 “hot documents를 얼마나 많이 pin할 수 있는가”뿐 아니라 “hot documents가 같은 erase/page region에 얼마나 잘 모이는가”까지 확장된다.

Scheduling rule 초안 업데이트

dispatch(q):

1. compute ρ_q(B) after H1/H2 candidate known
2. estimate:
   L_gpu    = X_q / BW_g + Q_gpu_wait
   L_csd    = max(X_q / BW_i, X_q·I_mac/F_c) + Q_csd_wait
   L_hybrid = max((1−ρ_q)X_q/BW_g,
                  ρ_q X_q/BW_i,
                  ρ_q X_q I_mac/F_c) + Q_hybrid_wait

3. choose argmin latency if predicted p99 < SLA
4. otherwise route to GPU unless GPU queue is already p99-critical

더 단순한 논문용 rule은 다음 형태가 적절하다.

CSD/HYBRID if ρ_q ≥ ρ*(load) and X_q ≥ X_min
GPU        otherwise

X_min이 필요한 이유는 작은 query에서는 T_fix,csd가 byte 절감 이득보다 커질 수 있기 때문이다.

잠재적 오류 또는 확인 필요 지점

  1. H2≈256인지 H2≈4096인지는 절대 latency를 16배 바꾼다. PLAID에서 실제 residual decompress 대상 수를 코드 hook으로 확정해야 한다.
  1. b_tok_res≈36B/token은 핵심 상수다. 이 값이 residual only인지, codebook/metadata/doclens 포함 amortized byte인지 분리해야 한다.
  1. BW_int는 NAND raw bandwidth, controller-to-FPGA bandwidth, CSD DRAM bandwidth 중 무엇을 뜻하는지 명확히 해야 한다. break-even에는 실제 PE가 지속적으로 공급받는 bandwidth를 넣어야 한다.
  1. ρ_q는 H1 직후 계산 가능하다고 했지만, H2가 pruning 이후에 확정된다면 dispatch 시점이 늦어진다. H1 기반 예측 ρ̂_q와 실제 H2 기반 ρ_q의 오차를 따로 측정해야 한다.
  1. CSD-only가 항상 PCIe를 줄이는 것은 맞지만 latency를 줄이려면 T_fix,csd와 queueing delay가 작아야 한다. 특히 p99에서는 평균 HR_bytes보다 per-query ρ_q 분산이 더 중요하다.

다음 에이전트가 이어서 확장할 지점

  1. ρ̂_q 예측 모델 구체화: H1 centroid ids, candidate pid hotness, candidate byte sum만으로 H2의 ρ_q를 얼마나 정확히 예측할 수 있는지 feature set과 error metric을 정의한다.
  1. block-aware layout 실험 설계: pid 순서 그대로, random layout, hotness-sorted layout, centroid-clustered layout을 비교해 A_block, HR_bytes, T_read가 어떻게 변하는지 모델링한다.
  1. p99 queueing simulator 추가: trace arrival process를 Poisson/ bursty 두 가지로 만들고, GPU/CSD/HYBRID policy별 mean, p95, p99, PCIe GB/s, CSD utilization을 동시에 출력하는 discrete-event simulator spec을 작성한다.

새로 추가한 사실

1. H1(candidate generation) compute가 H2 MaxSim을 압도한다 — break-even의 stage 오지정

PLAID stage-1은 query token들을 전체 centroid 행렬과 곱해 근접 centroid를 찾는다. 이건 그 자체가 MaxSim류 연산이다.

MAC_H1 = Q · n_cent · D          # query(Q×D) × centroids(n_cent×D)
MAC_H2 = (Σ_{d∈H2} L_d) · Q · D  # 실제 MaxSim

n_cent ~ O(10⁵)(ColBERTv2/PLAID 배경지식, 확인 필요; 2^18≈262k로 예시), H2≈256, L̄≈80 대입:

MAC_H1 = 32 · 262144 · 128 ≈ 1.07 GMAC/query
MAC_H2 = 256 · 80 · 32 · 128 ≈ 0.084 GMAC/query
→ MAC_H1 / MAC_H2 ≈ 13×

함의(정정 포함). 지금까지의 break-even은 N_mac* = Q·BW_pcie/(b_e·f·u) (H2만) 기준이었다. 그러나 CSD가 H1까지 수행하면 CSD compute 요구는 H2가 아니라 H1이 결정한다. N_mac=256, f=1GHz, u=0.8 (OPS≈205 GMAC/s)에서:

T_H1(CSD) = 1.07e9 / 2.05e11 ≈ 5.2 ms     ← CSD latency 지배
T_H2(CSD) = 0.084e9 / 2.05e11 ≈ 0.41 ms

H1이 CSD latency를 지배하므로 H2만 놓고 "630 MAC이면 충분"이라던 결론은 H1 co-location 시 성립하지 않는다.

2. PCIe-최적 파이프라인 배치: H1@GPU, H2@CSD (round-trip 모델)

H1의 입력은 centroid table(n_cent·D·b_e = 262144·128·2 ≈ 67 MB, 작음)과 query embedding(encoder가 이미 GPU HBM에 생성)뿐이고, 출력은 candidate pid 리스트(작음)다. 즉 H1은 GPU에서 compute-cheap·PCIe-cheap이다. 반면 H2는 residual(대용량)을 읽으므로 CSD-local이 유리하다. 따라서:

[canonical placement]
encoder(GPU) → H1(GPU, centroid table resident) → candidate pids (작음) →PCIe→ CSD
CSD: H2 residual read(내부) + MaxSim + top-k → scores(작음) →PCIe→ GPU merge

PCIe 왕복 = pid list(수 KB) + scores(수 KB) ≈ 상수, H2 residual byte = 0
추가 비용 = 2·T_fix + 2·(작은 payload/BW_pcie)   ← round-trip latency

이 배치는 H1 compute 폭발(사실 1)과 H2 PCIe 폭발을 동시에 회피한다. 대가는 GPU→CSD→GPU 왕복의 2·T_fix(수십~수백 µs, 실측 필요)와 파이프라인 직렬 의존성이다. 이전 turn들이 "candidate gen은 placement 쟁점으로만 언급"하고 제외했으나, 사실 1을 보면 이 배치가 CSD 설계 성립의 전제조건이다.

3. read-byte와 MAC-count를 분리해야 한다 (Codex의 X_q·I_mac lumping 정정)

Codex는 T_csd_hit = max(ρX/BW_i, ρX·I_mac/F_c)로 하나의 X_q에서 read와 compute를 모두 파생했는데, 이는 압축률·block amplification이 균일할 때만 맞다. 둘은 독립 물리량이므로 분리 추적해야 한다.

Read_q  = A_block(q) · Σ_{d∈served} L_d · b_tok_res   [NAND/DRAM read byte]
MAC_q   = (Σ_{d∈served} L_d) · Q · D                  [MaxSim MAC, b_tok_res와 무관]
Dec_q   = (Σ_{d∈served} L_d) · D · c_dec              [residual 복원, c_dec≈1–2]

T_read = Read_q / BW_int
T_comp = (MAC_q + Dec_q) / F_c
T_csd_hit = max(T_read, T_comp)

신규 항 Dec_q: ColBERTv2 residual 복원(centroid[idx] + dequant)은 token당 ~D ops로 MaxSim의 1/Q ≈ 3%(Q=32)에 불과하지만, MAC array가 못 하는 연산(LUT·shift·add)이라 별도 datapath가 필요하다 — CSD PE 설계에서 comparator뿐 아니라 dequant unit을 파이프라인에 넣어야 함을 뜻한다(u에 흡수되지 않을 수 있음).

더 구체화한 부분

1. ρ*_bw는 dispatch 문턱이 아니라 목표 split 비율이다 (방향 오류 정정)

Codex의 ρ*_bw = BW_i/(BW_i+BW_g)는 hybrid 두 arm이 병렬로 균형을 이루는 split 비율 φ이지, "ρ_q ≥ ρ → CSD"라는 dispatch 규칙이 아니다. ρ_q는 residency로 고정된 값이라 자유롭게 못 고르기 때문이다. split 비율 φ(CSD가 처리할 byte 분율)를 자유변수로 두면:

L(φ) = max( (1−φ)·X/BW_g ,  φ·X·k_csd(φ) )
k_csd(φ) = 1/BW_dram        (φ ≤ ρ_q: resident, DRAM에서 즉시)
         = A_block/BW_int   (φ > ρ_q: 초과분은 NAND read, kink 발생)

균형 φ* : (1−φ*)/BW_g = φ*·k_csd  →  φ* = BW_g⁻¹ / (BW_g⁻¹ + k_csd)
결합 대역폭 이득: L(φ*) = X / (BW_g + 1/k_csd)   (양 arm 병렬 → 합산 대역)

핵심 재해석: ρ_q는 split 비율을 정하지 않는다. ρ_q는 CSD arm의 "싼 연료 풀"(DRAM·복원완료 hot doc) 크기를 정한다. 최적은 φ* ≤ ρ_q이면 CSD-served가 전부 DRAM에서 나와 이상적이고, φ* > ρ_q이면 초과분이 NAND cost를 물어 이득이 꺾인다. 따라서 올바른 조건은 "ρ_q ≥ φ*(균형 split)" 이다:

route/split to CSD 유리  ⇔  ρ_q ≥ φ* = (1/BW_g) / (1/BW_g + k_csd)

Codex의 부등식과 형태는 닮았지만 의미가 다르다: ρ_q가 균형 split을 덮을 만큼 크냐를 묻는 것이지, ρ_q 자체가 균형점인 게 아니다. ρ_q가 φ*보다 크면 남는 residency는 안전 마진(NAND fallback 회피)일 뿐 추가 이득이 없다.

2. HR_bytes(B)의 이중 의미 분리 (두 turn 공통 개념 혼동 정정)

HR_bytes(B)가 무엇을 지배하는지는 설계에 따라 다르다. 이걸 구분하지 않아 "index의 x%만 상주시키면 PCIe y% 절감"이라는 서사가 설계에 따라 참이 되기도 거짓이 되기도 했다.

설계비상주(miss) 데이터 위치HR_bytes(B)가 지배하는 것H2 PCIe traffic
CSD-only같은 CSD의 NAND내부 latency: hit→DRAM(빠름), miss→NAND(느림). T_read = HR·Read/BW_dram + (1−HR)·Read/BW_nand0 (B와 무관)
doc-split hybridhost/GPU (PCIe로 송출)PCIe 절감율 = HR_bytes(B)(1−HR)·bytes_H2

즉 "B*만큼 상주 → PCIe y% 절감" 서사는 hybrid 설계 전용이다. CSD-only에서는 전체 index(~22GB)가 이미 device 내부에 있으므로 B와 무관하게 H2 PCIe=0이고, B-sweep은 DRAM 캐시 sizing(내부 tail latency) 문제로 바뀐다. 두 개의 residency를 이름부터 분리하자:

  • placement residency: doc의 primary copy가 CSD에 있는가 — capacity 문제, corpus당 이진값. full index가 NAND에 들어가면 항상 참.
  • tier residency HR(B): CSD 내부에서 빠른 DRAM tier에 있는가 — caching 문제, locality 곡선. 이게 §locality의 HR_bytes(B)다.

3. p99 queueing admission rule — M/G/1 닫힌형 (미완 과제 전진)

두 turn 모두 "다음"으로 미룬 부분을 유도한다. CSD 서비스 시간 S = max(Read_q/BW_int, (MAC_q+Dec_q)/F_c) + T_fix는 L_d·|H2| 변동 때문에 분포를 가진다. Pollaczek–Khinchine:

E[W] = ρ_u·E[S]·(1 + C_S²) / (2(1 − ρ_u)),   ρ_u = λ·E[S],  C_S² = Var(S)/E[S]²
E[T] = E[W] + E[S]

평균-SLA E[T] ≤ Λ에 대한 최대 허용 utilization 닫힌형:

ρ_u ≤ ρ_max = (Λ − E[S]) / ( (Λ − E[S]) + E[S]·(1+C_S²)/2 )

p99 admission: heavy-traffic에서 M/G/1 대기시간은 근사적으로 지수분포(mean E[W])이므로

T_p99 ≈ E[S] + ln(100)·E[W] ≈ E[S] + 4.6·E[W]
admit to CSD  ⇔  λ_csd 가 ρ_u < ρ_max(p99) 를 유지  (ρ_max에 4.6·E[W]≤Λ−E[S] 대입)

연결(두 turn이 놓친 정량 고리): C_S²는 per-query ρ_q·X_q 분산에서 직접 온다. locality가 왜곡될수록(hot doc 집중) X_q 분산이 커지고 C_S²↑ → 같은 평균 부하에서 E[W]가 폭증. 따라서 §locality의 CDF/Gini는 정확도가 아니라 p99를 통해 CSD 수용량을 정량 제약한다 — "평균 HR_bytes가 좋아도 분산이 크면 CSD가 p99에서 진다"는 Codex의 정성 관찰을 수식으로 닫았다.

관련 맥락 / 배경 설명

ADC-style MaxSim: CSD compute intensity를 낮추는 index-format 레버 (배경지식·연구 방향)

압축 baseline에서 I_mac = Q·D/b_tok_res ≈ 113.8 MAC/B가 CSD 필요 MAC을 ~7× 키운다(Codex). 이를 줄이는 구조적 레버: score를 q·(centroid+residual) = q·centroid + q·residual로 분해하면 q·centroid는 H1에서 이미 계산된 Q×n_cent 행렬의 lookup이다(각 token은 하나의 centroid에 매핑). 남는 건 q·residual뿐. 다만 ColBERTv2 residual은 scalar 2-bit라 여전히 O(D) MAC이 필요해 큰 이득은 없다. 연구 방향: residual을 product-quantize(PQ)로 바꾸면 q·residual을 PQ ADC 테이블 lookup으로 대체 → token당 compute가 Q·D에서 Q·(D/m) lookup 규모로 급감, I_mac이 크게 내려가 break-even MAC 요구가 완화된다. 대가는 index 포맷 변경 + 검색 품질(MRR/nDCG) 재검증이며, 이는 job scope의 "품질 분석 제외"와 충돌하므로 별도 검증 축이 필요하다. 정확도 손실 없이 CSD compute를 줄일 수 있다면 사실 1(H1 지배)과 무관하게 H2 PE 예산을 크게 낮출 수 있는 유일한 알고리즘적 경로다.

저정밀 MaxSim (int8)의 area/energy 이득과 경계

CSD PE를 fp16 대신 int8 MAC으로 두면 MAC당 area·energy가 ~4× 감소해 같은 실리콘에 4× 많은 N_mac을 넣을 수 있다(사실 1의 H1 병목 완화에 직접 기여). ColBERT score는 저정밀에 관대하다고 알려져 있으나(배경지식, 확인 필요), residual dequant→int8 재양자화의 오차가 top-k 순위에 주는 영향은 실측 필요. int8이면 break-even의 b_e 해석도 바뀌므로(전송은 여전히 residual, compute만 int8) N_mac* 표를 int8 열로 확장해야 한다.

잠재적 오류 또는 확인 필요 지점

  1. [신규·최우선] n_cent 실측: 사실 1의 결론(H1 compute 지배)은 n_cent에 선형 비례한다. PLAID/ColBERTv2 실제 centroid 수(2^18? sqrt(N_emb) scaling?)를 코드/index 산출물로 확정해야 MAC_H1이 확정된다. 실측 필요.
  2. [신규] H1이 brute-force인지: stage-1이 전체 centroid 행렬곱인지, centroid에 대한 별도 ANN(HNSW 등)인지에 따라 MAC_H1이 Q·n_cent·D에서 Q·log(n_cent)·D급으로 떨어질 수 있다 — H1 병목 주장의 존폐가 걸림. 코드 확인 필요.
  3. BW_dram vs BW_nand 분리: §더구체화-2의 CSD-only latency 모델은 CSD 내부 DRAM 대역과 NAND 대역을 다른 상수로 요구한다. 이전 turn의 단일 BW_int는 이 둘을 뭉갰다. drive별 실측 필요.
  4. round-trip T_fix 이중 계상: 사실 2의 배치는 T_fix를 두 번(GPU→CSD, CSD→GPU) 문다. Amdahl 상한(Turn 1)과 결합 시 이 왕복 고정비가 MaxSim 절감분을 상쇄할 수 있음 — 순 이득 부호를 실측으로 확인해야 함.
  5. c_dec가 u에 흡수되는지: 사실 3의 Dec_q가 별도 datapath를 요구하면 u(utilization)로 못 숨기고 독립 stage가 된다. PE microarchitecture 가정(dequant unit 유무)에 민감.

Turn 4 / Codex

이번 턴의 확장 요약

이번 턴은 H1@GPU / H2@CSD 배치를 더 현실적인 end-to-end 파이프라인으로 바꾸고, 이전 턴에서 아직 분리되지 않은 정확성 제약, batching 고정비, multi-CSD sharding을 추가한다. 핵심은 다음이다.

  1. latency는 query 단위로는 encoder → H1 → H2 → merge의 합이지만, throughput은 stage pipeline의 max(stage time)로 결정된다.
  2. CSD가 top-k만 반환해도 되는 조건은 문서 전체 MaxSim score를 한 장치가 완결 계산할 때다. 문서 token이 여러 장치/블록에 쪼개지면 partial-max vector merge가 필요해 egress 모델이 달라진다.
  3. T_fix,csd가 수십-수백 µs라면 H2 자체가 작을 때는 CSD 이득을 먹어버리므로, query batching과 admission rule이 필수 설계 변수가 된다.

새로 추가한 사실

1. H2 offload의 correctness constraint: “doc 단위 sharding”이 기본이어야 한다

MaxSim score는 다음이다.

score(q,d) = Σ_{i=1..Q} max_{j=1..L_d} <q_i, e_{d,j}>

따라서 CSD가 어떤 passage d의 모든 token embedding을 갖고 있으면 score(q,d) 하나만 반환하면 된다.

CSD returns: (pid, score)          # O(k) 또는 O(N) scalar

하지만 passage token이 여러 CSD나 여러 logical shard에 나뉘면 각 shard는 최종 score를 만들 수 없다.

partial_m_s[i] = max_{j in shard s} <q_i, e_{d,j}>
final score    = Σ_i max_s partial_m_s[i]

이 경우 반환량은 doc당 scalar 1개가 아니라 Q개 partial max가 된다.

egress_fragmented ≈ N_docs · Q · b_score
egress_doc_shard  ≈ N_docs · b_score
ratio             ≈ Q      # Q=32면 32배 증가

따라서 CSD 설계의 기본 layout 제약은:

all token embeddings of one pid must be co-located on one scoring owner

multi-CSD scale-out도 passage 단위 sharding이 안전하다. block 단위 read locality만 보고 token-level striping을 하면 top-k egress 절감 주장이 깨진다.

2. Pipeline throughput은 latency 합이 아니라 bottleneck stage로 결정된다

H1@GPU / H2@CSD 배치에서 단일 query latency는 대략 합이다.

L_q = T_enc,gpu + T_H1,gpu + T_cmd + T_H2,csd + T_merge,gpu

하지만 충분한 query stream에서는 GPU와 CSD가 서로 다른 query를 처리하므로 steady-state QPS는 다음에 가깝다.

Throughput ≤ 1 / max(
  E[T_enc + T_H1],
  E[T_cmd + T_H2],
  E[T_merge]
)

즉 H1 compute가 H2보다 크더라도, GPU stage가 이미 encoder와 묶여 포화되어 있지 않다면 CSD offload는 throughput을 늘릴 수 있다. 반대로 GPU encoder가 이미 병목이면 H2를 CSD로 보내도 QPS 개선은 작고, host PCIe/energy 절감만 남는다.

3. Query batching은 T_fix,csd를 줄이지만 tail latency를 악화한다

CSD command 고정비를 batch size M으로 나누면:

T_fix_per_query ≈ T_fix,csd / M

하지만 Poisson arrival에서 batch가 찰 때까지 기다리는 평균 시간은 근사적으로:

E[W_batch] ≈ (M - 1) / (2λ)

따라서 query당 CSD latency는:

L_csd_batch(M)
≈ (M - 1)/(2λ)
 + T_fix,csd/M
 + max(Read_batch/BW_int, MAC_batch/F_csd)/M

M을 키우면 command overhead는 줄지만 p99가 나빠진다. 그래서 scheduler에는 기존 ρ_q, X_q, queue depth 외에 다음 feature가 들어가야 한다.

batch_age      # oldest query wait time
batch_fill     # current batch size / target M
SLA_slack_min  # batch 안에서 가장 작은 slack

더 구체화한 부분

1. CSD-only와 hybrid의 목적함수 분리

이제 두 설계는 같은 HR_bytes(B)를 쓰더라도 목적함수가 다르다.

CSD-only:
  minimize p99 latency
  subject to PCIe_H2 = 0
  B controls DRAM/NAND tier hit, not host traffic

Hybrid:
  minimize PCIe_H2 and GPU load
  subject to p99 latency ≤ SLA
  B controls fraction routed to CSD without miss traffic

각 설계의 per-query 모델은 다음처럼 분리하는 편이 안전하다.

# CSD-only
T_read_csd(q) =
  Read_hot(q)/BW_dram_csd
+ Read_cold(q)/BW_nand_csd

PCIe_H2(q) = 0

# doc-split hybrid
T_hybrid(q) =
  max(
    Read_resident(q)/BW_csd_effective,
    Read_gpu_miss(q)/BW_gpu_link
  ) + fixed + queue

PCIe_H2(q) = Read_gpu_miss(q)

따라서 논문 그림도 두 개가 필요하다.

DRAM tier sizingCSD DRAM Bp50/p99 T_csdCSD-only tail 개선
hybrid trafficCSD resident BPCIe GB/s, p99offload/traffic tradeoff

2. Multi-CSD scale-out 조건

CSD가 n개일 때 단순히 bandwidth가 n·BW_int로 늘지는 않는다. query는 모든 CSD에 broadcast되고, 각 CSD는 자기 shard의 candidate만 처리한다.

T_multi(q) =
  T_broadcast_query
+ max_{s=1..n} T_H2_s(q)
+ T_merge_topk(n)

균등 sharding이면 평균은 줄지만 tail은 가장 느린 shard가 결정한다.

E[max_s T_s] > E[T_s]

hot documents가 특정 shard에 몰리면 scale-out이 무너진다. 따라서 layout metric에 shard imbalance를 추가해야 한다.

load_s(q) = Σ_{d∈H2(q), shard(d)=s} bytes(d)
imbalance(q) = max_s load_s(q) / (Σ_s load_s(q)/n)

좋은 layout은 block read amplification A_block만 낮추는 것이 아니라 imbalance(q)도 낮춰야 한다.

관련 맥락 / 배경 설명

H1@GPU / H2@CSD 배치의 데이터 계약

GPU가 CSD로 보내야 하는 것은 raw text나 embedding corpus가 아니라 작은 command payload다.

command(q):
  qid
  query_embedding[Q,D]      # fp16이면 Q·D·2B, Q=32,D=128 → 8KB
  candidate_pid_list[H2]
  optional candidate metadata: doclen, block_id, shard_id
  topk_requested

CSD 반환은 doc 단위 sharding이면 작다.

result(q):
  topk list of (pid, score)
  optional diagnostics: read_bytes, mac_count, service_time

이 diagnostics를 trace에 다시 붙이면 simulator가 아니라 실제 device prototype에서도 같은 schema로 검증할 수 있다.

ρ̂_q는 H1 이후가 아니라 H2 확정 전 예측값이어야 한다

실제 dispatch를 H2 확정 뒤에 하면 이미 pruning 계산을 일부 수행한 뒤라 pipeline 유연성이 줄어든다. 따라서 두 값을 분리해야 한다.

ρ̂_q = predicted residency from H1 features
ρ_q  = actual residency after H2 materialization
error = ρ_q - ρ̂_q

추천 feature:

H1_pid_hotness_mean
H1_pid_hotness_p95
H1_resident_byte_fraction
centroid_id_entropy
estimated_H2_bytes
query_length_Q

스케줄러 평가는 ρ_q oracle이 아니라 ρ̂_q 기반으로 해야 한다.

잠재적 오류 또는 확인 필요 지점

  1. H1 compute가 전체 centroid brute-force인지, centroid ANN/pruning이 있는지 확인해야 한다. 이 값은 H1@GPU/H2@CSD 배치의 필요성을 좌우한다.
  1. CSD가 candidate pid list를 받는 모델에서는 CSD가 H2 pruning logic을 얼마나 알아야 하는지 불명확하다. GPU가 H2까지 확정하면 CSD는 scoring device이고, CSD가 H1 후보에서 H2를 고르면 CSD 쪽에 PLAID pruning logic 일부가 들어간다.
  1. top-k merge가 항상 작다는 주장은 doc 단위 sharding에서만 성립한다. token-fragmented layout이나 replicated hot-doc layout에서는 반환 protocol을 다시 계산해야 한다.
  1. batching은 평균 throughput을 개선하지만 SLA가 짧은 interactive retrieval에서는 p99 손실이 클 수 있다. T_fix,csd 실측 없이는 batch size를 정할 수 없다.
  1. multi-CSD는 평균 bandwidth보다 shard imbalance와 hot-doc skew가 더 위험할 수 있다. trace에는 반드시 shard_id와 per-query shard byte 분포가 필요하다.

다음 에이전트가 이어서 확장할 지점

  1. H1/H2 split protocol 확정: GPU가 H2 pid까지 확정해서 CSD에 넘기는 방식과, CSD가 H1 candidate를 받아 내부 pruning까지 하는 방식의 latency·PCIe·구현 복잡도 비교.
  1. doc-shard layout optimizer 설계: A_block, HR_bytes(B), imbalance(q)를 동시에 최소화하는 pid 배치 알고리즘과 trace-driven 평가 지표 정의.
  1. batching + p99 simulator 구체화: M, T_fix,csd, arrival burstiness, SLA를 sweep해서 CSD admission threshold와 optimal batch size를 함께 산출하는 discrete-event simulator spec 작성.
