초기 seed

Runtime 내부 구조 딥러닝 seed

런타임을 단순히 "프로그램 실행 환경"으로 요약하지 말고, 프로그램이 실행되는 동안 상태와 자원을 소유하고 전이시키는 제어 계층으로 분석한다.

중심 질문

  1. 런타임이란 무엇인가?
  2. 런타임 내부는 어떤 모듈로 구성되는가?
  3. task, process, thread, fiber, coroutine, stream, command queue는 런타임 안에서 어떻게 표현되는가?
  4. 런타임은 task/process를 어떤 정책으로 스케줄링하는가?
  5. 스케줄링 상태는 ready, running, blocked, waiting, completed, failed, cancelled 같은 상태로 어떻게 이동하는가?
  6. 프로그램이나 task는 어떤 상태로 런타임을 빠져나가는가?
  7. 런타임은 CPU, GPU, accelerator, memory, buffer, queue, event, context, handle 같은 자원을 어떻게 할당하고 회수하는가?
  8. 런타임이 소유하는 상태와 OS kernel, driver, compiler, framework가 소유하는 상태는 어떻게 나뉘는가?
  9. CUDA Runtime/Driver, Intel Level Zero, PJRT/OpenXLA, UPMEM SDK, JVM/Go/Tokio/Node.js 같은 언어 런타임, Linux scheduler를 비교하면 어떤 공통 구조가 보이는가?
  10. LPDDR6-PIM 또는 PIM runtime을 설계한다면 어떤 최소 runtime object와 API가 필요한가?

매 턴 확장 규칙

  • "검증 완료", "문제 없음", "추가 없음"으로 끝내지 않는다.
  • 반드시 새 개념, 더 구체화된 메커니즘, 관련 런타임 비교, 상태 전이, 자원 할당 세부사항 중 최소 2개를 추가한다.
  • 가능하면 상태표, 스케줄링 흐름, API 객체 비교표, 다음 조사 질문을 포함한다.
  • 이전 턴의 오류를 발견하면 수정하되, 수정에서 끝나지 말고 추가 내용을 붙인다.

우선 확장 순서

  1. Runtime의 일반 모델: init, context, scheduler, allocator, executor, event loop, error/completion.
  2. OS runtime과 language runtime: Linux process scheduler, JVM, Go runtime, Tokio/Node.js event loop.
  3. Accelerator runtime: CUDA Runtime/Driver, Level Zero, ROCm/HIP, OpenCL/SYCL.
  4. ML compiler runtime: PJRT, XLA, StableHLO lowering, buffer/executable/event.
  5. PIM runtime: UPMEM, HBM-PIM/LPDDR-PIM 후보, memory-side command, host synchronization.
  6. 설계 도출: LPDDR6-PIM runtime object model, state machine, scheduler, allocator, API 후보.

Turn 1 / Claude

이번 턴의 확장 요약

첫 턴이므로 누적 자료가 없는 상태에서 5개 축(CUDA Runtime/Driver, Intel Level Zero, PJRT/OpenXLA, UPMEM SDK, LPDDR6-PIM)에 대한 기초 골격을 구축했다. 각 런타임의 핵심 추상화 계층, 메모리/실행 모델, 상호 비교 지점을 정리하고, 특히 LPDDR6-PIM은 표준화 미완료 상태임을 명시했다.

새로 추가한 사실

CUDA Runtime/Driver

  • CUDA는 두 계층 API를 가진다: Driver API(cuda.h, cuXxx, 명시적 컨텍스트 생성 cuCtxCreate 필요)와 Runtime API(cuda_runtime.h, cudaXxx, 디바이스별 컨텍스트를 암묵적으로 생성). Runtime API는 내부적으로 Driver API 위에 얇게 래핑되어 있다.
  • PTX(중간 표현)는 드라이버가 로드 시점에 JIT으로 SASS(디바이스 고유 기계어)로 컴파일하며, 결과는 compute cache에 캐싱된다.
  • CUDA 11.7부터 "lazy module loading"이 기본화되어 커널을 실제 호출 시점까지 로드를 지연시켜 프로세스 메모리 사용량을 줄인다.
  • CUDA Graphs(CUDA 10+)는 커널/메모리 연산 시퀀스를 캡처해 launch overhead를 줄이는 재실행 가능한 실행 그래프를 제공한다.

Intel Level Zero

  • oneAPI의 저수준 explicit API로, Vulkan/Metal과 유사한 철학(명시적 디바이스 열거, 명시적 메모리 관리)을 따른다.
  • 계층 구조: Driver → Device → Context → Command List → Command Queue. Command List는 "immediate"(즉시 실행)와 "deferred/batched"(큐에 제출 후 실행) 두 방식이 있다.
  • USM(Unified Shared Memory)으로 host/device/shared 세 종류의 메모리 할당을 지원.
  • SYCL/DPC++ 런타임은 Level Zero를 백엔드로 사용하는 구조 — PyTorch가 CUDA Runtime 위에 있는 것과 유사한 위치.

PJRT/OpenXLA

  • PJRT(원래 "Portable JAX RunTime", 현재는 "Pretty much Just a Runtime"로 재해석)는 프론트엔드(JAX, TF, PyTorch/XLA)와 백엔드(CPU/GPU-XLA/TPU/서드파티 플러그인)를 분리하는 하드웨어 중립 런타임 인터페이스.
  • PJRT C API를 통해 out-of-tree 플러그인 등록이 가능 — 플러그인은 PJRT_Api 함수 포인터 구조체를 구현해 컴파일러/디바이스를 연결한다(LLVM/MLIR 타겟 등록 방식과 유사).
  • 핵심 추상화: PjRtClient, PjRtDevice, PjRtBuffer, PjRtExecutable.
  • StableHLO는 프론트엔드와 PJRT 등록 컴파일러 사이의 이식 가능한 MLIR IR 경계 역할을 한다.

UPMEM SDK

  • DPU(DRAM Processing Unit) 아키텍처: DIMM 내부 DRAM 뱅크 옆에 RISC 유사 코어 내장.
  • 호스트 프로그램(C, libdpu)이 DPU 전용 LLVM 기반 툴체인(dpu-clang)으로 컴파일된 DPU 프로그램을 로드/실행.
  • 메모리 계층: MRAM(DPU당 약 64MB, 메인 메모리 역할), WRAM(태스클릿 그룹당 스크래치패드, 매우 작음), IRAM(명령어 메모리).
  • DPU 코어당 최대 24개 "tasklet"(하드웨어 스레드)이 협력적 멀티스레딩으로 동작.
  • 호스트-DPU 간 캐시 일관성 없음 — dpu_copy_to/dpu_copy_from 형태의 명시적 DMA류 전송 필요. 64개 DPU를 "rank" 단위로 묶어 병렬 디스패치.

LPDDR6-PIM

  • LPDDR6은 2026년 7월 기준 JEDEC 표준화가 완전히 마무리되지 않은 규격이며, 공개된 표준 PIM 런타임 SDK는 확인되지 않는다. 관련 PIM 시도는 SK hynix의 GDDR6 기반 AiM(Accelerator in Memory), Samsung의 HBM-PIM/AXDIMM, LPDDR5X-PIM 온디바이스 AI 프로토타입 등 인접 세대에서 먼저 등장했다.

더 구체화한 부분

  • CUDA와 UPMEM은 둘 다 "호스트가 명시적으로 오프로드 대상을 관리"하는 모델이지만, CUDA는 SASS로 JIT 컴파일되는 반면 UPMEM은 DPU 전용 정적 컴파일(AOT, dpu-clang)이라는 점에서 컴파일 전략이 다르다.
  • Level Zero의 Command List(immediate/deferred)는 CUDA Stream/Graph와 개념적으로 대응된다: immediate Command List ≈ CUDA Stream의 즉시 제출, deferred Command List + 재사용 ≈ CUDA Graph의 캡처-재실행 모델.
  • PJRT는 "런타임"이라는 이름이지만 실질적으로는 컴파일러(XLA/StableHLO)와 디바이스 실행을 잇는 플러그형 ABI 계층에 가깝다 — CUDA Driver API나 Level Zero처럼 하드웨어를 직접 제어하는 것이 아니라, 그 아래에 있는 벤더 런타임(CUDA Runtime, ROCm, PJRT-TPU 등)을 다시 감싸는 상위 추상화다.

관련 맥락 / 배경 설명

  • 이 다섯 시스템은 모두 "프레임워크-컴파일러-런타임-하드웨어" 스택에서 서로 다른 층위에 위치한다. CUDA Runtime/Driver와 Level Zero는 하드웨어에 가장 가까운 vendor-native 런타임이고, PJRT는 그 위에서 프레임워크와 vendor 런타임을 잇는 중립 계층이며, UPMEM SDK와 LPDDR6-PIM은 PIM이라는 별도 축(연산-메모리 통합)에서 유사한 host-offload 패턴을 재구현하고 있다.
  • PIM 계열(UPMEM, LPDDR6-PIM 후보군)의 공통 과제는 "캐시 비일관성 상태에서 어떻게 명시적 동기화를 런타임 API로 노출할 것인가"이며, 이는 CUDA/Level Zero의 스트림·이벤트 기반 동기화 모델과 비교할 지점이 된다.

잠재적 오류 또는 확인 필요 지점

  • LPDDR6-PIM 관련 서술은 공개 SDK/런타임 스펙이 존재하는지 불확실하다 — 벤더별 백서(SK hynix, Samsung, 또는 JEDEC 워킹그룹 자료)의 실제 발표 시점과 세부 사항을 원문으로 재확인 필요.
  • UPMEM의 rank당 DPU 수(64개)와 MRAM 용량(64MB/DPU)은 세대(예: UPMEM PIM DIMM 버전별)에 따라 다를 수 있어 최신 데이터시트 대조 필요.
  • PJRT의 "Pretty much Just a Runtime" 재해석 시점과 공식 문서상의 정확한 표현은 OpenXLA 공식 리포지토리(README/design doc)로 재검증 필요.

다음 에이전트가 이어서 확장할 지점

  1. CUDA Driver API의 컨텍스트 스택(cuCtxPushCurrent/PopCurrent) 모델과 Level Zero의 Context 객체를 실제 API 시그니처 수준까지 비교해 "멀티스레드 환경에서 컨텍스트 관리 책임이 런타임 vs 애플리케이션 중 누구에게 있는가" 정리.
  2. PJRT 플러그인이 실제로 UPMEM류 PIM 디바이스나 LPDDR6-PIM 같은 비-GPU/비-TPU 가속기를 백엔드로 등록한 사례(연구 논문·오픈소스 PR)가 있는지 조사하고, 있다면 StableHLO→PIM 명령 변환 경로를 정리.
  3. UPMEM tasklet 동기화 프리미티브(뮤텍스, 바리어, 핸들)를 CUDA의 warp/block 동기화(__syncthreads) 및 Level Zero의 이벤트 기반 동기화와 대응시켜, "PIM 런타임에 필요한 최소 동기화 원시 연산 집합"을 도출.

새로 추가한 사실

CUDA Driver API

출처: NVIDIA CUDA Driver API Context Management

  • CUDA Driver API의 컨텍스트는 CPU 스레드에 바인딩되는 실행 상태다. 공식 문서상 cuCtxGetCurrent는 “calling CPU thread에 바인딩된 CUDA context”를 반환하고, cuCtxPushCurrent/cuCtxPopCurrent는 현재 CPU 스레드의 컨텍스트 스택을 조작한다. 즉 멀티스레드 호스트 런타임에서는 “어느 CPU 스레드가 어느 GPU context를 current로 갖는가”가 명시적 관리 대상이다.
  • cuCtxCreate는 새 컨텍스트를 만들면서 호출 스레드에 associate하며, 이미 current context가 있으면 새 context가 이를 대체하고 이후 cuCtxPopCurrent로 복원될 수 있다. 이 동작은 CUDA Runtime API의 암묵적 primary context 모델과 대비된다.
  • CUDA context 생성 플래그에는 CPU 스레드가 GPU 완료를 기다릴 때의 정책이 포함된다. 예: CU_CTX_SCHED_SPIN, CU_CTX_SCHED_YIELD, CU_CTX_SCHED_BLOCKING_SYNC, CU_CTX_SCHED_AUTO. 따라서 CUDA Driver API의 context는 단순한 device handle이 아니라 host-side wait behavior까지 포함하는 런타임 실행 정책 단위다.

Intel Level Zero

  • Level Zero는 CUDA의 “thread-current context” 모델과 다르게, ze_context_handle_t를 대부분의 메모리/명령 API에 명시 인자로 넘기는 스타일이다. 이 차이는 멀티스레드 런타임 설계에서 중요하다. CUDA Driver는 thread-local current state를 활용하고, Level Zero는 객체 handle 전달을 통해 상태를 노출한다.
  • Level Zero의 command list는 kernel launch뿐 아니라 memory copy, barrier, event signal/wait를 append하는 기록 객체에 가깝다. 따라서 PIM 런타임을 설계할 때 “연산 명령”과 “데이터 이동 명령”을 같은 command buffer에 넣을지, 별도 DMA queue와 compute queue로 분리할지 비교 기준이 된다.

PJRT/OpenXLA

출처: OpenXLA PJRT C API header

  • PJRT C API는 ABI 확장을 염두에 두고 거의 모든 구조체에 struct_size와 extension_start를 둔다. 이는 out-of-tree plugin이 다른 버전의 프레임워크와 만날 때, 구조체 크기로 caller/callee가 아는 필드 범위를 협상하는 방식이다.
  • PJRT의 비동기 완료 모델은 PJRT_Event 중심이다. PJRT_Event_IsReady, PJRT_Event_Await, PJRT_Event_OnReady가 존재하며, enqueue된 비동기 작업의 완료와 오류 상태를 event로 돌려준다. 이는 CUDA event, Level Zero event와 같은 하위 런타임 완료 통지 모델을 상위 ABI로 재포장하는 지점이다.
  • PJRT_Client_Create는 optional key-value store callback을 받을 수 있다. 이는 단일 디바이스 런타임을 넘어서 multi-process/multi-node 클라이언트 생성 시 플러그인이 rendezvous 정보를 교환할 수 있게 하기 위한 장치다. PIM이 메모리 채널/랭크 단위로 분산된 경우에도 유사한 topology discovery와 coordination 인터페이스가 필요할 수 있다.

UPMEM / PIM

출처: UPMEM Unleashed: Software Secrets for Speed

  • 2025년 논문 UPMEM Unleashed는 UPMEM SDK 수준에서도 호스트-PIM 데이터 전송과 NUMA 배치가 성능 편차의 주요 원인임을 지적한다. 논문은 PIM allocation API가 NUMA 구조를 더 잘 반영하면 host-PIM transfer throughput과 consistency를 개선할 수 있다고 보고한다.

출처: PIM-SHERPA

  • 2026년 PIM-SHERPA는 on-device LLM inference에서 prefill과 decode가 요구하는 메모리 속성이 충돌한다고 설명한다. Prefill은 cacheable weight reuse가 유리하지만, PIM decode는 PIM 동작을 확실히 트리거하기 위해 non-cacheable region이 필요하다는 문제다. 이는 LPDDR 계열 PIM 런타임이 단순 malloc/free가 아니라 cacheability, layout, swizzle, double buffering을 API로 다뤄야 함을 시사한다.

LPDDR6-PIM

참고: TechRadar LPDDR6/JESD209-6 보도

  • 이전 턴의 “LPDDR6 표준화 미완료” 표현은 수정해야 한다. 2026년 7월 기준 공개 보도들은 JEDEC이 LPDDR6 JESD209-6 표준을 이미 발표했다고 설명한다. 다만 이는 LPDDR6 메모리 표준에 대한 이야기이며, LPDDR6-PIM 명령 ISA나 런타임 SDK 표준이 공개되었다는 의미는 아니다.

  • PJRT가 PIM을 직접 지원하려면 최소한 PJRT_Buffer가 “일반 device memory”뿐 아니라 “PIM-triggerable memory region”을 표현해야 한다. 또한 PJRT_Executable은 StableHLO의 dot_general, convolution, gather/scatter 중 어떤 연산을 PIM 명령으로 lower할 수 있는지 명시해야 한다.

관련 맥락 / 배경 설명

  • CUDA/Level Zero는 accelerator-native runtime이고, PJRT는 ML compiler runtime ABI이며, UPMEM/LPDDR-PIM은 memory-attached execution 모델이다. 이 차이 때문에 동일한 “runtime”이라는 단어를 쓰더라도 인터페이스가 해결하는 문제가 다르다.
  • GPU 런타임은 대체로 “연산 장치로 데이터를 보내고 커널을 실행”하는 모델이다. PIM 런타임은 “데이터가 이미 있는 곳에서 연산이 일어나도록 메모리 접근을 설계”하는 모델이다. 그래서 PIM에서는 kernel launch overhead보다 cache coherence, address mapping, bank locality, memory attribute mismatch가 더 전면에 나온다.
  • 2026년 공개 연구 흐름은 PIM을 단독 범용 가속기라기보다 LLM decode, embedding lookup, bandwidth-bound GEMV/GEMM처럼 메모리 병목이 강한 구간에 붙이는 방향으로 보인다. 따라서 LPDDR6-PIM 런타임 설계도 CUDA식 범용 kernel launch API보다 “연산 패턴 제한 + 메모리 배치 제어 + framework lowering”에 가까울 가능성이 높다.

잠재적 오류 또는 확인 필요 지점

  • LPDDR6 JESD209-6의 세부 사양은 JEDEC 원문 접근이 제한될 수 있으므로, 공개 기사만으로 command/address 변경, sub-channel 수, DVFSL, ECC/security 기능을 단정하기보다 JEDEC 문서 또는 벤더 PHY/controller 문서로 재확인해야 한다.
  • Level Zero의 immediate command list와 regular command list의 정확한 동기화/ordering 보장은 spec 버전별로 확인이 필요하다. 특히 event wait/signal이 같은 queue 내부 ordering만 보장하는지, 여러 queue와 여러 engine 사이에서 어떤 fence가 필요한지 정리해야 한다.
  • PJRT의 “PIM backend” 실제 구현 사례는 아직 명확하지 않다. AWS Neuron, Intel GPU, Apple Metal, TPU, ROCm, CUDA 계열 PJRT 플러그인은 확인 가능하지만 UPMEM 또는 LPDDR-PIM용 공개 PJRT backend는 별도 조사가 필요하다.
  • UPMEM의 tasklet 수, MRAM 크기, rank당 DPU 수는 제품 세대와 DIMM 구성에 따라 달라질 수 있다. 논문/SDK 예제의 숫자를 일반 규격처럼 쓰면 오류가 될 수 있다.

다음 에이전트가 이어서 확장할 지점

  1. CUDA cuCtxPushCurrent/cuCtxPopCurrent, Level Zero zeContextCreate, PJRT PJRT_Client_Create를 실제 API 인자 수준에서 나란히 놓고, 멀티스레드 런타임에서 thread-local state와 explicit handle 방식의 장단점을 비교한다.
  1. PIM-SHERPA의 cacheable/non-cacheable weight 문제를 LPDDR6-PIM 런타임 요구사항으로 일반화한다. 특히 pim_alloc, pim_map_cacheable, pim_map_uncacheable, pim_flush, pim_barrier, pim_layout_transform 같은 최소 API 후보를 도출한다.
  1. StableHLO 연산 중 PIM으로 lowering하기 좋은 후보를 분류한다. 예: decode 단계 GEMV, low-bit dot product, embedding lookup, elementwise activation, reduction. 각 연산이 요구하는 memory layout, synchronization, host fallback 조건을 정리한다.

이번 턴의 확장 요약

이전 두 턴은 주로 "런타임이 무엇을 소유하는가"(컨텍스트, 버퍼, command queue)에 집중했다. 이번 턴은 seed의 5-6번 질문("스케줄링 상태는 어떻게 이동하는가", "task는 어떤 상태로 런타임을 빠져나가는가")이 아직 다뤄지지 않았다는 점을 지적하고, CUDA stream/graph node, Level Zero command list, PJRT execution, UPMEM DPU, 일반 OS 스케줄러의 상태를 하나의 공통 상태 모델로 정렬했다. 또한 UPMEM DPU 내부(tasklet 간) 동기화 프리미티브를 구체적인 API 수준으로 추가했다. 마지막으로 이전 턴의 "PIM-SHERPA" 논문 인용을 재검증할 수단이 이번 턴에는 없었음을 명시한다.

새로 추가한 사실

UPMEM tasklet 동기화 프리미티브 (DPU-side runtime library)

  • UPMEM DPU 프로그램은 호스트 스레드가 아니라 DPU 코어 위에서 협력적으로 스케줄링되는 최대 24개 tasklet을 다룬다. 이 tasklet들 사이의 동기화는 호스트 API(dpu_copy_to/dpu_copy_from)가 아니라 DPU 내부 런타임 라이브러리가 제공한다.
  • 핵심 프리미티브는 세 계열로 나뉜다: (1) mutex_pool/mutex_id_t — DPU당 물리적으로 제한된 개수의 하드웨어 뮤텍스를 tasklet들이 나눠 쓰는 방식(소프트웨어 스핀락이 아니라 하드웨어 자원 예약에 가깝다), (2) barrier_t — 정해진 tasklet 수가 모두 도달할 때까지 대기하는 배리어, (3) handshake — 두 tasklet 간 1:1 신호/대기 프리미티브.
  • MRAM(DPU당 공유 메인 메모리 역할) 접근은 tasklet들 사이에 자동 캐시 일관성이 없으므로, mutex/barrier는 "연산 동기화"뿐 아니라 "MRAM 접근 순서 보장" 역할도 겸한다. 이는 GPU의 __syncthreads()(block 내부 shared memory 가시성 보장)와 목적은 유사하지만, GPU가 SIMT 그룹 단위로 하드웨어가 동기화를 강제하는 반면 UPMEM은 명시적 라이브러리 호출로 tasklet 스케줄러(협력적 멀티스레딩, preemption 없음)에 양보 지점을 만든다는 점이 다르다.

CUDA context 상태 관리의 두 API 경로

  • cuCtxPushCurrent/cuCtxPopCurrent는 호출 스레드가 가진 컨텍스트 "스택"을 조작하는 legacy 경로로, 라이브러리가 임시로 다른 컨텍스트로 전환했다가 원래 상태로 복원해야 할 때(예: 콜백 안에서) 쓰인다.
  • cuCtxSetCurrent/cuCtxGetCurrent는 스택을 거치지 않고 호출 스레드의 "current context" 슬롯을 직접 읽고 쓰는 경로다. 두 경로가 공존하는 이유는 CUDA Driver API가 하위 호환을 유지하면서도, 스택 기반 push/pop보다 단순한 get/set 모델로 전환하려는 설계 방향을 보여준다.
  • Runtime API(cudaXxx)는 이 위에서 디바이스별로 자동 생성/참조 카운트되는 "primary context"(cuDevicePrimaryCtxRetain)를 암묵적으로 사용한다 — 즉 사용자가 명시적 컨텍스트를 만들지 않아도 되는 이유는 Runtime API가 내부적으로 primary context를 대신 관리해주기 때문이다.

여기서 드러나는 중요한 비대칭은, 가속기/PIM 런타임 대부분은 "Cancelled"에 해당하는 1급 상태가 없다는 점이다. OS 스케줄러는 임의 시점에 태스크를 선점·종료시킬 수 있지만, GPU 커널이나 DPU tasklet은 일단 dispatch되면 완료되거나(성공/실패) 전체 컨텍스트/디바이스를 리셋해야만 멈출 수 있다. 이는 PIM 런타임을 설계할 때 "부분 취소"가 사실상 불가능하다는 제약으로 이어진다 — 즉 LPDDR6-PIM 런타임의 pim_cancel() API를 만든다면, 실질적으로는 "cancel"이 아니라 "reset + 재초기화"에 가까운 의미로 정의해야 한다.

또한 "Failed"의 관찰 방식도 갈린다. CUDA는 커널 오류가 나면 해당 컨텍스트 전체가 "sticky error" 상태(cudaErrorIllegalAddress 등)로 오염되어 이후 모든 호출이 실패하고 컨텍스트를 재생성해야 하는 반면, Level Zero/PJRT는 오류를 해당 호출의 리턴값/이벤트로 국소화해 전달하려 시도한다. PIM처럼 하드웨어 fault 격리가 약할 가능성이 있는 시스템에서는 CUDA식 "컨텍스트 오염" 모델을 채택할 여지가 크다 — MRAM/뱅크 상태가 불확실해지면 해당 rank/channel 전체를 재초기화해야 할 수 있기 때문이다.

관련 맥락 / 배경 설명

  • OS 프로세스 스케줄러는 "선점 가능, 취소 가능, 상태 조회 가능"을 기본 전제로 설계되지만, 가속기/PIM 런타임은 "일단 던지면 완료를 기다리거나 전체를 리셋"이라는 훨씬 거친 제어 단위로 동작한다. 이 차이는 단순한 구현 디테일이 아니라, PIM처럼 host-device 캐시 일관성이 없는 시스템에서 "취소"가 데이터 무결성 문제로 직결되기 때문에 생기는 근본적 제약이다 — 실행 중인 tasklet을 강제로 멈추면 MRAM에 일부만 쓰여진 상태(torn write)가 남을 수 있다.
  • UPMEM의 하드웨어 뮤텍스가 "개수 제한된 자원"이라는 점은, seed 질문 7번("자원을 어떻게 할당하고 회수하는가")과 직결된다 — CPU 스레드용 뮤텍스는 논리적으로 무한히 만들 수 있지만, DPU 뮤텍스는 물리적 슬롯을 tasklet들이 나눠 써야 하므로 런타임(또는 프로그래머)이 뮤텍스 슬롯 자체를 할당/회수해야 하는 별도 자원 계층이 생긴다.

  • UPMEM 하드웨어 뮤텍스의 정확한 개수(세대별로 다를 수 있음)와 API 함수명(mutex_pool vs 실제 헤더상의 정확한 심볼명)은 SDK 버전에 따라 다를 수 있어 UPMEM 공식 SDK 문서(dpu_include의 mutex.h, barrier.h, handshake.h)로 재확인이 필요하다.
  • 위 상태표의 "Cancelled" 관련 서술(CUDA에 커널 취소 API가 없다는 것, Level Zero의 zeCommandListReset 동작)은 CUDA/Level Zero 최신 스펙 버전에서 변경되었을 가능성이 있어 재검증이 필요하다.

  2. 위 통합 스케줄링 상태표를 확장해 "Waiting"과 "Blocked"를 세분화한다 — 예를 들어 CUDA에서 stream 간 dependency 대기(cudaStreamWaitEvent)와 host-side blocking sync(cudaDeviceSynchronize)는 둘 다 "Blocked"로 뭉뚱그렸지만, 전자는 device-side queue 대기이고 후자는 host CPU 스레드 자체가 블록되는 것이라 성격이 다르다. 이 구분을 UPMEM/PIM의 host-DPU 동기화 경계에도 적용한다.
  3. "Cancelled 상태 부재" 문제를 LPDDR6-PIM 설계로 구체화한다 — pim_reset(rank) API가 진행 중인 다른 채널/뱅크의 연산에 영향을 주지 않도록 하려면 rank/channel 단위 격리가 필요한데, 이것이 LPDDR6의 sub-channel 구조(공개된 JESD209-6 정보 기준)와 어떻게 맞물리는지 다음 턴에서 조사한다.

LPDDR6 상태 수정 보강

출처: https://www.tomshardware.com/pc-components/dram/jedec-publishes-first-lpddr6-standard-new-interface-promises-double-the-effective-bandwidth-of-current-gen

  • LPDDR6 자체는 JESD209-6로 공개된 상태로 보는 것이 맞다. 공개 보도 기준으로 LPDDR6는 10.6-14.4Gbps/pin급 속도, 더 세분화된 sub-channel 구조, DVFS 계열 전력 제어, ECC/신뢰성 기능을 강조한다.
  • 그러나 이것은 LPDDR6 메모리 인터페이스 표준이지, LPDDR6-PIM 명령 ISA, PIM runtime ABI, PIM SDK 표준이 공개되었다는 뜻은 아니다. 따라서 “LPDDR6 표준은 있음”과 “LPDDR6-PIM 런타임 표준은 확인되지 않음”을 분리해야 한다.

더 구체화한 부분

1. Blocked/Waiting을 하나로 뭉개면 런타임 분석이 흐려진다

가속기 런타임에서는 “대기”가 최소 네 층으로 갈라진다.

이 구분이 중요한 이유는 completed가 항상 “host가 결과를 안전하게 읽을 수 있음”을 뜻하지 않기 때문이다. PIM에서는 특히 연산 완료, writeback 완료, host-visible 상태, cacheable view 복구가 별도 단계가 될 수 있다.

2. PIM task 상태와 memory-region 상태를 분리해야 한다

LPDDR6-PIM 후보 런타임은 task 상태 머신만으로 부족하다. 메모리 객체 자체가 상태를 가진다.

객체상태 후보
pim_taskCreated → Submitted → Running → Completed/Failed → Reclaimed
pim_queueOpen → Accepting → Draining → Quiesced → Reset
pim_regionHostWritable → Flushed → PimOwned → PimDirty → HostVisible
pim_layoutHostLayout → Transforming → PimLayout → Restoring
pim_contextInitialized → Active → ErrorContaminated → Resetting → Destroyed

여기서 핵심은 pim_task=Completed여도 pim_region=HostVisible이 아닐 수 있다는 점이다. CUDA/Level Zero에서는 event completion이 비교적 강한 완료 신호로 쓰이지만, PIM에서는 cacheability와 layout 때문에 완료 신호가 더 많은 후처리를 요구할 수 있다.

3. 언어 런타임과 가속기 런타임의 공통 구조

Go, Tokio, Node.js 같은 언어 런타임은 GPU/PIM 런타임과 하드웨어 계층은 다르지만 공통 구조가 있다.

관련 맥락 / 배경 설명

PIM-SHERPA의 cacheable/non-cacheable 문제는 단순 최적화가 아니라 런타임 객체 모델을 바꾸는 요구사항이다. 일반 GPU 런타임의 buffer는 대개 “어느 device memory에 있는가”를 중심으로 표현되지만, PIM 런타임의 buffer/region은 다음 속성을 가져야 한다.

이 API에서 pim_prepare와 pim_make_visible은 성능 보조 함수가 아니라 correctness 경계다.

잠재적 오류 또는 확인 필요 지점

  • JEDEC 원문은 접근 제한/robots 차단 때문에 이번 턴에서는 공개 보도와 요약 자료로만 확인했다. LPDDR6의 정확한 sub-channel 수, command/address encoding, ECC/tag bit 의미는 JESD209-6 원문으로 다시 대조해야 한다.
  • PIM-SHERPA는 제품 수준 PIM-enabled system을 언급하지만, 그것이 곧 공개 범용 LPDDR6-PIM SDK가 있다는 뜻은 아니다. 논문 환경이 특정 벤더 내부/시제품 스택인지, Android/Linux memory attribute 제어를 어떻게 구현했는지 원문 세부 확인이 필요하다.
  • UPMEM의 NUMA-aware allocation 제안은 논문상 “minor API extensions”로 제시되지만, 이것이 공식 UPMEM SDK에 병합된 기능인지, 연구용 패치인지 구분해야 한다.

다음 에이전트가 이어서 확장할 지점

  1. PIM-SHERPA 원문에서 DDB와 OWR의 실제 단계별 흐름을 뽑아 HostLayout → PimLayout → HostVisible 상태 머신으로 정리한다.
  1. LPDDR6 JESD209-6 원문 또는 신뢰 가능한 컨트롤러 벤더 문서에서 sub-channel, burst, ECC/tag, memory carve-out 정보를 확인하고, PIM command queue 설계와 연결한다.
  1. Go/Tokio/Node.js의 ready/wake 모델을 CUDA stream event, Level Zero event, PJRT event와 나란히 놓고 “런타임의 공통 최소 스케줄러 인터페이스”를 submit, park, wake, poll, fence, cancel/reset 관점으로 도출한다.

새로 추가한 사실

Go 런타임 스케줄러 (GMP 모델)

  • Go 스케줄러는 G(goroutine), M(OS 스레드), P(논리 프로세서, GOMAXPROCS 개수만큼 존재)로 구성된다. 각 P는 로컬 run queue를 가지며, 비어 있으면 다른 P의 큐에서 절반을 훔쳐오는 work-stealing으로 부하를 분산한다. 이와 별도로 global run queue가 있어 로컬 큐가 가득 찼을 때 넘친 goroutine을 받는다.
  • goroutine이 blocking syscall에 들어가면, 해당 M은 P를 놓아주고(sysmon 백그라운드 스레드가 감지) 새 M이 그 P를 가져와 다른 goroutine을 계속 실행한다. 이는 Turn 4의 "host-blocking wait는 host worker thread를 점유해 thread pool을 고갈시킨다"는 우려에 대한 언어 런타임 차원의 해법 사례다 — CUDA cudaDeviceSynchronize나 UPMEM launch sync에는 이런 자동 P 이양 메커니즘이 없다.
  • network I/O 대기는 별도 netpoller(Linux epoll 기반)가 처리하며, I/O가 준비되면 해당 goroutine을 다시 run queue에 넣는 방식으로 "wake"가 일어난다.

Tokio (Rust async 런타임)

  • Tokio의 실행 단위는 Future + Waker로 구성된 task다. Poll::Pending을 반환하면 executor는 해당 task를 run queue에서 제거하고, 대신 해당 future 내부가 보관한 Waker가 나중에 wake()를 호출해야만 task가 다시 큐에 들어간다.
  • multi-thread flavor에서는 각 worker thread가 로컬 큐를 가지고 work-stealing하며, 넘치는 task를 받는 global injector queue가 별도로 존재한다. reactor(mio 기반)가 OS I/O readiness를 감지해 대응하는 Waker를 호출하는 방식으로 "device 완료 → CPU 재개" 흐름과 유사한 구조를 만든다.

Node.js (libuv 이벤트 루프)

  • libuv 이벤트 루프는 timers → pending callbacks → idle/prepare → poll → check → close callbacks의 phase로 구성되며, 각 phase 사이에 promise microtask queue가 전부 비워진 뒤 다음 phase로 넘어간다.
  • JS 실행 자체는 단일 스레드이지만, 파일시스템 I/O·DNS·일부 crypto 연산처럼 blocking 가능한 작업은 libuv의 기본 4-스레드 thread pool로 오프로드된다. 이는 "논리적으로 단일 실행 흐름을 유지하면서 blocking 작업만 별도 풀로 위임"하는 패턴으로, CUDA Runtime API가 명시 컨텍스트 관리 없이 primary context 하나로 사용자 경험을 단순화하는 것과 목적이 유사하다.

Turn 3 정정: "OS는 취소 가능, 가속기는 불가능"은 단위를 혼동한 비교였다

Turn 3의 통합 상태표는 "Cancelled" 행에서 Linux SIGKILL을 가속기의 커널 취소 불가와 대비시켰는데, 이는 서로 다른 단위를 비교한 것이다. SIGKILL은 프로세스 전체를 강제 종료하는 것이지, 프로세스 내부의 개별 스레드 하나만 골라 강제 종료하는 것이 아니다. 실제로 Go의 goroutine, Java의 스레드(Thread.stop()은 데이터 무결성 문제로 deprecated), Tokio의 task 모두 세부 실행 단위 차원에서는 강제 취소가 없고 협조적 취소(cooperative cancellation: context.Context, AbortController, CancellationToken)만 가능하다.

즉 더 정확한 일반화는 다음과 같다.

계층굵은 단위(강제 종료 가능)가는 단위(협조적 취소만 가능)

언어 런타임런타임 프로세스 종료goroutine/task/promise(협조적 취소만)

PIMrank/channel reset(가정)개별 tasklet/command(취소 API 없음)

이 대칭은 LPDDR6-PIM 설계에 직접 시사점을 준다: pim_cancel(task)이라는 세밀한 API를 만드는 것은 다른 모든 계층의 선례에 비춰볼 때 실현 가능성이 낮고, 대신 pim_task에는 협조적 취소 플래그(예: 다음 barrier 지점에서 확인하는 cancellation token)를, pim_rank/pim_channel 단위에는 강제 reset API를 두는 이층 구조가 더 현실적인 설계다.

관련 맥락 / 배경 설명

  • Go/Tokio/Node.js는 모두 "wake 조건을 등록하고 조건 충족 시 재스케줄"하는 동일한 골격을 공유하지만, wake의 트리거가 스케줄러 내부(netpoller, mio reactor, libuv)에 있는지, 외부 호출자가 poll로 직접 확인해야 하는지가 갈린다. 가속기/PIM 런타임은 인터럽트 배선 비용과 지연시간 특성 때문에 poll 우선 설계를 택하는 경향이 있고, 이는 언어 런타임이 CPU 사이클을 아끼기 위해 wake 우선 설계를 택하는 것과 대조된다 — 결국 "누가 대기 중인 작업을 재개시킬 책임을 지는가"라는 동일한 문제를 서로 다른 하드웨어 제약 아래서 풀고 있는 것이다.
  • "협조적 취소만 가능하다"는 원칙은 seed 질문 6번("프로그램이나 task는 어떤 상태로 런타임을 빠져나가는가")에 대한 답을 더 좁힌다 — 대부분의 런타임에서 task는 자신이 다음 체크포인트(barrier, await, syscall 복귀)에 도달하기 전까지는 외부에서 강제로 빠져나가게 만들 수 없다. 즉 "취소됨" 상태는 사실 "취소 요청이 수락된 뒤, 실행 중이던 작업이 스스로 안전 지점에서 종료를 확인하고 나가는" 상태에 가깝다.

새로 추가한 사실

PIM-SHERPA의 DDB/OWR 흐름 확인

출처: arXiv:2603.09216

  • PIM-SHERPA는 실제로 prefill/decode 사이의 두 불일치를 문제로 삼는다:
  1. prefill은 weight reuse 때문에 cacheable region이 유리함
  2. decode에서 PIM을 확실히 트리거하려면 non-cacheable region이 필요함
  3. host-friendly weight layout과 PIM-aware swizzled layout이 다름
  • 논문이 제안한 두 방법은 다음처럼 런타임 상태 전이로 해석할 수 있다.
  • DDB, DRAM Double Buffering: weight를 non-cacheable PIM-aware region에 유지하고, 다음 layer의 swizzled weight를 작은 cacheable buffer로 미리 가져온다.
  • OWR, Online Weight Rearrangement: GEMM 직전에 on-demand swizzled memory copy를 수행해 layout 불일치를 그 시점에 해소한다.

CUDA reset의 실제 범위

출처: [NVIDIA CUDA Runtime API, cudaDeviceReset](https://docs.nvidia.com/cuda/cuda-runtime-api/group__CUDART__DEVICE.html)

  • cudaDeviceReset()은 현재 프로세스의 current device에 연결된 allocation, stream, event, array, external resource 등 대부분의 CUDA runtime 상태를 정리하고 device 상태를 reset한다. 이후 같은 device에 대한 API 호출은 device를 다시 초기화한다. NVIDIA 문서는 reset 호출 시 다른 host thread가 device를 접근 중이지 않도록 caller가 보장해야 한다고 명시한다.
  • 중요한 예외도 있다. 문서상 cudaMallocAsync()/pool 기반 async allocation은 cudaDeviceReset()만으로 파괴되지 않고 명시적으로 해제해야 한다. 즉 “reset이면 모든 리소스가 항상 사라진다”는 단순화는 틀리다. 런타임의 reset은 allocation domain별로 다르게 동작할 수 있다.

CUDA async notification

  • CUDA Runtime API에는 cudaDeviceRegisterAsyncNotification()이 존재한다. callback은 async notification을 받지만, callback 내부에서 다시 CUDA API를 호출하는 것은 금지/제한된다. 이는 PIM 런타임 설계에도 시사점이 있다. 완료 interrupt/callback을 제공하더라도 callback은 “상태 기록 + 사용자 worker 깨우기” 정도로 제한해야지, callback 안에서 pim_submit()이나 pim_free()를 허용하면 재진입성과 deadlock 문제가 커진다.

더 구체화한 부분

관련 맥락 / 배경 설명

GPU 런타임은 보통 kernel completed를 중심으로 상태를 닫는다. 반면 PIM 런타임은 operation completed, memory write completed, host cache coherent, layout restored가 서로 다른 사건이다. 이 차이 때문에 PIM 런타임의 event는 CUDA event보다 더 풍부한 payload를 가져야 한다.

Event payloadGPU 런타임PIM 런타임 후보

partial-write 여부드묾필요

특히 PIM fault가 발생하면 “task 실패”만 기록하면 안 된다. 어떤 memory region이 부분 갱신되었는지, host가 그 region을 다시 읽어도 되는지, rank/channel reset이 필요한지까지 event/error 객체가 전달해야 한다.

잠재적 오류 또는 확인 필요 지점

  • Turn 5의 zeCommandListReset을 “큐 단위 reset/cancel”처럼 읽을 수 있는 표현은 정밀하지 않다. 이름 그대로 command list 객체 재사용/초기화 성격이 강하며, 이미 device에서 실행 중인 개별 command를 안전하게 취소하는 1급 API로 단정하면 안 된다. Level Zero의 queue, fence, event, command list reset 범위는 공식 spec 원문으로 다시 분리 확인해야 한다.
  • PIM-SHERPA의 DDB/OWR은 product-level PIM-enabled system을 대상으로 하지만, 이것이 공개 LPDDR6-PIM SDK 존재를 뜻하지는 않는다. 논문의 소프트웨어 방법을 LPDDR6-PIM API 후보로 일반화할 수는 있으나, 실제 JEDEC/벤더 ABI로 단정하면 안 된다.
  • CUDA reset 설명에서 async memory pool 예외가 중요하다. LPDDR-PIM도 region allocator, host pinned buffer, command descriptor pool, firmware queue memory가 서로 다른 lifetime을 가질 수 있으므로 “context reset = 모든 객체 소멸” 모델은 위험하다.

새로 추가한 사실

Linux 커널의 GPU 스케줄링 계층 (DRM scheduler)

  • Linux 커널에는 CFS(Completely Fair Scheduler)와 별개로, GPU 작업 제출을 다루는 커널 내장 스케줄러 프레임워크인 drm_sched(DRM GPU scheduler)가 존재한다. amdgpu, etnaviv, panfrost, lima, v3d 같은 오픈소스 GPU 드라이버가 이를 공유해서 쓴다. 이는 각 GPU 엔진(ring)별로 job run queue를 유지하고, job 간 의존성(fence)을 기다렸다가 순서대로 하드웨어 ring에 제출하는 커널 공간 스케줄러다.
  • dma_fence는 커널 서브시스템 간 완료 신호를 주고받는 표준 동기화 객체다. GPU 드라이버가 job 완료 시 dma_fence를 signal하면, 디스플레이 서브시스템이나 다른 GPU 엔진, 심지어 다른 드라이버(예: V4L2 캡처 장치)가 이를 기다릴 수 있다. 이는 CUDA event나 Level Zero event보다 한 계층 아래에 있는, 커널이 소유하는 완료 신호다 — 사용자 공간 이벤트는 결국 이 dma_fence를 감싼 것에 가깝다.
  • TDR(Timeout Detection and Recovery)은 GPU job이 정해진 시간 내에 끝나지 않으면 커널 드라이버가 해당 엔진 또는 GPU 전체를 강제로 리셋하는 메커니즘이다. 이는 Turn 3~6에서 다룬 "가속기에는 세밀한 취소가 없고 거친 reset만 있다"는 관찰의 실제 커널 구현 사례다 — cudaDeviceReset()이나 애플리케이션 판단 이전에, 이미 커널 드라이버가 자체적으로 정지된 job을 감지해 reset을 트리거하는 자동화된 계층이 존재한다.

NVIDIA 드라이버의 예외적 위치

  • NVIDIA의 전통적 독점 드라이버(nvidia.ko)는 오픈소스 드라이버들과 달리 drm_sched를 사용하지 않고 자체 채널/컨텍스트 관리 로직을 커널 모듈 내부에 둔다. 반면 최근 공개된 open-gpu-kernel-modules(2022년 이후 오픈소스화된 커널 모듈)는 GSP(GPU System Processor)라는 GPU 자체에 내장된 RISC-V 기반 마이크로컨트롤러로 상당한 관리 로직(클럭, 전력, 일부 스케줄링)을 오프로드하는 방향으로 가고 있다 — 즉 "커널 드라이버가 소유하는 상태"의 일부가 다시 GPU 온보드 펌웨어로 이관되는 추세다. 이 부분은 공개 정보가 제한적이라 아래 확인 필요 항목에 플래그한다.

TDR을 LPDDR-PIM reset 절차에 연결

Turn 6의 quiesce → drain → poison → reset(scope) → rebuild 절차는 애플리케이션이 능동적으로 호출하는 경로였다. 여기에 TDR과 동일한 자동 감지 경로를 추가해야 한다: PIM 커맨드 큐가 일정 시간 내 완료 이벤트를 내지 못하면, 커널 드라이버(또는 firmware) 계층이 스스로 PIM_ERROR_TIMEOUT을 발생시키고 해당 rank/channel을 자동으로 quiesce 상태로 전이시켜야 한다. 이는 사용자 코드가 hang을 명시적으로 감지해 reset을 호출할 것이라고 가정하면 안 되는 이유이기도 하다 — 애플리케이션이 죽거나 응답하지 않을 수 있기 때문에, 최소한의 워치독은 런타임/드라이버 내부에 있어야 한다.

관련 맥락 / 배경 설명

dma_fence가 시사하는 바는, "완료 통지"라는 개념이 실제로는 계층마다 다시 구현되는 것이 아니라 커널 최하단에서 한 번 정의된 뒤 위로 갈수록 더 풍부한 의미(디바이스 종속적 payload, 사용자 공간 편의 API)로 감싸진다는 점이다. CUDA event, Level Zero event, PJRT event는 결국 이 커널 fence 위에 각자의 어휘를 씌운 것에 가깝다. PIM 런타임을 설계할 때도 "PIM event"를 처음부터 새로 정의하기보다, 이미 존재하는 커널 fence 프레임워크(dma_fence) 위에 PIM 특화 payload(affected region, visibility state 등, Turn 6에서 도출)를 얹는 방식이 이식성과 커널 통합 측면에서 더 현실적일 수 있다 — 특히 LPDDR6-PIM이 표준 메모리 컨트롤러 드라이버(JEDEC 호환 DDR 컨트롤러)와 공존해야 한다면, 완전히 새로운 동기화 프레임워크보다 기존 커널 관용구를 재사용하는 편이 채택 장벽이 낮다.

새로 추가한 사실

Linux DRM GPU scheduler

Linux kernel 문서는 DRM GPU scheduler가 userspace job을 software queue에 넣고, scheduler가 이를 hardware run queue로 스케줄한다고 설명한다. software queue에는 priority가 있으며, scheduler는 dependency handling을 제공하고, 실제 hardware submit은 드라이버 callback이 담당한다. 출처: Linux kernel DRM Memory Management - GPU Scheduler

이 구조는 사용자 공간 런타임의 stream/queue와 커널 공간의 hardware queue 사이에 중간 software scheduling layer가 있음을 보여준다.

계층큐 객체하는 일

Kernel driverDRM scheduler entity/run queue프로세스 간 공정성, priority, fence dependency
Hardware/firmwarering buffer, doorbell, firmware queue실제 command fetch/dispatch

dma_fence는 성공/실패를 담는 completion 객체다

dma_fence_get_status_locked()는 fence가 아직 signaled되지 않았으면 0, 성공 완료면 1, error completion이면 음수 error code를 반환한다. 문서에는 예시로 -ECANCELED와 -ETIME이 언급된다. 출처: Linux kernel dma-buf / dma_fence documentation

따라서 fence는 단순 “완료됨/안 됨” 플래그가 아니다. 최소한 다음 세 상태를 갖는다.

상태의미

Signaled error작업은 종료 처리됐지만 취소/타임아웃/리셋 등 오류 동반

이 점은 PIM event 설계에 중요하다. pim_event도 boolean ready만으로는 부족하고, ready + status + affected_region + visibility_state를 가져야 한다.

DRM syncobj는 host wait와 queued dependency를 구분한다

DRM syncobj 문서는 GPU work가 syncobj를 wait하도록 enqueue될 때와, host가 DRM_IOCTL_SYNCOBJ_WAIT로 기다릴 때를 구분한다. 특히 WAIT_FOR_SUBMIT 플래그는 아직 fence가 붙지 않은 syncobj에 대해 “먼저 fence 제출을 기다린 뒤 그 fence를 기다리는” 동작을 제공한다. 출처: Linux kernel DRM sync objects

이것은 Turn 4~5에서 나눈 host-blocking wait와 queue dependency wait가 Linux 커널 동기화 객체에서도 실제로 분리된다는 근거다.

PIM 런타임에는 event와 reservation object가 모두 필요하다

GPU 메모리 관리에서 buffer object는 보통 “이 버퍼를 누가 쓰고 있는가”를 추적하기 위해 fence를 reservation object에 붙인다. PIM도 비슷한 구조가 필요하다.

관련 맥락 / 배경 설명

Linux DRM scheduler의 존재는 “런타임 스케줄러”를 하나로 보면 안 된다는 점을 보여준다. 실제 실행 경로는 보통 다층이다.

framework op
  -> compiler executable
  -> user-mode runtime queue
  -> kernel driver scheduler entity
  -> hardware ring / firmware queue
  -> engine execution
  -> fence/event completion
  -> memory visibility transition

PIM 런타임은 GPU보다 memory visibility 단계가 더 전면에 나온다. GPU에서는 command completion과 buffer visibility가 대체로 런타임/driver 내부 규칙으로 감춰지지만, PIM에서는 cacheable/non-cacheable alias, swizzled layout, rank/channel placement가 correctness 조건이 된다. 따라서 PIM의 scheduler는 “언제 실행할까”뿐 아니라 “이 region이 지금 실행 가능한 memory state인가”를 함께 판단해야 한다.

잠재적 오류 또는 확인 필요 지점

  • Level Zero의 zeCommandListReset, zeCommandQueueSynchronize, zeFenceHostSynchronize는 여전히 공식 spec 원문 대조가 필요하다. 현재까지의 안전한 해석은 zeCommandListReset을 실행 중 command 취소 API로 보면 안 된다는 것이다. command list 객체를 재기록 가능한 상태로 되돌리는 API에 가깝고, 이미 hardware queue에 제출된 작업의 세밀한 취소를 보장한다고 단정하면 위험하다.

출처: NVIDIA open-gpu-kernel-modules

  • NVIDIA open-gpu-kernel-modules README는 GSP firmware와 맞는 user-space driver component가 필요하다고 설명한다. 그러나 이것만으로 “스케줄링의 어느 부분이 GSP-RM으로 이동했는가”까지 확정할 수는 없다.
  • dma_fence error code를 PIM error model에 그대로 복사하면 안 된다. Linux kernel 내부 error vocabulary와 user-facing PIM SDK error vocabulary는 분리해야 한다. 예를 들어 kernel fence는 -ETIME을 줄 수 있지만, user API는 이를 PIM_ERROR_TIMEOUT과 required_reset_scope로 확장해 보여주는 편이 낫다.

다음 에이전트가 이어서 확장할 지점

  1. Level Zero 공식 spec 또는 헤더 원문에서 zeCommandListReset, zeCommandQueueSynchronize, zeFenceHostSynchronize, event reset/signal/wait의 정확한 lifetime과 “실행 중 취소 불가” 범위를 확정한다.
  1. Linux DRM의 dma_resv/reservation object 모델을 조사해, pim_region에 read fence/write fence/visibility fence를 붙이는 설계를 더 구체화한다.
  1. PIM 런타임 최종 상태 머신을 command state, queue state, region state, kernel fence state, user event state의 5개 축으로 나누고, timeout/reset이 각 축을 어떻게 전이시키는지 통합 다이어그램으로 확장한다.

새로 추가한 사실

Windows Fiber API — seed가 명시한 "fiber"의 실제 사례 (지금까지 다뤄지지 않음)

  • Windows fiber는 ConvertThreadToFiber로 현재 스레드를 fiber 실행 모드로 전환한 뒤, CreateFiber로 추가 fiber를 만들고 SwitchToFiber로 명시적으로 전환하는 순수 user-mode 협조적 스케줄링 단위다. 커널은 fiber의 존재 자체를 모른다 — 스케줄링 개입이 전혀 없고, 프로그래머가 직접 SwitchToFiber를 호출하는 지점에서만 전환이 일어난다.
  • 이는 Go goroutine과 결정적으로 다르다. Go 1.14부터는 goroutine에 async preemption(비협조적 선점, signal 기반)이 도입되어 무한루프 goroutine도 강제로 스케줄러가 끊고 다른 goroutine을 돌릴 수 있다. 반면 fiber는 명시적 SwitchToFiber 호출 없이는 영원히 선점되지 않는다 — "협조적 실행 단위" 안에서도 fiber(완전 수동 전환)와 goroutine(런타임이 개입할 수 있는 협조+선점 혼합)은 스펙트럼상 다른 위치에 있다.

Level Zero의 host-blocking 지점을 큐 레벨과 fence 레벨로 분리

  • zeCommandQueueSynchronize는 큐 전체가 처리한 모든 command list의 완료를 호출 스레드가 블로킹 대기하는 API로, CUDA cudaDeviceSynchronize에 대응하는 큐 단위 host-blocking wait다.
  • zeFenceHostSynchronize는 특정 fence 객체 하나의 신호를 기다리는 더 세밀한 host-blocking wait로, CUDA의 cudaEventSynchronize에 대응한다. Turn 4~5가 나눈 "host-blocking wait vs queue dependency wait" 구분을 Level Zero 안에서 다시 "큐 전체 대기 vs 개별 fence 대기"로 한 단계 더 세분화할 수 있다는 뜻이다. 다만 이 두 API의 정확한 동작(타임아웃 파라미터 의미, 부분 실패 시 반환값)은 원문 미확인 상태로 유지한다.

관련 맥락 / 배경 설명

Fiber(완전 수동, 커널 미개입) vs goroutine(런타임이 개입 가능) vs OS 스레드(커널이 강제 선점)의 3단 스펙트럼은, 가속기 계열에서도 유사하게 재현된다: PIM tasklet(barrier/handshake 지점에서만 양보, Turn 3)은 fiber에 가깝고, GPU TDR(워치독이 강제 개입, Turn 7)은 OS 스레드의 커널 선점에 가깝다. 즉 "누가 실행을 멈출 권한을 가지는가"라는 축은 언어 런타임과 가속기 런타임을 관통하는 동일한 설계 스펙트럼이며, LPDDR6-PIM은 이 스펙트럼에서 "협조적 지점(barrier)에서의 자발적 양보"와 "워치독의 강제 개입" 둘 다를 함께 가져야 한다는 것이 지금까지의 결론이다.

  • Windows Fiber API 서술은 일반적으로 잘 알려진 Win32 API 동작이지만, 이 프로젝트의 초점이 Linux/가속기/PIM 런타임이라는 점에서 Windows 사례가 LPDDR6-PIM 설계에 얼마나 직접 적용 가능한지는 추가 판단이 필요하다.

Turn 10 / Codex

이번 턴의 확장 요약

이번 턴에서는 Turn 9가 남긴 세 가지 축 중 두 가지를 실제 문서 기준으로 보강한다. 첫째, Level Zero의 zeCommandListReset, zeCommandQueueSynchronize, zeFenceHostSynchronize를 원문 헤더 기준으로 확인해 “reset은 실행 중 command 취소가 아니다”를 더 강하게 확정한다. 둘째, Linux dma_resv가 더 이상 단순한 “exclusive/shared fence” 모델만으로 설명되지 않고, KERNEL/WRITE/READ/BOOKKEEP usage 계층을 갖는다는 점을 반영해 PIM reservation 설계를 갱신한다.

새로 추가한 사실

Level Zero command list reset은 취소 API가 아니다

Level Zero 헤더의 zeCommandListReset 설명은 command list를 “initial empty state”로 되돌려 command append가 가능하게 한다고 되어 있고, 더 중요하게는 애플리케이션이 “device가 command list를 참조 중이 아님”을 보장해야 한다고 명시한다. 따라서 이미 queue에 제출되어 device가 실행 중인 command를 취소하는 API로 해석하면 안 된다. 출처: [oneAPI Level Zero ze_api.h](https://raw.githubusercontent.com/oneapi-src/level-zero/master/include/ze_api.h)

dma_resv는 단순 exclusive/shared fence보다 usage 계층 모델에 가깝다

Linux 문서 기준 dma_resv는 buffer에 붙은 fence들의 컨테이너이고, DMA_RESV_USAGE_KERNEL, WRITE, READ, BOOKKEEP을 구분한다. usage 순서는 KERNEL < WRITE < READ < BOOKKEEP이며, 특정 usage로 fence를 조회하면 그보다 낮은 usage의 fence도 포함된다. 예를 들어 READ fence를 요청하면 WRITE와 KERNEL fence도 함께 고려된다. 출처: Linux dma-buf / dma-resv 문서

CPU access coherency와 device completion은 별도다

Linux dma-buf 문서는 implicit fence poll이 DMA 완료를 알려줄 뿐, CPU 접근 전 필요한 cache flush나 준비 작업은 별도로 필요하다고 설명한다. 또한 CPU mmap 접근은 DMA_BUF_IOCTL_SYNC의 SYNC_START/SYNC_END로 감싸야 하며, 이 동기화는 cache coherency를 위한 것이지 다른 device/process의 동시 접근을 막는 lock은 아니다. 출처: Linux dma-buf CPU access / implicit fence 문서

핵심 수정은 visibility fence를 단순 write fence에 섞지 않는 것이다. PIM에서는 “PIM write 완료”와 “host가 cacheable view로 안전하게 읽을 수 있음”이 분리되므로, PIM_RESV_VISIBILITY를 별도 usage로 둬야 PimDirty → HostVisible 전이가 명확해진다.

CUDA warp/thread/block을 실행 단위 표에 추가

CUDA 문서 기준 개발자 관점에서 thread가 병렬성의 기본 단위이고, thread는 block과 grid로 조직된다. 각 thread는 자기 상태와 control flow를 가질 수 있지만, 성능 관점에서는 같은 warp 안의 divergence를 줄여야 한다. CUDA memory scope도 thread/register, block/shared memory, grid/global memory로 나뉜다. 출처: NVIDIA CUDA Programming Guide

claim/release와 make_visible의 정확한 위치

Turn 6과 Turn 9의 상태 머신을 결합하면 정상 경로는 다음처럼 된다.

Host wants PIM write
  pim_claim(region, DEVICE_WRITE)
    Region: HostWritable -> PreparedForPIM
    Reservation: add READ/WRITE dependencies that must complete first

  pim_prepare(region, noncacheable, pim_layout)

여기서 pim_wait(event)는 command completion만 보장하고, pim_make_visible()은 host-visible correctness를 보장한다. 이 둘을 합치면 사용자는 편하지만, 런타임 내부 상태 분석에서는 반드시 분리해야 한다.

Level Zero의 command list 모델은 PIM runtime 설계에 좋은 힌트를 준다. command list는 “실행 중인 작업” 자체가 아니라 “실행될 명령을 담은 재사용 가능한 기록 객체”다. 그래서 reset은 command list object를 비우는 동작이지, device engine에서 이미 실행 중인 command를 멈추는 동작이 아니다.

LPDDR-PIM도 비슷하게 pim_program 또는 pim_command_buffer와 pim_task를 분리하는 편이 좋다.

잠재적 오류 또는 확인 필요 지점

  • Turn 9의 “이번 세션은 웹 접근이 차단된 것 같다”는 메타 판단은 이번 턴 기준으로는 더 이상 맞지 않다. 이번 턴에서는 Level Zero 헤더, Linux kernel 문서, NVIDIA CUDA 문서를 직접 확인했다. 다만 UPMEM 공식 SDK 문서의 tasklet 세부 API는 이번 턴에서 원문 확인하지 못했고, 논문/일반 지식 기반으로만 연결했다.
  • dma_resv의 usage 계층을 PIM에 그대로 복제하면 안 된다. Linux의 BOOKKEEP은 implicit sync에 참여하지 않는 내부 fence까지 포괄하지만, PIM의 VISIBILITY는 correctness에 직접 관여한다. 따라서 PIM_RESV_VISIBILITY는 Linux BOOKKEEP의 단순 별칭이 아니라 PIM 전용 확장으로 봐야 한다.
  • CUDA global memory 설명에는 cudaDeviceReset이 allocation을 free한다고 되어 있지만, Turn 6에서 언급한 async memory pool 예외처럼 CUDA runtime 세부 기능별 lifetime 차이가 있을 수 있다. PIM 설계에서도 context reset이 모든 host-exported region, descriptor pool, kernel fence를 자동 소멸시킨다고 단정하면 위험하다.
  • UPMEM tasklet을 “fiber에 가깝다”고 보는 유비는 유용하지만 완전 동일하지 않다. fiber는 host user-mode stack switching 단위이고, tasklet은 DPU-side runtime/firmware/SDK가 관리하는 PIM 내부 worker다. 공통점은 host kernel scheduler가 직접 보지 못한다는 점이다.

다음 에이전트가 이어서 확장할 지점

  1. UPMEM 공식 SDK 문서에서 DPU tasklet 수, WRAM/MRAM/host transfer API, barrier/mutex/handshake primitive를 원문 기준으로 확인하고, CUDA thread/warp/block 표에 DPU, tasklet, rank, DPU set을 추가한다.
  1. pim_reservation 설계를 sub-region 단위로 확장한다. 같은 pim_region 안에서도 bank/channel이 다르면 동시 PIM write가 가능한지, 가능하다면 fence를 region 전체가 아니라 {region, offset, length, bank} 범위에 붙여야 하는지 검토한다.
  1. Level Zero의 command list/queue/fence/event 모델을 CUDA graph/stream/event, PJRT executable/buffer/event와 나란히 놓고, LPDDR-PIM의 최종 API를 command_buffer, queue, task, event, reservation, region 여섯 객체로 정리한다.

새로 추가한 사실

UPMEM DPU/rank/tasklet 구조 (일반 지식, SDK 원문 미검증)

  • 공개된 UPMEM 논문/자료 기준으로 각 DPU(Data Processing Unit)는 다수의 tasklet(하드웨어 스레드, 통상 최대 24개)을 실행하며, tasklet들은 하나의 DPU 안에서 WRAM(수십 KB 규모의 scratchpad)과 MRAM(수십 MB 규모의 DRAM-side 메모리)을 공유한다.
  • 여러 DPU가 모여 rank를 이루고, host는 보통 rank 단위로 DPU program을 launch한다 — 즉 개별 DPU 하나만 골라 실행을 트리거하는 것이 아니라 rank에 속한 DPU 전체가 동시에 프로그램 카운터를 시작점으로 이동한다.
  • dpu_launch()류의 API는 동기(호출자가 완료까지 블로킹) 또는 비동기(완료 여부를 별도로 poll/sync) 모드를 갖는 것으로 알려져 있다. 이는 Turn 5~10에서 반복된 "host-blocking wait vs 비동기 poll/event" 이분법이 PIM 진영에서도 실제로 존재함을 시사한다.

Linux drm_gpuvm과 range-based dma_resv 논의 (일반 지식, 미검증)

  • dma_resv는 원래 dma-buf(GEM object) 전체 단위로 fence를 붙이는 설계다. sparse/virtual GPU memory(가상 주소 공간에 여러 물리 backing을 매핑)가 널리 쓰이면서, 커널 GPU 드라이버 커뮤니티에서는 object 전체가 아니라 가상 주소 range 단위로 fence/의존성을 추적해야 하는 요구가 생겼고, 이를 다루기 위한 drm_gpuvm(GPU virtual memory manager) 같은 헬퍼가 커널에 추가된 것으로 알려져 있다.
  • 요점은, "object 전체 단위 reservation"에서 "sub-range 단위 reservation"으로 가는 것이 Linux 커널 GPU 드라이버 진영에서도 자명하게 이미 해결된 문제가 아니라 여전히 개발 중인 영역이라는 것이다.

더 구체화한 부분

Turn 10 정정: pim_reservation의 sub-region 세분화는 dma_resv 확장이 아니라 dma_resv가 아직 못 푼 문제를 새로 푸는 것이다

Turn 10 다음 과제 2번은 "같은 pim_region 안에서도 bank/channel이 다르면 fence를 {region, offset, length, bank} 범위에 붙여야 하는지 검토"였다. 이를 "Linux dma_resv 모델을 더 세분화하면 된다"는 식으로 접근하면 안 된다 — dma_resv 자체가 object 단위이지 range 단위가 아니기 때문에, PIM이 sub-region fence를 원한다면 이는 기존 커널 관용구를 그대로 확장하는 것이 아니라 GPU 진영도 아직 골머리를 앓는 문제(range-based tracking)를 PIM 도메인에서 새로 설계하는 것에 가깝다.

이 점을 반영해 pim_reservation을 2계층으로 나누는 편이 더 현실적이다.

즉 기본값은 region 전체 단위 fence(구현이 단순하고 correctness가 보수적)로 두고, 성능이 중요한 워크로드에서만 bank-level sub-range reservation을 opt-in으로 여는 2단 설계가, "복잡도 대비 이득이 확실할 때만 세분화한다"는 원칙에 맞다. 이는 애초에 커널 GPU 드라이버가 sparse memory라는 명확한 필요가 생긴 뒤에야 range-based 추적을 도입한 순서와도 일치한다.

관련 맥락 / 배경 설명

rank가 host launch의 최소 단위라는 점은, PIM 런타임 설계에서 "동시성의 기본 단위"가 GPU/CPU와 다르다는 것을 뜻한다. GPU는 개별 kernel launch를 스트림 단위로 세밀하게 스케줄링할 수 있고, CPU는 스레드 단위로 매우 세밀하다. 반면 PIM(적어도 UPMEM 모델)은 "rank 전체가 같은 프로그램을 동시에 시작"하는 SPMD에 가까운 굵은 launch 단위를 가진다. 이는 LPDDR6-PIM 설계에도 시사점을 준다 — pim_submit()이 개별 command 단위로 매우 세밀한 스케줄링을 제공한다고 가정하면 안 되고, 실제로는 "channel/rank 단위로 동일 연산을 broadcast하는" 형태가 하드웨어 현실에 더 가까울 수 있다. 즉 pim_queue가 다루는 "명령"은 GPU의 개별 kernel launch보다 UPMEM의 rank-wide launch에 더 가까울 가능성이 있다.

잠재적 오류 또는 확인 필요 지점

  • UPMEM의 tasklet 개수(24), WRAM/MRAM 용량, dpu_launch/동기화 API 이름은 모두 일반적으로 알려진 논문 수준 지식이며 이번 턴에도 SDK 원문(공식 API 헤더/매뉴얼) 대조를 하지 못했다. Turn 10에서 이미 요청된 항목이 세 번째 턴째 이월된다.
  • drm_gpuvm/range-based dma_resv 서술은 GPU 드라이버 커뮤니티의 일반적 동향에 대한 서술이며, 정확한 자료구조명과 현재 커널 버전에서의 지원 범위는 커널 소스나 커밋 로그로 재확인이 필요하다.
  • "rank 전체가 동시에 launch된다"는 서술이 모든 UPMEM 세대/구성에 동일하게 적용되는지, 아니면 부분 rank/부분 DPU 선택 launch가 가능한 옵션이 있는지는 미확인이다 — 이 부분을 사실인 것처럼 LPDDR6-PIM 설계 결론에 과도하게 반영하지 않도록 주의해야 한다.

관련 맥락 / 배경 설명

PIM 런타임의 scheduler는 CPU scheduler나 GPU stream scheduler와 성격이 다르다. CPU scheduler는 실행 단위의 레지스터/스택 상태를 전환하고, GPU stream scheduler는 command dependency와 engine occupancy를 관리한다. PIM scheduler는 여기에 더해 메모리 위치 자체가 실행 가능성의 일부가 된다.

따라서 LPDDR6-PIM의 실행 흐름은 다음처럼 정리하는 편이 낫다.

framework op
  -> compiler/runtime partitions tensor by topology
  -> region allocator places chunks on channel/rank/bank
  -> command buffer records PIM operation
  -> submit descriptor declares placement + read/write reservations
  -> scheduler checks deps + bank conflicts + visibility/layout state
  -> PIM engine executes memory-side command
  -> event signals command completion
  -> visibility fence makes result host/framework-visible

이 구조에서는 event ready와 tensor ready가 다르다. event ready는 command가 끝났다는 뜻이고, tensor ready는 host/framework가 읽어도 되는 visibility/layout 상태까지 회복됐다는 뜻이다.

새로 추가한 사실

Gang scheduling — Turn 11/12의 "rank 전체 broadcast launch"에 붙일 정확한 이름

  • Feitelson과 Rudolph가 병렬 시스템 스케줄링 문헌에서 정립한 gang scheduling(co-scheduling)은, 서로 통신/동기화하는 다수의 실행 단위(프로세스/스레드)를 하나의 그룹으로 묶어 반드시 동시에 스케줄하거나 전혀 스케줄하지 않는 정책이다. 개별 단위를 독립적으로 time-slice하면 barrier에서 서로를 기다리는 동안 다른 것들이 이미 선점되어 전체 처리량이 급락하기 때문이다.
  • 이는 Turn 11의 "rank 전체가 같은 프로그램을 동시에 시작"과 정확히 같은 문제의식이다. UPMEM DPU들이 같은 rank 안에서 WRAM/MRAM을 공유하며 barrier로 동기화한다면, host launch가 rank 단위(all-or-nothing)로 설계된 것은 우연이 아니라 gang scheduling이 요구하는 필연적 구조다 — 부분 DPU만 선택 launch하는 API가 잘 알려져 있지 않다는 Turn 11의 관찰도 이 틀에서 설명 가능하다.

Go 런타임의 GMP 모델과 async preemption의 정확한 메커니즘

  • Go 런타임은 G(goroutine)/M(OS thread)/P(processor, 스케줄링 컨텍스트)의 3요소로 구성된 M:N 스케줄러를 쓴다. 각 P는 자신의 로컬 run queue를 가지며, 자신의 큐가 비면 다른 P의 큐에서 goroutine을 훔쳐오는 work-stealing 정책을 쓴다.
  • Go 1.14의 async preemption은 별도 monitor 스레드(sysmon)가 goroutine이 일정 시간(기본 10ms) 이상 실행되면 해당 M에 SIGURG 시그널을 보내 강제로 goroutine을 중단시키고 스택 정보를 저장한 뒤 스케줄러로 제어를 되돌리는 방식이다. Turn 9의 "goroutine은 fiber보다 강한 선점을 가진다"는 서술을 이 구체적 메커니즘으로 뒷받침한다.

Tokio(work-stealing async executor) vs Node.js(단일 스레드 event loop + libuv 스레드 풀)

  • Tokio는 Go 런타임과 유사하게 다중 OS 스레드에 걸친 work-stealing 스케줄러를 쓴다. Future는 poll될 때까지 상태가 없다가, Poll::Pending을 반환하면 waker가 등록되고, I/O 준비 완료 시 waker가 executor 큐에 task를 다시 넣는다.
  • Node.js는 반대 극단이다: JS 실행 자체는 단일 메인 스레드에서만 일어나고(동시성 없음), 파일 I/O·DNS·일부 crypto 연산 같은 블로킹 작업만 libuv의 기본 4개짜리 스레드 풀(UV_THREADPOOL_SIZE)로 오프로드한 뒤 콜백을 이벤트 루프 큐에 다시 넣는다. 즉 "병렬성"이 아니라 "블로킹 회피"가 목적인 스레드 풀이다.

Turn 12 보정: pim_submit의 기본 스케줄링 단위는 "task 하나"가 아니라 "gang"이어야 한다

Turn 12의 pim_submit_desc_t는 하나의 pim_task_t를 반환하는 구조였는데, gang scheduling 관점에서 보면 이는 오해를 부를 수 있다. rank 단위로 강결합된 실행이라면, pim_submit()이 반환하는 것은 개별 DPU/engine 하나의 task가 아니라 해당 rank/gang 전체를 대표하는 단일 handle이어야 하고, 그 안의 개별 실행 단위(DPU별 진행 상황)는 host에 노출되지 않는 것이 하드웨어 현실에 더 가깝다. 즉:

typedef struct {

  uint32_t      gang_size;      // 이 launch에 참여한 DPU/engine 수
  bool          partial_select; // 부분 선택 launch 지원 여부(SDK별 상이, 미확정)
} pim_gang_result_t;

partial_select를 명시적 필드로 둔 이유는, Turn 11이 지적했듯 "부분 rank/부분 DPU 선택 launch가 가능한지"가 여전히 미확인이기 때문이다 — API가 이를 지원하지 않는 하드웨어에서는 이 필드가 항상 false로 고정된다.

관련 맥락 / 배경 설명

Gang scheduling은 역사적으로 BSP(Bulk-Synchronous Parallel) 모델과 짝을 이룬다. BSP는 계산을 "동시 실행 → 전역 barrier → 통신" 단계로 나누는데, 이는 정확히 Turn 3~9에서 반복된 "협조적 barrier 지점에서만 양보"라는 PIM/tasklet 서술과 같은 구조다. 즉 LPDDR6-PIM 런타임이 채택해야 할 실행 모델은 우연한 설계 선택이 아니라, 강결합 메모리 공유 + 저수준 통신 비용이라는 하드웨어 제약이 필연적으로 BSP/gang scheduling으로 수렴시키는 것으로 이해하는 편이 더 원칙적이다. 반대로 host 쪽 API(command 제출, 논블로킹 wrapper)는 이런 제약이 없으므로 Tokio/Go 스타일의 work-stealing이나 Node.js 스타일의 event loop을 자유롭게 참고할 수 있다 — "PIM 실행 자체는 BSP/gang scheduling, host-side 오케스트레이션은 async runtime 스타일"이라는 2단 구조가 이번 턴의 결론이다.

잠재적 오류 또는 확인 필요 지점

  • 이번 턴의 gang scheduling/BSP 연결은 병렬 컴퓨팅 이론의 일반 지식(Feitelson & Rudolph)이며, UPMEM이 공식적으로 자신의 launch 모델을 "gang scheduling"이라고 명명하는지는 확인하지 못했다 — 이는 서술의 정확성이 아니라 적용 프레임의 타당성에 대한 주장이므로, SDK 문서에서 이 용어를 쓰지 않는다고 해서 유비 자체가 틀린 것은 아니지만 과도하게 공식 용어처럼 인용하지 않도록 주의해야 한다.
  • partial_select 필드는 Turn 11~13에 걸쳐 세 번째로 "미확인"으로 남는 UPMEM 세부사항이다. 다음 에이전트는 이 항목을 별도로 추적하기보다, SDK 원문 확인이 계속 불가능하다면 "미확인 상태를 명시한 채로 설계를 확정"하는 방향으로 전환하는 편이 낫다.
  • Go async preemption의 10ms 기본 임계값과 SIGURG 사용은 일반적으로 알려진 Go 런타임 지식이나, 버전별 세부 구현(예: 특정 GOOS에서의 예외)은 재확인이 필요하다.
  • CUDA warp scheduler의 GTO 정책 서술은 학술 문헌에서 자주 언급되는 일반적 설명이며, 특정 세대(Volta 이후 independent thread scheduling)에서의 정확한 변형은 미검증이다.

다음 에이전트가 이어서 확장할 지점

  1. gang scheduling/BSP 프레임을 pim_context의 topology 모델(Turn 12)과 결합해, "어느 계층까지 gang이 강제되는가"(rank 전체인지, channel 내 특정 bank group인지)를 스케줄러 정책 파라미터로 명시하고, opt-in sub-range reservation(Turn 12)과의 상호작용(gang 내부에서도 bank-level 병렬은 허용되는지)을 정리한다.
  2. host-side PIM API를 Tokio/Node.js 스타일의 비동기 wrapper로 감쌀 때, pim_event의 waker 등록 방식(콜백 vs poll vs async/await 스타일)을 설계하고 이것이 5축 통합 상태 머신(Turn 9)의 어느 지점에서 트리거되는지 명시한다.
  3. UPMEM SDK 원문 확인이 계속 불가능할 경우를 대비해, 지금까지 미확인 상태로 남은 모든 UPMEM 세부사항(tasklet 수, launch granularity, partial selection 여부)을 별도 "가정 목록"으로 모아 명시적으로 확정/보류 상태를 구분한 최종 표를 만든다.

Turn 14 / Codex

이번 턴의 확장 요약

이번 턴에서는 Turn 13의 gang scheduling/BSP 관점을 유지하되, 이를 PIM 런타임 객체 모델의 정책 필드로 끌어내린다. 핵심 보강은 두 가지다.

  1. PIM launch 단위는 단순 task가 아니라 gang, placement, partition이 결합된 제출 단위로 봐야 한다.
  2. host-side async runtime은 pim_event를 단순 poll 객체로 두지 말고, callback/waker/completion queue로 연결할 수 있어야 한다.

추가로 UPMEM 공식 SDK 원문은 여전히 직접 확인하지 못했지만, SimplePIM/DaPPA 같은 공개 프레임워크가 UPMEM 위에서 분산, 회수, 통신, 메모리 관리를 추상화한다는 점은 확인된다. 출처: SimplePIM GitHub, SimplePIM 논문, DaPPA 논문

새로 추가한 사실

PIM 프레임워크의 실제 추상화는 launch보다 data orchestration에 가깝다

SimplePIM README는 UPMEM 기반 프레임워크가 management, communication, processing API를 나누고, gather/scatter/broadcast/allreduce/allgather 같은 통신 연산과 map/zip/reduction 같은 처리 연산을 제공한다고 설명한다. 즉 low-level dpu_launch 하나를 감싸는 수준이 아니라, 런타임이 다음을 함께 관리한다.

계층역할

communicationhost-PIM, PIM-PIM 데이터 이동과 collective
processingmap, zip, reduction 같은 data-parallel operator 실행

이 구조는 LPDDR6-PIM 런타임에서도 queue와 event만으로는 부족하고, partitioner, placement planner, collective runtime이 필요하다는 근거가 된다.

DaPPA는 하드웨어 자원 관리를 프로그래머에게서 숨기는 방향을 명시한다

DaPPA 논문은 프로그래머가 DPU 배치, 데이터 분산, 데이터 회수, 메모리 관리를 직접 하지 않도록 data-parallel pattern 기반 인터페이스를 제공한다고 요약된다. 출처: DaPPA

따라서 LPDDR6-PIM 런타임의 상위 계층은 다음 두 API 층으로 나뉘는 편이 자연스럽다.

API 층사용 대상노출 객체

2025년 UPMEM 관련 연구는 NUMA-aware PIM allocation까지 언급한다

UPMEM Unleashed는 UPMEM SDK를 기반으로 하되, PIM allocation에 NUMA 구조를 고려하는 API 확장이 host-PIM transfer consistency/throughput을 개선할 수 있다고 보고한다. 출처: UPMEM Unleashed

이는 Turn 12의 pim_topology_t가 단지 device 내부 topology만 담으면 부족하다는 뜻이다. host socket, NUMA node, memory controller와 PIM rank/channel의 관계까지 런타임 placement cost model에 들어가야 한다.

더 구체화한 부분

pim_submit은 task가 아니라 gang submission을 만들어야 한다

Turn 13의 지적처럼 PIM 실행이 rank/channel 단위로 묶이면 반환값은 개별 worker의 task가 아니라 gang 전체의 제출 인스턴스여야 한다.

typedef enum {
  PIM_GANG_RANK,
  PIM_GANG_CHANNEL,
  PIM_GANG_BANK_GROUP,
  PIM_GANG_EXPLICIT_MASK
} pim_gang_scope_t;

typedef struct {
  pim_gang_scope_t scope;
  pim_placement_t placement;
  uint32_t participant_mask;
  bool allow_partial;
} pim_gang_desc_t;

typedef struct {
  pim_command_buffer_t *cmd;
  pim_gang_desc_t gang;
  pim_reservation_key_t *reads;
  size_t num_reads;
  pim_reservation_key_t *writes;
  size_t num_writes;
  pim_event_t **deps;
  size_t num_deps;
} pim_submit_desc_t;

typedef struct {
  pim_task_t *task;
  pim_event_t *event;
  pim_gang_desc_t actual_gang;
  uint32_t rejected_participants;
} pim_submit_result_t;

여기서 중요한 필드는 actual_gang과 rejected_participants다. 사용자가 부분 실행을 허용했더라도 scheduler가 bank conflict, poisoned region, unavailable rank 때문에 실제 gang을 줄였을 수 있다. 반대로 allow_partial=false이면 일부 participant라도 준비되지 않았을 때 전체 제출은 NotReady 또는 Rejected가 된다.

gang scheduling과 sub-region reservation의 충돌 규칙

Turn 12의 sub-region reservation은 gang scheduling과 결합될 때 다음처럼 판정해야 한다.

상황스케줄링 판정
같은 gang 내부에서 서로 다른 bank write하드웨어가 bank 병렬 write를 보장하면 허용
같은 gang 내부에서 같은 bank write순서가 정의되지 않으면 거부 또는 command buffer 내부 ordering 필요
서로 다른 gang이 같은 region readvisibility/layout 상태가 같으면 허용
서로 다른 gang이 같은 sub-range writewrite-after-write hazard로 직렬화
region-level visibility transform pendingsub-range 병렬성보다 우선하여 전체 region block
poisoned sub-range 존재region 전체 poison인지 sub-range poison인지 정책 필요

즉 sub-region reservation은 “성능 최적화”이면서 동시에 “gang admission control”의 입력이다. scheduler는 단순히 queue head를 꺼내는 것이 아니라, 다음 함수를 수행한다.

admit(gang, reservations):
  deps_ready?
  placement_available?
  no_conflicting_write?
  required_layout_ready?
  visibility_state_compatible?
  fault_domain_healthy?

host-side async 연결: pim_event는 waker를 가져야 한다

Tokio/Node.js 비교를 PIM API로 내리면 pim_event는 다음 세 가지 사용 방식을 모두 지원해야 한다.

방식API 후보의미
blocking waitpim_event_wait(event, timeout)현재 host thread block
pollingpim_event_query(event, &status)event 상태 즉시 확인
async wakepim_event_set_callback(event, cb) 또는 fd exportexecutor/event loop에 completion 전달

PIM event 상태 전이는 다음처럼 host async runtime과 연결된다.

PIM engine completion
  -> kernel/driver fence signal
  -> runtime completion queue entry
  -> pim_event: NotReady -> ReadySuccess/ReadyError
  -> registered callback/waker fired
  -> async task resumes
  -> optional pim_make_visible()
  -> tensor/framework buffer becomes HostVisible

중요한 점은 waker가 깨어나는 시점이 반드시 “host가 데이터를 읽어도 되는 시점”은 아니라는 것이다. 기본 event는 command completion이고, tensor readiness는 make_visible까지 포함한 별도 future로 모델링하는 편이 안전하다.

pim_future_t pim_submit_async(...);        // command completion
pim_future_t pim_make_visible_async(...);  // host-visible completion

잠재적 오류 또는 확인 필요 지점

  • UPMEM 공식 SDK의 dpu_alloc, dpu_load, dpu_launch, dpu_sync 정확한 시그니처와 partial DPU selection 여부는 이번에도 공식 SDK 원문으로 확정하지 못했다. 다만 SimplePIM/DaPPA가 UPMEM 위에서 management/communication/processing 계층을 제공한다는 점은 확인했다.
  • Turn 13의 “gang scheduling”은 PIM 설계 프레임으로는 유용하지만, UPMEM 공식 문서가 그 용어를 쓰는지는 별개다. 최종 문서에서는 “gang scheduling에 해당하는 구조로 해석할 수 있다” 정도가 안전하다.
  • pim_event_set_callback 같은 callback API는 편하지만 callback 실행 스레드, lock ordering, reentrancy 문제가 생긴다. 실제 SDK에서는 eventfd/fd export 방식이 더 안전할 수 있다.
  • sub-range poison을 허용하면 복구는 정교해지지만 API 복잡도가 크게 늘어난다. 첫 버전은 region-level poison, 고급 모드에서 bank/sub-range poison을 여는 설계가 더 현실적이다.

  1. pim_gang_desc_t와 pim_reservation_key_t를 결합해 정상 제출, partial gang rejection, bank conflict, poisoned region, visibility pending의 admission-control 상태표를 만든다.
  1. UPMEM 공식 SDK 원문 확인이 계속 어렵다면, SimplePIM/DaPPA/UPMEM Unleashed 기반의 “확인된 사실”과 dpu_launch 세부사항 같은 “보류 가정”을 분리한 최종 PIM 런타임 근거 표를 작성한다.

이번 턴의 확장 요약

이번 턴에서도 WebSearch 권한이 거부되어(6회 연속 — Turn 3, 5, 7, 9, 13, 15) UPMEM SDK 원문 확인은 또다시 불발됐다. 대신 Turn 14가 남긴 세 과제 중 두 가지를 정면으로 다룬다. 첫째, pim_future_t를 command-completion/visibility-completion으로 분리하고 Linux의 실제 explicit-fencing 메커니즘(sync_file)과 io_uring의 SQ/CQ 모델을 근거로 fd 기반 async API를 구체화한다. 둘째, pim_gang_desc_t와 pim_reservation_key_t를 결합한 admission-control 상태표를 완성한다. 마지막으로 Turn 14가 요청한 "확인된 사실 vs 보류 가정" 분리표를 지금까지 13개 턴에 걸쳐 흩어진 UPMEM 관련 서술을 모아 정리한다.

새로 추가한 사실

sync_file — Linux에서 이미 존재하는 "fence를 pollable fd로 감싸는" 메커니즘

  • Linux 커널의 sync_file(drivers/dma-buf/sync_file.c, include/linux/sync_file.h)은 dma_fence를 file descriptor로 감싸 poll()/select()/epoll()로 대기 가능하게 만드는 explicit fencing 메커니즘이다. Android graphics stack과 DRM atomic commit(DRM_IOCTL_MODE_ATOMIC의 OUT_FENCE_PTR, IN_FENCE_FD)에서 GPU와 display controller 간 동기화에 실제로 쓰인다.
  • 이는 Turn 14가 제안한 "pim_event가 callback/waker/eventfd를 지원해야 한다"는 요구를 이미 커널이 GPU 진영에서 풀어놓은 문제로 연결한다 — PIM도 새 IPC 메커니즘을 발명하기보다 pim_event를 sync_file처럼 fd로 export하면, 기존 epoll, Tokio의 AsyncFd(tokio::io::unix::AsyncFd), Node.js의 uv_poll_t(libuv가 임의의 raw fd를 이벤트 루프에 등록하는 표준 방식)에 별도 브릿지 코드 없이 바로 꽂을 수 있다.
  • 다만 이번 턴에도 커널 소스나 최신 문서로 직접 재확인하지 못했으며, 이는 GPU 드라이버 스택에 대한 일반적으로 잘 알려진 지식에 기반한 서술이다.

io_uring의 SQ/CQ 분리 모델 — pim_command_buffer/pim_queue 설계와 구조적으로 유사한 선례

  • io_uring은 submission queue(SQ)와 completion queue(CQ)를 별도의 lock-free ring buffer로 분리하고, 애플리케이션이 SQ에 요청을 채워 커널에 알리면 커널이 비동기로 처리한 뒤 결과를 CQ에 채우는 모델이다. 필요시 eventfd를 CQ에 등록해 새 completion이 생길 때마다 fd가 신호되게 할 수 있어, epoll 루프와도 자연스럽게 통합된다.
  • 지금까지 설계해 온 pim_command_buffer(기록된 descriptor) + pim_queue(제출 순서) + pim_event(completion) 3분할은 io_uring의 SQ(descriptor 기록) / 커널 처리 / CQ(completion) 3분할과 구조적으로 대응한다. 차이는 io_uring이 범용 syscall batching을 목표로 한다면, pim_queue는 gang/placement/reservation이라는 PIM 특유의 admission 조건을 추가로 검사해야 한다는 점이다.

더 구체화한 부분

pim_future_t를 command-completion / visibility-completion으로 분리하고 fd 기반으로 통일

Turn 14가 제안한 pim_event_wait/pim_event_query/pim_event_set_callback 3원화 대신, 세 방식 모두를 하나의 fd 위에서 구현하는 편이 커널 선례와 일관된다.

typedef struct {
  int fence_fd;          // sync_file 스타일: dup 가능, poll/epoll 가능, close 시 fence는 유지
  pim_event_kind_t kind; // PIM_EVENT_COMMAND_COMPLETION | PIM_EVENT_VISIBILITY_COMPLETION
} pim_future_t;

관련 맥락 / 배경 설명

sync_file과 io_uring을 함께 놓고 보면, Linux 커널이 "비동기 완료를 어떻게 상위 계층에 통지할 것인가"라는 문제를 두 가지 서로 다른 방식으로 풀었다는 것이 드러난다 — sync_file은 "완료 하나당 fd 하나"(fine-grained, GPU/display용), io_uring은 "완료 다수를 링 버퍼 하나로 batch"(throughput 지향, 범용 I/O용). PIM 런타임은 이 둘을 계층별로 나눠 쓰는 것이 합리적이다: pim_future(개별 command/visibility completion)는 sync_file 모델을, pim_queue의 대량 completion 소비(호스트가 수백 개의 gang submission을 순회할 때)는 io_uring의 CQ 모델을 따르면, 소수의 중요 이벤트는 세밀하게 기다리고 다수의 루틴한 완료는 배치로 소비하는 두 워크로드 패턴을 모두 지원할 수 있다.

잠재적 오류 또는 확인 필요 지점

  • sync_file과 io_uring 서술은 Linux 커널의 잘 알려진 일반 지식이지만, 이번 턴에도 커널 소스나 최신 문서로 직접 재검증하지 못했다. 특히 sync_file이 최신 커널에서 dma_fence의 DMA_RESV_USAGE_* 계층(Turn 10)과 정확히 어떻게 상호작용하는지는 미확인이다.
  • admission-control 표의 "서로 다른 gang이 같은 sub-range write" 항목에서 "뒤에 온 gang을 큐에 hold"라고 서술했는데, 이 직렬화가 pim_queue 레벨에서 자동으로 일어나야 하는지, 아니면 사용자가 명시적으로 deps에 앞선 write event를 넣어야 하는지는 API 설계상 결정되지 않은 채로 남아 있다 — 자동 직렬화는 편의성을 높이지만 숨은 성능 저하(silent serialization)를 유발할 수 있어 opt-in/opt-out 정책이 필요하다.
  • pim_future_t의 fence_fd를 실제 하드웨어가 지원할 수 있는지(즉 PIM 컨트롤러의 completion 신호가 커널까지 올라와 fd로 export될 수 있는 인터럽트/폴링 경로를 가지는지)는 순수 소프트웨어 설계 관점의 가정이며, 실제 LPDDR6-PIM 하드웨어의 인터럽트 아키텍처에 따라 불가능할 수도 있다.
  • UPMEM 관련 미확인 항목이 이번 턴까지 5~6턴 연속 이월되고 있다는 점은, Turn 13이 이미 제안했듯 "확인 시도를 계속 반복하기보다 미확인 상태를 명시한 채 설계를 확정"하는 편이 이제는 더 합리적이라는 신호로 봐야 한다.

다음 에이전트가 이어서 확장할 지점

  1. pim_queue의 대량 completion 소비 경로를 io_uring CQ 스타일의 batch drain API(pim_queue_drain_completions(queue, out_events[], max_count))로 구체화하고, 개별 pim_future fd 기반 대기와 언제 어느 것을 써야 하는지 사용 가이드를 정리한다.
  2. admission-control 표의 "cross-gang sub-range write 자동 직렬화 vs 사용자 명시 의존성" 정책을 확정하고, 이를 pim_submit_desc_t의 새 플래그(예: auto_serialize_conflicts)로 반영한다.
  3. 지금까지 13개 턴에 걸쳐 설계된 8개 핵심 객체(context, topology, region, reservation, command_buffer, gang, queue, task/event/future)를 하나의 통합 헤더 초안(pim_runtime.h 형태)으로 처음 정리하여, 지금까지 분산된 구조체 정의들 간의 이름 충돌·필드 중복을 점검한다.

Turn 16 / Codex

이번 턴의 확장 요약

이번 턴은 Turn 15의 fd/future 설계를 한 단계 더 정리한다. 핵심 보강은 두 가지다.

  1. sync_file은 “PIM event fd”의 좋은 선례지만, 정확히는 dma-buf에 현재 붙어 있는 fence 집합의 snapshot fd이므로, PIM command 하나당 항상 1:1로 대응한다고 단정하면 안 된다.
  2. io_uring식 CQ는 pim_queue의 batch completion 모델에 적합하지만, eventfd 알림은 completion 개수와 1:1 대응하지 않는 hint로 봐야 한다.

따라서 LPDDR6-PIM 런타임은 개별 completion을 위한 pim_future와 대량 완료 소비를 위한 pim_completion_queue를 분리하는 편이 좋다.

새로 추가한 사실

dma-buf의 sync_file export는 “현재 fence 집합의 snapshot”이다

Linux dma-buf 문서는 DMA_BUF_IOCTL_EXPORT_SYNC_FILE이 dma-buf에 현재 붙은 fence들을 sync_file로 export한다고 설명한다. 중요한 점은 이것이 즉시 wait와 다르게 “현재 시점의 fence snapshot”을 나중에 기다릴 수 있게 만든다는 것이다. 또한 DMA_BUF_SYNC_READ만 지정하면 writer 완료를 기다리고, DMA_BUF_SYNC_WRITE를 지정하면 read/write 사용자 전체 완료를 기다린다. 출처: Linux dma-buf documentation

PIM 설계로 옮기면 다음 차이가 생긴다.

객체의미
pim_task_event_fd특정 제출 instance의 command completion
pim_region_sync_fd특정 region/sub-range에 현재 붙은 reservation fence snapshot
pim_visibility_fdcache/layout/visibility transform 완료 fence

즉 pim_future_t 하나로 모든 것을 표현할 수는 있지만, 내부 kind는 반드시 구분해야 한다.

dma-buf poll readiness와 CPU visibility는 여전히 별개다

dma-buf 문서는 poll이 DMA fence completion을 알려줄 뿐, CPU access 전 cache flush나 coherency preparation은 별도로 필요하다고 설명한다. DMA_BUF_IOCTL_SYNC도 cache coherency를 위한 것이며, 다른 device/process의 동시 접근을 막는 lock이 아니다. 출처: Linux dma-buf CPU access / implicit fence section

이는 Turn 14~15의 “command future ≠ visibility future” 구분을 더 강하게 뒷받침한다.

io_uring eventfd notification은 completion 개수와 1:1 대응하지 않는다

io_uring_register(2) 문서는 IORING_REGISTER_EVENTFD로 completion notification을 받을 수 있지만, batched CQE 여러 개가 하나의 eventfd notification만 만들 수 있으므로 알림 개수와 CQE 개수를 직접 대응시키면 안 된다고 설명한다. eventfd는 “CQ를 확인하라”는 힌트다. 출처: io_uring_register(2)

PIM completion queue도 같은 규칙을 따라야 한다. pim_queue_eventfd가 readable이면 pim_queue_drain_completions()를 호출해야지, fd read 값 하나를 task 하나 완료로 해석하면 안 된다.

관련 맥락 / 배경 설명

PIM 런타임의 completion 설계는 GPU fence 모델과 io_uring 모델의 중간에 있다.

선례PIM에 가져올 점그대로 가져오면 안 되는 점
dma-buf pollbuffer readiness와 fence wait의 fd 통합CPU visibility까지 보장한다고 보면 안 됨

따라서 LPDDR6-PIM의 완료 경로는 다음처럼 계층화된다.

PIM command finishes
  -> task event signaled
  -> queue CQE posted
  -> queue eventfd becomes readable as a hint
  -> host drains completions
  -> optional region visibility future submitted
  -> visibility CQE posted
  -> framework tensor marked HostVisible

이 구조에서 task completed와 tensor usable은 서로 다른 시점이다.

잠재적 오류 또는 확인 필요 지점

  • Turn 15의 sync_file 설명은 방향은 맞지만, “완료 하나당 fd 하나”로 일반화하면 부정확하다. dma-buf의 export sync file은 현재 fence set snapshot이며, PIM에서는 task event fd와 region reservation snapshot fd를 구분해야 한다.
  • pim_queue_get_eventfd()는 실제 LPDDR6-PIM 컨트롤러가 interrupt 또는 kernel polling thread를 제공한다는 가정이 필요하다. 하드웨어가 completion interrupt를 못 주면 eventfd는 runtime thread가 polling해서 신호하는 software emulation이 된다.
  • 자동 직렬화는 correctness에는 편하지만 성능 병목을 숨긴다. 기본 정책은 REJECT 또는 명시적 QUEUE_WAIT가 더 투명하다.
  • UPMEM SDK 세부 API는 여전히 최종 설계의 필수 근거로 삼기보다 “기존 PIM 플랫폼 참고 사례”로 격리하는 편이 안전하다.
