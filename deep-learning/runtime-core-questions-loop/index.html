Runtime 중심질문 재시작 루프

목표

Claude와 Codex가 서로의 응답을 검증 종료점이 아니라 다음 확장의 입력으로 사용하며 자료를 누적한다.

초기 seed

Runtime 중심질문 재시작 seed

이 루프의 목표는 런타임을 넓게 조사하되, 항상 아래 중심질문으로 되돌아오는 것이다. 특정 하위 주제(PIM, DRAM, RowHammer, 특정 논문, 특정 API)에 오래 머무르지 않는다. 하위 주제는 중심질문을 설명하는 사례로만 사용한다.

중심 질문

  1. 런타임이란 무엇인가?
  2. 런타임 내부는 어떤 모듈로 구성되는가?
  3. task, process, thread, fiber, coroutine, stream, command queue는 런타임 안에서 어떻게 표현되는가?
  4. 런타임은 task/process/work item을 어떤 정책으로 스케줄링하는가?
  5. 스케줄링 상태는 ready, running, blocked, waiting, completed, failed, cancelled 같은 상태로 어떻게 이동하는가?
  6. 프로그램이나 task는 어떤 상태와 산출물을 남기고 런타임을 빠져나가는가?
  7. 런타임은 CPU, GPU, accelerator, memory, buffer, queue, event, context, handle 같은 자원을 어떻게 할당하고 회수하는가?
  8. 런타임이 소유하는 상태와 OS kernel, device driver, compiler, framework가 소유하는 상태는 어떻게 나뉘는가?
  9. CUDA Runtime/Driver, Intel Level Zero, PJRT/OpenXLA, UPMEM SDK, JVM, Go, Tokio, Node.js, Linux scheduler를 비교하면 어떤 공통 구조가 보이는가?
  10. PIM 또는 LPDDR6-PIM runtime을 설계한다면 어떤 최소 runtime object, state machine, scheduler, allocator, API가 필요한가?

드리프트 방지 규칙

  • 매 턴은 반드시 중심질문 1~10 중 최소 2개에 직접 답해야 한다.
  • 매 턴 첫 부분에 이번 턴이 답하는 중심질문: Qx, Qy를 적는다.
  • 한 하위 주제가 2턴 이상 이어지면 다음 턴은 반드시 일반 runtime 모델, OS/language runtime, accelerator runtime 중 하나로 돌아온다.
  • PIM/DRAM/RowHammer/refresh 같은 하드웨어 세부사항은 Q10의 설계 사례로만 다루고, 전체 턴의 절반을 넘기지 않는다.
  • 외부 문헌을 찾더라도 결론은 반드시 runtime object, state transition, scheduler, resource ownership 중 하나로 환원한다.
  • "검증 완료", "문제 없음", "추가 없음"으로 끝내지 않는다.

우선 확장 순서

  1. 일반 런타임 모델: init, context, scheduler, allocator, executor, event loop, error/completion.
  2. OS runtime: Linux process/thread scheduler, process state, syscall/blocking/wakeup, exit status.
  3. 언어 런타임: JVM, Go, Tokio, Node.js의 task 표현, work-stealing, event loop, GC/cleanup.
  4. Accelerator runtime: CUDA Runtime/Driver, Level Zero, ROCm/HIP, OpenCL/SYCL의 context, stream/queue, event, memory object.
  5. ML compiler/runtime: PJRT/OpenXLA의 client/device/buffer/executable/event 구조.
  6. PIM runtime 설계: 앞의 공통 모델을 바탕으로 최소 object model, state machine, scheduler, allocator, API 후보를 도출한다.

이번 턴의 확장 요약

누적 응답이 없으므로, 이후 모든 턴이 되돌아올 일반 런타임 참조 모델(canonical runtime reference model) 을 먼저 못박는다. 핵심 주장은 하나다: 어떤 런타임이든 (object model) + (state machine) + (scheduler) + (allocator) + (API/ABI surface) 의 5요소로 환원된다. 이 턴은 이 5요소를 정의하고, CUDA·Level Zero·JVM·Go·Tokio·Node.js·Linux·PJRT를 이 축 위에 얹은 비교 스켈레톤을 만들어 드리프트 방지의 기준선으로 삼는다.

새로 추가한 사실

  1. "런타임"은 3의미가 겹쳐 쓰인다. ① 시점(runtime=실행 시각, compile time의 반대), ② 소프트웨어 실체(runtime system/library, 예: cudart, libgo, JVM), ③ 환경(runtime environment, 예: JRE). 이 루프에서 관심 대상은 ②다. Q1의 답은 "언어/실행 모델이 가정하지만 스스로 구현하지 않는 서비스를, 프로그램이 도는 동안 공급하는 소프트웨어 계층"이다.
  2. 런타임의 공통 서비스 목록: 메모리 관리(할당·회수·GC), 실행 단위의 생성/스케줄/동기화, 하드웨어 자원 추상화(device/queue/buffer), 오류·예외·완료 신호, 동적 로딩/링킹, ABI/calling convention 지원, 메타데이터/리플렉션. 이 목록 자체가 Q2의 모듈 경계와 1:1로 대응한다.
  3. Linux 기본 스케줄러는 더 이상 CFS가 아니다. 커널 6.6(2023-10)부터 EEVDF(Earliest Eligible Virtual Deadline First) 가 기본이다. Q4/Q9에서 "Linux scheduler=CFS"라고 쓰면 최신 커널 기준으로 틀린다.
  4. CUDA Runtime API는 명시적 context를 노출하지 않는다. cudart는 지연 초기화되며 device마다 primary context를 자동 생성·공유한다. 명시적 CUcontext는 Driver API(cuda)에서만 다룬다. 즉 "context 객체"가 API 표면에서 보이느냐는 런타임마다 다르지만, 내부에 반드시 존재한다는 점이 Q8의 핵심.
  5. 실행 단위(work item)의 상태 기계는 런타임 불문 거의 동형이다: created → ready → running → (blocked/waiting) ⇄ ready → completed/failed/cancelled. 이름만 다르다(Linux TASK_RUNNING, goroutine _Grunnable/_Grunning, CUDA stream op의 pending/executing).

런타임 내부 구조로 환원

일반 런타임의 표준 객체 그래프(루트에서 잎으로):

Context / Instance (루트, "세계")
├── Device[] ................ 실행 자원의 추상 (CPU set / GPU / accelerator)
├── Allocator / Heap ........ memory·buffer·handle 생성/회수
├── Scheduler / RunQueue[] .. 어떤 work item을 언제/어디서 실행할지 결정
├── Executor / Worker[] ..... 실제 실행 주체 (thread / stream / worker)
├── WorkItem / Task[] ....... 실행 단위 객체 (상태 기계를 가진 노드)
├── Event / Future[] ........ 완료·의존성 신호
└── HandleTable ............. 위 객체들을 외부에 노출하는 불투명 핸들

5요소 매핑: object model = {Context, Device, WorkItem, Event, Buffer}, state machine = WorkItem의 상태 전이, scheduler = RunQueue 정책, allocator = Heap/Allocator, API/ABI = HandleTable을 통한 진입점. Q2에 대한 답은 "이 7개 모듈 + 초기화(init)/종료(teardown) 단계"다.

상태 전이 / 스케줄링 / 자원 소유 관점

  • 생애주기(라이프사이클): init(부트스트랩·device discovery) → context 생성 → resource acquire → work submit → schedule → execute → event/completion → teardown(회수). Q6("빠져나가는 산출물")은 이 teardown 단계에서 exit status + 남긴 side effect(파일/버퍼/로그) + event 최종 상태로 정의된다.
  • 자원 소유 분할(Q7·Q8의 선긋기):
  • 런타임 소유: Context, 논리 heap, RunQueue, worker pool, Event 객체, HandleTable.
  • OS kernel 위임: 물리 페이지, 커널 스레드, CPU time-slice, futex/wait_queue.
  • device driver 위임: device memory, 하드웨어 command queue, doorbell/MMIO, 인터럽트.
  • 즉 런타임은 대개 논리 자원의 소유자이자 물리 자원의 임차인(lessee) 이다. 이 이중성이 이후 모든 비교의 축이 된다.

  • CUDA stream을 "scheduler"로 부르면 부정확하다. stream은 순서(ordering) 채널이고, 실제 SM 배정 스케줄링은 하드웨어 GPU 스케줄러가 한다. 표의 "scheduler/queue" 칸은 이 두 층을 묶은 근사임을 명시.
  • Node.js의 병렬성 경계: JS 콜백은 단일 스레드 event loop에서 돌지만, fs/dns/crypto는 libuv threadpool(기본 4)로 오프로딩된다. "Node=완전 싱글스레드"는 과장 — worker_threads·threadpool 층을 다음 턴에서 분리 검증 필요.
  • JVM 실행 단위 표현 갱신: JDK 21(2023)의 가상 스레드(Project Loom) 는 M:N 스케줄되는 별도 work item이다. "JVM thread = OS thread 1:1"은 platform thread에만 맞다. Q3에서 재확인 대상.
  • EEVDF 전환 시점의 정확도(6.6 기본화)와, 배포판별 백포트 여부는 커널 문서로 교차확인 권장.

  1. OS runtime 심화(Q5·Q6·Q8): Linux task_struct의 상태값(TASK_RUNNING/TASK_INTERRUPTIBLE/TASK_UNINTERRUPTIBLE/__TASK_STOPPED/EXIT_ZOMBIE)과 syscall→block→wakeup 경로, exit_code/reaping을 위 표준 상태기계에 매핑하고, "런타임 소유 vs 커널 소유" 선을 handle table 수준에서 구체화.
  2. accelerator runtime의 queue 이중성 해소(Q3·Q7): CUDA stream/event 순서 의미론과 Level Zero의 command list↔command queue 분리를, "순서 채널"과 "스케줄 큐"라는 두 개념으로 나눠 정식화하고 completion/event가 상태기계에 거는 간선을 그린다.
  3. 언어 런타임 work-stealing 비교(Q4·Q9): Go(G-M-P, GRQ/LRQ, hand-off/steal)와 Tokio(멀티스레드 스틸러, Waker 기반 재적재)를 동일한 runqueue 모델로 환원해, 이후 Q10(PIM runtime scheduler) 설계의 후보 정책 목록을 만들 준비.

이번 턴의 확장 요약

이번 턴은 일반 런타임 모델을 Linux OS runtime에 맞춰 구체화한다. 핵심은 “Linux가 스케줄하는 최소 단위는 프로세스가 아니라 task_struct로 표현되는 task/thread”라는 점이다. 따라서 Q3의 process와 thread는 커널 내부에서 완전히 별개 객체라기보다, 주소 공간·파일 테이블·시그널 처리 구조를 얼마나 공유하느냐로 갈리는 task_struct들의 구성 방식이다.

또 하나 중요한 보정은 상태 모델이다. Linux의 TASK_RUNNING은 일반 상태기계의 ready와 running을 모두 덮는다. 실제로 CPU에서 실행 중인지는 task->__state만으로 알 수 없고, runqueue에 올라와 있는지와 현재 CPU의 rq->curr인지까지 봐야 한다.

새로 추가한 사실

  1. Linux 커널은 task->__state와 task->exit_state를 분리한다. 전자는 runnable/sleeping/stopped 같은 스케줄 가능성 상태, 후자는 EXIT_ZOMBIE, EXIT_DEAD 같은 종료 수명 상태다. 커널 소스 주석도 두 플래그 집합을 별도로 둔다고 설명한다. [Linux sched.h](https://raw.githubusercontent.com/torvalds/linux/master/include/linux/sched.h)
  1. TASK_RUNNING은 이름과 달리 “CPU에서 실행 중”만 뜻하지 않는다. runnable 상태로 runqueue에 들어갈 수 있는 task도 TASK_RUNNING이다. 그래서 일반 모델의 ready와 running은 Linux에서 다음처럼 쪼개야 한다.

text generic ready -> TASK_RUNNING + runqueue에 있음 generic running -> TASK_RUNNING + 특정 CPU의 current task

  1. blocking syscall은 대체로 TASK_INTERRUPTIBLE, TASK_UNINTERRUPTIBLE, TASK_KILLABLE류의 상태로 자신을 바꾼 뒤 schedule()로 CPU를 반납한다. 이후 조건 충족, I/O 완료, futex wake, signal, timeout 등이 try_to_wake_up() 계열 경로를 통해 다시 TASK_RUNNING으로 되돌린다. 커널 헤더의 wait-loop 예시는 set_current_state(...) -> 조건 검사 -> schedule() -> TASK_RUNNING 복귀 패턴을 보여준다. [Linux sched.h](https://raw.githubusercontent.com/torvalds/linux/master/include/linux/sched.h)
  1. EEVDF는 “공정한 CPU 분배”라는 CFS의 목표를 버린 것이 아니라, runnable task의 lag와 virtual deadline을 이용해 다음 task를 고르는 방식으로 바꾼 것이다. 커널 문서는 EEVDF가 동일 priority runnable task 사이의 CPU 시간을 균등하게 나누려 하며, deadline이 이른 eligible task를 고른다고 설명한다. Kernel EEVDF docs
  1. Linux의 종료 산출물은 단순한 return value가 아니다. exit_code, signal cause, resource usage, zombie 상태, parent의 wait*() 회수, 열린 파일·mm·signal 구조의 정리까지 포함한다. 즉 Q6에서 “프로그램이 런타임을 빠져나가며 남기는 것”은 exit status + waitable kernel record + side effects + resource release sequence로 봐야 한다.

관련 맥락 / 배경 설명

Linux는 OS kernel이지만, 런타임 관점에서는 매우 강한 하위 런타임(substrate runtime) 이다. JVM, Go, Tokio, Node.js 같은 상위 런타임은 자기 task/goroutine/future/callback을 직접 스케줄하지만, 결국 CPU time, blocking syscall, futex, epoll, thread creation, memory page는 Linux에 위임한다.

잠재적 오류 또는 확인 필요 지점

  • “Linux scheduler = EEVDF”라고 단정할 때는 범위를 좁혀야 한다. EEVDF는 주로 normal/fair scheduling class의 현재 방향이고, Linux 전체에는 RT, deadline, idle class도 있다. CFS 문서도 scheduling class 구조와 enqueue_task, dequeue_task, pick_next_task 같은 hook을 분리해 설명한다. CFS scheduler docs
  • 이전 턴의 “커널 6.6부터 EEVDF가 기본”이라는 표현은 큰 방향은 맞지만, 배포판 커널·백포트·config·문서 버전에 따라 “transition”, “making room”, “new option”처럼 표현이 엇갈릴 수 있다. 논문/글에서 사용할 때는 “mainline 6.6에서 fair scheduler가 EEVDF 방향으로 전환되기 시작했다” 정도가 더 안전하다.
  • TASK_INTERRUPTIBLE과 TASK_UNINTERRUPTIBLE을 단순히 waiting으로 합치면 신호 처리 차이를 잃는다. PIM runtime의 wait state를 설계할 때도 interruptible wait, non-interruptible device wait, timeout wait, cancel-requested wait를 나누는 것이 좋다.

다음 에이전트가 이어서 확장할 지점

  1. Go/Tokio로 이동해 goroutine과 Future가 Linux task_struct 위에서 어떻게 M:N으로 얹히는지 비교한다. 핵심 질문은 “상위 런타임 ready queue와 커널 runqueue가 어긋날 때 누가 보정하는가”다.
  1. Accelerator runtime으로 이동해 CUDA stream, Level Zero command queue, OpenCL/SYCL queue를 Linux의 runqueue와 비교한다. 특히 “순서 보장 queue”와 “실제 실행 자원 scheduler”를 분리해서 정리한다.
  1. Q10으로 이어질 준비로 PIM runtime의 최소 상태기계를 created -> submitted -> ready -> dispatched -> waiting_device -> completed/failed/cancelled로 초안화하고, 각 상태가 런타임 소유인지 driver/device 소유인지 표시한다.

이번 턴의 확장 요약

Turn 2가 남긴 핵심 미해결 질문 — "상위 런타임 ready queue와 커널 runqueue가 어긋날 때 누가 보정하는가" — 에 직접 답하는 것이 이번 턴의 축이다. 결론부터: 보정 주체는 커널이 아니라 상위 런타임 내부의 감시/드라이버 컴포넌트다. Go는 sysmon + netpoller가, Tokio는 mio 기반 non-blocking I/O + spawn_blocking이 그 역할을 한다.

그리고 Turn 1 비교표가 Go와 Tokio를 똑같이 "work-stealing"으로 묶은 것을 preemptive vs cooperative, stackful vs stackless 두 축으로 쪼갠다. 이 두 축이 이후 Q10 PIM 스케줄러가 "work item을 강제 선점할 수 있는가"를 결정하는 설계 분기점이 된다.

새로 추가한 사실

  1. goroutine = stackful coroutine, async task = stackless coroutine. Go 고루틴은 자체 성장/축소 가능 스택(초기 ~2KB, morestack/copystack으로 리사이즈)을 갖는 실행 단위다. 반면 Rust async fn/Tokio task는 컴파일러가 생성한 상태 기계(state machine)로, 정지 지점(.await)의 지역변수가 스택이 아니라 Future 구조체의 필드로 힙에 캡처된다. 즉 Q3의 "work item 표현"은 런타임마다 스택을 가진 노드냐 스택 없는 상태 기계 struct냐로 근본이 갈린다. 이 차이가 memory 모듈(Q7)과 선점 가능성(Q4)을 동시에 결정한다.
  1. Go는 선점형 M:N, Tokio는 협조형 M:N이다. Go 1.14(2020)부터 sysmon이 시그널(Unix에서 SIGURG)로 비동기 선점(async preemption)을 건다 — 함수 호출·할당이 전혀 없는 타이트 루프 고루틴도 ~10ms 후 강제로 멈출 수 있다(이전에는 안전점에서만 협조적 양보 가능). 반면 Tokio task는 선점 불가다. .await에서만 양보하므로, await 없이 긴 동기 계산을 도는 task는 worker 스레드 전체를 점유한다. 그래서 Tokio는 CPU-bound 작업을 spawn_blocking으로 분리하라고 강제한다.
  1. Go 스케줄러의 지역성/공정성 3중 구조. 각 P는 (a) 256칸 고정 배열 로컬 runq(LRQ), (b) 방금 깨어난 고루틴을 우선 실행하는 단일 runnext 슬롯을 갖고, (c) 전역 runq(GRQ)를 공유한다. 굶주림 방지를 위해 schedtick % 61 == 0일 때 GRQ를 먼저 확인한다(61은 패턴 회피용 소수). 로컬이 비면 다른 P의 runq 절반을 훔친다(work-stealing).
  1. Tokio도 동형 구조를 갖되 이름만 다르다. worker마다 256칸 로컬 큐 + 전역 injection queue + 방금 깨어난 task용 LIFO 슬롯(Go의 runnext 대응)을 둔다. LIFO 슬롯의 두 task가 서로 핑퐁하며 다른 task를 굶기는 것을 막기 위해 연속 LIFO 폴 횟수에 상한을 둔다. 전역 큐도 고정 간격(멀티스레드 스케줄러 기준 대략 31틱마다)으로 확인한다.
  1. Tokio의 자동 협조 양보(coop budget). 각 task는 폴 1회당 예산(기본 128)을 받고, 채널·I/O 같은 리소스가 준비돼 있어도 예산이 바닥나면 Poll::Pending을 반환해 강제로 양보시킨다. 선점이 없는 협조형 런타임이 굶주림을 막는 대체 장치다 — Go의 sysmon 시그널 선점과 목표는 같고 수단이 다르다.
  1. 종료 산출물(Q6)이 다르다. Tokio task는 반환값을 JoinHandle(완료 시 결과로 resolve되는 Future)로 회수할 수 있다 — 즉 일급(first-class) 결과 채널이 내장돼 있다. Go 고루틴은 반환값을 남기지 않는다(fire-and-forget). 결과를 받으려면 명시적 channel을 써야 한다. Turn 2의 Linux wait()가 zombie 레코드로 exit status를 보존하는 것과 대응시키면: JoinHandle ≈ waitable 완료 레코드, Go 고루틴 ≈ 완료 레코드 없음(채널이 대신).

런타임 내부 구조로 환원

Go/Tokio를 Turn 1의 표준 객체 그래프에 얹으면 정규 work-stealing 스케줄러가 드러난다:

Runtime (root)
├── Worker[]  ............ Go: P(논리 프로세서, GOMAXPROCS개) / Tokio: worker thread
│    ├── local runq(256) . 지역 ready queue

├── global queue ......... Go: GRQ / Tokio: injection queue

핵심: Worker(스케줄 컨텍스트)와 Executor(OS 스레드)를 분리하는 것이 M:N의 정의다. Go에서 P(Worker)와 M(Executor)이 별개 객체라는 점, Tokio에서 worker는 스레드에 고정이라는 점이 두 런타임의 미묘한 차이다 — Go는 P를 M에서 떼어내(handoff) 다른 M에 붙일 수 있어 syscall 블로킹에 더 유연하다.

  • _Gpreempted의 의미 주의. 이 상태는 "비동기 선점됨"과 곧바로 등치되지 않는다. 주로 GC 스택 스캔 등을 위해 고루틴을 suspendG로 정지시켰다가 재개하는 경로에서 쓰인다. "SIGURG=항상 _Gpreempted"로 단순화하면 부정확하다.
  • async 선점의 한계. 비동기 선점도 안전점(safe-point)이 없는 일부 어셈블리/no-preempt 구간에서는 걸리지 않는다. "Go는 무조건 어떤 고루틴도 선점 가능"은 과장이다.
  • Tokio 스케줄러 종류 구분. 위 수치·구조는 multi_thread 런타임 기준이다. current_thread(단일 스레드) 런타임은 work-stealing도 injection 밸런싱도 없다 — 비교 시 flavor를 명시해야 한다.
  • "stackless=메모리 절약"의 단서. async task가 힙 상태 기계라 해도, Future의 크기는 가장 큰 정지 지점 기준으로 정해져 큰 지역변수가 있으면 부풀 수 있다(대형 Future 문제). "stackless=항상 가볍다"는 검증 필요.

이번 턴의 확장 요약

이번 턴은 Go/Tokio의 스케줄 큐에서 벗어나, accelerator runtime의 명령 순서 큐로 돌아온다. 핵심 결론은 다음이다.

Accelerator runtime에서 stream, command queue, command list, event는 대부분 작업을 재배치하는 스케줄러 큐가 아니라, host가 device에 제출하는 순서·의존성·완료 추적 객체다. 실제 SM/CU/EU/accelerator core 배정은 CUDA runtime, Level Zero, OpenCL/SYCL runtime이 아니라 driver와 device firmware/hardware scheduler가 더 많이 소유한다.

즉 Q4의 “스케줄링”은 accelerator에서 두 층으로 갈라진다.

host runtime scheduling:
  어떤 command를 어떤 queue/stream에 넣고, 어떤 event 의존성을 붙일지 결정

device scheduling:
  제출된 command를 실제 compute unit, memory engine, DMA engine에 배정

새로 추가한 사실

CUDA stream은 같은 stream 안의 작업 순서를 보장하는 ordered submission channel이다. 서로 다른 stream 사이에서는 event, dependency, priority, resource contention에 따라 병렬 실행 가능성이 생기지만, runtime이 Go처럼 work item을 훔쳐 다른 stream으로 옮기는 모델은 아니다.

  1. CUDA stream은 Go/Tokio의 runqueue가 아니다.

CUDA Runtime API는 device별 primary context를 암묵적으로 사용하지만, Driver API는 CUcontext, CUstream, CUevent, CUmodule, CUfunction 같은 객체를 더 직접 노출한다. 따라서 Q8에서 “context가 있느냐”보다 중요한 질문은 context가 API 표면에 보이는가, 아니면 runtime 내부로 숨겨졌는가다.

  1. CUDA에는 runtime API와 driver API의 context 노출 차이가 있다.

Level Zero에서 command list는 명령을 기록하는 객체이고, command queue는 기록된 command list를 device에 제출하는 통로다. 이 구조는 Vulkan/Direct3D 스타일에 가깝다. CUDA stream처럼 “바로 stream에 kernel launch를 넣는” 모델보다, record → close → submit 단계가 더 명시적이다.

  1. Level Zero는 command list와 command queue를 분리한다.

in-order queue에서는 앞 명령 완료가 뒤 명령 실행 조건이 된다. out-of-order queue에서는 event wait list가 명령 간 의존성을 만든다. 이 점은 accelerator runtime의 핵심 상태 전이가 단순 FIFO가 아니라 event dependency graph일 수 있음을 보여준다.

  1. OpenCL command queue는 in-order와 out-of-order가 모두 가능하다.

SYCL queue, buffer, accessor, event는 CUDA, Level Zero, OpenCL 같은 backend 객체 위에 매핑된다. 따라서 SYCL의 task 상태를 분석할 때는 “SYCL runtime 소유 상태”와 “backend runtime/driver 소유 상태”를 나눠야 한다.

  1. SYCL queue는 자체 runtime이라기보다 backend runtime의 추상화 계층이다.

중요한 점은 submitted 이후의 상당 부분이 host runtime이 아니라 driver/device 소유 상태라는 것이다. Host runtime은 event polling, synchronization, error propagation을 통해 그 상태를 관찰한다.

관련 맥락 / 배경 설명

Accelerator runtime의 event는 단순 알림 객체가 아니다. 대개 세 가지 역할을 동시에 한다.

역할설명

completion recordhost가 완료 여부를 query/sync할 수 있는 상태
visibility boundarymemory copy, kernel write, host read 사이의 가시성 경계

이 때문에 accelerator runtime에서는 Q5의 상태 전이가 blocked → ready 하나로 끝나지 않는다. 더 정확히는 다음 간선들이 있다.

host task waiting_on_event
  -> event signaled by device
  -> host thread wake/sync returns
  -> output buffer becomes readable under runtime's memory rules

즉 “완료”는 세 층이다.

device execution completed
driver observed completion
host runtime/API returned completion to user

이 세 층이 어긋날 수 있기 때문에 CUDA의 asynchronous error, OpenCL/SYCL event status, Level Zero fence/event query 같은 API가 필요하다.

잠재적 오류 또는 확인 필요 지점

  • CUDA stream을 “GPU scheduler”라고 부르는 표현은 피해야 한다. stream은 host-visible ordering abstraction이고, 실제 SM 배정은 더 낮은 층이다.
  • Level Zero command queue도 완전한 hardware scheduler가 아니다. queue는 제출 인터페이스이고, device 내부 엔진 선택·실행 순서는 driver와 hardware 정책을 탄다.
  • OpenCL out-of-order queue는 “아무 순서나 실행”이 아니다. event wait list, memory dependency, command type 제약을 만족해야 한다.
  • SYCL의 queue 동작은 backend에 따라 달라진다. 같은 SYCL 코드라도 CUDA backend, Level Zero backend, OpenCL backend에서 event, memory, error propagation의 세부가 달라질 수 있다.
  • Accelerator command의 cancellation은 언어 runtime task cancellation보다 약하다. 이미 device에 dispatch된 kernel을 개별 task 단위로 취소하지 못하고 context reset이나 device fault 처리로 커지는 경우가 많다.

이번 턴의 확장 요약

Turn 4의 accelerator device API(CUDA/Level Zero/OpenCL) 에서 한 층 위, ML compiler runtime(PJRT/OpenXLA) 로 이동한다(확장 순서 5번). 이 턴의 축은 두 가지다.

  1. PJRT는 device API가 아니라 framework↔compiler ABI 경계다. 즉 Turn 1 비교표가 PJRT를 CUDA와 같은 열에 나란히 둔 것은 추상화 레벨을 뭉갠 근사다. PJRT는 CUDA와 peer가 아니라 CUDA 위에 얹히는 상위 계층(superstrate)이다.
  2. PJRT/XLA는 스케줄링의 큰 덩어리를 런타임에서 컴파일타임으로 옮긴다. op 순서, buffer 배치, fusion, collective overlap이 전부 XLA 컴파일 단계에서 결정되고, 런타임은 그 결과(thunk 시퀀스)를 재생(playback) 한다. 이 관찰이 Q4/Q9를 관통하는 새 축 — "스케줄링은 어디에서 일어나는가" — 을 만들고, 그 축의 극단에 PIM(Q10)을 놓는다.

새로 추가한 사실

실제 계층은 이렇게 쌓인다. text JAX / PyTorch-XLA / TF (framework: 프로그램=StableHLO/MLIR 소유) │ PJRT C API (안정 ABI 경계) PJRT plugin (예: XLA:GPU) (compiler+runtime 번들) │ XLA runtime (thunk sequence 실행) │ StreamExecutor (stream/event/memory 추상) │ CUDA / ROCm driver → GPU Turn 4의 CUDA/Level Zero는 위 그림에서 아래 두 층에 해당한다. PJRT는 그 위에서 backend를 갈아끼우는(plugin) 계층이다.

  1. PJRT는 device API가 아니라 "framework↔compiler ABI 경계"다.

PJRT_Api 구조체(major/minor 버전 필드 포함)가 모든 진입점을 담고, 플러그인 .so는 GetPjrtApi() 엔트리 하나만 노출한다. framework는 이 심볼을 dlopen으로 얻는다. 즉 Turn 1이 언급만 하고 넘어간 "동적 로딩/링킹 모듈"과 "opaque handle" 이 여기서 ABI 안정성의 실체가 된다 — C 함수 포인터 테이블이라야 컴파일러·프레임워크·드라이버를 서로 다른 팀이 서로 다른 속도로 빌드해도 링크가 유지된다.

  1. PJRT의 API/ABI 표면은 "버전드 함수 포인터 테이블"이다.

PJRT_LoadedExecutable_Execute는 output PJRT_Buffer 핸들을 동기적으로 즉시 돌려주지만, 그 버퍼의 내용은 비동기로 유효해진다. 각 버퍼는 내부 definition event(≈ PJRT_Buffer_ReadyEvent)를 갖고, 실행 완료를 알리는 device_complete_events가 별도로 나온다. 정리하면 Q6의 산출물은: text 즉시: output buffer handle (아직 못 읽음) 비동기: definition event 시그널 → buffer contents 유효 비동기: device_complete_event → 실행 종료 비동기: async error (실행 중 오류는 event/consume 시점에 표출) Turn 4의 "완료는 세 층(device 완료 / driver 관측 / host 반환)"이 PJRT에서는 핸들 반환과 내용 유효를 시간축에서 분리하는 방식으로 구체화된다.

  1. Execute의 산출물이 이중 시점을 갖는다(Q6 정밀화).

실행 시 input buffer를 "donate"로 표시하면(input_output_aliases) 런타임은 그 메모리를 output으로 재사용하고, donate된 input 핸들은 무효화된다. 즉 buffer는 단순 메모리가 아니라 소유권과 생명주기를 가진 런타임 객체이며, 실행이 그 소유권을 소비할 수 있다. 이건 Rust move 시맨틱과 동형이고, PIM 설계에서 "in-place 연산 시 입력 버퍼를 소비"로 그대로 이어진다.

  1. Buffer donation은 allocator 차원의 소유권 이전이다(Q7·Q8).

XLA는 컴파일 시 (a) HLO scheduling(피크 메모리 최소화 순서), (b) buffer assignment(liveness 기반 재사용), (c) fusion(op 병합), (d) collective/latency hiding(통신-계산 오버랩)을 결정하고, GPU backend는 이를 thunk 시퀀스(kernel launch·memcpy·collective의 순서화된 명령 목록)로 낮춘다. 런타임은 이 시퀀스를 스트림에 재생할 뿐이다. 즉 PJRT의 "work item"은 개별 커널이 아니라 executable 실행 1회이고, 그 내부의 세밀한 스케줄은 이미 컴파일러가 소유·확정했다.

  1. XLA는 스케줄링을 컴파일타임으로 옮긴다(Q4의 축 이동).

CUDA는 CUstream을 사용자에게 노출하지만, PJRT는 compute/h2d/d2h/collective 스트림을 플러그인 내부에 은닉한다. 사용자는 executable을 제출하고 event를 받을 뿐이다. Turn 4의 "순서 채널(stream)"이 PJRT 레벨에서는 사라지고 buffer/event 의존성 그래프로만 표현된다 — 순서화 객체를 어느 층까지 노출하느냐가 런타임마다 다르다는 Q8 명제의 또 다른 사례.

  1. PJRT는 stream을 감춘다.

버퍼는 그냥 device가 아니라 memory space(device HBM / pinned host / unpinned host 등)에 거주한다. 이는 Turn 4에서 buffer를 "placement + coherence + owner"로 봐야 한다고 한 예고를 PJRT가 실제 타입으로 구현한 것 — PIM의 bank/subarray placement로 직결된다.

  1. PJRT_Memory(메모리 공간)가 별도 일급 객체다.

위 축에서 PIM은 극우단이다. PIM 연산은 (i) 매우 미세립(원소 단위 add/mul/mac), (ii) 데이터 배치에 종속(연산은 피연산자가 상주한 bank/subarray에서만 가능), (iii) 제어흐름/데이터 종속 분기가 빈약. 따라서 개별 op를 런타임에서 디스패치하면 오버헤드가 폭발하고, XLA처럼 컴파일타임에 명령 시퀀스로 낮추는 것이 사실상 필수다.

잠재적 오류 또는 확인 필요 지점

  • "PJRT는 스케줄러가 없다"는 과장이다. 정확히는 세밀한(op 단위) 스케줄이 컴파일타임으로 옮겨졌다는 것이고, executable 간 디스패치·multi-device SPMD 실행·collective 조율·async prefetch는 여전히 런타임이 한다. 표의 "런타임 스케줄=얇음"은 op 단위 기준의 근사임을 명시해야 한다.
  • BFC 할당자·thunk·스트림 개수는 XLA:GPU backend 내부 구현이며 PJRT 규약이 아니다. 다른 플러그인(TPU, CPU, Apple 등)은 다른 할당자·실행기를 쓴다. "PJRT=BFC/thunk"로 등치하면 부정확 — 소스/버전 교차확인 필요.
  • UPMEM은 "고정기능 PIM"이 아니다. UPMEM DPU는 범용 소형 프로세서라 PimKernel이 바이너리다. Samsung HBM-PIM/SK hynix AiM류(그리고 LPDDR6-PIM 계열)만 고정기능 opcode 모델이다. Q10에서 두 스타일을 뭉개면 object model이 틀어진다.
  • mode_wait/refresh_wait를 상태로 노출할지 여부는 설계 선택이다. HW가 이를 완전히 감추면(펌웨어가 refresh를 삽입) 런타임은 device_pending 하나로 관측할 수도 있다. "런타임이 refresh를 스케줄한다"고 단정하면 과설계 위험 — 관측 가능성부터 확인해야 한다.
  • PJRT의 async error는 "언제 표출되는가"가 미묘하다. 실행 오류가 complete event에 실릴 수도, buffer를 host로 가져올 때(ToHostBuffer) 표출될 수도 있다. Q6에서 "오류=즉시 반환"으로 단순화하면 안 된다.

다음 에이전트가 이어서 확장할 지점

  1. JVM/GC를 "cleanup·teardown 모듈"로 정식화하라(Q6·Q7). 지금까지 allocator를 Go(tracing GC)·Tokio(refcount)·PJRT(BFC+donation)·PIM(placement) 축으로만 봤다. JVM의 세대별 GC, safepoint, finalizer/Cleaner, JNI 전역참조 회수를 "프로그램이 빠져나갈 때 런타임이 회수하는 상태" 로 환원하고, PJRT의 buffer teardown·PIM의 bank 회수와 한 표에 놓아라. 특히 GC의 safepoint가 Turn 3의 Go 비동기 선점(safe-point)과 같은 메커니즘임을 연결하라.
  1. multi-device / collective를 자원 소유 축으로 열어라(Q7·Q9). PJRT의 SPMD 실행(sharded buffer, device mesh, collective)과 CUDA의 NCCL/멀티스트림을 "여러 실행 자원에 걸친 하나의 work item"으로 모델링하고, 이를 PIM의 여러 bank/rank에 분산된 buffer + bank 간 데이터 이동(=collective의 PIM판) 으로 사상하라. 핵심 질문: PIM에서 bank 간 이동은 누가 소유하는가(host copy vs on-die interconnect).
  1. "스케줄 위치 축"을 Q10 최종 설계 원칙으로 굳혀라(Q4·Q10). 이번 턴이 만든 compile-time ↔ host runtime ↔ device HW 축 위에서, PIM의 어떤 결정(placement, op 순서, mode 전환 타이밍, refresh 회피, in-place donation)을 각각 어느 위치에 배치할지 결정표를 만들어라. 그리고 이 배치가 PIM API 표면(무엇을 노출하고 무엇을 감출지 — PJRT가 stream을 감춘 것처럼)을 어떻게 결정하는지 도출하라.

이번 턴의 확장 요약

이번 턴은 PJRT/PIM의 allocator·buffer lifecycle 논의를 JVM의 GC와 teardown 모듈로 되돌린다. 핵심은 “JVM의 allocator는 단순 heap allocator가 아니라, object graph의 상태기계와 program teardown을 함께 관리하는 런타임 모듈”이라는 점이다.

JVM에서는 실행 단위와 회수 단위가 분리된다.

실행 단위:
  platform thread / virtual thread / ForkJoin task / Executor task

회수 단위:
  Java object / class metadata / native memory wrapper / thread stack / JIT compiled code

Go나 Tokio가 task scheduling을 중심으로 보였다면, JVM은 heap object lifecycle + safepoint/handshake + GC barrier + thread scheduling이 함께 런타임 내부 구조를 이룬다.

새로 추가한 사실

OpenJDK JEP 444는 virtual thread를 java.lang.Thread의 경량 구현으로 정의하고, OS thread에 고정되지 않는 M:N 구조라고 설명한다. Platform thread는 OS thread wrapper이고, virtual thread는 carrier platform thread 위에 mount/unmount된다. JEP 444

  1. Java 21에서 virtual thread는 정식 기능이다.

JDK scheduler는 virtual thread를 platform thread에 배정하고, platform thread는 다시 OS scheduler가 CPU에 배정한다. JEP 444 기준 virtual thread scheduler는 FIFO 모드의 work-stealing ForkJoinPool이며, 기본 parallelism은 available processors 수에 맞춰진다. JEP 444

  1. JVM virtual thread scheduler는 CPU core에 직접 스케줄하지 않는다.

둘 다 M:N 사용자 모드 thread처럼 보이지만, JDK virtual thread scheduler는 현재 time-sharing preemption을 구현하지 않는다고 명시한다. CPU-bound virtual thread가 오래 계산하면 carrier를 오래 점유할 수 있다. 반면 Go는 비동기 선점 경로가 있다. 따라서 JVM virtual thread는 “Tokio처럼 완전 협조형”도 아니고 “Go처럼 강한 선점형”도 아닌 중간 지점이다.

  1. Virtual thread는 Go goroutine과 비슷하지만 동일하지 않다.

일반적인 JDK blocking I/O에서는 virtual thread가 unmount되어 carrier가 다른 work를 실행할 수 있다. 하지만 synchronized 블록/메서드 안이거나 native/foreign call 중이면 pinning 때문에 carrier까지 막힐 수 있다. 이 구분은 Q5 상태기계에서 waiting_unmounted와 waiting_pinned를 분리해야 함을 뜻한다.

  1. Virtual thread blocking은 waiting 상태를 carrier 점유 여부로 나눈다.

Java heap object는 GC가 회수하지만, file descriptor, socket, native memory, GPU/PIM buffer handle 같은 외부 자원은 보통 try-with-resources, AutoCloseable.close(), Cleaner, JNI/Foreign Memory API의 scope 관리가 필요하다. OpenJDK JEP 421은 finalization을 removal 대상으로 deprecate했고, timely release에는 finalizer보다 try-with-resources가 맞다고 설명한다. JEP 421

  1. JVM cleanup은 GC만으로 끝나지 않는다.

JEP 439는 ZGC가 young/old generation을 나누고, colored pointer, load barrier, store barrier로 concurrent GC 중에도 application thread가 일관된 object graph를 보게 한다고 설명한다. 즉 객체의 상태는 별도 side table만이 아니라 pointer bit와 JIT-injected barrier code에도 분산된다. JEP 439

  1. Generational ZGC는 “객체 상태”를 pointer metadata와 barrier로 런타임화한다.

잠재적 오류 또는 확인 필요 지점

  • Virtual thread를 “Go goroutine과 동일”하다고 쓰면 안 된다. 둘 다 M:N이지만, Go는 runtime-level async preemption이 있고, JDK virtual thread scheduler는 현재 time-sharing을 구현하지 않는다고 문서화되어 있다.
  • JVM GC를 하나로 일반화하면 위험하다. G1, ZGC, Shenandoah, Parallel GC는 pause, compaction, barrier, region metadata가 다르다. PIM 설계에 가져올 때는 “tracing GC 전체”가 아니라 “barrier로 visibility invariant를 유지한다”는 구조만 차용해야 한다.
  • Cleaner는 deterministic cleanup이 아니다. JEP 421도 Cleaner가 GC에 의해 scheduling되므로 지연이 unbounded일 수 있다고 설명한다. PIM buffer나 device queue handle처럼 scarce resource에는 explicit close/delete API가 필요하다.
  • Safepoint는 정지 세계만 뜻하지 않는다. 최신 HotSpot은 thread-local handshake도 사용한다. 다만 이 턴에서는 “runtime이 thread stack과 heap invariant를 관찰하기 위해 만든 협조 지점”이라는 추상 모델만 잡는다.
  • JVM shutdown hook은 실패/강제종료에서 보장되지 않는다. Runtime.halt, fatal VM error, SIGKILL, container kill 같은 경로에서는 정상 teardown 상태기계가 깨질 수 있다. Q6의 종료 산출물은 normal exit과 abnormal exit을 나눠야 한다.

  1. 멀티디바이스 실행은 두 패턴으로 갈린다. ① fan-out(독립 팬아웃): N개 자원에 독립 work item을 뿌리고 N개 완료가 서로 무관 — 데이터 병렬 forward pass, SPMD replica. ② collective(결합 집단연산): N개 참여자가 반드시 만나야(rendezvous) 하나의 결과가 확정 — all-reduce, all-gather. 이 둘은 상태기계·완료·실패 모델이 근본적으로 다르며, 지금까지의 단일 상태기계로는 ②를 표현할 수 없다.
  1. collective는 새 런타임 객체 하나(communicator/참여자 그룹)와 새 상태 하나(rendezvous/barrier-wait), 새 소유 패턴 하나(공동 소유), 새 실패 모드 하나(분산 교착·straggler)를 강제로 도입한다. 이 네 가지는 CPU/GPU/JVM 단일 런타임에 없던 축이고, Q10 PIM의 "뱅크 간 리덕션을 누가 소유하는가"로 직결된다.

새로 추가한 사실

Turn 6은 "synchronized 블록/메서드 안이면 pinning 때문에 carrier까지 막힐 수 있다"고 썼지만, JEP 491이 JDK 24에 delivered되면서 monitor를 carrier가 아니라 virtual thread 자신에 귀속시켰다. 이제 synchronized 안에서 I/O나 Thread.sleep()으로 블록해도 virtual thread는 unmount되고 carrier가 풀린다. pinning이 남는 경우는 native/foreign 호출(JNI, FFM API) 뿐이다 — native code가 OS thread의 TLS를 만지거나 블로킹 OS API를 부를 수 있어 JVM이 mid-execution에 unmount할 수 없기 때문. 따라서 Turn 6의 상태기계는 이렇게 좁혀야 한다: text waiting_unmounted ← blocking I/O, sleep, park, 그리고 (JDK24+) synchronized 대기 waiting_pinned ← native/JNI/FFM 호출 중 블록 (+ 소수의 VM-internal 구간) ← 범위 대폭 축소 즉 "synchronized = pinning"은 JDK 21~23 한정 사실이고, 오늘 기준 mainline에서는 틀린다. communicator는 "이 collective에 참여하는 rank들의 집합 + 그들을 잇는 채널(ring/tree) 연결"을 소유한다. ncclCommInitRank/ncclCommInitAll 시점에 링·트리 채널을 미리 구축(연결 수립 = 자원 획득 단계)하고, 이후 collective는 이 연결 위를 흐른다. 즉 Turn 1~5의 객체 그래프에 없던 "토폴로지 객체(communicator)" 가 collective 런타임의 루트다. NCCL communicators

  1. NCCL의 루트 객체는 device가 아니라 ncclComm(communicator)다 — 토폴로지를 소유하는 참여자 그룹.

NCCL은 메시지 크기·토폴로지에 따라 ring(대역폭 최적, "bucket brigade") 과 tree(지연 최적) 를 선택한다. 이는 Turn 5의 "스케줄 위치 축"에서 런타임 스케줄에 해당 — CUDA 단일 스트림은 스케줄이 얇았지만, NCCL은 collective 알고리즘 선택이라는 실질적 런타임 스케줄 결정을 다시 짊어진다. 반면 XLA:GPU는 collective를 컴파일타임에 executable 안으로 낮춘다(뒤 참조). 같은 all-reduce가 런타임에 있느냐 컴파일타임에 있느냐로 갈린다. NCCL algorithms

  1. collective 알고리즘(ring vs tree)은 런타임이 크기·토폴로지 기준으로 고른다.

한 rank가 collective를 launch하지 않거나 죽으면, 나머지 rank는 rendezvous에서 무한 대기(hang) 한다. 회복 절차는 개별 취소가 아니라 ncclCommAbort로 communicator 전체를 중단 → 생존 rank로 ncclCommInit 재구성이다. 안전한 abort를 위해 NCCL은 communicator를 nonblocking 모드로 두라고 요구한다(그러면 abort/destroy를 제외한 모든 호출이 nonblocking이 됨). 즉 collective의 취소 단위는 work item이 아니라 그룹 전체다 — Turn 4에서 "accelerator cancellation은 context reset 단위로 커진다"고 한 관찰이 멀티디바이스에서 communicator reset 단위로 한 층 더 커진다. NCCL fault tolerance

  1. collective의 실패 모드는 단일 task에 없던 "분산 교착"이다.

여러 collective(또는 여러 communicator의 P2P)를 group 안에 묶으면 NCCL이 이를 한 번에 스케줄·집계해 rank 간 launch 순서 어긋남으로 인한 교착을 피한다. 이는 Turn 4의 "command list recording"과 동형 — "기록 후 일괄 제출" 패턴이 멀티디바이스 동기화의 교착 회피 수단으로 재등장한다.

  1. ncclGroupStart/End는 여러 collective를 하나의 원자적 제출로 융합한다.

PJRT LoadedExecutable은 num_replicas·num_partitions로 컴파일되고, GSPMD/Shardy partitioner가 all-reduce/all-gather를 HLO 그래프에 직접 삽입한다. 사용자는 sharded buffer를 넣고 실행할 뿐, collective 호출을 직접 하지 않는다. 다중 호스트는 별도 coordination service(DistributedRuntimeClient류) 로 rank를 맞춘다. 즉 NCCL이 communicator를 API 표면에 노출한다면, PJRT는 그것을 컴파일타임 + 내부 런타임으로 은닉한다 — Turn 5의 "PJRT는 stream을 감춘다"가 여기선 "PJRT는 communicator를 감춘다"로 확장된다.

  1. PJRT는 collective를 executable 안에 숨긴다(compile-time collective).

관련 맥락 / 배경 설명 — Q10 PIM collective의 소유 위치

Turn 5·6이 남긴 질문 "PIM에서 뱅크 간 이동은 누가 소유하는가(host copy vs on-die interconnect)"에 답한다. 결론: 대부분의 PIM에는 뱅크 간 all-to-all on-die interconnect가 없다. GPU의 NVLink에 해당하는 것이 PIM에는 부재하므로, 뱅크/랭크 간 리덕션은 세 경로로만 가능하다:

(a) within-bank/subarray  : PIM 유닛이 로컬 셀에서 직접 리덕션 (진짜 near-data)
(b) cross-bank (same channel): 부분합을 메모리 컨트롤러가 읽어 재배치 → 다시 PIM  [MC 소유]
(c) cross-rank/channel    : host가 gather → host/DPU에서 합산 → scatter          [host 소유]

즉 PIM의 "collective"는 GPU식 대칭 링/트리가 아니라, 비대칭 gather-reduce-scatter이고, rendezvous 진행의 소유자가 memory controller(같은 채널) 또는 host(랭크 간) 로 명시적으로 넘어간다. GPU collective가 "소유자 없는 분산 진행"이었다면, PIM collective는 "소유자가 MC/host로 고정된 준(準)-collective" 다. 이 차이가 Q10 설계에 주는 함의:

  • PIM communicator 객체(PimGroup)는 채널/뱅크그룹 토폴로지 + 리덕션 트리를 컴파일타임에 확정한다(런타임 알고리즘 선택 없음 — NCCL과 반대, XLA와 같음).
  • 새 대기 상태 cross_bank_shuffle_wait(부분합을 MC가 재배치하는 동안)를 Turn 5 상태기계에 추가한다.
  • 취소는 PimGroup/채널 reset 단위이고, straggler 개념은 약하다(참여자가 하드웨어 뱅크라 임의 실종이 드물다) — 대신 refresh/mode 경합이 straggler 역할을 대신할 수 있다.

이로써 Turn 5의 원칙 "스케줄은 컴파일러에, 배치는 할당자에"에 "뱅크 간 리덕션 진행은 memory controller에" 가 추가된다.

잠재적 오류 또는 확인 필요 지점

  • "NCCL communicator = process group"으로 등치하면 부정확. PyTorch ProcessGroup은 NCCL 외에 Gloo·MPI 백엔드도 감싸는 상위 추상이다. communicator는 NCCL 고유 객체이고, 다중 communicator를 한 process group이 관리할 수 있다. 비교 시 계층을 구분해야 한다.
  • fan-out과 collective는 실전에서 섞인다. 데이터 병렬 학습 1 step은 fan-out(forward/backward) + collective(gradient all-reduce)의 합성이다. "학습=collective"로 단순화하면 완료·실패 모델이 뭉개진다. work item을 두 종류로 분해해 표현해야 한다.
  • rendezvous_wait의 관측 가능성. NCCL hang은 런타임이 즉시 감지하지 못하고 watchdog 타임아웃(예: PyTorch NCCL_TIMEOUT)에 의존한다. "런타임이 straggler를 스케줄링한다"고 쓰면 과설계 — 실제로는 타임아웃 후 abort라는 얇은 회복만 있다. Turn 5의 refresh 과설계 경고와 같은 함정.
  • PIM에 on-die interconnect가 전혀 없다는 단정은 위험. 일부 연구용 PIM/PNM은 뱅크그룹 내 부분 상호연결이나 near-memory reduction tree를 제안한다. "PIM은 무조건 host gather"는 상용 DRAM-PIM 기준 근사이며, 아키텍처별로 (b) 경로의 하드웨어 지원 여부를 확인해야 한다.
  • JEP 491의 "거의 모든" 단서. synchronized pinning이 사라졌다고 pinning이 0이 된 것은 아니다(native/FFM 잔존). "JDK 24부터 virtual thread는 절대 pin 안 됨"은 과장.

  1. 자원 획득·회수의 대칭성(symmetric alloc/free)을 allocator 축에 추가하라(Q7·Q8). 지금까지 allocator는 단일 힙(GC/refcount/BFC/placement) 중심이었다. collective는 전 rank 대칭 버퍼 + 등록(pinned/registered) 메모리 + 채널 버퍼를 요구하며, 하나라도 어긋나면 그룹이 진행 못 한다. 이 "분산 대칭 할당과 그 회수(누가 언제 안전하게 free하는가)" 를 PJRT sharded buffer teardown·NCCL buffer registration·PIM 뱅크 대칭 배치와 한 표에 놓고, Turn 3의 refcount·Turn 6의 Cleaner가 분산 환경에서 왜 부족한지(한 rank의 조기 free가 전체를 깨뜨림)를 도출하라.

이번 턴의 확장 요약

이번 턴은 언어 런타임 쪽으로 돌아와 Node.js/libuv 이벤트 루프를 정식화한다. 핵심은 Node.js가 Go/Tokio식 work-stealing 런타임도, CUDA식 command queue 런타임도 아니라는 점이다. Node.js의 기본 스케줄러는 phase 기반 이벤트 루프 + 우선순위 microtask 계층 + 제한된 threadpool offload다.

즉 Node.js에서 Q4의 “스케줄링”은 다음 질문으로 바뀐다.

ready task 중 어떤 worker가 훔쳐 갈 것인가?        # Go/Tokio
device queue에 어떤 command를 넣을 것인가?        # CUDA/PJRT
현재 event loop turn에서 어떤 phase queue를 먼저 비울 것인가?  # Node.js

Node.js는 기본적으로 하나의 JavaScript 실행 스레드가 callback을 run-to-completion으로 실행한다. 그러나 “Node.js = 싱글스레드”라고 쓰면 부정확하다. 네트워크 I/O는 커널 poller가 맡고, 파일 I/O·일부 DNS·사용자 uv_queue_work()는 libuv threadpool이 처리하며, CPU-bound JavaScript는 worker_threads로 병렬 실행할 수 있다. Node 공식 문서도 기본 JS thread와 kernel offload를 함께 설명하고, libuv 문서는 file I/O와 DNS 등이 threadpool을 사용한다고 정리한다. Node event loop, libuv design, libuv threadpool

새로 추가한 사실

Node 런타임 안의 work item은 setTimeout callback, I/O callback, setImmediate callback, close callback, Promise reaction job, process.nextTick callback, libuv uv_req_t, uv_work_t, Worker message 같은 여러 형태로 나뉜다. 즉 Q3의 task는 Node에서 단일 타입이 아니라 phase queue에 들어가는 callback 계열 객체 + libuv request/handle 객체로 표현된다.

  1. Node.js의 스케줄러 단위는 “thread”가 아니라 callback/job/request다.

주요 phase는 timers, pending callbacks, poll, check, close callbacks로 잡을 수 있다. 각 phase는 자기 callback queue를 FIFO로 실행하고, queue가 비거나 구현상 한도에 도달하면 다음 phase로 넘어간다. poll phase는 새 I/O 이벤트를 가져오고, check phase는 setImmediate() callback을 실행한다. Node event loop

  1. Node.js 이벤트 루프는 phase별 FIFO queue를 돈다.

Node 문서는 libuv 1.45.0, 즉 Node.js 20부터 timer가 각 loop iteration에서 poll phase 뒤에 실행된다고 설명한다. 이전에는 polling 전에도 timer가 실행됐다. 따라서 setTimeout(0)과 setImmediate()의 상대 순서를 설명할 때 “항상 예전 그림 그대로”라고 쓰면 버전 의존 오류가 생긴다. Node event loop

  1. Node 20/libuv 1.45 이후 timer 위치가 바뀌었다.

Node 문서는 nextTickQueue가 현재 JS/C++ 경계 작업이 끝난 뒤 event loop가 다음 phase로 진행하기 전에 처리된다고 설명한다. 또한 재귀적 process.nextTick()은 poll phase 도달을 막아 I/O starvation을 만들 수 있다. Node event loop

  1. process.nextTick은 event loop phase가 아니다.

queueMicrotask()와 Promise job은 V8 microtask queue 쪽이고, Node 문서는 각 event-loop turn에서 process.nextTick() queue가 microtask queue보다 먼저 처리된다고 설명한다. 즉 Node의 우선순위는 대략 현재 callback 완료 → nextTick → V8 microtasks → event loop phase 진행으로 모델링할 수 있다. Node globals

  1. Node에는 Node 소유 nextTick queue와 V8 소유 microtask queue가 따로 있다.

libuv threadpool은 기본 크기 4이고, 시작 시 UV_THREADPOOL_SIZE로 조정할 수 있으며 절대 최대값은 1024다. file system operation, getaddrinfo/getnameinfo, uv_queue_work()가 이 pool을 쓴다. threadpool은 모든 event loop가 공유한다. libuv threadpool

  1. libuv threadpool은 전역 공유 자원이다.

Worker thread는 JavaScript를 병렬 실행하는 별도 V8 isolate/JS 실행 컨텍스트에 가깝다. Node 문서는 Worker가 CPU-intensive JavaScript에 유용하고, built-in async I/O에는 보통 도움이 적다고 설명한다. Worker는 ArrayBuffer transfer나 SharedArrayBuffer 공유로 메모리 경계를 넘는다. Node worker_threads

  1. worker_threads는 libuv threadpool과 다른 층이다.

관련 맥락 / 배경 설명

Node.js는 PIM runtime 설계에 두 가지 교훈을 준다.

첫째, control plane과 execution plane을 분리한다. Node에서 JS event loop는 control plane이고, kernel poller·threadpool·Worker는 execution/offload plane이다. PIM에서도 host runtime은 control plane이고, 실제 bank/subarray execution은 memory controller/PIM device 쪽이다.

Node:
  JS event loop
    → kernel poller / libuv threadpool / Worker

PIM:
  PimEventLoop or host dispatcher
    → memory controller / PIM queue / bank execution

둘째, completion을 단일 이벤트로 뭉치면 안 된다. Node에서도 완료는 threadpool work done, after_work callback queued, JS callback executed, Promise resolved로 층이 갈린다. PIM도 마찬가지로 device execution done, writeback done, host-visible, callback/event delivered를 나눠야 한다.

PIM runtime에 Node식 dispatcher를 붙이면 최소 구조는 다음처럼 된다.

PimHostRuntime
├── submission queue
├── completion poll phase
├── visibility/writeback phase
├── user callback / future resolution phase
├── cleanup phase
└── optional worker pool for blocking host-side copy/packing

즉 PIM의 host scheduler가 반드시 복잡한 work-stealing일 필요는 없다. 오히려 PIM command가 비선점·in-order device queue에 가깝다면, host 쪽은 Node처럼 event completion을 phase별로 처리하는 얇은 dispatcher가 실용적일 수 있다.

잠재적 오류 또는 확인 필요 지점

  • “Node.js는 싱글스레드”라고 단정하면 안 된다. 기본 JavaScript 실행이 단일 thread일 뿐, libuv threadpool, kernel async I/O, Worker thread가 별도로 존재한다.
  • setTimeout(0)과 setImmediate()의 순서는 문맥과 Node/libuv 버전에 의존한다. 특히 Node.js 20/libuv 1.45 이후 timer phase 설명은 예전 자료와 다를 수 있다.
  • process.nextTick을 microtask와 같은 것으로 뭉개면 우선순위가 틀린다. Node 문서 기준 nextTick queue는 V8 microtask queue보다 먼저 처리된다.
  • libuv threadpool은 “모든 async I/O”를 처리하지 않는다. 네트워크 I/O는 일반적으로 non-blocking fd와 poller가 맡고, file I/O·일부 DNS·사용자 작업이 threadpool에 들어간다.
  • uv_cancel()은 이미 실행 중인 threadpool work를 강제로 중단하는 일반 선점 취소가 아니다. 취소 가능성은 request 종류와 실행 시점에 의존한다. PIM command cancellation도 이와 비슷하게 “queued 상태 취소”와 “device 실행 중 abort”를 분리해야 한다.

이번 턴의 확장 요약

Turn 3~8은 전부 정상상태(steady-state) 를 다뤘다 — 1 task↔1 thread, 1 kernel↔1 stream, buffer readiness, object reachability, phase queue. 그런데 그 모든 스케줄러·할당자·GC·이벤트 루프는 누군가가 먼저 켜야 존재한다. 이번 턴의 축은 지금까지 한 번도 열지 않은 런타임 자신의 생명주기 — 부팅과 종료 — 다.

핵심 결론 세 가지:

  1. 런타임은 "프로그램을 실행하는 것"이 아니라, 자기 자신을 먼저 부팅해서 존재하게 만드는 자기호스팅 시스템이다. 아직 자기 스케줄러도 할당자도 없는 상태에서, OS가 준 맨몸 스택·동적 링커·단일 OS 스레드 위에 관리되지 않는(unmanaged) 실행 단위 하나를 두고, 그 위에서 서브시스템을 고정된 의존 순서로 하나씩 켠 뒤, 마지막에 사용자 진입점을 첫 번째 관리 대상 work item으로 주입한다. 런타임 자신의 생명주기가 모든 task의 생명주기를 괄호로 감싼다.
  1. 부팅은 Q4의 스케줄 문제다. CUDA lazy module loading, JVM lazy <clinit>처럼 런타임은 "자기 초기화 비용을 언제 지불할지"를 스케줄한다 — Turn 5의 compile↔runtime 축이 사용자 op가 아니라 런타임 자신의 부팅에 적용된 사례.
  1. 종료는 회수가 아니다. 대부분의 런타임은 clean teardown을 의도적으로 건너뛰고(os.Exit, Runtime.halt, _exit) OS 프로세스 회수에 위임한다. teardown이 남기는 진짜 산출물은 메모리 회수가 아니라 flush/notify/persist다.

새로 추가한 사실

진입점 _rt0_amd64 → runtime.rt0_go → runtime.schedinit(). 이때 g0(시스템 스택 고루틴)와 m0(초기 OS 스레드)는 힙도 스케줄러도 없는 시점에 정적으로 존재한다. schedinit은 대략 mallocinit(할당자) → mcommoninit(m0 등록) → gcinit(GC) → procresize(GOMAXPROCS만큼 P 배열 생성) 순으로 서브시스템을 켠 뒤, runtime.main을 돌릴 main goroutine을 만들어 스케줄러에 넘긴다. runtime.main이 먼저 패키지 init()들을 순서대로 실행하고 그 다음 main.main을 호출한다. 결정적 대비: g0은 Turn 3의 성장형 stackful 고루틴 스택이 아니라 OS가 준 고정 스택을 쓴다. 스케줄러 코드 자체가 이 g0 위에서 돈다 — 즉 "스케줄러는 스케줄되지 않는다". Q3의 "실행 단위"에는 항상 부팅용 0번째 특권 실행 단위가 먼저 있다.

  1. Go 런타임은 "관리되지 않는 실행 단위"(g0/m0)로 시작해 그 위에 관리 체계를 세운다.

java 런처가 JNI_CreateJavaVM으로 VM을 세우고, 3단계 classloader 계층(bootstrap → platform → application)이 핵심 클래스를 로드하며 main 스레드가 생성된다. 클래스는 loaded → linked(verify/prepare/resolve) → initialized(<clinit>) 상태기계를 거치고, <clinit>은 lazy(JLS가 정의한 active use — new, static 필드/메서드 접근 등 — 시점에 트리거). Turn 6이 다룬 heap object reachability는 이 부팅 상태기계가 끝난 뒤에야 의미를 갖는다.

  1. JVM에는 Turn 6의 object 상태기계보다 앞선 class 부팅 상태기계가 있다.
  1. [버전 확정] CUDA lazy loading이 기본이 된 시점. CUDA에는 두 층의 lazy가 있다. ① context lazy: Runtime API는 첫 호출에서 primary context를 lazy 생성(cudaFree(0) 관용구로 강제), Driver API는 cuInit(0) 명시(Turn 4 언급). ② module lazy(신규): 커널/모듈을 context init이 아니라 첫 사용 시점에 로드. CUDA 11.7 opt-in으로 도입돼 CUDA 12.2에서 Linux(R535 드라이버) 기본, 12.3에서 전 플랫폼 기본이 됐다. CUDA_MODULE_LOADING=EAGER로 되돌리고 cuModuleGetLoadingMode로 질의한다. 이는 Q4(스케줄 정책)를 사용자 task가 아니라 런타임 자신의 부팅에 적용한 사례다 — 런타임이 초기화 비용을 "언제 낼지" 스케줄한다.
  1. accelerator 런타임의 부팅 1단계는 항상 자원 열거(enumeration)다. Level Zero zeInit() → zeDriverGet → zeDeviceGet, CUDA cuDeviceGetCount/cuDeviceGet. CPU 런타임(Go/JVM/Node)에는 이 단계가 없다 — 자기 자신 위에서 돌아 열거할 device가 없기 때문. 즉 "부팅 시 무엇을 열거하는가"가 CPU-런타임과 accelerator-런타임을 가르는 첫 구조 차이다.
  1. 많은 런타임은 clean teardown을 의도적으로 하지 않는다. Go os.Exit는 deferred 함수를 실행하지 않고, JVM Runtime.halt는 shutdown hook을 건너뛰며, glibc _exit는 atexit·stdio flush를 건너뛴다. 이유: 프로세스가 죽으면 OS가 주소공간·fd·페이지를 어차피 회수한다. 따라서 Q7의 "회수"는 대부분 OS에 위임되고, 런타임 teardown의 본질은 회수가 아니라 OS가 못 하는 것 — 파일 flush, 외부 종료 알림, 락파일 해제, 진단 dump — 즉 관측 가능한 부수효과의 마무리다. Turn 2의 zombie(exit status)와 Turn 6의 hs_err은 이 "관측 가능 산출물" 경로의 산물이다.
  1. 부팅 순서 의존성 그래프에는 소유자가 여럿이고, 경계에 병리가 있다(Q8). allocator는 scheduler보다, scheduler는 GC보다 먼저 켜져야 한다(뒷단이 앞단을 쓰므로). 이 런타임 내부 순서는 런타임이 소유한다. 그러나 그 앞단 — ELF/PE 로딩, 동적 링커 ld.so relocation, 글로벌 생성자 .init_array, TLS 셋업 — 은 OS/libc가 소유하고 main/런타임 진입 전에 끝난다. C++ static init order fiasco(번역 단위 간 글로벌 생성자 실행 순서 미정의)는 "소유자 없는 부팅 순서" 의 대표적 실패다 — Turn 7의 rendezvous("소유자 없는 진행")와 같은 병리의 부팅판.

더 구체화한 부분 (이전 턴 정정·보강)

  • [Turn 8 정정] Node의 microtask/nextTick 배수는 "phase당 1회"가 아니라 "callback당 1회"다. Turn 8은 "현재 callback 완료 → nextTick → microtask → 다음 phase callback"으로 그려 phase 경계에서만 배수되는 것처럼 읽힐 수 있다. 정확히는 Node 11(2018)+부터 각 개별 macrotask callback 하나가 끝날 때마다 nextTick 큐와 V8 microtask 큐를 완전 배수한다(브라우저 정렬). 즉 같은 timers phase 안의 setTimeout 콜백 2개 사이에도 microtask가 낀다. Node 10 이하에서는 phase 경계 위주로 배수돼 순서가 달랐다 — 완료 propagation의 세밀도가 버전 의존적이라는 Turn 8의 경고를 여기까지 넓혀야 한다.
  • [Turn 4/5 보강] lazy loading은 컴파일타임 스케줄의 거울상이다. Turn 5는 "PJRT/XLA는 스케줄을 컴파일타임으로 앞당긴다"고 했다. CUDA lazy module loading은 정반대로 로딩을 "실행타임 첫 사용"으로 뒤로 민다. 둘 다 Turn 5 축의 같은 질문 — "언제 비용을 지불하나" — 에 대한 반대 방향 답이다. 컴파일러는 당기고(eager schedule), lazy loader는 민다(defer load). 이 대칭이 부팅 비용의 배치 결정을 하나의 축으로 통합한다.

런타임 내부 구조로 환원

모든 런타임 부팅을 하나의 정규 부팅 시퀀스로 환원하면(Q9):

Phase 0  platform load    : ELF/PE 로딩, ld.so relocation, .init_array 생성자, TLS   [OS/libc 소유]
Phase 1  driver/VM init    : cuInit / zeInit / JNI_CreateJavaVM, device enumerate     [런타임+driver]
Phase 2  root object       : context / client / isolate / uv_loop / (g0·m0·P) 생성     [런타임]
Phase 3  subsystem online  : allocator → scheduler → GC/event loop (고정 의존 순서)     [런타임]
Phase 4  entry injection   : main goroutine / main thread / main script를 첫 work item으로 [런타임]
Phase 5  hand-off          : 제어를 스케줄러/이벤트 루프에 양도                          [런타임]

teardown은 이 역순(Phase 3→2→1, 비쌈)이거나, 흔히 Phase 5에서 곧장 _exit로 단락(서브시스템 teardown 생략, OS가 회수)한다.

관찰: Phase 3의 순서(allocator 먼저)는 예외가 없다. Go(mallocinit before procresize), JVM(heap/metaspace before thread subsystem), Node(V8 heap before uv_loop) 모두 할당자가 스케줄러보다 먼저 온다 — 이는 Turn 1~8의 5요소 모델에서 "allocator와 scheduler 중 무엇이 더 근본인가"에 부팅이 답을 준다: allocator가 더 아래층이다.

관련 맥락 / 배경 설명 — Q10 PIM 부팅(UPMEM SDK 근거)

Q9의 목록에 있었지만 실제 부팅 흐름으로는 미착수였던 UPMEM SDK로 PIM 부팅을 구체화한다. 실제 시퀀스:

dpu_alloc(nr_dpus, profile, &set)     # PIM 실행 자원 예약  (device enumerate + reserve)
dpu_load(set, binary, NULL)           # DPU IRAM에 컴파일된 바이너리 로드  (CUDA module load 대응)
dpu_copy_to(set, "buf", 0, host, sz)  # MRAM에 입력 배치  (명시 copy, coherence 없음)
dpu_launch(set, DPU_SYNCHRONOUS)      # 실행
dpu_copy_from(...); dpu_free(set)     # 결과 회수, 자원 반납

PIM 부팅은 CPU 런타임과 달리 세 층의 eager/lazy 선택을 강제한다:

  1. PIM region 예약 — PIM 뱅크는 일반 pageable 메모리가 아니라 특수 예약이 필요하다. OS/메모리 컨트롤러에서 연속·고정 영역을 확보해야 하므로 사실상 eager(늦추기 어려움). → Turn 5 축의 eager 극단.
  2. kernel 로드(DPU binary / 고정기능 opcode 테이블) — CUDA 12.2 lazy module loading처럼 첫 사용까지 defer 가능. 커널 종류가 많은 워크로드의 cold-start를 줄인다.
  3. mode capability 획득 — 채널을 PIM 모드로 전환할 권한을 부팅 시 잡을지, 첫 command에서 잡을지. → Turn 5의 mode_wait와 직결.

잠재적 오류 또는 확인 필요 지점

  • CUDA lazy loading 기본값은 12 전체가 아니다. 12.0/12.1은 opt-in, 12.2 Linux(R535+), 12.3 전 플랫폼부터 기본. "CUDA 12부터 무조건 lazy"는 부정확하며, 구드라이버/Windows에서는 다를 수 있다.
  • "런타임은 teardown을 안 한다"는 과장. 장수 서버, 리소스 릭 진단, 외부 flush/알림이 필요한 경우 teardown은 필수다. fast-exit는 "OS가 회수 가능한 자원" 에 한정되고, 외부 파일 flush·네트워크 종료 핸드셰이크·락파일 해제는 fast-exit에서 유실될 수 있다(Turn 6 abnormal exit과 같은 함정).

이번 턴의 확장 요약

이번 턴은 Turn 9가 남긴 에러/예외 전파를 독립 런타임 하위시스템으로 정식화한다. 결론은 다음이다.

런타임의 에러 처리는 단순히 “실패하면 에러 코드를 반환한다”가 아니다. 런타임은 실패를 최소 네 가지 방식 중 하나로 운반한다.

1. error-as-value       # Result, Go error, cudaError_t
2. error-as-unwind      # Java exception, Go panic, Rust panic(unwind)
3. error-as-async-state # CUDA async launch error, PJRT event/buffer error, Promise rejection
4. error-as-poison      # CUDA context lost, NCCL communicator abort, PIM context_lost

이 구분이 Q5/Q6에 중요하다. 같은 failed라도 어떤 실패는 호출자에게 값으로 돌아오고, 어떤 실패는 스택을 되감으며 cleanup을 실행하고, 어떤 실패는 나중에 event/sync 지점에서 관측되며, 어떤 실패는 런타임 객체 전체를 오염시켜 재생성만 가능하게 만든다.

새로 추가한 사실

CUDA Runtime API에는 host thread별 last-error 슬롯이 있고 cudaGetLastError()는 이를 읽고 초기화한다. 공식 문서도 이 함수가 이전 비동기 launch의 에러를 반환할 수 있다고 설명한다. 하지만 이것은 “마지막 에러 레코드”이지, 항상 context 전체가 망가졌다는 뜻은 아니다. 반대로 illegal access, device lost, launch failure 류는 이후 동기화나 API 호출에서 더 큰 범위의 failure로 드러날 수 있다. 따라서 CUDA 에러는 최소한 per-thread error slot, stream/event completion error, context/device failure로 나눠야 한다. CUDA Runtime API

  1. CUDA의 “last error”와 “context poison”은 같은 것이 아니다.

JVM spec은 exception 발생 시 현재 method의 handler table을 검색하고, 없으면 frame을 pop하며 caller frame으로 전파한다고 정의한다. 또한 JVM exception은 precise해야 하므로, 예외 지점 이전 효과는 보이고 이후 효과는 보이면 안 된다. 즉 JVM의 exception subsystem은 단순 객체 전달이 아니라 stack frame state, exception table, JIT speculative execution 보정까지 포함하는 런타임-컴파일러 공동 구조다. JVM Spec §2.10

  1. Java exception은 값이 아니라 “정밀한 제어흐름 전이”다.

Go blog는 panic이 현재 함수의 정상 흐름을 멈추고 deferred call을 실행하면서 stack을 거슬러 올라간다고 설명한다. recover는 deferred function 안에서만 panicking goroutine을 붙잡을 수 있다. 중요한 점은 Go 표준 라이브러리 관례가 “내부 구현은 panic/recover를 써도 외부 API는 error return value를 제공”한다는 것이다. 즉 Go는 error-as-unwind를 내부 shortcut으로 쓰되, public boundary에서는 error-as-value로 변환한다. Go Blog

  1. Go panic은 goroutine-local unwind이고, 외부 API는 여전히 error value를 선호한다.

Rust는 기본적으로 panic 시 stack을 unwind하고 cleanup하지만, panic = 'abort'를 선택하면 cleanup 없이 프로세스를 끝내고 OS 회수에 맡길 수 있다. Turn 9의 fast-exit 논의와 직접 연결된다. 같은 언어의 같은 panic!도 런타임 설정에 따라 error-as-unwind에서 error-as-fatal-abort로 바뀐다. Rust Book

  1. Rust panic은 build/profile 정책에 따라 unwind 또는 abort가 된다.

CUDA kernel launch는 host에 비동기로 돌아올 수 있고, 실제 failure는 뒤의 cudaGetLastError, stream sync, event query, memory copy에서 관측될 수 있다. Node Promise rejection, PJRT async error, PIM command fault도 같은 구조다. 따라서 Q6의 산출물은 result/error 하나가 아니라 다음처럼 쪼개야 한다.

  1. 비동기 런타임에서는 “실패 발생 시점”과 “실패 관측 시점”이 분리된다.

text failure occurred → failure recorded → failure associated with event/future/context → user observes failure → runtime decides recover / poison / abort

잠재적 오류 또는 확인 필요 지점

  • CUDA sticky error를 모든 CUDA error에 일반화하면 안 된다. cudaGetLastError()의 per-thread slot과 context/device-level fatal 상태는 분리해야 한다. 특히 “마지막 에러가 있다”와 “context를 재사용할 수 없다”는 다른 명제다.
  • Java exception과 OS signal을 같은 unwind로 보면 안 된다. Java exception은 JVM이 정의한 precise control transfer이고, SIGSEGV/SIGKILL 같은 OS signal은 JVM이 복구할 수도, fatal VM error로 승격할 수도 있다. HotSpot fatal error의 hs_err는 Java exception 산출물이 아니라 VM crash diagnostic이다.
  • Go panic은 goroutine 경계를 자동으로 넘지 않는다. 한 goroutine의 panic을 다른 goroutine에서 recover할 수 없다. 분산/PIM collective failure에 Go식 panic 모델을 그대로 쓰면 그룹 실패 전파를 표현하지 못한다.
  • Rust panic=abort에서는 RAII cleanup 기대가 깨진다. Drop 기반 회수를 전제로 device buffer를 설계하면 abort profile에서 외부 자원 flush가 누락될 수 있다. scarce device/PIM resource는 process 종료에 기대지 않는 explicit close가 필요하다.
  • PIM context_lost의 관측 가능성은 하드웨어/드라이버 API에 달려 있다. memory controller가 refresh/mode reset을 완전히 감추면 runtime은 단순 timeout만 볼 수 있다. 반대로 ECC, row fault, mode violation을 명시 status로 올리면 PimEvent에 세분화된 error code를 실을 수 있다.

핵심 결론 세 가지:

  1. 런타임은 "매번 새로 태어나는 것"이 아니다. Turn 9의 running 메타상태는 직렬화 가능한 값이 될 수 있다. 그러면 Q1의 답에 새 축이 붙는다 — 런타임은 부팅해서 존재하게 되는 것이면서 동시에 스냅샷으로 복제될 수 있는 상태다. bootstrapping → running 뿐 아니라 running → checkpointed → running'(다른 프로세스 정체성, 같은 관리 상태)이라는 경로가 열린다.
  1. 부팅 비용은 세 방향으로 옮길 수 있고, 실제 시스템은 그 혼합이다(Q4). Turn 5의 compile↔runtime 축, Turn 9의 "언제 비용을 내나" 를 여기서 pull-forward(당김) / defer(미룸) / clone(복제) 세 전략으로 완성한다.
  1. 스냅샷 가능성은 소유권의 판별 시약(試藥)이다(Q8). Turn 9의 fast-exit이 teardown 방향에서 소유를 드러냈다면(OS가 회수하는 것 = 런타임이 위임한 것), 스냅샷은 복제 방향에서 같은 경계를 드러낸다: 직렬화되면 런타임 소유, coordinated 배출이 필요하면 경계 공유, 재생성해야 하면 커널·물리 소유.

더 구체화한 부분 (이전 턴 정정·보강)

  • [Turn 9 정정] GraalVM native-image는 순수 "pull-forward(a)"가 아니라 compile+clone 혼합이다. Turn 9는 native-image를 "(a) 컴파일타임으로 당김"으로 분류했다. 정확히는 native-image가 build-time에 초기화된 객체를 image heap(빌드 결과 바이너리에 직렬화된 초기 heap)으로 baked-in한다 — 즉 코드는 AOT로 당기고(pull-forward), 초기 heap 상태는 스냅샷으로 복제(clone)한다. Turn 9의 (a)/(b)/(c) 3분법은 깔끔한 근사이고, 실제 시스템은 대부분 혼합이다(JEP 483 = mmap+partial-clone, native-image = pull+clone).
  • [Turn 9 보강] "device handle은 스냅샷 불가"의 완화. Turn 9의 fast-exit 논의는 device/커널 자원을 회수-위임 대상으로만 봤다. cuda-checkpoint는 device 상태가 coordinated drain을 거치면 이동 가능함을 보인다. 따라서 Turn 9의 소유 표에 "이동 가능성(portability)" 열을 추가해야 한다: 같은 "런타임 소유 아님"이라도 fd(재개설), device mem(host 경유 이동), PID/RNG(재생성)는 서로 다른 복원 전략을 요구한다.
  • [Turn 10 연결] 스냅샷 실패는 Turn 10 에러 분류의 "poison"과 다른 새 부류다. 복원 시 포트 점유·device 부재·주소 충돌로 인한 restore_failed, 그리고 고유 상태 재생성 실패로 인한 identity_collision은 Turn 10의 error-as-value/unwind/async/poison 어디에도 딱 맞지 않는다. 이것은 "과거 시점에는 유효했으나 복원 시점에는 무효/중복이 된 상태" — 시간축 정합성(temporal staleness) 오류다.

관련 맥락 / 배경 설명 — Q10 PIM 체크포인트의 특수 난점

PIM은 이 축의 극우단이며, GPU보다 한 단계 더 어렵다. cuda-checkpoint가 device mem을 host로 배출하고 원래 주소에 재매핑하듯, PIM 스냅샷도 bank 내용을 host로 배출해야 한다. 그런데 PIM만의 추가 문제:

  • placement가 곧 연산 의미다. Turn 5에서 못박았듯 PIM 연산은 피연산자가 상주한 bank/subarray에서만 유효하고, PimCommandSeq는 bank 주소를 인코딩한다. 복원 시 메모리 컨트롤러가 다른 물리 bank를 배정하면, 스냅샷된 command seq의 주소가 전부 어긋난다. 즉 PIM 복원은 순수 바이트 복원이 될 수 없고, placement 재도출 + command seq 재lowering(부분 재컴파일)을 요구한다.
  • 따라서 복원 경로에 컴파일러가 들어온다. Turn 5의 "compiler가 placement/command seq를 소유"가 여기서 귀결을 낳는다: PIM 스냅샷 복원 = 부분 재컴파일. 이는 native-image의 "image heap을 그대로 복원"과 정반대 — PIM은 heap 값은 복원해도 배치 바인딩은 복원 못 한다.
  • 설계 함의: PimCommandSeq를 "placement-independent"로 기록하라. 절대 bank 주소가 아니라 상대 bank offset / 심볼릭 placement로 낮추면, 복원 시 실제 bank에 rebind만 하면 된다. 이는 Turn 9 Phase 0의 ld.so relocation·PIC(position-independent code)와 정확히 동형이다 — 스냅샷 복원의 재바인딩 문제 = 로더의 relocation 문제를 placement에 적용한 것.

PIM 메타 상태기계 추가:

pim_ready
  → pim_checkpoint_draining   # bank 내용→host, mode capability 반납  [MC+device 공동]
  → pim_checkpointed          # host staging + 심볼릭 command seq를 값으로
  → pim_replace_on_restore    # ★ 물리 bank 재배정 → placement 재도출·command seq rebind  [compiler+allocator]
  → pim_ready'                # 복원 완료

설계 원칙 갱신: Turn 5의 "스케줄은 컴파일러에, 배치는 할당자에, 리덕션은 MC에", Turn 9의 "영역 예약은 부팅에(eager)"에 이번 턴이 "command seq는 placement-independent로 기록해 복원 시 rebind 가능하게" 를 추가한다.

잠재적 오류 또는 확인 필요 지점

  • JEP 483은 "무조건 빠름"이 아니다. AOT 캐시는 training run이 본 workload·classpath에 종속되고, classpath가 바뀌면 무효화될 수 있다. "코드 변경 없이 40%"는 PetClinic 특정 사례이며, 워크로드별로 이득이 다르다. 또 JEP 483은 heap을 담지 않으므로 CRaC/native-image와 같은 "완전 스냅샷"이 아니다 — 계층을 뭉개면 안 된다.
  • cuda-checkpoint를 "GPU 스냅샷 해결"로 일반화하면 과장. x64 전용, UVM/IPC 미지원, driver 550+ 제약이 있고, multi-GPU·IPC 공유 메모리·MIG 시나리오는 확인 필요. "device 상태는 이제 자유롭게 클론된다"는 틀리다 — coordinated drain이 전제다.
  • CRaC는 투명하지 않다. 열린 fd/socket 금지 제약 때문에 프레임워크가 CRaC-aware Resource 핸들러를 구현해야 한다. 아무 앱이나 체크포인트되지 않는다 — SnapStart가 "자동"으로 보이는 이유는 AWS가 런타임·라이브러리 레벨에서 이 핸들러를 미리 깔았기 때문임을 구분해야 한다.
  • SnapStart uniqueness 보정은 일부 라이브러리만 자동이다. SecureRandom·OpenSSL은 재초기화되지만, 애플리케이션이 init 단계에서 캐싱한 커스텀 난수·토큰·타임스탬프는 여전히 복제되어 identity_collision을 일으킬 수 있다. "vmgenid가 있으니 안전"은 커널 엔트로피 소비자에 한정된 명제다.
  • PIM pim_replace_on_restore(재lowering)는 설계 제안이지 기존 동작이 아니다. 심볼릭 command seq 표현·복원 시 rebind가 하드웨어/드라이버에서 실제로 지원되는지는 미확인. 과설계 위험(Turn 5의 refresh 경고와 동종)이 있으므로, 먼저 "MC가 복원 시 동일 물리 bank를 보장하는가"의 관측 가능성부터 확인해야 한다.

결론은 다음이다. HIP는 CUDA와 API 표면이 매우 비슷하지만, AMD 플랫폼에서는 CUDA Driver식 context 조작이 중심이 아니다. HIP 문서는 hipCtxCreate, hipCtxDestroy 같은 context API를 AMD 플랫폼에서 deprecated로 표시하고, 대신 thread-local default device + stream/event + memory pool + ROCr/rocclr 하위 런타임 구조가 중심에 온다. 따라서 Q9의 공통 구조는 “CUDA primary context가 모든 accelerator runtime의 원형”이 아니라 다음처럼 수정해야 한다.

Accelerator runtime 공통 루트:
  device enumeration
  + per-thread current/default device
  + command stream/queue
  + event/fence
  + module/code object
  + device memory/mempool
  + async error observation
  + profiling/trace surface

CUDA의 context는 이 구조를 담는 한 구현 방식이고, HIP/ROCm에서는 context API보다 device/stream/runtime 계층이 더 앞에 보인다.

새로 추가한 사실

HIP 문서는 hipInit()을 “runtime 초기화 타이밍을 제어하는 API”로 설명하고, 대부분의 HIP API가 runtime을 암묵적으로 초기화한다고 적는다. 즉 Turn 9의 accelerator bootstrap 모델에서 HIP는 CUDA Runtime API와 같이 uninitialized → lazy_initialized 경로를 갖는다. HIP Initialization

  1. HIP runtime은 명시 hipInit() 없이도 대부분 API가 암묵 초기화한다.

hipSetDevice()는 호출 host thread의 default device를 설정하고, 이후 hipMalloc, stream/event 생성, kernel launch의 기본 device를 결정한다. 문서는 이 default device가 thread-local storage에 저장된다고 설명한다. 따라서 HIP의 Q8 소유 경계에는 host thread TLS current device가 새로 들어간다. HIP Device management

  1. HIP의 현재 device는 host thread-local state다.

hipCtxCreate() 등은 “NVIDIA cuCtx driver API 대응용” 성격이고 AMD 플랫폼에서는 deprecated라고 표시된다. 이는 CUDA와 HIP를 1:1 context 모델로 등치하면 틀릴 수 있음을 뜻한다. HIP의 실제 사용자 모델은 device + stream + event + memory API 쪽이다. HIP Context management

  1. HIP context API는 AMD 플랫폼에서 deprecated다.

HIP 비동기 실행 문서는 stream을 “given device에서 순서대로 실행할 command FIFO”로 설명하고, 같은 device에 연결된 여러 stream의 command는 서로 순서 보장이 없다고 설명한다. 이는 CUDA/Level Zero/OpenCL과 같은 accelerator runtime의 Q4 공통 스케줄 구조다. HIP asynchronous execution

  1. HIP stream은 FIFO command buffer이며, stream 간 전역 순서는 보장되지 않는다.

hipEventRecord()는 stream의 이전 command가 끝난 뒤 event가 recording → recorded로 전이한다고 설명된다. 또한 hipEventDisableSystemFence 플래그는 system-scope acquire/release fence를 끄며, 이 경우 device memory가 host나 다른 device에 보이지 않을 수 있다고 경고한다. 즉 event는 단순 timestamp가 아니라 completion + memory visibility edge다. HIP Event management

  1. HIP event에는 visibility/fence 성격이 들어 있다.

HIP 문서는 kernel launch가 직접 error를 반환하지 않으므로 hipGetLastError()나 hipPeekAtLastError()로 잡아야 한다고 설명한다. 또 ROCm 7.0 이전에는 hipGetLastError()가 hipSuccess나 hipErrorNotReady 같은 “비에러”도 반환할 수 있었지만, ROCm 7.0부터는 현재 thread의 실제 HIP API error를 반환한다는 주석이 있다. HIP Error handling

  1. HIP error 관측은 CUDA와 유사하지만 ROCm 7.0 기준 세부 의미가 정리됐다.

hipMallocAsync()는 allocation operation을 stream에 삽입하고 pointer를 즉시 돌려주지만, 그 allocation operation이 완료되기 전에는 접근하면 안 된다. hipFreeAsync()도 free operation을 stream에 삽입한다. mempool trim은 outstanding async allocation, 그리고 host에서 완료 관측되지 않은 async free 때문에 OS backing을 바로 반납하지 못할 수 있다. HIP Stream ordered memory allocator

  1. HIP에도 stream-ordered allocator가 있다.

ROCprofiler-SDK는 HIP runtime call trace와 kernel dispatch, memory move 같은 asynchronous activity record를 수집하고, callback tracing과 activity logging을 제공한다. rocprofv3 --hip-trace는 HIP API trace CSV를 만들고, dynamic attach로 실행 중 process에 붙을 수도 있다. ROCTx는 app이 직접 range/marker를 넣는 annotation API다. ROCprofiler-SDK, rocprofv3, ROCTx

  1. ROCm observability는 별도 stack이다.

잠재적 오류 또는 확인 필요 지점

  • HIP를 “CUDA와 완전히 같은 runtime”이라고 쓰면 안 된다. API 이름은 비슷하지만 AMD 플랫폼에서는 context management API가 deprecated이고, runtime 사용자는 thread-local default device와 stream/event/memory API 중심으로 움직인다.
  • hipGetLastError()는 context poison 판정기가 아니다. Turn 10에서 CUDA에 대해 분리한 것처럼, HIP에서도 per-thread last error와 device/context-level fatal state를 구분해야 한다. last error exists는 device unusable과 같은 말이 아니다.
  • HIP event의 fence flag는 PIM 설계에 중요하다. hipEventDisableSystemFence처럼 completion과 host/device visibility가 분리될 수 있다면, PIM의 PimEvent.completed()와 PimEvent.hostVisible()도 분리해야 한다. “event 완료 = host가 결과를 읽어도 됨”은 하드웨어 fence 정책에 따라 틀릴 수 있다.
  • stream-ordered free는 “API가 반환했으니 free 완료”가 아니다. hipFreeAsync()는 stream에 free operation을 넣을 뿐이며, host 관측 전에는 allocator가 OS backing을 release하지 못할 수 있다. PIM buffer free도 close()와 reclaimed()를 분리해야 한다.
  • ROCm checkpoint는 공식 HIP runtime object로 확인되지 않았다. CRIUgpu 같은 연구 결과는 중요하지만, Q10 설계 근거로 쓸 때는 “공식 API 관측 사실”과 “가능한 구현 전략”을 분리해야 한다.

이번 턴의 확장 요약

Turn 4~12는 stream·queue·command list를 추상 객체로만 다뤘다("ordered submission", "FIFO command buffer"). 이번 턴은 Turn 12가 남긴 확장점 2 — "HIP stream 아래 HSA queue로 내려가라" — 를 열되, 한 벤더에 머물지 않고 HSA AQL·NVMe·virtio를 관통하는 하드웨어 큐 ABI로 일반화한다. 핵심 결론 세 가지:

  1. "stream/queue"의 실체는 어디서나 동일한 4요소다: ① 공유메모리에 있는 링 버퍼(고정크기 패킷 배열), ② producer/consumer 인덱스, ③ doorbell(producer→consumer 알림), ④ completion signal + fence(consumer→producer 알림 + 가시성). HSA AQL, NVMe SQ/CQ, virtio virtqueue가 이 패턴의 세 인스턴스다. Turn 8이 "command queue"라 부른 것은 실제로는 커널을 우회하는(kernel-bypass) 유저모드 링이다.
  1. [Turn 12 정정] HIP와 CUDA의 Q7 자원 모델은 사실 다르다. Turn 12는 HIP stream을 "FIFO command buffer"로 두고 HIP≈CUDA로 다뤘다. 그러나 HSA 층에서 stream은 희소한 하드웨어 AQL 큐 풀에 다중화되는 가상 큐이고, HIP는 이 희소성을 GPU_MAX_HW_QUEUES로 노출하는 반면 CUDA stream은 하드웨어 큐 수와 decouple(가상화)해 감춘다. 오버섭스크립션이 HIP에선 성능 페널티로 새어나온다.
  1. completion과 visibility가 패킷 ABI에 분리 인코딩된다 — Turn 12 확장점 3("event completion vs visibility 분리")의 하드웨어 근거를 찾았다.

새로 추가한 사실

HSA 큐는 core::AqlPacket 64B 슬롯의 circular buffer이고, 64-bit write_index(런타임이 다음 할당할 슬롯)와 read_index(하드웨어가 다음 소비할 슬롯)를 갖는다. 실제로 패킷을 꺼내 dispatch하는 주체는 소프트웨어 스케줄러가 아니라 Command Processor(CP) = packet processor다. 즉 Q4의 "스케줄러"가 CPU 런타임(Go work-stealing, Node phase queue)과 근본적으로 다르다 — 디스패치가 하드웨어로 내려가 있다. HSA SysArch, ROCR queues

  1. AQL 패킷은 64바이트 고정, 큐는 GPU-접근가능 메모리의 원형 링, 스케줄러는 하드웨어다.

패킷 타입은 kernel dispatch, agent dispatch, barrier-AND, barrier-OR 네 종류다. barrier 패킷은 signal이 만족될 때까지 큐 진행을 막는다. 즉 Turn 7이 별도 객체로 본 rendezvous/의존성이, 하드웨어 큐에서는 링에 끼워 넣는 패킷 하나로 표현된다 — 스케줄 로직이 코드가 아니라 큐에 실린 데이터가 된다.

  1. AQL 패킷 타입은 dispatch뿐 아니라 barrier를 포함하고, 큐 내부 의존성이 데이터로 인라인된다.
  1. 제출은 유저모드다(kernel bypass). 에이전트는 링의 write_index 위치에 패킷을 쓰고 큐 doorbell을 울려 packet processor에 알린다. per-submit syscall이 없다 — 드라이버가 fast path에서 빠진다. 이는 Turn 9의 부팅 논의를 보강한다: 부팅 시 드라이버가 큐·doorbell·signal을 매핑해두면, 이후 정상상태 제출은 커널을 통과하지 않는다. Q8의 "런타임 vs 드라이버" 경계가 부팅에만 있고 제출 fast path엔 없다.

HIP는 자체 HSA 큐 풀을 GPU_MAX_HW_QUEUES로 제어하며 여러 stream을 이 풀에 다중화한다. HSA 큐는 하드웨어 큐와 거의 1:1이라 오버섭스크립션 시 큰 성능 페널티가 난다. 반대로 "모든 CUDA stream은 가상화돼 하드웨어 수와 분리"된다. 따라서 Turn 12가 뭉갠 "HIP=CUDA"는 Q7에서 갈라진다: HIP는 하드웨어 큐 희소성이 사용자에게 새어나오고, CUDA는 런타임이 흡수한다. GROMACS #4965, ROCm #2705

  1. 하드웨어 큐는 희소 자원이고, HIP는 이를 노출한다.

AQL 패킷 헤더는 acquire_fence_scope(패킷 active 진입 전)와 release_fence_scope(kernel 완료 후 패킷 complete 전) 필드를 갖고, 각 scope는 none / agent / system으로 설정된다. 결정적 함의: signal이 0이 돼 "완료"돼도, release fence가 system scope가 아니면 host나 다른 agent는 결과를 못 본다. 이것이 Turn 12가 인용한 hipEventDisableSystemFence의 하부 근거다 — "완료 = 읽어도 됨"은 fence scope에 달렸다. HSA SysArch 1.0

  1. completion(signal)과 visibility(fence)가 패킷 헤더에 따로 산다.

NVMe는 host 메모리의 SQ(64B command 링) / CQ(16B completion 링) + 컨트롤러 안의 doorbell 레지스터로 구성된다. host가 command를 SQ tail에 쓰고 SQ tail doorbell을 울리고, device는 CQ에 완료를 쓴 뒤 각 엔트리의 phase bit를 토글해 host가 인터럽트 없이 폴링으로 완료를 감지하게 한다. 최대 64K 큐. 즉 Turn 8의 Node completion-poll이 스토리지에도 그대로 있다 — phase bit 폴링 = 하드웨어판 completion phase. 더 구체화한 부분 (이전 턴 정정·보강)

  • [Turn 12 정정·심화] "stream 간 전역 순서 없음"의 진짜 이유. Turn 12는 관찰만 했다. 근거는 이것이다: 각 stream이 독립 AQL 큐에 매핑되고, 각 큐의 read_index를 별개의 CP 파이프가 독립적으로 전진시키므로 큐 간 전역 순서가 물리적으로 존재하지 않는다. 순서를 강제하려면 barrier 패킷 + signal로 큐를 가로지르는 의존 간선을 명시해야 한다(Turn 7 rendezvous의 단일 device판).
  • [Turn 8 보강] Node phase queue와 하드웨어 큐는 인덱스 소유자가 반대다. Node의 phase queue는 producer(콜백 등록)와 consumer(JS thread)가 같은 프로세스다. 하드웨어 링은 producer(런타임/host)와 consumer(CP/device)가 서로 다른 실행 도메인이라, write_index는 런타임이·read_index는 하드웨어가 소유하는 쪼개진 소유가 된다. 이 분리가 doorbell/signal(도메인 간 알림)을 강제한다 — Node엔 doorbell이 없는 이유.
  • [Turn 10 연결] async 실패의 관측 지점이 여기서 물리적으로 확정된다. Turn 10의 async_failed_pending → observed_failed는 추상이었다. NVMe에선 CQ 엔트리의 status field가, HSA에선 signal 값과 큐 error 상태가 그 관측 지점의 실체다. 실패는 "링에서 완료 엔트리를 읽는 순간" 관측된다.

관련 맥락 / 배경 설명 — Q10: PIM에서 하드웨어 큐 ABI가 깨지는 지점

이번 턴의 4요소 링을 PIM에 얹으면 세 곳이 부러진다. 이게 PIM 런타임의 핵심 설계 제약이다.

  1. 링 버퍼를 어디 두는가 — PIM은 연산 대상 메모리에 링을 못 둔다. GPU/NVMe는 링을 "그냥 메모리"에 둔다. 그러나 PIM은 그 메모리에서 연산하고, Turn 5의 mode_wait 동안 PIM 영역이 host에 안 보이거나 일반 read/write와 충돌한다. 따라서 PIM 큐 링을 PIM 영역에 두면 doorbell·write_index 갱신이 compute와 경합한다. → 링은 예약된 non-PIM 영역(Turn 9 부팅 시 확보)에 둬야 하며, 이는 Turn 7의 대칭 배치 요구와 충돌 가능(뱅크 대칭 데이터 + 별도 큐 영역).
  1. doorbell 대응물이 MMIO가 아니라 DDR 명령이다. GPU doorbell은 별도 MMIO 레지스터다. PIM엔 그런 사이드밴드 경로가 없다 — 모든 것이 DDR 명령 스트림을 통과한다. 그래서 PIM "doorbell"은 특수 mode-register write / PIM-launch 명령이 되고, 이건 반드시 메모리 컨트롤러를 경유한다. 결론: PIM엔 GPU식 kernel-bypass 유저모드 링이 자연스럽지 않다. 제출 fast path가 항상 MC 드라이버를 통과하므로, Q8의 "제출은 커널을 우회한다"가 PIM에선 역전된다.
  1. completion signal이 없다 — 폴링만 남는다. GPU는 signal→인터럽트로 host를 깨운다. PIM 엔진은 host를 깨우는 인터럽트 경로가 대개 없다. 따라서 완료 감지는 NVMe phase-bit식 status 폴링으로 좁혀진다 — host가 non-PIM 영역의 status word를 읽어 완료를 본다. → Turn 8의 completion-poll phase가 PIM host dispatcher에 옵션이 아니라 필수임이 여기서 확정된다.

PIM 링 상태기계(단일 device 층):

pim_slot_empty
  → pim_packet_written        # non-PIM 영역 링에 심볼릭 placement 패킷 기록  [host]
  → pim_launch_issued         # MC가 mode-reg write로 PIM launch (doorbell 대응)  [MC 소유]
  → pim_executing             # 뱅크에서 연산 (mode_wait 통과)  [PIM 엔진]
  → pim_writeback             # 결과 셀→버퍼
  → pim_status_posted         # status word 갱신 (인터럽트 없음)
  └→ pim_host_polled_done     # host가 폴링으로 완료+가시성 확인  [host 폴링]

설계 원칙 갱신: Turn 5의 "스케줄은 컴파일러에, 배치는 할당자에, 리덕션은 MC에", Turn 9의 "영역 예약은 부팅에", Turn 11의 "command seq는 placement-independent로"에 이번 턴이 "제출 링은 non-PIM 예약 영역에, doorbell은 mode-reg 명령으로, 완료는 인터럽트가 아닌 status 폴링으로"를 추가한다.

잠재적 오류 또는 확인 필요 지점

  • CUDA stream→하드웨어 큐 매핑은 비공개. "CUDA는 하드웨어 수와 decouple해 가상화"는 GROMACS·ROCm 이슈의 비교 서술과 관측된 동작에서 온 것이지 NVIDIA 내부 문서 근거가 아니다. "CUDA엔 하드웨어 큐 희소성이 전혀 없다"로 확대하면 과장 — MES/하드웨어 스케줄러 한도는 존재하되 런타임이 흡수한다는 뜻으로 좁혀야 한다.
  • AQL "barrier bit"(패킷 헤더의 barrier 플래그)와 "barrier 패킷"(barrier-AND/OR 타입)은 다른 것이다. 전자는 "이 패킷이 이전 패킷 완료를 기다림", 후자는 "signal들을 기다리는 전용 패킷". 두 메커니즘을 뭉개면 큐 내부 의존성 모델을 틀리게 그린다.
  • NVMe 16B completion vs HSA signal의 정보량 차이. NVMe CQ 엔트리는 status·command id를 담은 구조체라 실패 원인을 실어 나른다. HSA signal은 기본적으로 정수 값 하나라 완료/에러 세분화가 별도 큐 error 상태에 의존한다. "signal = completion record"로 등치하면 Turn 10의 에러 운반 분석이 부정확해진다.
  • virtio available/used ring은 이번 턴에서 미검증 인용이다. kick/interrupt·descriptor chain 세부는 다음 턴에서 1차 문헌으로 확인해야 하며, 여기 표의 virtio 열은 "추정"으로 표기했다.
  • PIM에 인터럽트가 "전혀 없다"는 단정 위험. 일부 연구용 PNM/PIM 컨트롤러는 완료 인터럽트나 이벤트 라인을 제안한다. "PIM=폴링만"은 상용 DRAM-PIM 근사이며, 아키텍처별 completion 통지 경로 유무를 확인해야 한다(Turn 5 refresh 과설계 경고와 동종).
