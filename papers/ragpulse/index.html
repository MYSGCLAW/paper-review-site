RAGPulse: An Open-Source RAG Workload Trace to Optimize RAG Serving Systems

저자 · Zhengchao Wang, Yitao Hu, Jianing Ye, Zhuxuan Chang, Jiazheng Yu, Youpeng Deng, Keqiu Li 출처 · arXiv preprint (arXiv:2511.12979v1 [cs.LG]) 연도 · 2025

핵심 요약

RAGPulse는 실제 운영 중인 대학 전체(university-wide) 정책 Q&A RAG 시스템 'Twen'에서 수집한 오픈소스 RAG 워크로드 트레이스(workload trace) 데이터셋이다. 저자들은 기존의 범용 LLM 추론 트레이스(generic LLM inference trace)가 RAG 고유의 다단계 파이프라인(multi-stage pipeline: retrieve, augment, generate)과 지식 의존성(knowledge dependency)을 포착하지 못해 학계 연구와 실제 배포 사이에 성능 격차(performance gap)가 생긴다고 주장한다. 이를 메우기 위해 2024년 4월부터 40,000명 이상의 학생·교직원에게 서비스해 온 Twen 시스템에서 1주일간 샘플링한 7,106건의 요청 레코드를 공개하며, 원본 텍스트를 모두 제거하고 remapped hash_ids로 대체하는 프라이버시 보호 해시 기반 포맷(privacy-preserving hash-based format)을 사용해 구조적·시간적 특성만 보존한다. 트레이스 분석 결과 (1) 주기적 부하 변동(periodic load fluctuation), (2) 입력 길이에 따라 변하는 동적 입력 구성(dynamic input composition), (3) 소수의 hot document에 접근이 집중되는 long-tail 문서 사용 빈도, (4) 요청 간 문서 중복(inter-request document overlap)에 따른 강한 시간적 지역성(temporal locality)이라는 네 가지 특성이 드러났고, 이는 retrieval caching과 content-aware batching이 유망한 최적화 방향임을 실증적으로 뒷받침한다. 또한 데이터셋의 실효성을 검증하기 위해 vLLM 기반 프로토타입 RAG 서비스와 클라이언트-서버 구조의 워크로드 리플레이(workload replay) 벤치마킹 플랫폼(TTFT/TPOT 측정)을 함께 구축·공개하였다. 다만 본문에는 벤치마킹 플랫폼의 실측 TTFT/TPOT 결과가 보고되지 않으며, 이 논문은 시스템 최적화 기법 제안이 아니라 데이터셋·인프라 제공(artifact) 논문에 해당한다.

해결하려는 문제

논문 주장

LLM의 근본적 한계와 RAG의 부상

LLM은 학습 시점 이후 정보를 알지 못하는 knowledge cutoff와, 사실이 아닌 내용을 확신 있게 생성하는 hallucination이라는 두 가지 내재적 한계를 가진다. RAG(Retrieval-Augmented Generation)는 LLM의 추론·생성 능력과 외부 지식베이스의 실시간 사실성을 결합해 이를 해결하는 방식으로, 지식 저장을 모델의 추론 능력으로부터 분리(decoupling)함으로써 LLM 애플리케이션의 신뢰성(reliability), 시의성(timeliness), 추적 가능성(traceability)을 크게 높인다. 논문은 RAG가 엔터프라이즈급 Q&A 시스템, 지능형 문서 분석 도구, 신뢰 가능한 AI 어시스턴트 구축의 표준(standard)이 되었다고 서술한다.

PDF p. 1

RAG 서빙의 병목은 복합적(composite)이다

순수 LLM 추론과 달리 RAG 서빙 시스템의 성능 병목은 복합적이다. 성능은 생성 단계(예: KV cache 효율)만이 아니라 검색 단계(예: vector database latency), 그리고 두 단계 사이의 복잡한 상호작용(예: adaptive batching)에 의해서도 크게 좌우된다. 따라서 생성 단계만 기술하는 트레이스로는 시스템 최적화 연구를 제대로 수행할 수 없다.

PDF p. 1

핵심 문제: RAG-specific 실세계 트레이스의 부재

논문이 정의하는 핵심 문제는 '기존 범용 LLM 추론 트레이스가 RAG 고유의 워크로드 동역학을 포착하지 못한다'는 것이다. 실세계 RAG 워크로드는 매우 편향된(highly skewed) 지식 청크 접근 빈도와 유의미한 요청 간 시간적 지역성(hot document 현상)을 보이는데, 이러한 특성은 retrieval caching과 content-aware batching 설계에 결정적임에도 범용 트레이스에는 전혀 존재하지 않는다. 이 격차(gap)는 연구 도구와 실제 배포 사이의 성능 괴리를 낳으며, 학계와 산업계 모두 실세계 RAG 워크로드 특성을 정확히 반영하는 공개 벤치마크 데이터셋을 시급히 필요로 한다는 것이 이 논문의 출발점이다.

기존 방식의 한계

논문 주장

다단계 파이프라인 복잡성 미반영

RAG 요청은 retrieval, re-ranking, generation 등 서로 구분되는 단계를 순차적으로 통과한다. 이 구조 때문에 시스템 지연시간, 자원 소비, 캐싱 동작이 복잡한 단계 간 상호작용(inter-stage interaction)에 의해 결정된다. 그러나 기존 범용 LLM 트레이스는 일반적으로 generation 단계에만 초점을 맞추므로 이러한 복합 효과(composite effect)를 특성화하지 못한다.

PDF p. 1, 2

지식베이스·쿼리 패턴 의존성과 요청 간 상관관계 누락

RAG 성능은 지식베이스와 쿼리 패턴에 강하게 의존한다. query similarity, embedding cache hit rate, retrieval latency 같은 워크로드 동역학은 실제 사용자 행동에 따라 크게 변동하며, 실무 환경에서 연속된 쿼리들은 유의미한 요청 간 문맥 의존성(inter-request contextual dependency)을 보인다. 이러한 동역학과 요청 간 상관관계를 포착하는 것은 KV cache 재사용, batching, scheduling 전략 최적화에 필수적이지만 범용 트레이스에는 담겨 있지 않다.

PDF p. 2

검색-생성 co-optimization의 차단

검색과 생성을 독립적인 블랙박스로 취급하는 전통적 접근은 시스템 설계의 잠재력을 제한한다. RAG 트레이스 데이터가 있어야 dynamic retrieval caching이나 query-adaptive batching 같은 두 컴포넌트의 협력 최적화(co-optimization)를 설계·평가할 수 있다는 것이 저자들의 주장이다.

PDF p. 2

합성(synthetic)·일반 채팅 기반 트레이스의 비대표성

기존의 RAG 트레이스 데이터는 흔히 synthetic하거나 일반적인 채팅 워크로드에서 파생된다. 비대표적(non-representative) 트레이스로 최적화 전략을 평가하면 연구 결과와 실제 시스템 성능이 단절되며, 이는 RAG 서빙 시스템의 효율성·안정성·신뢰성에 심각한 문제를 야기한다.

핵심 contribution

논문 주장

1) 실세계 RAG 워크로드 트레이스 공개

2024년 4월부터 40,000명 이상의 학생·교직원에게 서비스해 온 대학 전체 정책 Q&A RAG 시스템에서 수집한 오픈소스 실세계 RAG 워크로드 트레이스 RAGPulse를 소개하고 공개한다(코드: https://github.com/flashserve/RAGPulse). 현재 공개분은 1주일 운영 기간에서 샘플링한 7,106건의 요청 레코드다.

PDF p. 1, 2, 6

2) 데이터 소스 아키텍처와 프라이버시 보호 포맷 상세화

트레이스가 생성된 Twen 시스템의 아키텍처(오프라인 지식베이스 구축 + 온라인 Agent LLM 처리)를 상세히 기술하고, 원본 텍스트를 모두 제거해 remapped hash_ids로 대체하는 프라이버시 보호 해시 기반 데이터 포맷을 정의한다.

PDF p. 1, 3, 5

3) 네 가지 워크로드 특성의 통계 분석

트레이스에 대한 심층 통계 분석을 통해 (a) 주기적 부하 변동, (b) 동적 입력 구성, (c) 텍스트(문서 청크) 사용 빈도의 long-tail한 hot document 접근 패턴, (d) 요청 간 문서 중복으로 나타나는 강한 시간적 지역성이라는 네 가지 핵심 특성을 규명한다.

PDF p. 3, 4, 6

4) 벤치마킹 플랫폼 구축

데이터셋의 실효성을 검증하기 위해 vLLM 기반 프로토타입 RAG 서비스와, 클라이언트-서버 구조의 워크로드 리플레이 벤치마킹 플랫폼(TTFT/TPOT 측정 포함)을 구축한다.

PDF p. 3, 4

5) 최적화 연구 방향 제시

분석 결과가 content-aware batching과 retrieval caching 같은 RAG 시스템 최적화 연구에 실증적 근거(empirical support)와 연구 방향을 제공한다고 제시한다.

제안 방식의 전체 구조와 주요 구성요소

논문 주장

전체 구조 개요

RAGPulse는 알고리즘이 아니라 '데이터셋 + 이를 재생할 수 있는 벤치마크 인프라'라는 형태의 산출물이다. 구조는 크게 세 층으로 나뉜다. (1) 데이터 생산 층: 실제 운영 시스템 Twen이 오프라인 단계에서 벡터 지식베이스를 구축하고, 온라인 단계에서 사용자 요청을 처리하며 Langfuse를 통해 트레이스를 기록한다. (2) 데이터 층: 기록된 레코드가 해시 기반으로 익명화되어 RAGPulse 트레이스 파일이 된다. (3) 데이터 소비 층: 연구자가 클라이언트-서버 리플레이 메커니즘으로 트레이스를 재생하여 vLLM 기반 서버의 TTFT/TPOT를 측정한다.

PDF p. 1, 2, 3, 4, 5

RAGPulse 트레이스 레코드 포맷

각 레코드는 하나의 RAG 요청에 대한 시스템 레벨 런타임 정보를 담는다. 필드는 timestamp(트레이스 시작 시각 12:00:00 기준 초 단위 제출 시각), input_length(system prompt·검색 문서·대화 이력 등 모든 구성요소 토큰 길이의 합), output_length(모델 생성 출력 토큰 길이), hash_ids(sys_prompt, passages_ids, history, web_search, user_input 각각에 대한 해시 식별자 집합), session_id(요청이 속한 대화 식별자)로 구성된다. 즉 입력이 '무엇으로 이루어져 있는지'를 컴포넌트 단위로 분해해 기록하는 점이 범용 트레이스와의 핵심 차이다.

PDF p. 2, 3

Twen 시스템 아키텍처 (데이터 소스)

Twen의 핵심 설계 철학은 검색과 생성의 분리(separation of retrieval and generation)이며, 이를 위해 Python 기반 마이크로서비스 아키텍처(FastAPI, Docker)를 채택해 CPU 집약적 검색 작업과 GPU 집약적 생성 작업을 분리한다. 구성요소는 Generator Model(Qwen3-235B-A22B-Instruct-2507), Embedding Model(로컬 배포된 infgrad/stella-large-zh-v3-1792d), Orchestration(LangChain), Vector Database(Qdrant, 태그 기반 구조로 global/tagged/user-personal 데이터베이스로 조직화), Document Maintenance(Python 프로그램과 LLM을 이용한 내용 추출·형식 표준화·자동 분할)이다.

PDF p. 4, 5

Offline Stage: 벡터 지식베이스 구축

오프라인 단계의 핵심 과제는 고품질 벡터 지식베이스 구축이다. 대학 정책·정보 코퍼스를 (1) 교수·부서장의 수동 제출, (2) 인가된 관리자를 위한 업로드 시스템, (3) 공개 부서 웹사이트를 주기적으로 수집하는 대상 웹 크롤러의 세 채널로 확보한 뒤, 저장 형식에 따라 Python 추출 경로와 LLM 기반 OCR 추출 경로로 나누어 처리하고, 전용 정규화 에이전트와 전용 LLM 청킹을 거쳐 벡터 임베딩을 생성한다.

PDF p. 5

Online Stage / Agent LLM

온라인 단계에서는 중앙 LLM(Agent LLM, Qwen3-235B)이 자율적 컨트롤러(autonomous controller)로 동작한다. 요청이 도착하면 사용자 정보와 프롬프트를 바탕으로 최적 처리 전략을 스스로 계획하고, 멀티모달(이미지·오디오·텍스트) 관점에서 질의를 해석하며, 웹 검색·RAG·사전 정의된 외부 도구(API 호출, 프로그래밍 등)를 동적으로 오케스트레이션한다. 또한 능동적 self-reflection을 지원해, 자신의 응답이 불충분하다고 판단하면 multi-hop 방식으로 도구를 반복 사용한다.

PDF p. 5

프라이버시 보호 해시 기반 포맷

사용자 프라이버시를 엄격히 보호하기 위해 원본 텍스트 콘텐츠 전부를 제거하고 remapped hash_ids로 대체한다. 이로써 완전한 익명화를 보장하면서도 연구에 필요한 구조적·시간적 특성은 모두 보존되며, 시스템 성능 및 스케줄링 정책 분석에 적합한 형태가 된다.

PDF p. 3

벤치마킹 플랫폼과 텍스트 재구성(Decryption) 전략

프로토타입 온라인 RAG 서비스는 vLLM을 코어로 하며, Generator로 Qwen2.5-14B를 단일 NVIDIA A800(80GB VRAM) 서버에 배포해 요청 수신·텍스트 처리·결과 생성의 종단 간 파이프라인을 구성한다. 텍스트가 해시로 대체되어 있으므로 KV cache처럼 구체적 텍스트에 의존하는 컴포넌트를 평가하려면 텍스트 재구성 전략이 필요하다. 그 원칙은 고유한 hash ID마다 원본과 토큰 길이가 엄밀히 동일한 고유·랜덤 콘텐츠 placeholder 텍스트를 생성하고, 동일 hash ID는 모든 요청에서 항상 동일한 placeholder로 매핑되도록 하여 시뮬레이션의 일관성과 타당성을 보장하는 것이다.

PDF p. 4

클라이언트-서버 리플레이 메커니즘

서버 측은 vLLM 기반으로 핵심 추론을 담당하며, OpenAI 호환 API로 요청을 받고 streaming/non-streaming 생성 모드를 모두 지원한다(TTFT·TPOT 측정에 필수). 고부하에서도 안정성과 처리량을 유지하기 위해 비동기·동시성 설계를 채택한다. 클라이언트 측은 부하 생성기로서 Data Preprocessing → Request Scheduling(time-scaling factor로 재생 속도 조절 가능) → Metric Collection의 3단계 워크플로를 수행한다.

단계별 동작 과정

논문 주장

1단계 — 문서 수집 (오프라인)

대학 정책 문서를 수동 제출, 인가 관리자용 업로드 시스템, 대상 웹 크롤러의 세 가지 경로로 수집한다.

PDF p. 5

2단계 — 텍스트 추출과 정규화

수집된 문서를 저장 형식별로 구분한다. TXT류는 Python 프로그램으로 직접 내용을 추출하고, JPG/PDF처럼 내용이 명시적으로 저장되지 않은 파일은 LLM 기반 OCR로 추출한다. 추출된 원본 콘텐츠는 포맷이 지저분한 경우가 많으므로, 전용 정규화(normalization) 에이전트가 이를 표준화한다.

PDF p. 5

3단계 — 청킹과 벡터화

정규화된 텍스트를 또 다른 전용 LLM으로 의미적 무결성(semantic integrity)을 보장하도록 지능적으로 분할(chunking)한다. 비용 효율을 위해 하위 작업별로 7B~70B 범위의 다양한 LLM을 사용한다. 생성된 고품질 청크를 임베딩해 Qdrant 벡터 데이터베이스(global/tagged/user-personal)에 저장한다.

PDF p. 5

4단계 — 요청 수신과 전략 수립 (온라인)

사용자 요청이 Web Service(Nginx, Python APIs)를 통해 도착하면 Agent LLM(Qwen3-235B)이 사용자 정보와 프롬프트를 근거로 최적 처리 전략을 자율적으로 계획하고, 멀티모달 입력(이미지/오디오/텍스트)을 해석한다.

PDF p. 4, 5

5단계 — 도구 오케스트레이션과 multi-hop self-reflection

Agent LLM이 웹 검색, RAG 검색, 사전 정의된 외부 도구(API 호출, 코드 실행 등)를 동적으로 오케스트레이션한다. 이어서 자신의 응답을 스스로 평가해 현재 답변이 불충분하다고 판단되면 multi-hop 방식으로 여러 도구를 반복 사용하여 최종 응답에 도달한다. 이 과정에서 최종 프롬프트(Final Prompt)가 구성된다.

PDF p. 5

6단계 — 트레이스 레코드 생성

LangChain의 AsyncIteratorCallbackHandler로 Qwen3-235B를 비동기 스트리밍 호출하며, 이때 Langfuse CallbackHandler가 활성화되어 지연시간·토큰 길이·user_id·session_id·검색된 hash_ids를 포함한 전체 호출 메타데이터를 캡처한다. 캡처된 데이터는 Langfuse 데이터베이스로 전송되어 RAGPulse 트레이스 데이터셋을 형성하며, 원본 텍스트는 해시 ID로 익명화된다.

PDF p. 3, 5

7단계 — 리플레이를 위한 텍스트 재구성 (데이터셋 사용)

벤치마크 클라이언트가 트레이스 파일을 읽고, 각 hash ID의 원본 토큰 길이와 엄밀히 일치하는 랜덤 토큰 시퀀스를 생성해 텍스트 입력을 재구성한다(Data Preprocessing).

PDF p. 4

8단계 — 요청 재생과 지표 수집

Request Scheduling 모듈이 트레이스 파일의 원본 타임스탬프를 엄격히 따르며(필요 시 time-scaling factor로 가속·감속) vLLM 기반 서버로 요청을 디스패치하고, Metric Collection 모듈이 요청별 TTFT/TPOT 성능 데이터를 기록해 이후 분석을 위해 영구 저장한다.

PDF p. 4

핵심 메커니즘과 데이터 구조

논문 주장

해시 기반 익명화가 만드는 '구조 보존 트레이스'

이 논문에는 전통적 의미의 알고리즘이나 새로운 인덱스 구조가 없다. 대신 핵심 기술 장치는 hash_ids 기반의 레코드 표현이다. 요청의 각 입력 구성요소(system prompt, 검색된 문서들, 대화 이력, 웹 검색 결과, 사용자 질문)를 서로 다른 해시 ID 집합으로 표현함으로써, 텍스트 내용은 전혀 노출하지 않으면서도 '어떤 청크가 언제 어떤 요청에서 재사용되었는가'라는 관계 정보를 완전히 보존한다. hot document 분석과 요청 간 문서 중복 분석은 모두 이 hash_ids 필드의 재사용 패턴을 집계하여 도출된다.

PDF p. 2, 3, 4

길이 보존 placeholder 매핑 (텍스트 복원 규칙)

KV cache처럼 구체적 텍스트에 의존하는 컴포넌트를 평가하려면 텍스트가 필요하므로, 논문은 두 가지 불변조건(invariant)을 갖는 복원 규칙을 제시한다. (1) 각 고유 hash ID에 대해 원본과 토큰 길이가 엄밀히 동일한 랜덤 콘텐츠 placeholder를 생성한다. (2) 동일한 hash ID는 모든 요청에서 반드시 같은 placeholder 텍스트로 매핑되어야 한다. 두 번째 조건이 있어야 서로 다른 요청이 같은 청크를 참조할 때 prefix/청크 단위 캐시 재사용이 실제와 동일하게 재현되어 시뮬레이션의 일관성과 타당성이 보장된다.

PDF p. 4

임의 스케일 재생 (Scaling RAGPulse to Any Scale)

트레이스가 상당한 시간적 범위(temporal span)를 갖는다는 성질을 이용해, 연구자는 실험 목표에 따라 트레이스를 잘라내거나 스케일링할 수 있다. 예컨대 고트래픽 구간(1시간 등)만 분리해 피크 동시 처리 능력을 벤치마킹하거나, 전체 장기 트레이스로 KV cache 관리 같은 캐싱 정책의 장기 효과를 평가할 수 있다. 여기에 재생 시 time-scaling factor를 적용해 부하 강도를 조절한다.

PDF p. 4

실험 구성

논문 주장

분석 대상 워크로드

분석 대상은 RAGPulse 트레이스 자체다. Twen 대학 전체 정책 Q&A RAG 시스템(2024년 4월부터 40,000명 이상 사용자에게 서비스)의 1주일 운영 기간에서 샘플링한 7,106건의 요청 레코드로 구성된다. 논문에는 비교 대상 baseline 시스템이나 baseline 트레이스와의 정량 비교 실험이 제시되지 않는다.

PDF p. 2

벤치마킹 하드웨어·소프트웨어 구성

모든 실험은 단일 NVIDIA A800 GPU(80GB VRAM)를 갖춘 서버에서 수행되었다. 추론 프레임워크는 vLLM이며, RAG 파이프라인의 Generator로 Qwen2.5-14B를 배포했다. 이는 트레이스를 생성한 실제 운영 시스템 Twen의 Generator(Qwen3-235B-A22B-Instruct-2507)와는 다른 모델임에 유의해야 한다.

PDF p. 4, 5

측정 지표

서빙 성능 지표로는 LLM 서빙 분야에서 널리 인정되는 두 가지를 채택한다. TTFT(Time To First Token)는 요청 수신부터 첫 출력 토큰 반환까지의 총 경과 시간으로 시스템 응답성과 사용자 체감 지연의 핵심 벤치마크이고, TPOT(Time Per Output Token)는 첫 토큰 이후 각 토큰 생성에 걸리는 평균 시간으로 생성 단계의 처리량과 처리 효율을 반영한다. 데이터셋 특성 분석에는 입력/출력 토큰 길이 CDF, QPS 및 초당 생성 토큰 수(Tokens/s) 기반 처리량 시계열, 입력 길이 구간별 평균 입력 컴포넌트 비율, Passage 접근 빈도 CDF, 요청 간 문서 중복률(Document Overlap Ratio)이 사용된다.

PDF p. 4

주의: 실측 성능 결과의 부재

본문 §4.4~4.5는 TTFT/TPOT의 정의와 측정 방법론(클라이언트-서버 리플레이)을 제시하지만, 실제로 측정된 TTFT/TPOT 수치는 논문에 보고되지 않는다. 따라서 이 논문의 '실험' 부분은 시스템 성능 평가라기보다 데이터셋 특성 분석과 재생 인프라의 기능 설명에 해당한다.

논문 주장

논문의 결론

RAG는 신뢰할 수 있고 고성능인 LLM 애플리케이션 구축의 표준 패러다임이 되었지만, 검색·재순위·생성으로 이루어진 복합 아키텍처는 시스템 최적화에 전례 없는 도전을 제기한다. 이 논문의 중심 테제는 기존 범용 LLM 트레이스가 RAG 고유의 워크로드 동역학을 포착하지 못해 학계 연구와 실제 배포 사이에 심각한 성능 격차를 만든다는 것이다. 저자들은 이 격차를 메우기 위해 RAGPulse를 설계·공개하고, 심층 분석을 통해 고도로 편향된 hot document 접근 패턴과 유의미한 요청 간 시간적 지역성이라는 실세계 RAG 워크로드의 결정적 특성을 정량화했으며, 이것이 content-aware batching과 retrieval caching 같은 최적화에 명확한 실증적 근거와 연구 방향을 제공한다고 결론짓는다.

논문의 강점

  • 논문 주장 합성 데이터가 아니라 2024년 4월부터 40,000명 이상에게 실제로 서비스 중인 운영 시스템에서 수집된 트레이스라는 점에서 생태 타당성(ecological validity)이 높다.
    PDF p. 1, 2, 5
  • 분석자 해석 입력을 sys_prompt / passages_ids / history / web_search / user_input으로 분해해 기록하는 스키마 설계가 이 논문의 실질적 핵심이다. 총 입력 길이만 기록하는 범용 트레이스로는 불가능한 '동적 입력 구성' 분석과 캐시 재사용 잠재력 추정이 이 분해 덕분에 가능해진다.
    PDF p. 2, 3
  • 분석자 해석 프라이버시 보호와 연구 유용성 사이의 상충을 해시 익명화 + 길이 보존 placeholder 매핑 규칙으로 실용적으로 절충했다. 특히 '동일 hash ID는 항상 동일 placeholder'라는 불변조건은 prefix/청크 캐시 재사용 실험의 재현성을 보장하는 핵심 장치다.
    PDF p. 3, 4
  • 분석자 해석 데이터셋만 던지지 않고 vLLM 기반 서버와 클라이언트 리플레이 도구, time-scaling factor, TTFT/TPOT 수집기를 함께 공개해 후속 연구자의 진입 비용을 낮췄다.
    PDF p. 4
  • 분석자 해석 분석에서 도출한 각 특성이 곧바로 특정 시스템 최적화 기법(long-tail → retrieval caching, 요청 간 중복 → content-aware batching 및 KV cache 재사용)으로 연결되어, 관찰과 설계 함의 사이의 논리가 명확하다.

핵심 용어

RAG (Retrieval-Augmented Generation)
LLM의 추론·생성 능력과 외부 지식베이스의 실시간 사실성을 결합하는 패러다임. Retrieve(임베딩 모델이 질의를 벡터화해 벡터 DB에서 가장 의미적으로 관련된 지식 청크 검색), Augment(검색된 청크를 원 질의·시스템 프롬프트와 context로 결합), Generate(증강 프롬프트를 LLM에 제출해 사실에 근거한 출력 생성)의 다단계 파이프라인으로 동작한다. PDF p. 1
Workload Trace
실제 서비스에서 발생한 요청들의 도착 시각과 특성을 기록한 데이터. 시스템 연구자가 실제 부하를 재현(replay)해 스케줄링·캐싱 정책을 평가하는 데 사용된다. PDF p. 1, 2
hash_ids
RAGPulse 레코드에서 요청 입력의 각 구성요소(system prompt, 검색된 문서들, 대화 이력, 웹 검색 결과, 사용자 질문)를 나타내는 해시 식별자 집합. 원본 텍스트를 제거하고 이 ID로 대체함으로써 익명화하면서도 청크 재사용 관계를 보존한다. PDF p. 3
Hot Document / Long-tail Effect
소수의 핵심 지식 청크가 다수의 서로 다른 요청에 의해 빈번히 검색되는 반면 대다수 청크는 훨씬 드물게 접근되는 고도로 편향된 사용 분포. Retrieval Caching의 유효성을 뒷받침하는 근거다. PDF p. 3, 4
Temporal Locality / Document Overlap
임의의 시간 윈도우 내에서 동시에 도착하는 요청 다수가 동일하거나 매우 유사한 지식 청크에 접근하는 성질. 요청 간 문맥 의존성을 의미하며 content-aware batching과 KV cache 재사용의 근거가 된다. PDF p. 3, 4
TTFT (Time To First Token)
시스템이 요청을 수신한 순간부터 첫 출력 토큰이 생성·반환될 때까지의 총 경과 시간. 시스템 응답성과 사용자 체감 지연의 핵심 벤치마크다. PDF p. 4
TPOT (Time Per Output Token)
첫 토큰이 생성된 이후 후속 토큰 각각을 생성하는 데 걸리는 평균 시간. 생성 단계의 처리량과 처리 효율을 주로 반영한다. PDF p. 4
Content-aware Batching
동일한 문서(지식 청크)에 접근하는 요청들을 함께 배치(batch)로 묶어 처리하는 최적화 전략. 요청 간 문서 중복이 높을 때 이득이 크다. PDF p. 4, 6
Retrieval Caching
벡터 데이터베이스에서 검색되는 콘텐츠 청크를 캐싱하는 최적화 방향. hot document의 long-tail 접근 분포가 그 가치를 뒷받침한다. PDF p. 4, 6
Time-scaling Factor
벤치마크 클라이언트가 트레이스의 원본 타임스탬프를 따라 요청을 재생할 때 재생 속도를 가속·감속하도록 도입한 계수. 부하 강도를 조절해 유연한 실험을 가능하게 한다. PDF p. 4
Text Decryption (텍스트 재구성)
해시로 대체된 입력을 시뮬레이션에 쓰기 위해 각 고유 hash ID마다 원본과 토큰 길이가 엄밀히 동일한 랜덤 placeholder 텍스트를 생성하고, 동일 ID는 항상 동일 텍스트로 매핑하도록 하는 전략. PDF p. 4
Agent LLM
Twen 온라인 단계의 중앙 LLM(Qwen3-235B)으로, 자율적 컨트롤러로서 처리 전략을 계획하고 멀티모달 질의를 해석하며 웹 검색·RAG·외부 도구를 동적으로 오케스트레이션하고, 응답이 불충분하면 multi-hop self-reflection을 수행한다. PDF p. 5
vLLM
PagedAttention 기반의 널리 채택된 오픈소스 LLM 추론 프레임워크. RAGPulse 벤치마킹 플랫폼의 서버 측 코어로 사용된다. PDF p. 4
Langfuse
LLM 호출 메타데이터(지연시간, 토큰 길이, user_id, session_id, hash_ids 등)를 캡처하는 관측(observability) 도구. Twen 온라인 호출 중 CallbackHandler로 활성화되어 RAGPulse 트레이스를 형성한다. PDF p. 5

  • 트레이스에 포함된 요청의 언어 구성 및 세션당 평균 요청 수 등 세션 수준 통계 (본문에 제시되지 않음)
