TileMaxSim: IO-Aware GPU MaxSim Scoring with Dimension Tiling and Fused Product Quantization

핵심 요약

TileMaxSim은 ColBERT 계열 multi-vector retrieval의 핵심 연산인 MaxSim scoring을 GPU에서 IO-aware하게 재구현한 Triton kernel 계열이다. 저자는 현대 GPU에서 MaxSim에 대한 roofline analysis를 수행하여, naive 구현이 N_q×N_d 크기의 similarity matrix를 HBM에 명시적으로 materialize하기 때문에 peak HBM bandwidth의 5~18%만 사용한다는 점을 문제의 근본 원인으로 지목한다(pp.1-2). 해결책은 FlashAttention이 attention에 적용한 IO-aware paradigm을 MaxSim 고유의 matmul→max→sum reduction 구조에 맞게 재설계하는 것이다. 구체적으로 (1) query tile과 document tile을 동시에 tiling하여 BQ=N_q로 설정함으로써 각 document embedding을 HBM에서 정확히 한 번만 읽는 multi-query SRAM tiling(V2-MQ), (2) embedding dimension을 128-wide chunk로 분할해 d=64~768을 하나의 kernel family로 처리하는 dimension tiling, (3) shared-memory lookup table로 PQ 압축 문서를 명시적 decompression 없이 채점하는 fused PQ decompression+scoring(TileMaxSim-PQ)의 세 기법을 제시한다(pp.2-5). 실험(H100)에서 V2-MQ는 N_q=32, N_d=128, B=100K 조건에서 peak HBM bandwidth의 80.2%를 달성하고 PyTorch Loop 대비 220배, PyTorch Naive 대비 6.5배, PLAID의 실제 GPU scoring 경로 대비 1.7~1.9배 빠르며, 동일 노드에서 측정한 WARP(CPU) 대비 scoring throughput이 469배 높다(pp.6-10). 동시에 MS MARCO와 세 개 BEIR 데이터셋에서 reference PyTorch MaxSim과 동일한 ranking·retrieval metric을 산출하여 exact quality를 보존하며, PLAID의 scoring kernel을 drop-in 교체할 경우 100K candidates 기준 end-to-end pipeline latency가 약 98% 감소한다(pp.9-10).

해결하려는 문제

논문 주장

MaxSim scoring이 GPU 하드웨어를 활용하지 못한다

ColBERT와 같은 multi-vector retrieval 모델은 질의 토큰 임베딩 집합과 문서 토큰 임베딩 집합 사이의 MaxSim scoring(각 query token에 대해 document token들과의 유사도 최댓값을 구한 뒤 query token에 대해 합산)을 수행한다. 저자는 현대 GPU에서 MaxSim에 대한 roofline analysis를 수행한 결과, naive 구현들이 peak HBM bandwidth의 5~18%만 달성한다고 보고한다. 근본 원인은 N_q×N_d 크기의 similarity matrix S를 HBM에 명시적으로 materialize한다는 점이다. S는 한 번 max-reduction에 쓰이고 곧바로 버려지는 중간 데이터인데, 이를 HBM에 쓰고 다시 읽는 메모리 트래픽이 전체 성능을 지배한다.

PDF p. 1

대규모 검색에서 scoring 비용이 latency를 지배한다

질의 토큰 수 N_q, 문서 토큰 수 N_d, 임베딩 차원 d에 대해 naive MaxSim은 O(N_q·N_d·d)의 곱셈-누산(multiply-accumulate) 연산을 요구한다. 질의당 수백만 개의 후보 문서를 채점해야 하는 대규모 검색 환경에서 이 비용은 전체 retrieval latency를 지배하게 되며, 특히 candidate generation 단계가 개선될수록 scoring이 차지하는 상대적 비중은 더 커진다.

PDF p. 1, 2

CPU 중심 scoring이라는 아키텍처 불일치

저자는 MaxSim이 본질적으로 GPU 병렬성에 잘 맞는 batched matrix operation임에도 불구하고 PLAID, WARP, GEM 등 주요 multi-vector 엔진들이 scoring을 CPU에 국한시키고 있다는 점을 아키텍처 불일치(architectural mismatch)로 규정한다. 근거로 현대 GPU가 CPU보다 30~50배 높은 메모리 대역폭(H100 3.35 TB/s vs. dual-channel DDR5 약 100 GB/s)을 제공한다는 점을 든다.

PDF p. 1, 2

기존 방식의 한계

논문 주장

PLAID: centroid 기반 근사와 PyTorch scoring

PLAID는 centroid pruning과 product quantization을 도입해 ColBERTv2 대비 최대 7배의 latency 감소를 달성했다. 그러나 CUDA kernel로 처리하는 부분은 centroid 기반 근사 decompression뿐이고 최종 scoring은 PyTorch에서 실행하며, bandwidth utilization은 보고되지 않았다. 또한 exact MaxSim이 아니라 centroid-approximate 방식이다.

PDF p. 1, 5

WARP: CPU에 국한된 SIMD 최적화

WARP는 SIMD 가속 C++ kernel로 CPU scoring을 최적화하여 XTR reference 구현 대비 41배, PLAID 대비 3배의 속도 향상을 달성했다. 하지만 implicit decompression을 위한 lookup table과 two-stage aggregation을 사용하는 CPU 중심 기여이며, GPU의 높은 메모리 대역폭은 활용하지 못한다.

PDF p. 1, 5

GEM: CPU OpenMP 기반 graph index

GEM은 인덱스 구성(construction)에 Earth Mover's Distance를, 검색(search)에 Chamfer distance를 사용하는 graph-based index를 구축하지만 CPU 기반 OpenMP 병렬화에 의존한다. 결국 PLAID, WARP, GEM 세 시스템 모두 MaxSim scoring을 CPU에 국한시키고 있다는 것이 저자의 공통된 지적이다.

PDF p. 1, 5

GPU 라이브러리는 single-vector 거리만 지원

NVIDIA cuVS(CAGRA, IVF-PQ, HNSW)와 FAISS 같은 GPU 가속 벡터 검색 라이브러리는 단일 벡터(single-vector) 거리 계산만 지원하며, 임베딩 집합에 대한 MaxSim의 matmul-max-sum reduction을 네이티브로 계산할 수 없다. 이들 라이브러리를 MaxSim에 적용하려면 결국 전체 N_q×N_d similarity matrix를 materialize해야 하는데, 이것이 바로 TileMaxSim이 없애려는 병목이다.

PDF p. 5, 10

EMVB: 빠른 candidate generation, 표준 scoring

EMVB는 bitvector prefiltering과 product quantization으로 ColBERT retrieval을 가속하고 메모리 footprint를 줄여 빠른 candidate generation을 가능하게 하지만, 최종 MaxSim scoring은 IO-aware하지 않은 표준적인 CPU 또는 GPU 연산에 위임한다.

PDF p. 5

핵심 contribution

논문 주장

(1) Bandwidth gap의 근본 원인 분석

현대 GPU에서 MaxSim에 대한 roofline analysis를 수행하여, naive 구현이 similarity matrix materialization 때문에 peak bandwidth의 17.6%만 달성함을 보이고, memory-bound ceiling에 도달하기 위해 필요한 IO 감소량이 2.0배임을 정량화한다(Section 2.3). 이는 최적화의 방향이 compute 증대가 아니라 memory traffic 감소임을 뒷받침한다.

PDF p. 2

(2) d>128을 위한 dimension tiling

embedding dimension을 128-wide chunk로 분할하는 tiling 전략을 제시하여 register/SRAM 용량을 초과하지 않으면서 하나의 kernel family가 d=64부터 d=768까지의 임베딩을 모두 처리하도록 한다. d=768에서 peak bandwidth의 86%를 달성한다(Section 3).

PDF p. 2

(3) Fused PQ decompression+scoring

TileMaxSim-PQ는 명시적 decompression 없이 shared-memory lookup table을 통해 PQ 압축 문서를 채점하며, decompress-then-score 파이프라인 대비 최대 약 31배의 HBM IO 감소를 달성한다(Section 4).

PDF p. 2, 5

(4) 80% peak bandwidth utilization

multi-query tiled kernel(V2-MQ)이 각 document embedding을 HBM에서 정확히 한 번만 읽음으로써 H100 peak HBM bandwidth의 80.2%를 달성한다. 저자는 비교 맥락으로, compute-bound kernel인 FlashAttention-3가 peak compute의 약 75%를 달성한다는 점을 언급하며 TileMaxSim은 memory-bound kernel로서 compute가 아니라 bandwidth를 목표로 한다고 설명한다(Section 6).

PDF p. 2

(5) Exact quality 보존과 포괄적 평가

MS MARCO와 세 개의 BEIR benchmark에서 reference MaxSim과 동일한 ranking을 유지하면서, loop 기반 scoring 대비 220배, WARP 대비 469배(동일 노드 기준)의 scoring throughput을 달성한다. 또한 near-linear한 multi-GPU scaling, 100K~500K 문서 구간에서 일정한 83M docs/s throughput, dimension(64~768)·precision·token count·domain 전반의 견고성을 보인다(Sections 6, 6.10).

PDF p. 2

(6) Drop-in 시스템 통합

ColBERTv2/PLAID의 scoring kernel을 직접 대체하는 실험을 통해, 100K candidates에서 동일한 retrieval quality를 유지하면서 end-to-end pipeline latency를 최대 98%까지 감소시킴을 보인다(Table 15).

제안 방식의 전체 구조

논문 주장

설계 원리: MaxSim에 맞춘 IO-aware kernel

TileMaxSim은 FlashAttention의 IO-aware paradigm(중간 행렬을 HBM에 쓰지 않고 SRAM에서 tiling하여 처리)을 따르되, MaxSim 고유의 matmul→max→sum reduction 구조에 맞춰 kernel을 새로 설계한다. 저자는 이 구조가 softmax 정규화가 필요한 attention과도 다르고, scatter-sum 방식의 sparse retrieval과도 다르다고 명시한다. 핵심은 similarity matrix S를 HBM에 절대 materialize하지 않고, query token별 running maximum을 register에 유지한 채 document embedding을 SRAM으로 streaming하는 것이다.

PDF p. 1, 2, 3

세 가지 축의 기법

전체 설계는 세 축으로 구성된다. 첫째, multi-query SRAM tiling은 query tile과 document tile을 동시에 tiling하여 BQ=N_q일 때 각 document embedding을 HBM에서 정확히 한 번만 읽는 IO-optimal 패턴을 만든다. 둘째, dimension tiling은 임베딩 차원을 128-wide chunk로 나눠 부분 dot product를 누산함으로써 d>128인 모델에서도 register/SRAM 용량 제약을 넘지 않게 한다. 셋째, fused PQ decompression+scoring은 PQ code를 decompress하지 않고 shared-memory lookup table 조회로 곧바로 유사도를 누산한다.

PDF p. 2, 3, 4, 5

Kernel variant의 진화 경로

논문은 V1(per-query-token) → V2(per-document) → V2-MQ(multi-query tiling)의 순서로 kernel을 제시하며, 각 단계에서 어떤 IO 비효율이 남는지를 드러낸다. V1은 document tile을 query token마다 반복 로드하여 최적 대비 N_q배의 document read가 발생하고, V2는 중간 token-max buffer를 없애고 sum-reduction을 fuse하지만 여전히 document embedding을 N_q번 다시 읽는다. V2-MQ만이 single-pass IO-optimal 패턴을 달성한다.

PDF p. 3, 4

주요 구성요소와 단계별 동작 과정

논문 주장

구성요소 1 — TileMaxSim V1 (Per-Query-Token Kernel, Algorithm 1)

(batch, query-token) 쌍마다 하나의 thread block을 배정한다. 각 block은 하나의 query token을 register에 로드하고 BLOCK_Nd 크기의 document token tile들을 순회하며 SRAM에 로드해 dot product를 계산하고 running maximum을 갱신한다. 구조상 각 document tile이 query token마다 한 번씩 반복 로드되므로 최적 대비 N_q배 많은 document read가 발생한다.

PDF p. 3

구성요소 2 — TileMaxSim V2 (Per-Document Kernel, Algorithm 2)

문서 하나당 하나의 program을 배정한다. 각 program은 N_q개 query token 전체를 순회하며 document token tile들에 대해 running max를 갱신한 뒤 합산한다. 이로써 V1의 중간 token-max buffer를 제거하고 sum-reduction을 kernel 내부로 fuse하지만, 각 document embedding을 여전히 N_q번 다시 읽는다.

PDF p. 3, 4

구성요소 3 — TileMaxSim V2-MQ (Multi-Query Tiling, 최적 variant, Algorithm 3 / Figure 1)

query token과 document token을 동시에 tiling하는 최고 성능 variant이다. BQ×d 크기의 query tile을 한 번만 register에 로드하고, BN×d 크기의 document tile들을 SRAM으로 streaming하며 tl.dot(Tensor Core)로 부분 similarity S_t=Qtile·D_t^T를 계산한 뒤 BQ개의 running maxima로 reduction하고 atomic_add로 scores[b]에 누적한다. BQ=N_q로 설정하면 각 document embedding이 HBM에서 정확히 한 번만 읽히는 IO-optimal 패턴이 된다.

PDF p. 3, 4

구성요소 4 — TileMaxSim-PQ (Fused decompression+scoring)

Phase 1(Table Construction)에서 (N_q, M) 크기의 program grid가 query token과 sub-quantizer별 per-query distance(lookup) table T[i,m,k]를 미리 계산해 SRAM/L2에 상주시킨다. Phase 2(Fused Lookup+Max+Sum)에서 각 program이 (batch, query-token) 쌍을 담당하여 document token마다 M개의 1-byte PQ code를 로드해 table lookup으로 유사도를 근사 누산하고 running maximum을 갱신한다. decompressed vector는 HBM에 전혀 기록되지 않는다.

PDF p. 4, 5

동작 단계 1~2 — Grid 실행과 query tile 로드

Grid를 (B, ⌈N_q/BQ⌉)개의 program으로 실행한다. 각 program은 (batch 내 document, query-token tile) 조합 하나를 담당한다. 각 program은 자신의 BQ×d query tile(Qtile)을 HBM에서 register로 한 번만 로드하고, BQ개의 running maxima를 −∞로 초기화한다.

PDF p. 3, 4

동작 단계 3~4 — Document tile streaming과 running max 갱신

document token tile마다(총 ⌈N_d/BN⌉개) BN×d 크기의 document tile D_t를 HBM에서 SRAM으로 로드한다. 이어서 tl.dot(Tensor Core 연산)으로 부분 similarity S_t=Qtile·D_t^T(BQ×BN)를 계산하고, m ← max(m, rowmax(S_t))로 register 내 running maxima를 갱신한다. 이 S_t는 SRAM/register에만 존재하며 HBM에 기록되지 않는다.

PDF p. 3, 4

동작 단계 5 — 최종 score 누적

모든 document tile을 처리한 뒤, BQ개의 running maxima의 합을 atomic_add로 scores[b]에 HBM에 누적한다. 이 전체 과정에서 similarity matrix나 token-max buffer 같은 중간 텐서는 전혀 HBM에 다시 기록되지 않는다.

PDF p. 3, 4

동작 단계 6~7 — PQ 경로(Phase 1 / Phase 2)

PQ로 압축된 문서를 다룰 때는 Phase 1에서 (N_q, M) program grid가 각각 d_sub 차원의 K개 dot product를 계산하여 distance table T[i,m,k]=q_i[sub-vector m]^⊤·C[m,k]를 구축한다. 이 table은 H100의 50MB L2 cache에 들어갈 만큼 작아 해당 query의 scoring 단계 동안 L2에 상주한다. Phase 2에서는 각 program이 (batch, query-token) 쌍 하나를 맡아 document token tile마다 M개의 1-byte PQ code를 로드해 table lookup으로 유사도를 누산하고 running maximum을 갱신한다. 모든 문서가 동일한 table을 공유하므로 table access의 locality가 높다.

PDF p. 4, 5

핵심 알고리즘과 IO 분석

논문 주장

IO-optimal 조건: BQ = N_q

V2-MQ의 핵심 알고리즘적 통찰은 query tile 크기 BQ를 N_q와 같게 잡는 것이다. 이렇게 하면 하나의 program이 모든 query token을 한 번의 pass에서 처리하므로, document embedding을 다시 읽을 이유가 사라진다. 논문의 tile size ablation(Table 12)은 BQ=32(=N_q)로 설정하는 것이 가장 큰 효과를 내는 최적화이며 BQ=16 대비 4.5배 높은 bandwidth를 달성한다고 보고한다. BQ=32 계열 내에서는 BN=32가 최적으로 1,302 GB/s, 39.7M docs/s를 기록한다.

PDF p. 3, 4, 9

Arithmetic intensity와 roofline

논문의 Arithmetic Intensity Comparison(Section 3)에 따르면, TileMaxSim은 naive 구현 대비 HBM IO를 2.0배 줄이며 arithmetic intensity를 16.1에서 32.0 FLOP/byte로 높인다. 이는 이론적으로 2.0배의 speedup에 해당하고, N_q=64로 늘어나면 IO 감소량은 3.0배로 증가한다. 모든 MaxSim variant는 14~32 FLOP/byte 구간에 있어 H100의 crossover point인 591 FLOP/byte에 훨씬 못 미치므로 깊게 memory-bound이며, 따라서 최적화 방향이 memory traffic 감소여야 한다는 것이 저자의 논지이다.

PDF p. 3, 7

Dimension tiling 알고리즘

d>128인 경우 V2-MQ는 자동으로 dimension-tiled variant로 전환하여 embedding dimension을 128 크기의 chunk로 tiling한다. 각 chunk에 대한 부분 dot product를 누산한 뒤에야 max를 계산하므로, max reduction의 의미론이 보존되면서도 register/SRAM 용량 제약을 넘지 않는다. 실측에서 bandwidth utilization은 d가 커질수록 오히려 증가하는데(d=64의 44%에서 d=768의 86%), 저자는 큰 임베딩이 더 나은 memory coalescing을 제공하기 때문이라고 설명한다.

PDF p. 8

PQ fused kernel의 IO 이득

TileMaxSim-PQ는 decompress-then-score 파이프라인 대비 HBM IO를 최대 약 31.0배 줄인다(218,124,288 bytes vs. 6,758,400,000 bytes, B=100K, N_q=32, N_d=128, M=16, K=256 기준). 각 document token이 M바이트의 code로만 표현되므로 arithmetic intensity는 오히려 더 낮아지지만(batch size에 따라 5~31 FLOP/byte), decompressed vector의 HBM read를 shared-memory table lookup으로 대체하므로 memory-bound IO가 빠른 SRAM access로 전환된다.

Bandwidth utilization과 memory-bound 검증

V2-MQ(BQ=32, single-pass)는 B=100K에서 H100 peak HBM bandwidth의 80.2%(3,350 GB/s 중 2,687 GB/s)를 달성한다(Table 6). 전용 profiling run에서는 2,660 GB/s(peak 대비 79.4%)와 85.1 TFLOP/s의 achieved compute를 기록하는데, 이는 peak FP16 Tensor Core(1,979 TFLOP/s)의 4.3%에 불과하여 kernel이 완전히 memory-bound임을 확인시켜 준다(Table 7). torch.profiler와 nsys 프로파일링에서 V2-MQ Triton kernel이 전체 CUDA time의 99.7%를 차지하고 launch당 평균 589μs가 소요되며, scoring 중 device-to-host memory copy가 전혀 없고 cudaLaunchKernel 오버헤드는 call당 3.7μs(kernel time의 0.6%)이다.

Scaling 특성

throughput은 B≥100K에서 약 83M docs/s로 정체(plateau)되며 B=200K에서 peak bandwidth의 81.5%에 도달하고, 단일 H100에서 B=500K(16.4GB 임베딩)까지 scaling이 검증되었다(Table 8). embedding dimension을 64~768로 sweep하면 d=64에서 90.5M docs/s(peak의 44%), d=128에서 60.9M docs/s(60%), d=768에서 14.7M docs/s(86%)를 달성한다(Table 9). N_q를 8→64로 늘려도 BQ=N_q 설정 덕분에 throughput 감소가 완만하며(67.0M→35.5M docs/s), N_d를 32→512로 늘리면 throughput은 103.1M→22.0M docs/s로 감소하지만 achieved bandwidth는 845→2,881 GB/s로 오히려 증가한다(Tables 10, 11).

PQ scoring

TileMaxSim-PQ는 중간 batch size에서 PyTorch decompress-then-score 대비 1.9~2.9배의 speedup을 달성하며(N_d=64, B=10K에서는 1.0배), baseline이 OOM을 일으키는 B=100K까지 확장 가능하다(M=16 sub-quantizer, K=256 centroid, d_sub=8 기준). 저자는 speedup이 full-precision MaxSim만큼 크지 않은 이유를 PQ scoring이 문서당 데이터량이 작아 덜 memory-bound(낮은 arithmetic intensity)이기 때문이라고 설명한다(Table 5).

논문 주장

핵심 결론

저자의 결론은 MaxSim scoring이 CPU에 머물러야 할 본질적 이유가 없으며, similarity matrix materialization을 제거하는 IO-aware kernel 설계만으로 GPU peak bandwidth의 80% 수준을 달성할 수 있다는 것이다. TileMaxSim은 exact MaxSim을 계산하므로 어떤 근사도 도입하지 않고 기존 시스템의 scoring kernel을 그대로 대체할 수 있는 drop-in component이며, 83M docs/s 기준으로 100K candidates를 밀리초 단위에 채점할 수 있다.

대규모 외삽 추정

83M docs/s의 실측 throughput을 기준으로 외삽하면, 전체 8.8M-passage MS MARCO corpus에 대한 brute-force FP16 scoring은 단일 H100에서 약 107ms, 4×H100에 data-parallel로 분산할 경우 약 27ms가 소요된다. 다만 이 수치는 직접 측정치가 아니라 저자의 외삽 추정치임을 논문이 명시한다.

scoring과 torch.topk를 모두 GPU에서 수행하는 완전한 GPU-resident retrieval pipeline에서 500K-문서 corpus에 대한 end-to-end latency는 6.09ms이며(synthetic data 기준 82.0M docs/s, 실제 MS MARCO passage 기준 71.6M docs/s), top-k 선택이 추가하는 오버헤드는 0.1ms 미만이다. 이는 candidate generation 이후 단계 전체를 GPU에 두는 설계가 실현 가능함을 시사한다.

TileMaxSim V2-MQ 전체 동작 흐름

PDF p. 3, 4, 5

V2-MQ kernel의 실행 흐름. grid를 (B, ceil(N_q/BQ))로 launch하고, query tile을 register에 한 번 로드한 뒤 document tile을 SRAM으로 streaming하며 running maxima를 갱신한다. similarity matrix는 HBM에 기록되지 않으며, 최종 score만 atomic_add로 HBM에 누적된다. PQ 경로는 Phase 1의 lookup table 구축과 Phase 2의 fused lookup+max+sum으로 분기한다(pp.3-5).

기존 방식(naive/PLAID)과 TileMaxSim의 IO 경로 비교

PDF p. 1, 2, 3, 7

naive 구현은 전체 N_q×N_d similarity matrix를 HBM에 쓰고 다시 읽어 max/sum reduction을 수행하므로 중간 데이터에 메모리 트래픽을 낭비한다. TileMaxSim은 동일한 reduction을 SRAM/register 내부에서 fuse하여 document embedding을 HBM에서 한 번만 읽고 최종 score만 기록한다(pp.1-4).

  • 저자 명시 한계 TileMaxSim은 문서 임베딩이 GPU에 상주(GPU-resident)한다고 가정하며, 저자가 결론에서 이를 현재의 명시적 한계로 언급한다.

  • 저자 명시 한계 V2-MQ가 peak HBM bandwidth의 100%가 아닌 80.2%에 그치는 이유로 저자는 (1) N_q=32개의 동시 running maxima 유지에 따른 register pressure, (2) 2의 거듭제곱이 아닌(non-power-of-two) 차원이 vectorized load를 제한하는 점, (3) 최종 score 출력을 위한 atomic accumulation overhead를 제시한다.

  • 저자 명시 한계 V2-MQ kernel의 이론적 occupancy는 register 사용 제약으로 약 50%에 그친다. 다만 저자는 num_warps tuning으로 occupancy를 높여도 throughput이 개선되지 않았다는 점을 들어 kernel이 occupancy가 아니라 memory bandwidth에 의해 제한된다고 설명한다.
    PDF p. 8
  • 저자 명시 한계 저자는 TileMaxSim이 corpus가 GPU에 상주하고 배치 크기 B>1K일 때 가장 큰 이점이 있는 drop-in scoring kernel이라고 스스로 범위를 한정한다. 질의당 후보 문서가 매우 적을 때(약 64개)는 scoring 비용 자체가 미미하여 PLAID의 기존 kernel과 end-to-end latency 차이가 거의 없다고 보고한다.

핵심 용어

MaxSim (late interaction scoring)
ColBERT 계열 multi-vector retrieval의 스코어링 함수. 각 query token 임베딩에 대해 모든 document token 임베딩과의 유사도 중 최댓값을 구하고, 이를 모든 query token에 대해 합산하여 문서 점수를 만든다. 연산 구조는 matmul → max(문서 토큰 축) → sum(질의 토큰 축)이다. PDF p. 1, 2
IO-aware kernel
연산량(FLOP)이 아니라 HBM과 SRAM 사이의 메모리 이동량(IO)을 최소화하도록 설계된 GPU kernel. FlashAttention이 대표적이며, TileMaxSim은 같은 원리를 MaxSim의 matmul→max→sum 구조에 적용해 similarity matrix를 HBM에 materialize하지 않는다. PDF p. 1, 2, 3
Similarity matrix materialization
N_q×N_d 크기의 질의-문서 토큰 유사도 행렬 S를 HBM에 실제로 기록하는 것. S는 max-reduction에 한 번 쓰이고 버려지는 중간 데이터이므로, 이를 쓰고 다시 읽는 트래픽이 naive MaxSim의 주된 병목이라는 것이 이 논문의 핵심 진단이다. PDF p. 1, 2
Multi-query tiling (V2-MQ, BQ)
query token 축과 document token 축을 동시에 tiling하는 기법. query tile 크기 BQ를 N_q와 같게 설정하면 하나의 program이 모든 query token을 한 pass에 처리하므로 각 document embedding을 HBM에서 정확히 한 번만 읽는 IO-optimal 패턴이 된다. PDF p. 3, 4, 7
Dimension tiling
embedding dimension d를 128-wide chunk로 분할하여 부분 dot product를 누산한 뒤 max를 계산하는 기법. register/SRAM 용량을 초과하지 않으면서 하나의 kernel family가 d=64부터 d=768까지 처리할 수 있게 한다. PDF p. 2, 8
Product Quantization (PQ)
벡터를 M개의 sub-vector로 나누고 각 sub-space에서 K개의 centroid 중 하나로 근사하여 M바이트(K=256 기준)의 code로 압축하는 기법. 이 논문에서는 M=16, K=256, d_sub=8 설정을 사용한다. PDF p. 4, 5, 7
Fused PQ decompression+scoring
PQ code를 원본 벡터로 복원(decompress)하여 HBM에 쓰지 않고, query별 distance lookup table을 SRAM/L2에 상주시킨 뒤 code를 table index로 사용해 곧바로 유사도를 누산하는 방식. decompress-then-score 대비 최대 약 31배의 HBM IO 감소를 가져온다. PDF p. 4, 5
Roofline analysis / Arithmetic intensity
커널의 성능이 compute에 의해 제한되는지 memory bandwidth에 의해 제한되는지를 판별하는 분석. arithmetic intensity(FLOP/byte)가 하드웨어의 crossover point(H100에서 591 FLOP/byte)보다 훨씬 낮으면 memory-bound이다. MaxSim variant들은 14~32 FLOP/byte로 깊게 memory-bound이다. PDF p. 2, 3, 7
HBM / SRAM
HBM(High Bandwidth Memory)은 GPU의 대용량 off-chip 메모리(H100 기준 peak 3.35 TB/s)이고, SRAM(shared memory)은 SM 내부의 훨씬 빠르지만 작은 on-chip 메모리이다. IO-aware 설계의 목표는 HBM 접근을 SRAM/register 내 연산으로 대체하는 것이다. PDF p. 1, 3
Triton / tl.dot
Triton은 GPU kernel을 Python 문법으로 작성할 수 있게 하는 컴파일러/언어이며, tl.dot은 Tensor Core를 사용하는 행렬 곱 primitive이다. TileMaxSim의 모든 kernel은 Triton으로 구현되었다. PDF p. 3, 4
PLAID / colbert_score
PLAID는 centroid pruning과 PQ로 ColBERTv2 retrieval을 가속하는 엔진이고, colbert_score는 그 실제 GPU scoring 경로(batched D@Q.T 후 max-over-tokens 및 sum-over-query)로서 이 논문의 주요 실제 시스템 baseline이다. PDF p. 1, 5, 7
WARP
SIMD 가속 C++ kernel로 CPU MaxSim scoring을 최적화한 최신 multi-vector 엔진. XTR reference 구현 대비 41배, PLAID 대비 3배 빠르며, 이 논문에서 동일 노드 CPU 비교 대상으로 사용된다. PDF p. 1, 5, 10
Drop-in scoring kernel
기존 retrieval 파이프라인의 candidate generation과 decompression 단계를 바꾸지 않고 scoring 단계만 교체할 수 있는 구성요소. TileMaxSim은 exact MaxSim을 계산하므로 품질 변화 없이 PLAID의 scoring kernel을 대체할 수 있다. PDF p. 2, 10

  • Table 13에는 NFCorpus의 질의 수가 323개로 기재되어 있으나 Section 6.11 본문에서는 동일한 BEIR-NFCorpus 데이터셋에 대해 324개로 서술한다. 두 수치의 불일치에 대한 설명이 논문에 없다.
