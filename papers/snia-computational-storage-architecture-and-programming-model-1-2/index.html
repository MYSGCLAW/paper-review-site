핵심 요약

이 문서는 학술 논문이 아니라 SNIA(Storage Networking Industry Association)의 Computational Storage Technical Working Group이 작성한 표준 명세(standard specification)로, Computational Storage 장치와 호스트 사이의 아키텍처 모델(architecture model)과 프로그래밍 모델(programming model)을 정의한다. 핵심 아이디어는 전통적인 compute & memory 아키텍처 바깥의 연산 자원을 스토리지에 직접 결합하거나 host와 storage 사이에 배치함으로써 application 성능 및 infrastructure efficiency를 개선하고, parallel computation을 가능하게 하며 기존 compute/memory/storage/I-O 제약을 완화하는 것이다(17페이지). 이를 위해 문서는 CSx(Computational Storage Device: CSP/CSD/CSA)라는 상위 개념 아래 CSR(Computational Storage Resource), CSE(Computational Storage Engine), CSEE(Computational Storage Engine Environment), CSF(Computational Storage Function), Resource Repository, FDM/AFDM(Function Data Memory / Allocated Function Data Memory)이라는 계층적 용어 체계를 정의한다(14, 17–19페이지). 동작 이론(Theory of Operation)은 Discovery → Configuration/Activation → Security → CSF Usage(Direct / Indirect) → Sequencing(aggregator CSF) → Completion Handling의 흐름으로 구성되며(20–39페이지), 특히 CSEE를 CSE에 activate하고 그 위에 CSF를 activate하는 2단계 활성화 구조와, 여러 CSF를 host 개입 없이 순차·병렬로 묶어 실행하는 aggregator CSF 개념이 프로그래밍 모델의 중심축이다(19, 33–36페이지). 문서는 host–CSx 인터페이스의 물리적 특성이나 storage protocol을 특정하지 않는 벤더 중립적(vendor-neutral)·상호운용 가능(interoperable) 추상화를 목표로 하며(11–12페이지), 따라서 성능 실험이나 정량적 평가 결과는 포함하지 않는다.

해결하려는 문제

논문 주장

명세의 목적: 벤더 중립적 Computational Storage 모델 정의

이 명세의 목적은 CSx(Computational Storage Processor, Computational Storage Drive, Computational Storage Array)와 Host Agent 또는 다른 CSx 사이의 인터페이스에서 구현 가능한 capability와 action을 정의하는 것이다. 정의되는 action은 크게 Management(Discovery, Configuration), Security, Usage의 세 범주로 나뉜다. 이를 통해 특정 벤더 구현에 종속되지 않는(vendor-neutral) 상호운용 가능한(interoperable) Computational Storage 구현이 가능해지는 것을 목표로 한다.

PDF p. 11, 12

동기: 전통적 compute & memory 아키텍처 바깥의 연산 자원

Computational Storage 아키텍처는 전통적인 compute & memory 아키텍처 바깥의 compute 자원을 스토리지에 직접 결합하거나 host와 storage 사이에 배치함으로써 application 성능 및/또는 infrastructure efficiency의 향상을 가능하게 한다. 문서는 이러한 아키텍처의 목표를 parallel computation을 가능하게 하고 기존 compute, memory, storage, I/O에 대한 제약(constraints)을 완화하는 것으로 명시한다.

PDF p. 17

다중 CSF 조합 및 순서 제어 문제

단일 CSx 내에 있거나 여러 CSx에 걸쳐 있을 수 있는 다수의 CSF로 데이터를 전달하는 action과, 다수의 CSF에 작업 집합(set of tasks)의 수행을 요청하는 action을 정의하는 것도 이 명세가 다루는 문제 범위에 포함된다. 즉 개별 연산 하나를 offload하는 수준을 넘어, 연산들의 sequence를 host 개입 없이 장치 내부에서 구성·실행하는 방법이 필요하다는 점이 문제로 설정되어 있다.

PDF p. 12, 33

이 문서가 다루지 않는 범위

명세는 Host Agent와 CSx 간 인터페이스의 물리적 특성(physical characteristics)에 대해 어떠한 가정도 하지 않으며, 양자가 사용하는 storage protocol에 대해서도 가정하지 않는다. 즉 문제 정의 자체가 특정 전송 매체나 프로토콜과 분리된 추상 모델 수준에서 이루어진다.

제안 방식의 전체 구조

논문 주장

전체 구성: Host Agent + 하나 이상의 CSx

Computational Storage 아키텍처는 Host Agent(및 CS Driver)와 하나 이상의 CSx로 구성되며, 양자는 Fabric(PCIe, Ethernet 등)을 통해 연결된다. Figure 4.1은 CSP, CSD, CSA 세 가지 CSx 형태를 나란히 보여준다. 각 CSx는 MGMT(관리) 인터페이스와 I/O 인터페이스를 통해 host에 노출된다.

PDF p. 17, 18

CSx의 내부 구성요소

CSx는 (a) Computational Storage Resource(CSR), (b) CSD의 경우 Storage Controller 또는 CSA의 경우 Array Controller, (c) Device Memory, (d) CSD와 CSA의 경우 Device Storage로 구성된다. CSR은 다시 Resource Repository(활성화 대상 CSF/CSEE image 저장), Function Data Memory(FDM, 일부가 AFDM으로 할당됨), 하나 이상의 Computational Storage Engine(CSE)을 포함한다.

PDF p. 18

CSE / CSEE / CSF의 계층적 활성화 관계

CSE는 하나 이상의 CSF를 제공하도록 프로그램될 수 있는 CSR이다. CSEE는 CSE를 위한 운영 환경이고, CSF는 CSEE 안에서 CSE에 의해 configure 및 execute될 수 있는 specific operation 집합이다. Activation은 CSEE를 CSE에 연결하거나 CSF를 CSEE에 연결하는 과정이며, 활성화 시 필요한 자원이 할당되고 deactivation 시 해제된다. CSE가 CSF를 실행하려면 반드시 CSEE가 먼저 activate되어 있어야 한다. CSEE는 그 안에 추가 CSEE를 activate할 수 있으며, CSEE에 embedded된 CSF는 CSEE 활성화 시 암묵적으로 함께 활성화될 수 있다.

PDF p. 18, 19

제조 시 고정 CSF와 download 가능한 CSF

CSE는 제조 시점에 이미 하나 이상의 CSEE와 CSF가 활성화된 상태로 출하되어 host가 관리·I/O 인터페이스를 통해 곧바로 사용할 수 있다. 한편 host는 하나 이상의 CSEE와 CSF를 Resource Repository로 download한 뒤 activate할 수 있다. 제조 시 프로그램되어 변경 불가능한(즉 Resource Repository에 저장되지 않는) CSF도 존재할 수 있으며, compression, RAID, erasure coding, regular expression, encryption 등이 예로 제시된다. 활성화된 CSF는 CSx가 보고한 정의된 operation만 수행하며 하부 동작 자체는 변경되지 않는다.

PDF p. 19

FDM / AFDM의 접근 격리 규칙

FDM은 CSF가 사용하거나 생성하는 데이터를 담는 device memory이며, 그중 특정 CSF operation instance에 할당된 부분이 AFDM이다. 특정 CSF operation instance는 자신에게 할당된 AFDM에만 접근할 수 있어야 하며(shall), aggregator CSF와 그 constituent CSF들도 해당 aggregator CSF operation instance에 할당된 AFDM에만 접근할 수 있어야 한다. AFDM은 명시적으로 해제되거나 power cycle/reset 시 해제되며, 해제 시 물리 메모리를 clear하여 후속 CSF instance가 이전 사용자 데이터에 접근하지 못하게 해야 한다(should).

PDF p. 19

CSx의 세 가지 형태: CSP, CSD, CSA

CSP(Computational Storage Processor)는 영구 저장소를 제공하지 않으면서 연관된 storage system을 위해 CSF를 실행하는 component로, CSR과 Device Memory를 포함한다(스토리지 시스템과의 연결 방식은 구현 종속). CSD(Computational Storage Drive)는 CSF 실행과 영구 저장을 함께 제공하며 Storage Controller, CSR, Device Memory, 영구 저장소를 포함하고, 표준 host 인터페이스를 통해 일반 Storage Drive로도 동작할 수 있다. CSA(Computational Storage Array)는 control software를 통해 storage service·storage device·CSR에 가상화를 제공하는 storage array이며, CSA 내 CSR은 중앙집중형이거나 배열 내 CSD/CSP에 분산될 수 있다. CSA는 Transparent Storage Access와 Proxied Storage Access 경로를 함께 갖는다.

단계별 동작 과정

논문 주장

1단계: CSx Discovery 및 CSR Discovery

CSx 자체의 discovery는 fabric에 종속적이며 본 아키텍처의 scope 밖이다. CSx가 discover된 이후에는 해당 CSx의 특성을 파악하기 위한 CSR discovery process가 수행되며, 이 과정에서 CSE, CSEE, CSF, FDM을 포함한 사용 가능한 모든 자원이 발견된다. CSE의 discovery에는 그 CSE에 활성화된 CSEE 및 그 CSEE 안에 활성화된 CSF 정보가 포함된다. Resource Repository 내의 CSEE와 CSF도 discover될 수 있으나, 사용하려면 반드시 activate되어야 한다. CSR discovery process의 구체적 절차는 API 명세(예: SNIA Computational Storage API)에서 정의된다.

2단계: Configuration 및 2단계 Activation

5단계: CSF Usage — Indirect Usage Model

Indirect usage model에서는 host storage request와 연관된 데이터에 대해 CSF가 실행되며, 연관 기준은 (a) storage request 내 parameter(예: 실행할 CSF를 지정하는 field), (b) data locality(예: 특정 address range), (c) data characteristics(예: 지정 데이터의 크기)이다. locality나 characteristics 기반의 경우 storage request 전에 Storage Controller가 미리 CSF와 연관되도록 configure되어야 한다. 7단계: Sequencing (Aggregator CSF)

여러 CSF를 순차적 또는 병렬적으로 실행하여 host 개입을 최소화하거나 없애는 것이 바람직할 수 있다. 이 sequence 실행은 host가 개별 CSF 실행을 각각 개시하는 방식으로도, 또는 discrete CSF들을 캡슐화·집약하는 aggregator CSF로도 달성할 수 있다. CSx 자원 제약 내에서 어떤 sequence도 가능하며, discrete CSF든 다른 aggregator CSF든 aggregator CSF에 의해 여러 번 호출될 수 있다.

8단계: Completion Handling

CSF 실행 결과로 (a) CSF completion status(필수), (b) return value, (c) output data 세 종류의 정보가 생성될 수 있다. completion status는 성공/비성공(기대와 일치하지 않는 완료) 여부를 나타내며 실행을 요청한 entity에게 반환된다. return value는 연산에서 반환되는 값(예: 압축률 percentage)이고, output data는 연산 결과(예: 압축된 데이터)로 AFDM, Local Storage, Host Memory, 또는 host memory address로 지정 가능한 기타 목적지(예: 다른 장치의 메모리)에 놓일 수 있다. 비성공 완료는 CSEE 또는 host가 처리해야 하며, CSEE 처리는 host 부담을 offload하지만 host가 더 넓은 가시성과 자원을 가질 수 있고 일부 비성공 완료는 host만 처리 가능하다. CSEE는 알려진 상태로 cleanup/rollback할 능력이 있을 수도 없을 수도 있으며, 비성공 완료의 결과로 대응이 필요한 보안 고려사항을 CSEE 또는 host가 점검해야 한다.

Aggregator CSF의 기본 실행 규칙

Aggregator CSF는 지정된 sequence로 CSF들을 invoke한다. constituent CSF는 순차적으로, 병렬적으로, 또는 병렬과 순차의 조합으로 invoke될 수 있다. sequence 내 각 CSF가 성공적으로 완료될 때마다 aggregator CSF가 그다음 CSF를 호출하며, sequence 내 모든 CSF가 완료되면 aggregator CSF가 완료된다. 이 방식은 discrete CSF들이 독립적으로 존재·호출되면서도 sequence로 결합될 수 있게 하며, aggregator CSF가 다른 aggregator CSF를 호출하여 복잡한 sequence를 구성할 수도 있다.

  • 분석자 해석 용어 정의(3장)와 동작 이론(4장)을 분리하고, CSx/CSR/CSE/CSEE/CSF 각각에 번호가 붙은 형식 정의를 부여함으로써 후속 API 명세나 벤더 문서가 동일한 어휘를 참조할 수 있는 안정적 기반을 만든다.
    PDF p. 14, 18, 19
  • 분석자 해석 CSEE라는 중간 계층을 도입한 것이 설계상 강점이다. OS, Container Platform, Container, eBPF, FPGA Bitstream처럼 실행 모델이 근본적으로 다른 환경들을 동일한 activation 절차 아래 통합하여, 하드웨어 고정 기능과 download 가능한 소프트웨어 기능을 하나의 모델로 다룰 수 있게 한다.
    PDF p. 19, 43
  • 분석자 해석 Direct와 Indirect 두 usage model을 모두 정의하여, 기존 storage I/O 경로를 유지하면서 점진적으로 연산 offload를 도입하는 경로(Indirect)와 스토리지와 무관한 순수 연산 경로(Direct)를 동시에 지원한다.
    PDF p. 29, 30, 31
  • 분석자 해석 Aggregator CSF가 다른 aggregator CSF를 호출할 수 있게 하여 조합 가능성(composability)을 열어 두면서도, discrete CSF가 독립적으로 존재하고 호출될 수 있도록 유지한다. 모듈성과 파이프라인 효율을 동시에 확보하려는 설계로 볼 수 있다.
    PDF p. 33
  • 분석자 해석 AFDM 접근 제한을 shall 수준의 요구로, 해제 시 메모리 clear를 should 수준의 권고로 구분하여 명시함으로써, 데이터 잔류(data remanence)로 인한 정보 노출 위험을 모델 수준에서 선제적으로 다룬다.
    PDF p. 19
  • 분석자 해석 보안을 일률적 요구사항이 아니라 환경·상호연결성·데이터 민감도에 따라 달라지는 '고려사항'으로 제시하고, 단일 tenant와 다중 tenant 각각의 가정을 명시적으로 나열하여 독자가 자신의 환경에 어떤 항목이 적용되는지 판단할 수 있게 한다.
    PDF p. 23, 24
  • 분석자 해석 비성공 완료(unsuccessful completion) 처리에서 CSEE 처리와 host 처리의 장단점을 양쪽 모두 서술하고(CSEE 처리는 host 부담 offload, host는 더 넓은 가시성과 자원 보유, 일부는 host만 처리 가능), 특정 방식을 강요하지 않는다. 구현 다양성을 허용하면서도 설계 시 고려해야 할 축을 명확히 한다.

  • 저자 명시 한계 본 명세는 Host Agent와 CSx 간 인터페이스의 물리적 특성(physical characteristics)에 대해 어떠한 가정도 하지 않으며, Host Agent와 CSx가 사용하는 storage protocol에 대해서도 가정하지 않는다.

    PDF p. 20
  • 저자 명시 한계 시스템 관리자 등 privileged 개체/entity가 수행하는 privileged operation과 관련된 보안은 데이터·자원·구성·상태 보호에 critical하다고 문서가 인정하면서도, 본 문서의 scope 밖으로 명시적으로 제외된다.
    PDF p. 24
  • 저자 명시 한계 multi-host 환경 및 데이터 보호(protection of data)에 참여하는 CSx는 훨씬 더 복잡한 보안 고려사항과 요구사항을 초래한다는 점이 note되어 있으나, 그 복잡성이 완전히 해소되지는 않는다.
    PDF p. 24
  • 저자 명시 한계 CSR discovery process의 구체적 절차는 이 문서가 아니라 API 명세(예: SNIA Computational Storage API)에서 정의된다고 위임되어 있어, 이 문서만으로는 실제 구현이 완결되지 않는다.
    PDF p. 20
  • 저자 명시 한계 Figure 4.2의 discovery/configuration 흐름은 모든 action이 오류 없이 완료된다고 가정한 하나의 예시일 뿐이며, configuration이 완료될 수 있는 유일한 방법이 아니라고 문서가 스스로 밝힌다. 즉 오류 경로가 규정되지 않았다.
    PDF p. 21

  • 분석자 해석 구조상 예상되는 한계(분석자 추론): aggregator CSF는 자신과 동일한 CSx 상의 활성화된 CSF만 호출한다고 규정되어 있어(19페이지), 여러 CSx에 걸친 sequence는 host 또는 상위 계층이 조율해야 한다. Scope에서 언급된 '여러 CSx에 걸친 다수 CSF로의 데이터 전달'(12페이지)과 aggregator CSF의 단일 CSx 제약 사이에 추가 메커니즘이 필요해 보인다.
    PDF p. 12, 19
  • 분석자 해석 구조상 예상되는 한계(분석자 추론): CSF는 CSx가 보고한 정의된 operation만 수행하며 하부 동작은 변경 불가능하다(19페이지). 이는 안전성 측면에서는 이점이지만, host가 임의의 사용자 정의 연산을 표현하려면 eBPF 같은 특정 CSEE에 의존해야 하며, 그 경우 CSEE별 프로그래밍 모델 차이가 다시 이식성 문제로 되돌아올 수 있다.
    PDF p. 19, 43
  • 분석자 해석 구조상 예상되는 한계(분석자 추론): 문서는 실행 스케줄링, 자원 경합, 우선순위, QoS를 규정하지 않는다. 여러 CSF가 동시에 CSE와 FDM을 사용할 때의 조정 정책이 정의되지 않아 벤더 구현별로 성능 특성이 크게 달라질 수 있다.
    PDF p. 18, 19
  • 분석자 해석 구조상 예상되는 한계(분석자 추론): 규범 표현이 shall과 should로 혼재되어 있다(예: AFDM 접근 제한은 shall, 해제 시 메모리 clear는 should). should 항목은 준수하지 않아도 규격 위반이 아니므로, 보안적으로 중요한 메모리 clear 동작이 구현마다 달라질 여지가 있다.

핵심 용어

CSx (Computational Storage Device)
Host Agent 또는 다른 CSx와 인터페이스하는 Computational Storage 장치의 총칭. Computational Storage Drive(CSD), Computational Storage Processor(CSP), Computational Storage Array(CSA) 중 하나이다. PDF p. 14, 18, 19
CSP (Computational Storage Processor)
영구 데이터 저장을 제공하지 않으면서 연관된 storage system을 위해 하나 이상의 CSF를 실행할 수 있는 component. CSR과 Device Memory를 포함하며, storage system과의 연관 방식은 구현 종속적이다. PDF p. 19
CSD (Computational Storage Drive)
하나 이상의 CSF를 실행할 수 있으면서 영구 데이터 저장도 제공하는 component. Storage Controller, CSR, Device Memory, 영구 저장소를 포함하며 표준 host 인터페이스를 통해 일반 Storage Drive로도 동작할 수 있다. PDF p. 19
CSA (Computational Storage Array)
하나 이상의 CSF를 실행할 수 있는 storage array. control software를 통해 storage service, storage device, CSR에 가상화를 제공하며, 내부 CSR은 중앙집중형이거나 배열 내 CSD/CSP에 분산될 수 있다. PDF p. 19
CSR (Computational Storage Resource)
CSx가 CSF를 저장하고 실행하는 데 필요한 자원. Resource Repository, FDM, 하나 이상의 CSE로 구성된다. PDF p. 14, 18
CSE (Computational Storage Engine)
하나 이상의 CSF를 제공하도록 프로그램될 수 있는 CSR. 예로 CPU, FPGA가 제시된다. CSF를 실행하려면 CSEE가 활성화되어 있어야 한다. PDF p. 14, 18, 19
CSEE (Computational Storage Engine Environment)
CSE를 위한 운영 환경. 예로 Operating System, Container Platform, Container, eBPF, FPGA Bitstream이 제시된다. 사전 설치되거나 host가 download할 수 있으며 사용하려면 activate되어야 한다. 내부에 추가 CSEE를 activate할 수도 있고 CSF가 embedded되어 있을 수도 있다. PDF p. 14, 19, 43
CSF (Computational Storage Function)
CSEE 안에서 CSE에 의해 configure 및 execute될 수 있는 specific operation 집합. 예로 compression, RAID, erasure coding, regular expression, encryption이 제시된다. 활성화된 CSF는 CSx가 보고한 정의된 operation만 수행하며 하부 동작은 변경되지 않는다. PDF p. 14, 19
Activation / Deactivation
Activation은 CSEE를 CSE에 연결하거나 CSF를 CSEE에 연결하는 과정이며, 이때 사용에 필요한 자원이 할당된다. 더 이상 필요하지 않으면 deactivate할 수 있고 이 과정에서 할당된 자원이 해제된다. PDF p. 19
Resource Repository
CSx 내에 위치한 memory 및/또는 storage 영역으로, activation 대상이 되는 CSF와 CSEE의 image를 담을 수 있다. 여기 저장된 CSF/CSEE는 사용을 위해 반드시 CSE에서 activate되어야 한다. PDF p. 18, 19, 20
FDM (Function Data Memory)
CSF의 동작 과정에서 사용되거나 생성되는 데이터를 위해 CSF가 사용할 수 있는 device memory. PDF p. 14, 19
AFDM (Allocated Function Data Memory)
FDM 중 하나 이상의 특정 CSF operation instance에 할당된 부분. 특정 instance는 자신에게 할당된 AFDM에만 접근할 수 있어야 하며, 해제 시 물리 메모리를 clear하여 이전 사용자 데이터 노출을 방지해야 한다. PDF p. 14, 19
Aggregator CSF
CSF들의 ordered sequence를 호출하는 CSF. constituent CSF를 순차·병렬 또는 그 조합으로 invoke하며, 다른 aggregator CSF도 호출할 수 있다. 자신과 동일한 CSx 상의 활성화된 CSF를 호출한다. PDF p. 14, 19, 33
Fixed sequence aggregator CSF
하드코딩된 특정 CSF sequence를 실행하는 aggregator CSF. constituent CSF나 순서가 다르면 서로 다른 fixed sequence aggregator CSF로 간주된다. 병렬 실행을 포함할 수 있다. PDF p. 34
Variable sequence aggregator CSF
invocation 시 제공되는 parameter로 지정된 임의의 CSF sequence를 실행하는 aggregator CSF. 병렬 실행과 fixed sequence aggregator CSF를 포함할 수 있으며 조건 분기와 반복을 표현할 수 있다. PDF p. 36
Direct CSF Usage Model
host가 실행할 CSF와 대상 데이터 위치(AFDM, device storage, system memory 중)를 명시하는 computation request를 직접 보내고, 출력 위치도 지정하는 사용 모델. PDF p. 29
Indirect CSF Usage Model
host storage request와 연관된 데이터에 대해 CSF가 실행되는 사용 모델. 연관 기준은 storage request의 parameter, data locality(주소 범위), data characteristics(데이터 크기 등)이며, locality/characteristics 기반의 경우 사전에 Storage Controller를 configure해야 한다. PDF p. 29, 30
CSF Completion Output
CSF 실행 결과로 생성될 수 있는 정보. CSF completion status(필수, 성공/비성공 표시), return value(예: 압축률), output data(예: 압축된 데이터)로 구성되며 output data는 AFDM, Local Storage, Host Memory 등에 배치될 수 있다. PDF p. 32
Host Agent
CSx와 인터페이스하는 host 측 주체. 이 명세는 Host Agent와 CSx 간 인터페이스의 물리적 특성이나 사용 storage protocol에 대해 어떠한 가정도 하지 않는다. PDF p. 12
