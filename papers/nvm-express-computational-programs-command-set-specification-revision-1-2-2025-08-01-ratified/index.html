핵심 요약

본 문서는 연구 논문이 아니라 NVM Express가 2025년 7월 30일자로 발행한 산업 표준 규격서(Revision 1.2, ratified)이며, host에서 NVM subsystem으로 프로그램 실행을 오프로드(offload)하기 위한 Computational Programs Command Set을 정의한다(pp.9-10). 핵심 모델은 host-driven 방식이다. 즉 프로그램(program)은 device-defined이거나 host가 download한 것일 수 있고, 프로그램은 하나 이상의 memory namespace, 즉 Subsystem Local Memory(SLM) 내 데이터에 대해서만 동작하며, 프로그램의 입력 및 출력 데이터는 host가 발행하는 NVMe 커맨드를 통해 memory namespace로 staging된다(pp.9-10). 규격은 이를 위해 (1) 프로그램을 실행하는 주체인 compute namespace, (2) 데이터를 담는 memory namespace/SLM, (3) 프로그램이 접근 가능한 SLM 영역을 (NSID, offset, length) range들의 집합으로 한정하는 Memory Range Set, (4) compute namespace가 어떤 memory namespace에 도달 가능한지를 규정하는 reachability라는 네 가지 개념을 축으로 구조를 정의한다(pp.9-13). 커맨드 수준에서는 I/O 커맨드인 Execute Program(opcode 01h)과 Admin 커맨드인 Load Program, Program Activation Management, Memory Range Set Management가 정의되며, 프로그램은 PUID(64비트 Program Unique Identifier)로 기능이 식별되고 compute namespace 내에서는 PIND(Program Index)로 참조된다(pp.13-16, 25-27, 34-39). 동작 흐름은 capability 탐색 → 프로그램 load → activation → Memory Range Set 생성 → 입력 데이터 staging 후 Execute Program 발행 → completion queue entry에서 상태와 64비트 return value(RVAL) 확인 → 결과 데이터 회수의 순서로 진행된다(pp.17-22). 규격 문서의 특성상 정량적 실험 평가나 baseline 비교는 포함되어 있지 않다.

연산의 오프로드 경로 표준화

본 규격이 다루는 문제는 host에서 NVM subsystem으로 프로그램 실행(program execution)을 오프로드하기 위한 표준 메커니즘을 정의하는 것이다. 이때 실행되는 프로그램은 device-defined program(장치가 제공하는 프로그램)일 수도 있고, host가 download한 downloadable program일 수도 있다(pp.9-10).

PDF p. 9, 10

host-driven 실행 모델

규격이 채택한 실행 모델은 host-driven이다. 프로그램은 하나 이상의 memory namespace(Subsystem Local Memory, SLM) 내 데이터에 대해 동작하며, 프로그램의 입력 및 출력 데이터는 host가 발행하는 NVMe 커맨드를 통해 memory namespace로 staging된다. 즉 연산 자체는 device 안에서 일어나지만, 데이터의 배치와 실행 시점은 host가 명시적으로 통제한다(pp.9-10).

규격이 정의하는 것

본 규격은 Computational Programs Command Set의 기술적 아키텍처, 커맨드 동작(command behavior), 그리고 운영 워크플로(operational workflow)를 정의한다(pp.9-10).

개체 모델 정의

compute namespace(프로그램 실행 주체), memory namespace/SLM(입출력 데이터 저장소), Memory Range Set(프로그램의 SLM 접근 범위 한정), reachability(compute namespace가 도달 가능한 memory namespace 집합)라는 개체 및 관계 모델을 정의한다(pp.8-13).

커맨드 집합 정의

I/O 커맨드인 Execute Program(opcode 01h)과 Admin 커맨드인 Load Program, Program Activation Management, Memory Range Set Management를 정의하여 프로그램의 적재-활성화-실행 전 주기를 표준 커맨드로 표현한다(pp.9, 25-27, 34-39).

탐색 가능성(discoverability) 정의

Program List, Downloadable Program Types List, Memory Range Set List 로그 페이지와 CNS 05h/06h Identify 데이터 구조를 정의하여, host가 subsystem이 어떤 프로그램과 자원 한계(MAXACT, MAXMEMRS, MRSG, MAXMEMR, MAXPB, LPG 등)를 갖는지 표준 방식으로 조회할 수 있게 한다(pp.28-34).

PDF p. 28, 29, 30, 31, 32, 33, 34

프로그램 식별 체계 정의

64비트 PUID(IEEE organization identifier + Universal Program Identifier)로 프로그램 기능을 전역적으로 고유하게 식별할 수 있게 하고, compute namespace 내부에서는 PIND(Program Index)라는 논리 식별자로 프로그램을 참조하게 한다(pp.13-16).

제안 방식의 전체 구조

논문 주장

전체 개요

Computational Programs Command Set은 host가 NVM subsystem으로 프로그램 실행을 오프로드하는 메커니즘이다. 프로그램은 device-defined이거나 host가 download한 것일 수 있으며, host-driven으로 동작한다. 프로그램은 하나 이상의 memory namespace(SLM) 내 데이터에 대해 동작하고, 입출력 데이터는 host가 발행하는 NVMe 커맨드로 memory namespace에 staging된다. Figure 2는 하나의 domain 안에 compute namespace, memory namespace, NVM namespace가 공존하는 예시를 보여준다(pp.9-10).

PDF p. 9, 10

Compute Namespace

프로그램을 실행할 수 있는 domain 내 entity이다. Endurance Group이나 NVM Set에 속하지 않으며, 하나 이상의 프로그램 타입을 지원할 수 있고, compute namespace마다 서로 다른 memory namespace reachability 특성을 가질 수 있다(pp.9, 13).

PDF p. 9, 13

Subsystem Local Memory (SLM) / Memory Namespace

프로그램의 입력 및 출력에 사용되는 메모리이다. NSID로 식별되는 memory namespace로 구성되며, (NSID, offset, length)로 주소가 지정된다. 접근 단위에는 차이가 있어서, 프로그램 실행과 관련된 접근은 byte granular이지만 SLM Command Set 커맨드는 dword granular이다(pp.8, 10).

PDF p. 8, 10

Memory Range Set

SLM의 하나 이상의 range(각각 NSID, offset, length로 지정)로 구성되며 특정 compute namespace 안에서 생성된다. 프로그램의 SLM 접근을 특정 부분집합으로 제한하는 역할을 하며, Memory Range Set Management 커맨드로 생성·삭제된다(pp.10-12).

PDF p. 10, 11, 12

Reachability (Reachability Groups / Associations)

compute namespace가 어떤 memory namespace에 도달(reach) 가능한지를 결정하는 개념이며, 결과적으로 하나의 Memory Range Set에 포함될 수 있는 memory namespace의 범위를 제한한다(pp.10-13).

PDF p. 10, 11, 12, 13

Program, PUID, PIND

프로그램은 compute namespace 상에서 실행되는 기능 단위이다. namespace가 제공하며 host가 변경할 수 없는 device-defined program과 host가 로드/언로드할 수 있는 downloadable program으로 구분된다. 프로그램 기능은 64비트 PUID로 고유 식별될 수 있고, compute namespace별 논리 식별자인 PIND로 커맨드에서 참조된다. 프로그램은 실행 전에 Program Activation Management 커맨드로 활성화되어야 하며, 활성화는 필요한 compute resource를 예약하는 과정을 포함할 수 있다(pp.13-16).

PDF p. 13, 14, 15, 16

탐색용 로그 페이지 및 Identify 구조

Program List, Downloadable Program Types List, Memory Range Set List 로그 페이지는 각각 compute namespace에 연계된 프로그램 목록, 지원되는 downloadable program type 목록, 생성된 Memory Range Set 목록을 Get Log Page 커맨드로 조회하게 한다(pp.28-32). CNS 05h/06h Identify 데이터 구조는 compute namespace 특성(MAXACT, MAXMEMRS, MRSG, MAXMEMR, MAXPB, LPG 등)과 controller가 지원하는 명세 버전을 조회하게 한다(pp.32-34).

PDF p. 28, 29, 30, 31, 32, 33, 34

단계별 동작 과정

논문 주장

1단계: Compute Namespace Capability 탐색

host는 Controller Capabilities의 CAP.CSS 필드로 다중 I/O command set 지원 여부를 확인하고, Identify 커맨드(CNS 1Ch)로 이 I/O Command Set의 지원 여부를 확인한다. 이어 CNS 2h(Active Namespace ID list)와 CNS 3h(Namespace Identification Descriptor List) 조합, 또는 CSI=04h를 지정한 CNS 7h로 compute namespace 목록을 얻는다. 각 compute namespace에 대해서는 I/O Command Set specific Identify Namespace 데이터 구조, Program List 로그 페이지, Downloadable Program Types List 로그 페이지, Reachability Groups/Reachability Associations 로그 페이지를 조회한다(p.17).

PDF p. 17

2단계: Downloadable Program 로드

host는 Load Program 커맨드를 compute namespace에 발행하며, PIND와 실행 파일(executable program)을 가리키는 포인터를 지정한다(Figure 7: Load Program Flow, p.18). 프로그램은 LOFF/NUMB 필드를 사용해 여러 조각(piece)으로 나뉘어 개별 커맨드로 전송될 수 있다(pp.34-36).

PDF p. 18, 34, 35, 36

3단계: Program 활성화

host는 Program Activation Management 커맨드를 compute namespace에 발행하며 활성화할 프로그램의 PIND를 지정한다(Figure 8: Program Activation Flow, p.19). 활성화는 실행에 필요한 compute resource 예약을 포함할 수 있다(pp.15-16).

PDF p. 15, 16, 19

4단계: Memory Range Set 생성

host는 Memory Range Set Management 커맨드를 compute namespace에 발행하며 create 동작과 Memory Range 목록을 지정한다. 이때 지정 가능한 memory namespace는 해당 compute namespace가 reachable한 것으로 한정된다. 커맨드 완료 시 host는 completion queue entry에서 실행 상태와 새로 생성된 Memory Range Set Identifier를 확인한다(Figure 9: Create a Memory Range Set Flow, p.20). create 동작에서는 최대 128개의 Memory Range Descriptor를 지정할 수 있다(pp.36-38).

PDF p. 20, 36, 37, 38

5단계: Program 실행

실행 단계는 다섯 하위 단계로 구성된다. (1) host는 적절한 I/O command set 커맨드로 프로그램 입력 데이터를 Memory Range Set에 포함된 memory namespace 위치에 저장한다(이미 저장되어 있으면 생략 가능). (2) host는 PIND, Memory Range Set ID, Parameter data를 지정해 compute namespace에 Execute Program 커맨드를 발행한다. (3) 프로그램이 실행된다. (4) 완료 시 host는 completion queue entry에서 실행 상태와 프로그램의 return value를 확인한다. (5) 필요하면 host는 memory namespace에 대한 커맨드로 결과 데이터를 host 메모리로 전송한다(Figure 10: Execute Program Flow, pp.21-22).

PDF p. 21, 22

핵심 커맨드 및 자료 구조

논문 주장

Execute Program command (I/O, opcode 01h)

지정된 namespace에서, 지정된 Memory Range Set의 데이터와 커맨드에 담긴 parameter(CPARAM1/CPARAM2 필드 또는 데이터 버퍼)를 사용하여, 이미 활성화된 프로그램을 실행하는 I/O 커맨드이다. 완료 시 CQE의 Dword0/Dword1에 프로그램의 64비트 반환값(RVAL)을 담아 반환할 수 있다(pp.9, 25-27). 이 구조상 프로그램의 '주소 공간'은 Memory Range Set으로 명시되고, '인자'는 커맨드 필드 또는 별도 버퍼로 전달되며, '결과'는 RVAL과 SLM에 남는 출력 데이터로 나뉜다.

PDF p. 9, 25, 26, 27

Load Program command (Admin)

downloadable program을 특정 PIND에 다운로드(load)하거나 언로드(unload)하는 Admin 커맨드이다. 프로그램은 LOFF(offset)와 NUMB(길이) 필드를 사용해 여러 조각으로 나뉘어 개별 커맨드로 전송될 수 있어, 큰 실행 이미지를 분할 전송할 수 있다(pp.9, 34-36).

PDF p. 9, 34, 35, 36

Program Activation Management command (Admin)

프로그램을 활성화(activate)하거나 비활성화(deactivate)하는 Admin 커맨드이다(pp.9, 38-39).

PDF p. 9, 38, 39

Memory Range Set Management command (Admin)

Memory Range Set을 생성(create)하거나 삭제(delete)하는 Admin 커맨드이다. create 동작 시 최대 128개의 Memory Range Descriptor를 지정할 수 있다(pp.9, 36-38).

PDF p. 9, 36, 37, 38

식별자 체계

PUID는 64비트 값으로 IEEE organization identifier와 Universal Program Identifier로 구성되며 프로그램 기능을 고유하게 식별한다. PIND는 compute namespace별로 프로그램을 식별하는 논리적 식별자이며 Load/Activation/Execute 커맨드가 공통으로 사용하는 참조 키이다(pp.13-16).

PDF p. 13, 14, 15, 16

자원 한계 파라미터

Identify 데이터 구조는 MAXACT(최대 활성화 수), MAXMEMRS(최대 Memory Range Set 수), MRSG(granularity), MAXMEMR(최대 memory range 수), MAXPB, LPG 등의 필드를 통해 compute namespace의 자원 한계를 host에 노출한다. 이 값들은 host가 위 워크플로를 계획할 때의 제약 조건이 된다(pp.32-34).

규격서에서 실험 대신 제공되는 것은 커맨드 필드 정의, 상태 코드(Computational Programs Specific Status Values), I/O Controller Requirements(커맨드 지원, 로그 페이지 지원, feature 지원) 같은 적합성(conformance) 요구사항이다(pp.22-24). 이는 분석자의 해석이며, 성능 주장과는 성격이 다르다.

주요 결과와 의미

분석자 해석

정량 결과 없음

extraction.json에 key_results가 비어 있으므로 정량적 성능 결과는 제시하지 않는다. 그래프 수치를 눈으로 추정하지 않는다는 원칙에 따라 charts 배열은 비워 둔다.

구조적 결과 1: 연산 대상이 SLM으로 한정됨

프로그램은 NVM namespace의 데이터에 직접 임의로 접근하는 것이 아니라 memory namespace(SLM) 내 데이터에 대해 동작하고, 데이터는 host 커맨드로 staging된다(pp.9-10). 이는 device 내 연산의 부수효과 범위를 명확히 제한하는 대신, 저장 매체와 SLM 사이의 데이터 이동을 host가 명시적으로 조율해야 함을 뜻한다. 이 해석은 분석자의 추론이다.

PDF p. 9, 10

구조적 결과 2: 이중 격리(reachability + Memory Range Set)

compute namespace의 SLM 접근은 두 단계로 제한된다. 먼저 reachability가 해당 compute namespace가 도달 가능한 memory namespace 집합을 결정하고, 그 안에서 Memory Range Set이 실제 접근 가능한 (NSID, offset, length) 부분집합을 확정한다(pp.10-13). 결과적으로 여러 tenant나 여러 프로그램이 같은 subsystem을 공유할 때의 격리 근거가 규격 수준에서 제공된다. 이 해석은 분석자의 추론이다.

PDF p. 10, 11, 12, 13

구조적 결과 3: 실행 전 활성화의 의미

프로그램은 실행 전 반드시 활성화되어야 하며 활성화가 compute resource 예약을 포함할 수 있다(pp.15-16). 이는 Execute Program 커맨드 경로에서 자원 할당 비용을 제거하고 실행 경로를 짧게 유지하려는 설계로 읽힌다. 이 해석은 분석자의 추론이다.

PDF p. 15, 16

구조적 결과 4: 반환 채널의 이원화

프로그램 결과는 CQE Dword0/1의 64비트 RVAL과, SLM에 남아 host가 별도 커맨드로 읽어 가는 출력 데이터라는 두 경로로 나뉜다(pp.21-22, 25-27). 작은 스칼라 결과는 추가 전송 없이 즉시 회수되고, 큰 결과는 별도 read 경로를 탄다. 이 해석은 분석자의 추론이다.

규격의 결론적 위치

Revision 1.2는 Computational Programs Command Set Revision 1.1에 TP4153, ECN128, ECN129를 통합한 ratified 규격이며(p.2), computational storage를 위한 프로그램 오프로드의 아키텍처, 커맨드 동작, 운영 워크플로를 표준화한다(pp.9-10). 대학원생 관점에서 이 문서의 가치는 '어떤 성능이 나오는가'가 아니라 '연산 오프로드를 host-device 인터페이스로 어떻게 형식화하는가'에 대한 참조 모델을 제공한다는 점이다(분석자 해석).

  • 분석자 해석 제어 평면과 실행 평면이 명확히 분리되어 있다. 프로그램 적재·활성화·메모리 범위 설정은 Admin 커맨드로, 실제 실행은 단일 I/O 커맨드(Execute Program)로 수행되므로 실행 경로가 짧게 유지된다.
    PDF p. 9, 25, 34, 36, 38
  • 분석자 해석 reachability와 Memory Range Set이라는 두 단계 제한으로 프로그램의 메모리 접근 범위를 규격 수준에서 한정하여, 공유 subsystem에서의 격리 근거를 제공한다.
    PDF p. 10, 11, 12, 13
  • 분석자 해석 PUID(전역 고유 식별)와 PIND(namespace 지역 식별)를 분리해, 동일 기능의 프로그램을 벤더 간에 식별하면서도 커맨드 필드는 짧게 유지할 수 있다.
    PDF p. 13, 14, 15
  • 분석자 해석 MAXACT, MAXMEMRS, MRSG, MAXMEMR, MAXPB, LPG 등 자원 한계가 Identify 구조로 노출되어 host가 사전에 워크플로를 계획할 수 있고, 로그 페이지로 프로그램 목록과 지원 타입을 탐색할 수 있어 구현 간 이식성이 높다.

한계

  • 저자 명시 한계 저자(NVM Express)가 문서 내에서 명시한 한계 항목은 추출되지 않았다. extraction.json의 author_stated_limitations는 비어 있으므로 저자 명시 한계는 확인되지 않음으로 표시한다.
  • 분석자 해석 host-driven 모델이므로 프로그램의 입력·출력 데이터를 memory namespace로 staging하는 책임이 host에 있다. 저장 매체와 SLM 사이의 데이터 이동이 자동화되지 않으므로, 데이터 이동량이 큰 워크로드에서는 오프로드 이득이 host 측 staging 비용에 상쇄될 수 있다(구조상 예상되는 한계).
    PDF p. 9, 10, 21, 22
  • 분석자 해석 프로그램은 실행 전 활성화가 필요하고 활성화가 compute resource 예약을 포함할 수 있으며, MAXACT 같은 상한이 존재한다. 따라서 동시에 활성 상태로 둘 수 있는 프로그램 수가 제한되어, 많은 종류의 프로그램을 번갈아 쓰는 경우 activate/deactivate 전환 비용이 발생할 수 있다(구조상 예상되는 한계).
    PDF p. 15, 16, 32, 33, 34
  • 분석자 해석 Memory Range Set은 create 시 최대 128개의 Memory Range Descriptor로 제한된다. 접근 패턴이 산발적이거나 많은 수의 불연속 영역을 필요로 하는 프로그램은 여러 Memory Range Set으로 분할하거나 데이터를 재배치해야 할 수 있다(구조상 예상되는 한계).
    PDF p. 36, 37, 38
  • 분석자 해석 Execute Program의 즉시 반환값은 CQE Dword0/1의 64비트 RVAL로 한정된다. 그 이상의 결과는 SLM을 읽는 추가 커맨드가 필요하므로, 결과가 작지만 개수가 많은 워크로드에서는 왕복 횟수가 늘어날 수 있다(구조상 예상되는 한계).
    PDF p. 21, 22, 25, 26, 27
  • 분석자 해석 규격은 커맨드 인터페이스와 개체 모델을 정의하지만 실행되는 프로그램의 내부 실행 모델(ISA, 샌드박싱 방식, 스케줄링 정책 등)이나 성능 특성은 본 문서 범위에서 확인되지 않는다. 따라서 구현 간 성능 이식성은 규격만으로 보장되지 않는다(구조상 예상되는 한계).
    PDF p. 9, 13, 16
  • 분석자 해석 SLM 접근 단위가 문맥에 따라 다르다(프로그램 실행 관련 접근은 byte granular, SLM Command Set 커맨드는 dword granular). 이 비대칭은 host 소프트웨어가 버퍼 정렬과 부분 dword 갱신을 별도로 처리해야 함을 뜻한다(구조상 예상되는 한계).
    PDF p. 8, 10

핵심 용어

Compute Namespace
프로그램을 실행할 수 있는 domain 내 entity. Endurance Group이나 NVM Set에 속하지 않으며, 하나 이상의 프로그램 타입을 지원하고 memory namespace reachability 특성을 가진다. PDF p. 9, 13
Subsystem Local Memory (SLM)
프로그램의 입력 및 출력에 사용되는 subsystem 내부 메모리. NSID로 식별되는 memory namespace로 구성되고 (NSID, offset, length)로 주소가 지정된다. PDF p. 8, 10
Memory Namespace
SLM을 구성하는 namespace 단위. host가 NVMe 커맨드로 프로그램의 입출력 데이터를 staging하는 대상이다. PDF p. 8, 9, 10
Memory Range Set (MRS)
SLM의 하나 이상의 range로 구성되며 특정 compute namespace 안에서 생성되는 집합. 프로그램의 SLM 접근을 특정 부분집합으로 제한하고, create 시 최대 128개의 Memory Range Descriptor를 지정할 수 있다. PDF p. 10, 11, 12, 36, 37, 38
Reachability (Reachability Groups / Associations)
compute namespace가 어떤 memory namespace에 도달 가능한지를 결정하는 개념. Memory Range Set에 포함될 수 있는 memory namespace의 범위를 제한한다. PDF p. 10, 11, 12, 13
Device-defined Program
compute namespace가 제공하며 host가 변경할 수 없는 프로그램. PDF p. 13, 14
Downloadable Program
host가 Load Program 커맨드로 로드하거나 언로드할 수 있는 프로그램. PDF p. 13, 14, 34, 35, 36
PUID (Program Unique Identifier)
IEEE organization identifier와 Universal Program Identifier로 구성된 64비트 식별자로, 프로그램의 기능을 고유하게 식별할 수 있다. PDF p. 13, 14, 15
PIND (Program Index)
compute namespace별로 프로그램을 식별하는 논리적 식별자. Load, Activation, Execute 커맨드가 공통으로 사용하는 참조 키이다. PDF p. 15, 16
Program Activation
프로그램을 실행 가능한 상태로 만드는 절차. Program Activation Management 커맨드로 수행되며 필요한 compute resource 예약을 포함할 수 있다. PDF p. 15, 16, 38, 39
Execute Program command
opcode 01h의 I/O 커맨드. 지정된 Memory Range Set의 데이터와 CPARAM1/CPARAM2 또는 데이터 버퍼의 parameter로 활성화된 프로그램을 실행하고, CQE Dword0/1에 64비트 RVAL을 반환할 수 있다. PDF p. 9, 25, 26, 27
RVAL (Return Value)
Execute Program 완료 시 completion queue entry의 Dword0/Dword1에 담겨 반환될 수 있는 프로그램의 64비트 반환값. PDF p. 25, 26, 27
Load Program command
downloadable program을 특정 PIND에 다운로드하거나 언로드하는 Admin 커맨드. LOFF/NUMB 필드로 프로그램을 여러 조각으로 나누어 전송할 수 있다. PDF p. 9, 34, 35, 36
Memory Range Set Management command
Memory Range Set을 생성하거나 삭제하는 Admin 커맨드. PDF p. 9, 36, 37, 38
Program List / Downloadable Program Types List / Memory Range Set List
각각 compute namespace에 연계된 프로그램 목록, 지원되는 downloadable program type 목록, 생성된 Memory Range Set 목록을 Get Log Page 커맨드로 조회하는 로그 페이지. PDF p. 28, 29, 30, 31, 32
CNS 05h / CNS 06h
compute namespace 특성(MAXACT, MAXMEMRS, MRSG, MAXMEMR, MAXPB, LPG 등)과 controller가 지원하는 명세 버전을 조회하기 위한 I/O Command Set specific Identify 데이터 구조. PDF p. 32, 33, 34
byte granular / dword granular
SLM 접근 단위. 프로그램 실행과 관련된 접근은 byte 단위이고, SLM Command Set 커맨드는 dword(4바이트) 단위이다. PDF p. 8, 10
