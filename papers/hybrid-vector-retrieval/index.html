Hybrid-Vector Retrieval for Visually Rich Documents: Combining Single-Vector Efficiency and Multi-Vector Accuracy

핵심 요약

이 논문은 visually rich document retrieval(VDR)에서 나타나는 efficiency-accuracy trade-off를 해결하기 위해 HEAVEN(Hybrid-vector retrieval for Efficient and Accurate Visual multi-documENt)이라는 two-stage hybrid-vector framework를 제안한다. 저자들은 두 가지 관찰에서 출발한다. 첫째, single-vector retrieval은 query-page 쌍당 dot product 한 번만 계산하므로 multi-vector retrieval 대비 ViMDoc에서 최대 99.94% 적은 FLOPs를 요구한다(Obs.1). 둘째, single-vector retrieval은 Recall@1에서는 multi-vector 대비 22.5%의 성능 격차를 보이지만 Recall@200에서는 격차가 0.63%로 좁혀져, 넓은 candidate set을 대상으로 하는 coarse-grained retrieval에는 충분히 정확하다(Obs.2). HEAVEN은 이 관찰을 그대로 설계에 반영하여, Stage 1에서는 single-vector 모델(DSE)로 candidate page를 빠르게 선별하고 Stage 2에서는 multi-vector 모델(ColQwen2.5)로 fine-grained reranking을 수행한다. 각 stage에는 추가적인 redundancy 제거 장치가 들어간다. Stage 1에서는 Document Layout Analysis로 추출한 title layout을 여러 페이지에 걸쳐 조합한 visually-summarized page(VS-page)를 indexing 시점에 한 번만 만들어 비교 대상 페이지 수 자체를 줄이고, Stage 2에서는 POS tagging으로 전체 query token의 약 30%에 해당하는 key token만 골라 1차 reranking에 사용한다. 또한 저자들은 기존 VDR 벤치마크가 multi-document 설정과 long-context 설정을 동시에 다루지 않는다는 gap을 지적하며, 1,379개 문서·76,347페이지·10.9k 쿼리 규모의 ViMDoc 벤치마크를 함께 제안한다. 4개 multi-document 벤치마크(ViMDoc, OpenDocVQA, ViDoSeek, M3DocVQA) 평균에서 HEAVEN은 ColQwen2.5 대비 Recall@1의 99.87%를 유지하면서 쿼리당 FLOPs를 99.82% 절감했고, ViMDoc 기준 온라인 검색 latency는 2.412초로 ColQwen2.5의 2006.361초 대비 99.88% 빠르다. HEAVEN은 plug-and-play 구조로 각 stage의 base encoder를 추가 학습 없이 교체할 수 있다.

해결하려는 문제

논문 주장

연구 문제: VDR의 efficiency-accuracy trade-off

논문이 다루는 핵심 연구 문제는 visually rich document retrieval(VDR)에서 single-vector retrieval과 multi-vector retrieval 사이에 존재하는 efficiency-accuracy trade-off이다. Single-vector retrieval은 효율적이지만 부정확하고, multi-vector retrieval은 정확하지만 계산 비용이 크다. 논문은 이 두 성질을 하나의 파이프라인 안에서 결합하는 two-stage hybrid-vector framework인 HEAVEN을 제안하여 이 trade-off를 해소하고자 한다.

PDF p. 1, 2

동기: RAG 시대의 문서 검색과 시각적으로 복잡한 페이지

문서 검색(document retrieval)은 legal discovery, scientific literature search, enterprise knowledge management 등에 널리 응용되며, 최근에는 LLM 기반 Retrieval-Augmented Generation(RAG)의 핵심 구성요소로 다시 주목받고 있다. 그런데 실제 문서 페이지는 차트, 표, 그림 등 시각적으로 복잡한 요소를 포함하고 있어 텍스트만으로는 표현되지 않는 정보가 많고, 이를 이미지 단위로 처리하는 visual document retrieval(VDR)이 중요한 과제로 부상했다.

PDF p. 1, 2

실제 응용 환경과 벤치마크 사이의 gap

실제 응용 환경에서는 개별 문서가 수십 페이지에 이르는 대규모 multi-document 컬렉션을 대상으로 검색을 수행해야 한다. 그러나 기존 VDR 벤치마크는 multi-document 설정과 long-document 설정을 함께 고려하지 않는다. 논문의 Table 2에 따르면 기존 벤치마크는 대부분 단일 gold-standard 문서 안에서의 검색을 가정하거나 문서 길이가 평균 1.0~18.4페이지로 짧다. 논문은 이 gap을 메우기 위해 ViMDoc 벤치마크를 함께 제안한다(문서당 평균 55.4페이지, multi-doc/long-doc/cross-query 세 조건을 모두 충족).

기존 방식의 한계

논문 주장

OCR·layout parsing 기반 파이프라인의 비용

시각적으로 복잡한 콘텐츠를 처리하기 위해 전통적으로 OCR이나 복잡한 layout parsing이 사용되어 왔으며, 이는 indexing 시간과 파이프라인 복잡도를 증가시킨다. 논문의 Table 5에서도 textual 모델(NV-Embed-V2, BGE-M3 multi-vector)은 OCR 전처리에 473.0분의 offline indexing 시간을 요구하는 반면, 시각 인코더 기반 방법은 OCR 단계 자체가 필요 없음(NA)이 확인된다.

PDF p. 1, 8

Single-vector retrieval: 효율적이나 fine-grained interaction 미포착

DSE 같은 single-vector retrieval 모델은 query와 page를 각각 하나의 벡터로 표현하고 dot product 한 번으로 점수를 계산한다(S_SV(q,P) = <Ẽq, ẼP>). 코퍼스 전체 채점 비용은 O(d·|P|)로 저렴하지만, token/patch-level의 세밀한 상호작용을 포착하지 못해 정확도가 낮다.

PDF p. 2, 3

Multi-vector retrieval: 정확하나 계산량이 폭증

ColQwen2.5 같은 multi-vector 모델은 query token과 page patch embedding의 모든 쌍에 대해 MaxSim을 계산하고 합산한다(S_MV(q,P) = Σ_i max_j <Eq^(i), EP^(j)>). 페이지 하나 채점에 O(d·nq·nP), 코퍼스 전체에는 O(d·nq·Σ nP)가 필요하여 계산 비용(computational overhead)이 훨씬 크다.

PDF p. 2, 3, 9

코퍼스 전체 스캔의 redundancy

기존 single-vector 모델은 코퍼스 내 모든 페이지와 쿼리 간 유사도를 계산하므로 비용이 페이지 수 |P|에 선형적으로 증가한다. 그러나 대부분의 페이지는 쿼리와 무관하며, 로고·헤더·boilerplate 같은 반복적이고 비정보적인 콘텐츠를 포함하고 있어 전체 코퍼스에 대한 계산 상당 부분이 redundant하다.

PDF p. 3, 4

query token 측의 redundancy

Multi-vector 모델은 stopword처럼 정보성이 낮은 토큰을 포함한 모든 query token에 대해 모든 page embedding과의 유사도를 계산한다. 논문은 이 역시 redundant computation의 원인이라고 지적한다.

PDF p. 5

벤치마크의 한계

MP-DocVQA, VisR-Bench, SlideVQA, MMLongBench-Doc, MMDocIR, LongDocURL, ViDoRe, ViDoSeek, OpenDocVQA, M3DocVQA, VisDoM 등 기존 VDR 벤치마크는 multi-document 설정과 long-context 설정을 동시에 만족하지 않는다. 대부분 단일 gold-standard 문서 내 검색을 가정하거나 문서 길이가 짧아, 대규모 long-document 컬렉션에서의 검색 효율성을 평가하기에 부적합하다.

PDF p. 5, 6

핵심 contribution

논문 주장

Method — HEAVEN

visual document retrieval의 efficiency-accuracy trade-off를 효과적으로 해결하는 plug-and-play two-stage hybrid-vector retrieval framework인 HEAVEN을 제안한다. 각 stage의 base encoder는 추가 학습 없이 교체 가능하다.

PDF p. 2

Benchmark — ViMDoc

visually rich, long-context, multi-document retrieval을 동시에 평가할 수 있는 벤치마크 ViMDoc을 제안한다. VisR-Bench, REAL-MM-RAG, MMLongBench-Doc, MMDocIR, LongDocURL 5개 소스에서 1,379개 문서·76,347페이지를 수집했으며, 총 10.9k 쿼리 중 0.7k가 cross-page 쿼리이고 문서당 평균 55.4페이지로 기존 벤치마크 중 가장 길다.

PDF p. 2, 5, 6

Experiments — 정확도 유지와 대폭적인 연산량 절감

4개 multi-document 벤치마크에서 HEAVEN은 평균적으로 multi-vector 모델 ColQwen2.5의 state-of-the-art Recall@1 성능의 99.87%를 유지하면서 쿼리당 FLOPs를 99.82% 절감한다. Recall@3 기준으로도 99.27%를 유지한다.

제안 방식의 전체 구조와 주요 구성요소

논문 주장

전체 구조: two-stage hybrid-vector framework

HEAVEN은 single-vector retrieval의 효율성과 multi-vector retrieval의 정확성을 결합한 two-stage 구조이다. Stage 1은 대규모 코퍼스에서 single-vector 모델로 coarse-grained candidate page 집합을 빠르게 선별하고, Stage 2는 축소된 candidate에 대해서만 multi-vector 모델로 fine-grained reranking을 수행한다. 이 설계는 Obs.2(넓은 top-K에서는 single-vector도 충분히 정확함)를 직접적으로 활용한 것으로, 비싼 multi-vector 연산이 적용되는 페이지 수를 구조적으로 제한한다. 또한 HEAVEN은 plug-and-play이므로 각 stage의 인코더를 재학습 없이 교체하여 특정 도메인에 적응하거나 더 강력한 모델로 업그레이드할 수 있다.

PDF p. 2, 3

구성요소 1: Visually-Summarized Pages (VS-pages)

indexing 단계에서 한 번만 구성되는 요약 페이지이다. Document Layout Analysis(DLA)로 각 페이지에서 title layout을 추출하고, 여러 페이지의 title layout들을 압축·조합하여 하나의 VS-page를 만든다. 문서를 더 적은 수의 요약 페이지로 표현함으로써 Stage 1에서 비교해야 할 대상 수 자체를 줄인다(|VS| < |P|). VS-page 구성은 query-time inference에 추가 overhead를 주지 않는다.

PDF p. 3, 4

구성요소 2: Stage 1 (Candidate VS-page Retrieval + Candidate Page Refinement)

single-vector 모델로 먼저 VS-page 수준 유사도를 계산해 candidate VS-page를 고르고, 이를 실제 page로 확장한 뒤 VS-page-level 점수와 page-level 점수를 가중합하여 상위 K개 페이지를 refined candidate set으로 확정한다.

PDF p. 3, 4

구성요소 3: Stage 2 (Reranking with Filtered Query Tokens + Reranking Refinement)

POS tagging으로 걸러낸 key query token만 사용해 candidate page를 1차 재정렬한 뒤, 전체 query token을 사용한 multi-vector 점수와 Stage 1의 single-vector 점수를 결합하여 최종 fine-grained reranking을 수행한다.

PDF p. 5

구성요소 4: Document Layout Analysis (DocLayout-YOLO)

VS-page 구성을 위해 각 페이지에서 layout을 검출하는 모듈이다. 8개 layout 카테고리 중 title 영역만 crop하여 사용한다.

PDF p. 4, 6, 12

구성요소 5: Key Token Filtering (POS tagging, nltk)

query token 중 명사·고유명사 등 linguistic 중요도가 높은 key token을 식별한다. key token은 전체 토큰의 약 30%(OpenDocVQA 31.5%, ViMDoc 30.3%)에 불과하며, 이를 통해 Stage 2 reranking의 연산량을 줄인다.

PDF p. 5, 12

단계별 동작 과정

논문 주장

Step 0 (Indexing 전용): VS-page 구성

문서 Dk의 각 페이지 Pk,i에 DLA를 적용해 title layout 집합 Tk,i = DLA(Pk,i)를 추출하고, 문서 전체의 layout을 Tk로 모은다. reduction factor r에 따라 Tk를 ⌈|Dk|/r⌉개 그룹으로 분할하고, 각 그룹을 Assemble(·) 함수로 하나의 VS-page로 조합한다: VSk^(j) = Assemble(Tk^(j)). 각 VS-page에 대해 그것이 요약하는 원본 페이지 집합 Γ(VSk^(j))를 기록해 둔다. 이 과정은 indexing 시점에 한 번만 수행되므로 query-time 추가 비용이 없다. 실험에서 r은 min(15, |Dk|)로 설정된다.

PDF p. 4, 7

Step 1: Candidate VS-page Retrieval

쿼리 q가 들어오면 single-vector 모델로 모든 VS-page와의 유사도 S_SV(q,VS) = <Ẽq, ẼVS>를 계산한다. 비교 대상이 원본 페이지가 아닌 VS-page이므로 계산량이 줄어든다. 점수 기준 상위 (p1×100)%를 candidate VS-page 집합 C̄로 선정한다(기본값 p1 = 0.5).

PDF p. 4, 7

Step 2: Candidate Page Refinement

선정된 VS-page들이 요약하던 실제 페이지들로 확장하여 C = ∪_{VS∈C̄} Γ(VS)를 만든다. 각 후보 페이지 P에 대해 VS-page-level 점수와 page-level 점수를 결합한 S*_SV(q,P) = α·S_SV(q,Γ⁻¹(P)) + (1−α)·S_SV(q,P)를 계산하고, 이 점수로 정렬해 상위 K개를 refined candidate set C_K로 확정한다(기본값 α = 0.1, K = 200; M3DocVQA는 document-level 평가이므로 α = 0.4).

PDF p. 4, 7

Step 3: Reranking with Filtered Query Tokens

POS tagging으로 쿼리에서 key token q_key(전체의 약 30%)를 식별하고, q_key만으로 C_K에 대한 multi-vector(MaxSim) 점수 S_MV(q_key, P)를 계산해 재정렬한다. 상위 (p2×100)%를 더 작은 candidate 집합 C*로 좁힌다(기본값 p2 = 0.25). 즉 비싼 multi-vector 연산을 '적은 페이지 × 적은 토큰'에 대해서만 수행한다.

PDF p. 5, 7

Step 4: Reranking Refinement

최종적으로 축소된 C*에 대해서만 전체 query token을 사용한 정밀한 multi-vector 점수 S_MV(q,P)를 계산하고, Stage 1에서 얻은 single-vector 점수 S*_SV(q,P)와 β로 가중합한 최종 점수 S*_MV(q,P)로 페이지를 재정렬하여 검색 결과를 산출한다(기본값 β = 0.3; M3DocVQA는 0.4).

PDF p. 5, 7

핵심 알고리즘 및 인덱스 구조

논문 주장

두 가지 scoring 함수와 복잡도

Single-vector scoring은 S_SV(q,P) = <Ẽq, ẼP>로 페이지당 dot product 한 번, 코퍼스 전체 O(d·|P|)이다. Multi-vector scoring은 S_MV(q,P) = Σ_{i=1..nq} max_{j∈{1..nP}} <Eq^(i), EP^(j)> (MaxSim)로 페이지당 O(d·nq·nP), 코퍼스 전체 O(d·nq·Σ_{P} nP)이다. HEAVEN의 절감 논리는 이 복잡도 식의 각 인자를 단계적으로 줄이는 데 있다.

PDF p. 2, 3

연산량 절감의 세 축

(1) VS-page는 Stage 1의 비교 대상 수를 |P|에서 |VS|(< |P|)로 줄인다. (2) Stage 1의 candidate 선별은 multi-vector 연산이 적용되는 페이지 수를 |P|에서 K(=200), 다시 p2를 통해 C*로 줄인다. (3) key token filtering은 1차 reranking에서 nq를 약 30%로 줄인다. 세 축이 곱셈적으로 작용하여 전체 FLOPs가 크게 감소한다. 이는 논문의 구조를 바탕으로 한 분석자의 정리이며, 각 요소의 개별 효과는 Table 4의 ablation으로 뒷받침된다.

PDF p. 4, 5, 8

인덱스 구조: VS-page와 원본 page의 양방향 매핑

인덱스는 원본 page embedding과 VS-page embedding을 함께 보관하고, VS-page → 연관 page 집합(Γ) 및 page → 소속 VS-page(Γ⁻¹)의 매핑을 유지한다. 이 매핑이 있어야 Step 2에서 VS-page-level 점수를 page-level 점수와 결합할 수 있다.

왜 key token만으로 충분한가

Figure 6에 따르면 key token은 전체의 약 30%(OpenDocVQA 31.5%, ViMDoc 30.3%)에 불과하지만 MRR 기준으로 전체 토큰 사용(OpenDocVQA 86.83, ViMDoc 81.39)에 비교적 근접한 성능(Key: 82.61, 71.56)을 내며, 같은 개수의 무작위 토큰 사용(Random: 72.80, 66.37)보다 뚜렷하게 우수하다. 즉 성능 유지의 원인이 단순한 토큰 수 감소 내성이 아니라 linguistic 중요도 기반 선택에 있음을 보인다.

구현 설정

HEAVEN은 Stage 1에 DSE, Stage 2에 ColQwen2.5를 사용하고, DLA에는 DocLayout-YOLO, POS tagging에는 nltk, textual baseline의 전처리 OCR에는 Tesseract를 사용한다. 하이퍼파라미터 기본값은 r = min(15,|Dk|), α = 0.1, β = 0.3, p1 = 0.5, p2 = 0.25, K = 200이며 M3DocVQA에서는 document-level 평가 때문에 α, β를 0.4로 둔다. 실험은 16개 Intel Xeon Platinum 8480C CPU와 단일 NVIDIA H200 GPU에서 수행되었고 모든 결과는 5회 실행 평균이다. Stage 1 단독 성능

Stage 1만 사용한 HEAVEN도 4개 벤치마크 평균에서 DSE 대비 Recall@1 101.74%, Recall@3 100.86%를 달성하면서 FLOPs는 49.31% 절감했다(0.079 vs 0.156). 즉 VS-page 기반 후보 선별은 기존 single-vector 검색을 대체하면서 더 싸고 더 정확할 수 있다. M3DocVQA에서는 Stage 1이 DSE를 Recall@1 기준 7.73% 초과하면서 FLOPs를 79.67% 절감했고, 이어서 Stage 2가 ColQwen2.5 대비 Recall@1을 2.27% 추가 향상시키면서 FLOPs를 99.81% 절감했다.

Ablation: 각 구성요소의 역할 분담

ViMDoc Stage 1 ablation에서 HEAVEN은 R@100 97.20 / R@200 97.96 / FLOPs 0.134이며, VS-page를 제거하면 정확도는 소폭 오르지만(97.68 / 98.51) FLOPs가 0.235로 크게 증가한다. 반면 candidate refinement를 제거하면 FLOPs는 0.017로 줄지만 정확도가 59.47 / 68.02로 급감한다. Stage 2에서도 HEAVEN(R@1 71.05 / R@3 86.41 / FLOPs 0.486) 대비 query filtering을 빼면 정확도는 거의 동일한데(71.08 / 86.38) FLOPs가 0.871로 약 1.8배 증가하고, reranking refinement를 빼면 정확도가 58.05 / 77.08로 급감한다. 즉 VS-page와 query filtering은 효율성 담당, refinement 두 가지는 정확도 담당으로 역할이 뚜렷이 나뉜다.

실측 latency

ViMDoc 기준 online retrieval 쿼리당 latency는 HEAVEN 2.412초로 ColQwen2.5의 2006.361초 대비 99.88% 빠르다(BGE-M3 multi-vector는 20953.468초). 다만 offline indexing에서는 VS-page 구성에 58.9분, 인코딩에 286.4분이 추가로 필요하여 DSE(146.7분)나 ColQwen2.5(129.1분)보다 전처리 비용이 크다. 즉 HEAVEN은 offline 비용을 지불해 online 비용을 사는 구조이다(후반부 해석은 분석자 정리).

효율화 baseline 및 plug-and-play 비교

Figure 7에서 pooling/pruning 기반 효율화 변형들은 FLOPs를 줄일수록 정확도가 하락하는 반면 HEAVEN은 훨씬 적은 FLOPs에서 더 높은 정확도를 달성하여 가장 좋은 efficiency-accuracy trade-off를 보인다. Plug-and-play 분석(Table 6)에서는 DSE(Stage 1) + ColQwen2.5(Stage 2) 조합이 ViMDoc R@1 71.05 / R@3 86.41 / FLOPs 0.486, OpenDocVQA R@1 71.56 / R@3 84.53 / FLOPs 0.541로 최고 성능을 보여 기본 설정으로 채택되었다. 하이퍼파라미터 분석(Figure 8, 9)에서는 p1, p2, r, α, β 전반에 걸쳐 견고하며, 특히 문서가 긴 ViMDoc에서 reduction factor의 영향이 더 두드러진다고 서술한다.

cross-page 쿼리에서의 거동

ViMDoc의 cross-page 쿼리(677개, 전체의 6.2%)에서 HEAVEN은 Recall@1 25.22로 ColQwen2.5의 25.92에 근접하고, single-page 쿼리에서는 73.58 vs 74.10으로 유사한 격차를 유지한다. 두 경우 모두 FLOPs는 0.486 대 407.320으로 크게 절감된다. 즉 여러 페이지에 정보가 흩어진 어려운 쿼리에서도 hybrid 구조가 multi-vector 대비 정확도 손실 없이 작동한다.

결론 및 종합 해석

분석자 해석

논문의 결론

논문은 HEAVEN이 multi-vector 모델과 비교 가능한 정확도를 유지하면서 single-vector 모델을 크게 상회하는 성능을 보여, efficiency와 accuracy 사이의 강력한 trade-off를 확보했다고 결론짓는다. 또한 ViMDoc 벤치마크를 통해 long-context multi-document 환경에서 textual 방법과 visual 방법 사이의 뚜렷한 성능 격차가 드러나며, 이는 visual document retrieval 고유의 난이도를 보여준다고 주장한다.

PDF p. 8, 9

분석자 해석: 새로운 모델이 아니라 새로운 배치

HEAVEN의 핵심 기여는 새로운 인코더를 학습시키는 것이 아니라, 기존 single-vector·multi-vector 인코더를 관찰(Obs.1, Obs.2)에 근거해 재배치한 retrieval 파이프라인 설계에 있다. 학습이 필요 없다는 점은 도입 장벽을 낮추지만, 동시에 최종 정확도의 상한이 사용된 base encoder의 성능에 종속된다는 뜻이기도 하다. 실제로 Table 3에서 HEAVEN의 Recall은 대부분의 경우 Stage 2 인코더인 ColQwen2.5를 넘지 못하고 그에 근접하는 형태를 보인다(M3DocVQA는 예외).

PDF p. 3, 7, 9

분석자 해석: 비용 이동의 관점

Table 5를 함께 읽으면 HEAVEN은 총 연산량을 없앤다기보다 query-time에서 index-time으로 비용을 옮긴 설계로 볼 수 있다. VS-page 구성(58.9분)과 인코딩(286.4분)이라는 일회성 offline 비용을 지불하는 대신, 쿼리마다 반복되는 online 비용을 2000초대에서 2.4초대로 낮춘다. 따라서 쿼리 볼륨이 크고 코퍼스가 상대적으로 정적인 환경에서 이득이 가장 크며, 문서가 빈번히 갱신되는 환경에서는 재indexing 비용을 별도로 고려해야 한다.

  • 저자 명시 한계 HEAVEN은 pretrained vision-language encoder에 의존하며, 성능이 모델 규모(model scale)와 도메인 적응 품질(domain adaptation quality)에 따라 달라질 수 있다.
    PDF p. 9
  • 저자 명시 한계 Stage 1에서 사용하는 visually-summarized page(VS-page)는 document layout analysis에 의존하므로, layout이 noisy하거나 불규칙한 문서에서는 민감하게 영향을 받을 수 있다.
    PDF p. 9
  • 저자 명시 한계 HEAVEN은 retrieval 효율성에 초점을 맞추고 있으며 아직 retrieval-augmented generation을 통합하지 않았다. 저자들은 이를 future work로 남긴다.
    PDF p. 9
  • 분석자 해석 VS-page는 title layout만 crop하여 구성된다(DocLayout-YOLO의 8개 카테고리 중 title). 따라서 제목이 없거나 제목이 내용을 잘 대표하지 못하는 문서 — 예를 들어 표·차트가 지배적인 페이지나 슬라이드형 문서 — 에서는 요약 품질이 떨어질 가능성이 있다. 논문은 title을 선택한 이유를 제시하지만 다른 layout 카테고리를 함께 쓴 비교는 확인되지 않는다.
    PDF p. 4, 6, 12
  • 분석자 해석 cross-page 쿼리의 절대 성능이 낮다. ViMDoc cross-page 쿼리에서 HEAVEN의 Recall@1은 25.22로 single-page의 73.58에 크게 못 미친다. 다만 ColQwen2.5도 25.92로 유사하게 낮으므로 이는 HEAVEN 고유의 결함이라기보다 여러 페이지에 정보가 분산된 쿼리 자체의 난이도로 보이며, 페이지 단위 검색 패러다임 전반의 구조적 한계로 해석된다.
    PDF p. 16
  • 분석자 해석 offline indexing 비용이 baseline보다 크다. HEAVEN은 VS-page 구성 58.9분에 encode 286.4분이 필요하여 DSE(146.7분)나 ColQwen2.5(129.1분)를 상회한다. 문서가 자주 추가·갱신되는 환경에서는 재indexing 비용이 online 절감분을 상쇄할 수 있으나, 논문은 동적 코퍼스 시나리오를 다루지 않는다.
    PDF p. 8
  • 분석자 해석 정확도 상한이 Stage 2 인코더에 종속되는 구조이다. Stage 1에서 탈락한 페이지는 Stage 2가 복구할 수 없으므로 최종 Recall은 Stage 1의 Recall@K(ViMDoc R@200 = 97.96)에 의해 상한이 정해진다. 실제로 Table 3의 대부분 벤치마크에서 HEAVEN은 ColQwen2.5에 근접하되 이를 넘지 못한다.

핵심 용어

HEAVEN (Hybrid-vector retrieval for Efficient and Accurate Visual multi-documENt)
논문이 제안한 two-stage hybrid-vector retrieval framework. Stage 1의 single-vector 후보 선별과 Stage 2의 multi-vector 재정렬을 결합하며, 추가 학습 없이 각 stage의 인코더를 교체할 수 있는 plug-and-play 구조이다. PDF p. 2, 3
Visual Document Retrieval (VDR)
차트, 표, 그림 등 시각적으로 복잡한 요소를 포함한 문서 페이지를 이미지 단위로 인덱싱하여 쿼리에 대응하는 페이지를 찾는 검색 과제. 텍스트 추출(OCR)에 의존하지 않고 페이지 이미지를 직접 인코딩하는 접근이 최근 주류이다. PDF p. 1, 2
Single-Vector Retrieval
쿼리와 페이지를 각각 하나의 d차원 벡터로 표현하고 dot product로 관련도를 계산하는 방식. S_SV(q,P) = <Ẽq, ẼP>이며 코퍼스 전체 채점에 O(d·|P|)가 든다. 효율적이지만 token/patch 수준의 세밀한 상호작용을 포착하지 못한다. PDF p. 2, 3
Multi-Vector Retrieval / MaxSim
쿼리 토큰과 페이지 패치의 모든 쌍에 대해 유사도를 구하고, 각 쿼리 토큰마다 가장 유사한 패치의 점수를 골라 합산하는 late interaction 방식. S_MV(q,P) = Σ_i max_j <Eq^(i), EP^(j)>이며 페이지당 O(d·nq·nP)가 든다. 정확하지만 매우 비싸다. PDF p. 2, 3
Visually-Summarized Page (VS-page)
HEAVEN이 도입한 요약 페이지. 여러 페이지에서 DLA로 추출한 title layout들을 Assemble(·)로 한 장에 조합한 것으로, indexing 시점에 한 번만 만들어지며 Stage 1의 비교 대상 수를 |P|에서 |VS|(< |P|)로 줄인다. PDF p. 3, 4
Reduction Factor r
몇 개의 연속 페이지를 하나의 VS-page로 요약할지 제어하는 하이퍼파라미터. 문서 Dk는 ⌈|Dk|/r⌉개의 VS-page로 표현된다. 실험 기본값은 min(15, |Dk|)이다. PDF p. 4, 7
Document Layout Analysis (DLA)
페이지 이미지에서 제목, 본문, 표, 그림 등 layout 영역을 검출하는 기술. HEAVEN은 DocLayout-YOLO를 사용하며 8개 카테고리 중 title 영역만 crop하여 VS-page 재료로 삼는다. PDF p. 4, 6, 12
Key Token Filtering
POS tagging(nltk)으로 쿼리에서 명사·고유명사 등 linguistic 중요도가 높은 토큰만 선별하는 기법. 전체 토큰의 약 30%(OpenDocVQA 31.5%, ViMDoc 30.3%)만 남겨 Stage 2 1차 reranking의 연산량을 줄인다. PDF p. 5, 12
Candidate Page Refinement
Stage 1에서 candidate VS-page를 원본 page로 확장한 뒤, VS-page-level 점수와 page-level 점수를 α로 가중합한 S*_SV(q,P)로 재정렬하여 상위 K개를 확정하는 단계. 제거 시 Recall@200이 97.96에서 68.02로 급락하는 핵심 정확도 요소이다. PDF p. 4, 8
Reranking Refinement
Stage 2 마지막 단계로, 좁혀진 후보 C*에 대해 전체 query token 기반 multi-vector 점수와 Stage 1의 single-vector 점수를 β로 가중합해 최종 순위를 정한다. 제거 시 ViMDoc Recall@1이 71.05에서 58.05로 급락한다. PDF p. 5, 8
ViMDoc
저자들이 제안한 visually rich, long-context, multi-document retrieval 벤치마크. 5개 소스에서 1,379개 문서·76,347페이지를 수집했고 총 10.9k 쿼리(cross-page 0.7k), 문서당 평균 55.4페이지로 기존 벤치마크 중 가장 길다. PDF p. 2, 5, 6
Cross-page Query
답을 얻기 위해 한 문서 내 여러 페이지의 정보를 함께 참조해야 하는 쿼리. ViMDoc에는 677개(전체의 6.2%)가 포함되며, 모든 모델에서 single-page 쿼리보다 성능이 크게 낮다. PDF p. 16
Plug-and-Play
HEAVEN의 설계 속성으로, 각 stage의 base encoder를 재학습 없이 다른 pretrained 모델로 교체할 수 있음을 뜻한다. 도메인 적응이나 모델 업그레이드가 용이하다. PDF p. 3, 9
FLOPs (per query)
쿼리 하나를 처리하는 데 필요한 부동소수점 연산 수(단위: billions). 논문의 주요 효율성 지표이며, batch padding이 결과를 왜곡하지 않도록 임베딩을 사전 처리한 뒤 계산했다. PDF p. 7, 13
