RAGCache: Efficient Knowledge Caching for Retrieval-Augmented Generation

핵심 요약

RAGCache는 Retrieval-Augmented Generation(RAG) 서빙을 위한 multilevel dynamic caching system이다. RAG는 검색된 문서를 요청에 주입(knowledge injection)하기 때문에 입력 시퀀스가 매우 길어지고, 그 결과 prefill 단계의 계산·메모리 비용이 급격히 커진다. 논문에 따르면 원 요청이 100 토큰이고 검색된 문서가 1000 토큰이면 augmented request의 비용은 원 요청 대비 10배 이상(>10×)이 될 수 있다. 저자들은 시스템 특성 분석(system characterization)을 통해 두 가지 관찰을 제시한다. 첫째, prefill이 지배적 병목이며 시퀀스 길이가 4000 토큰을 넘으면 추론 시간이 약 1초에 달한다(LLaMA2-7B, A10G GPU). 둘째, 실제 QA 워크로드의 문서 검색 패턴은 소수 문서에 심하게 편중(skewed)되어 있어 MMLU에서는 상위 3% 문서가 전체 요청의 60%를 차지한다. 이 두 관찰로부터 '자주 재사용되는 문서의 key-value(KV) tensor를 캐싱하면 중복 prefill 연산을 제거할 수 있다'는 설계 동기가 도출된다. RAGCache는 문서 ID 기반 prefix tree인 knowledge tree로 KV tensor를 조직하고, GPU 메모리와 host 메모리의 2계층에 걸쳐 배치한다. 핵심 기법은 (1) 노드의 frequency, size, prefix-aware recomputation cost, recency를 종합한 priority(=Clock + Frequency×Cost/Size)로 축출을 결정하는 prefix-aware GDSF(PGDSF) replacement policy, (2) 캐시된 길이 비율을 기준으로 대기 요청 순서를 바꾸는 cache-aware reordering, (3) CPU의 vector retrieval과 GPU의 generation을 오버랩하는 dynamic speculative pipelining이다. 평가에서 RAGCache는 Faiss와 결합된 vLLM 대비 TTFT를 최대 4배 낮추고 throughput을 최대 2.1배 높였으며, 최신 caching 시스템인 SGLang 대비로도 TTFT 최대 3.5배, throughput 최대 1.8배의 개선을 보였다.

해결하려는 문제

논문 주장

Knowledge injection이 만드는 long sequence 비용

RAG 시스템은 사용자 질의에 대해 외부 지식 데이터베이스에서 관련 문서를 검색한 뒤, 이를 프롬프트에 덧붙여 LLM에 입력한다. 이 knowledge injection 과정에서 입력 시퀀스가 크게 길어지며 계산 및 메모리 비용이 함께 증가한다. 논문은 원 요청이 100 토큰이고 검색된 문서가 총 1000 토큰인 경우, augmented request의 추가 계산/메모리 비용이 원 요청 대비 10배 이상(>10×)이 될 수 있다고 명시한다.

PDF p. 1

동일 문서에 대한 반복적 재계산

여러 요청이 동일한 문서를 반복적으로 검색하더라도, 기존 RAG 시스템은 매 요청마다 해당 문서의 key-value(KV) tensor를 처음부터 다시 계산한다. 즉 문서 부분의 prefill 연산이 요청 간에 중복 수행된다. 논문이 다루는 문제는 이 검색된 지식의 중간 상태(intermediate states, 곧 KV tensor)를 GPU 메모리와 host 메모리 계층에 걸쳐 효율적으로 캐싱하고 여러 쿼리에 공유하는 것이다.

PDF p. 1, 2

관찰 1: prefill이 지배적 병목

저자들의 시스템 특성 분석에 따르면 LLM 추론은 prefill 단계와 decoding 단계로 나뉘며, RAG처럼 입력이 긴 워크로드에서는 prefill이 지배적 병목이 된다. LLaMA2-7B와 A10G GPU 환경에서 시퀀스 길이가 4000 토큰을 넘으면 추론 시간이 약 1초에 도달한다(Figure 2). 또한 Wikipedia 기반 문서 집합(약 0.3 million documents)에서 문서 길이의 평균은 3717.52 토큰으로, MMLU 요청 길이 평균 348.04 토큰보다 훨씬 길다(Figure 3). 즉 전체 입력 길이의 대부분을 검색된 문서가 차지한다.

PDF p. 3

관찰 2: 편중된(skewed) 문서 접근 패턴

저자들은 MMLU, Google Natural Questions, HotpotQA, TriviaQA 네 개의 QA 데이터셋에 대해 문서 검색 패턴(retrieval pattern)을 분석하였고, 접근이 소수 문서에 강하게 편중되어 있음을 확인했다. MMLU의 경우 상위 3% 문서가 전체 요청의 60%를 차지하며, 이는 균등분포 대비 20배 낮은 수치이다(Figure 5(a)). 이 편중성은 캐싱이 높은 hit rate를 낼 수 있음을 시사하며 RAGCache 설계의 직접적 동기가 된다.

PDF p. 4, 5

기존 방식의 한계

논문 주장

RAG 특성을 고려하지 않는 GPU-only 캐싱

vLLM은 단일 요청 내 여러 generation iteration을 위해 intermediate state를 관리·공유하여 재계산 비용을 줄이고, SGLang은 이를 서로 다른 요청 간(across requests) KV cache 재사용으로 확장한다. 그러나 이들은 RAG의 특성을 고려하지 않고 intermediate state를 GPU 메모리에만 캐싱한다. Augmented request의 시퀀스가 매우 길다는 점을 감안하면 GPU 메모리 용량이 제한적이어서 캐시 가능한 문서 수가 적고, 결과적으로 최적화되지 못한 성능(suboptimal performance)에 그친다.

PDF p. 1, 2

Retrieval과 inference의 순차 실행

기존 RAG 시스템에서는 CPU에서 수행되는 vector retrieval과 GPU에서 수행되는 LLM inference가 서로 독립적인 두 단계로 순차 실행된다. 그 결과 retrieval이 진행되는 동안 GPU 자원이 유휴(idle) 상태로 남고, end-to-end latency가 불필요하게 길어진다.

PDF p. 2

단순 replacement policy

비교 대상인 SGLang은 GPU 메모리 내에서 요청 간 KV cache 재사용을 지원하지만 replacement policy로 LRU를 사용한다. LRU는 문서 prefix마다 다른 크기(size), 접근 빈도(frequency), 재계산 비용(recomputation cost)을 반영하지 못한다는 점이 RAGCache가 PGDSF를 제안하는 배경이 된다.

PDF p. 5, 11

핵심 contribution

논문 주장

RAG에 대한 상세한 system characterization

RAG 시스템에 대한 상세한 특성 분석을 수행하여 성능 병목(performance bottleneck)과 최적화 기회(optimization opportunities)를 규명한다. prefill 지배성, 문서 길이 분포, 편중된 검색 패턴, prefix caching의 효과가 여기에 해당한다.

PDF p. 2, 3, 4, 5

최초의 지식 캐싱 RAG 시스템

저자들이 아는 한, 외부 지식(external knowledge)의 intermediate states를 캐싱하여 여러 쿼리에 걸쳐 공유함으로써 중복 연산을 줄이는 최초의 RAG 시스템인 RAGCache를 제안한다.

PDF p. 2

PGDSF와 dynamic speculative pipelining

RAG의 특성을 활용하여 cache miss rate를 최소화하는 prefix-aware Greedy-Dual-Size-Frequency(PGDSF) replacement policy를 설계하고, end-to-end latency를 최소화하기 위해 retrieval과 generation을 오버랩하는 dynamic speculative pipelining 기법을 설계한다.

PDF p. 2

프로토타입 구현과 실증

RAGCache 프로토타입을 구현하고 평가하였다. Faiss와 결합된 vLLM 대비 TTFT를 최대 4배 낮추고 throughput을 최대 2.1배 향상시켰으며, LLM 애플리케이션용 최신 caching 시스템인 SGLang 대비 TTFT를 최대 3.5배 낮추고 throughput을 최대 1.8배 높였다.

제안 방식의 전체 구조

논문 주장

전체 개요

RAGCache는 RAG에 특화된 multilevel dynamic caching system이다. 검색된 문서의 KV tensor를 문서 ID 기반 prefix tree인 knowledge tree(Figure 8)로 구조화하고, GPU 메모리(빠르지만 작음)와 host 메모리(느리지만 큼)의 두 계층에 걸쳐 배치하여 여러 요청 간에 공유한다. 시스템은 기존의 CPU 기반 external knowledge database(Faiss 기반 벡터 검색)와 GPU 기반 LLM 추론 엔진(vLLM 기반)을 통합 대상으로 삼고, 그 위에 캐싱 계층과 스케줄링 계층을 추가하는 형태이다(Figure 7).

PDF p. 4, 5

RAG Controller

RAG controller는 외부 지식 데이터베이스와 LLM 추론 엔진 간의 상호작용을 조정(orchestrate)하는 중심 구성요소이다. PGDSF replacement policy, cache-aware reordering, dynamic speculative pipelining을 통합 관리한다.

PDF p. 4

Knowledge Tree와 Cache Retriever

LLM 추론에서 KV tensor는 앞선 토큰들에 의존하므로, 동일한 문서라도 앞에 어떤 문서가 놓였는지(문서 순서)에 따라 KV 값이 달라진다. Knowledge tree는 이 순서 민감성을 문서 ID 기반 prefix tree로 표현하여, 서로 다른 요청이 동일한 prefix를 공유하는 경우를 트리 경로의 중첩으로 자연스럽게 나타낸다. Cache retriever는 prefix matching을 통해 트리 높이 h에 대해 O(h) 시간복잡도로 재사용 가능한 KV tensor를 조회한다.

PDF p. 4, 5

PGDSF Replacement Policy

각 노드에 대해 접근 빈도(frequency), 크기(size), prefix를 고려한 재계산 비용(prefix-aware recomputation cost), 최근성(clock)을 종합한 priority를 Formula 1(priority = Clock + Frequency × Cost / Size)로 계산한다. GPU 메모리가 가득 차면 leaf 노드 중 priority가 가장 낮은 노드를 host memory로 swap-out한다(Algorithm 1). 이때 데이터 복사는 최초 1회만 수행되고 이후 축출 시에는 GPU에서 바로 free 한다.

PDF p. 5, 6

Cache-aware Reordering

OrderPriority = Cached Length / Computation Length 기준의 priority queue를 사용해 대기 중인 요청의 처리 순서를 재배열한다. 캐시된 비율이 높은 요청을 먼저 처리하여 cache hit rate를 높이고 cache thrashing을 방지하며, 재배열 window 크기를 제한하여 request starvation을 막는다.

PDF p. 7

Dynamic Speculative Pipelining

벡터 검색이 진행되는 도중 산출되는 중간 top-k 후보 문서를 이용해 LLM이 조기에 speculative generation을 시작하게 함으로써 retrieval 단계와 generation 단계를 오버랩한다. 무조건 speculation 하는 것이 아니라 대기 중인 prefill 요청 수(max_prefill_bs)를 기준으로 speculation 시작 여부를 동적으로 결정한다. Theorem 5.1은 vector search와 LLM이 각각 한 번에 하나의 요청만 처리하는 단순화된 G/G/1 모델에서 request pool이 비었을 때 speculation을 시작하는 전략의 최적성을 보이며, 실제 구현(Algorithm 2)은 이 결과를 바탕으로 대기 중인 prefill 요청 수가 max_prefill_bs보다 작을 때 speculation을 허용하는 정책을 사용한다.

단계별 동작 과정

논문 주장

1단계 — 문서 검색

요청이 도착하면 RAG controller가 external knowledge database에 질의하여 관련 문서를 검색한다.

PDF p. 4

2단계 — 캐시 조회

검색된 문서 목록이 cache retriever로 전달되고, cache retriever는 knowledge tree에서 prefix matching으로 일치하는 KV tensor를 탐색한다.

PDF p. 4, 5

3단계 — Cache miss 경로

해당 문서 시퀀스의 KV tensor가 캐시에 없으면 LLM 추론 엔진에 전체 prefill 계산을 지시하여 토큰을 생성한다.

PDF p. 4

4단계 — Cache hit 경로

KV tensor가 캐시에 있으면 해당 tensor와 함께 요청이 LLM 추론 엔진으로 전달되고, prefix caching kernel을 이용해 캐시되지 않은 나머지 부분만 계산하여 토큰을 생성한다.

PDF p. 4

5단계 — 캐시 갱신과 축출

첫 토큰 생성 후 KV tensor가 RAG controller로 반환된다. Controller는 접근된 문서의 tensor를 캐시에 저장하고 PGDSF policy(Formula 1, Algorithm 1)에 따라 priority와 clock 등 캐시 상태를 갱신한다. GPU 메모리가 가득 찬 경우 priority가 가장 낮은 leaf 노드를 host memory로 swap-out하며, 데이터 복사는 최초 1회만 발생하고 이후 축출 시에는 GPU 메모리에서 곧바로 해제한다.

PDF p. 4, 5, 6

6단계 — 요청 재정렬

요청 큐가 포화 상태일 때 cache-aware reordering이 OrderPriority(=Cached Length / Computation Length) 기준으로 요청 순서를 재배열하여 cache hit rate를 높인다. 윈도우 크기를 제한하여 starvation을 방지한다.

PDF p. 7

7단계 — Speculative pipelining

retrieval 지연이 큰 경우 dynamic speculative pipelining이 vector search의 중간 top-k 후보를 단계적으로 LLM에 전달하여 speculative generation을 조기 시작한다. 후보 목록이 갱신되면 이전 생성을 종료하고 재시작하며, 최종 검색 결과가 speculation에 사용한 후보와 일치하면 생성 결과를 그대로 반환하고 다르면 재생성한다.

PDF p. 7, 8

8단계 — 응답 반환

생성된 답변이 최종 응답으로 사용자에게 전달된다.

PDF p. 4

핵심 알고리즘 및 인덱스 구조

논문 주장

Knowledge tree (인덱스 구조)

Knowledge tree는 문서 ID를 키로 하는 prefix tree이다. 루트에서 특정 노드까지의 경로는 하나의 문서 시퀀스를 의미하며, 그 노드는 해당 시퀀스에 대응하는 KV tensor를 보관한다. LLM의 attention이 prefix에 의존하므로 [D1, D2]와 [D2, D1]의 KV tensor는 서로 다르며, 트리 구조는 이 순서 민감성을 정확히 표현한다. 조회는 prefix matching으로 이루어지며 트리 높이 h에 대해 O(h)이다(Figure 8).

PDF p. 4, 5

PGDSF priority 계산 (Formula 1, Algorithm 1)

각 노드의 priority는 priority = Clock + Frequency × Cost / Size 로 정의된다. Clock은 최근성(recency)을 반영하는 논리 시계 값이고, Frequency는 접근 빈도, Cost는 해당 prefix를 재계산하는 데 드는 비용, Size는 KV tensor가 차지하는 메모리 크기이다. 즉 자주 접근되고 재계산 비용이 크며 크기가 작은 노드일수록 높은 priority를 갖는다. GPU 메모리가 부족하면 leaf 노드 중 priority 최솟값을 가진 노드를 host memory로 swap-out 한다. leaf만 축출 대상으로 삼는 이유는 부모 노드가 자식 노드의 prefix이므로 부모를 먼저 제거하면 자식이 무효화되기 때문이다(분석자 해석). 데이터 복사는 최초 축출 시 1회만 수행되며, 이미 host에 사본이 있으면 GPU 메모리를 바로 해제한다.

PDF p. 5, 6

Cache-aware reordering 기준

OrderPriority = Cached Length / Computation Length. 이미 캐시된 토큰 길이의 비율이 높은 요청일수록 우선 처리된다. 이 기준은 실제로 계산해야 할 토큰이 적은 요청을 먼저 내보내 GPU 시간을 절약하고, 동시에 서로 다른 prefix를 오가며 캐시를 밀어내는 cache thrashing을 억제한다. 무한정 밀리는 요청을 막기 위해 재배열 대상 window 크기를 제한한다(평가에서는 window=32 사용).

PDF p. 7, 11, 12

Dynamic speculative pipelining (Algorithm 2, Theorem 5.1)

Speculation은 잘못된 후보에 대해 GPU 연산을 낭비할 위험이 있으므로 항상 유리하지는 않다. Theorem 5.1은 vector search와 LLM이 각각 한 번에 하나의 요청만 처리한다는 단순화된 G/G/1 queue 모델에서, request pool이 비었을 때 speculative generation을 시작하고 비어 있지 않으면 pool이 빌 때까지 vector search를 계속하는 전략이 최적임을 보인다. 실제 RAG 시스템은 batch size와 병렬 vector search로 인해 이 단순 모델보다 복잡하므로, Algorithm 2는 이 이론적 결과를 바탕으로 대기 중인 prefill 요청 수가 max_prefill_bs보다 작을 때에만 speculation을 허용하는 정책(휴리스틱)을 사용한다.

평가 지표

주 지표는 평균 Time To First Token(TTFT)과 throughput이다. 여기서 throughput은 TTFT가 특정 임계값(최저 request rate에서 측정된 TTFT의 5배) 미만으로 유지되는 최대 request rate로 정의된다. 추가로 ablation에서는 cache hit rate와 RAGCache의 scheduling time(knowledge tree 조회·갱신, 요청 재정렬, speculative pipelining 결정에 소요되는 시간)을 측정한다. Cache hit rate는 검색된 문서 수 대비 hit한 문서 수의 비율로 정의된다(예: 저장된 시퀀스가 [D1, D2]이고 요청이 [D1, D3]이면 hit rate 50%).

End-to-end TTFT 및 throughput

동일한 request rate 조건에서 RAGCache는 vLLM 대비 평균 TTFT를 1.2–4× 낮추고 SGLang 대비 1.1–3.5× 낮춘다. SGLang 대비 이득이 존재한다는 점은 GPU-only 캐싱을 host memory까지 확장한 것과 replacement policy 개선의 효과가 실제로 유효함을 시사한다(분석자 해석).

SGLang 대비로는 1.2–2.6× 우수한 평균 TTFT를 보인다(2×H800, Figure 16).

Ablation — PGDSF replacement policy

host memory 8–128 GiB 구간에서 PGDSF는 GDSF 대비 1.02–1.32×, LRU 대비 1.06–1.62×, LFU 대비 1.06–1.75× 높은 cache hit rate를 달성한다(top-2 retrieval, request rate=0.8 req/s, Figure 17). 이 hit rate 향상은 평균 TTFT 1.05–1.29× 감소로 이어지며, host memory 8 GiB·MMLU 조건에서 평균 TTFT는 PGDSF 1.38초, GDSF 1.68초, LRU 1.78초, LFU 1.81초이다(Table 2). 저자들은 PGDSF가 문서 prefix별로 다른 크기, 접근 패턴, 재계산 비용을 함께 포착하기 때문이라고 설명한다.

Ablation — cache-aware reordering

cache-aware reordering을 적용하면 RAGCache는 평균 TTFT를 1.2–2.1× 감소시킨다(MMLU request rate=2.5 req/s, Natural Questions request rate=1.4 req/s, reordering window=32, host memory 16–128 GiB, Figure 18). 실험 조건이 상대적으로 높은 request rate라는 점은 이 기법이 큐가 포화된 상황을 겨냥한 최적화임을 보여준다(분석자 해석).

결론과 시사점

논문 주장

논문의 결론

RAGCache는 RAG 시스템의 지배적 병목이 검색된 문서로 인한 긴 prefill이라는 점과 실제 문서 접근이 심하게 편중되어 있다는 점을 결합하여, 문서 KV tensor를 knowledge tree로 조직하고 GPU–host 두 계층에 캐싱하는 접근을 제시한다. PGDSF, cache-aware reordering, dynamic speculative pipelining 세 기법이 결합되어 vLLM 대비 TTFT 최대 4×, throughput 최대 2.1×, SGLang 대비 TTFT 최대 3.5×, throughput 최대 1.8×의 개선을 달성한다.

PDF p. 2, 10, 13

시스템 설계 관점의 시사점 (분석자 해석)

이 논문의 핵심 통찰은 RAG 워크로드의 입력이 '임의의 긴 텍스트'가 아니라 '재사용 가능한 문서들의 순열'이라는 구조를 가진다는 점이다. 이 구조 덕분에 일반적인 토큰 단위 prefix caching 대신 문서 ID 단위의 트리 인덱스를 쓸 수 있고, 결과적으로 조회 비용이 O(h)로 낮아지며 replacement policy도 문서 단위 통계(빈도, 크기, 재계산 비용)를 활용할 수 있다. 또한 캐시 계층 확장(GPU→host)과 스케줄링(재정렬, 파이프라이닝)을 함께 다루었다는 점에서, 캐시 정책만 개선하는 접근보다 넓은 설계 공간을 다루고 있다. 다만 이 구조적 이점은 문서 순열의 재사용 가능성에 의존하므로 top-k가 커질수록 약화된다.

한계

  • 저자 명시 한계 Dynamic speculative pipelining의 이론적 근거(Theorem 5.1)는 vector search와 LLM이 각각 한 번에 하나의 요청만 처리한다고 가정하는 단순화된 G/G/1 queue 분석에서 도출되며, 이 단순 모델에서의 pool-empty 전략의 최적성만을 증명한다. 실제 RAG 시스템은 더 큰 LLM batch size와 병렬 vector search로 인해 더 복잡하며, Algorithm 2의 max_prefill_bs 기반 정책은 이 이론적 결과를 바탕으로 한 휴리스틱으로 봐야 한다. 요청이 다른 요청들과 배치로 묶인 경우 잘못된 speculative generation을 즉시 종료할 수 없다.
    PDF p. 8
  • 저자 명시 한계 Knowledge tree는 LLM 추론의 prefix 민감성 때문에 GPU 장애 발생 시 하위 노드들이 무효화되어 트리 전체가 영향을 받을 수 있다. 저자들은 가장 자주 접근되는 상위 레벨 노드 일부(예: system prompt)만 host memory에 복제하고 timeout 기반 재시도 메커니즘을 사용하며, 전체 노드를 완전히 복제하지는 않는다고 명시한다.
    PDF p. 9
  • 저자 명시 한계 top-k 값이 커질수록 문서 순열(permutation) 조합이 factorial로 증가하여 문서 시퀀스의 재사용 가능성이 낮아진다. RAGCache는 실제 top-k보다 낮은 k로 문서를 캐싱하는 방식(예: top-5 요청에 대해 top-3 문서만 캐싱)으로 hit rate와 cache efficiency의 균형을 맞출 뿐, 문제를 근본적으로 해결하지는 않는다.
    PDF p. 12
  • 저자 명시 한계 RAGCache는 주로 TTFT(prefill 단계 지연) 감소에 초점을 맞춘다. TPOT(time per output token, decoding 단계)에 대해서는 prefill iteration 가속의 부수적 효과로 낮아질 수 있다고만 언급되며 별도의 직접적 최적화 기법은 제시되지 않는다.
    PDF p. 12
  • 분석자 해석 평가 워크로드가 MMLU와 Natural Questions 두 QA 데이터셋에 한정되어 있다. 동기 분석(3장)에서는 HotpotQA와 TriviaQA의 검색 패턴도 함께 분석했으나 end-to-end 성능 평가에는 사용되지 않았으므로, 접근 편중성이 약한 워크로드에서의 성능은 확인되지 않음.
    PDF p. 4, 5, 10
  • 분석자 해석 지식 베이스가 정적(static)인 상황을 전제로 한 것으로 보인다. 문서가 갱신·삭제되는 경우 해당 문서를 prefix로 포함하는 knowledge tree 하위 노드들의 무효화 처리 비용이 문제가 될 수 있으나, 이에 대한 메커니즘이나 측정은 본문에서 확인되지 않음.
    PDF p. 4, 5
  • 분석자 해석 top-5 실험에서 GPU 용량 제한으로 문서를 truncate 하였다고 명시되어 있어, 해당 조건의 결과는 다른 top-k 설정과 완전히 동일한 기준에서 비교되지 않을 가능성이 있다.
    PDF p. 10, 11
  • 분석자 해석 PGDSF의 priority 계산에서 Cost(재계산 비용)는 cached/non-cached token 길이별 LLM prefill 시간을 offline profiling한 뒤 bilinear interpolation으로 추정한다(Algorithm 1). 다만 이 추정 오차가 hit rate나 TTFT에 어떤 영향을 주는지에 대한 별도의 민감도 분석은 확인되지 않음.

핵심 용어

RAG (Retrieval-Augmented Generation)
사용자 질의에 대해 외부 지식 데이터베이스에서 관련 문서를 검색한 뒤 이를 프롬프트에 주입하여 LLM이 답변을 생성하도록 하는 방식. 생성 품질은 높아지지만 입력 시퀀스가 길어져 계산·메모리 비용이 증가한다. PDF p. 1, 2
Knowledge injection
검색된 문서를 원 요청에 덧붙여 augmented request를 만드는 과정. 논문 예시에서 100 토큰 요청에 1000 토큰 문서가 주입되면 비용이 10배 이상 증가할 수 있다. PDF p. 1
KV tensor / KV cache
Transformer의 attention 계산에 쓰이는 key-value 텐서. 한 번 계산해 두면 이후 토큰 생성에서 재사용할 수 있으며, RAGCache가 캐싱 대상으로 삼는 intermediate state가 바로 이것이다. PDF p. 1, 2, 3
Prefill / Decoding
LLM 추론의 두 단계. prefill은 입력 프롬프트 전체의 KV tensor를 한 번에 계산하는 단계이고, decoding은 토큰을 하나씩 생성하는 단계이다. RAG처럼 입력이 긴 워크로드에서는 prefill이 지배적 병목이 된다. PDF p. 3
TTFT (Time To First Token)
요청 도착부터 첫 출력 토큰이 생성될 때까지의 시간. prefill 단계 지연을 주로 반영하며 RAGCache의 주 최적화 지표이다. PDF p. 10
TPOT (Time Per Output Token)
출력 토큰 하나당 소요 시간으로 decoding 단계 성능을 반영한다. RAGCache는 이를 직접 최적화하지 않는다. PDF p. 12
Knowledge tree
문서 ID를 키로 하는 prefix tree. 루트에서 노드까지의 경로가 하나의 문서 시퀀스를 나타내며 해당 노드가 그 시퀀스의 KV tensor를 보관한다. prefix matching으로 O(h)(h=트리 높이) 조회가 가능하다. PDF p. 4, 5
PGDSF (Prefix-aware Greedy-Dual-Size-Frequency)
RAGCache의 cache replacement policy. priority = Clock + Frequency × Cost / Size(Formula 1)로 각 노드 우선순위를 계산하고, GPU 메모리가 가득 차면 priority가 가장 낮은 leaf 노드를 host memory로 swap-out 한다(Algorithm 1). Cost는 cached/non-cached token 길이별 prefill 시간을 offline profiling하고 bilinear interpolation으로 추정한다. PDF p. 5, 6
Cache-aware reordering
OrderPriority = Cached Length / Computation Length 기준의 priority queue로 대기 요청 순서를 재배열하여 cache hit rate를 높이고 cache thrashing을 방지하는 스케줄링 기법. window 크기 제한으로 starvation을 방지한다. PDF p. 7
Dynamic speculative pipelining
vector search의 중간 top-k 후보를 이용해 LLM 생성을 조기 시작하여 retrieval과 generation을 오버랩하는 기법. 대기 중인 prefill 요청 수(max_prefill_bs)에 따라 speculation 수행 여부를 동적으로 결정하며, 이 조건(Algorithm 2)은 단순화된 G/G/1 모델(Theorem 5.1)에서의 pool-empty 최적 전략을 참고한 휴리스틱이다. PDF p. 7, 8
Cache hit rate (본 논문 정의)
검색된 문서 수 대비 hit한 문서 수의 비율. 예를 들어 저장된 문서 시퀀스가 [D1, D2]이고 요청이 [D1, D3]이면 hit rate는 50%이다. PDF p. 11
Throughput (본 논문 정의)
TTFT가 특정 임계값(최저 request rate에서 측정된 TTFT의 5배) 미만으로 유지되는 최대 request rate. PDF p. 10
vLLM
iteration-level scheduling과 PagedAttention을 사용하는 최신 LLM serving 시스템. 본 논문에서는 Faiss와 결합하여 주 baseline으로 사용된다. PDF p. 10, 13
SGLang
GPU 메모리 내에서 요청 간 KV cache 재사용을 지원하며 replacement policy로 LRU를 사용하는 고성능 LLM serving 시스템. RAGCache의 caching baseline이다. PDF p. 10
Cache thrashing
서로 다른 prefix를 요구하는 요청들이 번갈아 처리되면서 캐시 내용이 반복적으로 축출·재적재되어 hit rate가 떨어지는 현상. cache-aware reordering이 이를 완화한다. PDF p. 7
