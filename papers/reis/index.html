핵심 요약

REIS는 RAG(Retrieval-Augmented Generation) 파이프라인의 retrieval 단계를 SSD 내부에서 처리하는 최초의 RAG 특화 ISP(In-Storage Processing) 시스템이다. 저자들은 자체 측정을 통해 4,150만 개 항목 규모의 wiki_en 데이터셋에서 storage로부터의 I/O 트래픽이 전체 RAG 파이프라인 지연시간의 84%를 차지함을 확인했고(Sec 3.1, Figure 2), 기존 ISP 기반 ANNS 가속기들이 (i) 그래프 기반 알고리즘의 불규칙(irregular) 접근 패턴 때문에 SSD 내부 대역폭을 살리지 못하고, (ii) ANNS만 가속할 뿐 document retrieval은 가속하지 않으며, (iii) 상당한 저장 공간·하드웨어 오버헤드를 유발한다는 점을 문제로 지목한다. REIS는 이에 대응해 세 가지 메커니즘을 제안한다. 첫째, NAND flash의 OOB(Out-Of-Band) 영역 일부(16KB page당 16바이트, OOB의 0.7%)를 재사용해 임베딩과 대응 document chunk 주소(DADR)를 직접 연결하는 embedding-document linkage와, 임베딩을 모든 plane에 고르게 분산시키는 Parallelism-First Page Allocation 기반 database layout을 도입한다. 둘째, ISP에 적합한 순차적(streaming) 접근 패턴을 갖는 IVF(Inverted File) 알고리즘을 채택하고 이에 맞춘 R-IVF 레지스트리와 데이터 배치를 설계하며, 이 과정에서 INT8 재채점 주소(RADR)와 cluster tag가 OOB 영역에 추가로 저장되도록 linkage를 확장한다(이 확장분의 저장 비용은 0.7% 수치와 별개이며 본문에서 정량화되지 않는다). 셋째, Binary Quantization(BQ)을 기반으로 MAC 유닛 같은 추가 연산 하드웨어 없이 기존 SSD 구성요소(Cache/Sensing/Data Latch, fail-bit counter, pass/fail checker, 임베디드 코어)만으로 거리 계산과 top-k 선택을 수행하는 in-storage ANNS 엔진을 구성하며, Enhanced SLC-mode Programming(ESP)으로 ECC 없이 오류 없는 plane 내부 연산을 가능하게 한다. 저자들이 보고한 평가 결과에 따르면 REIS는 고성능 CPU 서버(CPU-Real; 논문은 Introduction에서 256-core라고 서술하지만 Methodology의 Table 3 스펙은 AMD EPYC 9554 2소켓 128코어로 제시되어 있어 두 서술이 서로 다르다) 대비 평균 13배(최대 112배) 성능과 평균 55배(최대 157배) 에너지 효율을 제공하고, 최신 ISP 기반 ANNS 가속기인 ICE 대비로는 서술 위치(Abstract/Introduction vs. 본문 Section 6.4)에 따라 제시된 배수 값이 다르므로 단일 수치로 단정하기보다 SSD 구성과 목표 Recall@10에 따라 수 배에서 최대 20배 이상의 폭으로 RAG retrieval을 가속한다고 이해하는 것이 안전하다(세부 수치는 unresolved 참조). 또한 Dataset Loading과 Search가 차지하던 지연시간 비중을 20.3%-69.3%에서 0.02%-0.15%로 낮추어, RAG 파이프라인의 병목이 LLM generation 단계(전체의 92%)로 옮겨감을 보인다.

해결하려는 문제

논문 주장

RAG의 retrieval 단계가 critical path에 있다

LLM은 학습 시점 데이터에 지식이 국한되며 재학습 비용이 커서 최신·도메인 특화 지식을 반영하기 어렵다. RAG는 외부 지식 데이터베이스를 검색해 이를 보완하지만, retrieval 단계가 응답 생성의 critical path 위에 있으므로 recall(정확도)과 latency(지연시간)의 균형이 중요하다. 즉 retrieval을 느리게 만드는 요인은 곧바로 사용자 응답 지연으로 이어진다.

PDF p. 1, 2

I/O data movement가 지배적 병목이다

대규모 데이터베이스 임베딩에 대해 Approximate Nearest Neighbor Search(ANNS)를 수행하면 host와 storage 시스템 간 I/O data movement가 심각한 성능 병목이 된다. 저자들의 직접 측정(Sec 3.1, Figure 2)에 따르면 4,150만 개 문서 항목을 가진 wiki_en 데이터셋에서 storage로부터의 I/O 트래픽이 전체 RAG 파이프라인 지연시간의 84%를 차지하며, 데이터셋 크기가 커질수록 dataset loading이 차지하는 지연시간 비중도 함께 증가한다.

PDF p. 1, 2, 5

ISP는 유망하지만 기존 설계로는 부족하다

저자들은 선행연구 NDSearch가 storage I/O가 billion-scale ANNS end-to-end 지연시간의 최대 75%를 차지하고 ISP가 CPU 대비 ANNS 성능을 31.7배 향상시킬 수 있음을 보였다고 인용하며, ISP를 병목 해결의 유망한 접근으로 제시한다. 그러나 기존 ISP 기반 ANNS 가속기들은 이 병목을 근본적으로 해소하지 못한다는 것이 이 논문의 출발점이다.

기존 방식의 한계

논문 주장

ISP에 부적합한 알고리즘 선택

HNSW, DiskANN 같은 그래프 기반 알고리즘을 사용하는 기존 ISP ANNS 가속기는 ISP 시스템에 맞지 않는다. 그래프 순회는 현재 방문 중인 정점을 분석해 다음 방문 정점을 결정하는 순차적(sequential-dependent) 과정이므로 불규칙한(irregular) 데이터 접근 패턴을 유발하고, 이는 channel 및 NAND Flash chip 충돌(conflict)을 일으켜 SSD의 내부 대역폭을 충분히 활용하지 못하게 한다.

PDF p. 1, 6

Document retrieval 단계 미가속

기존 ISP 기반 ANNS 가속기는 ANNS(검색) 자체의 가속에만 집중하고, ANNS가 선택한 데이터에 대한 document retrieval(문서 검색) 단계는 가속하지 않는다. 그러나 저자들의 측정(Figure 2, 3)에 따르면 dataset loading 단계가 RAG retrieval에 상당한 지연시간을 유발하므로, ANNS만 빨라져도 전체 병목은 남는다.

PDF p. 1, 6

저장 공간 및 하드웨어 오버헤드

기존 가속기는 스토리지 시스템에 큰 오버헤드를 도입한다. ICE는 NAND flash die 내부 연산을 위해 오류 정정 없이 오류를 허용하는 데이터 포맷을 사용하여 8-bit(4-bit) 정밀도 데이터에 32×(8×)의 저장 오버헤드를 유발하고, DeepStore는 systolic array 기반 아키텍처를 도입해 상당한 면적·전력 오버헤드를 유발한다.

PDF p. 1, 6

Batching과 Quantization의 한계

여러 질의를 묶어 dataset loading 오버헤드를 amortize하는 batching은, 의료·법률·금융 등 서로 다른 도메인의 질의가 서로 다른 도메인 특화/멀티모달 데이터셋에서 서비스되어야 하므로 실제 효과가 제한적이다. Quantization(PQ, BQ)은 저장 공간과 recall 간 좋은 트레이드오프를 제공하지만 임베딩에만 적용 가능하고 document chunk 크기는 줄이지 못한다. 실험적으로 BQ 적용 시 I/O data movement 오버헤드가 17-29% 감소하지만, 대형 wiki_en 데이터셋에서는 dataset loading이 여전히 전체 지연시간의 67%를 차지하는 병목으로 남는다(BQ 적용 후에도 document chunk가 wiki_en 전체 전송량 14GB 중 9GB를 차지).

PDF p. 5, 6

정교한 인덱스와 CXL 메모리 확장의 한계

SPANN 같은 hybrid ANNS는 flat index보다 큰 자료구조를 요구해 I/O 병목을 오히려 악화시킬 수 있다. SPANN은 HotpotQA에서 0.92 Recall@10을 달성하려면 전체 임베딩의 24%를 centroid로 메모리에 상주시켜야 하고, 그럼에도 exhaustive search 대비 22% 속도향상만 제공한다. 또한 hybrid ANNS는 임베딩의 저장·검색만 최적화할 뿐 document chunk는 최적화하지 않는다. CXL 기반 Memory Expansion은 이론적으로 데이터셋을 메모리에 상주시킬 수 있으나, 메인 메모리가 flash storage보다 GB당 한 자릿수 이상 비싸고(약 3.10 대 0.1 USD/GB) 데이터셋 크기와 도메인 특화 데이터셋 수가 계속 증가하므로 지속 가능하지 않다.

RAG 특화 ISP retrieval 시스템 REIS 제안

RAG에 특화된 최초의 ISP 기반 retrieval 시스템 REIS를 제안한다. REIS는 (i) 스토리지 시스템 내부에서 임베딩과 문서 간 상관관계를 구축해 효율적인 document retrieval을 지원하고, (ii) ISP-친화적 알고리즘 도입으로 retrieval 성능을 향상시키며, (iii) 현대 스토리지 시스템에 이미 존재하는 계산 자원을 활용하는 맞춤형 in-storage ANNS 계산 엔진으로 에너지 및 면적 효율을 향상시킨다.

제안 방식의 전체 구조와 주요 구성요소

논문 주장

전체 구조 개요

REIS는 host로부터 query embedding을 수신하여 storage 내부에서 데이터베이스를 조회하고 관련 document chunk를 반환함으로써 host-storage 간 통신을 크게 줄이는 ISP 기반 retrieval 시스템이다. 별도 연산 하드웨어 추가 없이 (1) 임베딩과 문서를 연결하는 database layout, (2) ISP에 맞춰 조정된 IVF 알고리즘 및 데이터 배치 기법, (3) 기존 스토리지 내부 계산 자원(플레인 내부 peripheral logic, SSD controller의 임베디드 코어)을 활용하는 in-storage ANNS 엔진이라는 세 가지 핵심 메커니즘으로 구성된다.

PDF p. 1, 2, 6

구성요소 1: Database Layout (Sec 4.1)

벡터 데이터베이스를 embedding region과 document region으로 분리 배치하고, Parallelism-First Page Allocation으로 임베딩을 모든 plane에 고르게 분산시켜 plane/die/channel 병렬성을 극대화한다. NAND flash의 OOB(Out-Of-Band) 영역 일부(16KB page당 16 spare bytes, OOB의 0.7%)를 재사용해 각 임베딩에 대응하는 document chunk 주소(DADR)를 저장함으로써, 별도 자료구조 없이 저비용으로 embedding-document linkage를 만든다. 이는 4KB 단위 임베딩/문서와 4바이트 document 주소를 가정한 예시 비용이며, 이후 IVF 도입에 따라 OOB에 추가로 저장되는 항목(RADR, cluster tag)의 비용은 이 0.7% 수치에 포함되지 않는다. 또한 R-DB라는 coarse-grained 주소 레지스트리(SSD DRAM 상주)를 도입해 페이지 단위 FTL 유지 비용을 줄인다(1TB 데이터베이스 기준 page-level FTL 1GB 대비 21바이트로 축소). Hybrid SSD 설계로 Enhanced SLC-mode Programming(ESP)을 적용한 SLC 파티션에는 binary 임베딩을, 고밀도 TLC 파티션에는 document chunk 및 INT8 임베딩을 저장하여 ECC 없이도 오류 없는(zero BER) in-plane 연산을 가능하게 한다.

PDF p. 6, 7

구성요소 2: ISP-Friendly ANNS 알고리즘 (Sec 4.2)

cluster 기반(IVF), 그래프 기반(HNSW), hash 기반(LSH) 알고리즘을 CPU 상에서 QPS/recall 기준으로 비교(Figure 5)한 결과, HNSW와 IVF 모두 높은 recall을 제공하지만 그래프 기반은 불규칙 접근 패턴 때문에 ISP에 부적합하다고 판단해, 순차적(streaming) 접근 패턴을 갖는 IVF를 채택했다. 이를 위해 embedding region을 centroid / binary / INT8 세 서브 영역으로 나누고, 각 cluster의 centroid 주소·첫/마지막 임베딩 인덱스·8비트 tag를 기록하는 R-IVF 배열(SSD DRAM, 항목당 15바이트)을 도입했으며, OOB 영역에 INT8 임베딩 주소(RADR)와 cluster tag를 추가 저장하도록 embedding-document linkage를 확장했다.

PDF p. 7, 8

구성요소 3: In-Storage ANNS Engine (Sec 4.3)

Binary Quantization(BQ) 기반으로 설계되어 MAC 유닛 등 추가 하드웨어 없이 기존 SSD 구성요소(Cache Latch, Sensing Latch, Data Latch, fail-bit counter, pass/fail checker 등 peripheral logic과 임베디드 코어)만으로 거리 계산과 top-k 선택을 수행한다. plane/die/channel 수준 병렬성을 활용하며, 세 가지 최적화를 포함한다. (i) Distance Filtering(DF): 사전 정의된 threshold를 초과하는 임베딩을 pass/fail checker에서 조기 제거한다. (ii) Pipelining(PL): Read Page Cache Sequential 모드로 다음 page read와 이전 결과 readout을 중첩하고, 임베디드 코어의 quicksort/reranking 커널 실행과 플레인 내 거리 계산을 중첩한다. (iii) Multi-Plane Input Broadcasting(MPIBC): die 내 모든 plane에 query를 동시에 broadcast한다. 또한 fine-grained 임베딩 접근을 위해 물리 page 주소에 offset을 덧붙이는 Mini-Page 주소 방식을 사용한다.

단계별 동작 과정

논문 주장

1단계: Query 수신 및 Input Broadcasting

Host로부터 새 query embedding을 수신하여 SSD DRAM에 저장한 뒤 ANNS 커널 실행을 트리거한다. Query embedding을 각 NAND flash plane의 데이터 버퍼로 전송하고, 이를 여러 번 복제해 Cache Latch(CL) 전체를 채운다(Input Broadcasting, IBC). MPIBC 최적화를 통해 하나의 die 내 모든 plane이 동시에 query를 수신한다.

PDF p. 9, 10

2단계: Coarse-grained search (cluster 선정)

모든 cluster centroid가 저장된 page를 읽어 Sensing Latch(SL)에 적재하고, CL(질의)과 SL(centroid) 간 XOR 연산 결과를 Data Latch(DL)에 저장한 뒤 peripheral logic의 fail-bit counter로 논리적 1의 개수, 즉 Hamming 거리를 계산한다. 각 항목(DIST, EMB, EADR, TAG)은 SSD DRAM의 TTL-C(Temporal Top List for Centroids)에 기록되고, 임베디드 코어가 quickselect 커널로 query에 가장 가까운 N개 cluster를 선정한다. 이 과정은 다음 page read와 거리 계산을 중첩하며 반복된다.

PDF p. 8, 9

3단계: Fine-grained search (cluster 내부 탐색)

선정된 cluster 내 모든 임베딩에 대해 동일한 XOR + fail-bit counter 방식으로 거리를 계산한다. 이때 Distance Filtering을 적용해 사전 설정된 threshold를 초과하는(query와 거리가 먼) 임베딩은 pass/fail checker에서 조기에 걸러내 SSD controller로 전송하지 않는다. 남은 항목(DIST, EMB, RADR, DADR)은 TTL-E(Temporal Top List for Embeddings)에 기록된다.

PDF p. 8, 9, 10

4단계: Top-10k 선택

임베디드 코어가 TTL-E에 대해 quickselect를 수행하여 query에 가장 가까운 10k개 임베딩을 선택한다.

PDF p. 9

5단계: Reranking

임베디드 코어가 RADR(rescoring address)을 이용해 top-10k 임베딩의 INT8 정밀도 버전을 fetch하고 INT8 정밀도로 거리를 재계산한 뒤 quicksort로 정렬하여 최종 top-k 임베딩을 선택한다. 이 단계가 binary quantization으로 인한 정확도 손실을 보상한다.

PDF p. 9

6단계: Document Identification 및 반환

ANNS 엔진이 top-k 결과에 연결된 DADR(document address)을 이용해 document identification을 수행하고, 해당 document chunk들을 host 시스템으로 전송하여 이후 LLM generation 단계에 사용되도록 한다. 즉 host로 나가는 데이터는 전체 데이터셋이 아니라 최종 선택된 문서뿐이다.

PDF p. 9

핵심 알고리즘 및 인덱스 구조

논문 주장

IVF 인덱스: 왜 그래프가 아니라 cluster인가

IVF는 데이터베이스 임베딩을 cluster로 나누고 각 cluster를 centroid로 대표한다. 검색은 centroid와의 거리 비교(coarse-grained) 후 선택된 cluster 내부만 순회(fine-grained)하는 2단계로 이루어지며, cluster 내부 임베딩은 연속적으로 배치되어 streaming 접근이 가능하다. 논문은 이 순차성이 SSD의 plane/die/channel 병렬성과 잘 맞아 channel·chip 충돌을 피할 수 있기 때문에, recall이 유사한 HNSW 대신 IVF를 선택했다고 설명한다.

PDF p. 7, 8

BQ 기반 거리 계산: XOR + fail-bit counter

Binary Quantization된 임베딩 간 거리는 Hamming distance로 계산되며, 이는 XOR 후 1의 개수를 세는 연산으로 환원된다. REIS는 NAND flash plane에 이미 존재하는 bitwise 연산 로직과 fail-bit counter(원래 프로그램 검증용)를 그대로 재사용해 이 두 연산을 수행하므로, MAC 유닛이나 systolic array 같은 새 연산 하드웨어가 필요 없다. 이것이 기존 ISP 가속기 대비 면적·전력 오버헤드를 피하는 핵심 기제이다.

PDF p. 8, 9

보조 자료구조: R-DB, R-IVF, TTL, OOB linkage

R-DB는 데이터베이스 단위 coarse-grained 주소 레지스트리로 page-level FTL을 대체해 1TB 기준 1GB를 21바이트로 축소한다. R-IVF는 cluster별 centroid 주소·첫/마지막 임베딩 인덱스·8비트 tag를 담는 배열로 항목당 15바이트를 차지하며 SSD DRAM에 상주한다. TTL-C와 TTL-E는 각각 centroid·임베딩 후보의 (DIST, EMB, 주소, TAG) 항목을 담는 임시 top list이다. OOB 영역에는 각 임베딩의 document 주소(DADR, 16KB page당 16바이트/OOB 0.7% 규모의 기본 linkage)에 더해, IVF 도입 이후 INT8 재채점 주소(RADR)와 cluster tag가 추가로 저장되어 임베딩-문서 연결을 별도 인덱스 없이 구현한다.

PDF p. 6, 7, 8, 9

정확도 보존 전략

BQ는 저장·연산 비용을 크게 줄이지만 정밀도 손실이 있다. REIS는 binary 임베딩으로 넓게 후보를 좁힌 뒤(top-10k), INT8 정밀도 reranking으로 최종 top-k를 결정하는 2단계 정밀도 전략을 사용한다. 논문의 CPU 기반 측정(Figure 5)에서 IVF의 recall은 reranking 결합 시 BQ 적용 후 0.97, PQ 적용 후 0.96으로 높게 유지된다.

REIS는 RAG retrieval의 지배적 비용이 연산이 아니라 host-storage 간 데이터 이동임을 정량적으로 보이고, 알고리즘 선택(IVF)·데이터 배치(OOB 기반 embedding-document linkage)·연산 위치(NAND plane 내부 기존 로직 재사용)를 함께 재설계함으로써 추가 연산 하드웨어 없이 성능과 에너지 효율을 동시에 개선할 수 있음을 보인다.

논문의 강점

  • 분석자 해석 문제 정의가 자체 측정에 기반한다. 84%(무양자화), 67%(BQ 적용 후)라는 구체적 병목 수치를 먼저 제시하고 설계를 도출하므로, 제안 기법의 동기가 사후 정당화가 아니라 관측에서 나온다.
    PDF p. 5, 6
  • 분석자 해석 알고리즘·데이터 배치·연산 위치를 각각 따로 최적화하지 않고 일관된 하나의 제약(SSD 내부 병렬성과 plane 로직의 표현력)에 맞춰 공동 설계했다. IVF 선택 → binary 임베딩 → XOR/fail-bit counter → OOB linkage로 이어지는 결정들이 서로를 지지한다.
    PDF p. 6, 7, 8, 9
  • 분석자 해석 추가 연산 하드웨어(MAC, systolic array)를 도입하지 않고 SSD에 이미 존재하는 구성요소만 재사용하므로, ICE·DeepStore가 겪는 저장 공간(32×/8×) 및 면적·전력 오버헤드 문제를 구조적으로 회피한다. 이는 실제 제품 적용 가능성 측면에서 중요한 차별점이다.
    PDF p. 1, 6, 8
  • 분석자 해석 ANNS만이 아니라 document retrieval까지 storage 내부에서 완결시킨다는 점이 기존 ISP ANNS 가속기와의 본질적 차이다. BQ 적용 후에도 document chunk가 wiki_en 전송량 14GB 중 9GB를 차지한다는 측정이 이 설계 선택의 필요성을 뒷받침한다.
    PDF p. 6, 9
  • 분석자 해석 비교 기준이 공정한 방향으로 설정되어 있다. I/O 비용이 0인 이상적 No-I/O, ECC가 불필요한 이상화 ICE-ESP, 연산 오버헤드가 0인 이상적 REIS-ASIC 등 자신에게 불리한 이상화 baseline을 포함시켜 이득의 출처를 분해한다.
    PDF p. 11, 12, 13
  • 분석자 해석 end-to-end 파이프라인 관점의 평가(Table 4)를 포함해, 커널 수준 속도 향상이 실제 응답 지연으로 얼마나 이어지는지와 병목이 어디로 이동하는지를 함께 보고한다. 개선의 상한을 스스로 드러낸다는 점에서 정직한 평가다.
    PDF p. 12

한계

  • 저자 명시 한계 ISP 도입 자체의 근본적 설계 난제를 인정한다. (1) 임베디드 코어는 성능과 기능이 제한적이며 부동소수점 연산을 지원하지 않고, (2) flash channel 대역폭이 전체 NAND flash 읽기 대역폭보다 제한적이다. 또한 plane 내부 하드웨어 재사용으로 인해 flash die 내부 로직은 단순한 bitwise 및 bit-counting 연산만 지원하며, NAND flash 읽기는 본질적으로 unreliable하여 ECC가 필요한데 이는 plane 내부 연산 시 기존과 다른 오류 완화 메커니즘(ESP)을 요구한다.
    PDF p. 6
  • 저자 명시 한계 I/O data movement가 end-to-end RAG 성능에 기여하는 정도는 인코딩 및 생성 모델의 크기에 크게 좌우된다는 caveat을 인정한다. 예를 들어 Llama 3.2 90B 같은 대형 생성 모델을 쓰면 생성 지연시간이 늘어나 retrieval 단계의 상대적 영향이 줄어든다.
    PDF p. 5
  • 저자 명시 한계 Coarse-grained access(경량 L2P 매핑)는 연속된 미할당 물리 공간의 존재를 요구하며, 데이터베이스 배포 시 단편화 해소(defragmentation) 작업이 필요할 수 있다. 저자들은 이를 상각 가능한 초기 오버헤드로 설명한다.
    PDF p. 7, 14
  • 저자 명시 한계 embedding-document linkage와 문서 영역이 contiguity(연속성) 기반 접근을 사용하므로, 문서가 업데이트되어 재배치될 경우를 지원하려면 OOB 영역의 물리 주소를 매번 갱신해야 하는 추가 복잡성이 발생한다(비-contiguity 대안 접근에서).
    PDF p. 13, 14
  • 저자 명시 한계 메타데이터 필터링은 현재 REIS 설계에 기본 포함되어 있지 않으며, 이를 지원하려면 read-only 데이터베이스를 가정한 확장(메타데이터 tag를 OOB 영역에 저장)이 필요하다고 discussion에서 별도로 제안한다.
    PDF p. 13
  • 분석자 해석 REIS 적용 후 Generation 단계가 전체 지연시간의 92%를 차지하므로, retrieval 최적화의 end-to-end 이득에는 명확한 상한이 있다. HotpotQA에서 end-to-end 개선이 1.25배에 그친 것이 이를 보여준다. 즉 retrieval QPS의 큰 향상이 사용자 체감 지연 개선으로 그대로 전이되지 않는다.
    PDF p. 12
  • 분석자 해석 IVF 채택은 ISP 친화성을 얻는 대신 알고리즘 선택의 폭을 좁힌다. 그래프 기반 알고리즘이 유리한 데이터 분포나, 향후 더 나은 인덱스가 등장할 경우 REIS의 하드웨어-알고리즘 결합 설계가 유연하게 따라가기 어려울 수 있다. 논문은 IVF 이외 알고리즘으로의 확장 가능성을 평가하지 않는다.
    PDF p. 7, 8
  • 분석자 해석 Distance Filtering의 threshold가 사전 정의(pre-defined)되어 있다. 성능 기여도가 가장 큰 최적화(평균 4.7×-5.7×)인 만큼, 질의 분포나 데이터셋 특성에 따른 threshold 민감도와 recall 손실 가능성이 중요한데 이에 대한 분석은 확인되지 않는다.
    PDF p. 10, 12
  • 분석자 해석 평가가 시뮬레이션 기반(Flash-Cosmos 모델, CACTI7, Zsim, Ramulator)이며 실제 SSD 프로토타입 구현 결과는 확인되지 않는다. 특히 ESP 기반 zero-BER 가정과 OOB 영역 재사용의 실환경 신뢰성·수명(endurance) 영향은 실측으로 뒷받침되지 않는다.
    PDF p. 11
  • 분석자 해석 hybrid SSD 구성(SLC 파티션에 binary 임베딩, TLC 파티션에 문서·INT8 임베딩)은 SLC 영역이 TLC 대비 저장 밀도가 낮으므로 실효 용량 측면의 비용을 수반할 것으로 보이나, 이 트레이드오프의 정량적 분석은 확인되지 않는다.
    PDF p. 6, 7
  • 분석자 해석 ICE 대비 성능 향상 배수는 Abstract/Introduction과 Section 6.4 본문 사이에 값이 일치하지 않으며, 이 차이를 설명하는 조건이 원문에 명시되어 있지 않다. 따라서 특정 단일 배수를 REIS의 대표 성능 향상치로 단정할 수 없다.
    PDF p. 2, 13
  • 분석자 해석 CPU-Real 서버의 코어 수는 논문 내에서 256-core(Introduction/Key Results)와 2소켓 128코어(Methodology Table 3, AMD EPYC 9554)로 다르게 표기되어 있다. 어느 쪽이 실제 평가에 쓰인 구성인지 원문에서 확정할 수 없다.
    PDF p. 1, 11
  • 분석자 해석 Table 4의 두 번째 비교 데이터셋 명칭이 원문 내에서 NQ와 wiki_en으로 상충되게 표기되어 있어, 61.69초→19.0초(3.24배 감소) 수치가 정확히 어느 데이터셋에 해당하는지 단정할 수 없다.
    PDF p. 12

핵심 용어

RAG (Retrieval-Augmented Generation)
외부 지식 데이터베이스에서 질의와 관련된 문서를 검색해 LLM 입력에 추가함으로써, 재학습 없이 최신·도메인 특화 지식을 반영하는 기법. retrieval 단계가 응답 생성의 critical path에 있다. PDF p. 1, 2
ANNS (Approximate Nearest Neighbor Search)
고차원 임베딩 공간에서 질의 벡터와 가장 가까운 벡터들을 근사적으로 찾는 검색. 정확도(recall)를 일부 희생해 탐색 속도를 크게 높인다. PDF p. 1, 2
ISP (In-Storage Processing)
데이터를 host로 옮기지 않고 스토리지 장치 내부에서 연산을 수행하는 방식. host-storage 간 데이터 이동 비용을 제거하고 SSD 내부 대역폭을 활용할 수 있다. PDF p. 1, 2, 6
IVF (Inverted File)
임베딩을 cluster로 분할하고 각 cluster를 centroid로 대표하는 인덱스. 검색은 centroid 비교(coarse-grained) 후 선택된 cluster 내부 순회(fine-grained)로 진행되며, cluster 내부가 연속 배치되어 streaming 접근이 가능하다. PDF p. 7, 8
HNSW / DiskANN
그래프 기반 ANNS 알고리즘. 높은 recall을 제공하지만 순회 과정이 현재 정점 분석에 의존해 불규칙한 접근 패턴을 만들고, 이는 channel/chip 충돌을 유발해 ISP에 부적합하다. PDF p. 1, 6, 8
BQ (Binary Quantization)
임베딩 각 차원을 1비트로 양자화하는 기법. 거리 계산이 Hamming distance(XOR + 1의 개수 세기)로 환원되어 NAND plane의 기존 로직만으로 계산할 수 있게 한다. REIS는 INT8 reranking으로 정밀도 손실을 보상한다. PDF p. 6, 8, 9
PQ (Product Quantization)
임베딩을 부분 벡터로 나누어 각각 코드북 인덱스로 압축하는 양자화 기법. 저장 공간과 recall 간 좋은 트레이드오프를 주지만 document chunk 크기는 줄이지 못한다. PDF p. 6, 8
OOB (Out-Of-Band) 영역
NAND flash page에 부속된 spare 영역으로 통상 ECC·메타데이터 저장에 쓰인다. REIS는 이 중 16KB page당 16바이트(OOB의 0.7%)를 재사용해 document 주소(DADR)를 저장함으로써 embedding-document linkage의 기본 형태를 구현한다. IVF 도입 이후에는 INT8 재채점 주소(RADR)와 cluster tag가 OOB에 추가로 저장되도록 확장되며, 이 확장분의 정확한 저장 비용은 0.7% 수치에 포함되지 않고 본문에서 별도로 정량화되지 않는다. PDF p. 6, 7, 8
ESP (Enhanced SLC-mode Programming)
SLC 모드로 프로그램해 셀 상태 간 마진을 넓힘으로써 오류율을 사실상 0으로 만드는 기법. REIS는 이를 통해 ECC 없이 plane 내부 연산을 수행할 수 있다. PDF p. 6, 7
Parallelism-First Page Allocation
임베딩을 모든 plane에 고르게 분산 배치해 plane/die/channel 수준 병렬성을 최대한 활용하도록 하는 REIS의 페이지 할당 정책. PDF p. 6, 7
Embedding-Document Linkage
각 임베딩의 OOB 영역에 대응 document chunk의 물리 주소를 저장해, 별도 자료구조 없이 임베딩과 문서를 연결하는 REIS의 메커니즘. document retrieval을 storage 내부에서 완결시키는 근간이다. PDF p. 6, 7
R-DB / R-IVF
SSD DRAM에 상주하는 레지스트리. R-DB는 coarse-grained 주소 레지스트리로 page-level FTL을 대체해 1TB 기준 1GB를 21바이트로 축소한다. R-IVF는 cluster별 centroid 주소·첫/마지막 임베딩 인덱스·8비트 tag를 담는 배열로 항목당 15바이트를 차지한다. PDF p. 6, 7, 8
TTL-C / TTL-E (Temporal Top List)
SSD DRAM에 유지되는 임시 후보 목록. TTL-C는 centroid 후보(DIST, EMB, EADR, TAG), TTL-E는 임베딩 후보(DIST, EMB, RADR, DADR)를 기록하며, 임베디드 코어의 quickselect 대상이 된다. PDF p. 8, 9
CL / SL / DL (Cache / Sensing / Data Latch)
NAND flash plane 내부의 래치들. REIS는 CL에 broadcast된 query를, SL에 읽어온 데이터베이스 임베딩을 두고 두 값의 XOR 결과를 DL에 저장하는 방식으로 거리 계산을 수행한다. PDF p. 8, 9
Fail-bit Counter / Pass-Fail Checker
원래 프로그램 검증에 쓰이는 plane 내부 peripheral logic. REIS는 fail-bit counter로 XOR 결과의 1의 개수(Hamming distance)를 세고, pass/fail checker로 threshold 초과 임베딩을 조기 제거(Distance Filtering)한다. PDF p. 8, 9, 10
Distance Filtering (DF)
사전 정의된 threshold를 초과하는 먼 임베딩을 plane 내부에서 조기에 걸러내 SSD controller로 전송하지 않는 최적화. sensitivity study에서 가장 큰 성능 기여 요인으로 보고된다(평균 4.7×-5.7×). PDF p. 10, 12
MPIBC (Multi-Plane Input Broadcasting)
하나의 die 내 모든 plane에 query embedding을 동시에 broadcast하는 최적화. die당 plane 수가 많은 SSD에서 이득이 커지며, DF+PL 대비 평균 6%(SSD-1)/26%(SSD-2) 추가 향상을 준다고 보고된다. PDF p. 10, 12
Reranking
binary 임베딩으로 좁힌 top-10k 후보를 RADR로 fetch한 INT8 정밀도 임베딩으로 재계산·재정렬해 최종 top-k를 결정하는 단계. BQ의 정밀도 손실을 보상한다. PDF p. 8, 9
Recall@10
상위 10개 검색 결과 중 실제 정답(정확한 최근접 이웃)이 포함된 비율. 이 논문에서 성능 비교는 동일한 목표 Recall@10(0.90/0.94/0.98) 조건에서 이루어진다. PDF p. 8, 11, 13
No-I/O
스토리지 I/O로 인한 데이터 이동 오버헤드가 전혀 없다고 가정한 이상적 CPU 비교점. REIS가 이 대비로도 평균 1.8배 빠르다는 점이 내부 병렬성 활용 효과를 보여준다. PDF p. 11
Mini-Page 주소
물리 page 주소에 offset을 덧붙여 page보다 작은 단위의 임베딩에 접근할 수 있게 하는 REIS의 fine-grained 주소 지정 방식. PDF p. 8, 9, 10
