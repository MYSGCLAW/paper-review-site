CEMU: Enabling Full-System Emulation of Computational Storage Beyond Hardware Limits

저자 · Qiuyang Zhang, Jiapin Wang, You Zhou, Peng Xu, Kai Lu, Jiguang Wan, Fei Wu, Tao Lu 출처 · ASPLOS '26 (31st ACM International Conference on Architectural Support for Programming Languages and Operating Systems), Volume 2, Pittsburgh, PA, USA 연도 · 2026

핵심 요약

CEMU는 computational storage drive(CSD) 연구를 위한 완전히 소프트웨어 기반의 오픈소스 full-system 에뮬레이션 플랫폼이다. 기존 CSD 연구 플랫폼은 (1) 고가이고 hardware-bounded인 하드웨어 플랫폼(예: OpenSSD, SmartSSD), (2) host CPU 성능에 종속되는(CPU-bounded) 소프트웨어 에뮬레이션, (3) full-system 실행을 지원하지 않아 page cache 같은 시스템 구성요소가 누락되는 소프트웨어 시뮬레이션(예: MQSim)으로 파편화되어 있어, 재현성과 전체 설계 공간(full design space) 탐구가 어렵다는 문제를 안고 있다. CEMU는 QEMU와 FEMU를 기반으로 guest VM 안에 표준 NVMe drive로 마운트되며, 핵심적으로 새로운 VM-freezing 메커니즘으로 host CPU/FPGA 성능을 초과하는 configurable computing capability를 에뮬레이트하고, 새로운 파일시스템 FDMFS를 중심으로 SNIA/NVMe 산업 표준을 준수하면서도 전통적 POSIX I/O 스택과 호환되는 CSD-oriented 소프트웨어 스택을 제공한다. 저자들은 CEMU가 Samsung SmartSSD와 ScaleFlux CSD2000 실제 하드웨어 대비 네 가지 synthetic benchmark(grep, kNN, LZ4, TPC-H sql)에서 평균 95% 이상의 종단간 정확도를 달성했다고 보고하며, LLM training과 LevelDB 두 full-system case study를 통해 수백 줄의 코드만으로 CSD 시스템을 프로토타이핑할 수 있는 민첩성과, 기존 플랫폼으로는 얻기 어려운 통찰(예: CSD 컴퓨팅 파워를 수백 배 늘려도 speedup은 약 2배에 그침, cache bypassing·I/O interference·inter-CSD 전송 오버헤드 등의 부작용)을 제시한다.

해결하려는 문제

논문 주장

CSD 연구를 위한 적절한 research platform의 부재

데이터의 기하급수적 증가와 AI 확산으로 컴퓨팅 파워·에너지 효율 수요가 커지는 가운데, 전통적 컴퓨팅 모델은 스토리지에서 host로 대량의 데이터를 이동시켜야 하므로 성능 오버헤드, 에너지 소비, 시스템 병목을 유발한다. CSD(computational storage drive)는 데이터가 위치한 장치에서 연산을 수행하는 in-storage processing 모델로 이 데이터 이동 오버헤드를 제거하며, HPC·데이터베이스·압축·그래프·추천·ML·LLM/RAG 등 여러 분야에서 이점이 입증되었고 Samsung SmartSSD, ScaleFlux, Alibaba PolarDB 등 산업계 채택과 SNIA/NVMe 표준화도 진행되었다. 그러나 CSD 기술의 연구와 보급을 저해하는 major challenge는 적절한 연구 플랫폼의 부재이다.

PDF p. 323

파편화된 플랫폼과 세 갈래의 근본적 제약

기존 플랫폼은 하드웨어 플랫폼과 소프트웨어(에뮬레이션/시뮬레이션) 플랫폼으로 파편화되어 있다. 하드웨어 플랫폼은 비용이 높고 컴퓨팅 성능이 hardware-bounded이며, 소프트웨어 에뮬레이션은 host CPU 성능에 종속(CPU-bounded)되고, 소프트웨어 시뮬레이션은 full-system fidelity가 부족하다. 이 파편화 때문에 CSD 시스템의 전체 설계 공간(full design space)을 탐구하기 어렵다.

PDF p. 323, 324

기존 방식의 한계

논문 주장

하드웨어 플랫폼: 고비용·hardware-bounded

OpenSSD 같은 하드웨어 플랫폼은 compute/storage subsystem과 소프트웨어 스택을 모두 갖춰 full-system 연구가 가능하지만, 비용이 높고(SmartSSD 약 $2,400, multi-CSD 구성 시 더 고가) 컴퓨팅 성능이 hardware-bounded여서 물리적 플랫폼보다 높은 성능의 차세대 CSD를 구성할 수 없다.

PDF p. 324

소프트웨어 에뮬레이션: CPU-bounded

빠른 DRAM으로 느린 flash를 timing delay 모델로 에뮬레이트하는 방식은 비용이 없고 상용 서버에서 실행되지만, CSD 내 하드웨어 가속기 에뮬레이션이 서버 CPU의 성능·효율에 근본적으로 제한(CPU-bounded)되어 컴퓨팅 능력 구성이 제약된다.

PDF p. 324

소프트웨어 시뮬레이션: full-system 부재

RTL synthesis나 MQSim 같은 시뮬레이터는 하드웨어 한계 없이 CSD를 유연하게 구성하지만 full-system 실행을 지원하지 않는다. 그 결과 system-level page cache 같은 핵심 구성요소가 누락되어 부정확한 평가로 이어진다. 예컨대 LevelDB에 대한 CSD offloading 시뮬레이션은 page cache를 배제하므로, 논문 §6.2가 드러내는 cache bypassing으로 인한 예기치 않은 성능 저하를 반영하지 못한다.

PDF p. 324

부정확한 성능 모델과 낮은 충실도

기존 CSD 압축 성능 모델은 constant 압축 지연을 가정하지만, 실측에서 4KB Zstd 압축의 host CPU 지연은 데이터 패턴에 따라 18μs~25μs로 38.8% 변동한다. INSIDER는 FPGA compute subsystem을 쓰되 실제 flash 대신 DRAM에 constant I/O delay를 더해 storage를 단순화하므로 flash 병렬성이 재현되지 않아, 동일 FPGA의 SmartSSD와 grep/knn/filter로 비교 시 34%의 종단간 오차율을 보인다(Figure 2).

PDF p. 324, 326

느린 full-system 시뮬레이션과 기능 한정 에뮬레이터

gem5는 full-system 모드를 제공하나 마이크로아키텍처 시뮬레이션 목적의 단일 스레드 clock-driven 방식이라 매우 느리다(수십 초 SSD 시뮬레이션에 5시간 소요). blockNDP·Wilcox 등의 QEMU 기반 CSD 에뮬레이터는 CSD offloading의 기능적 에뮬레이션에 그치고 성능 모델링이 없으며 컴퓨팅 성능 구성이 host CPU에 종속된다.

PDF p. 326

표준 비준수·비오픈소스·인터페이스 혼재·multi-CSD 부재

기존 CSD 플랫폼 중 SNIA/NVMe 산업 표준을 준수하는 것이 없고 대부분 비오픈소스여서 생태계가 파편화되고 재현이 어렵다. 또한 λ-IO·INSIDER는 Linux pread 시스템 콜을 오버로드해 연산을 오프로드함으로써 compute semantics와 file read semantics를 하나의 인터페이스에 혼재시킨다. 대부분의 연구는 multi-CSD를 다루지 않으며, Smart-Infinity(SmartSSD 10개+확장 섀시)·BlueDBM처럼 구축한 경우도 고비용·저가용성 문제가 있다.

PDF p. 324, 326

핵심 contribution

논문 주장

무료·고configurability의 통합 오픈소스 플랫폼 CEMU

무료이고 높은 configurability를 갖춘 CSD 에뮬레이션과, 산업 CSD 표준 및 전통적 POSIX I/O 스택에 호환되는 CSD-oriented 소프트웨어 스택을 통합한 새로운 오픈소스 소프트웨어 플랫폼을 제시하여 정확한 full-system 및 multi-CSD 연구를 가능하게 한다.

PDF p. 325

실제 하드웨어 기반 검증

CEMU를 Samsung SmartSSD와 ScaleFlux CSD2000 실제 하드웨어로 검증했다. 저자들은 네 가지 synthetic benchmark에서 평균 95%의 에뮬레이션 정확도를 보고하며, LLM training과 LevelDB 두 full-system case study는 수백 줄의 코드만으로 CSD 시스템을 프로토타이핑하는 민첩성(agility)을 보였다.

PDF p. 325

통찰을 주는 case study

다양한 CSD 구성을 탐색하며 full-system 동작을 조사해 기존 플랫폼으로는 얻기 어려운 통찰을 도출한다. LLM training case study는 다중·초고속 CSD로 성능을 크게 개선할 수 있으나 CSD 컴퓨팅 파워를 수백 배 늘려도 speedup은 약 2배에 그쳐 holistic system design이 필수임을 시사한다. LevelDB case study는 host cache bypassing, CSD 내부 I/O interference, inter-CSD 전송 오버헤드 등 CSD offloading의 부작용을 드러내 CSD를 신중히 배치해야 함을 보인다.

PDF p. 325, 326

제안 방식의 전체 구조

논문 주장

두 축: CSD emulator와 CSD-oriented 소프트웨어 스택

CEMU는 QEMU와 FEMU를 기반으로 개발된 완전 소프트웨어 CSD 에뮬레이션 플랫폼으로 guest VM 안에 표준 NVMe drive로 마운트된다. 크게 CEMU CSD emulator(§3)와 CSD-oriented 소프트웨어 스택(§4)으로 구성된다. CSD emulator는 conventional NVM storage namespace, compute namespace, memory namespace 세 NVMe namespace를 노출한다.

PDF p. 324, 327

CSD emulator의 네 모듈

CSD emulator는 NVMe module, CSF module, scheduling module, FEMU module의 네 모듈로 이루어진다. 핵심 아이디어는 (1) 새로운 VM-freezing 메커니즘으로 host CPU/FPGA 성능을 초과하는 compute engine을 configurable computing capability로 에뮬레이트하는 것과, (2) 새로운 파일시스템 FDMFS를 중심으로 SNIA/NVMe 표준을 준수하면서 기존 I/O 스택과 호환되는 소프트웨어 스택을 제공하는 것이다.

PDF p. 327

단계별 동작 과정

논문 주장

1단계 — 마운트 및 namespace 노출

CEMU CSD emulator(QEMU/FEMU 기반)가 guest VM 안에 표준 NVMe drive로 마운트되어 conventional NVM storage / compute / memory 세 namespace를 노출한다(Figure 3).

PDF p. 327

2단계 — 메모리 사전할당 및 CSF 다운로드

사용자는 FDMFS에서 fallocate()로 input/output device memory 공간을 파일로 미리 할당하고, ioctl(IOCTL_DOWNLOAD)로 compute namespace에 CSF(eBPF/shared library/bitstream)를 다운로드한다(Figure 7a).

PDF p. 330, 331

3단계 — 입력 로드(direct/indirect usage model)

direct usage model에서는 copy_file_range() 시스템 콜로 flash 상 데이터 파일에서 input device-memory 파일로 데이터를 직접 복사한다(FDMFS가 이를 NVMe memory-copy 명령으로 변환해 host를 거치지 않고 전달). indirect usage model에서는 하나의 ioctl(IOCTL_EXECUTE, indirect=1) 호출로 태스크 전체를 제출하고 CSD firmware가 흐름을 관리한다(Figure 7b).

PDF p. 331

4단계 — VM freeze 및 실제 실행

입력 로드가 끝나고 CSF 태스크가 초기화되면 CEMU는 BQL을 획득하고 global VM clock을 정지시킨 뒤 모든 vCPU에 pause 요청을 broadcast하여 VM을 freeze하고, 태스크를 host CPU(또는 FPGA)에서 실제로 실행한다.

PDF p. 328

5단계 — stall time 계산 및 VM resume

CSD 스케줄러가 Texec 모델(Scsf × Thost 또는 실측 실행시간)에 기반한 stall time을 계산하고, host CPU 상 물리적 연산이 끝나면 global VM clock을 재활성화하고 모든 vCPU에 resume 요청을 broadcast하여 VM을 재개한다. VM 관점에서는 태스크가 거의 즉시 완료된 것처럼 보인다(Figure 4).

PDF p. 327, 328

6단계 — 결과 저장 및 파이프라인/P2P

연산 결과는 output device-memory 파일에 저장되어 copy_file_range()로 flash 데이터 파일에 기록되거나 pread()로 host에 직접 읽힌다. 여러 chunk/CU/CSD를 이용하면 입력·연산·출력 단계를 파이프라인/병렬 처리할 수 있고, CSD 간 데이터 이동은 PCIe P2P로 host를 거치지 않고 수행할 수 있다.

PDF p. 329, 331

핵심 알고리즘 또는 인덱스 구조

논문 주장

VM-Freezing 메커니즘

host CPU 성능을 초과하는 compute engine을 에뮬레이트하기 위한 핵심 기법이다. 입력 데이터가 device memory에 로드되고 CSF 태스크가 초기화되면 QEMU VM(vCPU와 clock)을 pause_all_vcpus(), cpu_disable_ticks()로 정지시켜, 실제 연산을 host CPU에서 수행하는 동안의 지연을 VM 관점에서 관찰되지 않게 숨긴다. 그 뒤 Texec 모델 기반 stall time만 VM에 노출하고 resume_all_vcpus(), cpu_enable_ticks()로 VM을 재개한다. 이로써 물리 하드웨어보다 빠른 컴퓨팅 성능을 자유롭게 configurable하게 표현할 수 있다.

PDF p. 328

Texec 타이밍 모델과 CSF 실행 환경

CSF module은 오프로딩된 연산 태스크를 위해 eBPF(ubpf), shared library(.so, libdl), FPGA bitstream의 세 실행 환경을 제공하고, functional behavior와 timing model을 에뮬레이트한다. 타이밍은 사용자 설정 가능한 scaling factor Scsf로 실제 host 실행시간을 스케일링하여 Texec = Scsf × Thost로 계산한다.

PDF p. 327, 328

스케줄링 모델과 이기종 컴퓨팅

scheduling module은 태스크의 queueing/waiting time(Tsched)을 모델링하며 enqueue_job(), dequeue_job(), pick_next_job(), switched_out()의 네 hook function으로 pluggable scheduling policy를 지원한다(기본 FIFO·round-robin). 여러 compute unit(CU)에 개별 scaling factor와 policy를 지정해 heterogeneous 컴퓨팅을 에뮬레이트한다.

PDF p. 328

FDMFS와 스토리지 서브시스템

FDMFS(Functional Data Memory File System)는 device memory를 파일로 추상화해 fallocate로 할당하고 pread/pwrite로 접근하며 파일 permission으로 접근을 제어한다. block device(memory namespace)와 character device(compute namespace) 위에서 동작하고 NAND flash용 전통 파일시스템(ext4 등)과 공존한다. 스토리지 측은 FEMU를 채택해 host memory에 병렬 flash를 충실히 에뮬레이트하고, FTL이 별도 스레드로 동작하며 대역폭 기반 queuing/propagation delay를 모델링하는 PCIe bus model로 보강된다. CSD 간 데이터 전송은 각 CSD의 device memory를 PCIe BAR로 노출해 host를 거치지 않는 단일 P2P I/O로 수행한다.

논문 주장

높은 에뮬레이션 정확도

저자들은 CEMU가 실제 CSD 하드웨어 대비 평균 95% 이상의 종단간 성능 정확도를 달성했다고 보고한다. 구성별로는 단일 SmartSSD 96%, 3중 SmartSSD 95%, CSD2000 97%이며, 논문은 Figure 8의 모든 구성에서 relative error rate가 10% 미만이라고 제시한다. 대부분의 경우 chunk를 2개 동시 처리하는 것만으로 최대 성능을 얻는다. GPT-2 학습에서 CEMU-aligned CSD(compute 대역폭 약 1.25GB/s로 SmartSSD 모사)는 실제 SmartSSD 대비 평균 편차가 2.4%에 그친다고 저자들은 보고한다.

CEMU 소프트웨어 스택(io_uring, vfs, block layer, FDMFS, ext4)의 오버헤드는 전체 태스크 완료 시간에서 7.5%를 넘지 않는다(Figure 9, LZ4 direct 모델 단일 chunk 순차). indirect usage model은 chunk 크기가 작을수록 direct 대비 유리하여 1MB에서 7%, 2MB에서 4%, 4MB에서 3%의 성능 차이를 보인다.

확장성과 병목

LZ4 압축은 CSD 수에 따라 throughput이 거의 선형 증가하다 약 6개 CSD에서 포화한다(Figure 10). LevelDB(CSD별 개별 인스턴스)는 compaction을 오프로딩해도 host 측 복잡한 데이터 처리가 병목으로 남아 LZ4보다 처리량 증가가 느리고 더 일찍 포화한다.

LLM training case study의 통찰

Smart-Infinity를 CEMU로 포팅하는 데 약 200 LoC만 필요했다. CSD 수를 1→3개로 늘리면 GPT-2 학습이 2~2.5× speedup되지만, CEMU-extreme CSD(compute 대역폭 700GB/s로 확장, 컴퓨팅 엔진이 수백 배 빠름)는 CEMU-aligned(1.25GB/s) 대비 최대 2.4×에 그친다. 이는 컴퓨팅 파워를 수백 배 늘려도 I/O 오버헤드가 병목이 되어 개선이 제한됨을 보여, holistic system design의 필요성을 시사한다(Figure 12).

LevelDB case study의 부작용과 최적화

background compaction을 shared library 타입 CSF로 오프로딩(LevelDB-CSD)하는 데 약 500 LoC가 들었다. 100% write에서 host와 CSD 컴퓨팅 능력이 같을 때(x=1) 처리량은 501→721 Kops/s로 향상된다고 논문은 제시하지만, 50% write 혼합 워크로드에서는 CSD 내부 I/O interference로 vanilla 대비 절반 수준으로 저하된다(Figure 13, 16). FTL에 internal I/O 전용 큐를 두고 external I/O를 우선하도록 최적화하면 throughput과 I/O latency가 개선된다(Figure 17). 또한 locality 기반으로 16개 연속 SSTable을 동일 CSD에 배치하면 5-CSD RAID-0 100% write에서 P2P I/O 트래픽이 17.3→12 GB로 줄고 처리량이 7.4% 향상된다고 논문은 보고한다(Figure 18).

결론

논문 주장

결론 및 의의

CEMU는 무료·오픈소스이면서 configurable하고 SNIA/NVMe 표준과 POSIX I/O 스택에 호환되는 CSD-oriented 소프트웨어 스택을 통합함으로써, 하드웨어 한계를 넘어선(beyond hardware limits) 정확한 full-system 및 multi-CSD 연구를 가능하게 한다. QEMU 8.0.0/Linux 6.8 기반으로 총 약 11,000 LoC(QEMU 약 8K + kernel 약 3K)로 구현되었으며, 저자들은 실제 하드웨어 대비 평균 95% 이상의 정확도와 수백 줄 수준의 프로토타이핑 민첩성, 그리고 기존 플랫폼으로는 얻기 어려운 시스템 수준 통찰을 함께 제공한다고 결론짓는다.

논문의 강점

  • 분석자 해석 VM-freezing이라는 단순하면서 강력한 메커니즘으로 host CPU 성능이라는 근본적 제약(CPU-bounded)을 우회하여, 물리 하드웨어보다 수백 배 빠른 compute engine까지 configurable하게 표현할 수 있게 한 점이 핵심 강점이다.
    PDF p. 328
  • 분석자 해석 FEMU 기반의 충실한 flash 에뮬레이션과 full-system 실행을 결합하여, 시뮬레이터가 놓치는 page cache·I/O interference 같은 system-level 효과를 관측할 수 있어 평가의 현실성이 높다.
    PDF p. 324, 329
  • 분석자 해석 FDMFS와 io_uring, SNIA direct/indirect usage model을 통해 표준 준수와 기존 POSIX I/O 스택 호환을 동시에 달성하여, 애플리케이션 포팅 비용(예: Smart-Infinity 약 200 LoC, LevelDB 약 500 LoC)이 낮다.
    PDF p. 330, 333, 334
  • 논문 주장 실제 하드웨어(SmartSSD, CSD2000) 대비 정량 검증으로 평균 95% 이상 정확도와 10% 미만 오차를 저자들이 제시하여 에뮬레이션 신뢰성에 대한 실증적 근거를 뒷받침한다.
    PDF p. 332

한계

  • 저자 명시 한계 CEMU의 확장성은 host 서버 한 대의 CPU·메모리에 제한된다. QEMU는 여러 host에 걸친 단일 VM 생성이 불가하고 flash를 host memory에 에뮬레이트하므로, multi-CSD의 총 용량과 I/O 대역폭이 host memory의 용량·대역폭을 초과할 수 없다(다중 QEMU 인스턴스, CXL/RDMA 원격 메모리, 실제 SSD 활용은 향후 방향으로만 언급).
    PDF p. 329, 330
  • 저자 명시 한계 QEMU는 fine-grained per-vCPU freezing을 지원하지 않아 global VM clock이 모든 vCPU에 대해 함께 정지된다(저자는 clock 정지 중 guest 실행이 진행되지 않으므로 정확성·timing observability에는 영향이 없다고 주장).
    PDF p. 328
  • 저자 명시 한계 VM pause/resume 절차가 저자 서버에서 약 21μs의 지연 오버헤드를 유발하여, VM-freezing 기반 모델링으로 표현 가능한 최소 태스크 실행시간이 제한된다.
    PDF p. 328
  • 저자 명시 한계 FDMFS에서 device memory 매핑 파일을 resize하려면 현재 구현상 삭제 후 재생성해야 한다. 비연속 메모리 할당을 이용한 on-the-fly 동적 크기 조정은 가능하다고만 언급되고 실제 구현되지 않아 future work로 남겨졌다.
    PDF p. 330
  • 분석자 해석 정확도 검증이 grep/kNN/LZ4/sql 등 비교적 단순한 데이터-병렬 커널과 두 종의 실제 하드웨어에 국한되어, 복잡한 제어흐름·이기종 가속기·비정형 워크로드에서의 정확도는 검증되지 않은 것으로 보인다.
    PDF p. 332

핵심 용어

CSD (Computational Storage Drive)
데이터가 저장된 장치 내부에서 직접 연산을 수행(in-storage processing)하여 storage-host 간 데이터 이동 오버헤드를 제거하는 스토리지 장치. PDF p. 323
VM-Freezing
CEMU가 host 성능을 초과하는 compute engine을 에뮬레이트하기 위해 실제 연산 중 QEMU VM의 vCPU와 clock을 정지시켜 물리 실행 지연을 감추고 Texec 모델 기반 stall time만 노출하는 메커니즘. PDF p. 328
CSF (Computational Storage Function)
CSD로 오프로딩되는 연산 태스크. CEMU는 eBPF(ubpf), shared library(.so), FPGA bitstream의 세 실행 환경을 제공한다. PDF p. 327
Texec / Scsf
CSF 실행시간 타이밍 모델. 사용자 설정 scaling factor Scsf로 실제 host 실행시간 Thost를 스케일링하여 Texec = Scsf × Thost로 계산한다. PDF p. 328
FDMFS (Functional Data Memory File System)
device memory를 파일로 추상화하는 CSD-oriented 파일시스템. fallocate로 할당, pread/pwrite로 접근, 파일 permission으로 접근 제어하며 전통 파일시스템과 공존한다. PDF p. 330
FEMU
host memory에 병렬 아키텍처의 flash storage를 충실히 에뮬레이트하는 SSD 에뮬레이터. CEMU의 스토리지 서브시스템 기반이며 별도 스레드의 FTL과 PCIe bus model로 보강되었다. PDF p. 329
direct / indirect usage model
SNIA가 표준화한 CSD 실행 모델. direct는 host가 각 단계를 명시적으로 제어하고, indirect는 CSD firmware가 전체 실행을 관리한다. PDF p. 330, 331
PCIe P2P (Peer-to-Peer)
각 CSD의 device memory를 PCIe BAR로 노출하여 host를 거치지 않고 CSD 간 데이터를 직접 전송하는 방식. PDF p. 329
hardware-bounded / CPU-bounded
각각 하드웨어 플랫폼의 컴퓨팅 성능이 물리 하드웨어에, 소프트웨어 에뮬레이션의 성능이 host CPU에 종속되어 구성이 제약되는 상태. PDF p. 324
