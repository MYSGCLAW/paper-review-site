핵심 요약

이 논문은 GPU 중심(GPU-centric) 이종 시스템에서 저장장치 접근 경로를 재설계하여 대규모 신경망 정보 검색(neural Information Retrieval)의 메모리 비용과 지연시간을 동시에 낮추는 방법을 제안한다. 문제의 출발점은 멀티벡터 검색 모델(ColBERT 계열)이 문서 토큰마다 임베딩을 저장하기 때문에 재랭킹(re-ranking) 인덱스가 BM25 대비 최대 210.9배까지 커지고(Table 3), 이를 전부 시스템 메모리에 올리는 기존 방식이 대규모 환경에서 비용상 실현 불가능하다는 점이다. 저자들은 GPUDirect Storage(GDS)를 이용해 SSD에서 GPU 메모리로 임베딩을 직접 전송하는 ESPN(Embedding from Storage Pipelined Network)을 제안한다. ESPN의 핵심은 세 가지다. 첫째, candidate generation용 ANN 인덱스는 CPU 메모리에 캐시하고 대용량 재랭킹 임베딩 테이블은 SSD로 오프로드한다. 둘째, IVF 기반 ANN 검색이 전체 nprobe의 일부(δ, PrefetchStep)만 탐색한 시점에 software prefetcher가 예상 top-K 문서 임베딩을 GDS로 미리 가져와 남은 ANN 탐색과 저장장치 지연을 중첩시킨다. 셋째, 도착한 임베딩에 대해 custom CUDA MaxSim 커널이 즉시 early re-ranking을 수행한다. 여기에 상위 64-128개 후보만 재랭킹하는 대역폭 절감 기법 ESPN-Lite와, 저장된 임베딩을 조회하는 대신 사전 토큰화된 문서를 GPU에서 즉석 인코딩해 임베딩을 실시간 합성하는 ESPN-LIVE를 확장으로 제시한다. MS-MARCO v1/v2 실험에서 저자들은 ESPN이 mmap 대비 종단간 지연시간을 3.1-3.9배 줄이고, 완전 캐시 방식 대비 인덱스의 6-19%만 메모리에 두면서 메모리 사용량을 5-16배 절감하며, 완전 캐시 대비 1.02배 수준의 지연시간을 유지한다고 보고한다. ESPN-Lite와 ESPN-LIVE는 MRR@10 기준 0.5%의 미미한 품질 저하만 발생시킨다고 저자들은 밝힌다.

해결하려는 문제

논문 주장

CPU 중심 데이터 경로의 비효율

AI/ML 워크로드가 급증하면서 CPU-centric 아키텍처가 요구되는 데이터 처리량(throughput)과 연산 효율성을 제공하지 못한다. GPU에서 연산을 수행하려면 SSD 등 2차 저장장치의 데이터가 CPU에 의해 먼저 메인 메모리로 이동한 뒤 다시 GPU로 복사되어야 하며, 이 이동을 CPU가 주도·관리하기 때문에 비효율적이고 이종 시스템(heterogeneous system)의 메모리 계층(memory hierarchy)을 충분히 활용하지 못한다.

PDF p. 2, 3, 5, 6

메모리 계층 간 대역폭 격차

Fig.1의 메모리 계층 수치는 계층 간 격차가 매우 큼을 보여준다. GPU SRAM은 19TB/s(20MB), GPU HBM/GDDR6는 0.7-1.5TB/s(24GB 이상), CPU DRAM은 12.8GB/s(128GB 이상), SSD는 순차 3-7GB/s·랜덤 2-4GB/s(4TB 이상)이다. 최신 SSD의 순차 대역폭(7GB/s)은 PCIe 대역폭보다 한 자릿수 낮고, PCIe Gen5의 이론상 32GB/s조차 GPU 연산 처리량(Table 2: A100 1555GB/s, H100 2000GB/s)보다 두 자릿수 느리다. 즉 저장장치-GPU 경로가 구조적 병목이다.

PDF p. 2, 3, 4

멀티벡터 검색 인덱스의 폭발적 크기

정보 검색 시스템에서는 재랭킹(re-ranking)에 필요한 대규모 멀티벡터 문서 임베딩을 매 쿼리마다 반복적으로 저장장치에서 가져와야 하므로 위 문제가 더욱 두드러진다. 멀티벡터 신경망 검색 모델(예: ColBERTv1)은 BM25 대비 인덱스 크기가 최대 210.9배까지 증가하여(Table 3), 이를 전량 메인 메모리에 두는 기존 방식은 대규모 환경에서 확장성 한계에 부딪힌다. 또한 LLM 자체도 GPU 메모리 용량 대비 사용량이 매우 커서(Table 1: Llama3 405b, 배치 256·시퀀스 16384에서 KV Cache만 1008GB) 메모리 압박이 가중된다.

CPU 매개 데이터 이동

일반적인 ML 워크플로우에서는 데이터가 저장장치에서 CPU로 로드된 후 GPU로 전달되며, 이 이동을 CPU가 주도·관리하므로 비효율적이고 이종 시스템의 메모리 계층을 완전히 활용하지 못한다.

PDF p. 2, 3

메모리 매핑(mmap)의 오버헤드

mmap은 파일을 프로세스 가상 주소 공간에 직접 매핑할 수 있게 하지만, ML 워크로드처럼 무작위 접근이 빈번한 경우 잦은 page fault와 OS 수준 관리로 인해 데이터 전송이 크게 느려지고 PCIe 기반 통신의 병목을 악화시킨다.

전량 인메모리 캐시의 비용

검색 인덱스와 문서 임베딩을 시스템 메모리에 전량 저장하는 접근법은 대규모 데이터셋에서 비용상 실현 불가능(cost-prohibitive)하다.

PDF p. 6

전통적 키워드 기반 IR의 표현력 한계

BM25와 같은 키워드 매칭 기반 IR 방법은 CPU 실행에 최적화되어 있으나 동의어, 문맥적 뉘앙스, 복잡한 의미 관계를 포착하지 못한다.

PDF p. 5

기존 SSD 기반 시스템의 범위 제한

DiskANN, SPANN과 같은 기존 SSD 기반 시스템들은 ANN 인덱스만 저장장치로 오프로드할 뿐, 재랭킹을 포함한 전체 neural IR 파이프라인은 다루지 않는다. 또한 토큰 드롭핑(token dropping), 양자화(quantization), 압축(compression)으로 메모리 요구량을 줄이는 최신 시스템들은 멀티벡터 임베딩 검색 자체를 직접 최적화하지 않는다.

제안 방식의 전체 구조와 주요 구성요소

논문 주장

전체 구조 개요

ESPN은 GPUDirect Storage(GDS)를 활용해 SSD에서 GPU 메모리로 데이터를 직접 전송함으로써 CPU 기반 파일 I/O 병목을 우회하는 GPU-centric 아키텍처이다. Candidate generation을 위한 ANN 인덱스는 CPU 메모리에 캐시하고, 재랭킹에 필요한 대용량 멀티벡터 임베딩(재랭킹 인덱스)은 SSD에 오프로드한다. Software prefetcher는 IVF 기반 ANN 검색이 일부 클러스터(δ)를 탐색한 시점에 상위 K개 후보 문서의 임베딩을 GDS로 미리 가져오기 시작해 나머지 클러스터(λ) 탐색과 병렬로 겹치게 하며, ANN 검색이 끝나는 즉시 MaxSim 연산자로 early re-ranking을 수행해 재랭킹 지연을 최소화한다.

PDF p. 7, 8, 9, 10

GPUDirect Storage (GDS)

SSD에서 GPU HBM으로 CPU 개입을 최소화하며 데이터를 직접 전송하는 경로로, ESPN의 모든 임베딩 조회(프리페치 및 critical-path 조회)에 사용된다.

PDF p. 3, 4, 7

ANN 인덱스 (candidate generation index)

FAISS IVF-Flat 또는 IVFPQ 인덱스로, 쿼리와 유사한 후보 문서를 k-means 클러스터링 기반으로 탐색한다. CPU 메모리에 캐시되어 상시 유지된다.

PDF p. 6, 7, 10

재랭킹 인덱스 (multi-vector re-ranking embedding table)

MaxSim 재랭킹에 필요한 문서별 토큰 단위 멀티벡터 임베딩을 저장하며, SSD에 오프로드되어 메모리 사용량을 줄인다.

PDF p. 7, 10

Software Prefetcher

ANN 검색이 δ개 클러스터를 탐색한 시점에 예상 top-K 문서 ID를 산출하고 GDS로 해당 임베딩을 미리 가져와 나머지 ANN 탐색과 지연시간을 중첩시킨다. PrefetchBudget = ANNSearchTime(nprobe=η) − ANNSearchTime(nprobe=δ) (Eq.1), PrefetchStep = (δ/η)×100% (Eq.2)로 제어되며, Query Batch threshold = BW_SSD·PrefetchBudget / (쿼리당 데이터 크기) (Eq.3)로 지연 없이 처리 가능한 최대 배치 크기가 결정된다.

PDF p. 7, 8

Early re-ranking (custom CUDA MaxSim 커널)

프리페치된 임베딩이 도착하는 즉시 MaxSim 유사도 연산을 시작해 ANN 검색과 재랭킹 연산을 중첩시키고 critical path 지연을 줄인다.

PDF p. 7, 8

ESPN-Lite 부분 재랭킹 모듈

전체 후보 대신 상위 64-128개 문서만 MaxSim으로 재랭킹하고 나머지 후보는 1단계 검색 점수로 집계하여, 임베딩 전송량을 줄이고 더 큰 쿼리 배치를 지원한다.

PDF p. 9

ESPN-LIVE 온디맨드 임베딩 합성 모듈

저장된 임베딩 대신 사전 토큰화된 문서를 가져와 ANN 검색과 중첩된 배치 BERT 인코딩(GPU)으로 임베딩을 실시간 생성하고, 놓친 문서는 critical path에서 인코딩한다. 전체 검색 시간 중 GPU 연산 비중이 15% 미만이라는 관찰에 기반해 유휴 GPU 자원을 활용한다.

PDF p. 9, 10

단계별 동작 과정

논문 주장

1단계: 쿼리 인코딩과 ANN 검색 시작

쿼리 임베딩을 생성하고 candidate generation ANN 인덱스에 대해 IVF 기반 클러스터 탐색(nprobe = η = δ + λ)을 시작한다.

PDF p. 7

2단계: PrefetchStep 도달 시 프리페치 개시

ANN 검색이 δ개 클러스터(PrefetchStep 지점)를 탐색하면, prefetcher 스레드가 현재까지의 결과로 top-K 후보 문서 ID를 선정하고 GDS를 통해 해당 멀티벡터 임베딩 프리페치를 시작한다.

PDF p. 7, 8

3단계: 남은 ANN 탐색과 프리페치의 중첩

ANN 검색은 나머지 λ개 클러스터를 계속 탐색하며 recall을 정교화하고, 임베딩 프리페치는 PrefetchBudget 내에서 이와 병렬로 진행된다.

PDF p. 7, 8

4단계: 미스 문서의 critical path 조회

ANN 검색(nprobe = η)이 종료되면 prefetcher가 놓친 소수 문서(약 10% 미만)에 대해서만 critical path에서 GDS로 임베딩을 추가 조회한다.

PDF p. 7

5단계: Early re-ranking

도착한 임베딩에 대해 custom CUDA MaxSim 커널이 즉시 early re-ranking을 수행하며, ANN 검색/프리페치와 중첩되어 최종 재랭킹 워크로드를 최소화한다.

PDF p. 7, 8

6단계 (ESPN-Lite 변형): 부분 재랭킹

재랭킹 대상을 상위 64-128개 문서로 제한해 MaxSim을 적용하고, 나머지 후보의 점수는 1단계 검색기 점수와 집계해 최종 순위를 산출한다.

PDF p. 9

7단계 (ESPN-LIVE 변형): 조기 문서 인코딩

GDS 프리페치 대상을 저장된 임베딩 대신 사전 토큰화된 문서로 바꾸고, ANN 검색과 중첩하여 GPU에서 배치 BERT 인코딩으로 임베딩을 실시간 생성하며, 인코딩되지 못한 소수 문서는 critical path에서 인코딩한다.

PDF p. 9, 10

핵심 알고리즘 및 인덱스 구조

논문 주장

인덱스 이원화 구조

인덱스는 두 계층으로 분리된다. (1) candidate generation용 ANN 인덱스는 FAISS IVF 계열로 CPU 메모리에 상주하며 후보 문서를 빠르게 좁힌다. (2) 재랭킹용 멀티벡터 임베딩 테이블은 문서별 토큰 임베딩을 담고 SSD에 상주한다. 이 분리가 메모리 절감(인덱스의 6-19%만 메모리 유지)의 직접적 근거다.

PDF p. 7, 10, 12

프리페치 제어 수식

프리페칭은 세 수식으로 제어된다. Eq.1 PrefetchBudget = ANNSearchTime(nprobe=η) − ANNSearchTime(nprobe=δ): 프리페치가 숨을 수 있는 시간 창. Eq.2 PrefetchStep = (δ/η)×100%: 전체 nprobe 중 몇 %를 탐색한 시점에 프리페치를 시작하는지. Eq.3 Query Batch threshold = BW_SSD × PrefetchBudget / (쿼리당 데이터 크기): 지연 없이 처리 가능한 최대 쿼리 배치 크기. PrefetchStep이 작을수록 예측 정확도(hit rate)는 낮아지지만 budget은 커지는 트레이드오프가 존재한다.

PDF p. 7, 8

MaxSim 기반 재랭킹

재랭킹은 멀티벡터 유사도 함수인 MaxSim 연산자를 custom CUDA 커널로 구현해 GPU에서 수행한다. 임베딩 전체가 모이기를 기다리지 않고 도착하는 대로 연산을 시작하는 early re-ranking 방식으로 파이프라인화된다.

PDF p. 7, 8

ESPN-LIVE의 계산-저장 교환

ESPN-LIVE는 저장(storage)을 연산(compute)으로 교환하는 설계다. 전체 검색 시간 중 GPU 연산 비중이 15% 미만이라는 관찰을 근거로, 유휴 GPU에서 사전 토큰화된 문서를 배치 BERT 인코딩하여 임베딩을 즉석 생성한다. 그 결과 재랭킹 임베딩 테이블(v1 16.8GB, v2 255.4GB)을 저장할 필요가 사라진다.

PDF p. 9, 10, 11

실험 구성

논문 주장

하드웨어 및 구현

ESPN 프레임워크는 약 600줄의 C++와 1100줄의 Python으로 구현되었으며, 직접 데이터 전송에 Nvidia GPUDirect Storage를, ANN 검색에 FAISS를 사용한다. 실험은 Intel Xeon W-2255 CPU, Nvidia A5000 GPU, Samsung PM983 SSD 환경에서 수행되었다.

PDF p. 10, 11

데이터셋과 인덱스 구성

MS-MARCO v1(8.8M 문서, in-domain)과 MS-MARCO v2(138.4M 문서, out-of-domain)를 사용한다. v1은 FAISS IVF-Flat 인덱스(4.3GB), v2는 IVFPQ(m=128, nbits=8) 인덱스(17.5GB)를 쓰며, 재랭킹 임베딩 테이블은 v1 16.8GB, v2 255.4GB로 SSD에 저장된다. 메모리 구성은 cgroups로 시뮬레이션되었다.

PDF p. 10, 11

비교 대상과 지표

비교 대상은 Mmap, Virtual memory(+swap space), naive GDS(프리페칭 미적용), 완전 인메모리(DRAM) 캐시 솔루션이며, 검색 품질 비교에는 Pyserini 1단계 검색기(docT5q, uniCOIL, DeepImpact)와 ColBERTv2/ColBERTer 재랭킹을 사용한다. 측정 지표는 종단간 쿼리 지연시간(ms), 메모리 사용량(GB), 쿼리 처리량, prefetcher hit rate(%), MRR@10/MRR@100, Recall@1K, critical path 임베딩 접근 지연시간(ms), 인덱스 크기(GB)이다.

PDF p. 10, 11

재랭킹 깊이 설정

정확(exact) 방식에서는 상위 1024개 문서를 재랭킹하고, 근사(approximate) 방식인 ESPN-Lite와 ESPN-LIVE에서는 ANN 검색 결과 상위 128개 문서만 재랭킹한다. 임베딩 조회와 프리페칭은 90% prefetcher hit rate를 달성하도록 구성되었다.

주요 결과와 의미

논문 주장

Prefetcher hit rate

Figure 7에 따르면 MS-MARCO v1에서 hit rate는 prefetch step 5%일 때 74%(nprobe=1000), 85%(nprobe=3000)로 측정되며, 30% 지점에서는 본문에 '90% 이상'이라는 근사 표현으로만 보고된다. MS-MARCO v2에서는 5%일 때 68%(nprobe=160), 70%(nprobe=200)이며, 30% 지점은 각각 '약 89%', '약 90%'로만 언급된다. 정확한 수치가 제시된 5% 지점 기준으로도 대부분의 임베딩을 미리 가져올 수 있어 critical path 접근 수요가 크게 줄어드는 경향을 보인다.

PDF p. 11

MS-MARCO v1 종단간 지연시간 (Table 5)

Table 5에 따르면 Mmap은 메모리 10GB에서 183.2ms로 매우 느리며 30GB에서 53.4ms까지 감소한다. Virtual memory는 76.0ms에서 45.6ms(이 시점에는 임베딩 테이블이 메모리에 완전히 캐시됨)로 감소한다. naive GDS는 10-30GB 전 구간에서 56.6-58.3ms로 메모리 증가에 거의 반응하지 않는다. ESPN은 46.5-47.2ms로 메모리 구성에 관계없이 거의 일정하며, ESPN-Lite(top-128 근사)는 45.2-45.4ms로 표 내 최저치를, ESPN-LIVE는 53.4-54.1ms를 기록한다.

PDF p. 12

MS-MARCO v2 종단간 지연시간 (Table 6)

인덱스가 물리 메모리를 크게 초과하는 v2에서는 격차가 더 벌어진다. Table 6에 따르면 Mmap은 32GB에서 271.5ms, 256GB에서도 188.6ms이며, Virtual memory는 32GB부터 256GB까지 전 구성에서 OOM이 발생해 동작하지 않는다. naive GDS는 94.1-95.3ms, ESPN은 84.9-85.8ms, ESPN-Lite는 81.7-82.3ms(최저), ESPN-LIVE는 85.2-85.6ms이다. ESPN은 이 구성에서 1ms 이하의 임베딩 접근 지연시간을 제공한다고 저자들은 보고한다.

종합 성능 지표

저자들은 ESPN이 mmap 대비 종단간 쿼리 지연시간을 3.1-3.9배 줄이고, 완전 캐시 방식 대비 검색 인덱스의 6-19%만 메모리에 두면서 메모리 사용량을 5-16배 절감하며, 완전 캐시 대비 1.02배 수준의 종단간 지연시간을 유지한다고 보고한다.

PDF p. 1, 12

검색 품질 유지

ColBERTv2와 ColBERTer에서 상위 64-128개 문서만 재랭킹해도 MRR@10 점수의 99-99.7%가 유지된다고 저자들은 보고하며(Figure 5), 이 부분 재랭킹은 쿼리당 임베딩 데이터 전송량을 8-16배 감소시킨다. ESPN-Lite와 ESPN-LIVE의 실제 품질 저하는 MRR@10 기준 0.5%로 보고된다. 메모리 기반 방식과 저장장치 기반 방식(GDS, ESPN)의 검색 점수는 동등하다고 저자들은 밝힌다.

배치 처리량과 확장성

종단간 시스템에서 ESPN은 배치 크기 12까지 DRAM 기반 솔루션과 경쟁력을 유지하며, GDS 기반 검색 대비 처리량을 68% 향상시킨다고 저자들은 Figure 8을 근거로 보고한다. naive GDS 솔루션은 critical path에서 DRAM 대비 7.7배 높은 임베딩 접근 지연시간을 보인다(Figure 9). ESPN-Lite는 대역폭 요구량을 8배 줄임으로써 배치 크기 96까지 완전 캐시 DRAM 솔루션과 경쟁력을 유지한다.

결론 및 시사점

분석자 해석

논문의 결론

저자들은 GPU-centric 저장장치 접근 최적화가 대규모 멀티벡터 IR의 메모리 비용과 지연시간 트레이드오프를 근본적으로 개선함을 보였다고 주장한다. ESPN은 GDS 직접 전송, ANN 진행도 기반 프리페칭, early re-ranking을 결합해 저장장치 지연을 ANN 검색 시간 뒤에 숨기며, ESPN-Lite는 대역폭을, ESPN-LIVE는 저장 용량 자체를 제거한다.

PDF p. 1, 15

분석자 해석: 세 변형의 선택 기준

세 변형은 서로 다른 자원 제약에 대응한다고 해석할 수 있다. 저장 용량이 충분하고 정확도 손실을 원치 않으면 ESPN(exact, top-1024), SSD 대역폭이 병목이고 배치가 크면 ESPN-Lite, 저장 용량 자체가 제약이고 GPU가 유휴 상태라면 ESPN-LIVE가 유리하다. v2 실험에서 ESPN-LIVE는 255.4GB의 임베딩 테이블 없이도 ESPN과 거의 동일한 지연시간(85.2-85.6ms vs 84.9-85.8ms)을 나타낸다. 다만 v1에서는 ESPN-LIVE가 ESPN보다 약 7ms 느려, 인코딩 비용이 저장장치 조회 비용을 상회하는 구간이 존재한다.

핵심 용어

GPUDirect Storage (GDS)
NVIDIA의 데이터 전송 기술로, CPU와 시스템 메모리를 경유하지 않고 SSD에서 GPU 메모리(HBM)로 데이터를 직접 전송한다. ESPN의 모든 임베딩 조회 경로에 사용된다. PDF p. 3, 4, 7
ESPN (Embedding from Storage Pipelined Network)
GDS 기반 직접 전송, software prefetching, early re-ranking을 결합해 저장장치 지연시간과 GPU 처리량을 정렬시키는 이 논문의 제안 시스템. PDF p. 1, 7
ESPN-Lite
ESPN의 대역폭 효율 확장. 전체 후보 대신 상위 64-128개 문서만 MaxSim으로 재랭킹하고 나머지는 1단계 검색 점수로 집계하여 임베딩 전송량을 8-16배 줄인다. PDF p. 9
ESPN-LIVE
ESPN의 저장 용량 절감 확장. 사전 계산된 임베딩을 조회하는 대신 사전 토큰화된 문서를 GPU에서 배치 인코딩하여 임베딩을 실시간 합성하므로, 재랭킹 임베딩을 저장할 필요가 없다. PDF p. 9, 10
멀티벡터 검색 (multi-vector retrieval)
문서를 단일 벡터가 아니라 토큰 단위 여러 벡터로 표현하는 신경망 검색 방식(ColBERT 계열). 검색 품질은 높지만 인덱스 크기가 BM25 대비 최대 210.9배까지 증가한다. PDF p. 5, 6
MaxSim 연산자
멀티벡터 재랭킹에서 사용하는 유사도 함수로, 쿼리 토큰마다 문서 토큰들과의 최대 유사도를 취해 합산한다. ESPN에서는 custom CUDA 커널로 GPU에서 실행된다. PDF p. 5, 7, 8
Candidate generation / re-ranking 2단계 파이프라인
1단계에서 ANN 인덱스로 후보 문서를 빠르게 좁히고, 2단계에서 연산 집약적인 멀티벡터 유사도로 후보를 정밀 재정렬하는 전형적인 neural IR 구조. ESPN은 1단계 인덱스는 메모리에, 2단계 임베딩은 SSD에 배치한다. PDF p. 6, 7
IVF / nprobe
FAISS의 inverted file 기반 ANN 인덱스 구조. 벡터를 k-means 클러스터로 나누고, 쿼리 시 nprobe개 클러스터만 탐색한다. ESPN의 프리페치 시점은 이 nprobe 진행도를 기준으로 결정된다. PDF p. 6, 7
PrefetchStep / PrefetchBudget
PrefetchStep = (δ/η)×100%는 전체 nprobe(η) 중 몇 %를 탐색한 시점에 프리페치를 시작할지를 나타낸다. PrefetchBudget = ANNSearchTime(η) − ANNSearchTime(δ)는 프리페치가 지연 없이 숨을 수 있는 시간 창이다. PDF p. 7, 8
Prefetcher hit rate
프리페처가 미리 가져온 임베딩이 실제 최종 top-K 후보에 포함된 비율. 값이 높을수록 critical path에서 추가 조회할 문서가 줄어든다. 실험에서는 약 90%를 목표로 구성되었다. PDF p. 8, 11
Early re-ranking
임베딩이 전부 도착하기를 기다리지 않고 도착하는 대로 MaxSim 연산을 시작하는 파이프라이닝 기법. ANN 검색과 재랭킹 연산을 중첩시켜 critical path 지연을 줄인다. PDF p. 7, 8
mmap (memory mapping)
파일을 프로세스 가상 주소 공간에 직접 매핑하는 OS 기능. 무작위 접근이 많은 ML 워크로드에서는 잦은 page fault와 OS 관리 오버헤드로 성능이 크게 저하되며, 이 논문의 주요 비교 대상이다. PDF p. 4, 12
MRR@10 / Recall@1K
검색 품질 지표. MRR@10은 상위 10개 결과 내 첫 정답의 역순위 평균이고, Recall@1K는 상위 1000개 결과 내 정답 포함 비율이다. ESPN-Lite·ESPN-LIVE의 MRR@10 저하는 저자들이 0.5%로 보고했다. PDF p. 9, 11, 12
MS-MARCO v1 / v2
평가에 사용된 대규모 문서 검색 벤치마크. v1은 8.8M 문서(in-domain, 재랭킹 테이블 16.8GB), v2는 138.4M 문서(out-of-domain, 재랭킹 테이블 255.4GB)이다. PDF p. 10, 11
