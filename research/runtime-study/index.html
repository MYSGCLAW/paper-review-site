핵심 요약

  1. 분석 프레임은 이미 완성도가 높다. 계획서는 Runtime 역할 13분류(§3), 자원 할당 7분류(§4), 외부 관찰 9단계 실행 흐름(§5.3), 내부 12단계 처리 흐름(§7), 계층 경계 5단(§5.5), 내부 인터페이스 구조체 5종(§6)을 갖추고 있다. 새 프레임을 만들 필요 없이 이 프레임을 4개 시스템에 그대로 적용하는 것이 다음 단계다.
  2. 공통 객체 6종이 비교의 축이다. Device, Context, Buffer, Executable, Queue/Stream, Event/Fence(/Future)는 계획서 §5.2의 객체 표 그대로 4개 시스템 모두에 대응 가능하며[계획서 §2 중점 객체 목록], 가장 먼저 정리할 용어는 계획서 Phase 0의 10개 용어다.
  3. 각 시스템은 LP6-PIM의 서로 다른 설계 질문에 대응한다. CUDA → API 2계층화(무엇을 숨기고 무엇을 노출할 것인가), Level Zero → 명령 기록과 제출의 분리(Stream Executor/Command Backend 구조), PJRT → Compiler-Runtime 계약, UPMEM → 실존 PIM의 자원 할당·실행 흐름 [계획서 §2 각 분석 목적].
  4. LP6-PIM의 근본 차이는 "메모리 자원과 연산 자원의 결합"이다. 별도 device memory를 가진 GPU/NPU와 달리 연산 대상 DRAM 공간 자체가 실행 자원이며, bank/row/register/timing 상태를 Runtime이 추적하고 bank/PU 충돌을 Runtime이 직접 판단한다는 것이 계획서의 핵심 가정이다[계획서 §8.5]. 단, 계획서 스스로 "가능성"이라는 표현을 쓰고 있어 이 가정은 검증 대상이다.
  5. 내부 모듈 파이프라인 초안이 존재하지만 경계가 한 곳 어긋난다. 계획서 §11은 Task Scheduler → Stream Executor → Command Backend → Driver 순서인데, §6.5는 Stream Executor가 이미 완성된 command_buffer를 Driver/HAL에 넘기는 것으로 기술한다. Command Lowering이 어느 모듈 소관인지가 설계 쟁점이다.
  6. 참고 원문이 계획서 1건뿐이다. CUDA/L0/PJRT/UPMEM에 대한 모든 API 수준 서술은 현재 [배경지식] 상태이며, 계획서 §13이 지정한 외부 공식 자료 6종과 내부 자료 4종(홍pim.pdf, Micron LPDDR6-PIM draft, Overview md, kickoff pdf)은 아직 reference로 확보되지 않았다.
  7. 수집 우선순위는 계획서 §10이 이미 확정했다. ① CUDA Runtime vs Driver API 차이 ② Stream/Event/Context/Memory 객체 관계 ③ Level Zero Command List/Queue ④ Runtime 내부 Task/Resource/Submission 구조 ⑤ Compiler-Runtime-Driver 경계 순이며, PJRT와 UPMEM은 그다음이다.
  8. Completion 모델이 LP6-PIM에서 독특해질 수 있다. 계획서 §8.1이 "Operation Completion Timing"을 Runtime 관리 상태로 지정한 것은, interrupt/polling 대신 DRAM timing 기반의 결정론적 완료 판정 가능성을 시사한다[추정 — LP6-PIM spec 확인 필요].

세부 정리

1. 공통 Runtime 분석 프레임

1.1 공통 객체 — 가장 먼저 정리할 용어 (질문 1에 대한 답)

계획서 Phase 0(§9)이 지정한 10개 용어가 출발점이다: Device, Context, Buffer/Memory Object, Module/Executable, Kernel/Primitive/Task, Stream/Queue, Event/Fence/Future, Command List/Command Buffer, Resource Reservation, Submission/Completion. 산출물은 runtime_common_terms.md.

이 중 시스템 간 의미 차이가 커서 정의를 먼저 고정해야 하는 것 세 가지를 뽑으면:

  • Context — CUDA는 프로세스별 주소 공간+실행 상태, Level Zero는 자원 소유 컨테이너, PJRT는 명시 객체가 없음(Client가 겸함) [배경지식]. LP6-PIM에서는 "Register + Bank + PIM Mode Context"로 재정의된다[계획서 §8.5]. 즉 Context는 시스템마다 격리 범위가 다르므로 "무엇으로부터의 격리인가"를 축으로 정의해야 한다.
  • Command List vs Queue/Stream — 명령의 기록 컨테이너와 제출 대상을 분리하는지(L0)와 합쳐 놓는지(CUDA stream)가 갈린다. LP6-PIM Command Backend 산출물의 형태를 결정하는 용어다.
  • Event vs Fence vs Future — 명령 단위 세밀 동기화(Event), 제출 단위 완료(Fence), 상위 계층으로의 비동기 결과 전달(Future)로 층위가 다르다[계획서 §2.2, §2.3의 분석 목적에서 이 구분을 요구]. Completion Tracker가 셋 중 무엇을 제공할지 정해야 한다.

공통 객체 표 템플릿은 §5.2에 있다(객체 | 역할 | 생성 API | 제거 API | 내부 상태). 시스템별 분석 시 이 표를 그대로 채운다.

1.2 공통 처리 단계

계획서는 두 개의 흐름을 제공하며, 둘을 겹쳐 읽어야 한다.

외부에서 관찰되는 9단계 [계획서 §5.3]: Device 선택 → Context 준비 → Memory 할당 → Binary/Module Load → Data Copy → Kernel/Task Submit → Queue 실행 → Completion 확인 → Resource 해제. 각 단계마다 호출 API, 입력 인자, 생성/변경 객체, 동기/비동기 여부, Driver 호출 여부, 실패 조건, 완료 확인 방법을 기록한다.

내부 12단계 [계획서 §7]와 §11 모듈 파이프라인의 대응 (이번 정리에서의 재구성):

12단계내용담당 모듈 [계획서 §11 기준 매핑, 재구성]
1–2API 호출, 입력·Capability 검증Runtime API + Request Validator
3Primitive/Kernel 요청 → 내부 Task 변환Task Normalizer
4데이터·제어 의존성 등록Dependency Manager
5–6자원 상태 조회, 실행 자원 예약Resource Manager
7실행 순서·병렬성 결정Task Scheduler
8Device-specific Command로 LoweringCommand Backend

이 매핑에서 8단계(Lowering)와 9단계(제출)의 담당 모듈 순서가 계획서 내부에서 일관되지 않다는 점이 드러난다(쟁점 1).

1.3 자원 할당 종류

"Runtime이 자원을 할당한다"를 7종으로 세분화한다[계획서 §4]. LP6-PIM 예시는 계획서가 직접 제시한 것이다.

핵심 관찰: 기존 GPU runtime에서는 메모리 자원(1)과 연산 자원(4)이 독립적으로 할당되지만, LP6-PIM에서는 데이터가 놓인 bank가 곧 연산 위치이므로 두 할당이 결합된다[계획서 §8.5 "Memory: 연산 대상 DRAM 공간과 결합", "Data Placement: Bank/Row/Layout Mapping 중요"]. 이것이 Resource Manager 설계의 출발점이다.

1.4 상위 계층 / Runtime / Driver / HAL 경계

계층 스택은 Application·Framework → Runtime API → Runtime 내부 모듈 → Driver/HAL API → Firmware/Controller → Hardware [계획서 §5.5]. 경계마다 오가는 정보를 계획서 §6이 구조체 5종으로 규정했고, 각 구조체에 붙은 "확인할 내용"이 곧 시스템별 분석 질문이다.

Scheduling 책임 분해 표 [계획서 §5.4]: Tiling, Memory Placement, Queue 선택, Compute Unit 배치, Register 사용, Command Timing, Completion 판단의 7개 결정 항목 × Compiler/Runtime/Driver/Firmware/Hardware 5주체. 계획서에서 이 표는 의도적으로 빈 템플릿이다 — 시스템별 분석에서 채워야 할 답이다. 다만 LP6-PIM 열에 대해서는 계획서 §8.5가 이미 가설을 제시했다: Compute Unit 배치와 충돌 판단은 Runtime, Command Timing은 Command Backend, Register 상태 추적은 Runtime "가능성". 이 가설의 검증이 전체 분석의 목적 함수다.

추적할 코드 흐름 [계획서 §11 작업 1]: cudaSetDevice → cudaMalloc → cudaMemcpyAsync → cudaLaunchKernel → cudaEventRecord → cudaEventSynchronize → cudaFree. 호출마다 입력/생성 객체/수정 상태/실행 주체/비동기 여부/완료 확인/오류(즉시 vs 지연)를 표로 기록.

채울 산출물: cuda_runtime_analysis.md, cuda_runtime_sequence.md, cuda_runtime_vs_driver_api.md [계획서 Phase 1]. 특히 Runtime↔Driver 대응표(작업 2)는 [배경지식] 수준에서 cudaSetDevice↔cuDevicePrimaryCtxRetain+cuCtxSetCurrent, cudaMalloc↔cuMemAlloc, cudaLaunchKernel↔cuLaunchKernel 대응이 예상되나 문서로 확정해야 한다[확인 필요].

LP6-PIM에 주는 힌트 (질문 2에 대한 답)

  • Runtime API는 Context를 암묵 관리(primary context)하고 Driver API는 명시 관리한다[배경지식]. 같은 방식으로 LP6-PIM도 ExecutionDomain을 암묵 관리하는 고수준 API(framework용)와 도메인·register context를 명시 제어하는 저수준 API(HAL 검증·튜닝용)의 2계층으로 나눌 수 있다.
  • cudaMalloc(메모리 자원)과 kernel launch(연산 자원 사용)가 분리된 구조를 기준선으로 삼아, LP6-PIM에서 두 할당이 결합될 때 API에 무엇이 추가로 필요한지(bank/row 지정, PU mask)를 역으로 도출한다.
  • 오류의 즉시 반환/지연 반환 구분[계획서 Phase 1 체크리스트]은 LP6-PIM Completion Tracker의 error 모델(제출 시 오류 vs 실행 중 오류)의 참조 모델이다.

2.2 Level Zero (우선순위 2)

왜인가: Command List(기록)와 Command Queue(제출)의 분리 구조가 CHAMP-PIM의 Stream Executor와 Command Backend 구조를 구체화하는 직접 모델이다[계획서 §2.2].

수집 순서

  2. 최소 예제(zello_world 류)로 초기화→모듈 로드→list 기록→queue 제출→동기화 전체 시퀀스 추적 [배경지식].
  3. 심화: Immediate Command List vs 일반 Command List, Command List 재사용 조건, Queue Group과 engine 종류(compute/copy ordinal) [계획서 Phase 2 체크리스트].

채울 산출물: level_zero_runtime_analysis.md, level_zero_command_flow.md, level_zero_event_fence.md.

CHAMP-PIM에 주는 힌트 (질문 3에 대한 답)

  • Command List = "lowering된 명령의 재사용 가능한 컨테이너". LP6-PIM Command Backend의 산출물을 일회성 제출이 아니라 PIM command list 객체로 만들면, 같은 GEMV tile 시퀀스를 반복 실행할 때 lowering 비용을 절약할 수 있다. 단, active row·register 상태가 동적으로 변하면 재사용성이 깨질 수 있다(쟁점 3).
  • Command Queue 제출이 별도 단계라는 점이 Stream Executor(제출 담당)와 Command Backend(생성 담당)를 다른 모듈로 두는 근거가 된다.
  • Event(명령 단위, list 간 의존성 표현)와 Fence(queue 제출 단위 완료)의 이원 구조[계획서 §2.2 분석 목적]는 Completion Tracker가 Task 단위 완료와 제출 단위 완료를 구분해야 함을 시사한다.
  • Immediate Command List는 기록-제출 분리를 생략하는 저지연 경로다[배경지식]. 작은 PIM 연산(단발 GEMV)용 fast path 설계의 참조가 된다.

LP6-PIM에 주는 힌트 (질문 4에 대한 답)

  • Executable(컴파일 산출물)과 LoadedExecutable(device에 로드된 상태)의 분리[계획서 Phase 3 체크리스트]는 CHAMP Compiler 산출물(Primitive Descriptor 등)과 "특정 ExecutionDomain에 로드·바인딩된 실행체"를 다른 객체로 관리하라는 힌트다.
  • Buffer가 device 위치·layout을 캡슐화하고 비동기 ready 상태를 갖는 구조는, LP6-PIM Buffer에 bank/row/layout mapping을 속성으로 넣고 준비 완료를 event로 표현하는 모델의 참조다.
  • Future 기반 완료/오류 전달[계획서 §2.3 중점 객체]은 Runtime 내부의 Event/Fence와 별개로 상위 계층에 노출할 비동기 결과 표현이 따로 필요함을 보여준다 — Completion Tracker의 외부 인터페이스.
  • ExecuteOptions는 계획서 §6.1 SubmitRequest.execution_hints 필드의 실존 사례다. Compiler가 Runtime에 넘기는 metadata 목록(shape, layout, donation 등)을 여기서 수집한다[확인 필요].

왜인가: 상용 PIM runtime의 실물로, 자원 할당 단위와 host-PIM 데이터 이동, binary load, launch 구조를 LP6-PIM과 직접 비교할 수 있다[계획서 §2.4].

수집 순서

  2. 공식 예제 [계획서 §13] — dpu_alloc → dpu_load → dpu_copy_to/dpu_push_xfer → dpu_launch → dpu_copy_from → dpu_free 전체 흐름이 있는 예제 1개를 기준 시퀀스로 [배경지식: SDK 동봉 예제, 추가로 PrIM 벤치마크 스위트].
  3. 심화: rank 단위 병렬 전송(dpu_prepare_xfer/dpu_push_xfer), 비동기 launch 지원 여부, DPU별/Rank별 자원 관리 단위, Runtime-Driver 경계 [계획서 Phase 4 체크리스트].

채울 산출물: upmem_runtime_analysis.md, upmem_vs_lp6_pim.md.

LP6-PIM과의 같고 다름 (질문 5에 대한 답, 현 시점 정리)

같은 점 [추정, Phase 4에서 검증]:

  • 자원을 개별 유닛이 아니라 집합 단위로 할당 (DPU Set ↔ LP6의 Bank Set/PU 집합).
  • 명시적 binary/program load 단계가 존재 (DPU Binary ↔ CONFIG_DEVICE/CONFIG_REGISTER 류의 준비 task [계획서 §8.3]).
  • host↔PIM 데이터 이동이 명시적 Transfer 단계로 분리되고, Launch 후 Synchronization으로 완료를 확인.

다른 점 [계획서 §8.5의 차이 표에 근거]:

  • UPMEM DPU는 독립 코어+전용 MRAM/WRAM으로 "코어 중심" 실행 단위인 반면, LP6-PIM은 Sub-channel+Bank Set 중심이며 연산 대상이 host가 쓰는 DRAM 공간과 결합된다.
  • UPMEM runtime은 DRAM row/timing/register 상태를 사용자에게 노출하지 않는 것으로 보이나[추정], LP6-PIM은 Active Row, IR/SR/OR, Command Timing을 Runtime이 추적할 가능성이 있다.
  • 데이터 이동이 UPMEM은 "복사" 모델인 반면 LP6-PIM은 Bank/Row/Layout mapping(placement) 문제가 중심이 된다.

3. LP6-PIM Runtime 설계 연결

기존 runtime 분석 결과를 계획서 §8의 LP6-PIM 항목과 §11 모듈 초안에 연결한다. 각 구성요소마다 "정의 초안(계획서 근거) — 참조 모델 — 열린 설계 질문" 순으로 정리한다.

3.1 ExecutionDomain

정의 초안 [계획서 §8.2]: { device_id, subchannel_id, bank_set(EVEN|ODD), active_row, pu_mask, register_context, pim_mode, pit_group }. 실행이 일어나는 격리·충돌 판단 단위.

참조 모델: CUDA Context(격리 범위) + Level Zero Command Queue(제출 대상 바인딩) + GPU SM(연산 자원 슬롯)의 결합체에 해당한다[재구성]. 기존 runtime에서 세 개념이 분리돼 있던 것이 LP6-PIM에서는 메모리-연산 결합 때문에 하나로 묶인다는 것이 계획서의 관점이다[계획서 §8.5 Context 행: "Register + Bank + PIM Mode Context"].

열린 질문: ① active_row는 도메인의 정체성인가, 도메인이 가진 동적 상태인가(도메인 수명 결정에 직결). ② 도메인과 Stream이 1:1인가 N:1인가. ③ 도메인 전환 비용(register 저장/복원, PREmb/ACTmb)은 누가 계산하는가. [모두 LP6-PIM spec 확인 필요]

3.2 Resource Manager

정의 초안: TaskResourceRequest를 받아 ResourceDecision(runnable 여부, reservation_token, conflicting_tasks, earliest_start_time)을 반환한다[계획서 §6.2–6.3]. 추적해야 할 상태는 계획서 §8.1의 11종: PIM Enable State, Sub-channel, Even/Odd Bank Set, Active Row, PU Ownership, IR/SR/OR Register State, RICR/RISR State, PIT Group, PIMX Mode, Command Timing, Operation Completion Timing.

참조 모델: UPMEM의 DPU Set 할당(집합 단위 할당·해제)이 가장 가까운 실물이고, CUDA cudaMalloc류의 메모리 할당과 GPU의 암묵적 SM 배치가 분리 기준선이다. LP6-PIM에서는 이 둘이 결합되므로 Allocation Table에 bank/row/layout이, Resource Table에 PU ownership과 register 상태가 함께 들어가야 한다[계획서 Phase 6 체크리스트: Resource Table/Allocation Table 필드 정의].

질문 6에 대한 현 시점 답 (Runtime 직접 추적 vs 위임): 계획서 §8.1 목록이 "Runtime이 관리해야 할 상태" 후보 전체다. 이 중 논리 소유권(PU Ownership, Bank Set 예약, PIT Group 구성, Buffer↔Row mapping)은 충돌 판단에 필요하므로 Runtime 추적이 불가피해 보이고[계획서 §8.5 Scheduling 행], cycle 수준 Command Timing 집행과 인터럽트 처리, 실제 register write의 물리 시퀀스는 Driver/HAL(또는 존재한다면 Firmware/Controller)로 내릴 수 있는 후보다[추정]. 단 계획서 §8.5는 Timing을 "Backend가 고려"라고 적어 Runtime 쪽에 두는 가정이므로, 이 경계는 쟁점 2로 남는다.

3.3 Task Scheduler

정의 초안: ResourceDecision을 받아 실행 순서와 병렬성을 결정하고, ScheduledTask{task, selected_execution_domain, wait_events[], signal_event}를 산출한다[계획서 §6.4]. 내부 Task 어휘는 계획서 §8.3의 13종(CONFIG_DEVICE, CONFIG_REGISTER, LOAD_VECTOR, LOAD_SCALE, CLEAR_OUTPUT, ACTIVATE_BANK_SET, GEMV_TILE, STORE_OUTPUT, PRECHARGE_BANK_SET, READ_REGISTER, WRITE_REGISTER, RESET_CONTEXT)이다.

참조 모델: CUDA Stream ordering(같은 stream 내 in-order 보장)과 Event 기반 cross-stream 의존성이 순서 의미론의 기준선. GPU에서는 SM 배치를 HW가 하지만 LP6-PIM에서는 Even/Odd Bank Set 병렬성과 Bank/PU 충돌 회피를 Scheduler가 직접 판단한다는 것이 계획서의 가정이다[계획서 §8.5 Scheduling 행].

열린 질문: Scheduler가 Queue(도메인)까지 선택하는가, Task batching(여러 GEMV_TILE 묶음 제출)을 지원하는가[계획서 §6.4 확인 내용]. Even/Odd 동시 활성화의 하드웨어 제약은 spec 확인 필요.

3.4 Stream Executor

정의 초안: 도메인/stream별 제출 창구. ScheduledTask를 받아 in-order 제출을 보장하고 stream 간 의존성을 event로 연결하며, DriverSubmission{command_buffer, referenced_buffers, wait/signal events, completion_cookie}를 Driver/HAL에 넘긴다[계획서 §6.5].

참조 모델: Level Zero Command Queue가 직접 모델이다[계획서 §2.2 분석 목적: "CHAMP-PIM의 Stream Executor... 구조 구체화"]. CUDA stream의 암묵 제출과 L0의 명시 제출 중 어느 쪽을 외부 API로 노출할지는 CUDA 2계층 분석(§2.1) 결과에 달려 있다.

열린 질문(쟁점 1과 직결): §6.5에서는 Executor가 이미 완성된 command_buffer를 넘기는데, §11 파이프라인에서는 Executor 다음에 Command Backend가 온다. Executor가 Backend를 호출해 lowering을 받아 제출하는 구조인지, Backend가 Driver 직전의 최하층인지 확정해야 한다.

3.5 Command Backend

정의 초안: 논리 Task를 PIM command 시퀀스로 lowering한다. 계획서 §8.4의 GEMV_TILE 예시: Device/GCR/RICR 설정 → ACTmb → LdIREG(또는 WR-R) → LdSREG(또는 WR-R) → PIMX-CLR → LdMAC × N → StOREG(또는 RD-R) → PREmb. DRAM/PIM Command Timing을 이 계층이 고려한다[계획서 §8.5 Timing 행].

참조 모델: Level Zero의 Command List encode(명령을 미리 기록해 두는 것)와 CUDA의 Launch Descriptor 생성[계획서 Phase 1: "cudaLaunchKernel 입력과 내부 Launch Descriptor"]이 참조. 기존 GPU에서는 timing을 HW가 처리하지만 LP6-PIM에서는 Backend가 command 간 간격까지 고려해야 할 수 있다는 점이 최대 차이다[계획서 §8.5].

열린 질문: ① 산출물 형태 — HAL이 받는 것이 command buffer인가, register write 목록인가, 고수준 요청인가(§6.5 확인 내용과 동일 질문). ② LdIREG vs WR-R처럼 같은 효과의 명령 변형 중 선택 기준(성능? 모드?)은 spec 확인 필요. ③ N(LdMAC 반복 수)은 Compiler가 주는가 Runtime이 계산하는가 — Compiler/Runtime 경계 질문.

3.6 Completion Tracker

정의 초안: DriverSubmission.completion_cookie로 제출과 완료를 매칭하고, polling/interrupt/callback 중 하나로 완료를 감지해[계획서 §3 Completion 처리 행] 객체 상태 갱신과 Event Signal(12단계 중 10–12)을 수행한다. 오류는 제출 시점의 즉시 오류와 실행 중 지연 오류를 구분한다[계획서 §3 Error 처리, §11 작업 1].

참조 모델: L0의 Event(세밀)/Fence(제출 단위) 이원 구조, CUDA Event의 record/query/synchronize, PJRT Future(상위 노출용)의 3층을 조합하는 것이 자연스럽다[재구성].

LP6-PIM 특수 가설: 계획서 §8.1이 "Operation Completion Timing"을 관리 상태로 지정했다. DRAM command timing이 결정론적이라면 interrupt 없이 시간 기반 완료 판정이 가능할 수 있다[추정]. 이 경우 Completion Tracker는 "완료 통지를 기다리는" 모듈이 아니라 "완료 시각을 계산하고 검증하는" 모듈이 되어 기존 runtime과 성격이 달라진다. LP6-PIM spec에서 완료 통지 메커니즘(RISR가 상태 register 역할인지 등) 확인이 필요하다[확인 필요].

쟁점과 불확실성

쟁점 1 — Stream Executor와 Command Backend의 순서/경계 (계획서 내부 불일치). §6.5는 Stream Executor가 완성된 command_buffer를 Driver/HAL에 넘긴다고 기술하는 반면, §11 모듈 파이프라인은 Executor 다음에 Command Backend를 둔다. Lowering(12단계 중 8)이 제출(9)보다 앞서야 한다는 §7과 조합하면, "Executor가 Backend를 호출해 lowering 결과를 받아 제출"하는 구조와 "Backend가 Driver-facing 최하층"인 구조 두 해석이 가능하다. Level Zero 분석(기록과 제출의 분리)이 이 결정의 근거 자료가 된다.

쟁점 2 — Scheduling 책임을 Runtime으로 끌어올리는 가정의 강도. 계획서 §8.5는 GPU/NPU 대비 LP6-PIM에서 Runtime이 Bank/PU 충돌을 직접 판단하고 Backend가 timing을 고려한다고 보지만, Register 행에는 "추적할 가능성"이라는 유보 표현을 썼다. Runtime이 timing까지 관리하면 유연하지만 software 오버헤드가 커지고, HAL/Controller가 흡수하면 Runtime은 단순해지나 하드웨어 요구가 커진다. 어느 쪽이 실현 가능한지는 Micron draft와 kickoff 자료의 HAL 구조를 봐야 판단할 수 있다[확인 필요].

쟁점 3 — 기록-제출 분리(L0 모델)와 동적 PIM 상태의 양립성. Command list 재사용은 lowering 비용을 줄이지만, LP6-PIM command 시퀀스는 active row, register 상태, PIM mode 같은 동적 상태에 의존한다(ACTmb/PREmb가 시퀀스에 포함됨 [계획서 §8.4]). 미리 기록한 list가 제출 시점 상태와 어긋날 수 있으므로, "완전 사전 기록 / parameterized template / 제출 시점 lowering" 중 선택이 필요하다[추정]. L0의 command list 재사용 조건(Phase 2 체크리스트)이 참고 답을 줄 것이다.

쟁점 4 — Buffer 객체의 정의 충돌. 기존 runtime의 Buffer는 위치가 추상화된 handle이지만, LP6-PIM에서는 bank/row placement가 연산 가능성 자체를 결정한다[계획서 §8.5 Data Placement 행]. 상위 계층(Compiler)이 placement를 지정해 내려보내는지, Runtime이 placement를 결정하고 Compiler는 제약만 주는지에 따라 SubmitRequest의 layout 필드 의미가 달라진다. 이는 Compiler/Runtime 경계 확인 필요 항목과 직결된다.

쟁점 5 — Completion 통지 메커니즘. 계획서 §3은 polling/interrupt/callback을 나열할 뿐 LP6-PIM의 방식을 정하지 않았고, §8.1의 "Operation Completion Timing" 상태는 시간 기반 결정론적 완료 판정 가능성을 시사한다[추정]. RISR가 상태 통지 register라면 polling 모델, timing 계산만으로 충분하다면 tracker의 성격이 완전히 달라진다. spec 확인 전까지 Completion Tracker 설계는 두 갈래를 모두 열어 두어야 한다.

쟁점 6 — 근거 자료의 공백. 이번 수집에서 시스템별(API 수준) 서술은 전부 [배경지식]이다. 계획서가 지정한 외부 공식 자료 6종과 내부 자료 4종이 모두 미확보 상태이므로, 본 보고서의 시스템별 내용은 "수집 로드맵"으로만 유효하고 "분석 결과"로 인용해서는 안 된다.

후속 질문

다음 정보 수집/분석 회차에서 다룰 질문:

  1. Level Zero에서 command list 재사용이 허용되는 정확한 조건은 무엇이며, LP6-PIM의 동적 상태(active row, register)와 양립하는 "parameterized command template" 설계가 가능한가? (쟁점 3)
  2. CUDA Runtime↔Driver API 대응표에서 runtime이 자동화하는 것(primary context, module 관리)의 목록을 확정하면, LP6-PIM 고수준 API가 자동화할 항목은 무엇인가?
  3. PJRT plugin이 반드시 구현해야 하는 최소 함수 집합은 무엇이며, CHAMP Compiler-Runtime 경계를 PJRT 호환으로 만들 실익이 있는가?
  4. UPMEM의 비동기 launch와 rank 단위 병렬 전송은 실제로 어느 계층(라이브러리/드라이버)에서 병렬화되는가? LP6-PIM의 sub-channel 병렬성과 대응되는가?
  5. LP6-PIM에서 완료 판정이 시간 기반으로 가능하다면, Event/Fence 의미론을 어떻게 단순화할 수 있는가? RISR polling과 혼합 모델이 필요한가? (쟁점 5)
  6. Bank/Row/Layout placement를 Compiler와 Runtime 중 누가 결정하는가에 따라 SubmitRequest와 TaskResourceRequest의 필드가 어떻게 달라지는가? (쟁점 4)
  7. Multi-tenant/다중 프로세스 상황에서 ExecutionDomain 격리는 어떻게 보장하는가 — CUDA context 격리 모델을 어디까지 차용할 수 있는가? (계획서 §10 "이후 분석"과 연결)
  8. 오류·Context 복구(계획서 §10 항목 10): PIM mode 중 오류 발생 시 RESET_CONTEXT[계획서 §8.3]만으로 복구 가능한가, DRAM 상태(row) 복구가 추가로 필요한가?
