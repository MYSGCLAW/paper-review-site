PJRT Runtime 상위/하위 API와 필드 구성 정리

OpenXLA PJRT를 "런타임이 위(framework/compiler)로 올려주는 API"와 "아래(backend/plugin/driver)로 내려 요구하는 API"의 두 방향으로 나누어 정리한 설계 노트다. 각 API가 주고받는 객체, 옵션 구조체, 소유권과 비동기 완료 모델, 그리고 plugin 구현자가 채워야 하는 함수 테이블을 표와 호출 흐름으로 정리했다.

job.json이 제시한 10개 질문을 3개 축으로 묶었다.

  2. 필드 축 — PjRtClient/PjRtDevice/PjRtMemorySpace/PjRtBuffer/PjRtExecutable/PjRtLoadedExecutable/PjRtFuture/PjRtCompiler가 각각 어떤 식별자·옵션·상태 필드를 갖는가.
  3. 이식 축 — 새 backend runtime(예: LPDDR6-PIM 계열)을 설계할 때 PJRT에서 그대로 가져올 수 있는 API 형태는 무엇이고, 어디를 조심해야 하는가.

소스 트리 접근: 부분 성공

/home/sam/xla/xla/pjrt에 대해 파일 목록(Glob)은 열람 가능하지만 파일 본문(Read/Grep)은 권한 거부 상태다. 이번 개정 시점에도 재시도했으나 동일하게 거부되었다. WebFetch도 승인되지 않아 상류 GitHub 헤더로 대체하지도 못했다.

이것이 이 보고서의 성격을 결정한다.

  • 확정 가능한 것 — 어떤 파일/디렉터리/extension이 존재하는가, 코드가 어떤 축으로 모듈화되어 있는가. → [트리]
  • 확정 가능한 것 — 공식 문서와 설계 문서가 명시적으로 적은 클래스·함수·구조체·필드 이름. → [문서]
  • 확정 불가능한 것 — CompileOptions/ExecuteOptions/LoadOptions의 전체 필드 목록, PJRT_Api 함수 테이블의 전체 항목, 각 메서드의 정확한 현재 시그니처. 문서에 인용된 범위 밖은 모두 [추정]으로 격리했다.

즉 "API 분류·경계·흐름·정책"은 근거 위에 서 있고, "필드 표의 완전성"은 아직 서 있지 않다. 필드 표는 채우되, 각 행마다 근거 등급을 붙여 어디까지 믿을 수 있는지 명시했다.

이번에 다루지 않은 것

  • PJRT의 성능 특성(지연/대역폭 측정), 스케줄링 내부 구현.
  • distributed/(coordination service, preemption)와 multi-host topology의 상세. 파일 존재만 확인했다. [트리]
  • IFRT 계층. PJRT 위에 있으므로 이번 경계 정의에서 제외했다.
  • LPDDR6-PIM의 구체 설계. job.json 지시대로 "PJRT에서 배울 수 있는 API 설계 포인트"까지만 적었다.

핵심 요약

  1. PJRT의 경계는 "클래스 계층"이 아니라 "ABI 경계"에 그어져 있다. framework는 C++ PjRtClient를 보지만, plugin은 C PJRT_Api 함수 테이블을 채운다. 두 세계는 pjrt_c_api_wrapper_impl(C++ → C)과 c_api_client/pjrt_c_api_client(C → C++)라는 양방향 어댑터 한 쌍으로 이어진다. [문서] [트리]
  1. plugin이 반드시 구현해야 하는 함수는 놀랄 만큼 적다. wrapper 경로를 쓰면 GetPjrtApi와 PJRT_Client_Create 두 개가 실질적 필수이고, PJRT_TopologyDescription_Create·PJRT_Plugin_Initialize·PJRT_Plugin_Attributes는 공식 문서가 명시적으로 [선택사항]이라고 적었다. 나머지 C API는 wrapper가 C++ 구현으로 전달한다. [문서]
  1. "무엇이 선택 기능인가"는 extension chain으로 표현된다. 각 optional feature는 PJRT_*_Extension 구조체로 만들어져 PJRT_Api.extension_start에서 시작하는 연결 리스트에 매달린다. framework는 체인을 순회해 해당 extension의 존재 여부로 기능 지원을 판정한다. 설계 문서는 "Args만 추가하는 extension이라도 빈 PJRT_Api extension 구조체를 항상 두라"고 못박는다. [문서]
  1. 비동기는 buffer가 아니라 future로 관측된다. 비동기 plugin은 계산을 큐에 넣고 아직 읽을 수 없는 buffer를 즉시 반환해도 되며, 값이 필요해지는 시점에 GetReadyFuture()가 반환한 future에서 블록된다. 오류도 이 future에서 관측된다. [문서]
  1. buffer는 memory space를 알지만 device를 반드시 알지는 않는다. 역참조 방향이 buffer → memory space → device → client로 정해져 있고, device는 자신에게 붙은 buffer를 알 필요가 없다. 새 런타임을 설계할 때 그대로 베낄 만한 소유권 그래프다. [문서]
  1. PjRtExecutable과 PjRtLoadedExecutable의 분리는 "직렬화 가능한 아티팩트" 대 "실행 가능한 인스턴스"의 분리다. 전자는 compile 산출물과 실행 옵션을 들고 serialize/deserialize를 알고, 후자는 그 subclass로서 입력 인자를 받아 즉시 실행 가능한 메모리 내 실행체다. AOT compile → 저장 → 나중에 load 하는 경로가 여기서 나온다. [문서]
  1. PjRtCompiler는 plugin이 구현할 필요가 없다. 공식 문서가 "XLA 백엔드에 유용한 구현 세부정보를 제공하지만 플러그인이 구현할 필요는 없다"고 명시한다. 컴파일 책임은 PjRtClient::Compile이 지면 되고, 입력 모듈을 받아 PjRtLoadedExecutable을 돌려주는 것이 계약의 전부다. [문서]
  1. 버전 호환은 "함수 테이블 + minor 버전 게이트"로 처리된다. framework가 pjrt_api->pjrt_api_version.minor_version >= N을 검사해 새 메서드/새 필드를 쓸지 결정하고, 아니면 unimplemented나 기본값으로 후퇴한다. 필드 추가·이름 변경·메서드 폐기는 minor, 삭제·타입 변경·구조체 필드 재배열은 major다. [문서]
  1. ABI 안정성의 부담은 struct에 몰려 있다. 모든 args 구조체가 struct_size를 첫 필드로 갖고, 과거 void* priv 자리가 extension_start로 재활용되었다. 크기 기반 버저닝 + 확장 체인이 PJRT의 ABI 전략의 핵심이다. [문서]
  1. job.json이 요구한 항목 중 네 가지는 근거를 확보하지 못했다. LoadOptions의 존재 여부, buffer donation의 API 형태, executable cost analysis, phase compile의 입출력 필드다. 창작으로 메우지 않고 쟁점과 불확실성과 후속 질문으로 넘겼다.

  2. 기존 메서드의 인자를 확장할 때는 args 구조체의 extension_start에 Args extension을 매단다.
  3. framework는 PJRT_Api의 extension chain을 순회해 기능 지원 여부를 판정한다.
  4. 따라서 Args만 추가하는 extension이라도 빈 PJRT_Api extension 구조체를 반드시 등록해야 한다. (안 그러면 framework가 탐지할 방법이 없다)

C++ API와 C API 대응표

왜 args struct + extension chain인가

C API의 목적은 plugin .so와 framework를 따로 빌드·배포하면서도 ABI를 깨지 않는 것이다. 그래서 세 가지 장치를 쓴다. [문서] D6, D7

장치목적깨지는 조건
함수 테이블 (PJRT_Api)심볼을 하나(GetPjrtApi)로 줄이고, 나머지는 구조체 안 함수 포인터로필드 재배열/삭제 → major
args 구조체 + struct_size인자를 늘려도 함수 시그니처가 그대로필드 재배열/타입 변경 → major. 필드 추가/이름 변경 → minor
extension_start 체인선택·실험 기능을 본체 밖에 격리experimental extension은 보장 없음

버전 게이트의 실제 코드 패턴 (D6 원문):

// Framework code — PJRT_Memory_Kind가 MINOR 14에 추가되었다고 가정
if (pjrt_api->pjrt_api_version.minor_version >= 14) {
    return pjrt_api->PJRT_Memory_Kind(...);
} else {
    return unimplemented or default;
}

그리고 이 shim이 사는 곳이 PjRtCApiClient다:

// D6 원문 — layout이 버전 N에 추가되었다고 가정
PjRtCApiClient::BufferFromHostBuffer(...) {
    BufferFromHostBuffer_Args args;
    if (minor_version_ > N) {
        args.device_layout = ...;
    }
    c_api_->PJRT_Client_BufferFromHostBuffer(args);
}

즉 "버전 분기"라는 지저분한 일을 PjRtCApiClient 한 곳에 몰아넣고, 위쪽 framework는 C++ 인터페이스만 보게 한다. 다만 D6은 한계도 적었다 — "plugin 버전에 따라 다른 메서드를 호출해야 하는 경우나, 어차피 전부 unimplemented가 뻔한 일련의 호출을 아예 건너뛰고 싶은 경우"는 shim으로 못 감추고 JAX가 api_version을 직접 봐야 한다. [문서] D6

새 Runtime / Backend 설계 체크리스트

PJRT를 참고해 새 backend runtime을 만들 때 먼저 결정해야 하는 것들이다. LPDDR6-PIM 같은 특정 하드웨어의 설계를 확정하는 것이 아니라, PJRT가 어떤 선택을 했고 왜 했는지에서 뽑아낸 체크 항목이다.

A. 최소 API 세트 (이것만 있으면 basic workload가 돈다)

  • [ ] 진입점 1개 — .so에서 내보내는 심볼 하나 (GetPjrtApi 대응물). 나머지는 전부 함수 테이블 안으로. [문서]
  • [ ] client 생성 — config를 key-value로 받는 생성자 (PJRT_NamedValue 대응물).
  • [ ] device 열거 + addressable 여부 + id/kind/위치.
  • [ ] memory space 최소 1개 (default) + device→default 연결.
  • [ ] buffer 생성 (host→device) + host buffer semantics (share/copy/mutate).
  • [ ] buffer 반출 (device→host) — future 반환.
  • [ ] compile — 모듈 + 옵션 → 실행 가능 객체.
  • [ ] execute — 인자 handle 배열 → 결과 handle 배열.
  • [ ] ready future — 완료·오류 관측 지점.
  • [ ] delete / is_deleted.

B. 선택 API 세트 (없어도 basic workload는 돈다 — PJRT가 명시적으로 선택으로 분류한 것들)

C. 객체 모델 체크리스트

  • [ ] client이 device와 memory space를 소유하는가?
  • [ ] buffer가 memory space를 가리키고, device는 전이적으로만 아는가? (buffer→device 직결을 피하면 unpinned 메모리와 PIM처럼 "메모리에 붙은 연산기"를 자연스럽게 표현할 수 있다)
  • [ ] executable을 "직렬화 가능한 아티팩트"와 "실행 가능한 인스턴스"로 분리했는가?
  • [ ] compiler를 필수 인터페이스에서 뺐는가? (PJRT는 뺐다. compile 책임은 client가 진다)
  • [ ] 역포인터를 어디까지 둘 것인가? (PJRT는 memory space→device, device→client를 둔다)

D. 필드 체크리스트

  • [ ] 모든 ABI 구조체에 struct_size 첫 필드.
  • [ ] 확장 여지를 위한 extension_start 포인터 (PJRT는 과거 void* priv를 이 용도로 재활용했다).
  • [ ] device: id, kind, 로컬/전역 위치, process index, addressable 플래그.
  • [ ] buffer: 정적 shape와 논리 shape를 분리할 것인가? (PJRT는 분리했고, PyTorch/XLA가 재컴파일 판정에 정적 shape를 필요로 했기 때문이다 — D9)
  • [ ] 옵션 구조체에 넣을 필드와 extension으로 뺄 필드의 기준.

E. 비동기·오류 체크리스트

  • [ ] "실행 요청 반환 ≠ 실행 완료" 를 API 계약으로 못박았는가?
  • [ ] not-ready handle을 반환하고 완료 관측은 future 한 곳으로 모았는가?
  • [ ] 오류를 future에서 관측하는가, 반환값에서 관측하는가? (PJRT는 생성 계열은 StatusOr, 완료 계열은 future로 이원화 — [문서] D1 + [추정])
  • [ ] C 경계의 오류 표현: 성공 = nullptr, 실패 = 오류 객체 포인터. [문서]
  • [ ] 미구현 기능은 UNIMPLEMENTED를 반환하는가, 아니면 아예 함수 포인터를 nullptr로 두는가? (PJRT는 experimental의 경우 nullptr — [문서] D6)

F. backend 구현 체크리스트

  • [ ] 버전 협상: major/minor를 분리하고, 필드 추가·이름 변경·메서드 폐기는 minor, 삭제·타입 변경·재배열은 major로 규정.
  • [ ] 호환 창(compatibility window)을 몇 주로 할 것인가? (PJRT는 6주 시작 → 12주 목표)
  • [ ] framework가 plugin보다 오래된 경우의 동작을 plugin이 정의한다 (abort할 것인가, 구버전처럼 행동할 것인가). [문서] D6
  • [ ] 적합성 테스트 하네스 제공 (RegisterPjRtCApiTestFactory 대응물).
  • [ ] 미구현 optional 기능이 있는 plugin에 대해 테스트를 건너뛰는 메커니즘. (D7이 "불완전한 plugin에게도 테스트 스위트가 쓸모 있게 해달라"는 plugin 저자들의 오랜 요청이라고 적었다)

G. 확인 필요 항목 (이 체크리스트를 확정하기 전에 소스로 검증해야 하는 것)

  • CompileOptions / ExecuteOptions의 실제 필드 전체 목록.
  • LoadOptions가 실제로 존재하는지, 존재한다면 어떤 필드를 갖는지.
  • donation의 API 형태와 소유권 이전 규약.
  • PjRtFuture의 오류 전파 규약 (Status를 품는가, 별도 채널인가).
  • memory space kind의 값 집합과 host offload 표현 방식.
  • PJRT_Api 함수 테이블의 실제 항목 수와 필수/선택 구분.
  • executable의 cost analysis / memory stats가 노출하는 필드.

쟁점과 불확실성

1. 소스 본문 미열람 — 이 보고서의 가장 큰 구멍

/home/sam/xla/xla/pjrt에 대한 Read/Grep 권한이 없어 헤더 본문을 한 줄도 읽지 못했다. 이번 개정에서 다시 시도했으나 여전히 거부되었다. Glob으로 파일 이름만 확인 가능했고, WebFetch도 승인되지 않아 상류 GitHub 헤더로 대체할 수도 없었다.

영향 범위:

  • API 분류·경계·정책·흐름은 공식 문서와 설계 문서로 충분히 근거를 확보했다. 신뢰 가능.
  • 필드 표의 완전성은 확보하지 못했다. 문서가 인용한 필드만 알고, 나머지는 모른다.
  • job.json이 명시적으로 요구한 항목 중 LoadOptions, donation, cost analysis, phase compile의 입출력 필드는 근거를 전혀 찾지 못해 [추정]으로도 채우지 않고 비워 두었다.

이 구멍을 메우는 방법은 후속 질문 Q1에 적었다. 권한 한 번이면 해결되는 문제다.

2. 설계 문서와 현재 헤더 사이의 naming drift

D7(2023-12)은 확장 기반 구조체를 PJRT_Structure_Base / PJRT_Structure_Type이라 부르면서, 본문에 "I also wanna rename this PJRT_Extension_Base" 라는 저자 메모를 남겼다. 그리고 현재 트리의 파일 이름은 전부 pjrt_c_api_*_extension.h 형태다. [트리]

→ 이름 변경이 실제로 반영되었을 가능성이 높다. 하지만 확인하지 못했다. 이 보고서의 C API 구조체 이름들(PJRT_Structure_Base, PJRT_Structure_Type)은 설계 문서 시점의 이름이며, 현재 헤더에서는 PJRT_Extension_Base / PJRT_Extension_Type일 수 있다. 코드를 쓸 때 그대로 복사하면 안 된다.

4. pjrt_c_api_client.h가 두 곳에 있다

xla/pjrt/pjrt_c_api_client.h와 xla/pjrt/c_api_client/pjrt_c_api_client.h가 공존한다. [트리] 리팩터링 중 남긴 forwarding 헤더로 보이지만, 둘 중 어느 쪽이 정본인지 확인하지 못했다. 새 코드가 어느 쪽을 include해야 하는지는 열어봐야 안다.

5. "필수 vs 선택"의 경계가 기계적으로 검증되지 않는다

D7이 인정한 문제다. optional feature를 extension으로 표현하기로 했지만, experimental 여부는 여전히 주석으로만 표시된다. 그리고 "지금은 사실상 선택인데 본체 API에 들어가 있는 기능들"이 남아 있고, D7은 "일부는 본체에 그냥 남겨두게 될 수도 있다"고 적었다.

→ 새 backend를 붙일 때 "이 함수를 구현 안 하면 뭐가 깨지는지"를 헤더만 보고는 알 수 없다. 실제로는 framework 테스트를 돌려보며 알아내야 한다. 새 런타임을 설계한다면 이 부분은 PJRT를 따라 하지 말고 개선해야 할 지점이다.

6. wrapper 왕복 오버헤드 미측정

옵션 B(C++ 구현 + wrapper)는 "C++ → C → C++" 왕복을 만든다. buffer 하나당, 실행 하나당 이 경계를 몇 번 넘는지, 그 비용이 얼마인지 측정된 자료가 참고 문서에 없다. D1의 호출 로그를 보면 print(y) 한 번에 on_device_shape()가 7회, memory_space()가 6회 호출된다 — 경계를 넘는 호출 수가 결코 적지 않다. [문서] D1

→ PIM처럼 메모리 접근당 비용이 중요한 backend라면 이 경계 비용을 먼저 재봐야 한다.

7. LPDDR6-PIM 관련: 지금 단계에서 결론 내리면 안 되는 것

job.json이 "구체 설계 확정처럼 쓰지 말라"고 못박았고, 실제로 그럴 근거도 없다. 다만 PJRT 구조에서 PIM류 backend에 특히 유리해 보이는 지점과 불리해 보이는 지점을 관찰로만 적어 둔다.

  • 유리: PjRtMemorySpace가 이미 "여러 device가 접근 가능한 unpinned 영역" 개념을 갖고 있고, buffer가 device에 직결되지 않는다. "메모리에 붙은 연산기"를 표현할 자리가 이미 있다. [문서] D1
  • 유리: AcquireExternalReference가 opaque device memory pointer를 노출하는 공식 탈출구를 제공한다. [문서] D8
  • 불리 (미확인): memory space가 "위치"만 표현하는지, "연산 능력"까지 표현할 수 있는지 불명. PIM은 memory space가 곧 연산 유닛이므로 이 모델이 맞는지 검증 필요. memory_descriptions extension이 답을 갖고 있을 수 있다. [추정]
  • 불리 (미확인): on_device_shape/layout이 PIM의 뱅크 인터리빙 같은 물리 배치를 표현할 수 있는지 불명. custom layout은 선택 기능이므로 표현력이 제한적일 수 있다. [추정]

Q3. buffer donation의 API 형태와 소유권 이전 규약은? → 참고 문서 11개 어디에도 없다. pjrt_client.h의 ExecuteOptions 주변과 PjRtBuffer의 donation 관련 메서드를 확인해야 한다. 새 backend 설계에서 가장 실수하기 쉬운 부분이므로 우선순위가 높다. 이것이 채워지면 buffer 수명 상태도의 점선 전이를 실선으로 바꿀 수 있다.

Q4. D7의 Proposal A(extension 기반 optional feature)는 실제로 어디까지 이행되었는가? → pjrt_c_api.h에서 PJRT_Extension_Base/PJRT_Extension_Type의 존재를 확인하고, extensions/ 디렉터리의 각 헤더가 "빈 PJRT_Api extension 구조체"를 갖는지 본다. 이행 정도가 곧 "새 backend가 무엇을 안 만들어도 되는가"의 답이다.

Q5. AcquireExternalReference는 Option 1(refcount 3함수)과 Option 2(PJRT_ExternalReference 구조체) 중 무엇으로 구현되었나? → pjrt_c_api.h에서 PJRT_Buffer_IncreaseReference 또는 PJRT_ExternalReference를 grep하면 즉답. zero-copy 경로를 쓸 계획이면 반드시 확인.

Q6. phase compile은 정확히 무엇을 나누고, 어떤 입출력 필드를 갖는가? → c/pjrt_c_api_phase_compile_extension.h, c_api_client/pjrt_c_api_phase_compiler.h, pjrt_phase_compile_sample_plugin.h를 읽는다. 컴파일을 단계별로 쪼개는 기능은 새 backend가 XLA 파이프라인의 일부만 재사용하고 싶을 때 핵심이 될 수 있으므로, 이번에 파일 존재만 확인한 것이 아쉽다.

Q7. PjRtFuture는 오류를 어떻게 실어 나르는가? → pjrt_future.h를 읽는다. Future<Status>인지 Future<>가 내부에 status를 품는지에 따라 backend의 오류 전파 구현이 달라진다.

Q8. memory space는 "위치"만 표현하는가, 아니면 능력(capability)도 표현할 수 있는가? → pjrt_client.h의 PjRtMemorySpace와 c/pjrt_c_api_memory_descriptions_extension.h를 읽는다. PIM류 backend의 모델링 가능성을 판정하는 질문이다.

Q9. wrapper 경계를 넘는 호출 비용은 얼마인가? → 측정 과제. D1의 로그처럼 단순 연산 하나에도 getter가 수 회 호출되므로, pjrt_c_api_cpu plugin으로 마이크로벤치를 돌려 경계 비용을 재는 것이 현실적이다.

Q10. xla/pjrt/pjrt_c_api_client.h와 c_api_client/pjrt_c_api_client.h 중 정본은? → 두 파일을 열어 forwarding 여부를 확인. 사소하지만 새 코드의 include 경로를 정하려면 필요하다.
