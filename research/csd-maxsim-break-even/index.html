CSD MaxSim Offload의 GPU 대비 MAC/PE/FLOPS Break-even 분석

한 줄 요약

Candidate embedding이 GPU-resident가 아닌 한 GPU MaxSim은 PCIe-bound이므로, CSD는 GPU의 FLOPS가 아니라 PCIe 전달률만 이기면 되고, 그 조건은 N_mac ≥ Q · BW_pcie / (b_e · f_csd · u) — fp16, Q=32, PCIe Gen4 x16 기준 약 630개 MAC(≈1 TFLOP/s, A100 peak의 0.3%) 수준이다. 단, 이는 BW_int ≥ BW_pcie(내부 read bandwidth 조건)와 "압축 전송 baseline과의 비교"라는 두 개의 더 어려운 전제를 동반한다.

수집 범위

  • 포함: MaxSim의 MAC/FLOP 연산 모델, GPU 실행 경로(T_gpu)와 CSD 실행 경로(T_csd)의 latency equation, break-even inequality와 required MAC/PE/OPS 유도, sanity-check 수치 예시, sensitivity sweep 설계.
  • 제외: 문서 로컬리티/캐시 히트율 자체 분석(job scope에서 명시적으로 제외), query encoder(BERT forward)의 비용 — 양 경로에 공통이므로 소거, candidate generation(ANN/centroid 단계)의 자체 비용 — 단 배치(placement) 쟁점으로만 언급.
  • 자료 유형: manifest의 참고자료 8건(ColBERT/ColBERTv2/PLAID/MS MARCO/BEIR/CUDA Guide/PCIe/SNIA)은 전부 URL 메타데이터만 확인 — 본문 텍스트가 references/text/에 없고, 이번 세션에서 WebFetch/WebSearch 권한이 거부되어 열람하지 못했다. 따라서 본 문서의 모든 수치 파라미터(Q=32, E=128, PCIe 대역폭 표, GPU 스펙, SmartSSD 스펙 등)는 배경지식이며, 불확실한 값은 개별적으로 "확인 필요"로 표시한다. 수식 유도 자체는 first-principles이므로 참조 의존성이 없다.

핵심 요약

  1. MaxSim은 batched small GEMM + row-wise max: FLOP_total ≈ 2·B·N·Q·D·E, max-reduction은 GEMM 대비 1/(2E) ≈ 0.4% (E=128)로 무시 가능하다.
  2. GPU 경로의 arithmetic intensity는 AI = 2Q/b_e (query는 on-chip 상주, fused kernel 가정). fp16·Q=32이면 32 FLOP/B로 A100 ridge(≈153 FLOP/B)에 한참 못 미쳐 HBM-bound이고, 데이터가 PCIe를 건너는 순간 BW_pcie ≪ BW_hbm(30~80배)이므로 항상 PCIe-bound가 된다.
  3. 따라서 break-even의 본질은 "CSD compute vs GPU compute"가 아니라 "CSD compute vs PCIe 전달률": required OPS = (Q/b_e)·BW_pcie [MAC/s]. 여기서 B, N, D, E가 전부 소거된다 — break-even MAC 수는 워크로드 크기와 무관하고 Q, precision, 링크 속도, clock, utilization에만 의존한다.
  4. 필요 MAC 수는 작다: N_mac* = Q·BW_pcie/(b_e·f·u) → fp16, f=1 GHz, u=0.8일 때 Gen4 x4에서 158개, Gen4 x16에서 630개, Gen5 x16에서 1,260개. SmartSSD급 FPGA(KU15P, DSP slice 약 2천 개 — 배경지식, 확인 필요)로도 도달 가능한 규모다.
  5. 진짜 병목은 두 번째 필요조건 BW_int ≥ BW_pcie: 단일 drive의 NAND 내부 대역폭(~7–16 GB/s)은 GPU x16 링크 유효 대역폭(25–54 GB/s)보다 낮다. 단일 CSD는 warm host DRAM baseline에 지고, cold data / multi-drive scale-out / HBM·host DRAM capacity 초과 corpus에서만 이긴다.
  6. Result traffic은 소거 가능: score만 반환 시 D·E·b_e/b_s = 8192배, top-k 반환 시 10⁵배까지 egress가 줄어 T_res는 µs 단위가 된다. CSD의 PCIe 비용은 사실상 0으로 모델링해도 된다.
  7. 최강 GPU baseline은 raw fp16 전송이 아니라 압축 전송(ColBERTv2 residual compression, ~36 B/vector — 배경지식, 확인 필요): 전송량이 ~7배 줄면 required N_mac*도 ~7배(Gen4 x16 기준 ~4.5K MAC) 커진다. CSD 제안의 공정한 비교 대상은 PLAID류 압축 경로다.
  8. GPU-resident(HBM 상주) baseline을 이기는 것은 불가능에 가깝다(required OPS ≈ 65 TFLOP/s + BW_int ≥ 2 TB/s). CSD 논거는 latency가 아니라 capacity(fp16 full index ~150+ GB > HBM 80 GB), cost, host 자원 절감으로 옮겨가야 한다.

프로토콜 / 동작 흐름

요청·핸들·상태·메모리 객체·명령 제출·완료 신호의 순서를 두 경로로 재구성한다(NVMe/CUDA 동작은 배경지식, SNIA CS command 세부는 확인 필요).

sequenceDiagram participant H as Host CPU participant S as SSD / CSD participant G as GPU Note over H,G: [GPU path] 이동 객체 = doc embeddings (BNDEb_e, MB~GB급) H->>H: candidate doc-id list (N ids, retriever 출력) H->>S: NVMe read SQ entries (LBA per doc) S-->>H: DMA → host DRAM staging buffer, CQE 완료 H->>G: cudaMemcpyAsync H2D (stream) 또는 GPUDirect로 S→G 직접 H->>G: MaxSim kernel launch (grid=N docs) G->>G: HBM read → tensor core GEMM → fused max/sum (smem/reg) G-->>H: score buffer D2H (4N B), CUDA event sync Note over H,S: [CSD path] 이동 객체 = query embedding(8 KB) + scores/top-k(≤16 KB) H->>S: CS command: query emb + candidate id/LBA list 제출 S->>S: FW가 LBA resolve → NAND channel 병렬 read → channel buffer S->>S: PE array가 stream 소비: dot(E) → Q개 partial-max reg → doc 종료 시 sum S->>S: on-drive top-k heap (N scores → k) S-->>H: result buffer DMA (k·8 B or N·4 B) + CQE H->>H: n_csd개 drive의 partial top-k merge

핵심 차이: GPU path는 큰 메모리 객체(embedding tensor)가 두 번 hop(SSD→host DRAM→HBM)하는 구조이고, CSD path는 큰 객체가 device 내부에 머물고 작은 상태(Q·E query matrix 8 KB, doc당 Q개 partial-max register, k-heap)만 유지된다. CSD에서는 Q×D score matrix가 아예 materialize되지 않는다(streaming max).

세부 정리

1. MaxSim Operation Model

변수 정의 (기본값은 ColBERT 논문 기준 배경지식):

기호의미대표값
Qquery token 수32 (ColBERT 고정 padding — 배경지식)
Dcandidate doc당 token 수60–180 (MS MARCO passage, max 180 — 배경지식; 평균값 확인 필요)
Eembedding dimension128
Nquery당 re-rank candidate 수10³–10⁴ (PLAID ndocs — 확인 필요)
Bquery batch size1–수십
b_eelement당 byte (precision)fp32=4, fp16=2, int8=1, 2-bit residual≈0.28
b_sscore당 byte4 (fp32)

연산 정의: S(q,d) = Σ_{i=1..Q} max_{j=1..D} <q_i, d_j> — candidate당 (Q×E)·(E×D) small GEMM 후 row-wise max, Q개 max의 합.

MAC / FLOP:

MAC_total  = B · N · Q · D · E                    (dot-product 곱셈-누산)
FLOP_gemm  = 2 · B · N · Q · D · E
FLOP_red   = B · N · (Q·(D−1) compare + (Q−1) add) ≈ B·N·Q·D
FLOP_total ≈ 2·B·N·Q·D·E · (1 + 1/(2E)) ≈ 2·B·N·Q·D·E     [E=128 → red 비중 0.4%]

Intermediate matrix: candidate당 Q×D score matrix. Materialize 시 B·N·Q·D·b_s bytes (예: N=4096, Q=32, D=128 → 67 MB, 입력의 절반 규모) — fused kernel이면 register/shared memory에서 소거된다. Max-reduction은 FLOP으로는 무시 가능하지만, unfused 구현에서는 HBM traffic을 ~2배로 만들 수 있는 memory-side 항이다.

데이터량과 arithmetic intensity (query는 on-chip 캐시, fused 가정):

Bytes_D = B·N·D·E·b_e        (doc embeddings — 지배항)
Bytes_Q = B·Q·E·b_e          (query, fp16 8 KB/query — 무시)
AI      = FLOP/Bytes_D = 2Q/b_e   [FLOP/B]   ← N, D, E에 무관, Q와 precision만의 함수

fp16·Q=32 → AI=32, int8 → 64, fp32 → 16, 2-bit → ~114.

2. GPU Latency Model

경로: NAND → (SSD 링크) → host DRAM → (GPU 링크) → HBM → SM → score 반환.

Bytes_in  = B·N·D·E·b_e + B·Q·E·b_e ≈ Bytes_D
T_pcie    = Bytes_in / (ε_pcie · BW_pcie),   BW_pcie = min(n_ssd·BW_ssd_link, BW_gpu_link, BW_host)
T_comp    = max( FLOP_total/(F_gpu·η_gpu),  Bytes_HBM/BW_hbm )      [roofline]
T_gpu     = T_fix,gpu + T_pcie + T_comp                (non-overlapped)
T_gpu     ≈ T_fix,gpu + max(T_pcie, T_comp)            (chunked pipeline)
  • ε_pcie ≈ 0.8–0.9 (protocol overhead, 대형 DMA 기준 — 배경지식), T_fix,gpu ≈ 10–30 µs (kernel launch + driver — CUDA Guide 계열 배경지식).
  • HBM constraint 두 가지: (bandwidth) Bytes_HBM ≥ Bytes_D — fused면 1회 read, unfused면 score matrix 왕복이 추가돼 ~2배. (capacity) resident baseline이 성립하려면 index size ≤ C_hbm — fp16 MS MARCO full index ~150+ GB(배경지식, 확인 필요)는 80 GB HBM에 안 들어가고, ColBERTv2 압축 index ~20–36 GB는 들어간다.
  • regime 판별: compute-bound 조건은 AI > F_gpu·η/BW_hbm (A100 fp16 ridge ≈ 153 FLOP/B — 배경지식). Q=32·fp16(AI=32)는 미달 → resident면 HBM-bound. 비-resident면 BW_pcie(≤63 GB/s) ≪ BW_hbm(2–3.35 TB/s)이므로 무조건 PCIe-bound: T_gpu ≈ Bytes_D/(ε·BW_pcie). 이때 GPU의 F_gpu는 1차 근사에서 결과에 등장하지 않는다.

3. CSD Latency Model

경로: NAND → internal channel → on-drive PE array → local max/top-k → 결과만 PCIe.

Bytes_int  = B·N·D·E·b_e                       (internal read bytes, GPU 입력과 동일)
T_read     = Bytes_int / BW_int                 (NAND channel 병렬 read)
OPS_csd    = N_mac · f_csd · u                  [MAC/s]  (= 2·N_mac·f_csd·u FLOP/s)
T_mac      = MAC_total / OPS_csd = B·N·Q·D·E / (N_mac·f_csd·u)
Bytes_res  = B·N·b_s   (전체 score)  또는  B·k·(b_id+b_s) ≈ 8k B  (top-k)
T_csd      = T_fix,csd + max(T_read, T_mac) + Bytes_res/BW_link
  • N_mac: MAC unit(PE) 수, f_csd: clock, u: utilization(스트리밍 정렬/뱅크 충돌/tail 포함 유효 계수). local max-reduction은 MAC 뒤 comparator를 파이프라인에 붙이면 추가 cycle이 없고 u에 흡수된다. top-k heap은 O(N log k) scalar 연산으로 무시 가능.
  • 필요한 on-chip 상태는 query matrix Q·E·b_e = 8 KB + doc당 Q개 partial-max register뿐 — score matrix를 만들지 않는다.
  • Result traffic 축소율: 전체 score 반환 시 R = D·E·b_e/b_s = 8192배 (D=E=128, fp16), top-k(k=100, N=4096) 반환 시 134 MB → 800 B ≈ 1.7×10⁵배. T_res는 Gen4 x4에서도 µs 단위 → 모델에서 소거 가능. 이것이 job 질문 "score/top-k 반환 시 traffic 모델 변화"의 답: CSD egress 항이 상수(≈0)로 바뀌고, break-even은 ingress(BW_int)와 compute(OPS_csd)만의 문제가 된다.
  • T_fix,csd: NVMe/CS command 왕복 + FW dispatch, 수십~수백 µs 추정(확인 필요 — SNIA CS API 문서와 실측 필요).

4. Break-even MAC/PE 조건

Break-even inequality (양쪽 모두 pipelined, 비-resident GPU baseline):

T_csd ≤ T_gpu
⇔ max( Bytes_D/BW_int ,  MAC_total/OPS_csd ) ≤ Bytes_D/(ε·BW_pcie)      [+ 고정항 무시]
⇔ (i)  BW_int ≥ ε·BW_pcie                        …… 대역폭 조건
   (ii) OPS_csd ≥ MAC_total/Bytes_D · ε·BW_pcie = (Q/b_e)·ε·BW_pcie      …… compute 조건

(ii)에서 MAC_total/Bytes_D = (B·N·Q·D·E)/(B·N·D·E·b_e) = Q/b_e [MAC/B] — B, N, D, E가 전부 소거된다. 전송되는 1 byte당 수행해야 할 MAC이 Q/b_e로 고정되기 때문이다(각 doc token vector가 Q개 dot product에 참여).

Required OPS / MAC units / FLOPS:

required OPS      = (Q/b_e) · BW_pcie                  [MAC/s]
required N_mac*   = Q · BW_pcie / (b_e · f_csd · u)    [MAC units]
required F_csd    = 2Q · BW_pcie / b_e                 [FLOP/s]
내부 BW 활용 설계점: N_mac** = Q · BW_int / (b_e · f_csd · u)   (이 이상은 read-bound라 낭비)

수치표 (Q=32, fp16 b_e=2 → 16 MAC/B; f_csd=1 GHz, u=0.8; PCIe는 nominal — 배경지식):

GPU 경로 병목 링크BW_pcierequired OPSrequired N_mac*= FLOP/s (A100 312 TFLOPS 대비)
Gen3 x43.94 GB/s63 GMAC/s790.13 T (0.04%)
Gen4 x47.88 GB/s126 GMAC/s1580.25 T (0.08%)
Gen4 x1631.5 GB/s504 GMAC/s6301.0 T (0.32%)
Gen5 x1663.0 GB/s1.01 TMAC/s1,2602.0 T (0.65%)
Gen6 x16121 GB/s1.94 TMAC/s2,4203.9 T (1.2%)

PCIe bottleneck effect: 위 표가 곧 그 정량화다 — GPU가 312 TFLOP/s를 가져도 비-resident 데이터에는 (Q/b_e)·BW_pcie ≈ 0.25–2 TFLOP/s만큼만 쓸 수 있으므로, CSD는 GPU peak의 0.1–1%대 compute로 대등해진다. 32×32 systolic array(1,024 MAC) @1 GHz ≈ 1 TMAC/s면 Gen5 x16급 전달률까지 커버한다. SmartSSD급 FPGA(KU15P DSP ≈ 2K개, 0.3–0.5 GHz — 배경지식, fp16 MAC은 DSP 2개 소요 가능성 있어 확인 필요)로도 Gen4 x16 조건은 충족 가능.

Sanity-check example (B=1, Q=32, E=128, D=128, N=4096, fp16):

Bytes_D = 4096·128·128·2 = 134.2 MB          MAC_total = 2.147 GMAC (FLOP 4.29 G)
[GPU-A] warm host DRAM, Gen4 x16 (eff 25 GB/s): T_pcie = 5.37 ms
        T_comp = max(134.2MB/2.0TB/s, 4.29G/(312T·0.5)) = max(67, 27.5) µs = 67 µs
        → T_gpu ≈ 5.4 ms   (compute는 transfer의 1/80 — PCIe-bound 확인)
[GPU-B] cold, 단일 SSD Gen4 x4 (eff 6.5 GB/s): T_gpu ≈ 20.6 ms
[GPU-C] HBM-resident: T_gpu ≈ 67 µs + fix ≈ 0.1 ms
[CSD]   BW_int=12 GB/s, N_mac=256(16×16), f=1 GHz, u=0.8 → OPS=204.8 GMAC/s
        T_read = 11.2 ms, T_mac = 10.5 ms, T_res = 16.4 KB → ~2 µs
        → T_csd ≈ 11.3 ms
판정: vs GPU-B 1.8배 승리(256 MAC만으로), vs GPU-A 0.48배 패배(BW_int 12 < 25 — 조건(i) 위반),
      vs GPU-C 100배 이상 패배.
4-drive CSD array(shard 1024 docs/drive): T ≈ 2.8 ms → GPU-A 대비 1.9배 승리, 8-drive → 3.8배.

교훈: compute 조건(ii)은 수백 MAC으로 싸게 충족되고, 승패는 조건(i) BW_int vs BW_pcie와 baseline 선택이 가른다. 승리 시 speedup은 S ≈ n_csd·BW_int/(ε·BW_pcie)로 대역폭 비에 수렴한다.

5. Sensitivity와 가능성 판단

Sweep 설계 (offload 가능성 판단에 필요한 축):

Sweep 축범위break-even에 미치는 효과
PCIe Gen sweepGen3 x4 → Gen6 x16 (3.94→121 GB/s)N_mac* ∝ BW_pcie (선형). Gen이 오를수록 GPU 경로가 빨라져 CSD 마진 축소 — Gen6에서 CSD가 이기려면 n_csd·BW_int > 100 GB/s급 필요
N sweep10²–10⁴절대 latency ∝ N. break-even 조건은 불변. 하한: N < T_fix·ε·BW_pcie/(D·E·b_e) ≈ 76 docs(@25 GB/s, T_fix=100 µs)이면 고정 overhead 지배 → offload 무의미
Q sweep16–128N_mac* ∝ Q. 긴 query/expansion은 CSD에 불리. GPU는 Q>~153(fp16, A100)이어야 compute-bound 진입 — 현실적 Q에서는 도달 안 함
D, E sweepD 60–180, E 64–256절대 latency만 변화(∝ D·E), break-even 불변. 단 D·E·b_e가 NAND page(16 KB)보다 작아지면 read amplification으로 유효 BW_int 하락(압축 시 doc당 4.6 KB — 쟁점 4 참조)
precision sweepfp32→fp16→int8→2-bitN_mac* ∝ 1/b_e: fp32 315개 → fp16 630개 → int8 1,260개 → 2-bit ~4,500개(Gen4 x16). 압축은 PCIe 병목 자체를 줄이므로 CSD의 존재 이유를 잠식하는 가장 위험한 축
CSD PE sweep64–16K MACN_mac < N_mac**=Q·BW_int/(b_e·f·u)이면 compute-bound(PE 부족), 초과면 read-bound(PE 낭비). 설계점은 등호
n_csd sweep1–16 drivesGPU 경로는 GPU x16 링크에서 포화, CSD는 선형 scale → 대수적으로 n_csd > ε·BW_pcie/BW_int가 승리 조건

Regime map:

GPU 경로:  PCIe-bound   ⇔ 데이터가 PCIe를 건넘 (비-resident; BW_pcie ≪ BW_hbm이므로 항상)
           HBM-bound    ⇔ resident & AI = 2Q/b_e < F_gpu·η/BW_hbm   (Q=32, fp16에서 성립)
           compute-bound ⇔ resident & Q > ridge·b_e/2 ≈ 153        (비현실적)
CSD 경로:  compute-bound ⇔ N_mac·f·u < (Q/b_e)·BW_int   (PE 부족 — 증설로 해결)
           read-bound    ⇔ 그 반대 (설계 목표 regime)
Offload 유리 ⇔ (비-resident or corpus > C_hbm/host DRAM) ∧ BW_int·n_csd > ε·BW_pcie ∧ Q/b_e가 과대하지 않음

가능성 판단 요약: latency 관점의 offload 정당화는 (a) index가 HBM(·host DRAM)에 안 들어가는 규모이고, (b) drive 수가 많아 Σ BW_int가 GPU ingest 한계를 넘고, (c) 비교 대상이 동일 precision일 때 성립한다. (c)가 깨지는 순간(GPU 측이 압축 전송) required MAC이 7배 커지고 대역폭 이점이 7배 줄므로, CSD 설계는 처음부터 압축 도메인(2-bit residual을 내부에서 decompress + MaxSim)을 전제로 잡는 것이 방어 가능한 포지션이다 — 이 경우에도 decompress(element당 centroid lookup + add ≈ FLOP의 1/Q)는 저렴하다.

쟁점과 불확실성

  • 쟁점 1 — baseline 선택이 결론의 부호를 바꾼다: cold-SSD(20.6 ms) vs warm-host(5.4 ms) vs HBM-resident(0.07 ms) vs 압축 전송(0.76 ms급) 중 무엇과 비교하느냐에 따라 동일 CSD가 1.8배 승리~100배 패배 사이를 오간다. 공정한 논문/제안이라면 압축 + host-cache baseline을 명시해야 한다.
  • 쟁점 2 — "internal BW > external link"가 실제 CSD에서 성립하는가: 본 모델의 승리 조건은 BW_int > ε·BW_pcie인데, SmartSSD 1세대류(FPGA가 PCIe switch 뒤에 붙은 구조)는 SSD↔FPGA P2P가 링크 속도로 제한된다는 것이 통설(배경지식, 확인 필요). 진짜 이점은 controller-integrated CSD(자체 NAND channel 직결)에서만 나온다. 대상 CSD 아키텍처의 실측 BW_int가 이 분석의 최대 미확인 변수다.
  • 쟁점 3 — random read pattern: candidate N개는 corpus에 산재하므로 read는 sequential이 아니다. fp16·D=128이면 doc당 33 KB(NAND page 2개)라 효율이 유지되지만, 압축 시 doc당 ~4.6 KB로 page(16 KB)보다 작아져 read amplification이 유효 BW_int를 최대 3~4배 깎을 수 있다(추정). embedding layout(후보 co-location)이 별도 설계 문제로 남는다 — 단 job scope상 로컬리티 분석은 제외.
  • 쟁점 4 — full-scan은 성립하지 않는다: candidate generation 없이 corpus 전체(≈180 GB fp16) MaxSim은 8-drive CSD로도 ~1.9 s. CSD offload는 어디까지나 re-rank 단계(N ≪ corpus)의 이야기이고, candidate generation(PLAID centroid 단계)의 배치는 미해결 설계 쟁점이다.
  • 확인 필요 목록: ColBERT 정확 파라미터(Q=32 padding, D max 180, index 154 GiB 여부), ColBERTv2 per-vector bytes(20/36 B)와 압축률(6–10×), PLAID의 기본 ndocs와 latency 수치, SmartSSD DSP 수·클록·P2P BW, SNIA CS command 오버헤드, GPUDirect Storage 실효 BW. — 전부 이번 세션에서 URL 메타데이터만 확인(웹 도구 권한 거부).
