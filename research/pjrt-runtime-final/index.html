PJRT 런타임의 역할과 인터페이스 최종 정리

기준 소스: OpenXLA openxla/xla 저장소, commit 82d89744a3b (로컬 sparse checkout repo/).
분석 대상: xla/pjrt/pjrt_client.{h,cc}, pjrt_compiler.h, pjrt_executable.h, common_pjrt_client.{h,cc}, pjrt_device_description.h, xla/pjrt/c/pjrt_c_api.h.
아래 모든 파일:라인은 이 커밋 기준이며, 라인이 흔들릴 수 있는 곳은 클래스/함수명을 함께 적었다.

3줄 요약

  • 컴파일러: StableHLO/MLIR 모듈 또는 XlaComputation을 받아 디바이스 코드와 실행 메타데이터(shape·layout·sharding·memory kind·alias)를 담은 PjRtExecutable(아직 로드되지 않은 아티팩트) 까지만 만들어 런타임에 넘긴다 (PjRtCompiler::Compile, pjrt_compiler.h:447-460).
  • 런타임(PJRT): 그 아티팩트를 디바이스에 Load하고, 디바이스/메모리 스페이스/버퍼 수명/H2D·D2H 전송/인자 검증/출력 할당/OOM 재시도/two-phase launch/비동기 이벤트를 오케스트레이션하여, 프레임워크에는 PjRtBuffer 핸들과 Future<> 를 돌려준다 (PjRtClient 계약 pjrt_client.h:546-1113, 공통 구현 CommonPjRtClient/CommonPjRtLoadedExecutable common_pjrt_client.h:62-755).
  • 드라이버/벤더 백엔드: 런타임이 정해준 크기의 raw 메모리 할당(AllocateRawBuffer), 호스트 데이터의 디바이스 레이아웃 기록(LinearizeInto), 실행 상태 재구성(LoadRawExecutable), 실제 커맨드 제출(PjRtRawLoadedExecutable::Execute)만 구현하며, PjRtRawBufferRef와 PjRtDeviceEventRef라는 raw 타입으로만 런타임과 대화한다 (common_pjrt_client.h:141-172, 453-471, 684-687).

수집 범위

  • 포함: 위 8개 파일의 실제 선언/구현을 근거로 한 (1) 컴파일러–런타임–드라이버 책임 경계, (2) 계층 간 데이터 타입, (3) 핵심 객체별 public API 카탈로그, (4) C API ABI 구조. 참고 문서 7건(PJRT Plugin Mechanism, ABI versioning ×2, optional/experimental features, AcquireExternalReference C API, dynamic shapes, Next Pluggable Device)은 배경과 버전 드리프트 확인에 사용.
  • 제외: 특정 벤더 구현(pjrt_stream_executor_client, CPU/GPU/TPU client)의 내부, StableHLO 자체, 분산 런타임(coordination service) 상세. sparse checkout에 없는 헤더(raw_buffer.h, device_event.h, abstract_tracked_device_buffer.h, future.h 등)의 내부 정의는 "쟁점과 불확실성"의 확인 필요 표로 분리.
  • 자료 유형: 소스 코드(진실의 원천), 설계 문서 텍스트(2022–2023년 작성, 드리프트 존재), 원본 요청문(references/text/pasted-text.txt).

핵심 요약

  1. PJRT는 2계층 구조다: pjrt_client.h는 순수 가상 계약(프레임워크가 보는 것), common_pjrt_client.{h,cc}는 raw buffer 기반 클라이언트들의 공통 파이프라인 구현이다 (common_pjrt_client.h:61-62 "A common base class for Pjrt clients based on raw buffers").
  2. 컴파일 산출물 PjRtExecutable은 실행 기능이 없고 메타데이터 조회와 직렬화만 제공한다 (pjrt_executable.h:356-449). 실행 가능해지는 것은 PjRtClient::Load/CompileAndLoad가 만드는 PjRtLoadedExecutable부터다 (pjrt_client.h:677-732).
  3. 버퍼 생성(H2D)의 공통 파이프라인은 5단계다: MakeDefaultShapeForMemorySpace(shape·layout 정규화) → GetOnDeviceBytesCount → AllocateRawBuffer → LinearizeInto/LinearizeHostBufferInto(H2D 전송, definition event 반환) → DefineBuffer(핸들 조립) (CommonPjRtClient::BufferFromHostLiteral, common_pjrt_client.cc:277-304).
  4. 실행은 Prepare(실패 가능·재시도 가능)와 Launch(성공 보장)로 분리된다: "Execute is split into Prepare and Launch. Prepare can fail and be retried, while Launch is guaranteed to succeed." (common_pjrt_client.h:658-659).
  5. 멀티 디바이스 Execute는 gang scheduling + two-phase launch로 동작한다: 모든 코어가 Prepare를 마칠 때까지 barrier로 기다렸다가, 하나라도 실패하면 전체를 포기하고, 전원 성공 시에만 Launch에 들어간다 (CommonPjRtLoadedExecutable::Execute, common_pjrt_client.cc:1969-2005).
  6. 런타임→프레임워크 출력과 런타임→드라이버 출력은 타입부터 다르다: 위로는 std::vector<std::vector<std::unique_ptr<PjRtBuffer>>>와 Future<>, 아래로는 absl::Span<const PjRtRawBufferRef>(입력·출력 자리)와 PjRtDeviceEventRefVector(의존성)이다 (pjrt_client.h:1470-1473 vs common_pjrt_client.h:464-470).
  7. 출력 버퍼 메모리는 드라이버가 아니라 런타임이 Prepare 단계에서 미리 할당해 드라이버에 내려보낸다. 드라이버의 Execute는 결과를 쓸 자리(results)를 받는다 (AllocateOutputBuffersWithInputReuse common_pjrt_client.cc:981-1153, ExecuteLaunch cc:1704-1735).
  8. OOM 처리도 런타임 책임이다: 출력 할당은 retry_on_oom=false로 즉시 실패시키고(cc:1071-1073), 상위 ExecutePrepareWithOomRetries가 ShouldRetryOnOom 정책에 따라 Prepare 전체를 재시도한다 (cc:1737-1762).
  9. C API(pjrt_c_api.h)는 이 C++ 계약을 함수 포인터 테이블 하나(PJRT_Api, pjrt_c_api.h:3026-3218)와 불투명 구조체들로 감싼 ABI 경계이며, struct_size 필드와 MAJOR 0/MINOR 113 버전(106, 121행)으로 전방 호환을 관리한다.
  10. 이 커밋에는 PjRtFuture<>가 한 번도 등장하지 않는다. 비동기 반환 타입은 전부 xla::Future<>/tsl::Future<>다 — 기존 문서·블로그와 비교할 때 가장 큰 버전 드리프트.

2.4 Future<>와 ready event, Prepare/Launch 분리

  • 두 개의 비동기 어휘가 공존한다. 프레임워크 경계는 xla::Future<>(값/에러가 한 번 fulfil되는 future), 디바이스 경계는 PjRtDeviceEventRef/PjRtDeviceEventPromiseRef(디바이스 이벤트 참조; 상태는 C API enum PJRT_DeviceEvent_State_{Error,Unavailable,Ready}로 판별 — common_pjrt_client.cc:919-928). 이 둘을 잇는 어댑터가 CommonPjRtClient::MakeTrackedReadyFuture(device_event, ...) → tsl::Future<> (cc:615-628)와 CreateLinkedEventPromise(h:195-201)다.
  • 버퍼의 ready: PjRtBuffer는 생성 즉시 반환되고(값은 아직 없음), definition event들이 완료되면 GetReadyFuture()가 fulfil된다. 구현은 tracked buffer의 이벤트에서 future를 파생한다 (CommonPjRtBufferImpl::GetReadyFuture, cc:3075-3087).
  • Prepare vs Launch (타입 관점): ExecutePrepare(...) -> absl::Status — 검증·할당·LoadRawExecutable까지 전부 실패 가능 경로이고 상태만 반환한다 (cc:1555-1610). 반면 ExecuteLaunch(...) -> absl::StatusOr<Result>는 이미 채워진 ExecuteLaunchArgs를 소비할 뿐이라 설계상 실패하지 않는 단계로 취급되며("Phase 2: Launch. It cannot fail." cc:1999), 드라이버 쪽 실패는 status가 아니라 error 상태의 event/버퍼로 전파된다(RawExecuteResult.inline_status와 is_predetermined_error, error buffer 경로 h:92-93).
  • OOM 재시도: Prepare가 ResourceExhausted로 실패하면 ShouldRetryOnOom(attempts, device, executable, status)가 허락하는 동안 ExecuteLaunchArgs를 새로 만들어 재시도한다 (cc:1737-1762). 출력 할당이 retry_on_oom=false인 이유가 이것이다 — 낱개 할당에서 기다리지 않고 Prepare 전체 단위로 재시도한다 (cc:1071-1073).
  • two-phase가 주는 보장: 멀티 디바이스에서 한 코어라도 Prepare에 실패하면 아무 코어도 Launch하지 않으므로, "일부 디바이스만 실행된" 상태가 생기지 않는다 (cc:1969-1996; supports_two_phase_launch() h:75).

2.5 PJRT C API — PJRT_Api와 opaque struct

  • 엔트리포인트: 플러그인 .so는 GetPjrtApi() 심볼 하나로 PJRT_Api*를 노출한다 (참고 문서 "PJRT Plugin Mechanism": dlsym(library_handle, "GetPjrtApi")).
  • PJRT_Api (pjrt_c_api.h:3026-3218): struct_size + extension_start + PJRT_Api_Version 뒤에, PJRT_Error_Destroy부터 PJRT_TopologyDescription_GetMemorySpaceKindIds까지 전 메서드가 함수 포인터 필드로 나열된 단일 vtable이다. 새 필드는 끝에만 추가한다 ("Always add new fields to the end of the struct", 3142-3143).
  • 불투명 구조체: PJRT_Client, PJRT_Device, PJRT_Memory, PJRT_Executable, PJRT_LoadedExecutable, PJRT_Buffer, PJRT_Event, PJRT_Error, PJRT_TopologyDescription 등은 forward 선언만 있다 (141, 308, 408-435행). C++ 객체의 소유는 플러그인 내부에 있고, 프레임워크는 포인터만 쥔다. C++ 클래스와 1:1 대응: PjRtClient↔PJRT_Client, PjRtBuffer↔PJRT_Buffer, PjRtLoadedExecutable↔PJRT_LoadedExecutable, Future/DeviceEvent↔PJRT_Event.
  • 호출 규약: 모든 메서드는 PJRT_Error* PJRT_Foo(PJRT_Foo_Args*) 형태. Args 구조체마다 struct_size가 있어 호출자와 구현체가 서로 아는 필드 범위를 협상한다(PJRT_DEFINE_STRUCT_TRAITS, 27-47행). 버전은 PJRT_API_MAJOR 0 / PJRT_API_MINOR 113 (106, 121행)이고, 참고 문서 기준 minor 추가에 6→12주 전방 호환 윈도우를 둔다.
  • 컴파일·실행의 C 형식: 프로그램은 PJRT_Program{ char* code; const char* format; }로, format은 "hlo"/"hlo_with_config"/"mlir" (739-754). 옵션은 serialized CompileOptionsProto 바이트 (765-766). PJRT_Client_Compile은 곧바로 PJRT_LoadedExecutable*을 내고(767), 토폴로지 기반 PJRT_Compile은 unloaded PJRT_Executable*을 내며(3012), PJRT_Client_Load가 둘을 잇는다(775-788). 실행은 PJRT_LoadedExecutable_Execute_Args{ argument_lists[num_devices][num_args]; output_lists(호출자 할당); device_complete_events; execute_device; } (2047-2080) — C++의 returned_futures가 PJRT_Event**로 바뀐 모습이다.
  • 확장: 코어 테이블에 없는 기능(RawBuffer, PhaseCompile, FFI 등)은 PJRT_Extension_Base 연결 리스트로 붙는다 (61-97행). 이는 참고 문서 "optional and experimental features"의 Proposal A가 채택된 형태다.
  • 역방향 연동(신규): 이 커밋에서는 C++ 쪽 PjRtMemorySpace가 ToCApiPtr()/FromC()를 갖고 PJRT_Memory_FunctionTable vtable(get_user_data/set_user_data)로 C 객체를 직접 들고 다닌다 (pjrt_client.h:93-95, 124-146; pjrt_client.cc:50-118; pjrt_c_api.h:413-427). C++ 인터페이스와 C API가 단방향 래핑이 아니라 상호 참조로 얽히기 시작한 신호다.
이 파트에서 발표할 핵심 한 문장: 런타임은 위(프레임워크)로는 PjRtBuffer와 Future<>라는 "핸들 언어"로, 아래(드라이버)로는 PjRtRawBufferRef와 PjRtDeviceEventRef라는 "바이트·이벤트 언어"로 말하며, PJRT C API는 그 핸들 언어를 함수 포인터 테이블 하나(PJRT_Api)와 불투명 포인터들로 직렬화한 ABI다.

쟁점과 불확실성

코드만으로 단정할 수 없거나 발표에서 반론이 나올 수 있는 지점을 쟁점(해석이 갈리는 것) → 확인 필요(정의를 못 본 것) → 추정(정황만 강한 것) → 버전 드리프트(문서와 코드가 다른 것) 네 층위로 분리했다.

쟁점 (해석이 갈릴 수 있는 지점)

  1. ExecuteLaunch의 "성공 보장"은 어디까지 유효한가. 코드 주석은 "Phase 2: Launch. It cannot fail." (cc:1999)라고 단언하지만, 드라이버 반환 타입에는 RawExecuteResult::inline_status(h:462)와 is_predetermined_error 인자가 존재한다. 즉 "실패하지 않는다"는 호스트 측 오케스트레이션 상태 코드 경로에 한정된 진술로 읽는 것이 자연스럽고, 디바이스 측 실패는 error event/error buffer로 비동기 전파된다고 보이지만, inline_status가 non-OK가 되는 실제 조건은 이 checkout으로 확정할 수 없다 (후속 질문 3과 연결).
  2. C++ 인터페이스와 C API의 레이어링 역전. 통념(그리고 Next Pluggable Device 문서의 "C++/C/C++ 샌드위치")은 C API를 C++ 계약 위의 얇은 단방향 래퍼로 그리지만, 이 커밋에서는 pjrt_client.h가 c/pjrt_c_api.h를 직접 include하고(73행), PjRtMemorySpace가 ToCApiPtr()/FromC()를 요구하며(93-95행), device event 상태 판별이 C enum PJRT_DeviceEvent_State로 이뤄진다(common_pjrt_client.cc:919-928). 이것이 일시적 마이그레이션 상태인지 의도된 최종 구조인지는 공개 설계 문서가 없어 판단을 보류한다.
  3. 버퍼 소속 모델의 이중성(디바이스 vs 메모리 스페이스). PjRtBuffer::device()는 deprecated(TODO b/277820585)이고 구현은 memory_space_->devices()[0]이지만(common_pjrt_client.h:789-792, cc:3003-3006), API 표면에는 device()가 여전히 남아 두 모델이 공존한다. 발표에서는 "버퍼는 memory space 소속"으로 말해야 하나, 다대다 매핑에서 devices()[0]을 대표로 고르는 것이 규약인지 임시 편의인지는 코드 주석만으로 확정되지 않는다.
  4. common_pjrt_client.* 파이프라인의 일반화 가능성. 이 계층은 Copyright 2025의 신설 코드이며 "raw buffers 기반 클라이언트의 공통 베이스"(h:61-62)라고만 선언한다. CPU/GPU/TPU 실물 클라이언트가 sparse checkout에 없어, 본 문서 Part 1·2의 Prepare/Launch·5단계 H2D 서술이 모든 백엔드에 적용된다고 일반화할 수 없다 — 이 계층을 채택한 클라이언트에 한정된 설명으로 읽어야 한다.
  5. 문서와 코드 중 무엇을 진실로 삼는가. 참고 문서 7건은 전부 2022–2023년 설계 시점 자료이고, 명칭 수준의 드리프트가 실측된다(PjRtFuture 등장 0회 등). 본 문서는 커밋 82d89744a3b의 코드를 진실의 원천으로 삼았으며, 문서에만 있는 주장은 아래 버전 드리프트 표에 격리했다.

  • PjRtRawLoadedExecutable::Execute가 rvalue 한정(&&)인 것은 "launch state는 1회용이며 매 실행마다 LoadRawExecutable로 재구성한다"(h:451-453 주석)를 타입으로 강제하기 위한 설계로 추정.

후속 질문

다음 스터디 세션에서 이어갈 질문 다섯 개. 각 질문은 위 "쟁점과 불확실성"의 미확정 항목을 닫는 것을 목표로 한다.

  1. PjRtRawBuffer의 전체 계약은 무엇인가? raw_buffer.h를 열어 CopyRawHostToDeviceAndReturnEvent/MultiSlice/CopyTo/GetRawBufferAsyncValue의 시그니처를 확인하고, RawBuffer C API 확장(PJRT_Extension_Type_RawBuffer)과 어떻게 연결되는지 추적한다.
  2. 실제 백엔드는 raw boundary를 어떻게 채우는가? CPU 클라이언트(xla/pjrt/cpu/)와 GPU 클라이언트에서 AllocateRawBuffer/LinearizeInto/LoadRawExecutable/PjRtRawLoadedExecutable::Execute 구현을 각각 찾아, "커맨드 제출"이 CPU 스레드풀과 GPU 스트림에서 어떻게 다르게 생겼는지 비교한다. 아울러 어떤 클라이언트가 CommonPjRtClient를 실제로 상속하는지 확인해 쟁점 4(일반화 가능성)를 닫는다.
  3. ExecuteLaunch의 "성공 보장"의 정확한 경계는? RawExecuteResult::inline_status가 non-OK가 되는 실제 케이스와, 디바이스 측 비동기 실패가 error event → error buffer → GetReadyFuture() 에러로 전파되는 전체 체인을 구현 코드에서 확인한다.
  4. donation 트랜잭션의 동시성 규칙은? CommonPjRtBuffer::ScopedHold(abstract_tracked_device_buffer.h)에서 kDonation hold 중 kUsage 요청이 블록되는지 실패하는지, ConvertUsageHold/ConfirmDonation의 원자성을 확인한다 (pjrt_client.h:1442-1452의 TODO(chky)와 연결).
  5. C API 래퍼의 왕복 비용과 계층 통과 방식은? PjRtCApiClient(프레임워크 쪽 C++)와 pjrt_c_api_wrapper_impl(플러그인 쪽 C++)이 PJRT_Buffer/PJRT_Event를 사이에 두고 어떻게 소유권·이벤트를 넘기는지, 특히 PJRT_LoadedExecutable_Execute_Args.device_complete_events가 C++ Future<>로 복원되는 지점을 따라가며 쟁점 2(레이어링 역전)의 방향성도 함께 판정한다.
