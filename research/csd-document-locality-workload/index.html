CSD Resident Embedding을 위한 Document Locality 측정 설계 — 워크로드 선정, Trace 생성, 지표 정의

한 줄 요약

PLAID engine + ColBERTv2 index + MS MARCO passage corpus 조합을 워크로드로 선정하고, query별로 residual decompression 단계에서 실제 읽히는 passage embedding block 집합을 trace로 남겨, access frequency / reuse distance / working-set / hot coverage를 계산하면 CSD residency hit ratio와 PCIe traffic 절감량을 곧바로 유도할 수 있다.

수집 범위

  • 포함: ColBERT/ColBERTv2/PLAID/MS MARCO/BEIR 중 워크로드 선정, query stream → access trace 생성 절차와 record schema, locality metric 수식, CSD residency benefit 추정 절차, scheduling 인계 factor 정의.
  • 제외: end-to-end 성능 모델, CSD 하드웨어 설계 자체, retrieval 품질(MRR/nDCG) 분석.
  • 자료 상태(중요): references/manifest.json의 5개 문서는 모두 "URL only; content was not fetched automatically" 상태이고, 이번 세션에서 WebFetch/WebSearch 권한이 거부되어 원문을 직접 열람하지 못했다. 따라서 5개 문서는 전부 "URL 메타데이터만 확인"이며, 본문의 기술 내용은 해당 논문·코드베이스에 대한 배경지식으로 작성하고 수치는 추정/확인 필요로 구분했다. 첨부 repo도 없음(repo-files.txt: "No repository attached").

핵심 요약

  1. 최종 추천 워크로드: PLAID(실행 엔진) + ColBERTv2(index 포맷) + MS MARCO v1 passage(약 8.8M passages, 배경지식). BEIR는 query 수가 적어 보조 검증용, ColBERT v1은 index가 비압축이라 CSD 시나리오와 거리가 있다.
  2. "접근"의 정의가 결과를 좌우한다: PLAID는 후보를 centroid 점수만으로 걸러내므로, storage에서 실제로 읽는 것은 최종 단계에서 residual을 decompress하는 passage 집합뿐이다. 이 집합(이하 H2)을 byte 단위 접근으로 정의하고, 1단계 candidate 집합(H1)은 scheduling feature용으로 별도 기록한다.
  3. 공개 query set은 dedup되어 있다(배경지식): 동일 query 반복이 없으므로 문서 재사용은 서로 다른 query의 candidate 중복에서만 나온다. 따라서 S_uniq(고유 query 스트림, locality 하한) 과 S_zipf(Zipf 재표집 스트림, 운영 환경 근사) 두 스트림을 모두 측정해야 주장이 방어된다 — 본 설계의 최대 쟁점.
  4. 지표는 6종으로 충분하다: access frequency, reuse ratio, temporal reuse distance(LRU stack distance), working-set size(Denning WSS + Heaps 지수), hot coverage(Lorenz/CDF), Zipf α. 문서 크기가 token 수에 비례하므로 전 지표를 byte-weighted 버전과 병행 계산한다.
  5. PCIe 절감 주장에 필요한 최소 metric은 두 개다: baseline bytes/query 분포와 byte-weighted residency hit ratio HR_bytes(C). CSD가 hit을 내부에서 처리하면 절감율 = HR_bytes(C)가 그대로 성립한다.
  6. 규모 감(추정): ColBERTv2 2-bit 압축 시 token embedding당 ~36B, passage당 평균 ~2.5KB. PLAID k=100 설정에서 query당 수백 개 passage를 decompress한다고 알려져 있어 압축 상태로 ~0.6MB/query, fp16 원본 전송이면 ~4.5MB/query 수준 — 1k QPS면 PCIe Gen4 x4 대역폭(~8GB/s, 배경지식)의 상당 부분을 embedding 이동이 차지한다.
  7. static hot-set 평가는 trace의 train/test 분할이 필수다. 전체 trace로 hot set을 뽑아 같은 trace로 hit ratio를 재면 oracle leakage다.
  8. scheduling에 넘길 핵심 online feature는 candidate set locality ρ_q(후보 중 CSD resident 비율) 이다. ρ_q는 stage-1 직후(어떤 residual도 읽기 전에) 계산 가능하므로 GPU/CSD dispatch의 자연스러운 결정점이 된다.
  9. PLAID의 단계별 파라미터(nprobe, ndocs, 최종 decompress 수)와 index 수치는 원문 미열람 상태의 배경지식이므로, 본문 말미의 확인 필요 리스트를 논문·코드로 검증한 뒤 수치를 확정해야 한다.

관계 요점: MS MARCO는 corpus+query pool을, ColBERTv2는 CSD에 상주시킬 저장 객체(residual blocks) 를, PLAID는 어떤 pid의 block이 언제 읽히는지를 결정하는 접근 패턴 생성기를 제공한다. BEIR는 corpus를 바꿔 같은 파이프라인을 돌리는 민감도 실험 축이다.

PLAID 한 query의 생애주기와 storage 접근 지점(배경지식 기반 재구성, 함수/단계명은 확인 필요):

  • 상태/객체 순서: query handle(qid, q_idx) → centroid score 행렬(임시) → H1 candidate pid 리스트 → pruning 후 H2 pid 리스트 → residual block read 요청(pid→(chunk, offset, len) 변환은 doclens 누적합) → decompress된 embedding(임시) → top-k 결과로 완료.

  • centroid table과 IVF/doclens는 크기가 작아(수십~수백 MB 추정) 무조건 resident로 두고, locality 측정 대상에서 제외한다(쟁점 2 참조).

선정 이유:

  1. PLAID의 staged pruning 때문에 "candidate로 잡힘"과 "실제 storage에서 embedding을 읽음"이 분리된다. CSD 논거에 필요한 것은 후자이고, PLAID는 그 집합을 코드 상 명시적으로 만든다(decompress 대상 pid 리스트).
  2. ColBERTv2 residual block(passage당 ~KB, 추정)은 CSD에 두기에 적절한 크기의, 잘 정의된 저장 객체다. dense single-vector 검색(embedding 1개 = 수백 B)보다 문서당 byte가 커서 residency 효과가 뚜렷하게 나타난다.
  3. MS MARCO만이 수십만 규모 query pool을 제공해 train/test 분할과 긴 스트림 구성이 가능하다.
  4. 전 구성요소가 공개(코드: stanford-futuredata/ColBERT GitHub repo — 배경지식, URL 미검증)라 계측 hook 삽입이 현실적이다.

필요 데이터:

  • MS MARCO v1 collection.tsv(~8.8M passages), queries.train.tsv / queries.dev.tsv(스트림 구성용; qrels는 선택),
  • ColBERTv2 공개 checkpoint, PLAID index 빌드 산출물: centroids, ivf(centroid→pid), doclens(pid→token 수), residuals(압축 embedding) — 파일명·포맷은 repo 확인 필요,
  • (선택) query 반복 모델 근거: ORCAS 등 공개 click/query log 통계(배경지식, 확인 필요),
  • 자원: index 빌드용 GPU, index 저장 ~20-25GB(2-bit 기준 추정).

2. Trace 생성 방법

절차 (①~⑥):

① Index 빌드: MS MARCO에 대해 ColBERTv2/PLAID index를 표준 설정(dim=128, 2-bit residual — 추정)으로 빌드하고, doclens 누적합으로 정적 매핑 pid → (byte_start, byte_len)을 만든다.

② Query stream 정의 — 두 개를 만든다:

  • S_uniq: dev-full(또는 train에서 표본) query를 무작위 순서로 1회씩 재생. 반복이 없으므로 corpus 내재 locality(후보 중복)만 측정하는 하한.
  • S_zipf: query pool에서 Zipf(α_q) 인기 분포로 N개(예: 1M)를 복원추출해 반복이 있는 스트림을 합성. α_q는 공개 검색 로그의 query 빈도 왜도에서 가져오되(배경지식상 0.7~1.0 부근, 확인 필요) 민감도 sweep(α_q ∈ {0.6, 0.8, 1.0})으로 방어한다.
  • 각 query에 q_idx(0..N-1, 논리 시간축)와 합성 t_arrival(Poisson 도착 등)을 부여한다. locality 계산은 q_idx 기준, latency 모델만 t_arrival 사용.

③ 엔진 계측 — 3개 hook:

  • H1: stage-1 candidate generation 직후의 candidate pid 집합(+ probed centroid ids).
  • H2: residual decompression에 들어가는 pid 집합 — 실제 storage read. 이것이 locality의 1차 접근 정의.
  • H3: 최종 top-k pid.
  • hook 위치는 ColBERT repo의 searcher/index-scorer 경로(배경지식상 Searcher.search → candidate generation → filter → residual lookup 순, 정확한 함수명 확인 필요).

④ embedding block ID 변환: 고정 block 크기 BS(4KB와 64KB 두 값으로 모두 산출 권장)에 대해 block_ids(pid) = { floor(byte_start/BS) .. floor((byte_start+byte_len-1)/BS) }. passage(~2.5KB 추정) < 4KB이므로 한 block에 여러 pid가 섞인다 → pid 단위와 block 단위 지표가 달라진다(block 단위는 spatial locality·배치 순서 효과 포함). query 내에서 block id는 dedupe한다.

⑤ 후처리: 스트림 순서대로 스캔하며 pid별 prev_q_idx, delta_q = q_idx − prev_q_idx, LRU stack distance(아래 §3)를 채운다. 한 query 내 동일 pid 중복 접근은 1회로 접는다(query가 timestep).

⑥ 저장: Parquet 3-table.

Trace schema:

# queries 테이블 (query당 1행)
q_idx        int64    # 스트림 내 위치 = 논리 시간
qid          str      # 원본 query id (S_zipf에서는 반복 등장)
t_arrival    float64  # 합성 도착시각 (s)
n_q_tokens   int16
nprobe, k, ndocs      # PLAID 설정 (sweep 축)
n_cand_H1    int32    # stage-1 candidate 수
n_scored_H2  int32    # decompress된 pid 수
bytes_H2     int64    # Σ byte_len, 해당 query의 storage read 총량
topk_pids    list<int32>

# accesses 테이블 ((q_idx, pid)당 1행; H2 기준이 기본, stage_max로 H1도 표현)
q_idx        int64
pid          int32
stage_max    enum {H1_CANDIDATE, H2_SCORED, H3_TOPK}   # 도달한 최고 단계
n_tokens_d   int16
bytes        int32    # n_tokens_d × B_emb
block_start  int64
n_blocks     int16
prev_q_idx   int64    # 직전 접근 (없으면 -1 = cold)
delta_q      int64    # temporal gap
sd_docs      int64    # LRU stack distance (pid 단위)
sd_bytes     int64    # byte-weighted stack distance

# docs 테이블 (정적, pid당 1행)
pid, n_tokens, bytes, byte_start, block_start, n_blocks

이 schema로 §3의 모든 지표가 trace 재스캔 없이 groupby/집계만으로 계산되고, §5의 scheduling feature도 같은 field에서 파생된다.

3. Locality Metrics

표기: 스트림 q_1..q_N, query i의 접근 집합 A_i(기본 = H2 pid 집합), 총 접근 수 T = Σ_i |A_i|, 총 byte T_B = Σ_i Σ_{d∈A_i} bytes(d), 고유 문서 수 U = |∪ A_i|, corpus 크기 |C|.

f(d) = Σ_{i=1..N} 1[d ∈ A_i], 정규화 p(d) = f(d)/T. byte 가중: f_B(d) = f(d)·bytes(d). 파생 — corpus utilization util = U/|C| (전체 corpus 중 한 번이라도 읽힌 비율; CSD에 "전체를 둘 필요가 없다"는 첫 근거).

  • access frequency:

RR = (T − U)/T (전체 접근 중 재접근 비율), byte판 RR_B = (T_B − Σ_{d:f>0} bytes(d))/T_B. 보조 지표: multi = |{d : f(d) ≥ 2}| / U (한 번이라도 재사용된 문서 비율). RR이 낮으면 어떤 캐시도 못 이긴다 — residency 논거의 관문 지표.

  • reuse ratio:
  • temporal reuse distance — 두 정의를 모두 기록:
  • query-gap: Δq(d, j) = j − prev(d, j) (같은 pid의 연속 접근 간 query 수).
  • LRU stack distance: sd(d, j) = |{ d′ ≠ d : last_access(d′) ∈ (prev(d,j), j) }| — 두 접근 사이에 등장한 고유 문서 수. last-access 위치에 대한 Fenwick tree로 O(T log U)에 계산(Mattson 계열 표준 기법, 배경지식).
  • 핵심 성질: LRU 용량 C(문서 수)에서의 hit ratio가 HR_LRU(C) = Pr[sd < C]로 즉시 나온다. byte 용량 B에 대해서는 sd_bytes(그 구간 고유 문서 byte 합)로 HR_LRU,B(B) = Pr_bytes[sd_bytes < B]를 쓴다(가변 크기 객체라 근사임을 명시; passage 크기 분산이 작아 오차 작음 — 추정).
  • 그래프: reuse distance CCDF(log-log). 꼬리가 짧을수록 작은 CSD DRAM으로 충분.

WSS(τ) = E_t[ | ∪_{i=t−τ+1..t} A_i | ] — 최근 τ개 query가 건드린 고유 문서 수(byte판 WSS_B(τ) 병행). τ ∈ {10, 100, 1k, 10k, 100k}로 곡선을 그린다. 보조: Heaps' law 적합 U(n) ≈ K·n^β (스트림 앞 n개 query까지의 누적 고유 문서). β < 1이 뚜렷할수록 접근이 좁은 집합에 갇힌다는 뜻으로, S_uniq에서도 성립하는지가 핵심 관찰점.

  • working-set size (Denning, 배경지식):

f(d) 내림차순 순위 r(d)에 대해 Cov(x) = Σ_{r ≤ x·U} f(d_(r)) / T (byte판은 f_B/T_B). 보고 값: Cov(1%), Cov(5%), Cov(10%)와 90% 접근을 덮는 최소 문서 비율 x_90 = min{x : Cov(x) ≥ 0.9}. 그래프(질문 4의 답): ① rank–frequency log-log(Zipf plot), ② 누적 곡선 "top x% 문서가 y% 접근을 커버"(x 로그축, 1/5/10% 마커 표시, Lorenz curve와 동치) — CSD 스토리에는 ②가 주 그림, ①은 근거 그림. 불평등 요약치로 Gini G = 1 − 2∫₀¹ L(x)dx.

  • hot coverage:

f(r) ∝ r^{−α} 가정 하에 α를 MLE(Clauset-Shalizi-Newman 방식/powerlaw 패키지, 배경지식)로 적합하고 log-log OLS와 병기. tail cutoff(f(d)=1 무리)는 적합에서 제외. 해석: S_zipf의 α는 주입한 query 반복 왜도 α_q에 종속되므로, S_uniq에서 측정된 α(순수 candidate 중복 유래)를 반드시 따로 보고하고 두 스트림을 분리 해석한다.

  • Zipf/skew:

모든 지표는 (a) pid 단위 / block 단위, (b) count / byte-weighted, (c) S_uniq / S_zipf의 세 축 조합으로 계산한다 — 표 하나에 요약 가능한 8~12개 셀이 결과의 뼈대가 된다.

4. CSD Residency Benefit

설정: 용량 B bytes의 CSD 내부 상주 영역(DRAM 또는 pinned 구역)에 embedding block 일부를 두고, 나머지는 miss로 처리. 세 정책을 시뮬레이션한다 — ① STATIC-HOT: train 구간 f(d) 상위 문서를 byte 예산 B까지 pin, ② LRU(online) — §3의 sd_bytes 분포에서 바로 계산, ③ (상한 참고용) offline optimal 근사.

스트림을 시간순 50/50으로 나눠 train으로 HotSet(B)를 만들고 test 구간에서만 평가한다(oracle leakage 방지 — 필수 절차). HR_doc(B) = Σ_{i∈test} |A_i ∩ HotSet(B)| / Σ_{i∈test} |A_i| HR_bytes(B) = Σ_{i∈test} Σ_{d∈A_i∩HotSet(B)} bytes(d) / Σ_{i∈test} Σ_{d∈A_i} bytes(d) LRU는 HR_LRU,B(B) = Pr[sd_bytes < B]. PCIe 주장에 쓰는 값은 HR_bytes다.

  • residency hit ratio:

baseline(embedding이 일반 SSD/host에 있고 scoring 위치로 이동): Base = Σ_{i∈test} bytes_H2(i). CSD가 hit을 내부에서 scoring하는 설계: PCIe에는 query embedding(≈ n_q·128·2B ≈ 8KB/query, 추정)과 top-k 결과(수 KB)만 흐르므로 Saved(B) = HR_bytes(B) · Base, Reduction(B) = Saved/Base = HR_bytes(B) (+ 고정 metadata 보정). miss를 host로 올려 scoring하는 hybrid 설계면 PCIe bytes/query = (1 − HR_bytes(B)) · E[bytes_H2].

  • traffic saved bytes:

miss 1건 비용 모델 — T_miss(d) = bytes(d)/BW_NAND_int + T_NAND_lat + [hybrid: bytes(d)/BW_PCIe + host 처리], hit은 T_hit(d) = bytes(d)/BW_DRAM_int ≈ 0. query 기대 추가지연: E[ΔT] = (1 − HR_bytes(B)) · E[bytes_H2] / BW_bottleneck + miss 건수 · T_NAND_lat. 절감 주장이 성립할 조건(hybrid 기준): HR_bytes(B) > 1 − BW_PCIe_gain_threshold 형태의 부등식으로 제시하되, BW 상수(PCIe Gen4 x4 ≈ 8GB/s, NAND/내부 DRAM 대역 — 모두 배경지식)는 별도 표로 분리해 가정임을 명시한다.

  • cold miss cost:

B ∈ {0.25, 0.5, 1, 2, 4, 8, 16} GB (index 총량 ~22GB 추정 대비 ~1%→~70%)에서 HR_bytes(B) 곡선과 Saved GB / 1k queries를 그린다. 곡선의 knee B*(한계 이득 < ε 지점)가 CSD DRAM sizing 근거가 된다. 목표 서사: "index의 x%인 B* GB만 상주시켜도 H2 byte의 y%가 장치 내부에서 끝난다".

  • hot document capacity sweep:
  • 최소 metric 세트(질문 5의 답): ① E[bytes_H2/query](baseline traffic), ② HR_bytes(B) 곡선(정책 1개 이상, train/test 분할), ③ miss 비용 가정 1세트. 이 셋이면 "PCIe traffic이 HR_bytes만큼 줄고 miss 페널티를 감안해도 net win" 주장이 완결된다. 나머지 지표(§3)는 이 곡선이 왜 그렇게 나오는지의 설명 변수다.
  • 규모 예시(전부 추정, 확인 필요): passage당 68 tokens × 36B ≈ 2.45KB(2-bit). k=100에서 query당 decompress ~256 pid라면 bytes_H2 ≈ 0.63MB/query; GPU에서 fp16으로 scoring하려고 원본을 옮기면 256B/token 기준 ~4.5MB/query. 1k QPS 시 각각 ~0.6GB/s, ~4.5GB/s — 후자는 PCIe Gen4 x4의 절반 이상을 embedding 이동에 소모한다는 계산이 되어 CSD 동기부여의 정량 예시로 쓸 수 있다.

5. Scheduling으로 넘길 Factor

이 trace 분석의 산출물을 이후 GPU/CSD scheduling 연구의 feature로 인계한다. 결정점은 stage-1 직후(candidate pid는 알지만 residual은 아직 안 읽은 시점)다.

  • online/offline feature 구분 원칙: offline = index 빌드/프로파일 시점에 고정되어 lookup만 하는 것(HotSet 멤버십, hotness seed, 문서 bytes). online = query 도착 후 갱신되는 것(EWMA hotness, residency bitmap, ρ_q, CSD queue depth). trace schema(§2)의 prev_q_idx / sd_bytes / stage_max / bytes가 이 feature들의 학습·재현 데이터가 된다.
  • 활용 sketch: route(q) = CSD if ρ_q ≥ θ and csd_queue < L else GPU 같은 threshold 정책의 θ 민감도를 이 trace로 offline 재생 평가할 수 있고, hotness/Δq는 promotion(승격)·eviction 정책의 입력이 된다.
  • 인계 시 함께 넘길 것: accesses/docs Parquet, HotSet(B) 리스트(각 B), HR_bytes(B) 곡선, ρ_q 분포(스트림별). ρ_q 분포가 bimodal이면 정적 라우팅으로 충분하고, 넓게 퍼져 있으면 동적 scheduling 연구의 여지가 크다는 신호다 — 이 관찰 자체가 다음 단계 연구의 motivation 그림이 된다.

쟁점과 불확실성

  1. query 반복성 (최대 쟁점): 공개 MS MARCO query set은 dedup되어 있어(배경지식) S_uniq에서는 문서 재사용이 candidate 중복에서만 발생한다. 운영 검색은 query 반복(Zipf)이 강해 locality가 훨씬 크다. S_zipf의 α_q 선택은 결과를 직접 좌우하므로 근거 로그(ORCAS 등) 통계 확인 필요, sweep으로 방어. 리뷰어 관점에서 "S_uniq에서도 HR이 의미 있게 나오는가"가 논문의 방어선이다.
  2. 접근 정의(H1 vs H2): H1 기준으로 재면 traffic이 수십 배 과대평가된다(대부분 centroid-only로 pruning되어 residual을 안 읽음). 본 설계는 byte 지표를 H2로 고정했지만, IVF list 읽기까지 CSD로 옮기는 설계라면 metadata 접근 스트림을 별도로 측정해야 한다 — 범위 밖으로 명시.
  3. PLAID 파라미터의 정확한 값 — 확인 필요: k별 preset(nprobe, ndocs), stage-3 이후 최종 decompress 후보 수(ndocs의 1/4 수준이라는 기억은 미검증), hook을 걸 정확한 함수 위치. 논문 §알고리즘과 repo 코드로 검증 필요.
  4. 수치 전반 — 확인 필요: 8.8M passages, query set 크기(6,980 / ~101k / ~503k), dim 128·2-bit·36B/embedding, 총 embedding 수 ~600M, index ~22GB, fp16 원본 ~154GB, 평균 doclen ~68 — 모두 배경지식이며 원문 미열람. 본문 채택 전 논문 표에서 재확인할 것.
  5. granularity: pid 단위 HR과 block 단위 HR은 다르다(4KB block에 여러 passage 동거). hot passage들이 물리적으로 흩어져 있으면 block HR < doc HR. 반대로 hot 문서를 모아 재배치(layout reordering)하면 block HR을 끌어올릴 수 있다 — 측정 후 후속 최적화 아이템으로 분리.
  6. "document" vs "passage": MS MARCO passage corpus에서 locality의 단위는 passage다. 같은 원문서(URL)의 passage를 묶으면 spatial locality가 추가로 나올 수 있으나(MS MARCO document corpus 매핑 필요), 1차 범위에서는 passage=document로 둔다.
  7. LRU byte-weighted stack distance는 근사: 가변 크기 객체의 정확한 LRU byte hit ratio와 약간 다르다. passage 크기 분산이 작아 오차는 작을 것(추정)이나, 최종 수치는 실제 캐시 시뮬레이션으로 교차 검증 권장.
  8. 참고문서 5건 전부 "URL 메타데이터만 확인": 이번 세션에서 WebFetch/WebSearch가 거부되어 본문 검증을 못 했다. 위 확인 필요 항목의 1차 해소 수단이 이 5개 URL이다.
