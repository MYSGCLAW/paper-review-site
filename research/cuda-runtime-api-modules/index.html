핵심 요약

  1. Runtime API는 "약 33개 module"의 평면 나열처럼 보이지만, 실제로는 6개 축(device/context, 실행, 메모리, 동기화, 프로그램 표현(graph), 외부 리소스)으로 접힌다. [B] 이 6축이 그대로 일반 GPU 런타임의 object model이 된다 → 정보 지도 §9.
  2. Runtime API와 Driver API의 진짜 차이는 "context를 누가 소유하는가"다. [B] Runtime은 device당 primary context를 암묵 생성·참조 카운트로 관리하고, Driver는 context를 사용자에게 명시 핸들로 준다. 나머지 차이(모듈 로딩, 커널 launch 문법)는 이 결정의 파생물이다.
  3. CUDA 12.0 이후 Runtime API는 "드라이버의 명시성"을 되찾는 방향으로 움직였다. [B?] Library Management(cudaLibraryLoadData, cudaKernel_t)와 Driver Entry Point Access(cudaGetDriverEntryPoint)는 "런타임이 숨겼던 것"을 다시 핸들로 노출한 사례다.
  4. 동기/비동기 판정은 함수 이름(Async 접미사)으로 결정되지 않는다. [B] cudaMemcpy는 host 기준 blocking이지만, pageable 메모리를 쓴 cudaMemcpyAsync는 사실상 동기적으로 동작할 수 있고, 작은 H2D 복사는 반대로 cudaMemcpy여도 비동기처럼 반환할 수 있다. 이 노트에서 실무 영향이 가장 크고 동시에 오류 위험도 가장 큰 지점.
  5. NULL(legacy default) stream은 런타임 설계의 대표적 실패 사례다. [B] 다른 blocking stream들과 암묵적으로 동기화되므로, 라이브러리를 조합하면 stream 오염이 발생한다. per-thread default stream(cudaStreamPerThread)과 cudaStreamNonBlocking은 이 실수의 사후 보정이다. → 새 런타임은 명시적 queue 핸들 + 기본값 non-blocking으로 시작해야 한다.
  6. Graph Management는 CUDA가 "API 호출 스트림"에서 "제출 가능한 프로그램"으로 이동한 사건이다. [B] capture → instantiate → (update) → upload → launch 라는 lifecycle은 PJRT의 compile → load → execute와 구조적으로 같다. 여기에 device-side launch와 conditional node가 더해지면 제어 흐름까지 표현하는 진짜 IR이 된다. [B?]
  7. Stream Ordered Memory Allocator(cudaMallocAsync/cudaFreeAsync + memory pool)는 런타임 설계에 가장 직접 이식할 만한 부분이다. [B] "할당을 stream 순서에 태운다"는 아이디어가 캐싱 할당자와 graph 통합 문제를 동시에 푼다.
  8. 에러 모델은 sticky/non-sticky 이원 구조이며, sticky error는 사실상 context 파괴다. [B] cudaGetLastError가 상태를 소비(clear)한다는 점 때문에, 이 API는 라이브러리 경계에서 매우 다루기 어렵다.
  9. job.json이 언급하는 Execution Context Management / Logical Endpoint / Error Log Management 3종 module은 이번 수집에서 확인하지 못했다. [J] 특히 Logical Endpoint가 멀티노드 fabric 관련이라면 object model에 "원격 endpoint" 축이 추가되어야 하므로, 1순위 확인 항목이다.
  10. 결론: 이 노트는 골격(지도·모델·쟁점)으로는 지금 바로 쓸 수 있으나, 개별 API 이름과 deprecated 표시를 그대로 인용해서는 안 된다.

3-4. Execution Context Management / Error Log Management [J — 미확인]

job.json이 이 두 module을 명시하지만 본 수집에서 확인하지 못했다. 가설:

  • Execution Context Management: Runtime이 그동안 숨겨온 primary context를 명시 핸들로 노출하려는 시도일 가능성. 사실이라면 §2의 "context 소유권" 축이 무너지므로 Driver/Runtime 비교표를 다시 써야 한다.
  • Error Log Management: sticky error 외에 구조화된 로그 채널을 주려는 시도일 가능성. 사실이라면 §3-2의 에러 모델이 3계층이 된다.

두 가설 모두 근거 없음. 확인 전까지 설계 결정의 입력으로 쓰지 말 것.

  • capture 중에는 동기 API를 호출할 수 없다 (cudaStreamSynchronize, cudaMemcpy 등). 캡처 대상 코드 전체가 "비동기 순수" 해야 한다. [B]
  • capture mode(Global/ThreadLocal/Relaxed)는 다른 스레드의 CUDA 호출까지 차단할지를 정한다. 멀티스레드 프레임워크에서 Global은 위험. [B?]
  • cudaGraphExec_t는 동시에 여러 번 launch할 수 없다 (같은 exec의 재진입 불가로 추정). [확인 필요] [B?]
  • cudaGraphExecUpdate는 topology가 같을 때만 성공한다. "topology 동일"의 정확한 정의가 동적 형상 워크로드의 성패를 가른다 → 후속 질문 3.

런타임 설계 관점: capture는 "기존 명령형 API를 그대로 두고 IR을 추출하는 트릭" 이다. 새 런타임을 설계한다면 처음부터 프로그램을 선언적으로 제출받는 편이 낫지만, 기존 코드베이스를 흡수해야 하는 상황에서는 capture가 유일하게 현실적인 마이그레이션 경로다. 이 트레이드오프가 CUDA graph 설계의 본질.

9-3. 새 런타임을 설계한다면 CUDA에서 가져올 것 / 버릴 것

가져올 것 [B]

  1. stream-ordered allocation (§6-3) — 할당을 시간축에 태우는 아이디어.
  2. cudaLaunchAttribute 식 key-value 확장 (§5-1) — C ABI를 깨지 않고 기능 추가.
  3. external semaphore signal/wait (§8-1) — 호스트 왕복 없는 런타임 간 동기화.
  4. object 핸들 방식 (§8-3) — 전역 바인딩 금지.
  5. graph의 instantiate/update 분리 (§7) — 컴파일과 파라미터 갱신의 분리.

버릴 것 [B]

  1. NULL default stream의 암묵 동기화 — 기본값을 non-blocking으로.
  2. cudaGetLastError의 상태 소비 방식 — 에러는 호출 반환값으로만.
  3. thread-local current device — 명시 핸들(client)로.
  4. 원시 포인터 buffer — 최소한 크기·소유자를 아는 핸들로.
  5. 동기 할당이 device 전체를 멈추는 것 — 애초에 pool 기반으로 시작.

쟁점과 불확실성

A. 설계 쟁점 (판단이 갈리는 지점)

쟁점한쪽 주장반대쪽 주장현재 판단근거
A-1. stream을 사용자에게 노출할 것인가CUDA: 명시적 cudaStream_t로 최대 제어권 + 성능 상한PJRT류: stream을 런타임 내부에 두고 사용자에겐 executable 실행만 노출CUDA 쪽이 성능 상한은 높지만, 라이브러리 조합 시 stream 오염·NULL stream 함정이 반복 발생한다. 런타임을 새로 설계한다면 "명시적 queue 핸들 + 안전한 기본값(non-blocking)" 이 절충안[B]
A-2. graph를 쓸 것인가launch 오버헤드가 지배적인 소규모 커널 파이프라인에서 큰 이득캡처 제약(동기 API 금지), 파라미터 갱신 비용, 디버깅 난이도반복 형상이 고정된 추론 루프에는 강하게 권장. 형상이 매 스텝 바뀌면 cudaGraphExecUpdate 실패 비용이 커진다[B?]
A-3. cudaMallocAsync vs 프레임워크 캐싱 할당자런타임 pool이 표준·이식성·graph 통합에서 유리PyTorch 등은 자체 할당자가 이미 최적화되어 있고, 두 pool이 공존하면 단편화·이중 캐싱 발생공존이 진짜 문제다. release threshold 튜닝 없이 섞으면 메모리 사용량이 늘어난다 (배경지식, 실측 필요)[B?]
A-4. managed memory를 쓸 것인가코드가 단순해지고 oversubscription 가능페이지 폴트 비용, concurrentManagedAccess 미지원 환경 제약프로토타이핑/대용량 oversubscription 외에는 명시적 복사가 여전히 우세[B?]
A-5. Runtime API vs Driver API 중 무엇 위에 런타임을 얹을 것인가Runtime: 개발 속도, nvcc 통합Driver: context·모듈 로딩 명시 제어, 다중 context 격리프레임워크/런타임 작성자라면 Driver API. 단 Library Management 등장으로 Runtime의 격차가 좁아지는 중[B?]
A-6. 에러를 프로세스 치명으로 볼 것인가sticky error 후 복구는 신뢰할 수 없으니 fail-fast서빙 환경에서 프로세스 재시작 비용이 큼 → 복구 시도 필요현재로선 fail-fast가 정직하다. cudaDeviceReset 후 재초기화의 신뢰도는 미검증 → 후속 질문 5[B?]
